1. Executive summary

Verdict. Optimystic's core primitives are well-chosen and, on the happy path, well-built: content-addressed versioned blocks, deterministic-replay validation, Ed25519-signed two-phase commit, a genuinely transport-free db-core, and unusually honest, high-quality documentation. The maturity gap is systematic and predictable: failure paths, adversarial paths, and rarely-exercised branches are markedly less sound than the happy path. The most serious issues cluster at trust boundaries (signatures verified against unbound keys, unverified peer claims driving restoration), at protocol seams (cluster membership never agreed upon, multi-collection commit not atomic, read dependencies lost on cache hits), and in rare-branch logic (B-tree rebalancing). None of these require a redesign — but several must land before the system faces churn, crashes, or adversaries.

The ten findings that matter most

  1. Consensus signatures aren't bound to peer identities. Cluster vote signatures are verified against public keys carried inside the record itself, never checked against the peer ID. A malicious coordinator can invent a cohort, sign all the votes itself, and every honest member accepts a forged super-majority. → P2P #1
  2. Cluster membership is a per-coordinator local view, not an agreed set — and record.peers is not in the message hash. Every safety theorem quantifies over "the cluster," but nothing agrees on who the cluster is; churn produces spurious Peers mismatch aborts at best and split quorums at worst. → Design H1
  3. Snapshot isolation is porous: cache hits record no read dependency. Reads served from CacheSource never reach TransactorSource, the only place a ReadDependency is recorded — so OCC validation never sees them. → Txn #1
  4. The B-tree has four real corruption/wrong-result bugs in rebalance and iteration edge cases (borrow-from-right partition key, ancestor separator corruption on deep merges, end-of-leaf crack returning empty range scans, missing awaits). A property/fuzz test would have caught all of them. → Foundations #1–5
  5. Multi-collection transactions can half-commit and report failure. Commits run sequentially per collection with no cross-collection 2PC journal; on mid-loop failure the already-committed collections stand while the client is told the transaction failed. → Txn #3, Design H4
  6. Byzantine tolerance is actually Byzantine detection. Dispute escalation is disabled by default, single-round, post-commit, and never wired into the live node; arbitrators are selected from the same XOR-neighborhood an attacker would capture. → Design H2/H3, P2P #3
  7. Storage can silently diverge or permanently corrupt. get()-driven promotion bypasses the per-block commit latch; commits accept non-contiguous revisions on stale bases; pends seed an open-ended ranges: [[0]] that disables restoration; the fs adapter's non-atomic writes make one crash permanently unreadable. → Storage #1–4
  8. The SQL layer returns wrong results. PK range filters are claimed as handled but never applied (WHERE pk > x returns every row); the tree comparator is raw lexicographic so INTEGER/DESC/collated PKs mis-sort — while the correct comparator exists as dead code. → SQL #1–2
  9. The cohort-topic renew path has no replay protection, and wire numerics are unbounded. A captured signed withdraw is valid forever; unvalidated treeTier/ttl allow unbounded attacker-keyed engine creation and unevictable topics. → Cohort #1–4
  10. Session-built transactions can't validate remotely, and a transient apply fault drops a transaction forever. engineId: 'unknown' means every validating node would reject session transactions; and the persistent "executed" marker is never rolled back after an apply failure, so redelivery is skipped permanently. → Txn #7, P2P #2

Cross-cutting themes

  • Happy path ≫ failure path. Coordinators, trackers, and stores handle the main line carefully; the crash, retry, partial-failure, and adversarial branches contain most of the HIGH findings. The pattern repeats in every subsystem.
  • Advertised guarantees run ahead of the implementation. correctness.md's theorems describe target mechanisms (dispute cascade, multi-collection recovery, client signatures) that are unimplemented or disabled by default. architecture.md §Status is honest about this; correctness.md is not annotated the same way and over-credits the shipping system.
  • The bugs live where the type system was turned off. The seam files — libp2p-node-base.ts (28 as any, six empty catch {}) and network-transactor.ts (13 as any) — are exactly where load-bearing wiring silently fails.
  • Duplication that must-stay-identical has already drifted. Wire validators exist in three copies; the index-value serializer in two (and the drift is finding SQL #3); operations hashing is duplicated between coordinator and validator with no canonical ordering; four storage adapters are ~150-line near-clones.
  • Unbounded growth in several ledgers. No sweep/archival exists (storage grows O(revisions) forever); tracker op-lists grow unboundedly per hot block; several attacker-keyed maps (replay guard, membership verifier, cold-start forwarders) have no caps.

2. Design-level review (distributed-systems assessment)

Docs reviewed: architecture, transactions, correctness, right-is-right, arachnode, repository, crdt-sync, partition-healing, cohort-topic — each spot-checked against the implementation. "Design flaw" = unsound even if built as written; "drift" = docs promise what code doesn't do.

HIGHCluster membership is a unilateral coordinator view — the foundation of every safety theorem is unagreed design flaw

correctness.md §2 and Theorem 1 assume "the cluster responsible for B" is a well-defined set with an honest majority. In code, the set is whatever the coordinator's own findCluster(blockId) returns at transaction start (packages/db-p2p/src/repo/cluster-coordinator.ts:126-166), frozen into record.peers. The message hash covers record.message only — not record.peers (createMessageHash, :117-121). Two coordinators (or one, before/after churn) can drive the same messageHash with different peer sets; ClusterMember.mergeRecords hard-throws 'Peers mismatch' on divergence (cluster/cluster-repo.ts:438-440), so honest members reject each other rather than converge. Under partition, a minority re-derives a smaller local cluster and can reach 75% of that — Theorem 2's arithmetic assumes a shared K that nothing enforces, and the validateSmallCluster backstop fails open when FRET confidence is low (:375-379), precisely the condition a partition induces.

Make membership part of what is agreed: fold a membership epoch/hash into the consensus record (include record.peers in the signed message hash), or anchor cluster identity on the threshold-signed MembershipCertV1 machinery that already exists in cohort-topic. Treat this as the highest-priority design item — the dispute cascade is meaningless if the base cluster set isn't agreed.

HIGHByzantine tolerance is detection-only; the escalation that would upgrade it is off by default, single-round, and post-hoc drift + design

The docs promise "any validity disagreement blocks the transaction and triggers cascading geometric escalation." The code does the opposite: with a minority rejection, ClusterCoordinator.executeTransaction commits anyway at 75% super-majority and merely sets record.disputed = true (cluster-coordinator.ts:299-332). The dispute service defaults to disputeEnabled: false (dispute/types.ts:124); even when enabled, dispute-service.ts:160-222 selects one ring of arbitrators, votes once, and resolves — no recursion, no geometric widening, and it runs after commit. The Adaptive-BFT cost model (Theorems 8, 10) describes a mechanism that is not implemented. Today's real guarantee is "Byzantine-detectable within a cluster that has an honest super-majority" — strictly weaker than the advertised f<N/2 tolerance.

Short term: annotate correctness.md theorem-by-theorem with target-vs-current (as architecture.md §Status already does) so the formal claims aren't over-read. Long term: implement the synchronous, cascading escalation — but only after H1, since the cascade inherits the membership problem.

HIGHArbitrator independence is defeated by ID-proximity selection design flaw

dispute/arbitrator-selection.ts:10-28 picks arbitrators as ring positions K+1…2K by XOR distance to the block ID — the peers immediately adjacent to the (possibly captured) cluster. An attacker who can place node IDs near a block to capture its cluster (the Sybil vector correctness.md §7.1 admits) captures the next ring just as cheaply. Geometric widening only helps if each wider ring samples a more representative population; concentric XOR-neighbors walk through the attacker's owned region first.

Sample escalation audiences from an unpredictable, non-local population — e.g. hash(blockId ‖ escalation-round ‖ epoch) → pseudo-random ring coordinates — so each round's audience is drawn from the whole keyspace rather than the attacker's neighborhood.

HIGHMulti-collection atomicity (Theorem 3) is not realized — sequential commits, no un-commit, no recovery journal drift

TransactionCoordinator.commitPhase (packages/db-core/src/transaction/coordinator.ts:732-804) loops over collections calling transactor.commit one at a time; on mid-loop failure, cancelPhase excludes already-committed collections (:604-610, 812-821) — those commits stand. Commit can genuinely fail after successful PEND (a racing transaction advances the log tail between prepare and commit), so partial commit is reachable in normal operation. The persisted-state recovery Theorem 3 relies on lives per-cluster in cluster-coordinator.ts and only resumes the broadcasting phase (:775-805); there is no multi-collection 2PC journal at all.

Either give the coordinator a durable cross-collection decision record (real 2PC over the supercluster), or explicitly downgrade Theorem 3 to "atomic-intent, eventual-visibility" per the crdt-sync.md direction — which is the more honest framing of what the code does. Either way, surface committedCollections in the failure result so callers know reconciliation is required.

MEDIUMHot log-tail is a structural per-collection throughput ceiling design limit

Every write to a collection funnels through one critical cluster; conflict losers are rejected outright (resolveRace, cluster-repo.ts:1142-1156). The acknowledged fix (crdt-sync.md HLC redesign) is entirely unimplemented. A correct CP tradeoff, but there is no throughput story for a hot collection today beyond "shard into more collections."

Document the sharding guidance for application authors now; sequence the crdt-sync migration stages when contention becomes real.

MEDIUMOCC starvation is real and systematically biased against large transactions design limit

Theorem 9 admits no deterministic starvation prevention. Block-granular read-dependency capture (including B-tree interior and log-chain nodes) makes large transactions structurally more likely to observe stale reads and lose races indefinitely to short ones — a fairness inversion, honestly flagged, with no fence primitive built.

Ship a coordinator-side exponential backoff-with-priority-aging default rather than deferring entirely to applications; consider excluding purely structural (interior-node) reads from the conflict set where safe.

MEDIUMArachnode ring-shifting omits the hard part: safe responsibility handoff design gap

arachnode.md describes nodes shifting rings on capacity/demand but no protocol for atomic responsibility transfer — a node moving outward sheds half its keyspace, and if inward/outward references haven't loaded that slice elsewhere, those keys lose a replica. ringDepth = ceil(-log2(capacity/demand)) is an undamped reactive control loop — a classic thrashing setup near ring boundaries. Mostly not-yet-implemented, but the design itself doesn't address handoff.

Specify a two-phase handoff (advertise-new-ring → confirm-replication → release-old-range) and add hysteresis/damping to the ring-depth signal before building the migration machinery.

MEDIUMClients are unauthenticated; transactions carry no client signature drift

right-is-right.md §Client Signatures and the recovery scenarios assume client-signed transactions; TransactionStamp has peerId but no signature field, and no client-signature verification exists on the pend/commit path. Until this lands, the "bad client gets isolated" story and Theorem 13's recovery anchoring are incomplete. Flagged as target in the docs.

Add a signature over the stamp + statements to TransactionStamp and verify at pend; this also unblocks the dispute-recovery paths that need to authenticate the original transaction.

LOWAssorted design notes design

Operations-hash spec: validator agreement rests on identical hashing, but transactions.md:792-801 shows a non-cryptographic 32-bit example hash; the canonical operation serialization should be specified and versioned with engineId (the implementation-level ordering bug is Txn #10). Threshold defaults diverge: ClusterMember.superMajorityThreshold defaults to 1.0 (cluster-repo.ts:192) while the coordinator policy defaults to 0.75 (coordinator-repo.ts:92) — a latent phase-disagreement misconfiguration; enforce config coupling. Reads are soft: non-responsible reads warn and serve anyway with a 10 s lazy read-repair window — fine per Theorem 15, a footgun for app authors who haven't read it. Ejection is local: reputation lives per-node with decay; an "ejected" peer reconnecting to fresh nodes carries no scarlet letter (right-is-right.md asks this open question itself).

Design assessment. Taken as "a CP, log-tail-ordered, optimistic-concurrency database with Byzantine detection inside honest-majority clusters," Optimystic delivers, and the code backs the claim — deterministic-replay validation, verified vote signatures (modulo P2P #1), and real equivocation detection. The weaknesses concentrate where the design leans on assumptions it doesn't enforce: cluster identity (H1) and the tolerance-vs-detection gap (H2/H3). Recommendation: fix membership agreement first, then either implement or honestly downgrade the multi-collection atomicity claim, and annotate correctness.md with target-vs-current so its theorems aren't read as descriptions of the shipping system.

3. db-core foundations (blocks · btree · chain · transform · log · collections · utility)

HIGHB-tree borrow-from-right writes the wrong partition key — borrowed entry becomes unreachable bug btree/btree.ts:658-664

In rebalanceLeaf, borrowing the right sibling's first entry updates the parent separator with the key of the entry that was moved out, not the right sibling's new first key. Since indexOfKey sends key === partition to the right child (:381), a subsequent find(borrowedKey) descends into the right sibling — where the entry no longer lives — and misses. The branch-level borrow (:714-721) does it correctly, confirming the leaf case is the anomaly. Current tests merge rather than borrow, so this is untested territory.

After the two apply calls, use this.keyFromEntry(rightSib.entries[0]!) (the sibling's post-shift first entry) as the new partition key.

HIGHrebalanceBranch right-merge corrupts an ancestor separator when merging at index 0 bug btree/btree.ts:741-743

After merging the right sibling into a branch at pIndex === 0, the code writes pNode.partitions[0] (post-delete: the separator to the merged child's next sibling — strictly greater than the merged subtree's minimum) into the nearest ancestor separator. Any key in [subtreeMin, pNode.partitions[0]) then routes into the wrong subtree and is unreachable. The subtree minimum is unchanged by an internal merge, so no update is needed at all. Requires a 3+ level tree (~2K+ entries) under delete load — beyond current test sizes.

Delete the if (pIndex === 0 …) updatePartition(...) block.

HIGHinternalNext can't advance off an end-of-leaf crack — range scans silently return nothing bug btree/btree.ts:391-397

For an off path, internalNext only tries to "get on" at the current position; when leafIndex === entries.length it returns without falling through to the leaf-advance logic. Such a crack is routinely produced by find(key) where the key falls between a leaf's last entry and the parent partition (e.g. a just-deleted leaf-max value). Consequence: range({ first: <missing key at leaf end> }) yields zero results even though later entries exist in the right sibling. internalPrior handles the symmetric crack correctly (:428, :450-453).

When the get-on attempt fails and leafIndex >= entries.length, fall through to the branch-popping advance instead of returning.

HIGHTwo missing awaits: prior() returns an un-moved path; internalUpdate fire-and-forgets a structural delete bug btree/btree.ts:278-282, 463

prior() calls this.movePrior(newPath) without await (compare next() at :265-269): it "works" mid-leaf because the decrement is synchronous, but crossing a leaf boundary awaits a store read — the caller receives the path before it moves, and it then mutates underneath them. Exposed publicly via Tree.prior. Separately, internalUpdate's key-changing path runs this.internalDelete(await this.find(oldKey)) un-awaited — the delete's async rebalance interleaves with the immediately following find(newKey) on the same nodes, and any rebalance error becomes an unhandled rejection that also defeats AtomicProxy rollback.

Add both awaits.

HIGHfilterConflict replacement actions are silently discarded bug collection/collection.ts:133-134, 327-338

doFilterConflict queues a replacement by pushing onto this.pending — but it is invoked from inside this.pending.map(...), whose result is then assigned back to this.pending. The push landed on the old array beyond map's fixed range and is dropped by the reassignment. Any collection relying on filterConflict returning a rewritten action loses that action entirely — this is the documented conflict-merge extension point.

Collect replacements into a local array (return {keep, replacement} from the filter) and append them to the new this.pending after the map completes.

MEDIUMChain.add passes the wrong predecessor to newBlock — Log priorHash breaks across multi-block appends bug chain/chain.ts:124 + log/log.ts:231-236

Inside the append loop every new block gets newBlock?.(newTail, oldTail) — always the original tail — instead of the running predecessor. A single variadic add(...entries) spanning two+ blocks hashes the wrong predecessor, silently breaking the hash chain (latent today since Log appends one entry at a time). Additionally the hash is computed before nextId is applied to the predecessor, so stored bytes never match hashed content unless verifiers strip link fields — undocumented.

Pass the loop-local tail; specify a canonical hash payload excluding mutable link fields.

MEDIUMsyncInternal can spin forever on persistent stale failures, holding the collection latch robustness collection/collection.ts:245-281

The retry loop has no attempt cap, no growing backoff (fixed 100 ms, and only when staleFailure.pending), and no abort signal. A transactor that persistently rejects turns sync() into a hot infinite loop that starves all act()/update() callers.

Bounded retry budget (count or deadline) with exponential backoff; surface a typed error when exhausted.

MEDIUMTracker.tryGet gives source blocks precedence over tracked inserts — delete-then-reinsert reads stale content bug transform/tracker.ts:14-27

The inserts map is consulted only on source miss. If a block exists in the source, is delete()d, then re-insert()ed, tryGet returns the source block, ignoring the inserted replacement and any ops routed into it — contradicting the transform semantics (struct.ts:4-5: insert wins).

Check transforms.inserts before falling back to the source.

MEDIUMReopened trees have weaker atomicity than freshly created ones design collections/tree/tree.ts:45-54, 69-70

Collection.internalTransact hands each action handler an Atomic precisely so failures leave no partial state — but the tree's "replace" handler ignores it and mutates through the captured outer btree. The reopen path constructs new BTree(collection.tracker, …) without an AtomicProxy (only BTree.create installs one), so on an existing collection a mid-action failure leaves half-applied node mutations staged with no rollback.

Construct the reopened BTree over an AtomicProxy(collection.tracker), or make the handler operate through the passed _trx store.

MEDIUMmergeTransforms clobbers instead of merging per-block ops; AtomicProxy breaks under interleaved atomic() calls; branch left-merge path index off by merged length bug/design

(a) transform/helpers.ts:74-80: updates: { ...a.updates, ...b.updates } replaces a block's whole op array rather than concatenating; safe for the current per-block-disjoint caller, but the general-sounding helper silently loses ops for any future overlapping use. (b) transform/atomic-proxy.ts:32-48: re-entrancy detection (_active !== _base) can't distinguish nesting from a second concurrent atomic(); two overlapping un-awaited inserts on a bare BTree share one Atomic, and the first commit flushes the second's half-finished mutations (class advertises "re-entrant safe"). (c) btree/btree.ts:751-755: the branch left-merge adjusts pathBranch.index after the merge already appended nodes, shifting the index by the merged length; the leaf-level equivalent (:688-690) gets the order right.

(a) Merge per block id and dedupe deletes. (b) Serialize atomic() behind a promise-queue mutex or throw on foreign overlap. (c) Capture leftSib.nodes.length before the apply calls.

MEDIUMRead path replays the full accumulated op log and deep-clones on every tryGet perf transform/tracker.ts:16-18, transform/cache-source.ts:33

Every tracked read structuredClones the block, then re-applies all ops ever recorded for it (each op cloned again in applyOperation). Hot blocks updated once per operation — the collection header, the log tail — accumulate unbounded op lists in a long-lived tracker: each read is O(ops), a session O(ops²), multiplied by tree depth on descents.

Compact per-block ops on write (materialize into a shadow updated-block cache past a threshold), or memoize the transformed block per id and invalidate on new ops.

LOWAssorted polish cleanliness

utility/pending.ts:7-9 — isResponse uses response !== undefined, so a Promise<void> never reads complete (affects incompleteBatches); track settlement with a flag. btree.ts:186 awaits a sync callback; :514 vs :594 inconsistently await a void insert; :656/666/712 non-null-assert legitimately-undefined edge values. tracker.ts:70-72 double-wraps an already-deduped Set. log/log.ts:164-183 per-entry unshift is O(n²) over the unsynced tail — push then reverse. collection.ts:301 reverses an array in place where toReversed() is clearer.

Subsystem assessment. Thoughtfully engineered layering with unusually good doc comments; solid coverage of the B-tree's common paths. The weak spot is exactly where testing thins: rare rebalance branches and iteration cracks hold four real correctness bugs, and two missing awaits undermine an otherwise careful atomicity story. Highest-leverage single action: a property/fuzz test (random insert/delete/find/range vs a reference sorted map, at multiple tree heights) — it would have mechanically caught findings #1, #2, #3, and the merge-index bug.

4. Transaction layer (db-core transaction · transactor · network · cluster)

HIGHRead-dependency capture misses all cache hits — a snapshot-isolation hole bug transactor/transactor-source.ts:33 + transform/cache-source.ts:20-34

TransactorSource.tryGet is the only place a ReadDependency is recorded, but reads flow Tracker → CacheSource → TransactorSource, and CacheSource returns cached blocks without calling the underlying source. Any block served from cache (read in a prior transaction on the same Collection instance, or refreshed via applyCommittedToCache) produces no read dependency — the validator's stale-read check never sees it, so a transaction can read stale cached state and still pass OCC validation. Snapshot isolation is currently enforced by luck of cache misses.

Record read dependencies above the cache: have CacheSource report every read (id + cached rev) to a listener, or capture deps in Collection/Tracker.tryGet.

HIGHpendPhase leaks pended actions when transactor.pend throws bug transaction/coordinator.ts:706-716

Cleanup of already-pended collections runs only when pend returns {success:false} — but NetworkTransactor.pend throws on non-stale failures (network-transactor.ts:489), and pendPhase has no try/catch. Earlier collections' successful pends are never cancelled and block other writers until TTL expiry.

Wrap the per-collection pend loop in try/catch and run the same cancel-already-pended cleanup on throw before rethrowing.

HIGHMulti-collection commit half-commits, then cancels the rest and reports failure design transaction/coordinator.ts:604-612, 745-801

commitPhase commits collections one at a time; if collection N fails after three attempts, coordinateTransaction cancels only the un-committed collections and reports the transaction failed — while earlier collections' commits are durable. The GATHER supercluster nominees are threaded into pend but play no role in making tail commits atomic. The client is told "failed" for a transaction that half-committed. (Design-level counterpart: Design H4.)

Real 2PC over the supercluster (all tails commit or none, durable decision record), or at minimum return committedCollections in the error so callers know reconciliation is required instead of assuming a clean abort.

HIGHTransactionSession.execute double-applies actions on the engine-translation path bug transaction/session.ts:90-98 + transaction/actions-engine.ts:51

When no pre-computed actions are passed, session.execute calls engine.execute(tempTransaction) and then applies the returned actions again via coordinator.applyActions — but ActionsEngine.execute already applies each statement's actions through the coordinator itself. Every statement executed via a session + ActionsEngine without explicit actions is applied twice. All tests pass explicit actions, so the path is untested and broken; the same tension exists in the deprecated TransactionContext.addAction path.

Make engines side-effect-free translators (return actions; application belongs solely to the session/coordinator); bind the engine to the isolated validation coordinator on the validator path.

HIGHFire-and-forget cancel can crash the process on rejection bug transactor/network-transactor.ts:573, 480

Promise.resolve().then(() => this.cancel({...})) has no .catch; cancel() calls findCoordinator, which rejects exactly when commit just failed (peers unreachable) → unhandled promise rejection, fatal in modern Node. The analogous void Promise.resolve().then(() => this.cancelBatch(...)) at :480 has the same shape.

Append .catch(e => log('cancel failed: %o', e)) to both.

MEDIUMcommitBlock non-null-asserts optional missing — TypeError masks the real stale failure bug transactor/network-transactor.ts:577

.flatMap(b => (b.request!.response! as StaleFailure).missing!) — but StaleFailure.missing is optional (network/struct.ts:61). A reason-only stale failure yields undefined elements and distinctBlockActionTransforms destructures them → TypeError. The pend path already filters (:486); commit doesn't.

Add the same .filter((x): x is ActionTransforms => x !== undefined).

MEDIUMSession transactions carry engineId: 'unknown' — remote validation is dead on arrival bug transaction/session.ts:53 + transaction/validator.ts:70-75

TransactionSession.create hardcodes 'unknown' (marked TODO), but validators resolve engines by stamp.engineId and pendPhase always attaches the transaction to the PendRequest. Any validating cluster node will reject every session-built transaction with "Unknown engine: unknown". It only works today because storage-only nodes skip validation — the security/validation path can never engage. Related placeholders: coordinator.ts:354-358 ('local', '').

Expose id on ITransactionEngine, require it in createTransactionStamp, and remove the placeholder defaults.

MEDIUMFailed commit() leaves the staged log entry; retry double-logs; rollback can no-op bug transaction/coordinator.ts:147-158, 223-224

commit() appends a log entry into each collection's tracker before consensus; on failure it throws without undoing the append, and a legal retry of session.commit() appends a second entry for the same actions. For direct-staged trees, rollback(stampId) finds no stampData and returns immediately, leaving poisoned tracker state with no cleanup path.

Snapshot tracker state before the log-append loop and restore it on any failure path.

MEDIUMcommitPhase retries a commit the transactor already auto-cancelled bug transaction/coordinator.ts:786-800 + network-transactor.ts:573

On failed commit, NetworkTransactor.commitBlock immediately fires cancel(...) for the whole action — yet commitPhase retries the same request up to three times, racing against (or following) the cancel of the very pend it needs. Stale failures are also permanent, making blind identical retries wrong in both cases.

Retry only thrown/transient errors; suppress the transactor's internal auto-cancel while a coordinator-level retry policy is active (or own retries in exactly one layer).

MEDIUMOperations hash is non-canonical — honest validators can diverge design transaction/coordinator.ts:162-174 vs transaction/validator.ts:149-170

The hash is JSON.stringify over ops in Map-insertion order: sender iterates action/collection-data order, validator iterates its own collection-registration order. Nothing canonicalizes collection order, per-collection block order, or key order — an honest re-execution can produce a different hash → spurious "Operations hash mismatch" rejections. The collect/hash logic is also duplicated verbatim between the two files, guaranteeing eventual drift.

Sort operations by (collectionId, blockId, type), use canonical JSON, and extract one shared hashing module used by both coordinator and validator.

MEDIUMPerf: authoritative "not found" triggers a full serial retry round; per-collection phases are serial; per-block findCluster uncached perf network-transactor.ts:118-158, 325-332, coordinator.ts:685-801

(a) hasBlockInResponse demands a materialized block, so a valid "block doesn't exist" response (normal for createOrOpen probing) is treated as retryable — a second full round of findCoordinator + get, serial per batch. (b) pendPhase/commitPhase await collections sequentially though pends are independent. (c) Every pend does per-block findCluster with no caching; commit re-resolves the same blocks, relying on a duck-typed as any hint to avoid re-routing.

Distinguish authoritative-absent from no-response; parallelize per-batch retries and per-collection pends (Promise.allSettled + cancel-all-on-failure); cache cluster lookups for the pend→commit window keyed by block id.

LOWAssorted: PeerId dedup by object identity; private-state pokes; dead deprecated path cleanliness

gatherPhase merges nominees into a Set<PeerId> but each is a fresh object from peerIdFromString — duplicates flow into every PendRequest (coordinator.ts:648-659); key by toString(). Coordinator reaches into Collection's private source via bracket access in five places, duplicating rev-bump logic; NetworkTransactor smuggles coordinatingBlockIds and recordCoordinator through as any (network-transactor.ts:434-455) — add the typed members. The deprecated TransactionContext/commitTransaction path (placeholder stamps, double-apply) is unused by production callers — delete or quarantine it. TransactorSource.tryGet's result[id]! TypeErrors on sparse results from non-Network transactors (transactor-source.ts:29-31).

Subsystem assessment. The batch/retry machinery is careful about root-cause preservation, and the tail-first commit protocol is correctly threaded. But the distributed failure paths lag the happy path badly: pends leak on throw, multi-collection commit half-commits, cleanup is fire-and-forget without rejection handling, and failed commits poison local staged state. Most critically, the headline guarantee is porous — read deps vanish on cache hits (#1) and the validation path is unreachable for session transactions (#7). There's also a two-generations-of-API smell (TransactionContext vs TransactionSession, duplicated hashing) worth consolidating before hardening, since several bugs live in those seams.

5. Cohort-topic substrate (+ matchmaking · reactivity, db-core side)

HIGHThe renew path has no freshness or replay protection — a captured withdraw is valid forever security cohort-topic/member-engine.ts:243, registration/renewal.ts:305-381

6. P2P network & consensus (db-p2p: libp2p wiring · cluster 2PC · dispute · repo · reputation · sync)

HIGHCluster vote signatures are verified against an unbound, attacker-supplied public key security cluster/cluster-repo.ts:578-589

verifySignature reads the signing key from record.peers[peerId].publicKey — data carried in the record itself — and never checks that the key corresponds to peerId. For libp2p Ed25519 identities the peer id is the multihash of the public key, so the binding is trivially checkable and is the entire basis of the scheme. Without it, a malicious coordinator can fabricate a cohort: invent N peer ids, attach keys it minted, sign every promise/commit itself — every honest member's validateSignatures passes and the member applies the operation as consensus-approved. It also enables reputation poisoning: forged equivocating records make honest nodes penalize honest peers (cluster-repo.ts:483-487, dispute-service.ts:394-407 key penalties on self-asserted ids).

In verifySignature, require the key to derive the peer id — compare peerIdFromString(peerId).publicKey?.raw (or derived multihash) to the embedded key bytes and reject on mismatch. This single check restores the meaning of every downstream signature and reputation decision.

HIGHPersistent "executed" marker is never rolled back — a transient apply fault permanently drops the transaction on that member bug cluster/cluster-repo.ts:753-768

handleConsensus fires stateStore.markExecuted(...) (fire-and-forget) then applies operations. On an unexpected throw it rolls back only the in-memory marker — the interface has no unmarkExecuted. On redelivery, wasTransactionExecutedAsync hits the durable marker → the member skips handleConsensus forever. The operation is silently dropped even though the rollback was written specifically to allow retry.

Add unmarkExecuted(messageHash) and call it in the catch — or persist the marker only after applyConsensusOperation succeeds (the synchronous in-memory guard already covers the apply-window race).

MEDIUMThe dispute subsystem is entirely inert in the live node design libp2p-node-base.ts:382-497, dispute/dispute-service.ts:130

DisputeService is constructed, but the /dispute/1.0.0 inbound handler (DisputeProtocolService) is never registered in the node's services, and initiateDispute is never called from the cluster path (only tests). MEDIUMNo message-size or frame-count caps on inbound consensus protocols security protocol-client.ts:149-163; cluster/service.ts:210-221; repo/service.ts:273-284; sync/service.ts:115-124; dispute/service.ts:101-112

MEDIUMRead-repair and reconcile trust unverified peer claims to drive restoration security repo/coordinator-repo.ts:265-326, libp2p-node-base.ts:577-600

queryClusterForLatest takes the max ActionRev any peer reports — no signature or commit-cert check — and fetchBlockFromCluster restores against it; a single lying peer steers spurious restoration and content acceptance. reconcileBlock likewise picks Math.max(revs) from cohort archives and saveReplicatedBlocks without content verification — a malicious cohort member can seed bogus replicas.

MEDIUMvalidateSmallCluster fails open; hot-path double peerStore reads correctness/perf repo/cluster-coordinator.ts:347-380, libp2p-key-network.ts:589, 627

LOWAssorted: untyped composition root; order-sensitive record equality; dead abort-signal plumbing cleanliness

7. Storage & adapters (db-p2p/storage · storage-fs · storage-ns · storage-rn · storage-web)

HIGHRead-driven promotion bypasses the per-block commit latch bug db-p2p/src/storage/storage-repo.ts:168

get() promotion calls internalCommit without the latch that commit(), saveReplicatedBlock, and the dispute path all hold — internalCommit's own comment asserts it runs "within the locked critical section of commit()". A promotion racing a concurrent commit interleaves two getLatest → save → promote → setLatest sequences: setLatest can regress non-monotonically and the writers can cross-write revs/<N> for different actionIds.

Wrap the promotion loop body in withBlockCommitLatch(blockId, ...) (already exported) and re-check the pending inside the latch.

HIGHfs adapter writes are non-atomic — one crash permanently corrupts a block integrity db-p2p-storage-fs/src/file-storage.ts:251-254, file-kv-store.ts:18-22

ensureAndWriteFile is truncate-then-write, no temp-file+rename, no fsync. meta.json is rewritten on every commit; a crash mid-write leaves a truncated file, and readIfExists rethrows the JSON parse error (only ENOENT maps to undefined) — getMetadata throws forever, and the block becomes permanently unreadable including by recover(), which reads metadata first.

Write to <path>.tmp then fs.rename (atomic on POSIX and NTFS), fsync for metadata; treat JSON parse failure as corrupt-but-recoverable instead of throwing indefinitely.

HIGHCommit accepts non-contiguous revisions — divergent materializations served silently integrity storage-repo.ts:385-411, 540-550

commit() only rejects latest.rev >= request.rev, never checks contiguity. A replica that missed intermediate commits accepts the pend and materializes the rev-N transform on top of its stale local base — a divergent materialization then served to readers and cached as rev N. Because pends seed ranges: [[0]] (next finding), the gap is also invisible to restoration.

Treat latest.rev < request.rev − 1 as a gap: trigger ensureRevision/restore for the predecessor or fail the commit with a distinct "behind" result.

HIGHPend seeds an open-ended ranges: [[0]] — an honest-coverage lie that disables restoration bug db-p2p/src/storage/block-storage.ts:57

savePendingTransaction claims every revision from 0 onward is locally present. Consequences: (a) ensureRevision's inRanges is always true, so the restore callback never fires — getBlock(rev) throws or silently serves an older revision; (b) it's the only thing making commits look "in range," since internalCommit never merges committed revs into meta.ranges. Contrast saveReplica/saveDeletion, which correctly seed [] and merge closed ranges.

Seed ranges: [] at pend; merge [rev, rev+1] at commit (setLatest/internalCommit) so ranges stay an honest statement of what is locally reconstructible.

MEDIUMNativeScript SQLite transactions can nest and cross-rollback — no mutex on the shared connection integrity db-p2p-storage-ns/src/ns-opener.ts:74-88

transaction() issues BEGIN/COMMIT on the single shared connection with no mutex, while StorageRepo allows concurrent commits on disjoint blocks. Two concurrent promotions nest BEGINs: the second errors, its catch runs ROLLBACK — rolling back the first transaction's writes; concurrent plain statements get swept into an open transaction and discarded with its rollback.

Serialize transaction() (and ideally all writes) behind an async mutex, or use BEGIN IMMEDIATE with a queued executor.

MEDIUMRestoration filters by the wrong coordinate space — "hash the block ID" doesn't hash bug storage/restoration-coordinator-v2.ts:136-153

extractBlockPrefix copies raw first bytes of the blockId string, but cohort assembly uses hashKey(blockIdBytes) and ring partitions are computed from hashed peer coordinates. filterByPartition therefore compares a raw-byte prefix against hashed-coordinate partitions — the entire inner-ring fallback path queries non-responsible peers and filters out responsible ones.

Compute the prefix from hashKey(encode(blockId)) — the same coordinate used for cohort assembly.

MEDIUMrecoverBlock has no production caller — the crash state it repairs permanently wedges the block bug storage-repo.ts:477-481

In the promote-landed-but-setLatest-lost crash state, a retry-commit hits "Pending action not found" (the pend was already promoted; the idempotent partition doesn't catch it since latest.rev < request.rev) — wedged until someone manually calls recover(), which only tests do.

In commit(), when the pend is missing but getRevision(request.rev) + getTransaction show the action durably promoted, call storage.recover() and re-partition — or run recover() lazily on first post-startup access.

MEDIUMNo sweep/archival exists — storage grows O(revisions × block size) forever; capacity checks full-scan the store perf storage-repo.ts:550; leveldb-storage.ts:122-140; file-storage.ts:129-158

Every commit persists the full materialized block and the transform, and nothing prunes — docs/repository.md promises archival "based on resource pressure" that isn't implemented. Meanwhile getApproximateBytesUsed is a full-database scan (LevelDB iterates every key and value; fs stats the whole tree), and RingSelector triggers it repeatedly (twice in createArachnodeInfo alone).

Implement checkpoint-cadence materialization sweeping (the transform chain already supports replay); cache the byte estimate with a TTL or maintain an incremental counter.

MEDIUMFour adapters are ~150-line near-clones — extract a shared ordered-KV kernel design all four adapter packages

Same JSON (de)serialization, key scheme, drain-before-yield iteration, and promote semantics, differing only in the KV primitive. The LevelDBLike interface (db-p2p-storage-rn/src/leveldb-like.ts) is already exactly the needed kernel surface (get/put/delete/range-scan/atomic-batch). Also fold the ~90% duplicated saveReplica/saveDeletion bodies (block-storage.ts:126-223) into one helper.

Ship a KvRawStorage in db-p2p over a minimal ordered-KV+batch interface; each package keeps only its ~100-line driver.

LOWAssorted: in-place sort of caller state; missing clone; dead metrics; error-swallowing readdir bug/cleanliness

storage-repo.ts:162-165: missing aliases context.committed and .sort() mutates the caller's shared request context — copy first. memory-storage.ts:102-104: saveTransaction stores the caller's reference without structuredClone, violating the store's own documented clone invariants. restoration-coordinator-v2.ts:23,188,196-203: failureByRing never incremented; bare console.log; getMetrics leaks shared Map instances. arachnode-fret-adapter.ts:59 / ring-selector.ts:95: as any pokes into FRET internals and a fabricated fake PeerId. file-storage.ts:72,79: listPendingTransactions swallows all readdir errors as "no pendings" (skipping conflict detection on transient EACCES) and regex-filters action-ids so a future id scheme silently vanishes — map only ENOENT to empty.

Subsystem assessment. 8. SQL integration & apps (quereus-plugin-optimystic · quereus-plugin-crypto · reference-peer · demo · substrate-simulator)

HIGHPK range filters are claimed as handled but never applied — WHERE pk > x returns every row bug quereus-plugin-optimystic/src/optimystic-module.ts:1765, 590-597

getBestAccessPlan marks a PK range filter as handled, but executeRangeQuery is a stub that falls back to full scan and ignores _filterInfo entirely. Per the Quereus contract, handled filters are not re-applied by the engine — so the predicate is simply lost.

HIGHTree comparator is raw lexicographic — INTEGER/DESC/collated PKs mis-sort; the correct comparator is dead code bug optimystic-adapter/collection-factory.ts:43, schema/row-codec.ts:155-184, 243-292

RowCodec.createPrimaryKeyComparator exists but is referenced only by tests, never passed to Tree.createOrOpen (and its deserializeKeyPart mis-classifies numeric-looking TEXT like '123').

HIGHIndex-key serialization is JS-type-sensitive — bigint round-trips orphan index entries bug schema/index-manager.ts:285-306 + schema/row-codec.ts:189-197

On INSERT, index keys are built from the raw Quereus row (bigint 5 → "5"); the stored row is normalized to number 5, so a later UPDATE/DELETE recomputes the old index key as "5.000000000000000e+0" — the delete misses, leaving a stale index entry that index scans then resurrect. Canonicalize values (the same normalizeValue/decode transform) before serializing index keys everywhere; export one shared serializeIndexValue(); add a bigint round-trip index test.

MEDIUMUnescaped \x00 separators and NULL markers corrupt composite keys and index ranges bug schema/row-codec.ts:100-102, 155-158; schema/index-manager.ts:88, 118, 216-217

Composite PKs join parts with \x00; index entry keys are indexKey\x00pk with prefix ranges [key\x00, key\x01); NULL is the literal '\x01NULL\x01'/'\x01'. MEDIUMLegacy-mode commit is not atomic across trees; savepoints are silent no-ops bug optimystic-adapter/txn-bridge.ts:235-249, 400-418; optimystic-adapter/vtab-connection.ts:50-65
