# Auto-resubmission of cascade-invalidated transactions (optimization)

Speculative follow-on to the transaction-invalidation cascade
(`7.5-invalidation-log-entry-and-reversal`, `7.6-invalidation-cascade-detection`,
`7.6-invalidation-client-notification`). Not required for correctness — park here until the core
cascade has landed and there is demand.

## Context

The resolved cascade design re-evaluates each read-dependent of an invalidated transaction and
**retains** the ones that did not truly depend on the invalid value (partial reapplication). Dependents
that *are* invalidated are **not** auto-reapplied with rewritten operations — doing so would change a
transaction's content id / identity. Instead the client is notified and resubmits through the normal
optimistic-concurrency loop.

This ticket asks whether the **client library** should, opt-in, automatically resubmit an
invalidated-but-still-intended transaction on the client's behalf — recompute its operations against the
reverted base and re-run the commit — rather than requiring the application to observe the invalidation
and retry. This is a convenience/automation layer strictly above the protocol.

## Open questions (for whoever promotes this)

- Which transactions are eligible for auto-resubmit? Only those whose *intent* (statements + params) is
  still well-formed against the reverted base, vs. those whose preconditions no longer hold.
- Retry/backoff and starvation bounds (reuse the existing commit retry policy?).
- Idempotency: rely on `StampId()` / content-addressing so an auto-resubmit cannot double-apply.
- Surfacing: how the app distinguishes "auto-resubmitted successfully" from "needs manual intervention".
- Interaction with the cascade horizon: transactions beyond the cascade horizon are escalated for
  re-sync, not auto-resubmitted.

This is a use-case / behavior spec, not a design — keep it that way until promoted.
