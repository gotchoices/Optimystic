# Partition Healing and Reconciliation

description: partition healing is designed (docs/transactions.md) but not implemented; divergent state after a network partition requires manual intervention
prereq: 2-2pc-state-persistence
files:

----

The partition healing protocol is designed in `docs/transactions.md` with three classifications — Behind (normal sync), Ahead (tentative), Forked (conflict) — but none of the reconciliation logic is implemented. Currently the system relies on normal sync after partition recovery; if both sides committed divergent transactions during the partition, the resulting conflict requires manual intervention.

The super-majority (75%) threshold prevents both sides from committing in a 50/50 split, but asymmetric partitions (e.g., 80/20 where 80% side commits) can still produce divergent state when the 20% side reconnects.

## ClusterMember integration points

`ClusterMember` already accepts a `PartitionDetector` (held as `_partitionDetector` and on the `ClusterMemberComponents` interface in `packages/db-p2p/src/cluster/cluster-repo.ts`) but does not yet consume it. When this ticket is picked up, the three plausible consumption points are:

- `processUpdate` / `handlePromiseNeeded` — reject incoming promises during a suspected partition rather than approving them, to avoid creating divergent committed state on the minority side.
- `propagateIfNeeded` — skip propagation during partition (mirrors the gating used in `packages/db-p2p/src/transactions/block-transfer.ts:59,84`).
- Status reporting — surface `detectPartition()` to coordinators so they can fall back to a slower / quorum-locking mode.

To consume the detector, rename `_partitionDetector` back to `partitionDetector` (and add `private readonly` to the constructor parameter) at that time.
