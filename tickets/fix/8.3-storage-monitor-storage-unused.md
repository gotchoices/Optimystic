# StorageMonitor ignores its IRawStorage — `estimateUsedSpace` is a stub

description: StorageMonitor accepts `IRawStorage` but never reads it; `estimateUsedSpace()` returns 0 with a TODO. Implement actual used-space accounting using the injected storage backend.
prereq: none
files:
  - packages/db-p2p/src/storage/storage-monitor.ts (constructor `_storage` line 21; `estimateUsedSpace` lines 55-57 contains the TODO)
  - packages/db-p2p/src/storage/i-raw-storage.ts (the IRawStorage interface — exposes `listRevisions`, `getMaterializedBlock`, `listPendingTransactions`, etc.)
  - packages/db-p2p/src/libp2p-node-base.ts (line 386 — constructs StorageMonitor with the live raw storage)
  - packages/db-p2p/src/storage/file-storage.ts (concrete implementation; could expose a faster size query)

## Investigation

`StorageMonitor` was added alongside `RingSelector` in commit `697c51f` (Oct 2025). The `storage` constructor parameter has been declared since the file was created but **was never read** — `git log -p -S "this.storage"` returns nothing. Commit `f5bb326` renamed it to `_storage` to silence the lint.

The smoking gun is in the file itself, lines 55-57:

```ts
private async estimateUsedSpace(): Promise<number> {
    return 0; // TODO: Implement actual space calculation
}
```

The class accepts an `IRawStorage` specifically as the means to compute used space, then doesn't use it. `getCapacity()` falls through to this stub whenever no override is supplied via `StorageMonitorConfig` — i.e., in production. With used = 0 always, `available` always equals `total` (default 10 GB), so `RingSelector.shouldTransition()` will never see a `usedPercent > moveOut` condition and `RingSelector.determineRing()` will always think the node has full capacity.

The single non-test caller is `libp2p-node-base.ts:386`, which passes the live `rawStorage`. The call site is correct; only the consumer is a stub.

## Verdict: forgotten wiring (acknowledged TODO)

The author left an explicit `// TODO: Implement actual space calculation`, so this isn't ambiguous — it's known incomplete work that was never picked up. The lint silencing in `f5bb326` papers over the symptom (unused field) rather than the cause (stubbed implementation).

## Recommended action

Implement `estimateUsedSpace()` against `IRawStorage`. Two viable approaches:

**Approach 1 — Iterate the storage.** Walk all known blocks via `IRawStorage` (e.g., enumerate metadata, sum revision and materialized-block sizes). This is exhaustive but O(n) per call; cache results with a TTL.

**Approach 2 — Add a `getDiskUsage()` method to `IRawStorage`.** Concrete implementations like `file-storage.ts` can answer this cheaply (e.g., `du`-style traversal of the storage root or maintained running counters); in-memory implementations return their tracked footprint. This is the cleaner long-term API.

`IRawStorage` (per `packages/db-p2p/src/storage/i-raw-storage.ts`) currently exposes per-block read methods (`getMetadata`, `listRevisions`, `getMaterializedBlock`) but no enumeration of all blocks, so Approach 1 needs an additional listing primitive too. Approach 2 is the smaller surface change.

Recommended: **Approach 2** — add `getApproximateBytesUsed?(): Promise<number>` (optional, so existing implementations don't break) to `IRawStorage`; have `file-storage.ts` implement it; have `StorageMonitor.estimateUsedSpace()` call it (with `?? 0` fallback for backends that haven't implemented it yet, optionally logging a one-shot warning).

This fix pairs with ticket `8.2-ring-selector-fret-adapter-unused`: ring selection is only meaningful when both used-space and network demand are real values. Land them together.
