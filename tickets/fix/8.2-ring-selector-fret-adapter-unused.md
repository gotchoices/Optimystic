# RingSelector ignores its FRET adapter — uses hard-coded network estimates

description: RingSelector accepts an ArachnodeFretAdapter but never reads it; ring-depth math uses a hardcoded `estimatedTotalBlocks = 1000` instead of querying the network. Wire up the adapter so ring selection reflects actual network demand.
prereq: none
files:
  - packages/db-p2p/src/storage/ring-selector.ts (constructor `_fretAdapter` line 29; `determineRing()` lines 37-60 contains the hardcoded estimate)
  - packages/db-p2p/src/storage/arachnode-fret-adapter.ts (provides `getRingStats()`, `findPeersAtRing()`, `getKnownRings()`, `getMyArachnodeInfo()`)
  - packages/db-p2p/src/libp2p-node-base.ts (lines 384-393 — constructs the adapter and passes it to RingSelector)
  - packages/db-p2p/test/ring-selector.spec.ts (existing tests, mock adapter)
  - docs/arachnode.md (lines 74-86 — Capacity Management spec: overlap-factor sampling)
----

## Investigation

`RingSelector` was added in commit `697c51f` ("Various mesh test cleanup", Oct 2025). The `fretAdapter` constructor parameter has been declared since the file was created but **was never read** — `git log -p -S "this.fretAdapter"` returns nothing. Commit `f5bb326` renamed the field to `_fretAdapter` to silence the lint.

Inside `determineRing()` (lines 47-49):

```ts
// Estimate total network size from FRET
// We use a simple heuristic: assume average block size and typical data distribution
const avgBlockSize = 100 * 1024; // 100KB typical block
const estimatedTotalBlocks = 1000; // Conservative estimate
const estimatedTotalData = estimatedTotalBlocks * avgBlockSize;
```

The comment claims "Estimate total network size from FRET" but the code never calls FRET — it uses two hard-coded constants (100 KB × 1000 blocks). This is the smoking gun: the dependency was added with the intent to compute network-wide demand, but the actual implementation shipped as a stub.

The injected `ArachnodeFretAdapter` exposes the methods that *would* let RingSelector compute real demand:
- `getRingStats(): Array<{ ringDepth, peerCount, avgCapacity }>` — peer counts and capacity per ring
- `findPeersAtRing(ringDepth)` — peers at a given ring depth
- `getKnownRings()` — currently observed ring depths
- `getMyArachnodeInfo()` — own announced state

`docs/arachnode.md` lines 74-86 ("Capacity Management and Ring Adjustment") describes the intended algorithm: sample overlap factor (storage nodes per key) and cache coverage across rings, then adjust ring depth accordingly. None of that is implemented; the current `determineRing` is a placeholder that ignores network state.

The single non-test caller is `libp2p-node-base.ts:387`, which passes a fully-wired `ArachnodeFretAdapter`. So the *call site* is correct — only the consumer is broken.

## Verdict: forgotten wiring (latent bug)

This is not "placeholder by design" the way `_partitionDetector` is — there's no other consumer of this state, no companion ticket gating it, and the comment inside the function asserts it does something it doesn't. The hardcoded constants quietly mislead any node that tries to pick a ring: a 10 GB node and a 100 GB node would compute the same ring depth because the supposed "total network demand" is a constant.

## Recommended action

Wire `_fretAdapter` into `determineRing()` to compute estimated total network demand from observed ring stats. A reasonable first cut:

1. Restore the field: `_fretAdapter` → `private readonly fretAdapter`.
2. In `determineRing()`, replace the hardcoded `estimatedTotalBlocks` with a value derived from `this.fretAdapter.getRingStats()` — for example, sum of `peerCount * avgCapacity` across known rings, or use `getKnownRings()` to estimate the active partition count and scale capacity from there.
3. Keep a sane default for the bootstrap case (no peers known yet → fall back to current heuristic).
4. Same treatment for `shouldTransition()` (lines 108-139), which currently reaches no further than `storageMonitor` — once we have real demand data, "should we move outward" should also factor in observed overlap.

Note: this fix depends on `StorageMonitor.estimateUsedSpace()` actually being implemented (see ticket `8.3-storage-monitor-storage-unused`). Without real used-space numbers, ring selection still uses fictitious data even with FRET wired up. The two fixes pair naturally.

If fixing properly is out of scope for now, the alternative is to **drop** `_fretAdapter` from the constructor and accept that `RingSelector.determineRing` is a stub — but only after creating a follow-up backlog ticket for "real arachnode capacity-driven ring selection" so the design intent isn't lost.
