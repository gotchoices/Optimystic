----
description: ClusterService membership/responsibility check is disabled; every peer processes all cluster updates
files: packages/db-p2p/src/cluster/service.ts, packages/db-p2p/src/libp2p-node-base.ts, packages/db-p2p/src/repo/service.ts
----
The cluster update path in `ClusterService` has its membership/responsibility scoping disabled, so any peer that receives a cluster update participates in consensus regardless of whether it is one of the peers responsible for the key. This removes the responsibility-based scoping that DHT-based sharding and routing are supposed to provide.

In `ClusterService.handleIncomingStream` (packages/db-p2p/src/cluster/service.ts:101-106), the `'update'` operation unconditionally calls `this.cluster.update(message.record)`. The surrounding comment marks this as temporary: "TEMPORARY: Disable cluster membership check to fix empty promises issue ... TODO: Re-enable and fix cluster membership logic for proper DHT routing." The original membership check was returning redirect responses (instead of processing updates), leading to empty promise arrays, and it was switched off rather than corrected. With the check off, every peer that receives an update processes it, and there is no responsibility-based scoping of consensus.

Compounding the problem, the cluster init options that advertise this behavior are never read. `ClusterServiceInit` defines `configuredClusterSize`, `allowClusterDownsize`, `clusterSizeTolerance`, `kBucketSize`, and `responsibilityK` (the latter with a detailed doc comment describing how it determines the replica set responsible for a key, distinct from `kBucketSize` for DHT routing and `configuredClusterSize` for consensus quorum) at packages/db-p2p/src/cluster/service.ts:24-38. None of these fields are referenced anywhere in `cluster/service.ts` — the constructor (service.ts:56-64) ignores them. Yet `createLibp2pNode` passes them through (packages/db-p2p/src/libp2p-node-base.ts:238-251), giving callers the false impression that these knobs affect cluster behavior when they have no effect.

The correct pattern already exists on the repo path. `RepoService.checkRedirect` (packages/db-p2p/src/repo/service.ts:139-164) derives the sha256 key, looks up the cluster, performs a membership check against the top `responsibilityK` peers (by XOR distance), and constructs a redirect payload when this node is not responsible — with a small-mesh bypass when the cluster is smaller than `responsibilityK`. The cluster update path is inconsistent with this repo path: the repo path scopes by responsibility while the cluster path scopes by nothing.

The consequence is that the documented DHT-based sharding/routing model does not hold for cluster consensus, and Sereus's resilience goal is undermined: a large mesh has no membership scoping for consensus, so consensus load and participation do not partition across the network by key responsibility.

Expected behavior: the cluster update path should re-enable a correct responsibility/membership check, consistent with `RepoService.checkRedirect`, so that only peers responsible for a key participate in that key's consensus — without regressing the original empty-promises symptom that motivated disabling the check. The advertised cluster-size init options (`configuredClusterSize`, `allowClusterDownsize`, `clusterSizeTolerance`, `kBucketSize`, `responsibilityK`) should be wired so they actually influence cluster membership and consensus quorum behavior, or otherwise reconciled so the surface area does not advertise knobs that have no effect.

Related: the responsibility-driven redirect work on the repo path is captured in tickets/complete/6.5-responsibility-driven-redirects.md, which is the reference implementation this fix should align with.
