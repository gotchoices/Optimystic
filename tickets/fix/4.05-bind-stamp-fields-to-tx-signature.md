## The problem

The client-transaction-signature feature (`implement-client-tx-signature-core`) signs a
payload built from **`stamp.id` + statements + reads** (`clientSignaturePayload`). The
`stamp.id` is a SHA-256 hash of the stamp's fields (`peerId`, `timestamp`, `schemaHash`,
`engineId`, `expiration`, `nonce`).

The signature therefore binds those stamp fields **only transitively, through the hash** —
and **nothing ever re-verifies that `stamp.id` actually equals the hash of the fields.**
`TransactionValidator.validate` reads `stamp.expiration`, `stamp.engineId`, and
`stamp.peerId` straight off the wire and trusts them:

- `isTransactionExpired(stamp)` reads `stamp.expiration` directly (step 0).
- `clientSignaturePayload(stamp.id, …)` uses the stored `stamp.id`, not a recomputed one.
- `engines.get(stamp.engineId)` selects the replay engine (step 1).

### Exploit

An attacker (malicious relay / MITM / replaying peer) captures a validly-signed
transaction and mutates a stamp field **without touching `stamp.id` or `signature`**:

- Set `stamp.expiration` to the far future → `isTransactionExpired` returns false, the
  signature payload (built from the unchanged `stamp.id`) still verifies → **accepted.**
  The transaction's lifetime is now extended arbitrarily. This defeats the TTL-bounded
  replay window that the `nonce` was added to reinforce.
- Similarly `stamp.engineId` / `stamp.schemaHash` / `stamp.timestamp` are unprotected.

With the **fake** verifier used in the core tests this is invisible, because those tests
only tamper the signed payload (statements/reads) or the `peerId` passed separately to the
verifier — never a stamp field while holding `stamp.id` constant.

### Why it isn't caught today

Verification is dormant in `db-core` (no verifier port is injected). It goes **live the
moment `implement-client-tx-signature-p2p` wires `verifyPeerSig`** and an operator flips
`requireClientSignature: true`. At that point operators will reasonably believe "signed ⇒
tamper-proof transaction", which is false for stamp fields. That is why this must land
**before** p2p enforcement — hence p2p now lists this slug as a prereq.

Note the `peerId` case is *partially* covered by the real scheme: `verifyPeerSig` derives
the public key from `stamp.peerId`, so tampering `peerId` makes a victim's signature fail.
But `expiration`/`engineId`/`schemaHash`/`timestamp` have no such second line of defense.

## Requirements

- A validating node must reject a transaction whose `stamp.id` is not the true hash of its
  stamp fields, so that every field the system trusts (`expiration`, `engineId`, …) is
  covered by exactly the value the signer committed to.
- The fix must hold whether or not a client-signature verifier is wired (a tampered
  `stamp.id`/field is an integrity violation independent of signatures — e.g. it corrupts
  expiration and engine selection regardless). Decide and document whether the check is
  always-on or gated; **recommended: always-on**, since it is cheap (one SHA-256) and the
  fields are trusted on every path.
- Ordering: the integrity check must run **before** the expiration check, since the
  expiration check itself consumes an as-yet-untrusted field.
- Do **not** change the `stamp.id` hash formula or the `clientSignaturePayload` format —
  changing either churns existing ids/signatures. This is purely an added verification.

## Suggested approach (implementer may refine)

- Extract the stamp-id hashing out of `createTransactionStamp` into a pure
  `computeStampId(fields): Promise<string>` (or sync if the hash allows) so both creation
  and validation derive the id identically. `createTransactionStamp` calls it after
  generating the nonce; the validator calls it over the received stamp.
- In `TransactionValidator.validate`, add a step **0.0** (before expiration): recompute the
  id from the received stamp fields and reject with a clear reason (e.g.
  `Tampered transaction stamp`) on mismatch.
- Tests: tamper each of `expiration`, `engineId`, `schemaHash`, `timestamp` while holding
  `stamp.id` + `signature` constant → reject; a legitimately-built stamp → still valid
  (regression). Add an explicit "expiration extension is rejected" case mirroring the
  exploit above.

## Out of scope

- The real Ed25519 signer/verifier wiring — that stays in `implement-client-tx-signature-p2p`.
- Canonicalizing `createTransactionId`'s read ordering (separate hygiene concern tracked in
  `design-consensus-hygiene-notes`).
