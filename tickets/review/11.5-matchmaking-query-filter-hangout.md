# Review: matchmaking query/filter + hang-out engine

Second matchmaking slice on top of `matchmaking-wire-anchor-registration`. Lands the cohort-side
`QueryV1` evaluation and the seeker hang-out-vs-continue decision/walk engine, per
`docs/matchmaking.md` §Seeker query / §Capability filter / §Hang-out vs. continue. The multi-cohort
root sweep, adversarial bounds, and public-module assembly remain deferred to
`matchmaking-sweep-adversarial-module`.

## What landed

**Cohort side (advisory query path).**
- `matchesFilter` (db-core, pure): every `must` tag present, no `mustNot` tag present,
  `capacityBudget >= minBudget` when set; `undefined` filter matches everything.
- `evaluateQuery` (db-core, pure): applies the filter, truncates to `limit`
  (≤ `query_limit_max` 256), sets `truncated`, builds `ProviderEntryV1[]`/`SeekerEntryV1[]` with
  `registrationSig` forwarded verbatim.
- `handleMatchmakingQuery` (db-p2p): decodes each registration record's `appState` via
  `decodeMatchAppPayload` (skip+log undecodable), runs `evaluateQuery`, attaches injected
  `topicTraffic`/`cohortEpoch`, signs with the cohort **primary's** key over
  `queryReplySigningPayload` (epoch + truncated + traffic + ordered participantIds; single-member,
  not threshold — the reply is advisory).

**Signing-scope tension — resolved as option (b).** The foundation review flagged that the signed
registration image included the matchmaking `correlationId`, which forwarded entries don't carry —
making seeker re-validation unbuildable. `correlationId` is now **excluded** from
`providerSigningPayload` (`(topicId, capabilities, capacityBudget)`) and `seekerSigningPayload`
(`(topicId, wantCount)`), so the signed image is fully reconstructable from a forwarded entry.
Replay-binding stays with the cohort-topic `RegisterV1` envelope (its own correlationId + replay
guard). Seeker-side `verifyProviderEntry`/`verifySeekerEntry` land in `wire.ts` (crypto injected via
`EntrySigVerifier`); `entry-verify.spec.ts` covers sign → forward → verify round-trip, tamper
rejection (capacityBudget mutation), wrong-topic rejection, and malformed-sig false (never throws).
Doc §Wire formats updated with a "Signing scope (resolved)" callout.

**Seeker decision engine (db-core, pure).** `decide()` implements the doc's three-step rule:
immediate-match → hang-out feasibility (`expectedNewMatches`, `contentionFactor` capped at 4.0,
threshold `wantCount × contentionFactor`) → escalate. `filterAcceptRatio` is a running cumulative
matched/returned (starts 1.0 before observation). Tuning constants in `config.ts` match the
simulator-folded doc: `patience_default_ms` 10 000, `patience_per_tier_fraction` 1.0 (fixed
strategy), `filter_accept_ratio_initial` 1.0, `contention_factor_cap` 4.0, `requery_interval_ms`
1 000, `mean_want_count` 3.

**Seeker walk client (db-p2p).** `SeekerWalkClient.run()` drives the walk from `d_max`: register →
on `Accepted` issue immediate `QueryV1` (filter + re-validate + dedupe by participantId) → `decide`
→ done / hang out (sleep `requery_interval_ms`, renew, re-query until met or patience drains) /
withdraw + re-register at `d − 1`. At `d = 0` it hangs out for remaining patience then returns the
partial set. Patience is a single wall-clock deadline draining across hops. Handles `NoState`
(walk down), `Promoted` (descend to target), `Unwilling*` (terminate, no hang-out — case 3).
Clock/sleep/transport/verifier all injected for testing.

**All five doc edge cases are explicit and tested:** (1) absent `topicTraffic` → zero-rate, walk
without hang-out; (2) stale `arrivalsPerMin = 0` → query first, done if it fills, never withdraw on
the single zero reading; (3) `UnwillingCohort` → hang-out never entered; (4) pathological filter →
ratio decay collapses `expectedNewMatches`, walks to root; (5) contention → capped at 4.0.

## Known gaps / review focus

- **No live libp2p protocol binding.** Both ends are seam-injected: the query handler takes
  `records`/`topicTraffic`/`cohortEpoch`/`sign` from the caller; the walk client takes a
  `SeekerWalkTransport` (register/query/renew/withdraw). No `/optimystic/matchmaking/...` query
  protocol stream exists yet — concrete FRET/`CohortTopicService` bindings and public-module
  assembly are the next slice (`matchmaking-sweep-adversarial-module`). Mock-tier e2e is therefore
  the doc-flagged pending item.
- **Cohort does not verify `QueryV1.signature`.** The handler trusts the decoded query (structural
  validation only). The doc doesn't currently mandate cohort-side requester-sig verification, and
  adversarial bounds belong to the next slice — but a reviewer should confirm that posture.
- **`queryReplySigningPayload` binds participantIds, not full entry bodies.** `contactHint` /
  `attachedAt` are in neither the reply image nor the per-entry signed image (the spec's signed
  tuple never included them). A tampering primary is the acknowledged advisory-trust bound; flag if
  this deserves tightening.
- **Withdraw is intent-only.** The substrate still has no TTL=0 renew realization (documented
  cohort-topic follow-on from the foundation slice); `withdraw()` stops renewing and the record ages
  out by TTL. Correctness doesn't depend on it (doc §Provider self-throttling GROUNDING note).
- **Hang-out loop never re-runs `decide()`.** Once hanging out, the client polls until met or
  patience drains — it does not re-evaluate feasibility on a mid-hang-out traffic collapse. The doc
  doesn't require re-evaluation (patience bounds the cost), but it's a judgment call worth a look.
- Backlog refinements intentionally not implemented here (cross-referenced in the doc):
  `matchmaking-per-tier-patience-splitting`, `matchmaking-contention-from-seeker-pool`,
  `matchmaking-cohort-push-on-arrival` (this slice is the polling baseline).
