# Review: local change-notifier → cohort-topic origination bridge

Connects the **existing local single-node** change-notification primitive (`StorageRepo` →
`IBlockChangeNotifier`) to the **networked** cohort-topic substrate. This is the origination point:
a commit that lands on a node which is a cohort member for the collection's reactivity topic is
emitted into `CohortTopicService.onLocalCommit`, carrying the cluster commit certificate **forwarded
unchanged**, so reactivity (next ticket) can fan out without re-signing. Build + tests green across
db-core, db-p2p, and quereus-plugin-optimystic.

## What landed

### db-core — the origination contract
- **`CommitCert`** (`transactor/change-notifier.ts`): `{ thresholdSig: Uint8Array; signers: readonly
  string[]; minSigs: number }`. The authoritative cluster commit certificate, opaque to the bridge,
  passed through unchanged. Peer ids are strings (cluster keys signatures by peer-id string) — db-core
  stays free of any libp2p `PeerId` type.
- **`LocalChangeHook`** refined from the core-module placeholder `(change: LocalCommitChange) => void`
  to **`(event: CollectionChangeEvent, commitCert: CommitCert) => void`**, matching exactly what the
  reactivity ticket (`reactivity-origination-replay-delivery`) said it consumes. The old
  `LocalCommitChange { topicId, appPayload? }` placeholder was removed (it was defined-but-never-used;
  no consumers existed). `CohortTopicService.onLocalCommit?` unchanged in spirit (still the
  reactivity/matchmaking-set hook), now with the richer signature.

### db-p2p — the bridge + cert extraction
- **`makeCohortTopicChangeNotifier(deps)`** (`cohort-topic/change-bridge.ts`): returns an
  `IBlockChangeNotifier` (the value `NetworkTransactor` takes as `localChangeNotifier`). Deps:
  `source` (the StorageRepo, as a `ChangeBridgeSource`), `service` (CohortTopicService),
  `selfIsCohortMember(collectionId)`, `extractCommitCert(event)`. On **every** commit (catch-all
  feed): member + hook-attached + cert-available → `service.onLocalCommit(event, commitCert)`;
  otherwise a no-op. Throws are isolated + logged (never break the commit). Its `onCollectionChange`
  delegates straight to `source`, so the Quereus reactive-watch path is untouched. Plus
  `attachCohortChangeBridge(node, deps)` for the post-assembly node wiring.
- **`StorageRepo.onAnyCollectionChange(listener)`** (`storage/storage-repo.ts`): a catch-all feed that
  fires for *every* collection's `(pending → committed)` transition (the per-collection
  `onCollectionChange` can't see all commits — the bridge can't enumerate collection ids). The emit
  path was restructured to fire both per-collection and catch-all listeners off one event, with a
  shared isolating `fireChangeListeners` helper. Per-collection semantics are byte-identical to before
  (covered by the unchanged storage-repo.spec, still green).
- **`buildCommitCert` / `CommitCertStore` / `makeClusterCommitCertExtractor`**
  (`cluster/commit-cert.ts`): `buildCommitCert(record, minSigs)` assembles the cert from a
  `ClusterRecord`'s `approve` commit signatures — decoded and concatenated in ascending signer order
  (so `signers[i]` aligns with chunk `i`, the cohort-topic collected-multisig convention). The store
  is a bounded, TTL'd `actionId → CommitCert` cache; the extractor reads it on a change event.
- **`ClusterMember.onCommitCertificate`** (`cluster/cluster-repo.ts`): an optional sink (mirrors the
  existing `reconcileBlock` injection pattern) fired in `handleConsensus` **before**
  `storageRepo.commit(...)` — critical, because `StorageRepo.commit` emits the change event
  synchronously at the end of that call, so the cert must already be retained when the bridge resolves
  it. Zero cost when no sink is wired; a throwing sink is isolated + logged (never breaks consensus).
- **`libp2p-node-base.ts`**: new `onCommitCertificate?` node option, threaded to `clusterMember(...)`.
  The cert-capture seam, ready for the node-wiring that constructs the cohort host to use.

## How to exercise / validate

- **New spec — `packages/db-p2p/test/cohort-topic/change-bridge.spec.ts`** (drives a real
  StorageRepo commit so the catch-all fires exactly as in production):
  - member commit → `onLocalCommit(event, cert)` with correct `collectionId`/`blockIds`/`actionId`/`rev`;
  - non-member commit → no-op;
  - **threshold sig forwarded byte-for-byte unchanged** (same object, same bytes, same signers/minSigs);
  - throwing downstream hook → commit still succeeds;
  - no retained cert → skip (never fabricate); no hook attached → cert extraction never even runs;
  - decorator delegation: per-collection subscribe/unsubscribe still works through the bridge;
  - `buildCommitCert` (approve-only, sorted, byte-exact concat), `CommitCertStore` (TTL + cap eviction),
    `makeClusterCommitCertExtractor`, and an end-to-end stored-cert → bridge pass-through.
- **Capture path — `cluster-consensus-divergence.spec.ts`** (added describe block): drives a real
  2-peer consensus to `handleConsensus` and asserts `onCommitCertificate` fires once with `minSigs=2`,
  both signers, and a 128-byte blob (two concatenated 64-byte Ed25519 commit sigs); plus a zero-sink
  no-op case.
- Commands: `yarn workspace @optimystic/db-core build && test`, `yarn workspace @optimystic/db-p2p
  build && test`. Build type-checks the test files too.

## Honest gaps / reviewer's starting points

1. **Phase 2 activation is dormant in the production node (the biggest gap).** `createCohortTopicHost`
   is wired **only in tests** today — `libp2p-node-base.ts` never constructs a `CohortTopicService`,
   so there is no service to gate on and the bridge is not yet set as the node's `blockChangeNotifier`.
   I deliberately did **not** add a speculative in-assembly `if (cohortService)` branch: the cohort
   host is constructed *post*-node (`createCohortTopicHost(node, fret, …)`), so activation belongs in
   the post-assembly wiring via `attachCohortChangeBridge`, not inside `createLibp2pNodeBase`. What I
   shipped is the exact seam: the `onCommitCertificate` node option (cluster side) + the
   `attachCohortChangeBridge` helper (notifier side). The whole-node wiring (build cohort host →
   create `CommitCertStore` → pass `onCommitCertificate: store.put` → `attachCohortChangeBridge(node,
   { source: storageRepo, service, selfIsCohortMember, extractCommitCert: makeClusterCommitCertExtractor(store) })`)
   lands with the cohort-host node-wiring / live-tier work. **Suggested action:** confirm this is the
   right division and that no in-assembly stub is expected; if a thin always-on path is wanted, file a
   follow-on to construct the cohort host in the node.

2. **`selfIsCohortMember` is fully injected — no concrete impl shipped.** The bridge is membership-gate
   agnostic by design (testable, decoupled). The real predicate (consult the cohort host registry /
   FRET membership for the collection's reactivity topic) is supplied by the node wiring in (1). No
   collection-id → reactivity-topic derivation is done here; the reactivity ticket owns
   `topicId = H(tailId ‖ "reactivity")`.

3. **Cert byte format vs reactivity verification.** `buildCommitCert` concatenates the cluster
   **commit** signatures, which are Ed25519 sigs over the cluster commit-hash payload
   (`computeCommitHash(record)` then `hash + ':approve'`), **not** over a notification payload.
   Reactivity reuses these bytes as-is and must verify them against *that* payload (carried via
   `NotificationV1.digest`) — not naively re-verify over the notification body. The bridge forwards
   bytes unchanged; reconciling the exact verify payload is the reactivity ticket's job. Worth a
   reviewer eye that the `(thresholdSig, signers)` alignment I chose (ascending signer order, 64-byte
   chunks) matches what the cohort-topic `ICohortThresholdCrypto.verify` / `MembershipVerifier` expect.

4. **Capture happens on every cluster peer that commits, not only the eventual originator.** Each
   consensus peer stores the cert locally; only the node whose StorageRepo emits AND is a cohort member
   originates. Unused certs TTL-expire (bounded store). Correct, but confirm the store sizing
   (`DEFAULT_COMMIT_CERT_TTL_MS = 60s`, `MAX_ENTRIES = 4096`) is sane for expected commit rates.

5. **Catch-all subscription lifetime.** `makeCohortTopicChangeNotifier` subscribes once and never
   unsubscribes (node lifetime by intent). No `stop`/`close` is exposed on the returned
   `IBlockChangeNotifier`. Fine for a long-lived node; flagged in case a teardown handle is wanted.
