# Review: unify the owned-block tracked set across SpreadOnChurnMonitor and RebalanceMonitor

## What changed (Option A — inject a bare `Set<string>`)

Before this ticket both monitors kept their **own** `private readonly trackedBlocks = new Set<string>()`,
each fed by a **separate** `storageRepo.onAnyCollectionChange` subscription in `libp2p-node-base`. The
two sets could drift, and there was no responsibility-loss eviction at all. Now:

- **`SpreadOnChurnDeps` / `RebalanceMonitorDeps`** gained an optional `trackedBlocks?: Set<string>`.
  Each monitor's field is now `private readonly trackedBlocks: Set<string>`, assigned in the
  constructor as `deps.trackedBlocks ?? new Set<string>()`. Injected ⇒ shared; omitted ⇒ a fresh
  private set (standalone construction / existing unit tests unchanged). `trackBlock` /
  `untrackBlock` / `getTrackedBlockCount` / the `for…of this.trackedBlocks` loops are untouched.
  `RebalanceMonitor.responsibilitySnapshot` stays **per-monitor** (it is rebalance's own
  was-responsible memory, not owned-block tracking).
- **`NetworkManagerService.initSpreadOnChurnMonitor` / `initRebalanceMonitor`** each take a new
  required `trackedBlocks: Set<string>` param and pass it straight into the respective `Deps`.
  (Only `libp2p-node-base` calls these — confirmed via reference search.)
- **`libp2p-node-base`** now constructs **one** `const ownedBlocks = new Set<string>()`, passes it to
  both monitor inits, and collapses the **two** `onAnyCollectionChange` subscriptions into **one**
  (`ensureOwnedBlockFeed()`, registered lazily the first time either monitor is wired, torn down once
  in a dedicated up-front stop wrapper). In the `onRebalance` handler, **alongside** dispatching to
  `coordinator.handleRebalanceEvent(event)`, it now drives eviction:
  `event.gained → ownedBlocks.add`, `event.lost → rebalanceMonitor.untrackBlock` (which clears both
  the shared-set entry and the snapshot). This makes responsibility-loss the **authoritative** untrack,
  complementing (not replacing) spread's lazy no-local-data self-prune.

### Architecture (after)

```
 storageRepo.onAnyCollectionChange  ──(single feed: e.blockIds → add)──►  ownedBlocks: Set<string>
                                                                            ▲        ▲
                                            deps.trackedBlocks ─────────────┘        └───────── deps.trackedBlocks
                                            SpreadOnChurnMonitor                       RebalanceMonitor
                                            (reads to re-push on churn)                (reads to detect gained/lost)
                                                                                              │
 RebalanceMonitor.onRebalance(event) ──► gained: ownedBlocks.add  /  lost: untrackBlock ──────┘ (+ snapshot clear)
                                     └──► coordinator.handleRebalanceEvent(event)  (independent, order-agnostic)
```

## Use cases / behaviors the reviewer should validate

- **Sharing:** one `Set` instance backs both monitors on a live node (arachnode-enabled boot wires both;
  see node-wiring assertion). Track via the feed → both `getTrackedBlockCount()` agree.
- **Eviction authority:** a `RebalanceEvent` with `lost: [b]` removes `b` from the shared set so a
  subsequent `SpreadOnChurnMonitor.checkNow()` no longer attempts to spread `b`. Also clears the stale
  `responsibilitySnapshot[b]` via `untrackBlock`.
- **Re-gain:** a `gained` block is re-added to the shared set in the same handler (currently a near-no-op
  because `gained ⊆ trackedBlocks` already, but kept for intent/future-proofing — see "gaps").
- **Backward compat:** standalone monitor construction (no injected set) preserves all prior behavior;
  the entire pre-existing unit suite passes unchanged.
- **Teardown / opt-out gates:** `spreadOnChurn: { enabled: false }`, `rebalance: { enabled: false }`, and
  arachnode-disabled paths still behave as before (no monitor leak; receive handler still registered).
  The single owned-block feed is torn down exactly once.

## Known gaps / things to scrutinize (treat tests as a floor)

- **No direct test for the "spread disabled + rebalance enabled" feed path.** The `ensureOwnedBlockFeed()`
  helper is idempotent and registered by whichever monitor wires first, and the dedicated up-front
  teardown wrapper unsubscribes regardless of which monitor subscribed — so rebalance-only nodes are
  handled in code, but there is no spec that boots with `spreadOnChurn:{enabled:false}` + arachnode
  enabled and asserts the rebalance monitor still gets fed + the feed is torn down. Worth adding or at
  least eyeballing the wiring in `libp2p-node-base.ts` around the two `ensureOwnedBlockFeed()` calls.
- **`gained → ownedBlocks.add` is presently redundant.** `gained` is only ever derived by iterating
  `trackedBlocks`, so a gained block is already in the shared set. The add is defensive/forward-looking;
  confirm you're comfortable keeping it (documented inline) vs. dropping it.
- **Best-effort iteration safety under shared mutation** is documented, not locked. The rebalance handler
  can `delete` from `ownedBlocks` while `SpreadOnChurnMonitor.performSpread` is mid `for…of` over the same
  set inside an `await` loop. JS `Set` mutation during iteration does not throw (entries visited
  best-effort); accepted for a resilience mechanism. Confirm this is acceptable for your correctness bar.
- **`responsibilitySnapshot` staleness** when the shared set is mutated externally (spread self-prune, or
  eviction) is documented inline as inert (only `trackedBlocks` is iterated; a lingering snapshot entry is
  re-derived if the block is re-fed). Sanity-check the reasoning.
- **Initial-scan seed still deferred** — blocks already durable from a previous run are not re-tracked
  until next touched. `IRawStorage` exposes no enumerate-all-block-ids API; filed as
  `tickets/backlog/optimystic-owned-block-initial-scan-seed.md`. The "not re-tracked until next touched"
  comments remain accurate.
- **LSP noise (not a real issue):** editing these files surfaces `Cannot find name 'setTimeout' /
  'clearTimeout' / 'TextEncoder'` and mocha-globals diagnostics in the in-editor language server. These
  are an ambient-lib resolution artifact shared by every file in the package; `tsc` (`yarn build`)
  resolves them and exits clean.
