## What this ticket did (plain terms)

A transaction records a "read dependency" on every block it reads, as `(blockId, revision)`.
At commit time a validator rejects the transaction if any of those blocks has since changed
(optimistic concurrency, `docs/correctness.md` Theorem 5). The problem: a B-tree point lookup
reads the *interior branch nodes* on the way down to the leaf it actually wants. Those interior
reads went into the conflict set, so an unrelated concurrent change to a structural block a
transaction merely *walked through* could reject it — a false positive that gets worse the more
blocks a transaction touches (the fairness inversion behind the starvation work).

This change lets those **navigation** reads be dropped from the conflict set — but *only* for a
value-of-key point lookup (`BTree.get` / `Tree.get`), and *only* the interior nodes strictly
between the root and the target leaf. The target leaf itself is always retained. Everything else
(range scans, `find`, `first`/`last`, counts, writes, the root, the collection header, and all
log-chain reads) stays retained. Default is always "retain" — a read must be *positively*
classified as navigation to be droppable, so a forgotten tag can only over-reject, never miss a
conflict.

### Why dropping the interior read is safe (the whole crux)

A point lookup's result depends only on the terminal leaf's content. Any concurrent change that
would alter the result — setting/deleting the key, or a split/merge/rebalance that relocates the
key — necessarily **modifies that leaf**, bumping its revision. The retained leaf read catches it.
A change confined to an interior node that leaves the leaf untouched cannot change the result, so
excluding it removes a false positive without weakening detection.

## How it works (mechanism)

- `ReadPurpose = 'value' | 'navigation'` (`block-store.ts`). `BlockSource.tryGet(id, purpose?)`
  gains an optional purpose, default `value` — so every existing caller is unchanged.
- `ReadDependencyCollector` now stores `{revision, purpose}` per block id. Revision is max-wins
  (as before); purpose is **value-wins** (once any path reads a block as `value` it stays value —
  order-independent, hence deterministic). `getReadDependencies()` returns only `value` reads.
  `markValue(id)` upgrades an already-recorded read to value (keeps its revision).
- Purpose threads through the read chain: `get()` → `tryGet()` → Tracker → CacheSource →
  TransactorSource, each recording with the caller's purpose.
- `BTree.get(key)` calls `find(key, /*navigate*/ true)`. In `getPath`, a `navigate` descent reads
  each interior child as `navigation`; on reaching the leaf it calls `markReadValue(leafId)`
  (through the store chain → collector) to upgrade the leaf back to a retained value read. Public
  `find` stays `navigate=false` (its returned Path can seed a scan, so its structure must be
  retained).
- `transaction.reads` is built from the filtered set, so the transaction id, the client
  signature, and the validator stale-check all use the identical reduced set. Determinism is a
  function of query shape (leaf vs branch), not cache/timing.

Coverage (this is the FLOOR, not a finish line):
- **Collector unit:** default→value retained; navigation excluded; value-wins both orders;
  `markValue` upgrade keeps revision; `markValue` no-op on unknown id; revision max-wins across
  purposes.
- **CacheSource:** a `navigation` cache read is dropped and `markReadValue` re-retains it; a
  default read is retained.
- **Tree integration (real store chain, `nodeCapacity=4` to force a 3-level tree with ~60
  entries):** `get()` drops the interior branch id(s) on the path and retains the leaf (and root);
  `find()` retains interiors; a range scan retains interiors; the read set is identical across two
  `get()` runs (determinism); an absent-key lookup still retains the leaf it descended to.
- **Validator consequence (hand-built reduced read sets against the unchanged
  `TransactionValidator`):** leaf bump → `Stale read` rejection (split-moves-key); interior-only
  bump with the interior *not* in the set → no stale rejection, commits (the false positive
  removed).

## Reviewer: start here — known gaps & things to probe

Treat the tests as a floor. Specific things worth adversarial attention:

1. **Root + collection-header reads are RETAINED, not dropped (conservative).** The interior
   exclusion is applied inside `getPath` (levels strictly below the root); the root is read via
   `trunk.get()` and the header via `trunk.getId()`, both as `value`. So every point lookup still
   false-positive-conflicts with a root split/collapse or header change. This is deliberate (avoids
   threading purpose through `CollectionTrunk` and keeps the O(1) reads safe by default) and is
   documented in Theorem 5's Bound. Confirm you agree the residual is acceptable, or that narrowing
   it is a separate follow-up — do not treat it as a bug.

2. **Chain / log reads are ALL retained — navigation tagging was deliberately NOT applied to
   chains.** The ticket floated tagging chain nav nodes; I did not, because the chain backs the log
   whose tail is the commit-ordering arbiter (OCC tail binding) and enumerations are shape-
   dependent — both load-bearing. See the `NOTE:` in `chain.ts` `getTail`. If the reviewer thinks
   chain exclusion is in-scope-required, it should be its own carefully-predicated ticket, not an
   inline add here.

3. **Only `BTree.get` opts in; `BTree.find` does not.** Rationale: `find` returns a navigable
   `Path` that a caller may feed to `ascending`/`next` (a scan), whose result depends on tree
   shape. Verify there is no code path that treats a `find` result as a pure point lookup and would
   benefit — and, more importantly, that nothing calls `get` and then keeps navigating (there is no
   such path today; `get` returns the entry and discards the path).

4. **Tracker memoization skips recording entirely (pre-existing, not introduced here).**
   `Tracker.tryGet` returns a materialized memo for op-carrying blocks WITHOUT consulting the
   source, so it records no dependency (and no purpose) at all. This only affects blocks the
   transaction itself has staged writes to (already in the write set), and is orthogonal to this
   change — but if the reviewer is auditing read-set completeness, it is worth knowing this path
   exists independent of purpose tagging.

5. **New API surface: `Tree.createOrOpen(..., nodeCapacity?)`.** Added mainly for testability
   (forcing a multi-level tree with few entries). It threads into ALL THREE BTree constructions —
   the create-header btree, the reopen btree, AND the write-path `actionTree` in the replace
   handler — because the read and write btrees MUST share fan-out or the physical tree and the read
   view disagree. Check that threading is complete (it was the source of a test bug caught during
   implementation: the write btree originally kept default 64).

6. **End-to-end OCC reject/commit is proven at the validator layer with hand-built read sets, plus
   the read-set *composition* is proven at the tree layer — but NOT through a full multi-collection
   session/coordinator or a real cluster.** The two are stitched (correct set + unchanged validator
   = guarantee), but a reviewer wanting belt-and-suspenders could add a `TransactionSession`-level
   test that stages a point lookup and a conflicting interior-only change and asserts commit.
