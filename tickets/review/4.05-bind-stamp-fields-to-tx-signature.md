## What was wrong

A transaction's `stamp` carries the metadata a validating node trusts: `peerId`,
`timestamp`, `schemaHash`, `engineId`, `expiration`, `nonce`. `stamp.id` is a SHA-256 hash
of exactly those six fields (`stamp:<base64url>`). The client signature is made over a
payload built from `stamp.id` + statements + reads — NOT over the raw fields.

Nothing re-verified that `stamp.id` actually equals the hash of the fields. So an attacker
who captured a validly-signed transaction could mutate a stamp field **while holding
`stamp.id` and `signature` constant** — the signature payload (built from the unchanged
`stamp.id`) still verified, so the forged field was accepted. The headline exploit:
extend `expiration` far into the future to defeat the TTL-bounded replay window.

## What changed

**`transaction.ts`** — extracted the id derivation into one shared pure helper so creation
and validation hash **identically**:

```ts
export type StampFields = Omit<TransactionStamp, 'id'>;

export async function computeStampId(fields: StampFields): Promise<string> {
	const { peerId, timestamp, schemaHash, engineId, expiration, nonce } = fields;
	const stampData = JSON.stringify({ peerId, timestamp, schemaHash, engineId, expiration, nonce });
	return `stamp:${await hashString(stampData)}`;
}
```

Field order (peerId, timestamp, schemaHash, engineId, expiration, nonce) and the `stamp:`
prefix are byte-identical to the old inline code, so existing stamp ids and client
signatures stay valid — this is purely an added verification. `computeStampId` destructures
and rebuilds the object in fixed order (rather than stringifying `fields` directly), so a
caller who slips an `id` or extra property into the object cannot change the serialized
bytes. `createTransactionStamp` now derives its id through `computeStampId`.

**`validator.ts`** — new **step 0.0**, before the expiration check:

```ts
const expectedId = await computeStampId(stamp);
if (expectedId !== stamp.id) {
	return { valid: false, reason: 'Tampered transaction stamp' };
}
```

Design decisions (from the ticket, implemented as stated):
- **Always-on**, not gated on the signature verifier — one SHA-256, and expiration + engine
  selection trust stamp fields on every validation path.
- **Ordered first** (step 0.0), because the very next step (expiration) already consumes an
  untrusted field.
- Hash formula and `clientSignaturePayload` format **unchanged**.

## Use cases / validation

Run from `packages/db-core`:
`node --import ./register.mjs node_modules/mocha/bin/mocha.js "test/transaction.spec.ts" --reporter spec`
→ **102 passing**. Also `yarn build` (tsc) → exit 0, new export typechecks.

New tests in the `Client Transaction Signatures` block (tamper a stamp field AFTER signing,
holding `stamp.id` + `signature` constant, assert reason `Tampered transaction stamp`):
- `expiration` extended far into the future — headline regression guard (mirrors the
  reproduced exploit).
- `engineId` changed.
- `schemaHash` changed.
- `timestamp` changed.
- Regression: a legitimately-built signed transaction still validates (`valid: true`).
- No-verifier: `makeValidator(undefined)` + tampered expiration still rejected — proves the
  check is not gated on signatures.

## Reviewer notes / known gaps

- **The existing peerId-tamper test changed meaning, and I updated it.** peerId is one of
  the six stamp fields, so step 0.0 now catches a bare peerId forge (id held fixed) BEFORE
  the signature check — the old test's expected reason `Invalid client signature` no longer
  applied. Rather than just flip the expected string, I rewrote the test so the attacker
  *also recomputes `stamp.id`* to match the forged peerId: that passes integrity but the
  signature (made over the ORIGINAL id/identity) then fails, so it genuinely exercises the
  signature layer as peerId's second line of defense. Worth a look — confirm this is the
  intent and that both layers are meaningfully covered (integrity for id-fixed forgery,
  signature for id-recomputed forgery). This is the only behavioral churn to a pre-existing
  test; the statement/read tamper tests are untouched (they mutate non-stamp fields, so
  integrity passes and the signature check still fires as before).
- The `nonce` field is protected by step 0.0 (it feeds the hash) but has no dedicated
  tamper test — the four field tests plus the no-verifier test cover the mechanism; a nonce
  case would be redundant. Flagging in case the reviewer wants exhaustive per-field
  coverage.
- Statement/read tampering remains covered only by the **signature** path (they are not
  stamp fields, so integrity does not touch them). Unsigned + no-verifier posture still
  accepts statement/read tampering — unchanged by this ticket, and out of scope (real
  Ed25519 verifier wiring is `implement-client-tx-signature-p2p`).
