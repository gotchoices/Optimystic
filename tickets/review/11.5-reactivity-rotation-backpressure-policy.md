# Review: reactivity tail rotation, slow-subscriber backpressure, Edge/Core policy

Implements the three remaining reactivity concerns on top of the origination/replay/delivery hot path and
the backfill/resume/checkpoint recovery path: the **rotating tail anchor + drain protocol**, **per-subscriber
bounded-queue backpressure**, and the **Edge/Core T3 policy + consolidated config**. All production code is
pure db-core modules (injected ports, no FRET/libp2p) plus thin db-p2p wiring; the design simulator side
(rotation burst, coverage, adaptive-W) was already landed by `simulator-reactivity-replay` and its doc
callouts already exist, so this ticket added **no** simulator code.

## What landed

**db-core (`packages/db-core/src/reactivity/`)**
- `rotation.ts` — the full lifecycle:
  - `detectRotation(tailIdAtAttach, n)` — subscriber-side: rotated when delivered `tailId` *or*
    `rotationHint.newTailId` differs from `tailIdAtAttach`; distinguishes a hard rotation from a pre-announce.
  - `buildRotationHint(newTailId, fillingRevision)` → `{ newTailId, effectiveAtRevision: fillingRevision+1 }`.
  - `BlockFillTracker` — counts transactions per tail block; emits `warmup` at `block_fill_size −
    warm_threshold` and `filling` at `block_fill_size` (then resets).
  - `TailDrainGate` — within `T_drain`: `new_subscribe` → `RotationRedirectV1` (Promoted-shaped redirect to
    the new tree's `topicId = H(newTailId ‖ "reactivity")`), `renew`/`replay` → `serve`; after `T_drain` →
    `drained`.
  - `planReRegistration` / `planReRegistrationWave` — derive the new `topicId`, carry `lastRevision`, and
    stagger over `T_rejoin_jitter` via the cohort-topic `RejoinJitter` (wave form hard-bounds the new tail's
    inbound to `cap_promote_fast` per window).
  - `buildRotationHandoffCheckpoint(state)` / `applyRotationHandoff(newTail, handoff)` — fold the outgoing
    tail's replay buffer into a final `CheckpointSummary` over `[lastCheckpoint.toRevision+1, rotationRevision]`
    and land it on the new tail's `PushState.inheritedCheckpoint`.
- `backpressure.ts` — `BoundedQueue` (drop-oldest + monotone `dropped`) and `SubscriberBackpressure`
  (per-subscriber map). Wired into `PushState.perSubscriberQueue` (replacing the reserved stub) with
  `PushState.enqueueForSubscribers`. Primary-local — **not** gossiped (absent from `PushStateGossipV1`).
- `policy.ts` — `mayServeAsReactivityForwarder(profile)` (Edge → false), `instantiateForwarderPushState`
  (returns `undefined` on Edge), `requireForwarderPushState` (throws `ReactivityForwarderForbiddenError`),
  `reactivityNodePolicy` (forwarder eligibility + `delta_max` Core 4096 / Edge 0).
- `config.ts` — this ticket now owns `queue_max`/`delta_max`/`T_drain`/`warm_threshold`/`block_fill_size`;
  added `T_REJOIN_JITTER_MS` (inherited) and `resolveQueueMax` (adaptive hook, parallel to `resolveW`);
  marked `T_drain`/`queue_max`/`block_fill_size` simulator-validated-pending.
- `checkpoint.ts` — **refactor (review hotspot):** extracted `buildCheckpointSummary` / `coalesceMergedDelta`
  so `RollingCheckpoint.summary()` and the rotation handoff fold identically. Behavior of the rolling
  checkpoint is unchanged; verify the resume/checkpoint specs still pass (they do).
- `push-state.ts` — real backpressure map; `inheritedCheckpoint` + `adoptRotationCheckpoint` (idempotent,
  only advances).

**db-p2p (`packages/db-p2p/src/reactivity/subscription-manager.ts`)**
- Rotation detection on `onNotification`: invalidates the sticky cohort-hint cache and surfaces a
  `RotationNotice` (with a jittered re-registration `plan`) once per successor tail, via a new `onRotation`
  option (+ `rejoinJitter`). Origination already passes `rotationHint` through unchanged (tested).

**Docs** — `docs/reactivity.md` §Slow-subscriber backpressure / §Tail rotation / §Per-cohort policy /
§Configuration got `Implemented` callouts; `docs/cohort-topic.md` §Application policies got the
rotation-is-a-fresh-topic interaction note (and a stale `"push"` → `"reactivity"` anchor-suffix fix).

## Known gaps / things to scrutinize (treat tests as a floor)

These are **deliberate boundaries**, not oversights — flagged honestly for the reviewer:

1. **No live db-p2p reactivity *forwarder host* exists yet** (db-p2p has only origination + subscription
   managers). So `TailDrainGate`, `PushState.enqueueForSubscribers`, and `BlockFillTracker` are exercised as
   pure logic, not driven from a cohort member's receive/fan-out loop. The drain-redirect transport, the
   per-subscriber dequeue→dial delivery loop, and a per-collection `BlockFillTracker` feeding the origination
   `resolveContext.rotationHint` (+ warm-up FRET pre-dial) are **host-integration follow-ons** that should
   land with that forwarder host. Consider whether a follow-on ticket should be filed now.
2. **Anticipatory warm-up is signal-only.** `BlockFillTracker` emits `warmup`; the actual FRET pre-dial
   biasing has no API wired (best-effort by spec). Confirm that's acceptable for this milestone.
3. **Handoff↔resume coordination is one-directional.** `applyRotationHandoff` records
   `inheritedCheckpoint`, but `classifyResume`/`serveResume` (owned by `reactivity-backfill-resume-checkpoints`)
   do **not** yet consult it, so a resume whose span crosses the rotation is not yet served from the inherited
   checkpoint. The `ResumeReplyV1.TailRotated` variant + `latestKnownTailId`-staleness already live in that
   sibling ticket. Cross-check whether the classifier should read `inheritedCheckpoint` (likely a small
   follow-on in that module).
4. **Re-registration timers are the host's job.** The subscription manager surfaces a `RotationNotice` with a
   `fireAt` plan and invalidates the sticky cache, but does not schedule the timer or perform the re-register
   itself. Verify the seam (host schedules + registers under `plan.newTopicId` carrying `plan.lastRevision`)
   is the intended contract.
5. **`RotationRedirectV1` is a reactivity-specific shape**, not the cohort-topic tier-based `Promoted`
   (rotation redirects to a *new topic*, not a deeper tier). Confirm the wire/transport plan for it when the
   forwarder host lands (it is not a framed wire message in this ticket — no codec).
6. **Provisional params.** `T_drain`/`queue_max`/`block_fill_size` are simulator-validated-pending;
   `resolveQueueMax` exists but is not yet wired to a measured cohort size. Sourced from the single config
   table so the fold-back can revise without protocol changes.
7. **`checkpoint.ts` refactor** touches the already-reviewed backfill/resume path — worth a focused diff read
   to confirm `buildCheckpointSummary` is byte-identical to the prior inline fold (merged-digest + delta
   coalescing). Existing checkpoint/resume specs pass unchanged.
