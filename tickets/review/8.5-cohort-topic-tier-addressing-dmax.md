# Review: tier addressing (coord_d), tier ladder, and d_max

Implements three coupled, **pure-logic** db-core modules. No FRET imports, no libp2p, no state
machines — everything is driven through the injected ports (`IRingHash`, `ISizeEstimator`) from the
package-layering ticket. db-p2p will later supply the FRET-backed port impls.

## What was built

- **`addressing.ts`** — `TierAddressing` interface + `HashTierAddressing` impl (factory
  `createTierAddressing`, default `F = 16`). Computes `coord_0 = H(0x00 ‖ topicId)` and
  `coord_d = H(d ‖ prefix(P, d·log₂F) ‖ topicId)` for `d ≥ 1`, hashing through the injected
  `IRingHash`. `coord(d,…)` dispatches `d === 0` to `coord0`. Exported `prefixBits(P, n)` extracts
  the `n` MSBs MSB-first into `⌈n/8⌉` bytes (trailing low bits zeroed), left-padding when `P` is
  shorter than `n` bits. The tier byte is one byte (guards `d ≤ 255`). Constructor validates `F` is
  a power of two `≥ 2`.
- **`tiers.ts`** — `Tier` (T0..T3), `ALL_TIERS`, `NodeProfile`, `NodeProfileOverrides`,
  `edgeProfile`/`coreProfile`. Edge base = `{T0,T1}`, Core base = `{T0,T1,T2,T3}`. Operator
  overrides can only **narrow** (intersection with the base), so an Edge node can never be widened
  to T2/T3. Only static classes + profile flags + override surface — no willingness/TTL runtime
  behavior (those are later tickets, per the doc).
- **`dmax.ts`** — `DMaxComputer` + `makeDMaxComputer({ estimator, F, confidenceMin?, dMaxCap? })`.
  `dMax()` reads the injected `ISizeEstimator` lazily, computes `max(0, ⌊log_F(n_est)⌋ − 1)` capped
  at `dMaxCap`, and clamps to `⌊dMaxCap/2⌋` when `confidence < confidenceMin`. Defaults exported:
  `DEFAULT_CONFIDENCE_MIN = 0.3`, `DEFAULT_D_MAX_CAP = 60`.
- **`index.ts`** — re-exports all three modules (already flowed to `db-core/src/index.ts` via the
  existing `cohort-topic` barrel).
- **`docs/cohort-topic.md`** — added an "Implementation" callout under §Tier addressing (names
  `addressing.ts`, confirms the formula, records the validated zero-collision result) and under
  §Maximum useful depth (names `dmax.ts`/`tiers.ts`, confirms defaults match §Configuration).

## Review focus / known gaps (treat as a floor, not a finish line)

- **`const enum Tier` → const-object pattern.** The ticket spec wrote `export const enum Tier`, but
  db-core's test runner uses Node's native **type-stripping** (`--import ./register.mjs`), which
  rejects `enum` (non-erasable syntax) and aborts the whole suite. I substituted the idiomatic
  erasable equivalent — `export const Tier = { T0:0,…} as const` + `export type Tier = …`. Same
  `Tier.T0` member access and `0|1|2|3` value-union type; `tsc` build and runtime both pass.
  Confirm this substitution is acceptable (it changes the literal syntax from the spec but not the
  surface or values). If a true `enum` is required, the test harness would need `tsx`/compiled output.
- **`prefix(P, n)` left-pad semantics.** The doc says "n most-significant bits, left-padded if
  shorter." I read "left-pad" literally: when `P` has fewer than `n` bits, the high `(n − bits(P))`
  positions are zero and `P`'s bits sit at the low end. In practice peer IDs are 32-byte hashes
  (256 bits ≫ any realistic `n = 4·d`), so this branch is never hit on the real path — it's
  exercised only by a dedicated unit test. Verify the chosen padding direction matches what the
  db-p2p FRET binding will feed in; if FRET right-pads short keys, this disagrees (but FRET keys are
  also full-width, so likely moot). Flagged because the doc phrasing is genuinely ambiguous.
- **Coord byte-layout compatibility is asserted, not enforced here.** `addressing.ts` hashes through
  `IRingHash` at whatever `ringBits` it was constructed with (`RING_BITS = 256` default lines up
  with FRET). db-core has **no** way to check the coords actually match FRET's ring on the wire —
  that contract lives in the db-p2p binding (ports.ts documents it). Reviewer: there is no test that
  a db-core coord equals a FRET coord for the same input, because db-core can't import FRET. The
  closest guarantee is `ring-hash.ts`'s "256-bit = full SHA-256 = FRET digest" comment.
- **`DMaxComputer` consumes `ISizeEstimator` (`{nEst, confidence}`), not FRET's `SizeEstimate`
  (`{n, confidence}`) directly.** The ticket's illustrative signature showed `estimate: () =>
  SizeEstimate` wrapping FRET; per the layering rule and Phase-1 note ("drive d_max off the injected
  `ISizeEstimator`, not FRET directly") I bound it to the port instead. The FRET→port adaptation
  (`n → nEst`) belongs to db-p2p. Confirm this is the intended seam.
- **`floorLogF` boundary correction.** `⌊log_F(n)⌋` uses `Math.log` then corrects with integer
  `Math.pow(F, d)` comparisons in both directions to defeat float error at exact powers (e.g.
  `log_16(4096)` can return `2.9999…`). Covered by the powers-of-F test, but worth a skim for very
  large `n` (near `Number.MAX_SAFE_INTEGER`) where `Math.pow` itself loses integer precision — the
  `d_max_cap` ceiling bounds the blast radius, and a test exercises the `MAX_SAFE_INTEGER`/`F=2`
  case, but the intermediate `Math.pow` is not exact there.
- **Default constants vs §Configuration.** `F=16`, `d_max_cap=60`, `confidence_min=0.3` match the
  folded-back doc defaults (all "confirmed as written" by the simulator). No revision from the
  simulator fold-back touched `F` or `d_max_cap`, so the defaults stand. Re-confirm against
  §Configuration if any later doc edit lands.
- **Tier ladder is static only.** The decline-policy *behavior* (T0 never declined while serving,
  T3 declined freely) and the Edge runtime overrides (`ttl=60s`, `ping_interval=20s`, sticky
  backups) are intentionally **not** here — they belong to the willingness/TTL tickets. This module
  surfaces only the profile flags. Confirm the boundary.
