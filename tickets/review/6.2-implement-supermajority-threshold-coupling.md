# Review handoff — single-source + fail-fast-couple the super-majority threshold

## What the ticket asked for

`superMajorityThreshold` is the fraction of a cluster's peers that must promise
before a transaction proceeds. Its **default-when-config-absent had drifted**
across components: member `?? 1.0` (unanimity), coordinator `?? 0.75`, node
composition root `?? 0.67`. A member expecting 1.0 while the coordinator commits
at 0.75 is a silent phase-disagreement — nothing caught it at startup.

Two coupled changes were requested and delivered:
1. **One shared default constant** referenced by every *fallback default*.
2. **A fail-fast startup assertion** so a node whose member and coordinator would
   run different thresholds throws at construction instead of disagreeing
   mid-consensus.

## What was implemented

- **`DEFAULT_SUPER_MAJORITY_THRESHOLD = 0.75`** — new exported constant in
  `packages/db-core/src/cluster/structs.ts`, next to `ClusterConsensusConfig`
  (whose doc already said 0.75). Re-exported through `@optimystic/db-core` via
  the existing `cluster/index.ts` barrel.
- **Three fallback defaults now reference the constant** (no more `1.0` / `0.75`
  / `0.67` literals as *defaults*):
  - `cluster-repo.ts:250` (member) — was `?? 1.0`.
  - `coordinator-repo.ts` policy — was `?? 0.75`.
  - `libp2p-node-base.ts` `consensusConfig` — was `?? 0.67`.
- **Effective-threshold getters** added to both `ClusterMember`
  (`effectiveSuperMajorityThreshold`) and `CoordinatorRepo` (stores
  `policy.superMajorityThreshold`, exposes the same getter). These report the
  value each component *actually* uses for vote counting.
- **`assertSuperMajorityCoupling(member, coordinator)`** — new tiny module
  `packages/db-p2p/src/cluster/supermajority-coupling.ts`. Throws if the two
  resolved thresholds differ, with a message naming **both values and their
  provenance** (`consensusConfig.superMajorityThreshold` /
  `DEFAULT_SUPER_MAJORITY_THRESHOLD`). Extracted into a helper specifically so it
  is unit-testable without spinning up libp2p.
- **`libp2p-node-base.ts` calls the assertion** right after both `clusterImpl`
  (member) and `coordinatedRepo` (coordinator) are constructed — the composition
  root, before transaction recovery / accepting requests.

## Design choices a reviewer should sanity-check

- **Value chosen 0.75** (per ticket): the coordinator that actually commits
  already used it and the type documents it. Raising the member from 1.0 to 0.75
  removes the unanimity-vs-supermajority split; the ticket judged nothing depends
  on the member being unanimous.
- **The assertion reads RESOLVED values, not config inputs.** In the current node
  both member and coordinator are fed from one `consensusConfig` object, so the
  assert cannot actually fail today — it is a *guard against future drift*
  (divergent defaults, a mis-threaded `clusterPolicy`, a per-deployment config
  reaching only one side). Reviewer should confirm this is the intended posture
  (startup guard) rather than expecting it to catch a live mismatch that the
  current wiring can already produce — it can't, by construction. That is the
  point: keep it un-failable-by-accident and let it catch regressions.
- **Explicit caller overrides were deliberately NOT touched** (per ticket):
  `collection-factory.ts`'s `0.51`, harness values, test specs' explicit
  thresholds. Only the *absent-config default* was unified.

## Harness audit result (ticket asked for this)

- `mesh-harness.ts` wires member (`:143-153`) and coordinator (`:260-271`) each
  with `options.superMajorityThreshold ?? 0.75` — **identical expression on both
  sides**, so they always match. No change made; they cannot trip a coupling
  check. (Note: the harness constructs member + coordinator directly and does
  NOT call `assertSuperMajorityCoupling` — only the real node path does. If the
  reviewer wants belt-and-suspenders, adding the assert call to the harness's
  per-node build would be a low-risk addition, but it is not required for
  correctness since both sides share one expression.)
- `collection-factory.ts:285` calls `createMesh(1, { superMajorityThreshold:
  0.51 })`; the harness threads that single value to both sides → matched.

Targeted tests (all green locally):

```
  test/cluster-repo.spec.ts test/cluster-membership-admission.spec.ts --reporter spec
```

Coverage delivered (ticket's a/b/c):
- **(a) mismatch throws at startup** — `supermajority-coupling.spec.ts`: helper
  throws on unequal thresholds; real `ClusterMember(1.0)` + `CoordinatorRepo(0.75)`
  rejected; error message asserted to name both values + provenance.
- **(b) consistent config starts cleanly, both read the same value** — helper
  passes on equal; real member+coordinator from a shared 0.6 config both report
  0.6 and couple; **regression case**: both default to 0.75 with no config (the
  old 1.0-vs-0.75 drift is gone).
- **(c) single-node bring-up still commits with the unified default** —
  `mesh-sanity.spec.ts` Suite 0: a 1-node mesh created *without* an explicit
  threshold pends + commits (`ceil(1 * 0.75) = 1`, unanimity-of-one still works).

## Known gaps / where to push (reviewer: treat tests as a floor)

- **No end-to-end test that drives `createLibp2pNode` through the real assertion.**
  The assert is exercised via the extracted helper + real class getters, not by
  booting a full libp2p node with a deliberately-divergent config. Booting a real
  node to force the mismatch is heavy (and the current wiring makes a genuine
  mismatch hard to inject without reaching into internals). If the reviewer wants
  true end-to-end coverage, that is the gap to close.
- **Vote-count arithmetic unchanged but not re-proven at every site.** The formula
  `Math.ceil(peerCount * threshold)` at `cluster-repo.ts:741`, `:1324` and
  `cluster-coordinator.ts:280` was not modified; only the default *source*
  changed. `cluster-coordinator-supermajority.spec.ts` still passes (0.67/0.51
  math), but no new test pins `cluster-repo.ts`'s two ceil sites specifically.
- **`docs/cluster.md`** already documented 0.75 in the relevant spots, so it was
  left as-is; it does not mention the new constant name or the coupling assert.
  Non-blocking, but a doc line pointing at `DEFAULT_SUPER_MAJORITY_THRESHOLD` and
  the startup guard would help future readers.
- **`reference-peer` CLI** threads `superMajorityThreshold` only via
  `clusterPolicy` when the operator passes `--super-majority-threshold`; otherwise
  it falls through to the node-base default (now the constant). Not re-tested here
  — it is an explicit-override path, unaffected by the default change.

## Tripwire recorded (not a ticket)

- The coupling assertion is by-construction un-failable on today's node wiring
  (both components share one `consensusConfig`). It only earns its keep if someone
  later threads member and coordinator from *separate* config sources. Noted here
  so a future reader knows the guard is dormant-by-design, not dead code. Parked
  as this finding line; no code `NOTE:` added because the helper's own doc comment
  already explains the "future drift" rationale at the site.
