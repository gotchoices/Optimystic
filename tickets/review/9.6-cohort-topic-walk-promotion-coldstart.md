# Review: walk / promotion / cold-start (db-core cohort-topic dynamics)

This wires together the dynamic behavior of the FRET-backed topic tree, all as **FRET-free db-core
logic over injected ports** (`ITopicRouter`, `TierAddressing`, `DMaxComputer`, `CohortSigner`,
`RegistrationStore`): how a participant *finds* its cohort (walk-toward-root), how a cohort
*grows/shrinks* (promotion/demotion), and how a cohort *comes into being* (cold-start).

## What landed

### Walk (`walk.ts`)
`WalkEngine.register(topicId, tier, appPayload?)` walks toward the root from `d_max`, one probe per
RPC over the injected `ITopicRouter` (db-p2p will bind it to FRET's `RouteAndMaybeAct`). Reply
dispatch matches `docs/cohort-topic.md` §Lookup:
- `accepted` → `{ kind: "accepted", reply }` (reply carries `primary`/`backups`/`cohortEpoch`).
- `no_state` → step inward (`d − 1`); at the root, re-issue **once** at tier 0 with `bootstrap: true`.
- `promoted(targetTier)` → the **one** outward move: recompute `coord_targetTier` and register there
  (default `followPromoted: true`); with `followPromoted: false` it surfaces `{ kind: "promoted",
  targetTier }` for a caller-driven loop.
- `unwilling_member(candidates)` → direct `dialMember` retry of a named sibling at the **same** coord.
- `unwilling_cohort(retryAfter)` → `{ kind: "retry_later", afterMs }`; the caller backs off in time
  and a fresh `register` restarts at `d_max` (decorrelates retries — never re-hits the declined coord).
- `RegisterV1` build + signing is delegated to an injected `RegisterMessageFactory` (participant
  identity/crypto live there, not in the engine).
- A `maxSteps` safety valve (scales with `d_max`) bounds pathological inward/outward oscillation.

### Promotion / demotion (`promotion.ts`)
`PromotionLifecycle` keyed by `topicId`:
- `onParticipantCountChange(topicId, now)` — eager per-arrival; fires the cap (`cap_promote = 64`),
  hot-fast-path (`cap_promote_fast = 32` at `bucket ≥ bucket_overload = 6`), and slope pre-promotion
  (predicts crossing within `T_promote_lookahead = 30s`) triggers; returns a threshold-signed
  `PromotionNoticeV1` (or `undefined`).
- `maybeDemote(topicId, now)` — gossip-tick driven; enforces `cap_demote = 16` held for
  `T_demote = 5min`, the no-live-children requirement, the `T_promote_sticky = 60s` floor, and
  **root-never-demotes** (tree tier 0); returns a threshold-signed `DemotionNoticeV1` to the parent.
- `isPromoted(topicId)`.

### Cold-start (`coldstart.ts`)
- `shouldInstantiate({ bootstrap, followOn, quorumWilling })` = `(bootstrap ∨ followOn) ∧
  quorumWilling`. A speculative `d_max` probe (neither flag) → `false` (gets `NoState`, no parallel
  branch).
- `createForwarder(tier)` / `ColdStartManager` — link-up state machine: a deeper forwarder starts
  `awaiting_parent` (accepts participants, **holds** parent-involving ops) and flips to `serving` when
  its `ParentRegistrar.registerWithParent` acks; the root (tier 0) serves immediately.
- `promotedRedirectReply(targetTier, traffic?)` — the **just-promoted-burst bounce** (GROUNDING
  resolution): a just-promoted cohort whose child tier isn't instantiated yet bounces the in-flight
  burst with `Promoted(d+1)`, never buffering, never `UnwillingCohort`.

## Use cases to probe in review
- **Walk single-direction discipline.** Sparse-regime fan-out: every participant starts at a
  *distinct* `coord_{d_max}` (covered) — confirm no path produces an outward move that isn't a
  `Promoted` follow. Adversarial: a tree that alternates `NoState`/`Promoted` at the same tier (does
  `maxSteps` terminate cleanly, and is the default bound generous enough for legitimate deep walks at
  `d_max ≈ 6`?).
- **Promotion overshoot.** The eager path promotes at exactly `cap_promote` (overshoot 0); the
  simulator's real bound is the *gossip-lagged* `< arrivalsPerRound`. Confirm the eager wiring is the
  intended production model and that a future gossip-lagged caller still respects the bound.
- **Demotion hysteresis.** Children-block, `T_demote`, sticky window, root-never-demote are each
  tested in isolation — check the **combined** interaction (e.g. children clear *during* the
  `T_demote` hold; does the low-load clock stay anchored?).
- **Cold-start gate.** `shouldInstantiate` truth table is covered; the live concern is the `followOn`
  source (see gaps).
- **Burst bounce.** `promotedRedirectReply` is `Promoted`, never `UnwillingCohort` (covered);
  confirm the promotion sticky window genuinely keeps the cohort promoted across a realistic burst.

## Known gaps / honest flags (treat the tests as a floor)
1. **No db-p2p binding exists yet.** `packages/db-p2p/src/cohort-topic/` is still empty — there is no
   concrete `ITopicRouter` (FRET `registerMaybeAct`/`sendMaybeAct`), `ParentRegistrar`, or
   `RegisterMessageFactory` implementation. All three modules are db-core logic exercised only against
   in-test fakes. End-to-end (mock-tier e2e, simulator-number cross-checks named in the ticket) is
   **not** wired and is deferred to the db-p2p layering work.
2. **`followOn` is an un-wired integration seam.** `RegisterV1` carries only `bootstrap` — there is no
   wire signal distinguishing a `Promoted` follow-on from a speculative probe at a cold tier-`(d+1)`
   cohort. `shouldInstantiate` takes `followOn` as an explicit input; **how the db-p2p cohort host
   derives it from routing context is unresolved** and may need a design decision (or a wire-format
   revisit, which would touch the wire ticket). This is the single biggest open item.
3. **Lifecycle is not yet *driven*.** Nothing calls `onParticipantCountChange` per arrival or
   `maybeDemote` on a gossip tick — the cohort-side activity handler / gossip loop that does so is
   db-p2p work, not present here.
4. **Interface deviation from the doc sketch (documented).** `PromotionLifecycle` methods are `async`
   and *return* the signed notice rather than the doc's `void` "may emit" sketch, because threshold
   signing is asynchronous. Confirm reviewers are comfortable with this shape.
5. **Cold-start parent-registration failure** is logged (`console.warn`) and leaves the forwarder
   accepting participants but holding parent-ops; there is **no retry loop** — a host must re-drive
   `instantiate`/registration. Reasonable for now; flagged in case review wants a built-in retry.
6. **`maxSteps` default is heuristic** (`2·(d_max+2) + maxMemberRetries + 8`), not derived from a
   proof of convergence. It only bites in malformed trees; worth a sanity check against the deepest
   realistic `d_max`.
