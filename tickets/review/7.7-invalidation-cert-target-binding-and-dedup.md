# Review: invalidation certificate тАФ target binding (#2) + per-arbitrator dedup (#3)

Part 1 of 2 of the forgeable-invalidation-certificate fix. Closes **#2 target binding** (a genuine
`challenger-wins` proof for transaction X can no longer be replayed to revert an unrelated, innocent
transaction Y) and **#3 per-arbitrator dedup** (one signed vote replicated N├Ч can no longer manufacture
a super-majority). The third binding тАФ **#1 arbitrator-set legitimacy** тАФ is the follow-up
`7.8-invalidation-cert-arbitrator-set-binding` and is intentionally still open.

## What changed (implementation summary)

### Wire format тАФ versioned, target-bound vote payload (v2)
- Arbitration votes now sign **`v2:${disputeId}:${vote}:${computedHash}:${targetHash}`**, where
  `targetHash = hashString(`${messageHash}|${invalidatedActionId}|${sortedBlockIds.join(',')}`)`.
  `sortedBlockIds` is lexically sorted so the binding is order-independent across members; `hashString`
  is the shared db-core helper, so every member recomputes an identical hash.
- A `version: 'v2'` discriminator was added to **`ArbitrationVoteProof`** (`db-core/src/log/struct.ts`)
  and **`ArbitrationVote`** (`db-p2p/src/dispute/types.ts`). The verifier rejects any vote whose
  version is absent/unrecognized **before** counting it тАФ v1/legacy is never accepted-by-default.

### Verifier тАФ `verifyInvalidationCertificate(proof, target: CertificateTarget)`
- New required second arg `CertificateTarget = { invalidatedActionId, blockIds }`. The verifier
  recomputes `targetHash` from `proof.messageHash` + `target` and checks every vote's signature over the
  v2 payload built from it. Feeding a different target makes every signature fail тЖТ 0 decisive тЖТ reject.
- **Dedup/equivocation:** decisive verdicts are accumulated per `arbitratorPeerId` in a
  `Map<peerId, 'agree-with-challenger' | 'agree-with-majority' | 'equivocated'>`. A repeated identical
  decisive vote counts **once**; an arbitrator that appears with two *different* decisive votes is marked
  `equivocated` and dropped from **both** tallies. `inconclusive` votes stay non-decisive. The rule is
  documented in the verifier doc-comment.

### Callers
- `applyInvalidation` (invalidation.ts): verifies against `params.certificateTarget ?? { invalidatedActionId, blockIds }`.
- `applyConsensusInvalidation` (cluster-repo.ts): the **network-facing** path тАФ verifies against the
  request's **own** `{ invalidatedActionId, blockIds }`. This is the replay boundary: a genuine proof in
  a request pointing at a different action/blocks fails here. No `certificateTarget` override.
- `DisputeService.makeVote` / `handleChallenge`: arbitrators derive the target from the challenge's
  `originalRecord` (reusing `extractInvalidationTarget`) and sign the v2 payload. `version: 'v2'` set.

### тЪая╕П Discovered integration gap NOT in the ticket тАФ the cascade (read carefully)
The ticket said `verifyInvalidationCertificate` "has exactly two production callers." True тАФ but
**`cascadeInvalidate` (cascade.ts) calls `applyInvalidation` for every cascade child**, reusing the
**root's** proof to authorize reverting read-dependents whose own `(invalidatedActionId, blockIds)`
differ from the root. Under naive target binding every child would fail verification (the votes are
bound to the *root's* target). Resolution:
- Added optional `certificateTarget?: CertificateTarget` to `ApplyInvalidationParams` (defaults to the
  call's own target).
- `cascadeInvalidate` computes `rootCertificateTarget = input.certificateTarget ?? { invalidatedActionId: rootActionId, blockIds: [...new Set(seed.map(s => s.blockId))] }`
  and passes it to each child `applyInvalidation`. (Also added `certificateTarget?` to `CascadeInput`
  for explicit production override.)

**Security rationale (please scrutinize):** this decouples *what is verified* (root target) from *what
is reverted* (child target) on the **local** cascade path only. It is safe because:
1. The network-facing path (`applyConsensusInvalidation`) verifies against the request's own target тАФ
   no decoupling there, so the #2 replay boundary is fully closed for anything an attacker can send.
2. Cascade children are derived **locally and deterministically** by each member from a
   consensus-verified root + local logs (per docs ┬зRead-Dependency Cascade), not supplied by a peer.
   The certificate on a child is defense-in-depth attesting the *root* is invalid; the child-specific
   justification is the deterministic derivation.

**Follow-up flag for 7.8 / cascade-wiring:** if cascade children are ever driven through consensus
(`applyConsensusInvalidation`, the "its own consensus" option the docs mention) rather than local
deterministic replay, they would be rejected by target binding unless given per-child proofs or the same
`certificateTarget` threading. Not a problem today (cascade is not production-wired; it only calls the
local `applyInvalidation`), but it must not be forgotten when the cascade is wired.

## Honest gaps / things for the reviewer to poke at (tests are a floor, not a ceiling)

- **The cascade decoupling is the riskiest part of this change.** Verify the security rationale above
  holds, and that there is no code path where a peer-supplied request reaches `applyInvalidation` with an
  attacker-chosen `certificateTarget`. (Today only `cascadeInvalidate` sets it, from locally-derived
  data; `applyConsensusInvalidation` never sets it.)
- **`rootCertificateTarget` is derived from `seed` blockIds** when not passed explicitly. This relies on
  `seed` being a faithful 1:1 image of the root's commit `blockIds` (it is, in the current code:
  `seed = result.reverted` and there is one reverted entry per root blockId, including the delete-restore
  sentinel). If a future change makes the seed a subset/superset, child verification would silently break.
  An explicit `CascadeInput.certificateTarget` override exists to sidestep this; consider whether
  production wiring should always pass it.
- **`invalidation-notification.spec.ts` was NOT modified.** The ticket claimed it "constructs
  `InvalidateRequest`s that flow through `applyConsensusInvalidation`" тАФ it does **not**. It builds a
  `CollectionChangeEvent` + a commit cert and tests `buildNotificationV1`; there are no arbitration-vote
  helpers or dispute proofs in it. It still passes unchanged. (If the reviewer wants belt-and-suspenders
  coverage of an end-to-end notification carrying a target-bound proof, that would be net-new.)
- **Compile-required test edits beyond the ticket's list:** `dispute.spec.ts` (inline `ArbitrationVote`
  literals for `resolveDispute`), `db-core/test/invalidation-client.spec.ts`, and
  `db-core/test/log-invalidation.spec.ts` each gained `version: 'v2'` on their proof literals тАФ required
  by the now-mandatory field, no behavioral change.
- **Inconclusive votes and dedup:** the dedup map only tracks *decisive* verdicts, so an arbitrator that
  casts an inconclusive + a decisive vote is counted as the single decisive vote (inconclusive ignored).
  This matches intent (inconclusive is non-decisive) but is a subtlety worth a second look.
- **No end-to-end disputeтЖТoriginateтЖТapply test** asserting a *real* `DisputeService`-produced proof
  verifies against the *real* `InvalidateRequest` target. The pieces are wired to use the same
  `extractInvalidationTarget` derivation on both sides (so they're consistent by construction), and
  `dispute.spec.ts` covers origination, but the full round-trip through `applyConsensusInvalidation` with
  a service-generated proof is not directly asserted. Reasonable target for an added integration test.
