# Review: Reactivity PushState gossip over the cohort bus

## What was built

`ReactivityPushStateGossipDriver` (`packages/db-p2p/src/reactivity/push-state-gossip.ts`, new; exported
from `reactivity/index.ts`) drives db-core's already-tested `PushState` gossip codec
(`serializeGossip`/`mergeGossip`, `encode/decodePushStateGossipV1`) over the wire on a periodic cadence, so
the replay ring + dedupe window converge across a whole forwarder cohort. This is the **replication**
backstop to the live fan-out (`reactivity-forwarder-host`): a member that missed an origin dial still
receives the entry via gossip and can serve a backfill (12.4) after a primary failover — making the doc's
"**any** cohort member can serve a replay/backfill" property real over libp2p.

The module is deliberately **transport-agnostic + node-assembly-agnostic**: it depends only on the
`broadcastOver` seam (a `Pick<FretCohortGossipTransport, "broadcastOver">`) and the db-core codec, so it is
unit-tested with a fake transport and **real** `PushState`s. The actual node wiring (registering the
inbound handler, supplying `liveCollections`/`pushStateForGossip`/`gossipTransport`/`isCohortMember`,
calling `start()`) is the downstream `reactivity-notification-transport` (12.33) ticket — see its step 5,
which names this exact seam.

### Shape (as built)

```
class ReactivityPushStateGossipDriver {
  start(): void                                  // unref'd cadence timer; idempotent; no-op after stop()
  round(): void                                  // broadcast each live collection's serialized PushState (clipped)
  deliver(fromPeerId, frame: Uint8Array): void   // decode → member gate → resolve → mergeGossip (never throws)
  stop(): void                                   // stopped short-circuit + clearInterval
}
```

Deps (`ReactivityPushStateGossipDriverDeps`): `gossipTransport` (`broadcastOver`), `liveCollections()`
(`Iterable<{ pushState, cohortCoord }>`), `pushStateForGossip(g) → PushState | undefined`, optional
`isCohortMember(fromPeerId, g)`, `intervalMs` (default `DEFAULT_GOSSIP_INTERVAL_MS` = 5 s), `maxBytes`
(default `DEFAULT_STREAM_MAX_BYTES` = 512 KiB), optional `clock` (reserved, unused), optional `onTruncate`
(see below).

- **Broadcast** rides `broadcastOver(PROTOCOL_REACTIVITY_PUSH_STATE_GOSSIP, cohortCoord, frame)` — the same
  cohort-peer-resolution + self-exclusion + per-peer failure swallowing the promote-notice broadcast reuses.
- **Cadence** is its own unref'd `setInterval` with the host tick's discipline verbatim: `stopped`
  short-circuit, `rounding` re-entrancy guard, `unref()` so it never pins an idle process, per-collection
  try/catch isolation, mid-loop `stopped` break.
- **Inbound** `deliver` decodes (try/catch → drop+log), runs the `isCohortMember` gate (drop non-member),
  resolves the owning collection (`undefined` ⇒ ignore), then `mergeGossip` (which independently re-guards
  collection/topic — defense in depth).

## Use cases / validation (tests are a floor — see gaps)

`packages/db-p2p/test/reactivity/push-state-gossip.spec.ts` (9 specs, all green) over real db-core
`PushState` + real `serveBackfill`, fake `broadcastOver` capturing frames:

- **Any member serves replay.** A ingests an origin notification → one `round()` → B `deliver`s the frame →
  `serveBackfill(B, …)` returns the same entry A would; the full signed `NotificationV1` crosses intact.
- **Frame-size bound (load-bearing).** A 60-notification ring exceeds the (small, test-only) bound; one
  `round()` ⇒ broadcast frame stays `≤ maxBytes`, the clip is **surfaced** (`onTruncate` + log, no silent
  cap), and the kept slice is the most-recent (high-revision) contiguous run.
- **Convergence (streaming).** One revision per round with a small bound ⇒ every frame stays bounded, the
  ring grows large enough to exercise clipping, and B converges on A's **full** ring (`serveBackfill(B)` ==
  `serveBackfill(A)`).
- **Unserved collection** (`pushStateForGossip → undefined`) ignored; **foreign collection** rejected by
  `mergeGossip` even if a resolver mis-routes it (defense in depth).
- **Membership gate** drops a non-member sender before resolve/merge; the same frame from a member merges.
- **Undecodable frame** dropped without throw or merge.
- **Timer** `start()` ticks (and is idempotent), `stop()` halts further rounds (and a manual `round()` after
  stop is a no-op); empty live set ⇒ no broadcasts, no throw.

Build (`yarn workspace @optimystic/db-p2p build`) clean; full suite `767 passing, 27 pending, 0 failing`
(`/tmp/ps-gossip-full.log`); type-check via the workspace-local `tsc --noEmit` exit 0 (note: `npx tsc`
resolves a newer global TS that errors on the **pre-existing** `downlevelIteration` tsconfig deprecation —
not a code issue; the build's own tsc 5.9.3 is clean).

## Honest gaps to flag for review

- **Node wiring is NOT in this ticket** (by design — it is 12.33's step 5). Reviewer should NOT expect a
  live timer or registered handler anywhere yet; the driver is inert until 12.33 constructs and `start()`s
  it. The fake-transport unit tests are the floor.

- **`ReactivityForwarderHost` exposes no live-collection iterator nor a `(collectionId,topicId)→PushState`
  lookup.** It has only `pushStateFor(topicId)` (single, private `served` map keyed by topicId-b64). 12.33's
  `liveCollections` / `pushStateForGossip` therefore need a new accessor on the host (or must resolve
  collectionId/topicId → topicId → `pushStateFor`, relying on `mergeGossip`'s own collectionId guard). I did
  **not** add that accessor (out of this ticket's scope — driver + tests only); flagging it so 12.33 doesn't
  rediscover it. A minimal `servedTopics(): Iterable<{ collectionId, topicId, pushState }>` on the host
  would be the clean addition.

- **`cohortCoord` is a dep this ticket cannot source itself.** Gossip must broadcast to each served
  collection's FRET cohort coord, but the host/`PushStateInit` carry no `RingCoord`. 12.33 must thread it
  (likely `createTierAddressing(ringHash).coord0(topicId)`, the same derivation the notification verifier
  uses). The driver just consumes whatever `liveCollections` supplies.

- **Most-recent-N convergence has a documented limit.** Convergence to a member's full ring requires the
  per-round ingest delta to fit one frame: a burst larger than the frame-fitting window within one gossip
  interval leaves older entries un-replicated **by that member's own gossip**. Those are covered by (a)
  other cohort members holding them and (b) the checkpoint/backfill (12.4) path — consistent with the doc's
  layering (replay = recent window; checkpoint = deep history). A **rotating-window** truncation (advance an
  older slice each round) would guarantee full *static*-ring convergence from a single member; I kept the
  spec's literal "most-recent-N" (newest-first is what a lagging subscriber backfills first) and flag the
  rotating window as a possible enhancement, not a defect.

- **Dedupe-window floor.** The driver always keeps the **full** dedupe window (per spec — "it is small").
  If `maxBytes` were ever smaller than the dedupe-window floor (~2 KB at `dedupe_window=64`; never true at
  the 512 KiB production bound), the floor frame is broadcast even if marginally over-bound — still surfaced
  via `onTruncate`/log, never silent. The tests use a bound comfortably above the floor for this reason. If
  a reviewer wants a hard guarantee, the dedupe window could also be clipped, but that deviates from the
  spec and is unnecessary in production.

- **`onTruncate` dep is a small addition beyond the ticket's interface sketch** — a test/diagnostic seam
  that makes the "no silent cap" rule verifiable without scraping `debug` output. Production wiring can omit
  it (the log still fires). `clock` is accepted for parity with the host tick but is currently unused
  (gossip frames carry their own `receivedAt`).

- **Envelope signing deferred (as designed).** PushState-gossip frames are authenticated by the
  `isCohortMember` gate + the fact that every replay entry is a full end-to-end-verifiable `NotificationV1`
  (a backfill *serve* re-verifies entries regardless, so a poisoned ring entry can only waste a slot, never
  forge a delivery). Full peer-sig envelope signing of the frame is filed as the backlog hardening ticket
  `reactivity-pushstate-gossip-envelope-signing` (per the implement ticket's instruction; 12.33 also
  references it).

## Suggested review focus

1. The frame-bound binary search in `boundedFrame`/`encodeSlice`: monotonicity assumption, the
   `kept < total` truncation signal, and that the broadcast frame is genuinely `≤ maxBytes` in the normal
   (dedupe-floor-fits) case.
2. Timer discipline parity with the host tick (`stopped`/`rounding`/`unref`/idempotent `start`).
3. `deliver`'s never-throw contract and the gate→resolve→merge ordering (gate before resolve before merge).
4. Whether the host-accessor + `cohortCoord` gaps should be closed here (minor host addition) or left to
   12.33 as currently flagged.
