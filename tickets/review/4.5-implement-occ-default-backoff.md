## What shipped

Backoff+jitter is now the built-in default retry policy for optimistic-concurrency losses. Two
gaps from the ticket are closed:

1. **Jitter added to the single-collection sync retry.** `Collection.sync` previously backed off on
   a bare exponential curve (`base·2^(n-1)`, capped) — deterministic, so a herd of clients that lost
   the same race re-attempted in lockstep. It now uses `jitteredBackoffMs`, which multiplies the
   exponential value by a random factor in `(0.5, 1]` so re-attempts scatter across a window.

2. **The multi-collection commit path gained a retry loop.** `TransactionCoordinator.commit()`
   previously threw on any loss and left retry to the caller. It now wraps the single-attempt work
   (`commitOnce`) in a bounded backoff+jitter retry that re-drives ONLY a *clean stale loss*.

### The shared helper (`utility/backoff.ts`)

`jitteredBackoffMs(attempt, {baseMs, capMs, factor?, jitterFraction?}, rand?)`:
```
exp   = min(baseMs · factor^attempt, capMs)          // factor default 2
delay = exp · (1 - jitterFraction · rand())          // jitterFraction default 0.5, rand() ∈ [0,1)
```
So with defaults the delay is uniform in `(0.5·exp, exp]` — never zero (given `baseMs > 0`), never
above `capMs`. `rand` defaults to `cryptoRand` (a uint32 from `@noble/hashes` `randomBytes` / 2³², NOT
`Math.random`) and is injectable for deterministic tests. The abort-sleep helper (`abortableDelay`)
and `makeAbortError` were also moved here and are now shared by `collection.ts` and `coordinator.ts`
(the previously-duplicated private copies in `collection.ts` were removed). `SyncOptions.rand` was
added so both retry loops can take an injected rand.

### The coordinator retry (the part to scrutinize hardest)

`commit(transaction, options?)` loops around `commitOnce`. It retries **only** a
`CoordinatorStaleLossError`; anything else — `CoordinatorPartialCommitError`, an expired transaction,
an unavailable transactor, a thrown cluster error — propagates immediately without retry. Between
attempts it backs off (jittered) and re-reads every registered collection via `collection.update()`
so the next attempt pends against fresh revisions. `maxAttempts`, `deadlineMs`, and an abort `signal`
bound it. Defaults are safe out of the box (`maxAttempts` 10, `baseBackoffMs` 100, `maxBackoffMs`
5000 — mirroring the sync defaults).

**How "clean stale loss" is distinguished from a hard failure** (this is the subtle bit): a new
`staleLoss` boolean is threaded up through `pendPhase`/`commitPhase`/`coordinateTransaction`, and the
EMPTY-committed-set branch of `commitOnce` throws `CoordinatorStaleLossError` iff `staleLoss` is true,
else a plain `Error` (fail-fast, as before). Classification:
- **PEND** is a retryable conflict only when the failure carries `missing` or `pending` (a committed
  or pending action collided) — surfaced via a new `PendRejectedError.conflict` flag. A bare `reason`
  (storage full, policy reject) is **not** retryable.
- **COMMIT** is a retryable stale loss on any returned `{success:false}` (the codebase's designated
  "someone committed a newer rev" signal); a thrown/transient-exhausted failure is not.
- `staleLoss` at the phase level holds only if **every** failure was stale and none was hard.

A **partial landing** (`committedCollections` non-empty) still throws `CoordinatorPartialCommitError`
and is never retried — blindly re-driving would re-log the already-durable half.

## Use cases / validation to exercise (the reviewer's floor, not ceiling)

Tests added/updated (all green — full db-core suite 1221 passing; quereus-plugin 308; db-p2p 1208):

- **`test/backoff.spec.ts`** — `jitteredBackoffMs`: rand=0 → full exponential; cap clamps; rand→1 →
  ~half; custom factor/jitterFraction; RangeError on bad attempt; property test `0 < delay ≤ cap` for
  all attempt/rand; expected-value non-decreasing; thundering-herd spread (200 draws land in
  `(0.5·exp, exp]` and are mostly distinct); `cryptoRand` ∈ [0,1) and spreads.
- **`test/transaction.spec.ts` → "Coordinator commit backoff+jitter retry"** —
  - clean multi-collection stale loss backs off, re-reads, then commits (bounded: 4 commit calls);
  - a PARTIAL landing throws `CoordinatorPartialCommitError` and is NOT auto-retried (exactly 2
    commit calls, winner stays durable);
  - always-losing txn exhausts `maxAttempts:3` → `CoordinatorStaleLossError`, exactly 3 attempts;
  - abort mid-backoff → `AbortError` promptly;
  - `deadlineMs` halts retry far below a 1,000,000 `maxAttempts`.
- **Updated existing tests**: the single-collection "failed commit" test now asserts auto-retry
  *success* + no double-log (was: throw-then-manual-retry); three restore-invariant tests pass
  `{maxAttempts:1}` so the always-losing case fails fast instead of retrying; the empty-committed test
  now asserts `CoordinatorStaleLossError` (still not the partial signal).

Existing single-collection sync retry tests (`test/collection.spec.ts` "bounded sync retry") were
kept green unchanged — the jitter change preserved `maxAttempts`/`deadlineMs`/abort semantics.

## Known gaps / things to challenge

Written honestly — treat the tests as a floor:

- **No genuine concurrent committer in tests.** The "re-read clears the conflict then commits"
  mechanism is exercised only against `FlakyCommitTransactor` (which refuses without actually
  advancing a rev). Nothing in these tests has a *real* racing transaction commit a competing
  revision between attempts, then verifies the loser's `update()` correctly rebases and wins. A
  stronger integration test (a transactor that commits a competing rev on attempt N) would close this.
- **Deliberate PEND asymmetry between the two retry loops.** `Collection.sync` retries *any* returned
  stale failure including a bare `reason`; the coordinator retries a PEND failure *only* on a
  `missing`/`pending` conflict. This was chosen so hard-failure paths (storage full / peer rejected /
  unavailable) fail fast rather than spinning the default 10× — but it means the two loops classify a
  bare-`reason` pend differently. Confirm that's the intended contract, not an accident.
- **Error payload on a PEND-phase loss.** `commitOnce` builds `CoordinatorStaleLossError` with
  `coordResult.failedCollections ?? new Set(allCollectionIds)`. `pendPhase` does not return
  `failedCollections`, so a pend-phase loss names *all* participants as failed. Fine for a diagnostic
  message; confirm no caller keys logic off that set.
- **Broad re-read** — the retry refreshes every registered collection, not just this transaction's
  participants. Parked as a `NOTE:` tripwire at the re-read site in `coordinator.ts` (harmless today,
  narrow it only if a coordinator ever holds many collections and retry latency shows up).
- **`makeAbortError` message changed** from `'The sync operation was aborted'` to the generic
  `'The operation was aborted'` when the abort helpers were shared. Only `err.name === 'AbortError'`
  is asserted anywhere, so no test broke, but it is a user-visible string change.
- **`execute()` is intentionally NOT retry-wrapped** (per the ticket). It still returns the
  committed/failed partition on `ExecutionResult` rather than throwing. Left as-is.
