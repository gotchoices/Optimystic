# Review: anti-flood and anti-DoS defenses for cohort-topic

The implement stage folded the spec's two defensive sections into db-core logic, FRET-free and
injection-driven, matching the existing cohort-topic module conventions (ES modules, tabs, no `any`,
no inline `import()`, byte-array peer ids, factory + deps pattern). `yarn build` and `yarn test` are
green for `db-core` (531 passing; 23 of them new).

## What was built

### Anti-flood — `cohort-topic/antiflood/`
- **`RejoinJitter`** (`jitter.ts`). Staggers a post-cohort-failure re-registration wave. Two forms:
  - `scheduleRejoin(now)` — one participant draws a uniform offset over the window (the production
    API a participant calls when it loses its cohort). Random ⇒ bound holds *in expectation*.
  - `scheduleWave(count, now)` — even spacing at `T_rejoin_jitter / cap_promote`, so any
    `T_rejoin_jitter`-long sliding window holds ≤ `cap_promote` arrivals *by construction* (the hard
    bound the storm test asserts).
  - Window widens via `failureRateScale` (claim 2: "scaled with cohort failure rate observed from
    FRET"). Defaults `T_rejoin_jitter = 30 s`, `cap_promote = 64`.
- **Invariant predicates** (`invariants.ts`). Pure, dependency-free checks over a recorded walk trace
  so unit + e2e suites can target the structural disciplines without re-implementing the walk:
  `outwardMovesArePromoted` (claim 3, no speculative outward probe), `inwardStepsFollowNoState` (walk
  discipline), `retriesRestartAtDMax` (claim 4), `stickyHolds` (claim 5 sticky window). Mirrors the
  simulator's `walk-metrics.ts` instrumentation.

### Anti-DoS — `cohort-topic/antidos/`
- **`RegisterRateLimiter`** (`rate-limiter.ts`). Sliding-window counter keyed by `(peer, topic)`,
  default 4/window, window 60 s. Over-rate ⇒ `{ ok:false, retryAfterMs }` with exponential back-off
  (reuses `backoffRetryMs` from `willingness.ts`). Strikes reset once the source goes quiet for a
  full window; idle keys pruned on access.
- **`TopicBudget`** (`topic-budget.ts`). LRU by participant count, `topics_max = 2048`. `admit` of a
  new topic when full evicts the coldest *zero-participant* resident; if every resident is populated,
  the new instantiation is refused (existing topics continue). LRU tiebreak via a monotonic access
  counter (deterministic, no wall clock).
- **`CorrelationReplayGuard`** (`replay-guard.ts`). Drops stale timestamps (`now − ts > maxAge`),
  implausibly-future timestamps (`ts > now + maxFutureSkew`), and replayed `correlationId`s.
  Remembers ids for one `maxAge` window (anything older is stale anyway); amortized pruning.
- **`BootstrapEvidence`** (`bootstrap-evidence.ts`). Tier *policy* over injected verifiers: T0/T1
  require a signed parent reference (no PoW); T2/T3 accept PoW OR reputation signature OR parent
  reference. Non-bootstrap registrations always pass. db-core embeds no specific PoW/reputation
  scheme — the verifiers are injected (db-p2p binds them).

## Use cases to validate (tests are a floor, not a ceiling)

- **Rejoin jitter bounds the storm.** A 2,000-participant wave through `scheduleWave` keeps peak
  arrivals per `T_rejoin_jitter` window ≤ `cap_promote`. The unstaggered baseline would be 2,000 in
  one instant.
- **Rate limiter.** ≤4 admits per minute per `(peer, topic)`; 5th rejected, `retryAfter` doubles;
  isolation across peers and topics; forgiveness after a quiet window.
- **Topic budget.** Evicts the cold zero-participant topic at `topics_max`; refuses new when all
  residents are populated; re-admit of a resident always allowed.
- **Replay guard.** Fresh accepted once; exact replay rejected; stale and future timestamps rejected.
- **Bootstrap evidence.** Tier policy matrix (T0/T1 parent-ref only; T2/T3 any of three; none ⇒
  reject; non-bootstrap ⇒ accept).

## Known gaps / reviewer attention

- **Invariants are not auto-wired into `WalkEngine`.** The predicates operate on a caller-supplied
  probe trace; the implement stage deliberately did **not** modify the proven `walk.ts` to record
  traces (the ticket asked for "thin wrappers ... so the e2e suite can target them"). The e2e/mock-
  tier ticket (`cohort-topic-e2e-mock-tier`) is the natural place to record a real trace from the
  engine and feed these predicates. Today they are validated against hand-authored traces only —
  confirm that satisfies the "assert the walk/promotion already enforce" intent, or wire a recorder.
- **`RejoinJitter` is not consumed by any caller yet.** Like the rest of the substrate it is a
  db-core primitive awaiting the db-p2p `CohortTopicService` host. No production call site exercises
  it; only specs do.
- **Replay-guard keying.** Keyed on `correlationId` alone (the peer id is recorded but not part of
  the key). A replay re-using the same 16 random bytes is caught regardless of source; a cross-peer
  collision of distinct honest ids is astronomically unlikely but worth a sanity check on the
  intended semantics.
- **Rate-limit window semantics.** Sliding window via retained accept timestamps (not a token
  bucket). The "forgiveness" test relies on accepts aging out, not on a hard counter reset — confirm
  this matches the intended "exponential `retryAfter`" curve under sustained vs. bursty abuse.
- **`F^d` aggregate DoS is unresolved by design** (documented as an open reputation-escalation
  question). Per-coord limits bound per-cohort cost only; the network-wide budget is deferred to the
  simulator adversarial scenario. Not a defect — verify the doc framing is acceptable.
- **No simulator cross-run in this ticket.** Values (`T_rejoin_jitter`, `register_rate_per_peer`,
  `topics_max`) were built to the already-folded-back §Configuration defaults; this ticket did not
  re-run the simulator. The simulator specs remain the source of truth for those numbers.
