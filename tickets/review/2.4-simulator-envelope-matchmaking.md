# Review: matchmaking operating-envelope boundaries

Two `BoundaryAxisSpec`s layered on the generic `findBoundary` harness (`simulator-envelope-core`,
already landed), folded into a `Metrics` sink and exported from `src/index.ts`. New **measurement**
over the existing matchmaking model — no changes to `SeekerWalk`, the decision math, or
`seekerPoolContentionWouldFlip`. Build + full test suite are green (256 passing, +12 new).

## What was built

`runMatchmakingBoundaries({ seed })` → `MatchmakingBoundaryReport { boundaries, metrics, …diagnostics }`.

**Boundary 1 — `bounded-harm` × `lyingFraction` (seeded seeker walks).**
- Axis `lyingFraction ∈ [0,1]`: fraction of the seeker's non-root path tiers whose primary mis-reports
  `topicTraffic`. `lyingTierCount = round(f · startTier)`; the topmost tiers lie.
- Lie model is the **per-query-flip** adversary (`simulator-strengthen-scenario-adversariality`): a
  lied-about tier flips on a per-query counter — even queries under-report (`arrivalsPerMin = 0`, force
  escalate), odd queries over-report (`arrivalsPerMin` huge, force a hang-out that then drains).
- `holds(f)` runs an honest `SeekerWalk` and a lied one over the same seeded world/model and returns
  `liedMatched ∧ extraHops ≤ lyingTierCount ∧ extraDrain ≤ patienceMs`. Harm is measured **relative to
  the honest baseline** (extra hops / extra drain), so the deadline doesn't trivially satisfy it.
- `designAssumption = 0` (all-honest). Located `f* ≈ 1.5 / startTier ≈ 0.1875`, `margin ≈ +0.19`.

**Boundary 2 — `hang-out-fairness` × `seekerProviderRatio` (pure function).**
- Axis `ρ`: seeker-pool demand per provider-arrival; drives the exact `Σ wantCount = ρ · arrivalsPerMin`.
- `holds(ρ) = !seekerPoolContentionWouldFlip(…)` — whether swapping the shipped capped approximation
  (`queriesPerMin × meanWantCount`, clamped at `contention_factor_cap = 4`) for the exact sum flips the
  hang-out decision. Isolates the **decision flip**, not "the raw factor exceeded 4".
- `designAssumption = cap − 1 = 3` (the ratio at which exact contention `1 + ρ` saturates the cap — the
  design's assumed worst case). Located `ρ* ≈ 2.5`, `margin ≈ −0.5` (**negative**).

## Use cases for testing / validation (the spec file is the floor, not the ceiling)

- **Finite edge + margin sign.** B1: positive margin (design inside envelope). B2: **negative** margin
  (design point sits *outside* the fair-decision envelope). Both `boundaryFound`, `monotoneViolated=false`.
- **Positive controls.** B1: `holds(0)=true`, `holds(1)=false`. B2: `holds(0)=true`, `holds(3.5)=false`.
- **De-tautologization (B1).** A *static* under-report still matches at `f=1` (`holds=true`); the
  per-query-flip at the same fraction fails (`liedMatched=false`). The flip is what gives the adversary
  teeth — see `staticUnderReportHoldsAtCeiling` and the `harmMechanismAtEdge='match-failure'` diagnostic.
- **Flip isolation + divide guard (B2).** `measureSeekerContention(2).wouldFlip=false` vs `(3.5)=true`;
  `{ arrivalsPerMin: 0 }` does not throw (the `max(arrivalsPerMin,1)` guard).
- **Determinism.** Two full runs are byte-identical (`deep.equal` on boundaries + diagnostics); a
  per-fraction B1 walk pair is identical.
- **Metrics folding.** Both boundaries land in the sink keyed by `(claim, axis)`.

## Known gaps / where to push (treat my tests as a floor)

1. **B1 is a stark, single-tier-deep margin — verify it's a *finding*, not a modelling artifact.** The
   edge is `≈ 1.5/startTier` because the per-query-flip's *second* path reporter over-reports, and under
   the default `patience_per_tier_fraction = 1.0` that one over-report commits the **whole** patience
   budget to a provider-less tier → the seeker never matches. So "bounded harm" breaks essentially at the
   *second* flipping reporter. This is deliberately the headline (it quantifies the need for
   `matchmaking-per-tier-patience-splitting`), but a reviewer should confirm the breaking mechanism is the
   genuine `1.0` fraction fragility and not an over-tuned tier model. Worth probing: sweep
   `patience_per_tier_fraction < 1` and confirm the edge moves out (harm becomes graded escalation rather
   than instant match-failure).

2. **B1 only exercises the *match-failure* and *over-report* harm dimension, not the `+1-hop-per-tier`
   under-report dimension.** The escalate-to-root honest baseline means a *static* under-report adds **0**
   extra hops (honest already escalates to root), so the `extraHops ≤ lyingTierCount` conjunct is satisfied
   with slack and never the breaking bound. This is an honest decomposition (the existing
   `AdversarialReportingScenario.measureUnderReport` already covers the `extraHops = #under-reported`
   bounded case via a thin+fresh model), but if the reviewer wants the boundary to *also* break via
   `excess-hops`, a thin+fresh hang-out-match baseline with `patience_per_tier_fraction < 1` would expose
   a patience-depletion cascade — at the cost of timing-sensitivity. I judged the robust match-failure
   break the better trade; flag if you disagree.

3. **B2 margin is negative *by construction*, and I diverged slightly from the ticket's literal framing.**
   The ticket phrases the edge as "the ratio at which real contention first exceeds the cap *and* flips
   the decision." Because `seekerPoolContentionWouldFlip` clamps **both** terms at the cap, the decision
   can only flip while the *exact* term is still rising through the divergence window (approx fixed at 2,
   exact rising toward 4) — i.e. necessarily **before** `1 + ρ` reaches the cap. So `ρ* (≈2.5) < cap−1
   (=3)` always, and the margin is structurally negative. I treat that negative margin as *the answer*
   (`contentionRefinementWarranted = margin < 0` ⇒ the misfire precedes the design's assumed worst case ⇒
   `matchmaking-contention-from-seeker-pool` is needed). Confirm you're comfortable with (a) the negative
   margin sign (unlike the other four boundaries) and (b) interpreting the flip *onset* as `ρ*` rather
   than the cap-crossing point. The chosen B2 constants (`arrivalsPerMin=60`, `queriesPerMin=20`,
   `currentMatches=4`, `patienceMs=24_000`, `wantCount=8`) place the LHS at `28 = 3.5·wantCount`, which
   fixes `ρ* = 2.5`; they are somewhat arbitrary — sanity-check the algebra in the `measureSeekerContention`
   doc comment and that the edge is robust to reasonable perturbation.

4. **Cross-refs assumed to land (designed-as-if).** `matchmaking-per-tier-patience-splitting` and
   `matchmaking-contention-from-seeker-pool` (the refinements these boundaries quantify the need for) are
   *not* implemented here — `refinement-signal.ts` is reused read-only. No code depends on them landing.
