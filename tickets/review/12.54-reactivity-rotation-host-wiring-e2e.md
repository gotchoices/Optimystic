# Review — reactivity rotation: origination observe-rotation + node composition + e2e

The capstone of the tail-rotation series landed. A `cohortTopic`-enabled node now **observes** a tail
rotation (the only signal available on a live node), starts the outgoing tail's drain, composes the
recover-redirect serve and the re-registration scheduler, and the full redirect → re-register → continue path
is proven end-to-end over the mock-tier mesh. Treat the implementation as a starting point and the tests as a
floor — the honest-gaps section below lists exactly where coverage stops.

## What was built

### A. Origination observes the tail-id change → starts the drain (`origination-manager.ts`)
- `ReactivityOriginationManager` tracks the **last-seen reactivity tail anchor per collection** (keyed by
  `event.collectionId`). On a commit whose resolved `ctx.tailId` (base64url) differs from the last-seen, it
  fires the new `markRotated(oldTopicId, { newTailId, effectiveAtRevision: event.rev }, now)` option.
  `oldTopicId = reactivityTopicId(b64urlToBytes(lastSeenTailB64))` — byte-identical to the topic a subscriber
  subscribed under. First commit records the baseline (no fire); a tail-less commit (`event.tailId ===
  undefined`) returns before observation (never records/clears the baseline).
- A per-collection `BlockFillTracker` is fed `onCommit()` each commit; `warmup`/`filling` are **logged only**
  (signal-only on a live node — the successor tail id is not knowable; see §"the load-bearing constraint").
- `markRotated` and `blockFill` are **optional** options — absent ⇒ origination is byte-for-byte unchanged
  (the rotation observation still tracks last-seen + feeds the tracker, but fires/ logs nothing material). The
  observe path is isolated in its own try/catch so it can never block the notification emit.

### B. Node composition (`libp2p-node-base.ts`, the `cohortEnabled` block)
- `origination.markRotated` bound → `forwarderHost.markRotated`.
- Recover serve handler's `rotationFor` bound → `forwarderHost.rotationRedirectFor`. A **resume** carries the
  stale topic (`req.topicId = reactivityTopicId(latestKnownTailId)`); a **backfill** has no topic, so the new
  `resolveCurrentServedTopic(forwarderHost, collectionId)` helper resolves the collection's current served
  topic first.
- A `RotationReRegistrationScheduler` is constructed with the default **unref'd `setTimeout`** timer and
  exposed as `node.reactivityRotation`. Its `reRegister(plan)` is a **logged no-op** until the deferred
  subscribe factory (`optimystic-network-reactive-watch-integration-test`) lands — exactly as 12.33 exposed
  `reactivitySubscribers`/`reactivityRecover` without a live manager constructor.
- Teardown: `reactivityRotation.stop()` added at the top of the reactivity stop wrapper (before `host.stop()`).
- **Jitter knob (per the corrected note):** the production manager uses the *single*-subscriber
  `planReRegistration` → `RejoinJitter.scheduleRejoin` (uniform over `T_rejoin_jitter`), which does **not**
  consult `capPromote`; the node deliberately does **not** add a `DEFAULT_CAP_PROMOTE_FAST` jitter comment. The
  load-bearing knob is the window, not `capPromote`.

### C. Mock-tier e2e (`reactivity-mesh-harness.ts` + `mesh-tail-rotation.spec.ts`)
- Harness models `markRotated` with a real `TailDrainGate` (per the ticket's "or model it"): `rotateTail(coll,
  { autoReattach: false })` arms the gate; the subscriber's `resumeTransport` raises a `RotationRedirectError`
  for a stale resume while draining. A per-subscriber `RotationReRegistrationScheduler` is driven over the
  harness **virtual clock** (`advanceTime` fires due timers); on fire it re-attaches the subscriber to the new
  tail. `rotateTail` is now **async** (it re-seeds the new cohort's willingness via `setupTopic` so a
  post-rotation subscribe can register) — all callers updated to `await`.
- New tests: (1) **redirect-driven re-registration with no gap** — `markRotated` → `kind:"rotated"` →
  `onRotation` (preAnnounced: false) → scheduler arms → `advanceTime(T_rejoin_jitter)` fires → re-attach →
  continuous 1..12; (2) **cross-rotation resume from the inherited checkpoint** — a fresh subscriber under the
  new tail whose `fromRevision` is below the new ring but within the inherited checkpoint gets
  `checkpoint_window` (not `out_of_window`) and ends current. `resumeTransport` now also passes
  `inheritedCheckpoint` + `rotationRevision`.

## The load-bearing constraint (why observe-on-tail-id-change, not pre-announce)
Block ids are random (`randomBytes(32)`; deterministic derivation is the blocked `6.5-block-id-derivation`),
so at the filling commit the host **cannot** know the successor tail id → the pre-announce `rotationHint{
newTailId }` cannot be built on a live node. The authoritative live signal is `event.tailId` **changing** (a
hard rotation; subscribers detect via the delivered `tailId` differing, `preAnnounced: false`). Pre-announce +
warm-up stay exercised in the mock harness + the simulator (both synthesize the successor id).

## Use cases to validate (reviewer focus)
- **Encoding pin (most load-bearing):** `markRotated`'s `oldTopicId` MUST equal `reactivityTopicId(
  reactivityTailBytes(tail))` — the topic a subscriber subscribed under — NOT `blockIdToBytes` (double-hash). A
  mismatch is a silent dead feature (green tests). Covered by `managers.spec.ts` → "fires markRotated with the
  correctly-encoded old topicId on a tail-id change". Re-verify the encoding is right and the test would catch
  a regression to `blockIdToBytes`.
- **No-gap continuity across the redirect:** delivered revisions stay monotone + gap-free across the boundary
  even though `tailId` differs across the seam. `mesh-tail-rotation.spec.ts` → "a recover redirect drives
  re-registration end-to-end with no gap".
- **Cross-rotation resume from the inherited checkpoint:** `checkpoint_window`, not `out_of_window`, while the
  inherited window abuts the new ring's low edge. `mesh-tail-rotation.spec.ts` → "a cross-rotation resume is
  served from the inherited checkpoint".
- **Scheduler teardown:** no pending timer pins the process after `node.stop()`. `node-wiring.spec.ts` → the
  Core-node test schedules a notice, asserts pendingCount 1, then after stop asserts the scheduler refuses to
  schedule (pendingCount stays 0).
- **markRotated absent ⇒ origination unchanged**, same-tail commits don't fire, tail-less commit doesn't
  disturb the baseline. `managers.spec.ts` observe-rotation block.
- **Backfill-redirect path** (the `rotationFor` backfill branch resolving the collection's current served
  topic) is wired in node-base but NOT directly asserted at the node level — see gaps.

## Known gaps / honest notes for the reviewer
1. **The end-to-end redirect over a real socket is NOT driven by a live node-internal manager.** The node
   exposes the scheduler but its `reRegister` is a logged no-op; no node-internal manager calls `schedule()`
   until the deferred Quereus `Database.watch` subscribe factory lands
   (`optimystic-network-reactive-watch-integration-test`). The full redirect → onRotation → scheduler →
   reRegister → re-attach path is proven only at the **mock tier** + unit level. This is by design (mirrors
   12.33/12.42) but means the production socket path for rotation is not yet exercised end-to-end.
2. **The `markRotated` encoding test lives in `managers.spec.ts`, not `node-wiring.spec.ts`.** The ticket
   listed it under node-wiring, but driving a real tail-id change needs two cohort-gated tail commits on a real
   node (heavy/flaky). The unit test pins the actual goal — the encoding equality — deterministically.
   `node-wiring.spec.ts` covers the scheduler exposed + torn down. Reviewer: decide if a real-node markRotated
   assertion is worth the cost or if the unit pin suffices.
3. **The mesh harness models `markRotated` with a `TailDrainGate`, not the real `ReactivityForwarderHost`.**
   The harness has no forwarder host (it drives a raw `PushState`+forwarder). "Or model it" per the ticket. The
   gate's redirect/draining semantics are the real db-core class, but the wiring is modeled.
4. **The harness `reRegister` keeps the same manager (re-attach) rather than constructing a fresh manager under
   `plan.newTopicId`.** The production factory builds a fresh manager; the harness re-attaches the existing one
   (its `lastRevision` is already contiguous and `rotationHandledFor` dedupes the new-tail re-detection), which
   is the equivalent move with no gap but is not byte-identical to production.
5. **`rotateTail` became async** (re-seeds the new cohort's willingness). A reviewer should confirm this models
   reality acceptably and that no caller relies on synchronous rotation.
6. **Re-register swap ordering** (register the new-topic handler in `ReactivitySubscriberRegistry` BEFORE
   unregistering the old, so a mid-swap notification is never dropped) is **spelled out at the exposure site**
   for the watch-bridge ticket but NOT yet implemented (no factory). The mesh e2e plays the factory role via
   re-attach, which has no registry swap, so the ordering is documented-but-unverified-in-code.
7. **Pre-existing flaky test:** `mesh-slow-subscriber.spec.ts` → "an Edge subscriber receives notifications…"
   fails ~1/3 of runs with `CohortBackoffError: no willing primary` when the random member layout puts the Edge
   node at the cohort lead (Edge declines T3 willingness). Documented in `tickets/.pre-existing-error.md`;
   outside this diff (the test does not rotate). The triage pass will pick it up.
