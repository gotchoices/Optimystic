# What shipped

Damping for `RingSelector`'s ring-shift decision (spec: `docs/arachnode-ring-handoff.md` § Part 1).
**Self-contained to `RingSelector` + its config + construction site — the two-phase handoff protocol
(Part 2) is the sibling ticket `arachnode-ring-handoff-protocol` and is deliberately untouched.**

The three mechanisms, all in `shouldTransition()`:

1. **Smoothing (EWMA).** Each tick folds the raw demand inputs — available capacity and estimated
   network data (`Σ peerCount×avgCapacity` from ring stats) — into two per-instance EWMAs
   (`smoothedAvailable`, `smoothedTotalData`, α default 0.2). The decision works off the smoothed
   continuous depth `d = clamp(-log2(smoothedAvailable/smoothedTotalData), 0, 16)`. Seeded from the
   first real sample, **not from 0** (a 0 seed would peg `d` at 16 for many ticks).
2. **Dead-band / hysteresis.** Anchored on the node's **currently-advertised** ring `R`
   (`fretAdapter.getMyArachnodeInfo().ringDepth`), not a freshly recomputed one. Move **out** only
   when `d ≥ R+1−h` **and** `usedPercent > moveOut`; move **in** only when `d ≤ R−1+h` **and**
   `usedPercent < moveIn` **and** `R>0`. Default `h=0.5` ⇒ a full ring separates the two triggers, so
   both can never fire.
3. **Minimum dwell + single-step.** No new move within `minDwellMs` (default 10 min) of the last; no
   move while `status==='moving'`; the target is always `R±1` (single step) even when `d` implies a
   larger jump.

Supporting changes:
- `RingSelectorConfig` gained `smoothingAlpha`, `deadband`, `minDwellMs`, and an injectable `now`
  clock (for testable dwell). Defaults documented as constants in the file.
- `determineRing()` is now a **pure** instantaneous read (refactored to share an `estimateNetworkData`
  helper; behavior byte-for-byte unchanged) — it does not touch EWMA state, which is what keeps the
  pre-existing `determineRing` tests valid. It remains the deterministic first-boot ring.
- `createArachnodeInfo(peerId, ringDepthOverride?)` — the transition site now passes
  `transition.newRingDepth` so the **advertised** ring can never disagree with the ring the decision
  just chose (the "don't report a smoothed ring while advertising a different one" edge case).
- `libp2p-node-base.ts` wires the new config (keeping `moveOut:0.85`/`moveIn:0.40`) and passes the
  explicit target at the flip site. The return shape `{shouldMove, direction, newRingDepth}` is
  unchanged, so the (future) handoff consumer is unaffected.

# How to validate

Build + tests (from `packages/db-p2p`):
```
yarn build      # tsc, clean
node --import ./register.mjs node_modules/mocha/bin/mocha.js "test/ring-selector.spec.ts" --reporter spec
```
Result on implement: ring-selector spec **29 passing**; full db-p2p suite **1221 passing, 36 pending
(gated integration), 0 failing**.

The 10 new `shouldTransition (damped)` tests (use a `FakeClock` + a `setSignal(usedPercent, d)` helper
that pins the *smoothed* depth on the first sample; **no wall-clock sleeps**):
- no move inside the dead-band at normal usage
- move-out past the outer boundary under pressure; **dead-band blocks** move-out when usage is high
  but `d` hasn't crossed
- move-in past the inner boundary at low usage; **never move-in from ring 0**
- `status==='moving'` blocks a new move
- **boundary hover** — 12-sample jitter of `d≈R±0.3` around the node's own ring integer produces zero
  moves
- **EWMA seeding** — a legitimate move-in that a 0-seed would have wrongly blocked
- **dwell** — first move fires; rapid re-flip within `minDwellMs` suppressed; sustained pressure moves
  again once dwell elapses
- **single-step** — from ring 0 with `d≈2.5`, one call targets ring 1 (not 2); ring 2 only on a later
  tick after dwell + advertising the intermediate ring

# Honest gaps / what the reviewer should probe (tests are a floor)

- **Dwell keys off *trigger*, not *completion*.** The advertise→confirm→release handoff (Part 2) isn't
  landed, so `lastMoveAt` is stamped when a move is *triggered*. If a triggered move later aborts, the
  node is still rate-limited for `minDwellMs` (conservative, but not the spec's "after a *completed*
  shift"). Tagged `NOTE:` at the stamp site in `shouldTransition`. **Tripwire, not a ticket:** the fix
  is to re-stamp on completion, which only becomes real when the sibling handoff ticket lands.
- **Smoothing choice.** I smooth the two inputs separately and divide (per the ticket's literal "both
  inputs smoothed via the same EWMA"), which is subtly different from smoothing the coverage ratio
  directly. Both are valid EWMAs; confirm the reviewer agrees the input-wise form is what's wanted.
- **`advertisedRing` fallback.** When `getMyArachnodeInfo()` is undefined (never in production — bootstrap
  advertises first), the anchor falls back to `round(d)`. Documented; exercised only implicitly.
- **No integration-level test of the live loop.** The 60 s `setInterval` glue in `libp2p-node-base.ts`
  that feeds real FRET ring-stats into the EWMA over time is unit-covered only at the `RingSelector`
  boundary. If the reviewer wants end-to-end confidence in the damped loop under real churn, that's a
  candidate follow-up (deferred — the interval body is thin).
- **`d`-to-`usedPercent` coupling in tests.** `setSignal` drives available capacity and network-data
  estimate to hit a target smoothed `d`, and sets `usedPercent` independently via the monitor. Worth a
  sanity check that the coupling in the real path (both derive from the same capacity snapshot) can't
  produce a regime the unit tests miss.
