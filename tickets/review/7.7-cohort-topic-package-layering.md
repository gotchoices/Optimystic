# Review: cohort-topic package layering (pure core vs. transport binding)

This ticket established the single-source-of-truth layering rule for the cohort-topic substrate:
pure logic in `db-core` behind ports, FRET/libp2p implementations in `db-p2p`. Every downstream
substrate ticket prereqs this and must follow the rule.

## What landed

**db-core (pure):**
- `cohort-topic/ports.ts` — the five interfaces verbatim from the implement ticket: `ITopicRouter`,
  `ICohortGossipTransport`, `IMembershipSource`, `ISizeEstimator`, `IRingHash`, plus the
  db-core-owned `RingCoord` (= `Uint8Array`) and `PeerRef` types. Interfaces only — no FRET/libp2p
  import.
- `cohort-topic/ring-hash.ts` — `RingHash` / `createRingHash` backed by `@noble/hashes/sha2`
  (db-core's own SHA-256, already a dependency). `RING_BITS` default 256 → the full digest, so
  `H(bytes)` is byte-identical to a raw SHA-256, which is what FRET produces for the same input.
  Sub-byte ring widths truncate to `ceil(ringBits/8)` bytes and zero the trailing partial byte's
  low bits (same big-endian prefix semantics the simulator's `prefixBits` uses).
- Exported from `db-core/src/index.ts`.

**db-p2p (transport, stubs):**
- `cohort-topic/{topic-router,cohort-gossip-transport,membership-source,size-estimator}.ts` —
  one class per port (`FretTopicRouter`, `FretCohortGossipTransport`, `FretMembershipSource`,
  `FretSizeEstimator`), each implementing the db-core interface and **throwing
  `notWiredToFret(...)`**. These are intentional compiling placeholders; the real FRET binding is
  the downstream `cohort-topic-core-module-fret-integration` ticket. They exist so db-core
  substrate tickets can build and type-check against the ports now.
- Exported from `db-p2p/src/index.ts`.

**Guardrail:** `db-core/test/no-fret-import.spec.ts` walks `db-core/src/**` and fails if any
`import`/`export … from`/dynamic `import()` resolves to `p2p-fret`, `libp2p`, or `@libp2p/*`.

**Docs:** architecture.md §Package Layering (substrate spans both layers; db-core stays
network-free; names the guard), internals.md (new §Cohort-Topic Port Boundary), cohort-topic.md
(§FRET integration pointer to where ports vs. impls live).

- `db-core` full suite: **308 passing** (no regressions from the index export).
- New specs pass: `ring-hash.spec.ts` (6 cases incl. FRET-compat full-digest, determinism,

## Use cases / what to check

- **Layering rule holds:** confirm `ports.ts` has zero transport imports and that the guard would
  actually catch a violation (try adding `import 'libp2p'` to a db-core src file → spec should
  fail). Verify the guard regex isn't trivially bypassable for the forms this repo uses
  (it matches `import`/`export … from` and dynamic `import()`).
- **Stub honesty:** every db-p2p adapter method throws with a clear pointer to the
  fleshing-out ticket — no silent no-ops that could mask missing wiring downstream.
- **Port shape vs. siblings:** the implement ticket lists corrections imposed on
  `cohort-topic-tier-addressing-dmax`, `cohort-topic-gossip-membership-certs`, and
  `cohort-topic-walk-promotion-coldstart` (H via `IRingHash`, gossip-merge/cert-verify in db-core
  over the ports). Sanity-check the port signatures are sufficient for those consumers as
  described.

## Known gaps / honest flags (reviewer: treat as a floor)

- **FRET byte-compatibility is asserted by design, not cross-checked in a test.** `RingHash` at
  256 bits equals a raw SHA-256; FRET's `hashKey` (used by `packages/substrate-simulator`) is also
  SHA-256, so coords line up — but there is no test asserting `RingHash.H(x)` equals FRET
  `hashKey(x)` byte-for-byte, because db-core cannot import FRET. A natural home for that assertion
  is a **db-p2p** test once the FRET adapter lands (it's the package allowed to import both). Worth
  a follow-up note in `cohort-topic-core-module-fret-integration`.
- **Stubs take no constructor dependencies yet.** They throw rather than holding a libp2p/FRET
  handle. The dependency surface (registerMaybeAct/sendMaybeAct, cohort-gossip channel, membership
  machinery, size estimator) is described in doc comments but will be shaped by the integration
  ticket; reviewer should not expect a final constructor signature here.
- **Guard scope is db-core only**, as specified. db-p2p is free to import FRET/libp2p (it does).
- **Sub-byte ring widths are implemented but unused** (`RING_BITS` is 256 everywhere). The masking
  path has unit coverage but no production caller.
