# Review: Invalidation log entry + single-collection reversal

This implements the **core durable reversal primitive** for a single invalidated transaction in one
collection: the `InvalidationEntry` log type, the compensating-state computation, the invalidation
certificate, and the consensus-apply path. The broader cascade (read-dependents, multi-collection)
and client notification are the two follow-on tickets that `prereq` on this.

The implementation is a **vertical slice**: the db-core primitive and the db-p2p compute/verify/apply
logic are fully built and unit-tested; the composition-root wiring (live multi-node) is deliberately
left as injection points and documented below — **treat the gaps section as the priority review
surface, not an afterthought.**

## What landed

### db-core (pure log layer — no libp2p)
- `LogEntry<TAction>` gains an `invalidation?` arm. New types in `log/struct.ts`:
  `InvalidationEntry`, `DisputeResolutionProof`, `ArbitrationVoteProof`, `RevertedBlock`.
- `Log.addInvalidation(invalidatedActionId, invalidatedRev, resolution, reverted, rev, cascadeRoot?, timestamp?)`
  appends the compensating entry (append-only, prior-hash chain preserved).
- `Log.findInvalidation(invalidatedActionId)` — durable `committed-invalidated` lookup (scans
  newest→oldest, returns the most recent matching entry).
- `getFrom` made robust: invalidation entries take a rev slot but are skipped in the pending/action set.
- `network/struct.ts`: `InvalidateRequest` (the consensus operation payload). `repo-protocol.ts`:
  added `{ invalidate: InvalidateRequest }` as a 4th `RepoMessage` operation arm.

### db-p2p (`src/dispute/invalidation.ts`, new)
- `buildDisputeResolutionProof(resolution, messageHash)` — projects `DisputeResolution` → the db-core
  `DisputeResolutionProof` (outcome + signed votes only).
- `verifyInvalidationCertificate(proof)` — recomputes from cryptographically valid votes alone:
  `challenger-wins` + 2/3 super-majority of decisive `agree-with-challenger` votes, each Ed25519
  signature verified against its arbitrator peer-id's embedded key. Forged/tampered votes are dropped.
- `computeRevertedBlock(blockStorage, invalidatedRev)` — recomputes as-if-`T_inv`-absent content from
  stored revisions (base at `invalidatedRev-1` + surviving later actions replayed). Returns
  `{kind:'restore',...}` or `{kind:'delete',...}`.
- `applyInvalidation(ctx, params)` — deterministic apply: **dedup** on `(invalidatedActionId, disputeId)`
  via `Log.findInvalidation` → **certificate** verify → write per-block compensating revisions via
  `IBlockStorage.saveReplica` → **append** the `InvalidationEntry`.

### db-p2p wiring
- `cluster-repo.ts`: injectable `onInvalidate?: InvalidationApplySink`; new `applyConsensusInvalidation`
  arm in `applyConsensusOperation` (dedup + cert-verify-before-apply + delegate to sink + tolerate
  faults, never reset the stream); `getAffectedBlockIds` now includes `invalidate.blockIds` for
  conflict serialization; dedup markers pruned on the executed-tx TTL.
- `dispute-service.ts`: injectable `onInvalidation?`; `maybeInvalidate` fires on `challenger-wins`
  (initiator path only — it holds the original record), builds the proof + `InvalidateRequest` via
  `extractInvalidationTarget`, fires once per messageHash.

## How to validate (tests are a FLOOR — extend them)

Build: `yarn build` in `packages/db-core` then `packages/db-p2p`. Both clean.
Dependent `packages/quereus-plugin-optimystic` builds clean (RepoMessage change is additive).

Test commands (mocha, from each package):
- db-core: `node --import ./register.mjs node_modules/mocha/bin/mocha.js "test/log-invalidation.spec.ts" --colors`
  (5 new; full db-core suite = **888 passing**)
- db-p2p: same runner on `test/invalidation.spec.ts` (14), `test/cluster-invalidation.spec.ts` (5),
  `test/dispute.spec.ts` (32, incl. 2 new origination tests).

Covered use cases:
- Append an `InvalidationEntry` for a single-block `T_inv`; pre-`T_inv` content restored as a new rev;
  prior-hash chain intact; status durable via `findInvalidation`.
- Certificate: `majority-wins`/`inconclusive`/forged-signature/tampered-payload/sub-threshold rejected;
  valid `challenger-wins` accepted.
- Idempotency: same resolution twice ⇒ one entry / one revision (log + cluster-repo dedup paths).
- Determinism: two independent members converge on the same `restoredContentHash` and rev.
- `T_inv` superseded: surviving later action replayed on the rolled-back base.
- Cluster apply arm: applies via injected sink; rejects forged cert; dedups across distinct message
  hashes; tolerates missing/throwing sink (rolls back marker for retry).
- Dispute origination: `challenger-wins` originates a proof-bearing `InvalidateRequest`; `majority-wins`
  does not.

## KNOWN GAPS — primary review focus

1. **No composition-root wiring (the big one).** `onInvalidate` (cluster-repo) and `onInvalidation`
   (dispute-service) are injection points; `libp2p-node-base.ts` does **not** wire them yet. In a live
   node today the in-memory dispute status still flips, but no durable reversal is originated or applied
   end-to-end. Still needed: (a) a concrete `InvalidationApplySink` that opens the collection `Log` over
   the node's `StorageRepo` and calls `applyInvalidation`; (b) a concrete `onInvalidation` originator
   that builds the `invalidate` `RepoMessage` and drives it through `CoordinatorRepo.executeClusterTransaction`.
   This was scoped out to avoid landing a half-wired protocol path; the reusable primitives are all
   present and unit-tested.
2. **`InvalidateRequest.collectionId` is derived approximately** (`commit.headerId ?? coordinatingBlockIds[0]`).
   Confirm against how the eventual sink opens the `Log` (collection id == log header id) before wiring.
3. **No multi-node / crash-recovery libp2p integration test.** Determinism is proven at the unit level
   (two independent storages converge). Full cross-member convergence + crash-mid-apply need the live
   harness and gap #1.
4. **Delete-restore deferred.** A `T_inv` that *created* a block (restore == deletion) is recorded with
   the `DEFERRED_DELETE_RESTORE` sentinel and logged, **not** physically removed (`IBlockStorage` exposes
   no delete-transform write). Cascade ticket.
5. **Blind later-action replay** (no read-dependent re-evaluation) — by design for the no-cascade core.
6. **`getActionContext` still lists the invalidated action in `committed`.** The reversal is surfaced via
   `findInvalidation`, not by removing the action from the committed set. Acceptable for this ticket
   (the notification ticket consults `findInvalidation`); flag if the reviewer disagrees with that split.
7. **Concurrent-commit-vs-invalidation interleaving** (both land, monotonic revs) relies on storage-level
   serialization + cluster conflict detection; not yet exercised end-to-end.

## Suggested adversarial checks
- `saveReplica`'s monotonic guard vs. the chosen `rev = max(fromRev)+1`: confirm a real consensus-assigned
  rev (passed in `params.rev`) can never collide below a block's tip.
- `stableStringify` hash determinism across engines/object key orders (cross-member `restoredContentHash`).
- The `RepoMessage` union widening: re-scan coordinator/validator paths for any operation handling that
  should explicitly account for `invalidate` (e.g. should an `invalidate` op be pend-validated, or skipped
  as it is today?).
- Certificate threshold edge: `ceil(totalDecisive * 2/3)` with `totalDecisive` of 1–2 — confirm it matches
  `DisputeService.resolveDispute` exactly.
