# Review: read-dependency cascade detection + re-evaluation

Builds on `7.5-invalidation-log-entry-and-reversal` (complete) — the durable single-collection reversal
primitive. This ticket adds the **cascade**: from one reversed `T_inv`, find its transitive
read-dependents across collections and re-evaluate each, invalidating only those that no longer hold.

Treat this as a starting point. The engine and its 12 tests pass and the build is green, but several
deliberate scope decisions and inherited limitations are flagged below — the reviewer should weigh them,
not assume they're closed.

## What was implemented

**Read-set persistence (db-core).** `ActionEntry` gained `reads?: ReadDependency[]` (`log/struct.ts`).
`Log.addActions` threads a new optional `reads` param (inserted before the trailing `timestamp` — no
existing caller passed `timestamp` positionally). `coordinator.ts` passes `transaction.reads` on commit.
The full transaction read set rides inside the log-block transform the cluster already replicates, so **no
cluster commit/checkpoint path change was needed** — the reads flow through PEND/COMMIT automatically.
Back-compat: an entry written without reads has `reads === undefined` (a *legacy / unknown dependency*),
distinct from `reads === []` (read nothing).

**Cascade engine (db-p2p `dispute/cascade.ts`).** `cascadeInvalidate(input)`:
- seeds an invalidated `(blockId, rev)` frontier from the root reversal's `reverted` blocks;
- each round walks every in-scope collection log forward, collecting unprocessed actions whose read set
  intersects the frontier (legacy reads-less entries are always candidates);
- processes candidates in `(rev, collectionId, actionId)` order, deduped by `actionId`; re-evaluates each
  against the reverted state — **retain** leaves it (revisited next round if a later ancestor reverts),
  **invalidate** appends a child `InvalidationEntry` (`cascadeRoot = T_inv`) via the 7.5 `applyInvalidation`
  primitive and feeds its reverted blocks back into the frontier;
- runs to fixpoint or a hard horizon; a same-collection back-edge throws (DAG guard).

**Re-evaluation.** Pluggable `Reevaluate`. The default `contentEqualityReevaluator` is engine-free and
deterministic from stored revisions: compares the content a dependent *observed* (`blockId@rev`) against the
`restoredContentHash` the invalidation recorded (shared `hashBlockContent` helper added to `invalidation.ts`
so both sides hash identically).

**Config + plumbing.** `ClusterConsensusConfig.maxCascadeDepth` (default 32) / `maxCascadeTransactions`
(default 1000). `ApplyInvalidationParams.cascadeRoot` threaded into `addInvalidation`. Cascade API exported
from `dispute/index.ts`. Docs updated: `right-is-right.md` (new "Read-Dependency Cascade" subsection + file
map) and `correctness.md` (new "Theorem 8b: Invalidation-Cascade Termination & Convergence").

## Known gaps / decisions the reviewer should scrutinize (tests are a floor, not a ceiling)

1. **Default re-evaluator is block-granular conservative.** It never wrongly retains, but because reads are
   block-granular it *over-invalidates* a dependent that read an unchanged **field** of a changed block. Safe
   (over-invalidation just resubmits), but it means the ticket's literal "mixed chain — `T2` reads a block
   `T_inv` wrote but a field `T_inv` did not change → retain" is achieved two ways here: (a) the default
   retains when `T_inv` didn't change the block's *content* (the `shared`-value test), and (b) the
   *injected* re-evaluator retains on an unread field (the engine-style test). The default cannot retain the
   literal unread-field case. **Decide** whether to ship an engine-backed re-evaluator before live use.
2. **`TransactionValidator` is NOT literally reused.** Its read check is revision-equality vs the current
   tip — incompatible with the reverted-state model (the compensating revision is a *new, higher* rev), so it
   would always report stale after a revert. The cascade uses content comparison / an injected re-evaluator
   instead. This diverges from the ticket's "reusing `TransactionValidator`" wording — intentional; confirm
   it's acceptable.
3. **Children applied deterministically, not via per-child consensus.** Every honest member replaying the
   same seed + logs with a deterministic re-evaluator converges on the same children (matches 7.5's
   replica-apply model), so no per-child consensus round is wired. A **non-deterministic / engine-based**
   re-evaluator would break that and require each child driven through consensus. Documented in
   `cascade.ts` / `right-is-right.md`.
4. **Not wired into a live composition root.** Like 7.5, origination/apply is still not end-to-end (no
   `DisputeService`/`ClusterMember` wiring). The engine is a validated library; `cascadeInvalidate` is
   unreferenced by production code so far. The caller must compute the root seed (per-collection
   `(blockId, rev, restoredContentHash)`) and supply the collection `envs` universe.
5. **Multi-collection atomic root is documented, not implemented here.** The root reversal's cross-collection
   atomicity is the caller's job via existing GATHER/PEND/COMMIT 2PC; the per-collection root rev seeding is
   left to the caller (each collection has its own rev sequence). Only the *cascade* (children) is built here.
   No "multi-collection atomic root under induced partial failure" test exists — it belongs at the
   composition-root layer, not the engine.
6. **Cross-collection dependent discovery needs a read index.** The engine consumes a provided `envs` list
   rather than discovering which collections read a block. Cross-collection diamond ancestor-ordering is
   best-effort (retained candidates re-examined each round until fixpoint) vs exact rev-order within a
   collection.
7. **Inherited `computeRevertedBlock` limitation (from 7.5).** A block *created* by an action at `rev > 1`
   makes `computeRevertedBlock(storage, rev)` throw (it only recognizes creation at `invalidatedRev <= 1`,
   else it reads `rev-1` and the materialize fails) — surfaced while writing the idempotent test (worked
   around by pre-creating the block at genesis). Delete-restore physical removal is still the `DEFERRED_DELETE_RESTORE`
   sentinel. Worth a follow-up fix ticket so cascades over freshly-created blocks don't throw.
8. **Resumability = log idempotency, not a separate progress cursor.** Restartability relies on
   `applyInvalidation`'s `(actionId, disputeId)` dedup + `findInvalidation` (durable). There is no separate
   persisted "where it stopped" checkpoint beyond the log itself plus the returned `escalation` report.
9. **`maxCascadeDepth` = rounds, not graph depth.** A round is one rescan generation; a linear chain
   processed in rev order collapses into ~one progressing round, but the horizon counts rounds, not the
   longest dependency chain. Documented; confirm the semantics are acceptable as a horizon.
