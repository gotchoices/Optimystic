# What shipped

The **advertise → confirm-replication → release** handoff from `docs/arachnode-ring-handoff.md`
§ Part 2, plus the concurrent-move / crash / partition rules in § Part 3. A ring shift (or a
lost-block rebalance) can no longer drop a key below its replication floor `N`, because a node keeps
serving a range until it has confirmed ≥ N *other* holders have a current replica, then releases.

Two release paths that previously flipped responsibility before confirming are now gated:

1. **Ring shift** (`RingShiftCoordinator`, new). The unilateral `setArachnodeInfo` flip in
   `libp2p-node-base.ts` is replaced by a three-phase state machine driven off the (damped)
   `shouldTransition()` trigger:
   - **Phase A (advertise):** set `status='moving'`, publish the *target* ring/partition, and record
     the old range in the new `ArachnodeInfo.moveFrom` field. Keep serving everything.
   - **Phase B (confirm):** enumerate the shed range (blocks the old partition covers but the new one
     does not) and confirm every one is replicated to ≥ N qualifying post-move holders.
   - **Phase C (release):** set `status='active'` at the new ring (clear `moveFrom`), stop
     serving/spreading the shed range, and mark it GC-eligible.
   Any Phase-B failure rolls back to `active` at the old ring. **No shed block is released unless
   every shed block confirmed.** Move-**in** sheds nothing → Phase A only (advertise; the gained half
   is pulled by the restoration/rebalance path).
2. **Rebalance release** (`libp2p-node-base.ts` `onRebalance` handler). A `lost` block is no longer
   `untrackBlock`ed synchronously. `BlockTransferCoordinator.handleRebalanceEvent` now returns
   `{ pulled, released, retained }`; only `released` (confirmed to ≥ floor new owners) is untracked +
   marked GC-eligible. `retained` blocks stay tracked/served and retry next rebalance.

Supporting pieces:
- `BlockTransferCoordinator.confirmReplicated(blockIds, owners, floor)` — the confirm primitive:
  pushes to candidate owners, counts holders reporting the block *not* `missing`, requires ≥ floor.
- `arachnode-partition.ts` — the single source of truth for responsibility derivation:
  `extractPrefix`, `partitionCovers`, `isServingHolder` (fail-toward-old-holder: a `moving` peer
  still covers its **old** range via `moveFrom`), `qualifiesForFloor` (a peer counts toward a
  confirmer's floor only if its **target** partition covers the key — this excludes a concurrent
  same-range mover). `RingSelector` and `RestorationCoordinator` were refactored onto it (they each
  had a private copy of `extractPrefix`).
- `RebalanceEvent.floor` (= `getCohortSize()`, now public) carries N to the reaction.
- `RingShiftCoordinator.reconcileOnStart()` — refreshes a stale `moving` advertisement (crash between
  advertise and release) back to `active` at the old ring.
- `RingSelector.recordShiftSettled()` — re-stamps the dwell timer when a shift *settles* (spec §1.3,
  "after a completed shift"), discharging the tripwire the damping ticket left for this one.
- `libp2p-node-base` exposes `node.gcEligibleBlocks` and `node.ringShiftCoordinator`.

# Test coverage (the reviewer's floor — extend it)

`test/ring-shift-coordinator.spec.ts` drives the state machine with a stub confirmer and **real**
`hashKey`/`extractPrefix` (no faked coords). Covered:
- **Replication-floor invariant** — counts `active`-serving holders for a shed key at each phase
  transition (before / during Phase B via a confirmer hook / after Phase C) and asserts ≥ N; the
  mover counts until release, ≥ N others after. Also: only the shed half is released.
- **Concurrent adjacent moves** — a same-range mover (`status='moving'`, target partition sheds the
  overlap) is excluded from the floor; with too few qualifying holders the shift rolls back and keeps
  the range. Asserts the confirmer saw only the stable holder.
- **Partition mid-handoff** — Phase B aborts → rollback, range retained, confirmer never invoked.
- **Confirm floor unmet** — rollback, nothing released.
- **Move-in** — advertise only; never confirms/releases.
- **Crash / reconcileOnStart** — resumes the old range and refreshes to `active`; a `moving` peer is
  still a holder for its old range but is excluded from another mover's floor.
- **Membership superset** — during A–B the shed key is covered by old ∪ new holders (bounded delta).

`test/block-transfer.spec.ts` — `confirmReplicated`: confirms to floor, leaves unconfirmed below
floor / during partition / with no owners. `test/ring-selector.spec.ts` — `recordShiftSettled`
re-anchors dwell.

# Known gaps / flags (treat my tests as a floor, not a finish line)

- **`reconcileOnStart` is effectively a no-op in production today.** Arachnode `ArachnodeInfo` lives
  in FRET's in-memory metadata (`setMetadata`), which is not persisted across restarts, and bootstrap
  writes a fresh `active` info before `ringShift` is even built. The method + call are correct and
  unit-tested, and become load-bearing the moment arachnode metadata is persisted. It is wired now so
  that persistence is a one-line change, not a re-architecture. **Not exercised on a real restart.**
- **Membership-tolerance (`admitMembership`) is NOT exercised end-to-end.** The admission gate keys on
  `keyNetwork.findCluster` (a layer separate from arachnode partitions), so a faithful test needs a
  `findCluster` that reflects the ring shift — a larger integration than this ticket built. I proved
  the *mechanism* the spec relies on (the superset-cohort / bounded single-step delta) at the helper
  level only. A reviewer wanting the § Part 3 membership claim fully nailed should drive a real
  cluster-repo `admitMembership` across a simulated shift.
- **Behavior change: ring shifts now require the rebalance reaction to be wired.** A node with
  `rebalance:{enabled:false}` (or arachnode without the reaction) no longer shifts rings — previously
  it did the unilateral unsafe flip. A move-out is unsafe without the confirm/release path, so this is
  deliberate. Confirm it is acceptable for those configs.
- **No real-libp2p integration test drives an actual shift** over sockets (the production trigger is a
  60 s interval). The unit tests call the coordinator directly. A real-node test that injects a
  `shouldTransition` result and observes the handoff would be the natural next reinforcement.
- **`confirmReplicated` pushes rev-1 replicas** (no source `blockMeta`), matching the existing
  `pushBlocks` path; `saveReplicatedBlock` is idempotent and never downgrades, so a holder with a
  newer rev still confirms. Correct, but noted.

# Tripwires (recorded here as the index; homes noted — do NOT file as tickets)

- **`gcEligibleBlocks` grows unbounded until a sweep consumes it.** No sweep exists yet; this set is
  the coordinated GC-eligibility handoff `st-storage-sweep-archival-and-capacity-estimate` will read.
  `NOTE:`-tagged at the declaration in `libp2p-node-base.ts` (~903). Bound it when the sweep lands.
- **`RestorationCoordinator.filterByPartition` uses a peer's *advertised (target)* partition**, so a
  `moving` peer is not matched for the shed half it still physically holds. Restoration also queries
  `assembleCohort` + inner rings, so this is a minor coverage gap, not a correctness bug. Lives at the
  `partitionCovers` call site in `restoration-coordinator.ts`; note here is the index entry.
- **The gated rebalance reaction is fire-and-forget with per-block retry+backoff**, so a failing
  confirm leaves lingering timers after the triggering call returns — a pre-existing pattern
  (`executePush` did the same); harmless (the promise is `.catch`-guarded).
- **Qualifying-holder discovery excludes candidates with no advertised `ArachnodeInfo`** (their
  coverage can't be verified). In a mixed network this could make the floor hard to reach → the shift
  rolls back and retries. Safe (rolls back), and consistent with rebalance being a resilience
  optimization. At `buildOwnersMap` in `ring-shift-coordinator.ts`.
