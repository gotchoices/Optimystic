# Align `NotificationV1.digest` with the threshold-signed commit-vote bytes — REVIEW

This closes the cluster↔reactivity authenticity seam left open by `11-reactivity-origination-replay-delivery`.
Before, origination set `digest = b64url(utf8(event.actionId))` — a value the threshold signature was
**never** computed over — so every crypto-path test had to stub `ICohortThresholdCrypto.verify = () => true`.
Now `digest` carries the exact bytes each cohort member signed, so the existing `verify.ts` (which already
recomputes the threshold check over `b64urlToBytes(n.digest)`) succeeds against **real** Ed25519.

Implemented as the ticket's **Option 1** (surface the signed preimage on `CommitCert`). The ticket explains
why Option 2 (`actionId == commitHash`) is impossible — `actionId` is the transaction id, a different value,
and the signed preimage is `utf8(commitHash + ":approve")`, not the bare hash.

## What landed

**The signed value chain.** Each approving cohort member signs `utf8(commitHash + ":approve")`
(`cluster-repo.ts:528-537`, `signVote`), where `commitHash = base64url(sha256(messageHash ‖
canonicalJson(message) ‖ canonicalJson(promises)))`. This preimage is byte-identical across all approving
signers and reproducible from the agreed `ClusterRecord`. The chain now carries it end-to-end:

- **`CommitCert.signedPayload: Uint8Array`** (db-core, *required* field) — opaque to db-core, minted only by
  the cluster layer.
- **`buildCommitCert(record, minSigs, signedPayload)`** (db-p2p) — takes the preimage as a param and sets it
  on the cert. Deliberately does NOT recompute `commitHash` (the canonical-JSON helper is private to
  `ClusterMember`; duplicating it risks drift).
- **`cluster-repo.ts` commit branch** — computes
  `this.computeSigningPayload(await this.computeCommitHash(record), 'approve')` **before** the synchronous
  `captureCommitCert` + `StorageRepo.commit()` sequence, then threads it through `captureCommitCert` →
  `buildCommitCert`. The single `await` lands before the synchronous capture, so the
  "cert retained before commit() emits its change event" invariant is preserved.
- **`buildNotificationV1`** (db-core) — `digest: bytesToB64url(commitCert.signedPayload)`.

`verify.ts` is **unchanged** — it already did `payload = b64urlToBytes(n.digest)`. This was the whole point:
fix origination + the cert shape, and the verify side just starts succeeding against real crypto.

The headline proof is `reactivity-real-crypto.spec.ts`: it builds real Ed25519 keypairs, produces real
per-member `approve` commit signatures over the preimage via `buildCommitCert`, originates a notification,
and verifies it through `createNotificationVerifier` → `MembershipVerifier` whose
`ICohortThresholdCrypto.verify` is the **real** `verifyCollectedMultisig` (no `() => true`). Positive ⇒
`"verified"`; four negatives ⇒ `"untrusted"`, including **case (b): reconstruct the OLD
`digest = b64url(utf8(actionId))` and watch real crypto reject it** — direct evidence the fix is load-bearing.

## Review these adversarially (work is a starting point, not a finish line)

1. **Unconditional `await computeCommitHash` on the commit hot path.** I followed the ticket's explicit
   design (compute the preimage unconditionally before capture; the ticket calls the extra `sha256`
   "negligible"). Consequence: every cluster commit on **every** node — including nodes with **no** reactivity
   sink wired — now pays one extra `sha256` + one extra `await` microtask, even though `captureCommitCert`
   then no-ops. `computeCommitHash(record)` was *already* computed earlier in the same flow inside
   `validateSignatures` (`cluster-repo.ts:498`) but is not retained. **Decide:** is gating the computation on
   `this.onCommitCertificate` (skip when no sink) worth the extra branch, or is threading the already-awaited
   hash from `validateSignatures` down to here the better fix? I left it unconditional per the ticket; both
   alternatives are low-risk if you prefer them.
2. **Preimage correctness depends on `computeCommitHash` reading only `messageHash + message + promises`
   (not `commits`).** That's what makes `computeCommitHash(currentRecord)` (after self adds its own commit)
   equal the hash every signer signed. Confirm this holds and that no path mutates `record.promises` between
   signature validation and `applyConsensusOperation` — if it did, the captured preimage would diverge from
   what the signers signed and verify would (correctly) fail-closed. The cluster-consensus spec exercises the
   2-peer happy path; a >2-peer or merge-heavy path is not directly asserted here.
3. **Required-field migration completeness.** `signedPayload` is a *required* `CommitCert` field, so any
   missed literal is a compile error — and `db-p2p`'s `tsc` build (which includes `test/`) is green, so the
   compiler vouched for full coverage. Still worth a grep sanity pass for `CommitCert` literals in any package
   not built here (none found in db-core/db-p2p; storage packages don't reference it).
4. **Integration test uses a cached `MembershipCertV1` with a dummy `thresholdSig`.** This is deliberate and
   documented in-test: `MembershipVerifier.verifyMessage` trusts a `cache()`d cert without the
   self-consistent-quorum check (that runs only on the fetch path), so the test exercises the *notification's*
   real Ed25519 check, not the cert's. Production fetch still self-validates. Confirm you're comfortable the
   test isn't accidentally bypassing real crypto somewhere it shouldn't (the `verify` fn is the real
   `verifyCollectedMultisig`, imported from `threshold-crypto.ts` — grep to confirm no stub leaked in).
5. **`mergedDigest` ripple (checkpoint.ts).** Only doc wording changed; the fold still folds
   `b64urlToBytes(e.payload.digest)`. The new digest is still per-revision deterministic and
   cross-member-identical, so the fold stays convergent under gossip and remains a hint (not crypto-verified).
   No logic change — confirm the doc accurately describes the new digest.
6. **Stale doc line.** `docs/reactivity.md` still says (≈L147, outside this ticket's scope) that a subscriber
   "automatically trusts notifications without additional verifiers." With a real verify path that's now loose
   wording, but it predates this ticket and I left it untouched. Flag if you want it tightened.
