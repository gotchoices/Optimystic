# Review: reactivity change-event tail + bridge membership-gate seam

This was the prereq seam for `cohort-topic-host-node-activation` (13.6). It was **completed alongside
13.6** in the same run: the original db-core/storage-repo half was already in the working tree
(uncommitted) but its `change-bridge.ts` half was undone, and â€” critically â€” implementing the gate for
13.6 revealed the seam's premise was **incomplete**: `StorageRepo.commit` never actually received a
`tailId` on the production network path. Both halves are now done and the gap is closed.

## What the seam delivers

`CollectionChangeEvent` now carries the committed collection's chain `tailId` (the reactivity topic
anchor `H(tailId â€– "reactivity")`), and the cohort-topic bridge gates on the **whole event** so the
node-activation predicate can derive the rotating reactivity topic. `attachCohortChangeBridge` returns
`{ notifier, unsubscribe }` so the node can release the catch-all origination feed on stop.

## âš ď¸Ź The non-obvious fix the reviewer should scrutinize first (db-core core commit path)

The seam's TODO ("thread `request.tailId` through `StorageRepo.commit`") was satisfied in the working
tree, **but it didn't work end-to-end**, because `IRepo.commit` takes `RepoCommitRequest` =
`{ blockIds, actionId, rev }` â€” which deliberately has **no `tailId`**. `NetworkTransactor.commitBlocks`
built that bare per-block request, so the collection tail was dropped before reaching any
`StorageRepo.commit`; `request.tailId` was always `undefined` at the emit site (the type said
`CommitRequest` but the runtime object was a `RepoCommitRequest`). Confirmed live: a real `Tree.replace`
emitted change events with `tailId === undefined`.

Fix (closes the gap):
- `RepoCommitRequest` gains optional `tailId?: BlockId`.
- `NetworkTransactor.commit â†’ commitBlock â†’ commitBlocks` thread `request.tailId` into every per-block
  `getRepo(peer).commit({ actionId, blockIds, rev, tailId })`.
- `CoordinatorRepo.commit` and `RepoClient.commit` were **already** typed `CommitRequest` and forward the
  whole request into the consensus commit op (`{ commit: request }`) / wire, and `op.commit.tailId` was
  already read by `repo/client.ts` + `cluster/client.ts` â€” so no further plumbing was needed; once the
  transactor stops dropping `tailId`, it flows to every committing node's `StorageRepo.commit`.

**Adversarial review targets** (this touches EVERY commit):
- Backward compatibility: `tailId?` is optional everywhere; bare per-block paths (churn replication,
  block-transfer) still pass no tail and emit `tailId: undefined` (correctly never originate). Confirm no
  caller now *requires* it.
- Multi-collection / multi-block commits: every per-block commit in one `NetworkTransactor.commit` carries
  the SAME top-level `request.tailId` (the collection chain tail). Confirm that's correct for the
  header/tail/remaining-block split, and that a delete commit (materialized block undefined) still gets
  the tail from `request.tailId`, not the block.
- Serialization: the remote path JSON-serializes `{ commit: request }`; `tailId` is a plain string â€” fine,
  but worth a glance at the repo protocol codec.
