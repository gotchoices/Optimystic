## What the feature is (plain terms)

When a reactivity subscriber falls behind or wakes from sleep, it asks its serving cohort "what did I
miss?" — a **backfill** request (replay a revision range from the ring) or a **resume** request (classify my
lag against the stacked recovery windows). This ticket owns the **pure db-core wire substrate** for those
requests: the on-the-wire message shapes, the exact bytes each request signs (so signer and verifier agree
byte-for-byte), and a discriminated envelope (`RecoverRequestV1`/`RecoverReplyV1`) that the single libp2p
recover protocol frames. The live libp2p transport + node wiring that consumes this lives in db-p2p
(`reactivity-recover-rpc-transport`, already landed).

## Use cases to exercise / validate (the test floor — `recover.spec.ts`, `backfill.spec.ts`)

These 29 tests pass (per-test confirmation captured during implement). They map 1:1 to the originating
ticket's "Key tests":

- **Signing-payload determinism** — byte-for-byte stable for a fixed body; a change to **any** non-signature
  field (`fromRevision`, `toRevision`, `timestamp`, `collectionId`; resume: `latestKnownTailId`,
  `subscriberCoord`) changes the bytes; a backfill image and a resume image with otherwise-identical shared
  fields **never collide** (the type tag differs). These pin the field-order contract — a later reorder
  breaks them.
- **Real Ed25519 round trip** — `recover.spec.ts:79-88` signs `backfillSigningPayload` with a fixed-seed
  `@noble/curves/ed25519` key, verifies the signature, and asserts the **same** signature fails over a
  body with a mutated `timestamp` (the forged-fresh-timestamp replay attack).
- **`BackfillV1` validation** — a frame missing `timestamp` is rejected; a non-finite `timestamp` is
  rejected; a valid frame round-trips `encode/decodeBackfillV1`; `toRevision < fromRevision` still rejected
  (the ordering guard was not regressed by inserting `timestamp`).
- **Recover envelope** — `kind: "backfill"`+`backfill` and `kind: "resume"`+`resume` round-trip; a
  `checkpoint_window` resume reply round-trips through the wrapper (pins that the summary + recentEntries are
  not dropped); a kind/branch mismatch (backfill kind carrying a resume body, no branch present, both
  branches present, unknown kind) is rejected on decode; the `maxMessageBytes` bound is respected like the
  sibling codecs.
- **`createBackfillRequester`** — `backfill.spec.ts:144` asserts the request handed to the transport carries
  the injected clock's `timestamp` value, in the signed image.

## What to scrutinize (suggested adversarial angles)

- **Field-order / canonical-bytes contract is load-bearing.** Signer (subscriber) and verifier (serving
  cohort) must produce identical bytes. The determinism + collision tests pin it for db-core; confirm there
  is no *second* place that re-derives these bytes with a divergent order. (The db-p2p serve handler verifies
  by recomputing the **same** `backfillSigningPayload`/`resumeSigningPayload` — i.e. it imports these, it
  does not re-implement them. Worth confirming.)
- **Cross-package signature compatibility is the real risk, and it lives in db-p2p.** db-core tests use a
  dummy/fixed peer key — they prove the *payload bytes* are stable and an Ed25519 sig over them verifies, but
  not that libp2p's signer agrees. The byte-identity of noble's synchronous `signPeerSig` vs libp2p's async
  signer (and that `createRecoverRequestSigners` produces requests the inbound serve handler accepts) is
  tested in **db-p2p** `test/reactivity/recover-signers.spec.ts` (`byte-identical to the async signPeer …`,
  `signBackfill/signResume produces a request the handler verifies + serves`). If you want end-to-end
  confidence in the signing story, that's the spec to read — it is outside this ticket's package but is the
  load-bearing consumer of these helpers.
- **No `correlationId` nonce — deliberate.** The recover requests intentionally carry no random
  `correlationId` (unlike `RegisterV1`/`RenewV1`); replay protection keys on the **signature bytes +
  timestamp** at the db-p2p node-level guard. This was a fixed design decision (rationale in the originating
  ticket). Confirm you agree the signature-as-token argument holds, but do not treat the absence as a gap.
- **Protocol-id string discrepancy (cosmetic, db-p2p-owned).** The originating ticket's prose wrote
  `/optimystic/reactivity/recover/1.0.0`; the implemented code + docs use
  `/optimystic/reactivity/1.0.0/recover`. Both ends (handler + dialer) agree on the implemented string, so
  it is internally consistent — but the doc/ticket prose mismatch is worth a glance. The protocol id lives in
  db-p2p, not db-core.
