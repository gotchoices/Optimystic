# Review: pin and version the operations-hash byte format

## Plain summary

The **operations hash** is a short fingerprint one database node sends and every
other node recomputes to confirm they agree on exactly what a transaction changes.
The byte layout that fingerprint is built from had no version tag. So if two nodes
ran different software versions that layout the bytes differently, they'd compute
different fingerprints and reject each other with a generic "Operations hash
mismatch" — the same error you'd see for a genuine disagreement or a lie. This
change stamps a version into the fingerprint string (`ops.v1:…`) and makes the
recomputing node say "unsupported operations-hash format version" when the tag
differs, so version skew is diagnosable instead of looking like a content fault.

## What shipped

**Token change.** `hashOperations` now emits `ops.v1:<hash>` instead of the bare
`ops:<hash>`. The hashed body is unchanged — sort by `(collectionId, blockId,
type)`, `canonicalStringify`, SHA-256/base64url. Only the prefix moved.

- `OPS_HASH_VERSION = 'v1'` — single source of truth for the current format version.
- `OPS_HASH_PREFIX = 'ops.v1:'` — the wire token prefix.
- `opsHashVersion(token)` — total parser: returns the `vN` segment, or `null` for a
  bare legacy `ops:` token, an empty string, non-strings, or any garbage. The `.`
  delimiter is outside the base64url alphabet, so the version always slices cleanly.
- `canonicalOperationsPayload(operations)` — returns the **exact string**
  `hashOperations` feeds into SHA-256 (sorted + canonicalized, WITHOUT the token).
  Exposed for the future `design-client-transaction-signatures` ticket so a client
  signature can bind the identical bytes the validators hash.

**Validator split** (`validator.ts` step 9). When the recomputed hash ≠ the sender's:
- sender's version (via `opsHashVersion`) ≠ local `OPS_HASH_VERSION` → distinct
  `Unsupported operations-hash format version: local=v1, sender=<vN|unrecognized>`.
- same version, different bytes → the unchanged `Operations hash mismatch`.

**Docs** (`docs/transactions.md`). New normative section **"Operations Hash —
Canonical Serialization"** specifies the sort key, `canonicalStringify` rules, the
hash step, and the `ops.vN:` token, and names the two dimensions that must both match
(`engineId` for content, the format-version token for serialization). The two stale
illustrative `hashOperations` bodies (the 32-bit `((hash<<5)-hash)` sketch and the
`hash(JSON.stringify(...))` sketch) were replaced with pointers to the real module,
and the validator pseudocode now shows the version-skew/content-mismatch split.

## Why the change rippled into other tests

The wire prefix changed `ops:` → `ops.v1:`. Several tests hand-built the expected
operations hash as `` `ops:${await hashString(JSON.stringify([]))}` `` and fed it to
`validator.validate(...)` expecting `valid: true`. For an EMPTY operation list,
`JSON.stringify([]) === canonicalStringify([]) === '[]'`, so those hashes matched the
old validator purely by luck of the shared prefix. With the new prefix they no longer
match. Fixed by replacing the hand-built literals with the real helper
`await hashOperations([])` (in `transaction.spec.ts` and `client-tx-signature.spec.ts`),
which stays correct across any future prefix/format change.

Tests that pass a garbage hash like `'ops:abc'` and expect failure at an EARLIER step
(unknown engine, schema mismatch, stale read, missing/invalid signature) were left
untouched — they never reach the hash comparison. `quereus-engine.spec.ts` is entirely
in this category and was not modified.

Cases worth an adversarial eye:

- **Token classification** (`opsHashVersion`): current `ops.v1:` → `v1`; a bumped
  `ops.v2:` → `v2` (≠ local); bare legacy `ops:` → `null`; `''`, `'ops.'`, `'ops.:x'`,
  `'ops.v1'` (no colon), non-strings → `null`. All covered in
  `operations-hash.spec.ts`. Confirm none of these is ever mistaken for a content
  match — a `null`/foreign version must route to the version-skew error, never `valid`.
- **`transaction.id` did not move.** `createTransactionId` is a *persisted* identity
  and must be byte-identical for the same inputs; the version token lives only in the
  ops-hash string. `operations-hash.spec.ts` asserts stability; also spot-check that
  no coordinator/log path started using the ops-hash where the tx id is expected.
- **Payload derivability.** `canonicalOperationsPayload(ops)` + `OPS_HASH_PREFIX` +
  `hashString` reconstructs `hashOperations(ops)` exactly (asserted). This is the seam
  the future client-signature ticket binds; verify the payload carries NO token.

## Known gaps / limits (honest floor, not a finish line)

- **Detection, not cross-computation.** A node only *detects* a foreign version; it
  cannot reproduce another version's bytes. Mixed-version clusters fail legibly, not
  interoperably. Multi-version compatibility is deliberately out of scope (future work
  if rolling upgrades ship). Stated in the code NOTE and the doc.
- **No version whitelist.** `opsHashVersion` returns whatever version segment it
  parses; the validator only checks equality with the local version. There is no
  registry of "known" versions — any non-matching version is simply "unsupported."
  Fine today (only `v1` exists); revisit only if several versions must coexist.
- **Full db-p2p and quereus suites not run end-to-end.** Only the specs that touch the
  ops-hash wire format were run (above). The db-p2p src is untouched by this change and
  its other ops-hash reference (`signature-validation-integration.spec.ts`) uses a mock
  validator / tests `SignatureValid` directly, so it is prefix-independent. A reviewer
  wanting full assurance can run the whole db-p2p suite (network-mock heavy; use a
  streaming reporter, not `min`, to avoid idle-timeout).

## Review findings

- **Tripwire (parked in code, not a ticket):** the "detect-not-cross-compute" boundary
  and the "no version negotiation" residual concern live as a `NOTE:`-style paragraph in
  the `hashOperations` doc comment in `operations-hash.ts` and in the docs' normative
  section. If rolling upgrades are ever supported, that NOTE is where the multi-version
  work gets picked up.
- **Tripwire (parked in code):** `opsHashVersion` accepts any well-formed `vN` without
  checking it against a set of known versions — noted above under "No version
  whitelist"; recorded here as the index entry, analysis is in that bullet.
