# Review — reactivity rotation: re-registration timer host seam (db-p2p)

Implements the standalone, unit-testable host that schedules the jittered move of a subscriber to a rotated
tail. The `ReactivitySubscriptionManager` already surfaces a `RotationNotice{ newTailId, preAnnounced, plan:
ReRegistrationPlan }` once per successor; nothing previously scheduled the timer or performed the move. The
new `RotationReRegistrationScheduler` does. The node composition that binds it to the manager's `onRotation`
observer is **deferred to `reactivity-rotation-host-wiring-e2e`** (not in this diff, by design).

## What was built

- **`packages/db-p2p/src/reactivity/rotation-rereg-scheduler.ts`** — `RotationReRegistrationScheduler` +
  `RotationReRegistrationSchedulerOptions` + `RotationTimerCancel`.
  - `schedule(notice)`: de-dupe by successor `newTopicId` (base64url) → arm a one-shot timer for
    `max(0, plan.fireAt − now())` → on fire invoke injected `reRegister(plan): Promise<void>`.
  - Injected `setTimer(fn, delayMs) => cancel` + `now()` for deterministic tests. **Both are optional**, with
    production defaults: an **unref'd** `setTimeout`/`clearTimeout` and `Date.now`.
  - `cancel(newTopicId?)` / `stop()` for teardown; `pendingCount` getter (diagnostic/test seam).
- **`index.ts`** — re-exports the module.
- **`docs/reactivity.md` §Tail rotation step 3** — host scheduler seam + the load-bearing jitter contract.
- **Spec** — 13 cases (fake clock + fake timer queue), all passing.

### Use cases covered by tests (the floor — extend, don't trust as exhaustive)

- Fires exactly at `plan.fireAt` (delay = `fireAt − now`), with the notice's `plan`.
- Past `fireAt` → delay clamped to **0** (fires next tick, never negative).
- Duplicate successor (`newTopicId`) ignored — the redirect-vs-pre-announce race moves once.
- De-dupe **survives firing**: a notice for an already-fired successor is ignored.
- Chained OLD→A→B before A fires → **two** independent timers, both fire (superseded A is *not* cancelled).
- `stop()` cancels pending timers and refuses further scheduling; a timer racing `stop` never invokes
  `reRegister`.
- `cancel(newTopicId)` drops one timer; `cancel()` drops all; cancel-after-fire / cancel-twice / cancel-unknown
  are safe no-ops.
- `cancel(newTopicId)` **forgets** the successor (drops it from the de-dupe ledger), so a fresh notice
  reschedules it.
- A `reRegister` that **rejects** *or* **throws synchronously** is isolated+logged; other timers still fire and
  no unhandled rejection escapes.
- Production defaults (real `setTimeout(0)` + `Date.now`) actually fire the move.

## Areas warranting reviewer scrutiny (design choices / honest gaps)

- **`setTimer`/`now` are optional with production defaults (deviation from the ticket's literal wording).** The
  ticket said "the production node binds `setTimer` to `setTimeout`/`clearTimeout` with an unref'd timer." I put
  that default *in this module* (optional deps) rather than requiring the composing site to supply it — so the
  unref discipline lives in one place and `reactivity-rotation-host-wiring-e2e` can `new
  RotationReRegistrationScheduler({ reRegister })` and get correct behavior for free, while tests still inject
  fakes. The composing site may still override. If the reviewer prefers the deps mandatory, that's a one-line
  change. The unref mirrors `ReactivityPushStateGossipDriver`'s timer (`push-state-gossip.ts:130-135`).
- **`cancel(newTopicId)` clears the de-dupe ledger entry; `stop()` does too.** This means a cancelled successor
  can be rescheduled by a later notice (intentional — `cancel` = "forget this move"). Not spelled out in the
  ticket; flagged for confirmation. If a cancelled successor should stay permanently de-duped, move the
  `seen.delete` out of `cancel`.
- **No retry on `reRegister` failure** (the ticket allowed either; I chose none). A missed move is recovered by
  the next notification re-detecting the still-current rotation, or the subscriber's normal recover/re-walk.
  Documented in the class JSDoc. If a bounded retry is wanted it's an additive change.
- **Load-bearing jitter contract is enforced only by JSDoc + docs prose** — it cannot be enforced here because
  the scheduler only *consumes* `plan.fireAt` (drawn by the *manager's* `rejoinJitter`). The contract: the
  construction site (subscribe factory / `reactivity-rotation-host-wiring-e2e`) MUST build the manager's jitter
  as `createRejoinJitter({ capPromote: DEFAULT_CAP_PROMOTE_FAST })` (= 32), **not** the default (`cap_promote =
  64`), or the rotation burst silently overruns `cap_promote_fast`. **The reviewer should confirm the composing
  ticket (`reactivity-rotation-host-wiring-e2e`) actually carries this** — it is the real place the bug could
  land, and is out of this diff's scope. (Today `subscription-manager.ts:198` defaults to plain
  `createRejoinJitter()`; whether that default is acceptable when no host overrides it, vs. should itself be
  fast-capped, is a judgment call worth a look — but it is the manager's/composing ticket's call, not this
  scheduler's.)
- **Microtask isolation of `reRegister`.** `fire` wraps the call in `try/catch` (sync throw) *and* `.catch`
  (rejection). The synchronous-throw guard is belt-and-suspenders against a mis-implemented seam (the type is
  `Promise<void>`); confirm you're comfortable keeping both rather than only the `.catch`.
- **The default-timer test uses real `setTimeout(0)` + a `flush()`** (one `setTimeout(0)` await). Deterministic
  in practice but the only test not on the fake clock.
