# Review: Wire RebalanceMonitor + react via BlockTransferCoordinator on a live node

## What landed

The rebalance path is now activated end-to-end on a real node. All changes are confined to
`createLibp2pNodeBase` (`packages/db-p2p/src/libp2p-node-base.ts`) plus two new test specs; no changes
to `rebalance-monitor.ts`, `block-transfer.ts`, or `network-manager-service.ts` were needed (the
wiring uses their existing surfaces).

### Source changes (`libp2p-node-base.ts`)

- **Imports**: added `BlockTransferCoordinator` (value) from `./cluster/block-transfer.js` and
  `RebalanceMonitorConfig` (type) from `./cluster/rebalance-monitor.js`.
- **`NodeOptions.rebalance?`**: new optional `Partial<RebalanceMonitorConfig> & { enabled?: boolean }`
  field, mirroring `spreadOnChurn`. `enabled` is consumed by the gate; the rest (debounceMs,
  minRebalanceIntervalMs, suppressDuringPartition) is passed straight to the monitor. Default on.
- **Wiring block**: inserted inside the arachnode `if (fret) { … }` gate, immediately after the
  `restorationCoordinatorV2` construction + restore-callback replacement and before the ring-capacity
  `setInterval`. The block is gated `if (networkManager && (options.rebalance?.enabled ?? true) !== false)`
  and does, in order:
  - constructs `new BlockTransferCoordinator(storageRepo, keyNetwork, restorationCoordinatorV2,
    partitionDetector, protocolPrefix)` — **local** `storageRepo` (not repoProxy/coordinatedRepo) and the
    `/optimystic/<networkName>` prefix that matches the registered block-transfer receive handler;
  - `networkManager.initRebalanceMonitor(partitionDetector, fretAdapter, options.rebalance)` then
    `await rebalanceMonitor.start()`;
  - `rebalanceMonitor.onRebalance(event => { void coordinator.handleRebalanceEvent(event) })`;
  - a **separate** `storageRepo.onAnyCollectionChange` subscription feeding `trackBlock` for each
    `e.blockIds` (same feed spread uses; the unify follow-on collapses the two);
  - exposes `(node as any).rebalanceMonitor` and `(node as any).blockTransferCoordinator`;
  - installs a `node.stop` wrapper (captured-`previousStop`-last, composing with the spread / arachnode /
    clusterMember wrappers) that releases the subscription and stops the monitor;
  - wraps the whole thing in a non-fatal `try/catch` (log + continue) — a wiring failure leaves rebalance
    inert and does NOT hard-fail node startup, unlike the operator-opted-in cohortTopic block.

When arachnode is disabled or FRET is absent (`fretAdapter`/`restorationCoordinatorV2` don't exist),
the rebalance path stays inert — by design.

## Known gaps / honest flags (reviewer: treat tests as a floor)

- **"Started" + listener-registration is asserted reflectively / transitively.** The node-wiring spec
  checks `(monitor as any).running === true` and relies on `rebalance-monitor.spec.ts` to prove that
  `start()` registers the `connection:open`/`connection:close` listeners. I did **not** assert a real
  listener count on the live node: this libp2p version's emitter is a bare `EventTarget` (no
  `listenerCount`). A stronger end-to-end proof (emit a real connection event on a multi-node mesh and
  observe a pull/push over sockets) is integration-scale (see `real-libp2p.integration.spec.ts` /
  `substrate-real-libp2p.integration.spec.ts`) and was not added here.
- **No live socket-level rebalance E2E.** Nothing proves that a genuine topology change on a running
  multi-node mesh moves blocks over the wire. The unit + wiring specs are the floor; the reaction
  primitive itself is socket-tested elsewhere only for spread/block-transfer, not for a monitor-driven
  rebalance. Candidate for an env-gated integration test if the reviewer wants belt-and-suspenders.
- **Initial-scan gap (inherited, intentional).** Blocks already durable from a previous run are not
  re-emitted by `onAnyCollectionChange` on startup, so they are not tracked until next touched — same
  caveat the spread ticket documented; the deferred seed is `optimystic-owned-block-initial-scan-seed`.
- **Two tracked sets, double-stop.** This ticket deliberately keeps a **separate** owned-block
  subscription + tracked `Set` per monitor (spread vs rebalance). `unify-monitor-tracked-block-set` is
  the follow-on that collapses them onto one shared owned-block set and drives eviction from the `lost`
  signal — **not done here**. Also: both `NetworkManagerService.stop()` and the new `node.stop` wrapper
  call `rebalanceMonitor.stop()`; this is idempotent (early-returns when not running) — belt-and-
  suspenders, same as spread.
- **`enabled` passthrough.** `options.rebalance` (including `enabled`) is passed wholesale as the third
  arg to `initRebalanceMonitor`, whose param type is `RebalanceMonitorConfig`. `enabled` is an excess
  property the monitor ignores; harmless, but a reviewer may prefer destructuring `{ enabled, ...cfg }`
  before forwarding `cfg`. (Mirrors how spread forwards `options.spreadOnChurn`.)
- **Deployment assumption.** Rebalance is live only when arachnode+FRET are present. The arachnode-
  enabled deployments (e.g. sereus-health mobile, clusterSize=1) get it; arachnode-disabled nodes do
  not. Confirm this matches intended rollout.
