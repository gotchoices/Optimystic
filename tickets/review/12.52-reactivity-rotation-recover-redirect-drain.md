# Review ŌĆö reactivity rotation: recover drain-redirect serve + subscriber honor (db-p2p)

Adversarial review of the drain-redirect path that moves a stuck/waking subscriber off a **rotated, still-draining** tail. The old serving cohort now records that it rotated and answers recover requests (resume, and best-effort backfill) with a `kind: "rotated"` redirect for `T_drain`; the subscriber honors that redirect through the same jittered `onRotation` re-registration seam a notification pre-announce uses.

## What landed

### A. Rotation state on the forwarder host (`forwarder-host.ts`)
- `markRotated(oldTopicId, { newTailId, effectiveAtRevision }, now)` ŌĆö starts a db-core `TailDrainGate` keyed by the **old** topicId. db-core derives `newTopicId` internally via `reactivityTopicId(newTailId)`. **Idempotent**: a second mark to the same successor (or an *earlier* one, lower `effectiveAtRevision`) is a no-op (does **not** restart the drain window); a mark to a **later** successor (higher `effectiveAtRevision`) replaces the gate and restarts drain from `now`.
- `rotationRedirectFor(oldTopicId, now): RotationRedirectV1 | undefined` ŌĆö returns `gate.rotationRedirect` while `gate.isDraining(now)`; once `T_drain` elapses, evicts the gate **and** the topic's served `PushState` + `ingestTails` entry (reclaiming the maps the 12.31 review flagged as un-evicted on rotation), then returns `undefined`.
- Trigger is **not wired here** (see gaps): `markRotated` is invoked directly in tests.

### B. Recover serve returns the redirect while draining (`recover-transport.ts`)
- New `RecoverServeDeps.rotationFor?(req: { topicId?, collectionId }, now)` dep. Node wiring binds it to `forwarderHost.rotationRedirectFor`, resolving the old topic: resume passes `topicId = reactivityTopicId(latestKnownTailId)`; backfill passes **collectionId only** (the binding resolves the collection's current served topic).
- **Resume** serve: when `rotationFor` returns a redirect, reply `{ kind: "rotated", rotated }` ahead of the windows classification; otherwise serve as before **and now pass `ps.inheritedCheckpoint` into `serveResume`** (cross-rotation resume to the new tail serves from the inherited handoff).
- **Backfill** serve: same redirect-first branch (keyed by collectionId); else `serveBackfill` as before.

### C. Outbound transport surfaces the redirect (`recover-transport.ts`)
- `exchange` decodes a `kind: "rotated"` reply and throws the new exported `RotationRedirectError` (carries the `RotationRedirectV1`). **Terminal** in the cohort-walk ŌĆö it does not fall through to the next member (contrast a dial failure, which still does). `index.ts` re-exports it via the existing `export * from "./recover-transport.js"` (no explicit change needed).

### D. Subscriber honors the redirect (`subscription-manager.ts`)
- `escalateBackfill` and `resume()` catch `RotationRedirectError`, route it through a shared `surfaceRotation(newTailId, preAnnounced=false)` helper (extracted from `checkRotation`, so notify-driven and recover-driven rotation share one once-per-successor seam): invalidate the sticky cohort-hint cache, then fire `onRotation({ newTailId, preAnnounced: false, plan: planReRegistration(...) })` guarded by `rotationHandledFor`. `resume()` resolves the redirect as `ResumeApplyOutcome` `"tail_rotated"` (never throws out); the gap seam stays detached + catch-guarded.

## Use cases / behavior to scrutinize

- **The core race the ticket fixes**: a subscriber reaching the *old* draining cohort with `latestKnownTailId == oldTail` must get redirected, not served stale data up to `rotationRevision` then silence. Verify the redirect fires *ahead of* the windows classification.
- **Asymmetry to confirm**: the backfill redirect is emitted only when the node serves **solely** the old draining tail ŌĆö once both tails coexist, `pushStateForCollection` resolves the new tail (highest `lastRevision`) and backfill is served normally. This is documented as a deliberate best-effort secondary path (primary path is notify-driven detection).
- **Self-dial / co-located**: `markRotated`/`rotationRedirectFor` are node-local, so a co-located subscriber's in-process recover sees the redirect. Not exercised end-to-end here (no node-wiring).

## Known gaps / honest flags (treat tests as a floor)

1. **No live e2e of the full loop.** The `markRotated` **trigger** (origination observing `event.tailId` change) and the **`rotationFor` binding** (node wiring binding `RecoverServeDeps.rotationFor` ŌåÆ `forwarderHost.rotationRedirectFor`, including the backfill topic-resolution adapter `pushStateForCollection ŌåÆ topicId ŌåÆ rotationRedirectFor`) are explicitly **deferred to `reactivity-rotation-host-wiring-e2e`**. Unit tests inject `rotationFor` directly and call `markRotated` directly, so the *binding adapter itself is unverified here*. The existing `mesh-tail-rotation.spec.ts` exercises a hand-built `TailDrainGate`, not the host's `markRotated`/recover path. Reviewer: the seam contracts are pinned, but the integration is not.
2. **Lazy, on-demand eviction.** The drained gate + served `PushState` are reclaimed only on the **next** `rotationRedirectFor` call after `T_drain`. If no recover request ever reaches the old topic post-drain, reclamation waits for the host-wiring demotion / a future sweep ŌĆö the 12.31 leak is *reduced* (now reclaimable) but not proactively swept. Confirm this is acceptable or wants a follow-on.
3. **Chained-rotation ordering assumes monotonic `effectiveAtRevision`.** `markRotated` replace/no-op decision compares `effectiveAtRevision` only; a (physically implausible) later successor with an equal-or-lower `effectiveAtRevision` is treated as a no-op. Confirm the invariant (successor rotations strictly increase `effectiveAtRevision`) holds upstream.
4. **`resume()` maps a redirect to `"tail_rotated"`.** Semantic choice ŌĆö a redirect is treated as a tail rotation outcome (so `escalateBackfill`'s `await this.resume(); return;` treats it as success and skips `onChainRead`). Reasonable, but worth a sanity check that no caller distinguishes a real `tail_rotated` reply from a redirect-derived one.
5. **Forward-compat is decode-closed only.** A peer predating `kind:"rotated"` fails the reply decode ŌåÆ the transport treats it as a dial failure ŌåÆ falls through / chain-reads. Safe, but not directly tested at the db-p2p layer (db-core `recover.spec.ts` covers the codec's fail-closed decode).
