# Reactivity recover RPC transport — review handoff

## What this delivers (and the unusual prereq situation — read first)

This ticket (`reactivity-recover-rpc-transport`, 12.41) is the db-p2p transport for the reactivity
**recovery (pull)** direction. It was dispatched **out of order**: both of its prereqs were still unlanded
in `implement/` when this run started —

- **`reactivity-recover-wire-signing` (12.4)** — the pure db-core wire foundation this ticket *consumes*
  (`recover.ts` envelope, `backfillSigningPayload`/`resumeSigningPayload`, `BackfillV1.timestamp`). It did
  **not** exist in the codebase.
- **`reactivity-notification-transport` (12.33)** — the db-p2p node-wiring composition (the `fwd` forwarder
  host, subscriber registry, notify transport, origination manager, `resolveCohort`/`CohortPeerResolver`,
  subscribe factory inside `libp2p-node-base.ts`). It **errored mid-flight** (resume-note present) and its
  node wiring is **absent** — the `cohortEnabled` block in `libp2p-node-base.ts` still only installs the
  cohort-topic host + change bridge (its own comment at ~L764 says "installing the origination manager +
  emit transport is a sibling ticket").

Per the workflow rules, a tess-ticket prereq still in `implement/` is **not** a blockable condition
("deferred automatically… neither is 'an upstream tess ticket isn't done yet'"), and the implement stage
requires a green build. Since the db-p2p transport cannot compile without 12.4's symbols, this run
**materialized 12.4's db-core foundation** (exactly as that ticket specifies) so the build is green, then
built the genuinely-12.41 transport machinery on top, and **split the live node wiring** (which truly needs
12.33's composition) into a prereq-chained follow-on `reactivity-recover-node-wiring` (in `implement/`).

> **Reviewer / runner note on 12.4:** the `reactivity-recover-wire-signing` ticket is still sitting in
> `implement/`, but its entire scope (wire framing, `timestamp`, signing payloads, `createBackfillRequester`
> clock, tests, docs) is **already landed in-tree** by this run. When that ticket is picked up it should
> find its work complete (verify + advance), not re-implement it. This was unavoidable given the
> out-of-order dispatch + the no-block-on-tess-prereq rule.

## What was built

### db-core wire foundation (12.4's scope — landed here out of necessity)

- **`BackfillV1.timestamp`** (`backfill.ts`): new required `number` field (unix-ms), validated with
  `reqFiniteNumber`, `signature` kept last. Bound into the signing payload so freshness is authenticated.
- **`backfillSigningPayload` / `resumeSigningPayload`** (`backfill.ts` / `resume.ts`): canonical signed byte
  images — explicitly-ordered, type-tagged JSON arrays (`["BackfillV1", v, collectionId, fromRevision,
  toRevision, timestamp]` and `["ResumeV1", v, collectionId, fromRevision, latestKnownTailId,
  subscriberCoord, timestamp]`), mirroring cohort-topic `registerSigningPayload`. The leading tag prevents
  backfill/resume image collision.
- **`createBackfillRequester` `clock`** dep: stamps `timestamp: deps.clock()` into the unsigned image
  (no ambient clock in db-core proper).
- **`recover.ts`** (new): the discriminated `RecoverRequestV1` / `RecoverReplyV1` envelope + four
  `encode/decode` functions, reusing the inner per-message validators. Decode enforces the populated branch
  matches `kind` and the other branch is absent.

### db-p2p transport machinery (12.41 proper, minus node wiring)

- **`protocols.ts`**: promoted the reserved recover slot to `PROTOCOL_REACTIVITY_RECOVER =
  ${REACTIVITY_BASE}/recover`; added `recover` to `ReactivityProtocols`, `DEFAULT_REACTIVITY_PROTOCOLS`,
  `makeReactivityProtocols`, and `reactivityProtocolList` (so the existing `node.unhandle(...)` teardown
  already covers it once it is handled).
- **`recover-transport.ts`** (new):
  - `Libp2pReactivityRecoverTransport` — exposes `backfillTransport(topicId, collectionId)` /
    `resumeTransport(topicId, collectionId)` (the db-core seams). Target order: **sticky cohort-hint primary
    first**, then **cohort-walk** members; self is never dialed; a dial failure falls through to the next
    candidate (sticky → walk), a wrong-`kind` reply is terminal. The wire exchange is injected as a
    `RecoverDialer` (production impl = `createLibp2pRecoverDialer` over `requestResponse`) so target
    selection + framing unit-test without a real socket.
  - `decodeCohortHintTarget(primary)` — the **load-bearing encoding bridge**:
    `bytesToPeerIdString(b64urlToBytes(primary))` converts the sticky `primary` (base64url-of-member-bytes)
    to a `peerIdFromString`-dialable string. Pinned both ways by tests.
  - `createRecoverRequestHandler(deps)` / `registerRecoverHandler(node, protocol, deps)` — the inbound
    serve handler: bounded decode → `verifyPeerSig` against the **dialing peer** (the dialer *is* the
    signer) → node-level `CorrelationReplayGuard` keyed on the **signature bytes** + timestamp → resolve the
    live `PushState` (`pushStateFor(reactivityTopicId(latestKnownTailId))` for resume, falling back to
    `pushStateForCollection` for backfill and rotated-tail resume) → `serveBackfill`/`serveResume` → encode
    reply. Any decode/verify/replay/foreign-collection failure produces **no reply** and never throws out of
    the handler.
- **`forwarder-host.ts`**: `pushStateForCollection(collectionId)` — scans the topic-keyed served map for
  the matching collection, returning the **highest-`lastRevision`** served topic (the current tail across a
  rotation drain).
- **`subscription-manager.ts`**:
  - `onBackfillGap` now passes `clock` to `createBackfillRequester` and replaces the bare fire-and-forget
    with `void requester(from,to).catch(() => void escalateBackfill(from,to))`.
  - `escalateBackfill` — bounded retry (`backfillMaxRetries`, default 1) on a **transport failure**, then
    escalate to `resume()` (when configured) else `onChainRead`. Never rejects, never touches the
    commit/delivery path. The `available`-window **underflow** path stays separate (`onBackfillUnderflow`),
    so the two escalation causes don't double-fire.
  - `subscriberCoord` — dropped the `?? collectionIdB64` inline placeholder; the manager now uses the
    supplied real ring coordinate and only falls back to the collectionId placeholder with a logged warning.

## Use cases to exercise in review

- **Subscriber gap → backfill**: a gap fires `onBackfillGap`; the signed `BackfillV1` reaches the handler,
  serves from a seeded ring, and the reply replays through delivery (verify→contiguity→deliver→dedupe).
- **Mobile wake → resume**: each of the four `serveResume` classifications carried faithfully over the
  transport; a subscriber whose tail rotated away (exact topic unserved) still gets `tail_rotated` via the
  collection lookup.
- **Adversarial**: wrong-key signature → no reply → dialer sees a dropped stream → manager escalates;
  identical frame twice → second rejected (serve ran once); stale timestamp → rejected.
- **Encoding regression**: the sticky primary round-trips to a dialable peer-id string; the raw
  base64url-of-bytes fed to `peerIdFromString` throws (the silent-no-dial trap, pinned both ways).

## Honest gaps / what the reviewer should scrutinize

- **Live node wiring is deferred** to `reactivity-recover-node-wiring` (in `implement/`, prereq
  `reactivity-notification-transport`). Nothing in `libp2p-node-base.ts` constructs the recover transport,
  registers the handler, or feeds `backfillTransport`/`resumeTransport`/real `subscriberCoord` into the
  subscribe factory yet — because 12.33's `fwd`/registry/`resolveCohort`/subscribe-factory composition does
  not exist. The transport + handler are structured as injectable units precisely so that wiring is a thin
  composition step once 12.33 lands.
- **Outbound dial path has no real-socket test here.** The `RecoverDialer` seam is exercised with an
  in-process handler-piping fake; the real `requestResponse`/`readAllBounded` path over a socket is the
  env-gated integration test owned by the follow-on. Review whether `createLibp2pRecoverDialer` (a one-liner
  over `requestResponse`) needs more than the integration test will give it.
- **`RecoverDialer` injection deviates from the ticket's literal `constructor(node, opts)` shape** — chosen
  for testability (the node-wiring builds `createLibp2pRecoverDialer(node, ...)`). Confirm this is the
  preferred seam vs. taking the raw `Libp2p`.
- **Rotation drain-window serving** (both old + new tail serving, `RotationRedirectV1` honoring) is
  `reactivity-tail-rotation-transport` (12.5)'s. This serves `tail_rotated` from the current
  `PushState.tailIdAtJoin` + the highest-revision collection lookup; the precise drain/redirect choreography
  is 12.5's.
- **`subscriberCoord` reply routing** is unused by this same-stream request-reply transport; the field is
  sourced + signed correctly for a future out-of-band reply path.
- **Replay-guard scope**: a single node-level guard keyed on signature bytes (globally unique). Per-topic
  guards / a tuned window are a follow-on if recover volume ever warrants it.
- **`checkpoint_window` resume** is not exercised over the transport here (it needs a ring-eviction fixture);
  it is covered in db-core `resume.spec.ts`, and the transport's faithful-carriage tests cover the other
  three variants. Worth a glance whether a transport-level checkpoint_window case is wanted.
