## What was built

A transaction now carries an **advisory, aged `priority`** (default 0) that rises by one per failed
retry attempt and is capped at `MaxPriority` (8). A cluster member consults it as the **first
tiebreak** when deciding a race between two conflicting, simultaneously-pending transactions, so a
transaction that keeps losing eventually out-ranks fresh (priority-0) rivals instead of coin-flipping
forever on the message hash.

Priority is **fairness-only** and was kept entirely out of the safety argument: it does not enter the
transaction id, the client signature, the operations hash, stale-read checks, or validity. It rides
*inside* the signed cluster `message`, so it is integrity-protected in transit but the originating
coordinator can still self-assert it (a bounded fairness concern, not a safety hole — see tripwire).

### Data flow (how priority reaches the race)

```
multi-collection:  TransactionCoordinator.commit (staleLosses)
                     → transaction.priority = clampPriority(staleLosses)
                     → pendCollection builds PendRequest{ transaction }   ← priority on the tx
single-collection: Collection.sync (consecutiveFailures)
                     → TransactorSource.transact(..., clampPriority(consecutiveFailures))
                     → PendRequest{ priority }                            ← priority top-level

both → RepoMessage.operations[0].pend  →  computeClusterMessageHash(canonicalJson(message))
     → ClusterRecord.message (signed, JSON over the wire — new field survives)
     → member ClusterMember.resolveRace → recordPriority(record):
           clampPriority(pend.transaction?.priority ?? pend.priority)   // absent ⇒ 0
```

New `resolveRace` order: **(1) higher priority, (2) more promises, (3) higher message hash.**

### Progress bound (Theorem 9, updated)

- **Concurrent contention: now deterministic.** Once a starved transaction's priority clears a fresh
  rival's 0 — at most `MaxPriority` losses — every honest member picks it in every subsequent
  concurrent race. "2⁻ᵏ but memoryless" → "bounded losses, then a deterministic win."
- **Sequential starvation: explicitly NOT solved here.** A stream of quick transactions that each
  commit-and-finish inside the window never co-pend with a slow transaction, so they never meet it in
  `resolveRace`. That residual is deferred to backlog `feat-occ-priority-reservation` (already filed).

## How to validate / test use cases

Automated (added, green):
- **db-core** `test/transaction.spec.ts` → `describe('Aged priority ...')` and the stale test in
  `Transaction Validation`:
  - `clampPriority` bounds `[0, MaxPriority]`, floors, maps +Inf/over-cap → cap, NaN/≤0 → 0.
  - tx id **identical** for two attempts differing only in priority (identity stability).
  - client signature payload **identical** across priority (anti-replay/OCC binding never churns).
  - a `priority == MaxPriority` transaction with a **stale read** is still rejected at validation.
- **db-p2p** `test/cluster-repo.spec.ts` → `describe('priority-aged race resolution')`:
  - priority beats a rival with **more promises**.
  - reads priority from the multi-collection carrier (`pend.transaction.priority`) too.
  - **livelock guarantee**: an aged transaction wins within `MaxPriority + 1` concurrent rounds.
  - **cap symmetry**: two capped-out conflicts fall back to the hash tiebreak, order-independent;
    an over-cap self-assert clamps down to `MaxPriority`.
  - **mixed-version**: a record with no priority field = priority 0; race stays deterministic.
  - **in-transit tamper**: bumping priority after signing ⇒ `Message hash mismatch`.

## Honest gaps for the reviewer (treat tests as a floor)

1. **No live end-to-end integration test.** The guarantee is proven at the `resolveRace` unit level
   and the two retry loops are shown to set priority, but there is **no test that runs a real
   multi-node cluster through `NetworkTransactor` and observes an aged transaction actually winning a
   live concurrent race and committing.** The pieces are tested; their composition through the wire is
   not. This is the biggest coverage gap — a good place for an adversarial pass.
2. **Two carriers, by necessity.** The ticket framed priority as "a field on `Transaction`," but the
   single-collection `Collection.sync` pend has no `Transaction`, so that path uses a top-level
   `PendRequest.priority` instead. `resolveRace` reads `pend.transaction?.priority ?? pend.priority`.
   Confirm this dual-carrier is acceptable (vs. e.g. always mirroring onto one field).
3. **`cluster-coordinator.ts` needed no hashing change.** The ticket listed it as the message builder
   to modify; because priority already lives inside `message` (via the pend), `computeClusterMessageHash`
   covers it automatically. Only a breadcrumb comment was added there — verify that reasoning.
4. **Only targeted specs were run**, not the full db-p2p suite or the whole monorepo. CI / a broader
   `yarn test` pass is worth doing.
5. **`Collection.sync` priority path is thinner-tested than the coordinator path** — it is exercised
   indirectly (transact signature + resolveRace read the top-level carrier) but there is no test that
   drives `Collection.sync` through repeated stale failures and asserts the pend's `priority` climbs.

## Review findings (tripwire index — do NOT file as tickets)

- **Byzantine self-assert of priority (fairness DoS, not safety).** A coordinator can stamp
  `priority == MaxPriority` on every transaction. `recordPriority` clamps to the cap and priority never
  touches validity/operationsHash/stale-read, so it can only win races it might have ~50% won anyway —
  at-worst-status-quo fairness (same class as spam under honest-majority). Hardening (bind priority to
  *provable* age) is deliberately out of scope. Parked as a `NOTE:` at the `resolveRace` site in
  `cluster-repo.ts` and in `docs/correctness.md` Theorem 9's Byzantine note; no code change wanted now.
- **Hot-log-tail composition.** `resolveRace` + the record message are the same path a future
  HLC/crdt-sync redesign (`design-hot-log-tail-sharding-guidance`) will touch. Priority was kept a
  self-contained additive message field + one comparison key so it composes with that redesign; parked
  as a `NOTE:` cross-reference at the `resolveRace` site.
- **Fairness-vs-throughput bound.** Aging a transaction up makes short rivals lose that block's races
  while it pends; the cost is bounded (that block, that pend duration, ≤ `MaxPriority` extra losses per
  aged transaction). Parked as a `NOTE:` at the `resolveRace` site.
