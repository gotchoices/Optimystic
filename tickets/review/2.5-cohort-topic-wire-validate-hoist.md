# Review: hoist duplicated wire-validation primitives into one shared module

## What this was

The same generic structural-validation helpers (`asObject`, `requireV1`, `reqString`, `reqFiniteNumber`,
`reqStringArray`, the base64url helpers, the fail-with-`CohortWireError` shim, etc.) were copy-pasted
across multiple wire codecs and had begun to diverge. This ticket consolidated them into one module —
`packages/db-core/src/cohort-topic/wire/primitives.ts` — that every codec now imports. Consolidation
only; the only intended behavior delta (the fixed-length `b64urlFixedLen` byte-width check) already
landed in the prior `cohort-topic-wire-hardening` ticket and is preserved here.

## What actually changed vs. the plan (read this first)

The ticket named **three** copies. There is a **fourth**: `reactivity/push-state.ts` carried its own
private `fail` / `asObject` / `reqString` / `reqFiniteNumber` / `b64urlField` (the ticket listed it only
as a "consumer" of copy 3, but it never imported from `wire-validate.ts` — it re-declared its own). Since
the ticket's acceptance gate is "no duplicated primitive set remains," I folded those five into the
shared import as well. **`push-state.ts` keeps one local helper, `reqInt`** — a range-free integer check
(`Number.isInteger`, no lower bound). The shared module only has `reqIntInRange`, which *requires* a
`min`; forcing `reqInt` through it would add a `>= 0` bound and change behavior, so it stays local. This
is documented in a code comment at `push-state.ts` (the `reqInt` definition site).

## Canonical signatures chosen (the real hazard the ticket flagged)

The copies were not textually identical; the reconciliation is where a bug could hide. Decisions:

- **`failWire`** is the canonical export name (reactivity's name). The modules that internally prefer the
  terser `fail` alias it on import (`import { failWire as fail }`), so their ~40 call-site bodies were
  untouched — lower risk than renaming every call.
- **`reqBool` / `optBool`** canonical (paired naming; matched 2 of 3 modules). Reactivity's single
  `reqBoolean` call migrated to `reqBool`.
- **`reqIntInRange(obj, key, what, min, max?)`** — the object+key form (reactivity's), which reads and
  finite-checks the field itself. Matchmaking's value-form `reqIntInRange(value, key, …)` and its **9**
  call sites migrated. Behavior is identical because matchmaking always wrapped the argument in
  `reqFiniteNumber(obj, key, what)` first, and the obj-form does exactly that internally.
- **`reqStringArray(obj, key, what)`** — the object+key form (cohort-topic/matchmaking's). Reactivity's
  single value-form call site (`wire.ts`, NotificationV1 `signers`) migrated.

Every error-message string is byte-identical across the old copies, so no test that asserts on message
text can observe the move (spot-checked: `reqIntInRange`, `reqStringArray`, `b64urlField` messages match).

## CohortWireError relocation

`CohortWireError` moved from `validate.ts` into `primitives.ts` (so the primitives that throw it and the
class live together, and there's no cross-package coupling — `b64urlToBytes` is also in
`cohort-topic/wire`). `validate.ts` re-exports it (`export { CohortWireError } from "./primitives.js"`)
so every existing importer — `codec.ts`, `wire/index.ts`, matchmaking, reactivity, push-state — is
unaffected. **Circular-import note:** `primitives.ts` imports `b64urlToBytes` from `codec.ts`, and
`codec.ts` imports `CohortWireError` (via `validate.ts`) back — but this exact cycle already existed
between `validate.ts ↔ codec.ts`, and all usages are runtime (a class + functions), never at module-init,
so it is safe. Worth a reviewer's eye if you distrust ESM cycles.

## Where a reviewer should focus (test floor, not ceiling)

- **The 9 migrated `reqIntInRange` call sites in `matchmaking/wire.ts`** are the highest-risk change —
  the argument shape flipped from a pre-extracted number to (obj, key). Confirm each still targets the
  right field name and min/max (esp. `limit` → `1..QUERY_LIMIT_MAX`, the `minBudget` optional-branch case
  at `validateCapabilityFilter`, and the three `AggregateBucketV1` buckets). Existing matchmaking wire
  specs pass, but the tests may not cover every out-of-range rejection path — a targeted "negative /
  non-integer / over-max on each numeric field still throws `CohortWireError`" sweep would raise the
  floor.
- **`reqStringArray` obj-form on reactivity `NotificationV1.signers`** — verify a non-array / mixed-type
  `signers` still rejects (the value-form → obj-form flip reads `obj["signers"]` internally now).
- **`push-state.ts` `reqInt` staying local** — confirm you agree it should NOT be tightened to `>= 0`.
  If the team decides a shared range-free `reqInt` primitive is wanted, that's a follow-up, not part of
  this consolidation.
