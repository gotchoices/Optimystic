# Review: multi-collection atomicity — honest downgrade + reporting-contract tests

## What this ticket did (and the decision it must NOT re-litigate)

The plan settled that Optimystic commits a multi-collection transaction **collection-by-collection with no cross-collection undo**, so a mid-loop failure can half-land. It chose **Option B — honest downgrade** (formally: *atomicity of intent + eventual, reported visibility*) over Option A (real cross-collection 2PC). The reasoning is in the implement ticket's "Decision (settled in plan)" section; the trigger is a *permanent stale loss* (a racing txn steals a collection's log-tail slot between its PEND and COMMIT), which a 2PC journal cannot heal. **Do not re-decide the model.** Option A is parked as backlog `feat-cross-collection-atomic-commit` (already on disk, `prereq: design-cluster-membership-agreement`).

This ticket was mostly **making the promise honest in the docs** and **proving the existing reporting surface with tests** — the runtime machinery (`CoordinatorPartialCommitError`, `ExecutionResult.committedCollections/failedCollections`, the `commit()`/`execute()` partial-path splits, the `txn-bridge.ts` no-false-rollback branch) already shipped under the earlier `txn-multi-collection-commit-honest-reporting` ticket. No runtime behavior was changed here.

## Use cases the tests pin (the contract a caller can rely on)

- A multi-collection `commit()` that half-lands throws `CoordinatorPartialCommitError` naming the durable set vs the reverted set; the winner gets the success-path local treatment (fold+reset+clear), the loser is restored for clean retry, and `stampData` is dropped.
- The `execute()` path surfaces the same partition on `ExecutionResult` **without throwing** (it is non-retryable and not snapshot-wrapped), applying `recordCommitted`+`tracker.reset` to the committed subset only.
- A clean failure (PEND fails, or every collection loses) throws a **plain** `Error`, never the partial signal, and leaves every tracker pristine for retry.
- Re-driving `commit()` after a partial landing re-attempts **only** the failed collection — no duplicate log entry on the winner.

## Known gaps / things for the reviewer (treat tests as a floor)

- **Bridge-level partial-commit test is still absent.** The `txn-bridge.ts` `CoordinatorPartialCommitError` catch branch (no false `rollbackTransaction()`) has **no direct test** — covered only by the plugin build (type-check) and by mirroring the tested legacy `PartialCommitError` branch. This is already tracked by the existing backlog ticket **`debt-bridge-partial-commit-branch-test`** (needs a real session-mode + fault-injection vtab harness); I deliberately did **not** re-file it or attempt it here. If the reviewer wants it in-scope, that debt ticket has the full recipe.
- **No portable coordinator-crash-mid-loop test.** Nothing can portably kill the process mid-commit-loop. The doc states the outcome (partial landing, reconcile); the tests instead assert local memory matches durable state on the committed/failed split, which is the observable invariant regardless of when the loop stops. If the reviewer wants a crash-injection harness, that's a new ticket, not an inline fix.
- **The no-double-apply test drives `coordinator.commit()` twice directly** (a real end-to-end retry), relying on: users being excluded from the retry's `collectionData` because its tracker was reset+cleared; posts being re-pended cleanly because its first pend was cancelled by `coordinateTransaction`'s targeted cancel. Worth scrutinizing that retry path — it is the least-exercised code shape in the suite.
- **Two capability-table cells still use the short label "atomic"** — `docs/optimystic.md:15` ("Multi-collection atomic") and `docs/architecture.md:278` ("Multi-collection atomic mutation"). Left unedited: they name a feature *family* in a quick-reference table, not a formal guarantee, and both files are outside this ticket's scope. Flagging so the reviewer can decide whether to relabel (e.g. "Multi-collection (atomic-intent)").
- **Pre-existing tripwire carried in the prior complete ticket** (not re-filed): both `commit()`'s and `execute()`'s committed-subset loops iterate `result.actions`/`collectionData`, not a deduped collection set, so an engine emitting two `CollectionActions` for the *same* collection would double-bump its rev via `recordCommitted`. Pre-existing on the success paths, harmless today (nothing asserts the rev). Not this ticket's code.
