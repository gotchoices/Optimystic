# Review: surface invalidation to clients (push + pull)

Builds on `7.5-invalidation-log-entry-and-reversal` and `7.6-invalidation-cascade-detection` (both in
`complete/`). Implemented as a **library-stage vertical slice**, matching those tickets: the types, pure
assembler, durable status path, client reaction, and commit-cert capture are built and unit-tested; the
final node-side **emit** of an invalidation change event is composition-root wiring deliberately left as
an injection point (see "Known gaps" — it's the same gap #1 the parent tickets carry, gated on the
certificate-binding fix).

## What landed (verified building + tested)

### Push — typed invalidation marker, reused cert
- `CollectionChangeEvent` (`change-notifier.ts`) and `NotificationV1` (`reactivity/wire.ts`) gain an
  `invalidation?: boolean` flag + `invalidatedActionId?`. Wire codec validates + round-trips them
  (`reqBoolean` added to `wire-validate.ts`).
- `buildNotificationV1` (`reactivity/notification.ts`) threads the marker from the event. The signature is
  still the reused commit cert bit-for-bit — for an invalidation it's the **invalidation's** commit cert.
- `ClusterMember.applyConsensusInvalidation` (`cluster-repo.ts`) now captures that cert via `buildCommitCert`
  over the consensus record, keyed by the deterministic `invalidationActionId(invalidatedActionId, disputeId)`
  (`commit-cert.ts`), gated on the `onCommitCertificate` sink (zero-cost when no reactivity is wired).

### Pull — durable committed-invalidated
- `BlockActionStatus['statuses']` (`network/struct.ts`) gains `'committed-invalidated'`.
- `NetworkTransactor.getStatus` refines an otherwise-`aborted` slot to `committed-invalidated` by consulting
  the owning collection's log via `Log.findInvalidation` (durable, restart-surviving). Only `aborted` slots
  are checked (cheap for ordinary commit/pending queries); `pending` is left alone (not mislabeled).

### Client reaction
- `Log.getInvalidationsFrom(startRev)` (`log.ts`) surfaces invalidations since the client's last sync
  (`getFrom` deliberately skips them).
- `Collection.updateInternal` drops reverted blocks from the read cache and replays pending work against the
  reverted base — invalidation collapses into the existing stale-read re-sync path.
- `coalesceInvalidatedActionIds` (`reactivity/notification.ts`) collapses cascade-child notifications by
  `invalidatedActionId` so a client re-derives each affected unit once.

## How to exercise / validate

- **db-core**: `yarn test` → 905 passing, 0 failing. Targeted:
  - `test/reactivity/wire.spec.ts` — marker round-trip + rejects non-boolean/non-string marker fields.
  - `test/reactivity/notification.spec.ts` — `buildNotificationV1` marker threading + `coalesceInvalidatedActionIds`.
  - `test/log-invalidation.spec.ts` — `getInvalidationsFrom` (since-rev, newest-first, empty when up to date).
  - `test/invalidation-client.spec.ts` — **the integration**: `getStatus` reports `committed-invalidated`
    from the durable log and survives a fresh transactor over the same store (node restart); a
    never-committed action stays plain `aborted`; `updateAndSync` drops the reverted block from cache so a
    re-read observes the reverted base.
- **db-p2p**: `yarn test` → 895 passing / 30 pending / 0 failing. Targeted:
  - `test/cohort-topic/invalidation-notification.spec.ts` — an invalidation notification carries the marker,
    reuses the invalidation's commit cert, **verifies with real Ed25519** against the tail-cohort membership
    cert exactly like a commit notification, and a forged threshold sig (and a sub-threshold cert) is rejected.
  - `test/cluster-invalidation.spec.ts` — the invalidation commit cert is captured under `invalidationActionId`
    (not the original action id) and is NOT captured when the certificate is rejected.
- Dependent builds clean: `quereus-plugin-optimystic`, `reference-peer` (the `BlockActionStatus` widening is
  additive — no exhaustive-switch consumer breaks).

## Where a reviewer should push hardest (honest gaps / weak spots)

1. **Node-side emit is NOT wired (the big one).** Nothing yet emits a `CollectionChangeEvent` with
   `invalidation: true` from a live node. The `InvalidationApplySink` (composition root, parent gap #1, gated
   on `tickets/fix/invalidation-certificate-arbitrator-binding.md`) must, when applying an invalidation,
   emit a change event with `invalidation: true`, `invalidatedActionId`, and `actionId =
   invalidationActionId(invalidatedActionId, disputeId)` so the bridge's cert extractor resolves the captured
   cert. Until then the **push** path is proven only at the assembler+verify level, not end-to-end through a
   `StorageRepo` change feed. The reactivity-mesh-harness has no `invalidate()` driver for the same reason —
   adding one needs the emit. Confirm the cert-capture key and the documented emit contract line up so the
   wiring ticket is a drop-in.
2. **`getStatus` opens a `Log` per aborted ref.** Correct and durable, but a status query for an aborted
   action now triggers extra chain reads over the network (cached per `(collection, action)` within one
   call). Faulty log reads are swallowed (best-effort → stays `aborted`, logged). Sanity-check the failure
   isolation and whether the cost is acceptable, or whether the answer should ride an extended block-state
   field instead.
3. **Client reaction is over-inclusive.** `updateInternal` replays pending work whenever ANY invalidation
   landed since the client's rev and there is pending work — it does not check the pending action's read set
   against the reverted blocks. Safe by design (over-invalidation just resubmits, never wrongly retains;
   matches the cascade re-evaluator's stance) but coarser than necessary. The `updateAndSync` test asserts
   the cache-drop (re-read ≠ stale value) but does not assert a full strong round-trip of "resubmit produces
   the reverted-base result" — strengthen if desired.
4. **`getInvalidationsFrom` adds a tail walk to every `update`.** Even with no invalidations it walks the log
   tail back to the client's rev. Could be folded into `getFrom`'s single pass; kept separate for clarity.
5. **Marker is hint-only by contract.** Verify nothing was made to depend on push delivery (a dropped/forged
   notification must lose the client nothing — the pull + authoritative re-read always recover). The forged
   and below-threshold rejection tests cover the crypto; double-check no code path treats the flag as
   authoritative.

## Key use cases (the spec's "Key tests", mapped to where they live)

- Connected subscriber gets an invalidation notification with the reused cert that verifies like a commit →
  `invalidation-notification.spec.ts`.
- Offline→online client gets `committed-invalidated` from `getStatus`, durable across restart →
  `invalidation-client.spec.ts`.
- Notification marks invalidation distinctly; generic subscriber still refreshes, invalidation-aware client
  drops + resubmits → marker tests in `notification.spec.ts` / `invalidation-notification.spec.ts` +
  `coalesceInvalidatedActionIds`.
- `updateAndSync` resubmits against the reverted base → `invalidation-client.spec.ts` (cache-drop assertion;
  see gap #3 on strength).
- Forged invalidation notification rejected → `invalidation-notification.spec.ts`.
