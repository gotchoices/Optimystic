## What changed

`RingSelector` constructor field is now `private readonly fretAdapter: ArachnodeFretAdapter` (was `_fretAdapter`, never read).

In `determineRing()` (`packages/db-p2p/src/storage/ring-selector.ts:37`), the hard-coded `avgBlockSize`/`estimatedTotalBlocks` constants are replaced with:

```ts
const ringStats = this.fretAdapter.getRingStats();
let estimatedTotalData: number;
if (ringStats.length > 0) {
  estimatedTotalData = ringStats.reduce(
    (sum, stat) => sum + stat.peerCount * stat.avgCapacity,
    0
  );
} else {
  // bootstrap fallback: 100KB × 1000 blocks ≈ 100MB
  const avgBlockSize = 100 * 1024;
  const estimatedTotalBlocks = 1000;
  estimatedTotalData = estimatedTotalBlocks * avgBlockSize;
}
if (estimatedTotalData <= 0) {
  estimatedTotalData = capacity.available; // pathological-zero guard → ring 0
}
```

The downstream `coverage = capacity.available / estimatedTotalData` and `ringDepth = ceil(-log2(coverage))` math is unchanged. `Math.max(0.001, coverage)` floor and `Math.min(ringDepth, 16)` cap remain. A doc comment now references `docs/arachnode.md` §"Capacity Management and Ring Adjustment".

`shouldTransition()` was not modified — it routes through `determineRing()` and benefits transparently.

`libp2p-node-base.ts:387` already passed a real adapter; no signature change needed there.

## Review checklist

- Confirm the formula `Σ (peerCount × avgCapacity)` is the right proxy for network demand vs. an alternative (e.g., max single-peer capacity, or median). The ticket settled on sum because it gives "my share of network spare capacity" which maps cleanly to keyspace partitioning. Document if the reviewer disagrees.
- Confirm bootstrap-fallback (`100MB`) constant is acceptable. It matches prior behavior exactly so first-boot is unchanged from before this ticket.
- Verify the pathological-zero guard (`estimatedTotalData <= 0` → fall back to `capacity.available`, yielding coverage = 1, ring 0) is appropriate. Alternative would be to return `-1` or use the bootstrap constant; chose ring 0 because a network of zero-available peers is degenerate and a node with capacity should claim the full keyspace.
- Confirm no need to change `shouldTransition()` — `currentRingDepth` already flows from `determineRing()`.
- Cross-ring test math: Ring 0 (1 × 1GB) + Ring 4 (8 × 100MB) = 1.8GB; 100MB / 1.8GB ≈ 0.0548; `ceil(-log2(0.0548)) = 5`.
- Single-ring test math: 4 × 100MB = 400MB; 100MB / 400MB = 0.25; `ceil(-log2(0.25)) = 2`.

## Use cases / scenarios

1. **Cold start** (no peers visible yet): `getRingStats()` returns `[]` → bootstrap fallback (~100MB target) → behavior identical to pre-ticket. Newly-joining nodes pick a sensible default ring without needing FRET to be populated.
2. **Established network, single ring**: ring 0 has N peers with average available `A`. New node with available `a` picks ring `ceil(-log2(a / (N×A)))`. Larger N → higher ring → finer partition.
3. **Multi-ring network**: aggregation includes capacity at all rings, including outer rings, so a node with modest capacity correctly chooses a deep ring.
4. **Capacity pressure on existing node**: as `capacity.used` rises, `available` falls, `coverage` shrinks, `determineRing()` returns deeper ring; `shouldTransition()` (driven by `usedPercent > moveOut`) signals 'out'.
5. **Network shrinks**: peers leave, `getRingStats()` aggregate falls, `coverage` grows, `determineRing()` returns shallower ring on next sample.
