# Review: invalidation certificate — bind votes to the legitimately-selected arbitrator set (#1)

This is **part 2 of 2** of the forgeable-invalidation-certificate fix and the critical binding. Part 1
(`invalidation-cert-target-binding-and-dedup`, now in `complete/`) closed cross-target replay (#2) and
vote-duplication (#3) and established the versioned vote payload. This ticket bumped that wire format to
**v3** and bound every counted vote to the legitimately-selected arbitrator set, so a peer that can mint
Ed25519 keypairs can no longer stuff a synthetic cohort and forge a passing certificate.

## What was implemented

### Wire format (v3) — `db-core/src/log/struct.ts`
- `ArbitrationVoteProof.version` bumped `'v2'` → `'v3'`. The signed vote payload is now
  `v3:${disputeId}:${vote}:${computedHash}:${targetHash}:${setHash}` — bound to BOTH the reversed
  transaction (`targetHash`, #2) and the arbitrator set (`setHash`, #1).
- `DisputeResolutionProof` gained three fields: `challengerPeerId`, `arbitratorSet` (the K selected
  peer-id strings), and `arbitratorSetSignature` (the challenger's Ed25519 sig over
  `(disputeId, target, arbitratorSet)`). All three are **required** — a pre-set-binding proof is rejected.
- Doc-comments updated to the **restored** unqualified "cannot forge" guarantee with the residual note.

### Verifier — `db-p2p/src/dispute/invalidation.ts`
`verifyInvalidationCertificate(proof, target, options?)` now enforces, in defense-in-depth layers:
- **Layer 1 (always):** structural presence of the binding; challenger signature valid over the
  recomputed `(targetHash, setHash)` (so a tampered set breaks the cert); every counted vote's
  `arbitratorPeerId ∈ arbitratorSet`; v3 vote payload (incl. `setHash`); part-1 dedup/equivocation; 2/3
  super-majority among the in-set deduped votes.
- **Layer 2 (best effort):** an injected `recomputeArbitratorSet` capability re-derives the eligible set
  from the member's topology; `{feasible:true, legitimate:false}` rejects even a layer-1-valid cert
  (closes a fully-malicious challenger). `{feasible:false}` → degradation.
- **Degradation (none-available):** accept on layer 1 alone and **log** (plus invoke `onUnanchored`) —
  never silent-accept, never false-reject. Interim posture until the trust-anchor chain lands.
- New exported helpers `computeArbitratorSetHash`, `voteSigningPayload`, `arbitratorSetSigningPayload`
  (shared by origination + verification so signed/verified preimages cannot drift); new types
  `ArbitratorSetBinding`, `ArbitratorSetRecompute`, `ArbitratorSetRecomputeContext`,
  `ArbitratorSetVerdict`, `UnanchoredAcceptanceInfo`, `VerifyCertificateOptions` (all re-exported from
  `dispute/index.ts`).

### Origination — `db-p2p/src/dispute/dispute-service.ts` + `types.ts`
- `initiateDispute` now selects arbitrators **first** so the challenge carries `arbitratorSet`
  (`DisputeChallenge.arbitratorSet`); each arbitrator folds `setHash` into the v3 vote it signs.
- `handleChallenge`/`makeVote` thread `setHash`; `maybeInvalidate` signs the set with the challenger key
  and builds the proof binding via the updated 3-arg `buildDisputeResolutionProof`.

### Cluster apply path — `db-p2p/src/cluster/cluster-repo.ts`
- `ClusterMember` gained an optional injected `recomputeArbitratorSet`; `applyConsensusInvalidation`
  threads it (plus an `onUnanchored` logger) into the verify call. This is the authoritative network-path
  gate (runs before the sink).

## Known gaps / honest flags for the reviewer (treat tests as a floor)

1. **The live FRET recompute (layer 2) is NOT wired at the composition root.** The injection point on
   `ClusterMember` and the cluster-path gate exist and are tested with a **mock** recompute, but
   `libp2p-node-base.ts` intentionally leaves `recomputeArbitratorSet` unset (there is a code comment
   there). Consequence: a *live* node today runs layer 1 + degradation accept-and-log; the
   fully-malicious-challenger vector is closed only where a recompute is injected (tests), not yet in
   production. This was deliberate — a naive FRET recompute without a churn-tolerance window would
   false-reject legitimate certs from late-joiners (a liveness regression). **Reviewer decision:** is the
   interim degradation posture acceptable until the trust-anchor ticket, or should a conservative live
   recompute be wired now? If the latter, it likely warrants a new `fix/`/`plan/` ticket (needs
   live-topology + churn-tolerance validation that is not unit-testable here).

2. **Layer 3 (trust anchor) is unimplemented** — depends on
   `tickets/plan/cohort-topic-membership-cert-trust-anchoring.md`. Until it lands, the
   no-recompute-no-anchor case is the documented residual. The code cross-references that plan ticket
   (verifier doc-comment + the libp2p-node-base comment) so the eventual hard-gate upgrade is
   discoverable.

3. **Challenger anchoring depth (layer 1).** The pure verifier validates `arbitratorSetSignature` against
   the embedded key in `challengerPeerId` but does **not** confirm `challengerPeerId ∈
   originalRecord.peers` (the `InvalidateRequest`/proof does not carry the original cohort). So layer 1
   alone does not stop a challenger that legitimately holds its own key — exactly why layers 2/3 exist.
   A cheap hardening worth a look: thread the committed transaction's cohort into the cluster-path verify
   and assert the challenger was a member (the applying member honored that commit cert). Not done here.

4. **`applyInvalidation`'s internal re-verify runs with no recompute (degradation).** It is defense in
   depth behind the network gate (`applyConsensusInvalidation` does pass the recompute). Cascade children
   reuse the root proof through `applyInvalidation`, so they verify at degradation strength — acceptable
   because the root was already gated at the cluster path, but confirm that reasoning holds for the
   cascade.

5. **No durable end-to-end origination/emit wiring yet** (unchanged from prior invalidation tickets): the
   composition-root wiring (`onInvalidate` → `applyInvalidation`, invalidation change-event emit) is still
   pending, so this binding is not yet on a live exploit path. Consistent with the §Durable Invalidation
   "Wiring status" note in `docs/right-is-right.md`.

## Suggested reviewer focus
- Re-derive the v3 signed-payload strings by hand and confirm origination (`makeVote`/`signArbitratorSet`)
  and verification (`voteSigningPayload`/`arbitratorSetSigningPayload`) cannot drift.
- Adversarially probe the layering order in `verifyInvalidationCertificate` (does an early-return ever
  skip a needed check? is the super-majority computed only over in-set deduped votes?).
- Decide the disposition of gap #1 (wire live recompute now vs. defer) and gap #3 (challenger ∈ cohort).
