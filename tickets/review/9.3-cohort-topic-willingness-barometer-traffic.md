# Review: willingness / barometer / topic-traffic

Implements the admission/load brain of the cohort-topic substrate: the **capacity barometer**
(per-tier log-bucketed load), per-member **willingness** (the admit decision), and the
**topic-traffic** counters (advisory flow signal on replies). All three are pure db-core logic over
injected ports (gossip view, registration store) — no FRET/libp2p imports.

## What landed

### Phase 1 — barometer (`load/barometer.ts`)
- `utilizationBucket(u)` = `clamp(7 + ⌊log₂ u⌋, 0, 7)` — each bucket a doubling, bucket 7 = at/over
  capacity, bucket 0 = idle/`u<1/64`. `u ≤ 0` / NaN → 0.
- `LoadBarometerState`: `observe(tier, utilization)`, `bucket`, `reading`, `loadBuckets()` (the
  4-entry gossip array), `isOverloaded(tier)` (= `bucket ≥ overloadBucket`, the early-promote signal
  the promotion ticket consumes), `loadWilling(tier)` (= `!isOverloaded`).
- `DEFAULT_OVERLOAD_BUCKET = 6` — single threshold for **both** willingness-flip and early-promote,
  matching `bucket_overload` in docs §Configuration and the simulator's `DEFAULT_OVERLOAD_BUCKET`.

### Phase 2 — willingness (`willingness.ts`)
- `WillingnessOutcome` = `accepted` | `unwilling_member {candidateMembers: Uint8Array[]}` |
  `unwilling_cohort {retryAfterMs}`. (Note: `candidateMembers` are **raw peer-id bytes**, the
  substrate's internal convention; the wire bridge base64url-encodes them into
  `RegisterReplyV1.candidateMembers`. The ticket's type sketch said `PeerId[]` — I used `Uint8Array`
  to match the rest of cohort-topic, see registration/* which uses raw bytes.)
- `createWillingnessCheck(deps).evaluate(reg, self, now)`:
  - live self-willing = `profile.willingTiers.has(tier) ∧ barometer.bucket(tier) < overloadBucket ∧
    primaryTopicCount(tier) < maxPrimaryTopicsPerTier`;
  - willing-member count = gossiped-willing siblings (from `CohortView`, self excluded) + (self live);
  - **quorum gate**: `willingCount < quorum` → `unwilling_cohort`; else self-willing → `accepted`;
    else → `unwilling_member(willing siblings)`. Mirrors the simulator's `classifyAdmission`/`admit`.
- Bit packing: `tierBit`, `packWillingnessBits`, `willingnessBitsHex`, `selfWillingnessBits`
  (gossiped coarse bit = profile ∧ load only; budget is the live-only finer gate). Bit `t` = tier `t`,
  matching the simulator and the existing gossip-view decode.
- Back-off: `backoffRetryMs(attempt)` = `min(⌊base·2^attempt⌋, cap)`, `base=1s factor=2 cap=60s`
  (`DEFAULT_BACKOFF_CONFIG`), folded back verbatim from `simulator/backoff.ts` settled params.

### Phase 3 — traffic (`traffic.ts`)
- `TrafficCounters`: `recordArrival` (fresh **and** renewal — combined), `recordQuery`,
  `publish(topicId, now)` (freezes windowed counts into the gossiped summary each round),
  `published`, `snapshot(topicId)`, `reset()` (epoch-change zero).
- `snapshot` is **gossip-derived**: own last-`publish`ed counts + siblings' last-gossiped
  `topicSummaries` (exact-integer sums), `directParticipants` from the replicated store; lags ≤ one
  round, no reply-time recompute.
- `attachTopicTraffic(reply, traffic)` sets `topicTraffic` **only** on `accepted`/`promoted`.
- `toCohortTopicSummary` builds the gossip-frame summary from published counts + caller tier/promo.

### Phase 4 — docs
- §Willingness, §Capacity barometer, §Topic traffic signal each got a "Resolved (decided)" /
  "Implementation" note pinning: 1-bit-per-tier willingness, `bucket_overload=6` shared threshold,
  and traffic-on-accepted+promoted-only.

## Known gaps / reviewer focus (treat tests as a floor)
- **Quorum default is invented.** docs §Tier ladder says "a quorum" but pins no number; I defaulted
  to strict majority `⌊k/2⌋+1` (= 9 for k=16), configurable via `WillingnessConfig.quorum`. Confirm
  this is the intended semantics vs. the cohort-signature threshold `k−x=14`. **Worth a second look.**
- **Per-tier primary-topic budget default is invented** (`maxPrimaryTopicsPerTier = 2048`, = the
  cohort-wide `topics_max`); the doc names the budget but pins no per-member-per-tier number.
- **`countWithin` assumes events arrive in ascending `now`** (monotonic clock). True in production;
  an out-of-order `recordArrival` would under-prune. Documented; verify the assumption holds for all
  callers, or sort.
- **`snapshot` aggregation interpretation.** docs say "the same per-topic entry the member last
  gossiped"; I read this as own-last-published **+** sibling summaries summed (cohort-wide exact
  integers), since sharded landing means one member sees ~1/k of arrivals. An alternative literal
  reading is own-summary-only. Confirm the aggregation is desired (matchmaking consumes
  `arrivalsPerMin` as a cohort rate).
- **No integration wiring yet.** `publish` cadence, calling `recordArrival` from the register/renew
  handlers, `reset` on the gossip bus's drift handler, and folding `selfWillingnessBits`/`loadBuckets`
  into the outbound `CohortGossipV1` are **not** wired — these belong to the transport-host /
  promotion / register-handler tickets. This ticket delivers the pure logic + the seams only.
- **`childCohortCount`** is sourced from gossip summaries (max) or an injected override; the promotion
  ticket owns the authoritative value.
