# Review: registration storage, primary/backup sharding, TTL renewal/failover

Implements ticket 8.7. New module `packages/db-core/src/cohort-topic/registration/` plus two
specs. `yarn build` and `yarn test` are green for db-core (**402 passing**, 25 new). All logic is
db-core-pure (no FRET/libp2p; the `no-fret-import` guard still passes); every transport is injected.

## What was built

- **`types.ts`** — `RegistrationRecord`, the `RegistrationStore` interface, TTL constants
  (`DEFAULT_TTL_MS=90000`, `EDGE_TTL_MS=60000`, `MAX_PING_FAILURES=3`, `pingIntervalMs(ttl)=⌊ttl/3⌋`).
  **Decision:** peer ids are raw `Uint8Array`, not a new `PeerId` alias — `db-core/src/network/types.ts`
  already exports a structural `PeerId`, and a second one re-exported through the barrel would be an
  ambiguous export. This matches the substrate's existing `PeerRef.id: Uint8Array` convention.
- **`store.ts`** — in-memory `RegistrationStore`, doubly indexed (topic → participant maps).
  `evictStale` is strict `now − lastPing > ttl`. Added `listAll()` to the interface (the handoff
  inventory pass needs to enumerate all records) — this is the one extension beyond the ticket's
  literal interface.
- **`sharding.ts`** — `createSlotAssigner(hash).assignSlots(participantId, cohortEpoch, members)`:
  ascending member sort, `H(participantId ‖ cohortEpoch) mod k` (full-digest MSB-first mod, no
  bigint), wrap-around backups capped at `k − 1`.
- **`renewal.ts`** — participant `pingLoop`/`reattach` (3-fail → promote `backups[0]`, then next
  backup, then `relookup`); cohort `onRenew` (touch+gossip / `primary_moved`) and `sweepStale`.
- **`handoff.ts`** — per-rotation state machine: `start` (inventory broadcast + mark dual-serve),
  `onInventory` (pull records now assigned here, re-stamp `primary`/`backups`, ack), `onPull`,
  `onAck`, `isServing`.

## Resolved open questions (grounded, as the ticket required)

1. **Dual-serve until ack** — the previous primary keeps serving until the new primary acks the
   pulled record. Implemented in the handoff (`dualServing` set, cleared on `onAck`) **and wired
   into `onRenew`** via an optional `isServing` predicate so the old primary actually keeps serving
   during the window instead of redirecting (see registration.spec "keeps serving a moved record…").
2. **Lazy `cohortEpoch` refresh on backup failover** — promotion of `backups[0]` deliberately does
   *not* touch the epoch hint; a later `primary_moved` reply refreshes it. The participant tracks a
   separate `cohortEpochHint` (it is not a field on `RegistrationRecord`) so this is observable.

Both are annotated as resolved in `docs/cohort-topic.md`.

## Use cases to validate the review against

- **Sharding determinism / distribution** (sharding.spec): same `(participant, epoch, members)` →
  same primary+backups regardless of input order; epoch change reshuffles; 16k participants over 16
  members land within ±30% of the mean and use every member; k=2→1 backup, k=1→0 backups; wrap-around.
- **Store** (registration.spec): dual indexing, put-replaces, delete-prunes-empty-topic, `evictStale`
  boundary is strict-greater-than per-record-ttl.
- **Renewal participant**: success resets failures; exactly 3 consecutive primary failures promote
  `backups[0]` (and only `backups[0]`, not `backups[1]`); all-fail → one `relookup`; epoch hint
  stays put at failover and only moves on a subsequent `primary_moved`.
- **Renewal cohort side**: touch+gossip on the computed primary; `unknown_registration`;
  `primary_moved` when self is no longer primary; dual-serve override; `sweepStale` evicts+gossips.
- **Handoff**: 40 records survive a full epoch rotation with each landing on its new primary (no
  loss); old primary dual-serves until ack then stops; no-op when the slot doesn't move.

## Known gaps / things to probe (tests are a floor, not a ceiling)

- **No cross-member replication.** This ticket is the *local* store + deterministic functions only.
  `RegistrationRecord` is replicated across the `~k` members by cohort gossip in a later ticket; the
  `RenewalGossip`/`HandoffTransport`/`RenewalParticipantTransport` interfaces are the seams for it.
  The "backup verifies it sees the record in its local replica" step is therefore modelled as the
  reattach reply (`ok` vs `unknown_registration`); the actual replica check lands with replication.
- **No signature verification.** `onRenew` trusts the decoded `RenewV1`; signature checking is a
  separate layer. A reviewer should confirm that's an acceptable boundary here.
- **No FRET/libp2p binding.** db-p2p must implement the injected transports; not in scope.
- **`pingLoop` returns `Promise<void>`** (the ticket interface wrote `void`). Changed so a scheduler
  / test can await one tick; there is no internal timer (tess forbids background loops and the
  cadence is `pingIntervalMs`, exposed for an external scheduler). Confirm this is the intended shape.
- **Handoff concurrency / partial failure.** The state machine is tested on a synchronous in-memory
  harness. Real gossip is lossy/reordered: a dropped ack leaves the old primary dual-serving
  indefinitely (no timeout yet), and a pull that fails mid-flight is retried only on the next
  inventory. Worth a hard look for the replication ticket.
- **Stale `appState` on pull.** The handoff moves the record as-is and re-stamps only
  `primary`/`backups`; `appState`/`lastPing` carry over from the previous holder unchanged.
