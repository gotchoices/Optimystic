# Review — Reactivity rotation: resume↔handoff coordination + `RotationRedirectV1` recover codec (db-core)

This is the **pure db-core** half of wiring the tail-rotation lifecycle onto a live node. Two seams were
opened for the follow-on db-p2p tickets (`reactivity-rotation-recover-redirect-drain`,
`reactivity-rotation-rereg-scheduler`, `reactivity-rotation-host-wiring-e2e`). No host/transport code was
touched here.

## What was built

### A. Resume serves from the inherited (cross-rotation) checkpoint — `resume.ts`

- `classifyResume` gained an optional trailing `inherited?: CheckpointSummary` parameter (kept last so all
  existing positional callers still compile). New decision order: `tail_rotated` → `backfill` →
  `checkpoint_window` (rolling) → **`checkpoint_window` (inherited)** → `out_of_window`. The inherited branch
  fires only when the rolling checkpoint misses but `checkpointCovers(inherited, fromRevision)` (imported from
  `rotation.ts`) is true.
- `ResumeServingDeps` gained an optional `inheritedCheckpoint?: CheckpointSummary` (the new tail's
  `PushState.inheritedCheckpoint`).
- `serveResume`'s `checkpoint_window` branch now picks the rolling summary when `deps.checkpoint.covers(fromRevision)`,
  else the inherited summary; both serve the live ring's `recentEntries` (identical reply shape). **Precedence:
  rolling wins when both cover** — it is the fresher, narrower window.
- `applyResumeReply` is **unchanged**: an inherited checkpoint is a `CheckpointSummary` like any other, so the
  existing endpoint-verification and non-abutting guards apply verbatim.

### B. `RotationRedirectV1` codec on the recover reply envelope — `recover.ts` + `rotation.ts`

- `validateRotationRedirectV1` was added in `rotation.ts` next to the `RotationRedirectV1` interface (uses the
  shared `wire-validate.ts` helpers): requires `v === 1`, `result === "rotated"`, base64url
  `newTailId`/`newTopicId`, non-negative-integer `effectiveAtRevision`. It is exported from `index.ts`
  **transitively** via `export * from "./rotation.js"` (no explicit index edit needed; the test imports it
  from `index.js` and resolves — confirmed green).
- `RecoverKind` stays `"backfill" | "resume"` for **requests**. A new `RecoverReplyKind = RecoverKind | "rotated"`
  widens **replies** only. `RecoverReplyV1` gained `kind: RecoverReplyKind` and an optional `rotated?:
  RotationRedirectV1`, present iff `kind === "rotated"`.
- `validateRecoverReplyV1` handles the `"rotated"` branch via a new `reqReplyKind` narrower and rejects stray
  `backfillReply`/`resumeReply` on it (mirrors the existing `rejectStrayBranch` discipline both ways). Encode/decode
  round-trip it; replies carry no signature (unauthenticated routing hint — trust comes from re-walking the new tree).
- **Forward-compat is fail-closed**: an old decoder's `reqKind`/reply narrower rejects `"rotated"`, so a peer
  predating this change treats the reply as malformed and chain-reads (safe). Documented in the JSDoc.

### Test use-cases added (the floor — extend, don't trust as a ceiling)

- `resume.spec.ts` (new `inherited (cross-rotation) checkpoint` block): inherited classification + serve (a
  `fromRevision` below both ring and rolling checkpoint now served, vs `out_of_window` without inherited);
  rolling-wins precedence (incl. the rolling-low-edge case); ring-wins (backfill above ring low); inherited-undefined
  unchanged; non-abutting inherited summary still → `checkpoint_untrusted` subscriber-side.
- `recover.spec.ts`: `"rotated"` reply round-trip; stray `resumeReply`/`backfillReply` on a `"rotated"` reply →
  `failWire`; stray `rotated` on a `"resume"`/`"backfill"` reply → `failWire`; missing `rotated` body; malformed
  `newTopicId`; negative `effectiveAtRevision`.
- `rotation.spec.ts` (new `validateRotationRedirectV1` block): happy path + non-object / wrong-version /
  wrong-result / malformed-id / missing-field / negative & non-integer-revision rejections.

## Known gaps / honest scope boundaries (for the reviewer)

- **The seams are not yet consumed.** `db-p2p`'s `serveResumeReply` (recover-transport.ts) does **not** pass
  `inheritedCheckpoint` from the live `PushState`, and nothing yet emits a `kind:"rotated"` recover reply over
  the wire. So a cross-rotation resume reaching a real new-tail node over libp2p still falls to `out_of_window`,
  and the drain redirect is not yet framed end-to-end. **This is intentional** — it is the explicit follow-on
  work of the three db-p2p tickets named above (esp. `reactivity-rotation-recover-redirect-drain`). The db-core
  contract (classifier reads `inherited`; `validateRotationRedirectV1`; reply `kind:"rotated"`) is what lands here.
- **`checkpointCovers` import direction.** `resume.ts` now imports `checkpointCovers` from `rotation.ts`. Verified
  no import cycle (rotation.ts imports neither resume nor recover; index.ts evaluates resume→recover→rotation and
  `checkpointCovers` is a hoisted function decl, so no TDZ). Worth a second look if the reviewer is cycle-averse —
  an alternative would be relocating `checkpointCovers` to `checkpoint.ts`, but the ticket specified the rotation.ts source.
- **No new behavior in `applyResumeReply`.** The inherited path deliberately reuses the existing subscriber guards;
  the reviewer may want to confirm there is no inherited-specific edge (e.g. an inherited summary that *abuts* but
  carries a stale digest) that warrants distinct handling — current design treats it identically to a rolling summary.
- **docs/reactivity.md** updated in §Resume (new "Cross-rotation resume" note + `CheckpointWindow`/`OutOfWindow`
  bullets) and §Tail rotation steps 2 & 5. The §Wire formats section was **not** given a standalone
  `RotationRedirectV1` schema block — the redirect shape is documented inline in step 2; consider whether a formal
  wire-format entry is wanted.
