# Reactivity recovery: backfill RPC, parent checkpoints, resume protocol ‚Äî review

Implements recovery beyond the live notification stream: a short gap served from the replay ring
(**Backfill**), a long sleep served from a parent **Checkpoint**, out-of-window (**chain read**), and
across a tail rotation (**TailRotated**). Built on the `PushState` / `replayBuffer` substrate from
`reactivity-origination-replay-delivery` and consumes the `requestBackfill` seam that ticket defined.

`yarn build` + `yarn test` are green for **db-core** (763 passing) and **db-p2p** (643 passing, 0 failing).

## What landed

The on-disk draft files from commit `acf00eb` were finished, exported, and tested (they were
**unexported and untested** starting points, not completed work).

**Phase 1 ‚Äî Backfill** (`backfill.ts`)
- `BackfillV1` / `BackfillReplyV1` codecs + `serveBackfill` (cohort side): returns the **intersection**
  of the requested range with the replay ring and reports `available: { fromRevision, toRevision }`.
- `createBackfillRequester` (subscriber side): the `requestBackfill(from, to)` seam ‚Üî RPC ‚Äî signs, sends
  via an injected transport, replays the reply entries through `subscriber.onNotification` (so they are
  verified + deduped), and fires `onUnderflow` when `available.fromRevision > from` (lag fell past the ring
  ‚Üí escalate to checkpoint/chain).

**Phase 2 ‚Äî Checkpoints** (`checkpoint.ts`, `push-state.ts`, `replay-buffer.ts`)
- `RollingCheckpoint` fed by **replay-ring eviction**: `createReplayBuffer(capacity, onEvict)` now hands
  each evicted entry to `RollingCheckpoint.retire`, so revisions leaving the `W`-deep ring roll into the
  `W_checkpoint`-span summary sitting **immediately below** it (the doc's authoritative stacked semantics ‚Äî
  total one-RT range `W + W_checkpoint`). `PushState.parentCheckpoint` is now a getter over the live rolling
  checkpoint (the old `{fromRevision,toRevision}` stub interface was removed; the rich `CheckpointSummary`
  from `checkpoint.ts` is the single source).
- `verifyCheckpointEndpoints`: runs the standard notification verifier over **both** full bracketing
  endpoint notifications + checks their revisions equal `from`/`to`. A forged/tampered endpoint is rejected.
- `defaultDigestFold` (deterministic running hash; per-collection override) and the `mergedDelta` policy
  (omit when coalesced size > `delta_max`, omit when `delta_max == 0`, never split).

**Phase 3 ‚Äî Resume** (`resume.ts`, `subscriber.ts`, db-p2p `subscription-manager.ts`)
- `ResumeV1` / `ResumeReplyV1` codecs; `classifyResume` + `serveResume` ‚Äî the four-way classification over a
  `PushState` snapshot (TailRotated checked **first**, then Backfill / CheckpointWindow / OutOfWindow by lag
  against the stacked windows).
- `applyResumeReply` (subscriber side): backfill/checkpoint entries re-enter through `onNotification`; a
  **verified** checkpoint applies its digest then advances the contiguity head via the new
  `ReactivitySubscriber.rebaseline(toRevision)` before replaying `recentEntries` (so the digest-covered range
  doesn't trigger a spurious gap; recent entries at/below the head dedupe); an **untrusted** checkpoint or
  out-of-window escalates to `onChainRead`; tail_rotated escalates to `onTailRotated`.
- `createStickyCohortHintCache` (Edge one-RT-after-flap); db-p2p `ReactivitySubscriptionManager` gained a
  `resume()` driver + backfill-seam wiring (both behind injected transports/signers) and invalidates the
  sticky cache on `tail_rotated`.

**Phase 4 ‚Äî Docs** (`docs/reactivity.md`): resolved-design-question notes + Implemented callouts; wire
formats aligned to the implementation (see the resolved divergence below).

## Resolved design questions (verify these are the right calls)

1. **`mergedDigest` ‚Üí system-level deterministic fold, per-collection override.** `NotificationV1.digest` is
   the per-revision commit-vote signed payload (already system-level + identical across members), so the
   default fold is a deterministic running hash that converges under gossip. A hint only ‚Äî never
   cryptographically trusted; the bracketing endpoints are the anchor.
2. **`mergedDelta` vs `delta_max` ‚Üí omit when oversize, never split.**
3. **Resume-on-rotation ‚Üí keep the simple full walk.** The Edge sticky `cohortHint` cache shortcuts the common
   no-rotation flap; pre-dial toward the pre-announced successor coord is deferred to the rotation ticket
   (needs rotation-side announce state).
4. **Wire-format divergence resolved (doc + code now agree):** the checkpoint's bracketing endpoints are the
   **full endpoint `NotificationV1`s** (`bracketingEntries`), **not** bare `bracketingSigs`. Rationale: a bare
   threshold signature is not independently verifiable ‚Äî verifying an endpoint is a real committed revision
   needs the signed payload + signers, which only the full notification carries. The doc's `CheckpointSummary`
   and `ResumeReplyV1.checkpoint` were updated to match. **Reviewer: confirm you accept this divergence from
   the original ticket sketch.** Also note `BackfillReplyV1.available` uses `{ fromRevision, toRevision }`
   (not the ticket's `{ from, to }` sketch) ‚Äî doc + codec agree.

## Known gaps / honest caveats for the reviewer

- **No concrete on-wire RPC handler.** db-core provides `serveBackfill` / `serveResume` (pure) and the
  requester/`resume()` drivers; the db-p2p manager wires them **given an injected transport + signer**. There
  is **no** libp2p reactivity-application-protocol endpoint yet that decodes a `BackfillV1`/`ResumeV1` frame
  off the wire and calls the server, nor a dial path. That transport/protocol is follow-on work (coordinate
  with the rotation ticket). The injected-seam shape is deliberate so it stays unit-testable.
- **Request signing is injected, not canonical.** `BackfillV1`/`ResumeV1` take a `sign(unsigned) => string`
  hook; there is no canonical signing-payload helper (cf. cohort-topic's `registerSigningPayload`) and the
  requests are not yet wrapped in a cohort-topic replay-protection envelope (the doc says replay protection is
  the cohort-topic layer's job ‚Äî that wrapping is unwired here). Tests use dummy signers.
- **Checkpoint is per-member, not gossiped.** Each member rebuilds its rolling checkpoint deterministically
  from its own ring evictions (works because the ring is gossiped and the fold is deterministic), but the
  checkpoint is **not** in `PushStateGossipV1`. A member that never observed the lower revisions won't hold
  them in its checkpoint. Confirm this is acceptable vs. gossiping the summary.
- **`rotationRevision` / current-tail are caller-supplied** to `serveResume` (the rotation lifecycle owns the
  real values). The staleness check is `req.latestKnownTailId !== currentTailId`; coordinate the source of
  `currentTailId` with `reactivity-rotation-backpressure-policy`.
- **`W_checkpoint` adaptive hook** (`resolveWCheckpoint`) exists but isn't wired to a live cps measurement
  (simulator fold-back pending; default 4096 sourced from the shared config table).
- **Backfill seam is fire-and-forget** in the manager: a transport failure surfaces as an unhandled rejection
  rather than being retried (acceptable for a hint-only path, but confirm).

## Suggested review focus

- The **stacked-window classification** boundary conditions in `classifyResume` (empty ring, subscriber
  already current, exact edges `from == ringLow`, `from == checkpoint.from`).
- The **rebaseline** correctness: does advancing the contiguity head past a verified checkpoint ever skip a
  real un-summarized revision? (Checkpoint `toRevision` is `ringLow ‚àí 1` by construction, so `recentEntries`
  abut it ‚Äî verify that invariant holds when the ring/checkpoint are fed irregularly.)
- The **forged-endpoint rejection** path ‚Äî is there any code path where an unverified checkpoint advances
  subscriber state?
- The **CheckpointSummary single-source reconciliation** (push-state stub removed) ‚Äî confirm no external
  consumer depended on the old `{fromRevision,toRevision}` shape.
