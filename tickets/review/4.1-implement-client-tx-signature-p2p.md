## What this ticket did

The db-core half (`implement-client-tx-signature-core`) defined the signer/verifier **ports** and the
canonical signing payload but left them un-backed — so transactions were neither signed nor verified.
This ticket supplies the real libp2p Ed25519 backing and the rollout switch:

- **Signing (client side).** `collection-factory.ts` gained `getSigner(options)`, which returns a
  `TransactionSigner` bound to the node's libp2p identity key: `async payload =>
  bytesToB64url(await signPeer(nodeKey, payload))`. `txn-bridge.ts` passes it into
  `TransactionSession.create(...)` (6th arg). When no node/key exists (legacy `local`/`test`/`mesh-test`
  transactors, or an injected node) it returns `undefined` and the session is unsigned — unchanged
  behavior. Signing is always injected when a key exists; it only *adds* a field.

- **Key exposure.** libp2p does not surface the private key on its public interface, so
  `libp2p-node-base.ts` now attaches it as `(node as any).peerPrivateKey = nodePrivateKey`, on the same
  `(node as any).*` surface it already uses for `coordinatedRepo`/`keyNetwork`/etc.

- **Verifying + enforcement (node side).** `quereus-validator.ts` gained a
  `requireClientSignature?: boolean` option (**default false**). When `true`, it constructs the
  `ClientSignatureVerifier` port — `(peerId, payload, sig) => { try { return verifyPeerSig(peerId,
  payload, b64urlToBytes(sig)); } catch { return false; } }` — and passes it as the 4th
  `TransactionValidator` arg, so `validate()` rejects missing (`Missing client signature`) or invalid
  (`Invalid client signature`) signatures at pend. When `false`, the port is omitted: unsigned AND
  signed transactions both pass the signature step (migration / single-node-dev posture).

- **Signer-failure safety.** `session.commit()` now wraps the signer call in try/catch and returns a
  clean `{ success: false, error: 'Failed to sign transaction: …' }` — the transaction object is local
  and `coordinator.commit` is never reached, so a failed sign can never leak a partial unsigned tx into
  consensus.

**Identity model (settled, do not re-open):** for an Ed25519 libp2p identity, `stamp.peerId` string
embeds the raw public key, so a verifier derives the client key straight from it — no key distribution.
Non-Ed25519 identities are out of scope: `verifyPeerSig` returns `false` for them, so under enforcement
they are rejected.

## Known gaps — treat tests as a floor, not a finish line

- **No live cross-node enforcement test.** The plugin mesh tests run with enforcement OFF (the default),
  so signed transactions flow but are never *verified across the wire*. The enforcement path is tested at
  the validator seam (unit/integration), NOT through the live cluster PEND path. There is currently **no
  production wiring that flips `requireClientSignature: true`** — `createQuereusValidator` is only called
  from tests today, and the mesh harness does not thread the flag. A high-value follow-on is a mesh test
  that constructs the validator with `requireClientSignature: true` and asserts an unsigned client is
  rejected at pend across nodes. This is a real gap, not covered.
- **Recovery re-verification (Theorem 13 anchor) only JSON-deep.** The ticket said "a round-trip
  assertion through the transactor's serialization is enough here", and that is what landed (JSON
  round-trip + a real `session.commit` via `TestTransactor`). Whether the `signature` field actually
  survives the **on-disk cluster record / transaction log** serialization (db-p2p cluster-repo) so a
  future recovery reader can re-run `verifyPeerSig` is **not** verified — no recovery path exists yet.
  Confirm when recovery lands.
- **Signer bound to `(node as any).peerPrivateKey`.** A node injected via `registerLibp2pNode` that was
  not built by `createLibp2pNode` carries no key → signing silently disables for it. Harmless today
  (enforcement off), flagged at the code site with a `NOTE:`.

## Review findings

- **Tripwire (parked as `NOTE:` at the code site, `collection-factory.ts` getSigner):** a node injected
  without going through `createLibp2pNode` has no `peerPrivateKey`, so signing silently disables. Fine
  now (enforcement is off by default); becomes work only if a deployment enforces verification against
  such an injected node.
- **Tripwire (documented in the ticket, not a code site):** non-Ed25519 identities are rejected under
  enforcement (`verifyPeerSig` → false). This matches the cohort-topic substrate's standing Ed25519
  assumption; only becomes work if the project ever needs non-Ed25519 client identities.
- Stamp-field tampering (expiration/engineId/schemaHash mutated while holding stamp.id + signature) is
  handled by the prereq `bind-stamp-fields-to-tx-signature`'s step-0.0 integrity check, with existing
  db-core tests — not re-covered here.
