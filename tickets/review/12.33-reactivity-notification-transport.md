# Reactivity notification transport — live node wiring (review handoff)

## What landed

The `cohortTopic`-enabled production node (`libp2p-node-base.ts`, the `if (cohortEnabled)` block) now composes
the reactivity transport stack so origination → fan-out → inbound delivery → push-state gossip are **live** for
every collection. Built on top of the already-firing `attachCohortChangeBridge` (which installs the membership
gate + cert extractor) and the now-exposed `host`:

1. **Notify transport** — `new Libp2pReactivityNotifyTransport(node, { selfPeerId })`.
2. **Forwarder host** — `ReactivityForwarderHost` with:
   - `profile` from `host.profile` (newly exposed); `pushStateInit` deltaMax from `reactivityNodePolicy(profile)`.
   - `verifierFor` = `createNotificationVerifier({ verifier: host.service.verifier(), tier: T3 })`.
   - `directSubscribers(topicId)` = `host.registry.findServing(topicId, 0)?.records(topicId)` via
     `reactivityDirectSubscribers` (filters to reactivity `appState`, maps `participantId` → peer-id string).
   - `resolveChildPrimary(ref)` = FRET-nearest member of `b64urlToBytes(ref.coord)` as a peer-id string (no-op
     until the parent/child link lands; wired anyway).
   - `deliverLocal` → the node-level `ReactivitySubscriberRegistry`.
3. **Origination emit** — `ReactivityOriginationManager.install()` sets `service.onLocalCommit`; `resolveContext`
   derives `tailId = reactivityTailBytes(event.tailId)`, `emit` binds to `forwarderHost.ingest(reactivityNotificationTopicId(n), n)`.
4. **Subscriber registry** — new `ReactivitySubscriberRegistry` (`reactivity/subscriber-registry.ts`),
   keyed by topicId-b64, exposed as `node.reactivitySubscribers`. Inbound notify frames route
   `notify.onNotification → forwarderHost.onInbound → deliverLocal → registry.deliver`.
5. **PushState gossip** — `ReactivityPushStateGossipDriver` over `host.gossipTransport` (newly exposed
   `broadcastOver`), `liveCollections` from the new `forwarderHost.livePushStates()`, `isCohortMember` from
   FRET assembly. New `registerPushStateGossipHandler` (in `push-state-gossip.ts`, mirrors `registerNotifyHandler`).
6. **Teardown** — the existing `node.stop` wrapper now stops the gossip timer, unsubscribes inbound notify,
   `node.unhandle(reactivityProtocolList(...))`, then the change-bridge unsub + `host.stop()`, then previousStop.

Shared encoding: `reactivityTailBytes` moved to `reactivity/topic-bytes.ts` (single source of truth); the gate
re-exports it for back-compat. `ReactivitySubscriptionManager.tailIdAtAttach` JSDoc now documents the contract.

## DEVIATION from the ticket spec (please verify)

The ticket's pseudo-code said `host.registry.findServing(topicId, T3)`. **That is wrong and I used tree tier 0.**
`findServing(topicId, treeTier)` matches on the engine's **tree tier** (the coord depth `d`), not the capacity
tier. Reactivity's forwarder cohort sits at `coord_0` ⇒ tree tier **0**; T3 (=3) is the *capacity* tier (it gates
willingness + the verifier). Passing `T3` would match no engine (`engine.treeTier === 3` is never true for a
reactivity cohort) → silent no-subscribers — exactly the "green tests, dead feature" failure mode the ticket warns
about, one level up. Evidence: `reactivity-mesh-harness.ts:432` reads reactivity records off the tree-tier-0
deciding engine; `substrate-real-libp2p.integration.spec.ts` registers `tier:2`/`tier:3` participants at
`treeTier:0`. The constant `REACTIVITY_FORWARDER_TREE_TIER = 0` in `libp2p-node-base.ts` documents this. **Reviewer:
confirm this reading** — it is load-bearing; if reactivity registrations could ever land at a deeper tree tier
(multi-tier fan-out, which is post-`cohort-topic-parent-child-link`), `findServing(_, 0)` would miss them.

The ticket also said the `directSubscribers` adapter might still emit base64url; it already emits peer-id strings
(`reactivityDirectSubscribers` line 96, `bytesToPeerIdString`), so I reused it directly rather than rewriting.

## Honest gaps for the reviewer

- **Inbound forwarder-role gating (confirm intent):** every inbound notify runs `forwarderHost.onInbound`
  unconditionally. A Core node that receives a notify for a topic it does **not** serve will instantiate a
  PushState and fan out to an empty subscriber set (harmless + cached, but wasteful). The host self-gates only on
  the Edge policy + empty `directSubscribers`. Left as-is (matches the forwarder-host design); flag if you want
  inbound routed to `onInbound` only for served topics.
- **`push-state-gossip` purity:** the driver class stays libp2p-free (unit-testable with a fake transport); the new
  co-located `registerPushStateGossipHandler` imports libp2p (mirrors `registerNotifyHandler` in notify-transport).
  Module doc updated to say so. Reviewer may prefer the handler live elsewhere.
- **PushState-gossip frame peer-sig envelope signing is deferred** — the `isCohortMember` gate authenticates by
  cohort membership around the frame's coord, but per-frame signature verification is the `reactivity-pushstate-gossip`
  hardening backlog item. Frames are otherwise self-verifying (the merged entries carry their own threshold sig).
- **Child-cohort fan-out** is wired but exercised only synthetically until `cohort-topic-parent-child-link` populates
  `PushState.childCohorts` (single-tier-0 reach today). `resolveChildPrimary` returns a peer-id string ready for it.
- **Backpressure model** unchanged from `reactivity-forwarder-host`: `fanOut` enqueues then drains inline +
  fire-and-forget sends, so drop-oldest is only reachable under a pre-full queue. Genuine in-flight backpressure
  (a per-subscriber async delivery worker) is a follow-on redesign, not this ticket.
- **Quereus `Database.watch` → manager bridge** remains the backlog `optimystic-network-reactive-watch-integration-test`:
  this ticket delivers the node-level subscriber registry + socket routing it will plug into, NOT the bridge that
  *constructs* a manager from a watch. The integration test constructs the subscriber directly against the registry
  to prove the socket path.
- **Protocol namespacing:** reactivity uses `DEFAULT_REACTIVITY_PROTOCOLS` (network-agnostic), matching how the
  production cohort-topic host uses `DEFAULT_COHORT_TOPIC_PROTOCOLS`. If a named network should namespace reactivity
  protocols, that would need to thread through both families together — out of scope here, flag if desired.
- **Gated test is real-network timing-sensitive:** the integration test relies on the suite's `before` mesh
  converging (bounded polling, generous timeouts). It passed locally at N=4; at larger N or under load it inherits
  the same real-TCP/FRET stabilization tolerances as its sibling tests in that file.
