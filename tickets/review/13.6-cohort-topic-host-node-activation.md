# Review: cohort-topic host + change-bridge production-node activation

Flips reactivity origination from **dormant** to **live** on opt-in nodes. When
`NodeOptions.cohortTopic.enabled` is set, `createLibp2pNodeBase`:
1. creates a `CommitCertStore` early and composes its `put` into the `onCommitCertificate` sink threaded
   to `clusterMember(...)` (so the consensus cert is retained synchronously before the change event emits);
2. post-assembly (after `node.start()` + FRET) constructs the host via `createCohortTopicHost(node, fret,
   { privateKey: nodePrivateKey, wantK })`, exposed as `node.cohortTopicHost`;
3. builds `selfIsCohortMember(event)` over real FRET membership for `coord_0(H(event.tailId ‖
   "reactivity"))` (db-core default hashes; same `wantK` as the host);
4. installs the bridge as `node.blockChangeNotifier` via `attachCohortChangeBridge`, replacing the bare
   `storageRepo` notifier — so origination is live for ALL collections (the Quereus collection-factory
   captures that notifier once);
5. wraps `node.stop` to `unsubscribe()` + `host.stop()` before transports close.

A node **without** `cohortTopic.enabled` is byte-identical to today: bare `blockChangeNotifier =
storageRepo`, no host, no cert store. Activation is **hard-fail** when enabled but FRET is missing / the
host throws (operator opted in).

## Decisions worth an adversarial look

- **`resolveFretEngine` (libp2p-node-base.ts).** `node.services.fret` is the `Libp2pFretService` *wrapper*,
  which re-exports only a subset (`assembleCohort`, `routeAct`) and does **not** expose `setActivityHandler`
  that `createCohortTopicHost` needs (the host had only ever been tested against the raw engine / a mock).
  I reach the engine via the wrapper's lazy `ensure()` accessor (already initialized after `node.start()`;
  engine + wrapper share one store). This relies on a p2p-fret-internal method — review whether a cleaner
  public accessor (upstream in p2p-fret, or a typed adapter) is warranted, or whether the host should
  instead accept the wrapper subset.
- **Pinned `BlockId → tail bytes` encoding (`reactivityTailBytes`).** `event.tailId` is a `BlockId` string;
  the gate feeds `reactivityTopicId` `new TextEncoder().encode(tailId)` (the synchronous convention used by
  `cluster/client.ts` / `repo/client.ts`). db-core's async `blockIdToBytes` is deliberately NOT used (it
  `sha256`s first → double-hash + async). **Load-bearing:** the subscriber-side reactivity transport (ticket
  `12.3-reactivity-notification-transport`, owned elsewhere) MUST adopt the same encoding when deriving its
  `tailIdAtAttach` bytes from a `BlockId`, or origination and subscription resolve different coords and
  origination silently never reaches subscribers. Only the origination side exists today — confirm the
  subscriber wiring lands on the same encoding.
- **wantK / hash alignment.** Gate and host both use `cohortWantK` (one resolution) and db-core default
  `createRingHash()` / `createReactivityTopicAnchor()` / `createTierAddressing` — verified byte-identical to
  the host's internal `new RingHash()` and the subscriber anchor (and `coord_0` is fan-out independent). A
  future change to the host's default `wantK` or ring hash must update the gate in lockstep.
- **Gate extracted to its own module** (not inlined in node-base as the ticket sketched) so the
  encoding + coord derivation are unit-testable and reusable by the future origination wiring. Reasonable,
  but confirm you agree with the seam location.

## ⚠️ The honest gap the reviewer must weigh: solo node can't produce a consensus cert

The headline "probe fires once with a cert whose thresholdSig/signers match the consensus cert
byte-for-byte" is **NOT** provable on a single node, and the integration test does not claim it.
`CoordinatorRepo.commit` short-circuits consensus when the cluster is `<= 1` peer (`peerCount <= 1` →
direct `storageRepo.commit`), so `applyConsensusOperation`/`onCommitCertificate` never runs and **no
authoritative cert exists** on a solo node. The bridge therefore correctly cert-gates origination to a
**no-op** there (never fabricates an unsigned cert).

What the solo integration test DOES prove (all green):
- host constructed + exposed; bridge replaces the bare notifier (no `onAnyCollectionChange` on the decorator);
- a real `Tree.replace` commit emits a change event carrying a **real tail id** (the seam threading works
  through the production transactor — this is the proof the seam fix landed);
- the node's **real FRET** membership gate returns `member === true` for the committed tail (forming cohort
  = {self}), so the gate passes origination through (not short-circuiting);
- with no consensus cert, origination correctly no-ops.

Coverage of the cert-bearing delivery is split, honestly:
- **cert → onLocalCommit delivery** with byte-identical pass-through is proven with REAL components
  (real `StorageRepo` + real `buildCommitCert` + real `CommitCertStore`/extractor + real bridge) in
  `change-bridge.spec.ts` ("end-to-end: a stored cluster cert flows through the bridge unchanged on commit");
- **full real-consensus-cert origination on an activated node** (≥2-node consensus produces a cert, a
  member node originates) belongs to `14-substrate-e2e-real-libp2p-tier` — the multi-node real-libp2p tests
  here are env-gated (`OPTIMYSTIC_INTEGRATION=1`) and discovery-based, intentionally out of this ticket.

A reviewer who wants the headline proven before merge should consider whether a (gated) 2-node origination
test belongs here or is acceptably deferred to 14. My call: deferred, because solo cannot produce a cert
and the multi-node tier owns that.

## Other edge cases / interactions (covered or noted)

- **Tail-less / promotion events** → gate short-circuits on `event.tailId === undefined` (unit-tested).
- **Non-member** → gate returns false when FRET cohort excludes self (unit-tested with a stub FRET; cannot
  be forced on a solo node, where self is always a member).
- **onLocalCommit not yet installed** → bridge no-ops at `if (!hook) return;`. The
  `ReactivityOriginationManager` install + notification emit transport are owned by the reactivity
  origination wiring, NOT this ticket — "live" means the bridge INVOKES `onLocalCommit` whenever a consumer
  attaches one. The test attaches a probe to prove invocation.
- **Teardown** → `node.stop()` calls `unsubscribe()` + `host.stop()` (gossip timer cleared, protocols
  unhandled) before transports close, composing with the existing arachnode + clusterMember stop wrappers
  (unit-tested via a host.stop spy). No dangling `setInterval` (also `.unref()`'d in the host).
- **Disabled default** → existing suites stay green unchanged (db-p2p 639 passing; quereus 212 passing).
