# Reactivity recover RPC — live node wiring + real-socket integration test (review handoff)

The prereq (`reactivity-notification-transport`, 12.33) has **landed** (implement `b6c9351` + review `2d171b5`),
so the old "BLOCKED ON 12.33" preamble that no-op'd this ticket for 7 runs is resolved. The recover RPC is now
composed into the running node and exercised over real sockets. **Build + full db-p2p suite green; the env-gated
real-libp2p suite (including the new recover test) green.**

## What landed (the implementation)

All node wiring is in the `if (cohortEnabled)` block of `packages/db-p2p/src/libp2p-node-base.ts` (the same
block 12.33 built notify/forwarder/origination/gossip into), as a new "5. Recover RPC" step after
`pushStateGossip.start()` and before the teardown wrapper:

- **Node-level sticky cohort-hint cache** — `reactivityCohortHintCache = createStickyCohortHintCache()`. Keyed
  by collectionId, so one node-level instance is shared between the outbound transport's sticky-primary lookup
  and a future manager's rotation-invalidation. Starts empty ⇒ the transport falls straight through to the
  cohort-walk (the ticket's recommended, correct default; sticky-primary population is a one-RT optimization).
- **`resolveReactivityCohort(topicId)`** = `fret.assembleCohort(reactivityAddressing.coord0(topicId), cohortWantK)`
  — the **same** coord_0 FRET assembly the push-state-gossip authenticity gate uses, returning peer-id strings
  (the recover dialer's `peerIdFromString` space, matching the notify dial-target space).
- **Outbound transport** — `recover = new Libp2pReactivityRecoverTransport({ dialer: createLibp2pRecoverDialer(
  node, reactivityProtocols.recover), selfPeerId, cohortHintCache, resolveCohort })`. `maxBytes` omitted so the
  dialer + handler default to `DEFAULT_STREAM_MAX_BYTES`, **matching the notify transport's default** (one frame
  ceiling across the reactivity family).
- **Inbound serve handler (the live production behavior)** — `registerRecoverHandler(node,
  reactivityProtocols.recover, { pushStateFor: forwarderHost.pushStateFor.bind(...), pushStateForCollection:
  forwarderHost.pushStateForCollection.bind(...), replayGuard: createCorrelationReplayGuard() })`. One node-level
  replay guard (a pruned-on-access map — **no new timer to tear down**). This is what makes the node *answer*
  recover requests against its live forwarder `PushState`s.
- **Request signers** — `recoverSigners = createRecoverRequestSigners(nodePrivateKey)` (the already-landed
  synchronous signer seam from 12.41/the recover-request-signers split).
- **Exposed on the node** — `reactivityRecover`, `reactivityRecoverSigners`, `reactivityCohortHintCache`
  (mirrors how 12.33 exposes `reactivitySubscribers`).
- **Teardown** — the recover protocol unhandle rides the existing `node.unhandle(reactivityProtocolList(
  reactivityProtocols))` in the stop wrapper (recover is in the list — confirmed in `protocols.ts`). No change
  to teardown needed.

Docs updated (`docs/reactivity.md`): §Authentication recover note + §Backfill RPC note moved from
"transport built" → **"node wiring live"**, and §Real-libp2p e2e coverage gained a recover-socket paragraph.

## ⚠️ Honest gap — the ticket's "Feed the subscribe factory" step could NOT be done as written

**The ticket assumed 12.33 builds a "production subscribe factory" that CONSTRUCTS `ReactivitySubscriptionManager`
instances on the node. It does not.** 12.33 wired only the serving/forwarding side (forwarder host, notify
transport, origination, push-state-gossip) plus a node-level subscriber **registry** that receives inbound
frames. Constructing managers is the explicitly-deferred backlog item
`optimystic-network-reactive-watch-integration-test` (the Quereus `Database.watch` → manager bridge) — see the
comment in `forwarder-host.ts` and the note at `substrate-real-libp2p.integration.spec.ts` line ~486.

Consequences a reviewer must weigh:

- The ticket's bullet "feed `backfillTransport` / `resumeTransport` / `signBackfill` / `signResume` /
  `subscriberCoord` / shared `cohortHintCache` into the production subscribe factory" has **no factory to feed**.
  Instead, all of those seams are **exposed on the node** (`reactivityRecover` exposes
  `.backfillTransport(topicId, collectionId)` / `.resumeTransport(topicId, collectionId)`;
  `reactivityRecoverSigners` carries `signBackfill`/`signResume`; `reactivityCohortHintCache` is the shared
  cache) so the deferred factory wires them in one place. **`subscriberCoord`** is a per-request field the
  *manager* stamps into the `ResumeV1` (not a signer concern), so it correctly belongs to that future factory,
  not here.
- Therefore the **outbound** transport is constructed + exposed but **not yet called by any node-internal code**
  — exactly as 12.33's notify *subscriber* side has no node-internal constructor either. The **inbound serve
  handler IS live production wiring.** This split (serve-live, subscribe-exposed-for-factory) is the honest
  state; it is not a regression and matches how the whole reactivity subscribe path currently sits.

This is the one place the ticket's plan diverged from the codebase, and the cause is the prereq, not a design
problem here. If the reviewer judges the subscribe-side wiring in-scope, the right move is to advance the
`optimystic-network-reactive-watch-integration-test` backlog item, not to re-open this ticket.

## Test / validation use cases

**Live, real-socket (new, env-gated):** `substrate-real-libp2p.integration.spec.ts` →
*"a remote subscriber resumes past the tail over a real recover socket and is brought current (backfill)"*.
A remote node originates rev 1 on the origin (filling the origin's forwarder `PushState` replay ring), then a
remote subscriber sends a real `ResumeV1` over `/optimystic/reactivity/1.0.0/recover` and gets a `backfill`
reply carrying rev 1. Exercises the **production** registered serve handler, real peer-key signing/verify, the
replay/freshness guard, and `serveResume` end-to-end. Passed in ~331 ms at N=4.
  - Gate: `OPTIMYSTIC_INTEGRATION=1 yarn workspace @optimystic/db-p2p test:integration`. If unset, the suite is
    `describe.skip`'d and recover coverage rests on the in-process specs below.
  - **Determinism note:** the test pins the recover transport to the origin (`resolveCohort: () => [origin]`)
    rather than letting the production transport walk the real cohort. The sticky-primary → cohort-walk target
    selection (and the no-reply→fall-through behavior) is unit-covered by `recover-transport.spec.ts`. A
    reviewer wanting fuller real-socket fidelity could let `remote.node.reactivityRecover` walk the live cohort
    — the risk I avoided is the `readAllBounded` empty-frame-vs-throw semantics on a no-reply member over a real
    socket (a non-serving cohort member that closes without replying); worth a look if deeper fidelity is wanted.

**In-process floor (existing, always-on):**
- `reactivity/recover-transport.spec.ts` (18 tests) — serve handler (backfill, all serveResume classifications,
  stale-tail collection fallback, bad-sig/replay/stale/no-PushState rejections, undecodable frame) + outbound
  transport (sticky-primary-first, cohort-walk fallback, never-dial-self, no-target rejection, dial-target
  encoding bridge).
- `reactivity/recover-signers.spec.ts` (7 tests) — `signPeerSig` round-trip + async byte-identity, and the
  signers integrating with the serve handler.

## Suggested reviewer probes

- **Coord agreement:** confirm `resolveReactivityCohort` derives the *same* cohort the origination gate and the
  gossip authenticity gate use (all three go through `reactivityAddressing.coord0` → `fret.assembleCohort`), so
  a recover walk reaches exactly the members that hold the gossiped `PushState`.
- **Serve resolution:** `pushStateFor`/`pushStateForCollection` are bound to the **one** `forwarderHost`
  instance that notify fan-out + origination ingest also fill — so a recover request resolves against live
  served state (validated by the passing integration test).
- **maxBytes parity:** all recover seams omit `maxBytes` → `DEFAULT_STREAM_MAX_BYTES`, equal to the notify
  transport's (also omitted). Confirm no divergent literal slipped in.
- **Self-dial:** the transport excludes `selfPeerId`; a co-located subscriber falls through to other cohort
  members (the ticket's accepted "cohort-walk fallback" decision — no in-process short-circuit added).
