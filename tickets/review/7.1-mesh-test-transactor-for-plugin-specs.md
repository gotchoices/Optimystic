## Summary of what shipped

- **New factory path** `'mesh-test'` in `CollectionFactory.createTransactor` builds a
  1-node mock mesh via `createMesh(1, …)` + `buildNetworkTransactor(mesh)` — the
  exact production chain `StorageRepo → CoordinatorRepo → NetworkTransactor` that
  `'test'` (StorageRepo pass-through) never exercised. `'test'` is preserved for
  the narrow specs that explicitly want the pass-through.

- **Discovery / fix along the way**: `default_transactor` / `default_key_network`
  in the plugin config (`register(db, { default_transactor: … })`) were previously
  *dead* — Quereus surfaced them as `tableSchema.vtabAuxData` but
  `parseTableSchema` never read it, so every spec that "used" `'test'` was
  silently booting `createLibp2pNode` via the default `'network'` path. The spec
  suite's fast runtime masked this. `parseTableSchema` now reads
  `tableSchema.vtabAuxData` as a fallback below explicit `transactor='…'` vtab
  args, so plugin-level config finally drives the factory.

- **Harness relocation**: `packages/db-p2p/test/mesh-harness.ts` →
  `packages/db-p2p/src/testing/mesh-harness.ts` (+ `index.ts` barrel) and exposed
  via a new `"./testing"` subpath export in `packages/db-p2p/package.json`. All
  seven existing in-package consumers under `packages/db-p2p/test/*.spec.ts` were
  updated to import from the new path; the old `test/mesh-harness.ts` was
  deleted. The plugin imports through the subpath export:
  `import { createMesh, buildNetworkTransactor } from '@optimystic/db-p2p/testing'`.

- **Spec migrations**: three specs switched from `default_transactor: 'test'` →
  `'mesh-test'`:
    - `schema-support.spec.ts`
    - `engine-execution.spec.ts`
    - `schema-migration.spec.ts` (two sites)
  The fourth spec the original ticket listed, `distributed-quereus.spec.ts`, was
  **not** migrated: it doesn't use `default_transactor: 'test'` at all — it
  drives real libp2p and passes `transactor='network'` per-table. Migrating it
  to the mock mesh would contradict its purpose (multi-node real-libp2p
  distribution). The ticket's "four specs" count was off by one; flag if a
  reviewer disagrees.

- **Sanity spec**: `packages/quereus-plugin-optimystic/test/mesh-test-transactor.spec.ts`
  asserts DDL + DML round-trip via the `'mesh-test'` factory with a per-test 5s
  timeout.

- **Pre-existing typecheck failures**: `yarn typecheck` on the plugin reports
  three errors in `src/schema/index-manager.ts` (105, 157) and
  `test/manual-mesh-test.ts` (65). All three exist on main pre-ticket (verified
  via `git stash` + run). Not introduced by this work — flag separately if
  desired.

## Use cases for review

- **Interface discipline**: `'mesh-test'` is production wiring behind an
  `ITransactor` surface — no `.coordinatorRepo` / mesh internals leak into the
  plugin. Confirm the factory cache key (`${transactor}:${keyNetwork}`) still
  uniquely distinguishes `'mesh-test'` vs `'test'` vs `'network'`.

- **Cleanup**: the `'mesh-test'` factory does not currently shut down the mesh —
  there's no libp2p process or external resource to stop, so `CollectionFactory.shutdown`
  (libp2p-only) is unaffected. If a future reviewer wants the mock mesh to be
  released on shutdown, it's a one-line addition, but not required today.

- **Spec behavior under the new stack**: each migrated spec's assertion surface
  is now exercised against `NetworkTransactor + CoordinatorRepo`. Since all
  assertions still pass, no latent bugs surfaced; the ticket's "file a fix
  ticket per failure" clause did not trigger.

- **Subpath export reachability**: `@optimystic/db-p2p/testing` only exports
  `./dist/src/testing/*`. Consumers should never import from the test/ folder
  directly. Verify no such imports remain (I updated all seven in-package
  specs).
