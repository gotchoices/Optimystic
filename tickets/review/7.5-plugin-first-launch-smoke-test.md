## What was built

A single spec file at `packages/quereus-plugin-optimystic/test/plugin-first-launch-libp2p.integration.spec.ts` with three `it` blocks under a single `describe('Plugin first-launch over real libp2p')`:

1. **happy path: CREATE TABLE, INSERT, SELECT round-trip** — cold-boot a real libp2p node, register the plugin with an injected `NetworkTransactor`, run `CREATE TABLE canary (id INTEGER PRIMARY KEY, value TEXT NOT NULL) USING optimystic('tree://canary-smoke/canary')` → `INSERT (1, 'first')` → `SELECT value WHERE id = 1`, expect `'first'`.
2. **cold-restart over shared storage preserves rows** — boot node A over a pre-constructed `MemoryRawStorage` + pre-generated `Ed25519` private key, create+insert, stop node A, boot node B over the **same** storage+identity with a fresh `Database`, re-declare the table, and `SELECT` the original row.
3. **two sequential DDLs in one session** — two `CREATE TABLE` + `INSERT` pairs in one `Database` session, exercising the "second DDL after first DDL" path.

Gating:
- `before(function () { if (!process.env.OPTIMYSTIC_INTEGRATION) this.skip(); })` at the suite level.
- Top-of-file comment documents the POSIX invocation and the Windows PowerShell `$env:` alternative (verbatim copy of the block from `db-p2p/test/real-libp2p.integration.spec.ts`).

Transactor-injection route (Route A from the ticket): built `NetworkTransactor` externally in the spec (same wiring as `fresh-node-ddl-libp2p.spec.ts`) and registered it via `plugin.collectionFactory.registerTransactor('network:libp2p', transactor)`. No plugin-source changes required. The plugin's `SchemaManager` + `CollectionFactory` resolve the injected transactor via the existing cache key because `parseTableSchema` defaults `transactor` to `'network'` and `keyNetwork` to `'libp2p'`.

## Package wiring

Added to `packages/quereus-plugin-optimystic/package.json`:

```json
"test:integration": "OPTIMYSTIC_INTEGRATION=1 npm test"
```

POSIX-form, mirroring `db-p2p`. Windows users run either via bash/WSL or set `$env:OPTIMYSTIC_INTEGRATION=1` before `npm test` (documented in-spec).

## Interfaces for the reviewer to verify

- **Gate semantics.** `npm test` (no env var) must show the three canary `it`s as `pending` (mocha blue dash). `OPTIMYSTIC_INTEGRATION=1 npm test` must show them as passing.
- **Cross-platform doc.** The top-of-file comment must document both the POSIX and Windows PowerShell invocations.
- **No plugin-source changes.** `optimystic-module.ts`, `schema-manager.ts`, `collection-factory.ts`, and `plugin.ts` are untouched — the spec integrates against the existing `registerTransactor` + default-name-resolution surface.
- **Transactor cache key.** The spec relies on the plugin resolving `transactor: 'network'` + `keyNetwork: 'libp2p'` to the injected instance via `registerTransactor('network:libp2p', ...)`. If a future change renames the cache key convention in `CollectionFactory.getTransactorKey`, this spec breaks with a clear error (custom-transactor-not-found), not a hang.
- **Cleanup.** `afterEach` stops every tracked node via `Promise.allSettled`. The cold-restart `it` splices node A out of the tracked list after stopping it, so `afterEach` does not double-stop.
- **Fail-fast behavior.** Ticket's intentional-break check verified: breaking `SchemaManager.storeSchema` makes the canary fail with a surfaced error, not hang to timeout. This was confirmed empirically; see table above.

## Usage

```sh
# From repo root (POSIX):
npm run test:integration --workspace @optimystic/quereus-plugin-optimystic

# Windows PowerShell:
$env:OPTIMYSTIC_INTEGRATION=1; npm test --workspace @optimystic/quereus-plugin-optimystic
```

Note: `dist/plugin.js` is imported by the spec (matching the convention of other plugin specs), so `npm run build --workspace @optimystic/quereus-plugin-optimystic` must be run before the test scripts pick up plugin-source changes.
