## What shipped

### Helper — `packages/db-p2p/test/mesh-harness.ts`

Two new exports alongside `createMesh`:

- `buildNetworkTransactor(mesh, options?) → ITransactor` — single transactor over the mesh; `getRepo` looks up each peer's `coordinatorRepo`. `timeoutMs` / `abortOrCancelTimeoutMs` default to 5s.
- `buildNetworkTransactors(mesh, options?) → Map<peerIdStr, ITransactor>` — one transactor per mesh node, keyed by peer-id string, so multi-node specs can semantically say "driven by node A" vs "driven by node B."

`fresh-node-ddl.spec.ts` now consumes `buildNetworkTransactor` from the harness (its local copy was deleted).

### Spec — `packages/db-p2p/test/fresh-node-ddl-multi.spec.ts`

- **Scenario A (passing, 10s timeout)**: 3-node mesh, `responsibilityK: 3`, `clusterSize: 3`, `superMajorityThreshold: 0.67`. Node A does first-ever `Tree.createOrOpen` + `tree.replace`; nodes B and C independently `createOrOpen` the same tree id and `tree.get` — both must see A's write.
- **Scenario B (skipped via `describe.skip`, 15s timeout when enabled)**: 5-node mesh, `responsibilityK: 5`, `clusterSize: 5`, `superMajorityThreshold: 0.6`. `mesh.failures.failingPeers` marks node E unreachable before the first transaction. Node A writes; node B reads. Super-majority (3-of-5) is satisfied by the 4 reachable peers, but **on `main` today the commit phase fails with `Some peers did not complete: <peerE>[blocks:1](in-flight) cause=Pending action … not found`**. Reduced repro + stack trace are in `tickets/fix/5-multi-node-cold-start-commit-with-failing-peer.md`; removing `describe.skip` is the acceptance test for that fix.
