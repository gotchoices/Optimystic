# Review: reactivity notify protocol surface + transport primitives

## What was built

Pure transport plumbing for the 12.3 reactivity chain — frame → dial → deliver of `NotificationV1`
bytes over libp2p. **No fan-out orchestration, no role decision, no gossip** (those are the siblings
`reactivity-forwarder-host` and `reactivity-pushstate-gossip`). Mirrors `packages/db-p2p/src/cohort-topic/`
in style.

### `packages/db-p2p/src/reactivity/protocols.ts` (new)
Reactivity protocol family, mirroring `cohort-topic/protocols.ts`:
- `REACTIVITY_BASE = "/optimystic/reactivity/1.0.0"`
- `PROTOCOL_REACTIVITY_NOTIFY` = `${BASE}/notify` (one-way `NotificationV1`)
- `PROTOCOL_REACTIVITY_PUSH_STATE_GOSSIP` = `${BASE}/push-state-gossip` — **declared here** (the family
  owns it in one place) but **consumed by the sibling `reactivity-pushstate-gossip` ticket**, unused in
  this diff.
- `${BASE}/recover` is **reserved** (named in a comment, intentionally NOT defined) — 12.4 backfill/resume.
- `ReactivityProtocols` interface, `DEFAULT_REACTIVITY_PROTOCOLS`, `makeReactivityProtocols(networkName)`,
  `reactivityProtocolList(p)`.

### `packages/db-p2p/src/reactivity/notify-transport.ts` (new)
- `ReactivityNotifyTransport` interface + `Libp2pReactivityNotifyTransport`:
  - `send(target, n)` — **unicast** (vs cohort-gossip's broadcast). Reuses `sendOneWay` +
    `encodeNotificationV1`. Self-target guard skips the dial when `selfPeerId` is set.
  - `onNotification(handler)` — subscribe, returns unsubscribe.
  - `deliver(fromPeerId, frame)` — `decodeNotificationV1` in a try; decode failure → log + drop (never
    throws); fans the decoded notification to every subscriber, building `from: PeerRef = { id:
    peerIdToBytes(fromPeerId) }` (mirrors `FretCohortGossipTransport.deliver`).
- `registerNotifyHandler(node, protocol, transport, maxBytes?)` — one-way inbound handler mirroring
  `stream-util.ts`'s `handleRequestResponse` but **with no reply frame**: read one bounded frame
  (`readAllBounded` / `DEFAULT_STREAM_MAX_BYTES`) → `transport.deliver(...)` → close; abort on error.

### `packages/db-p2p/src/reactivity/index.ts`
Added `export * from "./protocols.js"` and `export * from "./notify-transport.js"`.

## Validation done

- **Build**: `yarn workspace @optimystic/db-core build` clean (db-core rebuilt so its latest reactivity
  exports — `encodeNotificationV1`/`decodeNotificationV1`/`NotificationV1`/`PeerRef` — are in `dist`).
- **Type-check**: `yarn workspace @optimystic/db-p2p exec tsc --noEmit` clean.
- **Tests**: `test/reactivity/notify-transport.spec.ts` — 9 passing. Re-ran `managers.spec.ts` +
  `notify-transport.spec.ts` together → **28 passing**, no regression from the index export change.

Test coverage (new spec):
- round-trip `send → deliver` via a capturing fake node; the framed bytes are byte-identical to
  `encodeNotificationV1(n)` and the decoded notification deep-equals the original; `from.id` decodes back
  to the inbound peer-id string.
- fan-out to multiple subscribers + unsubscribe handle.
- malformed inbound frame → dropped, no throw, no subscriber fire.
- unreachable target → `send` resolves (rejection swallowed); a later send to a live target still fires
  (failure isolation).
- self target skipped when `selfPeerId` set; non-self still dialed.
- `registerNotifyHandler` reads one inbound frame and delivers it; asserts the handler sends **no reply
  frame** (one-way).
- protocol namespacing: `DEFAULT_*` omits the network segment; `makeReactivityProtocols` inserts it,
  matching `makeCohortTopicProtocols`; `reactivityProtocolList` enumerates the family.

## Known gaps / things to scrutinize (tests are a floor, not a finish line)

- **`send` swallows MORE than just the dial rejection.** The ticket said swallow the dead-target
  rejection; the implementation wraps `encodeNotificationV1` + `peerIdFromString` + the dial all in
  try/catch so `send` **never rejects** (the load-bearing "one bad target never breaks a commit/fan-out"
  property). The tradeoff: a genuinely malformed *local* notification (a caller bug) is logged + dropped
  rather than surfaced as an exception. Decide whether an encode/peer-id error should instead propagate
  (it is a local programming error, "an exception is exceptional" per AGENTS.md) vs. stay swallowed for
  fan-out safety. Currently it is swallowed.
- **`deliver` passes the transport's stream-level `maxBytes` (default 512 KiB `DEFAULT_STREAM_MAX_BYTES`)
  as `decodeNotificationV1`'s message-codec ceiling.** These are conceptually distinct bounds (stream
  read bound vs db-core's `DEFAULT_MAX_MESSAGE_BYTES`). Unified to match the cohort host's
  `decodeCohortMessage(frame, maxBytes)` pattern; confirm that's intended and that 512 KiB is the right
  notification ceiling.
- **No real-libp2p / on-wire coverage.** All tests use a fake node (no real streams). The oversized-frame
  abort path is delegated to FRET's `readAllBounded` (its own tested behavior) and is **not** re-tested
  here — there is no explicit oversized-frame test. The existing `mesh-*` reactivity specs predate notify
  and do not exercise it. Real-wire round-trip coverage is deferred until `reactivity-forwarder-host`
  wires `registerNotifyHandler` into an actual node.
- **`registerNotifyHandler` is exported but nothing wires it yet.** No host registers the notify protocol
  in this diff; that is the forwarder-host ticket's job. Verify the handler signature/lifecycle matches
  what that ticket will expect.
- **Full `yarn workspace @optimystic/db-p2p test` was NOT run end-to-end.** Only db-core build +
  db-p2p type-check + the reactivity unit specs (managers + new spec). The full suite includes
  long-running real-libp2p/mesh/integration specs that exceed the agent idle window; a human/CI should
  run the whole suite. Nothing in this diff touches those subsystems (two new files + one index export
  line), so regression risk is confined to the reactivity module.
- **Self-delivery is out of scope.** The `selfPeerId` guard only *skips the dial*; in-process delivery of
  a co-located subscriber is the forwarder-host ticket's call (the transport never dials self and never
  in-process-delivers). Confirm that boundary reads correctly.
