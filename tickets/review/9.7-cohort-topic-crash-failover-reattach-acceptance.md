# Crash-failover backup promotion ↔ deterministic slot assignment (implemented)

## What was built

The TTL-failover participant path and the cohort `onRenew` were incoherent when composed: on a
**crash** (no membership rotation, `cohortEpoch` unchanged) `assignSlots` still names the dead member
as primary, so a re-attach to `backups[0]` was answered `primary_moved { newPrimary: <dead primary> }`
and the participant bounced between the promoted backup and the corpse until FRET rotation eventually
changed the epoch. This ticket closes that loop. All three resolved design questions from the source
ticket were implemented as specified (no re-litigation).

**Wire (`types.ts`, `validate.ts`).** `RenewV1` gains an optional, **signed** `reattach?: boolean`.
It lives inside the signed `UnsignedRenew` body (the participant's `buildRenew` includes it before
signing), so a member can trust the attestation and a MITM cannot flip a ping into a promotion.
`validateRenewV1` accepts it via `assignDefined(out, "reattach", optBool(...))` (mirrors `bootstrap`
on `RegisterV1`): absent → field stays off the decoded object; non-boolean → `CohortWireError`. No
codec framing change. Backward-compatible (old encoder omits it → decodes as a normal ping).

**Participant (`renewal.ts`).** `buildRenew(reattach: boolean)` sets `reattach: true` only on the
failover path (a plain ping omits it entirely → falsy). `trySend` → `buildRenew(false)`; `reattach()`
→ `buildRenew(true)`. `failover()` rewritten:
- `ok` → `promote(backup)` (existing).
- `primary_moved` with `newPrimary ≠ just-failed primary` → `applyPrimaryMoved(reply)` (adopt the
  rotated payload, refresh the lazy epoch hint), **not** a blind promote of the contacted backup.
- `primary_moved` with `newPrimary === just-failed primary` → **ignored** (bounce guard); continue.
- `unknown_registration` / anything else → try next backup; all exhausted → `relookup()` + reset.

**Cohort (`renewal.ts`).** `StoreRenewalCohortSide` gains `failoverServing: Map<recordKey, epoch>`
(epoch-scoped override) and a factored `touchAndServe(rec, now)`. `onRenew` branches on `msg.reattach`:
- **reattach=true:** `self == computed primary` → `touchAndServe` (no override); `self ∈ computed
  backups` (and record held) → re-stamp `primary = self` (drop self from backups), `gossip.touch`,
  set override under current epoch, reply `ok`; else `isServing?` → `touchAndServe`; else
  `primary_moved` (redirect — do **not** promote a non-backup).
- **plain ping:** serve if `self == computed primary` **or** `isServing?` **or** an override tagged
  with the *current* epoch exists. Housekeeping deletes a redundant override (self is computed primary
  again) or a stale one (epoch mismatch) so it never serves across a rotation. Else `primary_moved`.

The re-stamp drives gossip convergence + the next rotation's `handoff.start()`; it deliberately does
**not** drive the serve decision (the override does) — preventing the rotation-handoff regression the
source ticket calls out (an old primary whose stored `record.primary == self` would otherwise serve
forever).

## Key use cases / behaviors to validate (the floor, not the ceiling)

Participant side (`registration.spec.ts`):
- Re-attach sends `RenewV1.reattach === true`; a normal `pingLoop` sends it falsy (asserted on the
  signed body the mock transport receives).
- **Anti-bounce (the core bug):** primary fails 3× → re-attach `backups[0]` returns `ok` →
  `record.primary == backups[0]`, and 4 subsequent plain pings keep returning `ok` with
  `record.primary` pinned to the backup (no relookup, no bounce).
- `failover()` honors `primary_moved { newPrimary: live-member }` → adopts the live member + refreshes
  the epoch hint (not a blind promote).
- Bounce guard: `primary_moved { newPrimary: <just-failed primary> }` → ignored; falls through to the
  next backup.
- `unknown_registration` on re-attach → next backup tried; all unknown → `relookup`.

Cohort side:
- reattach=true, `self ∈ computed backups`, record held → `ok`, `lastPing` touched, gossip fired,
  `primary` re-stamped to self, self removed from backups, **and a subsequent plain ping serves `ok`
  via the override** under the unchanged epoch.
- Plain ping on a backup holding the record → `primary_moved` (no promotion, no override; a further
  plain ping still redirects).
- reattach on a member that is neither primary nor a computed backup → `primary_moved` (redirect, not
  re-stamped).
- Override epoch-scoped: accept under epoch-1, rotate to epoch-2 → plain ping no longer serves on the
  stale override (`primary_moved`, or `ok` only if self is the new computed primary).
- reattach where `self == computed primary` → `ok` without a redundant override.

Wire (`wire.spec.ts`): `RenewV1` with `reattach: true` round-trips; absent `reattach` decodes without
the field; non-boolean `reattach` throws `CohortWireError`.

## Honest gaps / reviewer attention

- **Cascading failover is covered by logic, not a dedicated cohort test.** The source ticket's
  `s → s+1 → s+2` cascade relies on `self ∈ computed backups` accepting for *any* computed backup
  (`backups.some(...)`), so the `backups[0]`-as-self acceptance test exercises the same branch a
  `backups[1]`-as-self (s+2) re-attach would. Participant-side iteration through multiple backups is
  tested (bounce-guard + unknown_registration fall-through). A reviewer wanting belt-and-suspenders
  could add an explicit cohort test with `self = backups[1]`.
- **Gossip / cross-member replication is mocked.** `gossip.touch` is a recording stub here; real
  convergence of the re-stamped record onto other replicas depends on the `cohort-topic-gossip-
  membership-certs` prereq (already landed) and is not re-exercised end-to-end in this unit suite.
  The override itself is intentionally **not** gossiped (only the contacted backup is pinged).
- **`primary_moved` with no `newPrimary`.** The bounce guard only compares when `reply.newPrimary` is
  defined; a `primary_moved` lacking `newPrimary` would fall into `applyPrimaryMoved` (which then
  leaves `primary` unchanged). Real cohort replies always carry `newPrimary`, so this is a benign
  defensive edge, but it is not asserted.
- **Delivery-level dual-serve window** (a revived dead primary plus the promoted backup both pushing)
  is the same bounded window the doc already tolerates for the rotation handoff; application-level
  delivery dedup owns it. Not exercised here (out of this layer's scope).
- **No new threshold/tie-break primitive** was introduced, per the resolved design — the single-primary
  invariant for renewal rests on the participant pinging exactly one member, plus epoch-scoped
  override clearing. A reviewer should sanity-check that this soft-state argument holds under the
  revived-primary scenario (covered by the epoch-clearing test + the TTL-staleness argument in docs).
