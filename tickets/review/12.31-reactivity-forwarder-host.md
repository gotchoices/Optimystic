# Review: Reactivity forwarder host (receive → forward → fan-out orchestration)

## What was built

`packages/db-p2p/src/reactivity/forwarder-host.ts` (new) — the orchestrator that turns db-core's
in-process forward *decision* into live fan-out over the `ReactivityNotifyTransport`:

- **`ReactivityForwarderHost`** with:
  - `ingest(topicId, n)` — the unified receive→forward→fan-out, bound to both the origination `emit` seam
    and the inbound forwarder path. Lazily instantiates `{ pushState, forwarder }` per topic behind the
    Edge gate (`instantiateForwarderPushState` → `undefined` on Edge ⇒ never forwards, cached as `null`),
    serializes ingests **per topic** (a promise-chain tail per `topicKey`), runs `forwarder.receive`, and
    fans out only on `"forward"`. Never rejects (defensive isolation) so a fan-out fault can never surface
    as a commit failure.
  - `onInbound(from, n)` — derives the topic from the notification, runs the **subscriber role**
    (`deliverLocal`, in-process) and the **forwarder role** (`ingest`); both idempotent via the forwarder
    dedupe + the subscriber's `(collectionId, revision)` dedupe.
  - `fanOut` (private) — reconciles per-subscriber queues (lazy GC of departed subscribers),
    `enqueueForSubscribers`, drains each queue and delivers (self → `deliverLocal`, others →
    `transport.send`, per-target isolated), then dials each child cohort with the **unmodified** frame.
  - `pushStateFor(topicId)` — test/diagnostic accessor.
- **`reactivityNotificationTopicId(n)`** — `reactivityTopicId(b64urlToBytes(n.tailId))`, the single
  derivation point (matches the verifier's coord derivation and origination's tail encoding).
- **`reactivityDirectSubscribers(source, topicId)`** — adapter over `CoordEngine.records(topicId)`:
  keeps records whose `appState` decodes to a reactivity `SubscribeAppPayloadV1`, maps to
  `bytesToB64url(participantId)`.
- Exported from `packages/db-p2p/src/reactivity/index.ts`.

`ReactivityOriginationManager.emit` needs **no change** (confirmed — it just calls `emit(n)`); the wiring
binds `emit: (n) => host.ingest(reactivityNotificationTopicId(n), n)`.

### Test use cases covered (the floor — extend, don't trust as exhaustive)

- emit → fan-out dials each direct subscriber exactly once, **byte-identical** (same-reference) frame.
- `"forward"` fans out; `"duplicate"` and `"untrusted"` do not (and untrusted never reaches the ring).
- node both subscriber + primary: delivers in-process AND fans out; **self never dialed**.
- `onInbound` drives both roles.
- slow subscriber drop-oldest (`dropped` increments) without stalling a fast subscriber.
- departed subscriber's queue reclaimed on the next fan-out (queue map bounded by live set).
- child-cohort primary dialed — both `CohortRef.primary` and the `resolveChildPrimary` fallback.
- Edge profile → no PushState, no fan-out, but still delivers as a subscriber via `deliverLocal`.
- per-target send failure isolated (one dead subscriber doesn't break the rest).
- concurrent ingest for one collection keeps the ring contiguous + dedupe correct.
- verifier-throw → `ingest` resolves (never rejects), no fan-out.
- `reactivityDirectSubscribers` filters to reactivity records only; `reactivityNotificationTopicId`
  matches the tail-anchor derivation.

## Known gaps / things to scrutinize (honest)

1. **Subscriber-id encoding reconciliation — the load-bearing integration risk.** The host operates in a
   single string space for `selfPeerId`, `directSubscribers()` output, child primaries, and every
   `transport.send` target: **base64url of the member-id bytes** (matching the `perSubscriberQueue` key and
   `reactivityDirectSubscribers`). But the real `Libp2pReactivityNotifyTransport.send` calls
   `peerIdFromString(target)` — it expects a **raw peer-id string**, not base64url. The host is
   deliberately encoding-agnostic, so the node-wiring ticket (`reactivity-notification-transport`) **must**
   close this loop consistently across all four seams — e.g. feed the host peer-id strings
   (`selfPeerId` = peer-id string, `directSubscribers` via `bytesToPeerIdString(record.participantId)`,
   child primaries decoded likewise) **or** wrap the transport to decode base64url→peer-id string. Reviewer:
   confirm the wiring ticket picks one space end-to-end; a mismatch here is a silent no-dial bug.

2. **Slow-subscriber backpressure is not a live loop.** `fanOut` drains each queue inline right after
   enqueue and fires `transport.send` fire-and-forget (`void`). So in steady state the per-subscriber queue
   is a pass-through and drop-oldest is **not organically reachable** — there is no "slow dial backs the
   queue up" worker (the mesh harness models a slow subscriber via `autoDrain=false`, which the host has no
   equivalent of). The drop-oldest test simulates a backed-up queue by pre-filling it. This matches the
   ticket's pseudo-code (enqueue → drain inline) and the harness's `enqueue → drain` semantics, but if the
   intent is genuine in-flight backpressure (bounded concurrent dials per subscriber, queue absorbs the
   lag), that's a follow-on redesign — a per-subscriber async delivery worker. **Decide which model is
   intended** before this is wired live.

3. **`onInbound` forwarder-role gating is upstream.** `onInbound` calls `ingest` unconditionally; `ingest`
   self-gates only via the Edge policy and an empty `directSubscribers` for non-members. A **Core** node
   that receives a notify for a topic it does *not* serve would instantiate a `PushState` and fan out to an
   empty set (harmless, but wasteful, and it caches served state for a topic it shouldn't). The deps carry
   no `servesTopic` predicate — the host trusts the routing layer to deliver forwarder-role inbounds only
   for topics the node serves. Reviewer: confirm the node-wiring ticket routes by topic (it says it does),
   or consider adding a `servesTopic` guard to the host.

4. **Child fan-out is wired but a no-op today.** `childCohorts == []` until
   `cohort-topic-parent-child-link` (backlog) populates it. Exercised only by the synthetic-childCohorts
   test. The dial path is correct but unexercised by any end-to-end mesh until that link lands.

5. **`reactivityDirectSubscribers` does not cross-check `collectionId`.** It filters by "decodes to a
   reactivity `SubscribeAppPayloadV1`" but doesn't compare the payload's `collectionId` to the topic's
   collection. Safe **today** because `topicId = H(tailId ‖ "reactivity")` and tail block ids are unique
   per collection, so all records under one `topicId` are one collection's subscribers. If that invariant
   ever loosens, add a `collectionId` filter.

6. **Docs intentionally not touched.** `docs/internals.md` §Cohort-Topic Origination Bridge tracks the
   *live* node path; this host is an unwired building block (the libp2p node assembly is
   `reactivity-notification-transport`). I left the docs alone rather than document a piece that isn't yet
   in the live commit/notify path — the node-wiring ticket should update internals.md when fan-out goes
   live. Reviewer may disagree and want a forward-reference note added now.

7. **Per-topic serialization map is unbounded-by-topic.** `ingestTails` / `served` hold one entry per topic
   the node ever ingested (overwritten, not appended). Bounded by the node's served-topic count, which is
   itself bounded by the cohort-topic `topics_max` budget — but nothing here evicts a topic whose cohort
   the node has *left* (e.g. after a tail rotation moves the topic elsewhere). Not a leak in any current
   test, but worth a glance for the rotation interaction.

## Suggested review focus

- Trace the encoding in gap (1) against `notify-transport.ts` and the (in-flight) node-wiring ticket —
  this is where a real bug would hide.
- Decide the backpressure model in gap (2): inline-drain + fire-and-forget (current) vs. a live
  per-subscriber delivery worker. The current model is honestly a pass-through for the queue.
- Sanity-check the per-topic promise-chain serialization (`ingest`) for correctness under interleaving and
  that it never leaks a rejection (both `then` arms call `ingestSerialized`, which catches all).
