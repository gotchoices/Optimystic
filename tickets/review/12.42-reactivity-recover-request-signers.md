# Review — reactivity recover request signing (synchronous Ed25519 signer seam)

This is the **written, committed, green** sub-deliverable split out of `reactivity-recover-node-wiring`
(12.42). The rest of that ticket — composing the recover transport + handler + signers into the live
running node, plus the real-socket integration test — stays in `implement/` (slug
`reactivity-recover-node-wiring`), genuinely gated on `reactivity-notification-transport` (12.33), whose
node-wiring block does not exist yet. This review covers **only** the signing seam, which is self-contained
(no node wiring, no libp2p dial) and unit-testable in isolation.

## What this code does (plain terms)

The reactivity recovery RPC lets a subscriber that fell behind ask a serving cohort member "send me what I
missed" (a `BackfillV1` or `ResumeV1` request). Each request must be **signed** so the server can verify the
asker is who they claim and reject replays. The signing seam the subscription manager exposes
(`signBackfill` / `signResume`) is **synchronous** — `(unsigned) => string` — because db-core's
`createBackfillRequester` builds the unsigned image internally (a pre-signed value is impossible) and the
port shape is `=> string`, not `=> Promise<string>`. But libp2p's `PrivateKey.sign` is **async**. This code
bridges that gap by signing synchronously with `@noble/curves/ed25519` directly over the node's raw Ed25519
seed, instead of forcing the seam (and a cross-package db-core change) async.

## What landed (the diff under review)

- **`signPeerSig(privateKey, payload): Uint8Array`** in `packages/db-p2p/src/cohort-topic/peer-sig.ts` —
  the synchronous mirror of the existing async `signPeer`. Reads the libp2p Ed25519 `raw` key (64 bytes =
  32-byte seed ‖ 32-byte public key; also accepts a bare 32-byte seed), signs with noble over the 32-byte
  seed. **Throws** on a non-Ed25519 key / unexpected raw length (a programming error on the node's own key —
  never attacker-reachable; the verify side, by contrast, is total and returns `false`).
- **`createRecoverRequestSigners(privateKey): { signBackfill, signResume }`** in
  `packages/db-p2p/src/reactivity/recover-transport.ts` — builds the manager's `(unsigned) => string` seam
  over db-core's canonical `backfillSigningPayload` / `resumeSigningPayload`, base64url-encoded. This is the
  exact glue the production subscribe factory will feed the manager once 12.33's node wiring exists.
- **Tests**: `packages/db-p2p/test/reactivity/recover-signers.spec.ts` (7 cases).
- **Doc**: `docs/reactivity.md` §Authentication — the recover-RPC bullet documents the synchronous
  subscriber-side signing path and that wiring it into the running node rides 12.33.

## Design decision (resolved — confirm it holds)

**Synchronous noble sign over the node's raw Ed25519 seed, keeping the existing `(unsigned) => string` seam
unchanged.** The async-seam alternative was rejected: it would force `createBackfillRequester`'s `sign`
async — a cross-package db-core change overlapping `reactivity-recover-wire-signing` (12.4) — for no
benefit. The sync path needs **no** db-core change, mirrors the existing synchronous `verifyPeerSig`, and
round-trips with it (both `@noble/curves/ed25519`, RFC8032).

The load-bearing correctness claim: **noble's sync sign is byte-identical to libp2p's async `PrivateKey.sign`
(Node/WebCrypto) for the same key + payload**, because RFC8032 Ed25519 is deterministic. The serving side
verifies under `verifyPeerSig` (noble, ZIP215 — a strict superset of RFC8032), so the round trip is
symmetric in both directions (sync-signed verifies under both libp2p and noble; libp2p-signed verifies under
noble).

## Review focus / adversarial angles

- **Seed extraction correctness.** `signPeerSig` slices `raw.subarray(0, 32)` for the 64-byte libp2p form.
  Confirm libp2p's Ed25519 `raw` layout is seed‖pub (not pub‖seed or a PKCS8/DER wrapper) for the libp2p
  version in use — a wrong slice would sign with garbage yet still "verify" only if the same wrong bytes
  feed both sides (they don't here: verify derives the pubkey from the peer-id multihash, independent of the
  signer's seed). The byte-identity test against `signPeer` is the guard; sanity-check it actually exercises
  a 64-byte key (it does — `generateKeyPair('Ed25519')`).
- **Determinism assumption.** The whole sync↔async equivalence rests on RFC8032 determinism. If any path
  (a future libp2p backend, a non-WebCrypto polyfill) produced a non-deterministic or differently-encoded
  signature, the byte-identity test would catch it — confirm the test is not accidentally comparing a
  re-derived value to itself.
- **Throw vs. total.** `signPeerSig` throws on non-Ed25519; `verifyPeerSig` is total (returns `false`).
  Confirm the asymmetry is intended and that the throw path is only ever reachable from the node's **own**
  key (the subscribe factory feeds `nodePrivateKey`), never from wire input.
- **base64url consistency.** `createRecoverRequestSigners` base64url-encodes the signature; the handler
  `b64urlToBytes` it back before `verifyPeerSig`. Confirm the encode/decode pair matches (both db-core
  `bytesToB64url`/`b64urlToBytes`) and that the signing payload (`backfillSigningPayload` /
  `resumeSigningPayload`) is byte-identical on both sides (it is the same db-core function — confirm no
  field-ordering or canonicalization drift).
- **Replay-guard interaction (out of scope but adjacent).** The serve handler keys anti-replay on the
  signature bytes. The signer does not control that; just confirm a correctly-signed-but-replayed request is
  rejected (it is — covered in the handler spec, not this signer spec).

## Honest gaps to flag

- **No live-node coverage.** These tests exercise the signer against the **in-process** handler only. The
  end-to-end "signed request over a real socket" path is the gated node-composition ticket's
  (`reactivity-recover-node-wiring`) env-gated integration test — not exercised here.
- **Ed25519-only.** The substrate assumes Ed25519 identities (libp2p default). A node configured with a
  non-Ed25519 key would throw at first recover-request sign. This matches the cohort-topic substrate's
  standing assumption (`verifyPeerSig` is also Ed25519-only), but it is an unguarded assumption at the node
  level — flag if the project intends to support other key types.
- This sub-deliverable is small (~60 lines + spec); it is reviewed in isolation precisely because the node
  wiring that consumes it cannot land until 12.33 does. The consuming call-site (`createRecoverRequestSigners(nodePrivateKey)`
  inside the subscribe factory) is described in the gated ticket but **not yet present** in the tree.
