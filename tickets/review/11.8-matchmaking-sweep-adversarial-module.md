Final implement slice of the matchmaking application. Builds on `matchmaking-wire-anchor-registration` (foundation) and `matchmaking-query-filter-hangout` (query + hang-out engine). Three deliverables landed; everything is unit-tested behind injected ports (no live libp2p ÔÇö mock-tier e2e is the documented next milestone).

## What landed

### 1. Multi-cohort sweep + aggregate counts (hot-topic representative sample)
- **`db-core/multi-cohort-seeker.ts`** (pure): `selectShards(aggregate, {wantCount,ÔÇŽ})` ranks tier-1 prefix shards high-population-first, accumulating *log-bucketed* counts until they cover `wantCount`, capped at a fan-out ceiling (default 16). `runMultiCohortSweep(ports, opts)` fetches the root aggregate via an injected `MultiCohortSweepPorts`, optionally threshold-verifies it, queries each elected shard, and returns the **deduped + filtered + `registrationSig`-re-validated** union. A cold/untrusted aggregate Ôçĺ empty set (caller falls back to single-cohort).
- **`db-p2p/aggregate-counts.ts`**: `buildAggregateCount(ctx)` ÔÇö the root-side producer. **Depth-gated** (returns `undefined` when `treeDepth < aggregate_count_minimum_tier`, default 1), **log-bucketed** (`logBucketCount` = largest power of two ÔëĄ n; rounds *down* so the seeker over-provisions, never under-selects), **threshold-signed**.
- **Wire change (flag for reviewer):** `AggregateCountV1` gained a `signers: string[]` field. The previously-frozen type had only `signature`, but a threshold-sig blob is unverifiable without the signer set (chunk `i` Ôćö `signers[i]`) ÔÇö exactly like `PromotionNoticeV1`/`DemotionNoticeV1`. Documented in `matchmaking.md` ┬žWire formats ("Threshold-sig envelope (resolved)"). Confirm this divergence from the prior shape is acceptable.

### 2. Adversarial traffic bounds + reputation hooks
- **`db-p2p/traffic-validation.ts`**: `boundReportedTraffic(reply, walkState, cfg)` returns `capPatienceMs = max(0, patienceRemaining)` (over-report bound) and `escalateAfterTiers = 1` (under-report bound); `maxWalkHops(dMax) = (dMax+1)`. Emits `TrafficCrossCheckSignal`s; `reportTrafficCrossCheck(reputation, signals)` bridges to `IPeerReputation.reportPeer(ProtocolViolation)`. The bounds hold **by construction** of the existing toward-root-only walk ÔÇö this function documents/asserts them and adds the reputation emission point.

### 3. Public module + integration
- **`db-p2p/module.ts`**: `MatchmakingProviderSession` (register/renew/withdraw/setCapacity/signalFull) and `MatchmakingSeekerSession` (register/query/walk ÔÇö walk escalates to the sweep on a hot topic). `createMatchmakingQuorumDiscovery` binds the db-core voting `QuorumDiscovery` port (walk + sweep) ÔÇö the binding `voting-quorum.ts` already documented.
- `SeekerWalkResult.maxChildCohortCount` added (the hot-topic / sweep-escalation signal).

## Known gaps / honest flags (tests are a floor)
1. **No FRET-host wiring (the big one).** Every substrate seam is an *injected port*: `aggregate-counts` takes `shardCounts`/`treeDepth`/`thresholdSign`; `module.ts` takes `walkTransport`/`queryCohort`/`estimateDMax`/`sweepPorts`. **None are bound in `db-p2p/src/cohort-topic/host.ts` yet** ÔÇö the root `CoordEngine` does not yet feed `shardCounts` from child-cohort accounting, and there's no libp2p matchmaking-query/arrival protocol handler driving the seeker transports. So the public module + producer + sweep are not reachable from a running node. That binding is the `matchmaking-*-e2e-mock-tier` follow-on. This is the reason all of ┬žmatchmaking stays "mock-tier e2e pending".
2. **AggregateCountV1 threshold *verification* path is unbuilt.** The producer sets `(signature, signers)`; the consumer's `verifyAggregate` port is optional and **no concrete db-p2p binding splits the blob + checks it against the membership cert** (the `createVerifyOnlyThresholdCrypto`/cohort signer exist but aren't wired to a sweep `verifyAggregate`). Today the sweep trusts the aggregate unless a verifier is injected.
3. **`boundReportedTraffic` is not *called inside* `SeekerWalkClient`.** The walk already enforces the bounds structurally (deadline + toward-root), and the e2e test proves it, but the explicit signal-emission/reputation reporting is a standalone function not yet invoked in the live walk loop. Wiring it in (to actually emit cross-check signals during a walk) is a small follow-on.
4. **Untuned heuristics.** `overReportSuspectRatio = 8` and the under-report `arrivalsPerMin===0 && observedMatches>0` rule are advisory and have **no simulator backing** (unlike `contention_factor_cap`). `reportTrafficCrossCheck` forwards every suspect signal with no dampening ÔÇö a single divergence is not proof; the reputation subsystem must aggregate/decay. Reviewer may want to gate emission behind repeated divergence.
5. **`logBucketCount` rounds down (power-of-two floor).** Selection's `overprovision = 1` default relies on this (real Ôëą reported). If a future bucketing changes to round-to-nearest, revisit `selectShards`.
6. **Sweep queries all selected shards (no early-stop)** for representativeness ÔÇö up to `maxShards` RPCs even if an early shard already satisfies `wantCount`. Bounded but not adaptive.
