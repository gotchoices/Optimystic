## Why

Ticket 5-get-block-throws was a contract-drift bug: `StorageRepo` returned an error for a block that existed in pending-only state at one layer, while the caller at the next layer up expected `{state:{}}`. Each layer pair has two distinct "empty-ish" states — **does not exist** and **pending-only metadata** — and each must round-trip through every boundary without throwing and without being confused with "committed but empty."

## What to build

One spec, one file: `packages/db-p2p/test/empty-state-contract.spec.ts`. For each layer pair below, assert **both** empty-states round-trip correctly.

### Layer pairs

1. **`BlockStorage ↔ StorageRepo`**
   - Unknown block: `storageRepo.get({ blockIds: [<never-seen-id>] })` → `{ [id]: { state: {} } }`. No throw.
   - Pending-only block: insert a pending transform via `storageRepo.pend(...)` but never commit. Call `.get(...)` for the affected block — must return a response with `state.pending` populated, `state.latest` undefined, and `block` reflecting the pending apply (or undefined, depending on contract). Document the answer this spec pins down.

2. **`StorageRepo ↔ CoordinatorRepo`**
   - Same two scenarios, through `coordinatorRepo.get(...)` on a 1-node mesh. Assert the coordinator does NOT escalate an unknown block to a cluster fetch loop (or if it does, it terminates cleanly with `{state:{}}`).

3. **`CoordinatorRepo ↔ NetworkTransactor`**
   - Same two scenarios via `networkTransactor.get([{ blockIds: [...] }])`. Assert the merge path in `network-transactor.ts:118-133` does not raise "missing" for a canonical `{state:{}}` response.

4. **`NetworkTransactor ↔ Collection` (via `TransactorSource` + `Tracker`)**
   - `TransactorSource.tryGet(<unknown>)` → `undefined`.
   - `TransactorSource.tryGet(<pending-only>)` → either the pending-applied block or `undefined`, matching whatever contract Collection expects. Pin the answer.
   - `Tracker.tryGet` on a fresh tracker with an empty source: returns `undefined`, does not throw.

### Shape

Use the Phase 1 `createMesh(1, ...)` + `buildNetworkTransactor` helpers. Each layer-pair test should be small and self-contained; group them in `describe` blocks named for the pair. Per-test timeout 3-5s.

### Explicit non-goals

- Not a replacement for layer-specific unit tests. This spec is about the *seam* between layers, not the implementation of either side.
- Not multi-node. Solo mesh is enough to exercise the contract; multi-node empty-state behavior is covered in Phase 3.
- Not the StorageRepo-only legacy tests (already exist in `storage-repo.spec.ts`).

## Expected outcomes

- All four contracts are pinned by tests. Any future refactor that shifts one layer's empty-state representation fails here first, not in a canary.
- Likely surfaces at least one latent contract mismatch (pattern: this is how ticket-5 was discovered).
