# Reactivity tail rotation, slow-subscriber backpressure, and Edge/Core policy

This ticket implements the three remaining reactivity concerns that ride on the hot path built
by [reactivity-origination-replay-delivery](#): the rotating tail anchor and its drain protocol,
slow-subscriber backpressure that isolates a flaky phone from the rest of the fan-out, and the
Edge/Core T3 willingness policy plus the consolidated configuration surface.

Grounded in `docs/reactivity.md` §Rotating tail anchor (L54–60), §Slow-subscriber backpressure
(L175–189), §Tail rotation (L263–285), §Per-cohort policy (L302–308), §Configuration (L430–452),
and the §Worked scenarios for rotation (L478–482).

## Architecture

### Rotating tail anchor

`topicId = H(tailId ‖ "reactivity")` is derived per emission using the shared helper defined in
the origination ticket. When the tail block fills (`block_fill_size` transactions, default 64) a
new tail block is born and `tailId` — hence `topicId` — changes. From the cohort-topic layer's
perspective the new `topicId` is an **entirely new topic** (a fresh tree at a new ring coord);
reactivity manages the subscriber/replay-state migration explicitly. Rotation is detected
subscriber-side when the delivered `tailId` (or `rotationHint.newTailId`) differs from
`ActiveSubscription.tailIdAtAttach`.

Rotation trades a periodic, bounded re-registration storm for distributing the root duty across
the ring over time (avoiding a permanent hotspot/attack target).

### Tail rotation protocol (`docs/reactivity.md` L263–285)

1. **Pre-announce.** While committing the block-filling transaction, the outgoing tail cohort
   embeds `rotationHint{ newTailId, effectiveAtRevision }` in that notification (this fills the
   `rotationHint?` field reserved on `NotificationV1` by the origination ticket). The hint reaches
   every active subscriber via the existing tree.
2. **Drain.** The outgoing tail cohort keeps accepting **renewals** and serving **replays** for
   `T_drain` (default 60 s) after rotation, but rejects **new subscriptions** with a
   `Promoted`-shaped redirect to the new `topicId`'s tree.
3. **Jittered re-registration.** On the hint, subscribers schedule re-registration at the new
   `topicId` with random jitter over `T_rejoin_jitter` (default 30 s, the cohort-topic value).
   Re-registration carries the subscriber's existing `lastRevision`; revisions are continuous
   across rotations, so no replay confusion.
4. **Forwarder draining.** Forwarders under the old tail observe their direct-subscriber count
   drop and **demote naturally** per the cohort-topic demotion protocol. They migrate no state;
   the new tree rebuilds via re-registration.
5. **Buffer-to-checkpoint handoff.** As it drains, the outgoing tail folds its replay buffer into
   a final `CheckpointSummary` covering `[lastCheckpoint.toRevision + 1, rotationRevision]` and
   hands it to the new tail cohort — the **only** state migration across rotations. The new tail
   holds the old checkpoint to serve `ResumeV1` spanning the rotation. (The `CheckpointSummary`
   type and the `ResumeReplyV1.TailRotated` variant are owned by
   [reactivity-backfill-resume-checkpoints](#); this ticket produces the handoff checkpoint and
   triggers the `TailRotated` condition. Coordinate the `latestKnownTailId`-staleness check
   across the two tickets.)

**Anticipatory warm-up** (`docs/reactivity.md` L279–281): when the tail block reaches
`block_fill_size − warm_threshold` (default 56 of 64) transactions, the outgoing tail
opportunistically biases FRET pre-dialing toward likely-successor coords. Best-effort; no state
migrates until the actual rotation.

### Slow-subscriber backpressure (`docs/reactivity.md` L175–189)

The forwarder primary maintains a per-subscriber bounded queue with drop-oldest semantics
(fills the `perSubscriberQueue` field reserved on `PushState`):

```ts
interface BoundedQueue {
	capacity: number;            // queue_max, default 32 revisions
	entries: NotificationV1[];
	dropped: number;             // monotone counter
}
```

When a subscriber's queue is full and a new notification arrives, the **oldest** entry is
dropped and `dropped` increments. The subscriber detects the gap on the next delivery (a revision
jump) and issues a `BackfillV1` against the replay buffer (the backfill RPC is owned by the
sibling ticket; this ticket relies on it). This isolates a slow subscriber so one flaky phone
does not stall fan-out to the rest of the cohort. Memory is bounded by
`cohort_subscribers × queue_max × notification_size`.

### Edge/Core policy (`docs/reactivity.md` §Per-cohort policy L302–308)

Reactivity is **T3 (luxury)** at the cohort-topic layer:

- A cohort under heavy T0 load reports willingness=false for T3 → `UnwillingCohort` on subscribe;
  subscribers back off and retry (cohort-topic standard).
- **Edge nodes never serve as reactivity forwarders** — only as subscribers (T3 *consumer* is
  fine; T3 *producer* is off in the Edge profile). Enforce this where a node decides whether to
  instantiate `PushState`/become a forwarder.
- `deltaMaxBytes`: Edge = 0 (rejects inbound `delta`, re-reads chain), Core = 4096. This is the
  authoritative home for the `delta_max` Edge/Core plumbing the origination ticket only consumed.
- Per-cohort topic budget (`topics_max`) bounds collections served; no additional admission
  policy beyond the cohort-topic default.

### Configuration surface

This ticket is the home for the consolidated reactivity defaults table (`docs/reactivity.md`
§Configuration L430–452), shared by all three reactivity tickets:

| Parameter | Default | Owner ticket |
|---|---|---|
| `W` | 256 | origination |
| `W_checkpoint` | 4096 | backfill-resume |
| `dedupe_window` | 64 | origination |
| `queue_max` | 32 | this ticket |
| `delta_max` (Core) | 4096 / Edge 0 | this ticket |
| `T_drain` | 60 s | this ticket |
| `warm_threshold` | 8 | this ticket |
| `block_fill_size` | 64 | this ticket |

TTL/ping/T_rejoin_jitter are inherited from the cohort-topic defaults.

### cohort-topic.md interaction note

Add to `docs/cohort-topic.md` a short reactivity-interaction note: the new `topicId` produced by
a tail rotation is treated as a **fresh topic** from cohort-topic's perspective (new tree, new
coord), and the old tree drains/demotes via the standard demotion protocol while the new tree
forms via standard registration/promotion.

### Simulator-dependent parameters

`T_drain` (60 s), `queue_max` (32), and `block_fill_size` (64) are **provisional** pending
[simulator-reactivity-replay](#) (which measures the rotation re-registration burst vs
`cap_promote_fast` inside `T_drain`, and whether `queue_max` should scale with cohort size/tier)
and the [fold-simulator-findings-into-design-docs](#) bridge. Source them from the shared defaults
table so the fold-back can revise them without touching the protocol code.
