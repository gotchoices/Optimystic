## Background

Read-dependency capture is **block-granular**: `ReadDependencyCollector.record(blockId, revision)`
fires for *every* block read, whether the transaction cared about the block's *value* or merely
walked *through* it to reach a value. B-tree navigation reads interior nodes (separator keys /
child pointers); log-chain walks read chain nodes. All of them land in `transaction.reads`, which
the validator checks for staleness at PEND (Theorem 5). A transaction that touches more blocks
therefore observes more revisions and is more likely to be stale-rejected by an unrelated
concurrent change to a *structural* block it never depended on — the exact fairness inversion the
starvation ticket flags (Theorem 5 already calls these "false positives from structural block
reads … a performance concern, not a correctness concern").

This ticket lets purely-structural navigation reads be excluded from the conflict (read) set —
but ONLY where dropping them cannot admit a correctness violation. Getting the safety predicate
exactly right is the whole difficulty; this is `hard`.

## The safety predicate — when a structural read may be dropped

Define a read's **purpose** at capture time:

- **VALUE read** — the transaction's result depends on this block's *content* (a leaf/entry it
  read, a log entry it consumed, an interior node whose separator keys / child-counts it used as
  data). ALWAYS retained in the conflict set.
- **NAVIGATION read** — an interior B-tree node or log-chain node read *only* to descend toward a
  target that is itself separately captured as a VALUE read. Eligible to be dropped **iff** the
  descent terminates in a VALUE read whose block id + revision is in the conflict set.

**Why the navigation read is then safe to drop:** any concurrent restructuring that would change
the queried result necessarily also bumps the revision of a block that *stays* in the conflict
set. Concretely, for a point lookup that descends to a leaf:

- A split/merge/rebalance that moves the queried key to a different leaf must **modify the leaf
  the transaction actually read** (remove/redistribute its entries), bumping that leaf's revision.
  The retained leaf VALUE read catches it → still stale-rejected. The *interior* node revision was
  redundant.
- A change to the value at the queried key bumps the leaf revision directly → caught.

So the interior read is load-bearing only when the retained VALUE read does **not** cover the
structural change. Enumerate the cases where it is **NOT** safe and the structural read MUST stay:

1. **Range / scan / iteration** (`BTree.range`, `Tree.range/ascending`, `chain` head/tail walks
   that enumerate). The *result set* depends on tree/chain *shape*: a concurrent split that adds a
   new leaf inside the scanned range, or a chain append that extends the tail, changes the answer
   without necessarily modifying any leaf/entry the scan already read. Interior/boundary structural
   reads for a range are VALUE reads — retain them.
2. **Aggregates / counts / existence-by-structure** — anything that reads an interior node's
   child-count, subtree size, or separator key *as data* (not just to pick a child pointer).
   Retain.
3. **Boundary probes** — `first()`, min/max, "does a key exist in this range" answered from a
   separator without descending to a leaf. The interior node IS the value. Retain.
4. **Descent that does NOT terminate in a captured leaf VALUE read** — e.g. a lookup for an
   absent key (miss) that stops at an interior node, so there is no leaf in the read set to carry
   the dependency. Retain the deepest structural read reached (it is load-bearing for the
   negative result).
5. **Log-tail / critical-block reads** used for commit ordering (the log tail is the race arbiter,
   `transactor-source.ts` tail binding). These are load-bearing for OCC "tail binding" — retain.

Only the **point-lookup interior-descent** case (1 target key, terminates in a captured leaf
VALUE read, value-of-key semantics) is safe to drop. Implement conservatively: default every
read to retained; drop ONLY when the reader can positively assert the safe case.

## Design

### Tag reads by purpose

Extend the capture path so a read can be marked `navigation` vs `value`:

- `ReadDependencyCollector.record(blockId, revision, purpose)` — default `value` (fail-safe: an
  un-tagged read is retained). Track purpose per block id; if the SAME block is ever recorded as
  `value` from any path, it stays `value` (value-wins over navigation, analogous to the existing
  max-wins on revision).
- The B-tree point-lookup path (`Tree.find/get` → `BTree` descent in `tree.ts` / `btree/`) tags
  interior-node reads `navigation` and the terminal leaf read `value`. Range/scan/boundary/miss
  paths tag everything `value` (i.e. do not opt in).
- Chain node walks (`chain.ts`) tag intermediate nav nodes `navigation` only where the walk
  terminates in a captured entry the transaction consumed; enumerations tag `value`.

### Exclude safe navigation reads from the conflict set

`getReadDependencies()` gains a mode (or a second accessor) that returns only the
conflict-relevant set = all `value` reads + any `navigation` read whose descent did NOT terminate
in a retained `value` read. The transaction's `reads` (used for the id, client signature, and PEND
stale-check) is built from this filtered set. Structural reads that are safely covered are dropped
before `reads` is assembled — so the id/signature are computed over the reduced set consistently
on both client and validator (both re-derive the same filtered set deterministically).

**Determinism requirement:** validators re-execute and must produce the identical filtered read
set (Theorem 4). The purpose tagging must be a deterministic function of the query shape, not of
timing/cache state — a validator re-running the same point lookup must tag the same interior
reads `navigation`. Keep the tagging in the query logic (tree/chain), not in the cache layer.

## Edge cases & interactions

- **Split moves the key (the load-bearing case).** Test: T reads key K via point lookup (interior
  reads dropped, leaf L retained @rev r). A concurrent txn splits L, moving K to a new leaf and
  bumping L's revision. T must STILL be stale-rejected because L@r is retained. Assert rejection —
  this proves dropping the interior read did not admit a lost update.
- **Interior change that does NOT affect the queried key.** Test: a rebalance elsewhere bumps an
  interior node T only navigated through, without touching leaf L. T must NOT be rejected (this is
  the false-positive we are removing). Assert commit succeeds.
- **Range scan retains structure.** Test: T does a range scan; a concurrent split adds a leaf in
  the scanned range. T MUST be rejected (range reads are `value`). Assert the structural read was
  retained.
- **Absent-key (miss) descent.** Test: T looks up a non-existent key, descent stops at an interior
  node (no leaf captured). A concurrent insert of that key bumps the interior node. T MUST be
  rejected (deepest structural read retained for the negative result).
- **Boundary probe / first()/aggregate.** Test: T's result comes from an interior separator/count;
  a concurrent change to that interior node MUST reject T.
- **Value-wins tagging.** A block read once as `navigation` and once as `value` (same txn) is
  retained. Unit-test the collector's purpose merge (value beats navigation; revision still
  max-wins).
- **Determinism across re-execution.** A validator re-executing the same statements produces the
  identical filtered read set as the coordinator (so `createTransactionId` and the client
  signature match). Test client vs validator agreement on the reduced set.
- **Client-signature / id interaction.** Because `reads` now excludes safe structural reads, both
  `createTransactionId` and `clientSignaturePayload` are computed over the reduced set — verify the
  round-trip and that a node with a signature verifier accepts the reduced-set signature. (This
  couples to the signature work already in the tree — do not regress it.)
- **Log-chain tail binding preserved.** The commit-ordering tail/critical-block read stays in the
  set; a concurrent tail advance still stale-rejects. Test the tail is never dropped.
- **Interaction with invalidation cascade (Theorem 8b).** The cascade matches read-dependents by
  `(blockId, revision)` in `ActionEntry.reads`. Dropping safe structural reads shrinks that set
  identically on the recording side, so a dependent that only *navigated* through an
  invalidated-but-value-unchanged block is no longer spuriously re-invalidated — this is the SAME
  false-positive reduction (Theorem 8b's Bound), and is desirable. Verify a cascade test still
  catches genuine value-dependents (leaf/entry reads retained).
- **Conservative default.** Any read the query logic does not positively classify stays `value`.
  A regression that forgets to tag = more retained reads = more false-positive rejections, never a
  missed conflict. Add a test asserting the un-tagged default is `value`.
