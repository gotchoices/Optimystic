# Single-source and fail-fast-couple the super-majority threshold

## Background — the drift

`superMajorityThreshold` is the fraction of a cluster's peers that must agree
(promise) before a transaction can proceed. It is read in several components,
and the **fallback default when config is absent diverges across every one of
them**:

- `packages/db-p2p/src/cluster/cluster-repo.ts:250` — `ClusterMember` defaults
  to **1.0** (unanimity).
- `packages/db-p2p/src/repo/coordinator-repo.ts:97` — `CoordinatorRepo` policy
  defaults to **0.75**.
- `packages/db-p2p/src/libp2p-node-base.ts:606` — the node composition root
  defaults to **0.67**.
- test/harness/plugin sites (`mesh-harness.ts`, `collection-factory.ts`) use
  **0.75** and **0.51**.
- the `ClusterConsensusConfig` doc comment (`db-core/src/cluster/structs.ts:51`)
  says "default 0.75 = 3/4".

The danger the plan ticket names: a **member expecting unanimity (1.0) while the
coordinator commits at 0.75** is a latent phase-disagreement — the coordinator
declares a transaction committed on a super-majority the member does not accept
as sufficient. Today this only bites when the two are configured from *different
sources* (e.g. one takes its explicit config, the other falls through to its
divergent default). It is silent: nothing at startup catches the mismatch.

## Design decision (resolved)

Two coupled changes:

1. **Single source for the default.** Define one exported constant —
   `DEFAULT_SUPER_MAJORITY_THRESHOLD = 0.75` — next to `ClusterConsensusConfig`
   in `packages/db-core/src/cluster/structs.ts` (the type's own doc already says
   0.75). Every fallback (`?? 1.0`, `?? 0.75`, `?? 0.67`) that is a *default*
   for a real member/coordinator becomes `?? DEFAULT_SUPER_MAJORITY_THRESHOLD`.
   - Value chosen 0.75 because the coordinator — the component that actually
     commits — already uses it, and the type documents it. Raising the member
     from 1.0 to 0.75 removes the unanimity-vs-supermajority split. (1.0 was
     "backward compatibility" per its comment; nothing depends on it being
     unanimous, and the coupling check below guarantees agreement regardless.)
   - **Explicit caller overrides are NOT touched.** `collection-factory.ts`'s
     `0.51` and any harness value are deliberate explicit configs, not
     fallbacks — leave them. This ticket only unifies the *default* and enforces
     *agreement*, it does not force one global value on every deployment.

2. **Fail-fast coupling at the composition root.** In a live node, the
   `ClusterMember` and the `CoordinatorRepo` must run on the **same**
   `superMajorityThreshold`. `libp2p-node-base.ts` (~605-616) is where the node
   assembles `consensusConfig` and wires both. Ensure ONE resolved threshold
   value flows to both (single object, or derive both from one source), and add
   a **startup assertion**: if the member's effective threshold and the
   coordinator policy's effective threshold are not equal, throw a clear error
   at construction/startup naming both values — never let the node come up
   mismatched. The check must hold however the value arrives (shared default,
   explicit `clusterPolicy` override, or per-deployment config).

## Expected behavior

- One exported `DEFAULT_SUPER_MAJORITY_THRESHOLD` constant; member, coordinator,
  and node-base fallbacks all reference it — no more `1.0` / `0.75` / `0.67`
  literals as *defaults* (explicit overrides retained).
- A node whose member and coordinator would run different super-majority
  thresholds **fails at startup** with a message naming the two conflicting
  values and where each came from — it does not start and later disagree
  mid-consensus.
- When configured consistently (the normal path), no behavior change beyond the
  member no longer silently defaulting to unanimity.

## Edge cases & interactions

- **Coupling holds across all config sources.** The startup check must fire for
  a mismatch introduced via defaults, via `options.clusterPolicy`, or via a
  per-deployment config object — validate at startup, **not** at commit-time
  (a commit-time mismatch is already a failed consensus, far too late).
- **Single-node / local / test meshes.** Confirm the unified default and the
  coupling assert do not break the `allowUnvalidatedSmallCluster` / single-node
  paths — a 1-peer cluster with threshold 0.75 still needs `ceil(1 * 0.75) = 1`
  promise, so unanimity-of-one still works. Add/adjust a test so single-node
  bring-up is not regressed by dropping the member's 1.0 default.
- **Vote-count arithmetic unchanged.** `Math.ceil(peerCount * threshold)` at
  `cluster-repo.ts:741`, `:1324` and `cluster-coordinator.ts:280` must keep
  producing the same counts for the same threshold — only the *default source*
  changes, not the formula.
- **Do not conflate with `simpleMajorityThreshold` or
  `membershipAdmissionFraction`.** Those are separate thresholds (commit
  quorum / which-set-may-vote) and are out of scope — only `superMajority`
  couples here.
- **Harness values.** `mesh-harness.ts` and `collection-factory.ts` pass
  explicit thresholds; if any of those are members+coordinators in the same
  in-process mesh, they must pass the *same* value to both or the new startup
  assert will (correctly) reject them — audit those call sites and align them.
