# Wire RebalanceMonitor + react via BlockTransferCoordinator on a live node

## Background

`RebalanceMonitor` (`packages/db-p2p/src/cluster/rebalance-monitor.ts`) is entirely inert in
production, exactly like `SpreadOnChurnMonitor` was before
`optimystic-spread-on-churn-monitor-wiring` (now complete):

- `NetworkManagerService.initRebalanceMonitor(...)` (`network-manager-service.ts:72`) is **never
  called**.
- The monitor is never `start()`ed, so its `connection:open` / `connection:close` listeners
  (`rebalance-monitor.ts:73-74`) are never registered.
- Nothing feeds its `trackBlock` (`rebalance-monitor.ts:99`).
- Its `onRebalance` handler (`rebalance-monitor.ts:95`) is never connected to anything.
- `BlockTransferCoordinator` (`block-transfer.ts:31`) — the existing reaction primitive that pulls
  gained blocks and pushes lost ones (`handleRebalanceEvent`, `block-transfer.ts:102`) — is
  **never constructed** anywhere in `src` (only in `test/block-transfer.spec.ts`).

This ticket activates the rebalance path end-to-end: drive the monitor, and connect its
`RebalanceEvent { gained, lost, newOwners }` to `BlockTransferCoordinator.handleRebalanceEvent`.

This is the **wire + react** half. A follow-on (`unify-monitor-tracked-block-set`) collapses the two
monitors onto a single shared owned-block set and drives eviction from the `lost` signal — do **not**
do that here; here each monitor keeps its own `Set`, both fed from the same
`storageRepo.onAnyCollectionChange` feed.

## Where to wire it

Both dependencies are constructed **only inside the `enableArachnode` block** in
`createLibp2pNodeBase` (`libp2p-node-base.ts:731-792`):

- `fretAdapter = new ArachnodeFretAdapter(fret)` (`libp2p-node-base.ts:736`) — required by
  `RebalanceMonitorDeps.fretAdapter`.
- `restorationCoordinatorV2 = new RestorationCoordinator(...)` (`libp2p-node-base.ts:755`) —
  required by `BlockTransferCoordinator` (it pulls gained blocks through `restore()`).

So the wiring belongs **inside that `if (fret) { ... }` block, after `restorationCoordinatorV2` is
constructed**. That matches the ticket's "when arachnode/FRET are available" gate naturally — when
arachnode is disabled or FRET is absent, there is no `fretAdapter`/`restorationCoordinator` and the
rebalance path stays inert (acceptable: rebalance is a resilience optimization).

Mirror the spread wiring's robustness (`libp2p-node-base.ts:670-727`): a wiring failure must be
**non-fatal** (log + continue, rebalance inert), unlike the operator-opted-in `cohortTopic` block
which hard-fails.

## Reaction semantics

`BlockTransferCoordinator` constructor:
`(repo, peerNetwork, restorationCoordinator, partitionDetector, protocolPrefix, config?)`.

- `repo` → the **local** `storageRepo` (not `repoProxy`/`coordinatedRepo`): a pulled/pushed replica
  must land in / be read from this node's own storage, same reasoning as the `blockTransfer`
  service registration (`libp2p-node-base.ts:451-459`).
- `peerNetwork` → `keyNetwork`.
- `restorationCoordinator` → `restorationCoordinatorV2`.
- `partitionDetector` → the existing `partitionDetector`.
- `protocolPrefix` → `protocolPrefix` (`/optimystic/<networkName>`), so pushes dial the same
  block-transfer protocol the node registers its handler under — the exact failure the spread
  ticket fixed (`block-transfer.ts:194` builds `BlockTransferClient` with this prefix).

Connect via `rebalanceMonitor.onRebalance(event => { void coordinator.handleRebalanceEvent(event) })`.
`handleRebalanceEvent` already pulls `gained` and pushes `lost` to `newOwners` concurrently
(`block-transfer.ts:102-115`), each guarded by `partitionDetector`.

## Owned-block feed

Feed `rebalanceMonitor.trackBlock` from the **same** `storageRepo.onAnyCollectionChange` feed that
drives spread (`libp2p-node-base.ts:698`). For this ticket, add a **separate** subscription inside
the arachnode block (the unify follow-on collapses the two into one). Same caveat the spread ticket
documented applies: blocks already durable from a previous run are not re-emitted on startup, so
they are not tracked until next touched — acceptable here, and `unify-monitor-tracked-block-set`
references the deferred initial-scan seed (`optimystic-owned-block-initial-scan-seed`).

## Teardown

Add a `node.stop` wrapper (compose with the existing wrappers — each calls its captured
`previousStop` last, see `libp2p-node-base.ts:717-727`, `783-788`, `828-834`) that releases the
rebalance owned-block subscription and stops the monitor. Note `NetworkManagerService.stop()`
already stops `rebalanceMonitor` (`network-manager-service.ts:149-151`); `RebalanceMonitor.stop()`
early-returns when not running, so the double-stop is idempotent — belt-and-suspenders, same as
spread.

## Exposure

Expose `(node as any).rebalanceMonitor = rebalanceMonitor` and
`(node as any).blockTransferCoordinator = coordinator` for tests/diagnostics, mirroring
`(node as any).spreadOnChurnMonitor` (`libp2p-node-base.ts:710`).

## Optional config gate

Consider a `NodeOptions.rebalance?: { enabled?: boolean }` flag mirroring `spreadOnChurn`
(`libp2p-node-base.ts:166`) so a node can opt out. Default on (when arachnode+FRET present). Keep it
thin — do not invent tuning beyond `RebalanceMonitorConfig` which already exists.
