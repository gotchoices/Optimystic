# Walk-toward-root lookup, promotion/demotion lifecycle, and cold-start instantiation

This ticket wires together the dynamic behavior of the tree: how a participant **finds**
its cohort (walk-toward-root), how a cohort **grows/shrinks** (promotion/demotion), and how
a cohort **comes into being** (cold-start). They fold together because the walk's
`Promoted`/`NoState` replies are produced by the promotion lifecycle, and a `Promoted`
redirect (or a `bootstrap` root request) is exactly what triggers cold-start
instantiation.

## Walk-toward-root lookup

Per `docs/cohort-topic.md` §Tree growth and lookup (L90-144). A participant starts at
`d_max` (from the addressing ticket) and walks toward the root, handling the reply set:

```ts
export type WalkOutcome =
	| { kind: "accepted"; reply: RegisterReplyV1 }
	| { kind: "promoted"; targetTier: number }   // recompute coord at targetTier, register there
	| { kind: "retry_later"; afterMs: number };  // unwilling_cohort

export interface WalkEngine {
	register(topicId: Uint8Array, tier: number, appPayload?: Uint8Array): Promise<WalkOutcome>;
}
```

Reply handling (single-direction semantics — the anti-flood backbone):
- `NoState` at tier `d` → walk one tier toward the root (`d-1`); no traffic signal.
- `Promoted(d+1)` → recompute `coord_{d+1}(self, topicId)` and register there (move
  outward only on explicit redirect).
- `UnwillingMember(candidates)` → retry the **same** coord at a named alternative member.
- `UnwillingCohort(retryAfter)` → back off in **time**, then restart from `d_max` (not the
  same coord) — decorrelates retries across the ring.
- `bootstrap: true` set on a root cold-start request (see cold-start below).

Routing goes through the db-core `ITopicRouter` port (see `cohort-topic-package-layering`),
whose db-p2p implementation wraps FRET's `RouteAndMaybeAct` (`registerMaybeAct`/`sendMaybeAct`):
`key = coord_d(self, topicId)`, `activity = RegisterV1`, `wantK = k`, `minSigs = k − x` (used
for promotion/demotion responses). Post-registration pings call `ITopicRouter.dialMember` to
reach the cached `primary` directly, falling back to `routeAndAct` only when the primary is
unreachable. The walk logic itself stays FRET-free in db-core.

## Promotion / demotion lifecycle

Per §Promotion and demotion lifecycle (L346-381):

**Promote** when, for a quorum:
- `directParticipants(T) ≥ cap_promote` (default 64), OR
- `loadBarometer[tier(T)].bucket ≥ bucket_overload` AND `directParticipants(T) ≥
  cap_promote_fast` (default 32) — the hot-load fast path from the barometer ticket.

Threshold-sign a `PromotionNoticeV1` (via the signer from the gossip ticket); future
registrations get `Promoted(d+1)`; existing participants unaffected.

**Pre-promotion on slope:** if the `directParticipants(T)` growth slope predicts crossing
`cap_promote` within `T_promote_lookahead` (default 30s), promote now — avoids overshoot
from gossip lag.

**Sticky anti-flap:** promotion state is sticky for ≥ `T_promote_sticky` (default 60s)
before re-evaluating, so transient drops don't flap back to accepting.

**Demote** when, for a quorum: `directParticipants(T) ≤ cap_demote` (default 16) AND held
for ≥ `T_demote` (default 5 min) AND no live child cohorts. Threshold-sign a
`DemotionNoticeV1` to the parent ("drop me as a tier-(d+1) child"); release forwarder
state.

```ts
export interface PromotionLifecycle {
	onParticipantCountChange(topicId: Uint8Array, now: number): void;  // may emit PromotionNoticeV1
	maybeDemote(topicId: Uint8Array, now: number): DemotionNoticeV1 | undefined;
	isPromoted(topicId: Uint8Array): boolean;
}
```

## Cold-start instantiation

Per §Cold-start instantiation (L369-377): a cold cohort instantiates as a forwarder for `T`
when it receives a `RegisterV1` for a `T` it doesn't serve AND either the registration's
`bootstrap: true` is set (root case) or it arrives as a follow-on to a parent's `Promoted`
redirect, AND a quorum of members is willing to serve `T` at that tier. The new forwarder
registers with its tier-`(d−1)` parent at first opportunity; until that registration acks,
it accepts participants but holds notifications/queries that need parent involvement.

### Resolved open question (GROUNDING): burst at a just-promoted cohort

A cohort that has just promoted but whose tier-`(d+1)` isn't fully instantiated yet, on a
burst of new same-tier registrations: **bounce with `Promoted(d+1)`** (cheap single-RPC) —
do not buffer and do not fail with `UnwillingCohort`. The sticky window (`T_promote_sticky`)
keeps it in promoted mode through the burst. Document this resolution.

## Simulator dependency

`cap_promote`, `cap_promote_fast`, `T_promote_lookahead`, `T_promote_sticky`, `T_demote`
are the headline simulator-validated parameters (`simulator-promotion-convergence` measures
overshoot and convergence; `simulator-participant-walk` measures hop counts and the
sparse-regime fan-out). Build to the folded-back values. The walk-fans-across-ring and
"promotion fires before overshoot exceeds bound" claims are checked against simulator
numbers in the mock-tier e2e suite.
