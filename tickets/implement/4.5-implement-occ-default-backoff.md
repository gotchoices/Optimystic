## Background

When two transactions touch the same block, one loses. The loser must retry with a fresh
transaction id. Today the single-collection retry loop (`Collection.sync`) already does
exponential backoff (`baseBackoffMs * 2^(n-1)`, capped at `maxBackoffMs`, `collection.ts:345`),
with a consecutive-no-progress cap (`maxAttempts`) and an optional wall-clock `deadlineMs`.
Two gaps make this not yet a *safe, fair default*:

1. **No jitter.** All retries fire at deterministic delays. Under load, a set of clients that
   lost the same race re-attempt in lockstep — a thundering herd that re-collides on the next
   round. The codebase already has a jittered helper, `backoffRetryMs` in
   `cohort-topic/willingness.ts` (exponential, capped, with proportional random jitter); the
   sync loop should use the same shape.

2. **The multi-collection path has no retry loop at all.** `TransactionCoordinator.commit()`
   and `execute()` (coordinator.ts) throw on a stale loss / `CoordinatorPartialCommitError`
   and leave retry entirely to the caller. The shipped default should retry a *clean,
   fully-retryable* stale loss (nothing durably committed) with the same backoff+jitter policy,
   while a **partial landing** (`committedCollections` non-empty) must NOT be blindly retried —
   it requires reconciliation (Theorem 3). This ticket adds the retry only for the clean-loss
   case and leaves the partial-landing case throwing exactly as today.

This ticket ships backoff+jitter as the default retry policy. It does NOT add priority aging
(that is `implement-occ-priority-aging`, which builds on this) and does NOT change race
resolution at the cluster.

## Design

### Jittered backoff helper (db-core)

Introduce one shared helper (or reuse `backoffRetryMs`) computing the delay for attempt `n`:

```
delay(n) = min(base * 2^n, cap) * (1 - jitterFraction * rand())   // "full-ish" jitter, decorrelated
```

- `jitterFraction` default ~0.5 (delay ∈ [50%, 100%] of the exponential value).
- Randomness uses the same CSPRNG path db-core already uses (`@noble/hashes` `randomBytes`),
  NOT `Math.random`, to stay consistent with the rest of the package.
- Keep it pure/injectable enough that a test can supply a deterministic rand for assertions.

Wire this into `Collection.sync`'s existing delay computation (replace the bare
`Math.min(base * 2^(n-1), cap)` at `collection.ts:345`). Preserve the existing
`consecutiveFailures` reset-on-progress semantics, `deadlineMs`, abortable sleep, and
`SyncRetryExhaustedError` contract untouched.

### Multi-collection retry loop (db-core coordinator/session)

Add a bounded retry around the coordinate step in `commit()` (and, if it is the retryable
entry point, `execute()`) that:

- Retries ONLY when the failure is a clean, fully-retryable stale loss — i.e. the trackers
  were restored to their pre-append snapshot (the EMPTY-committed-set branch, coordinator.ts:239)
  so re-driving re-appends cleanly.
- Re-reads / re-derives fresh revisions before each re-attempt (mirroring how `sync` calls
  `updateInternal()` before retrying) so the next attempt pends against current state rather
  than immediately re-failing stale.
- Uses the same backoff+jitter helper, `maxAttempts`, and `deadlineMs` knobs, exposed through
  the same options shape as `SyncOptions` (share the type; do not fork two policy vocabularies).
- Throws immediately (no retry) on `CoordinatorPartialCommitError` — a partial landing is not
  retryable. Also honor an abort signal.

Defaults must be safe out of the box (a caller that passes no options gets bounded,
jittered retry). Keep the knobs configurable per call.

## Edge cases & interactions

- **Partial landing must not retry.** A `CoordinatorPartialCommitError` (committedCollections
  non-empty) escapes the retry loop unchanged; blindly retrying re-logs already-durable actions.
  Test: a two-collection commit where one collection stale-loses at COMMIT throws the partial
  error and the coordinator is NOT re-driven.
- **Clean stale loss retries then succeeds.** A stale loss at PEND (nothing durable) backs off
  and, after a re-read that clears the conflict, commits. Test asserts bounded attempt count and
  eventual success (mirror `collection.spec.ts` "recover from transient stale failures").
- **Budget exhaustion is bounded.** An always-losing transaction exhausts `maxAttempts` (or
  `deadlineMs`) and surfaces a terminal error rather than spinning forever, without holding a
  latch open. Test: attempt count == cap, no infinite loop.
- **Abort mid-backoff.** An abort during the backoff sleep rejects promptly with an AbortError
  (single- and multi-collection paths), matching the existing sync abort test.
- **Jitter never zero-delays and never exceeds cap.** Property test: for all attempts,
  `0 < delay(n) <= cap` and delay is non-decreasing in expectation; jitter fraction respected.
- **Thundering-herd shed.** With jitter, N clients that collide at t=0 do not all re-attempt at
  the identical next tick (deterministic-rand test asserting spread), so offered load per window
  does not cascade (cf. the willingness.spec.ts "synchronized burst sheds offered load" test).
- **Deadline vs attempt cap independence.** Preserve the existing distinction: `deadlineMs`
  stops on wall-clock regardless of a large `maxAttempts`, and vice versa (existing
  collection.spec.ts tests must still pass).
- **Healthy multi-batch sync not tripped.** The cap counts consecutive *no-progress* failures;
  a legitimate large multi-batch sync that commits progress each round must still pass under a
  small `maxAttempts` (existing test locks this — keep it green).
