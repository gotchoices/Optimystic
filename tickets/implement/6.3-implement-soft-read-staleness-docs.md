# Document soft-read staleness (and cross-ref ejection-locality)

## Background — the footgun

`CoordinatorRepo.get` (`coordinator-repo.ts:181-242`) does a **soft** proximity
check: if this node is not formally responsible for a block, it logs
`proximity:get-warning` ("serving read for non-responsible block") and **serves
the read anyway** for graceful degradation. Separately, in the default `lazy`
read-repair mode, a *present* block is only re-verified against the cluster once
its local copy is older than `readRepairWindowMs` (**default 10000 ms**); within
that window a read is served from local state without consulting peers.

Both are **safe by design** — this is the read-path behavior formalized in
`docs/correctness.md` Theorem 15 (Read-Path Integrity): a transaction that acts
on stale data is caught at *validation* time via read-dependency revision
checks, so stale reads never corrupt committed state. But a read issued *outside*
a transaction (or whose result a caller trusts without committing) can quietly
return **data up to one commit stale within the ~10 s window**, with no
error surfaced to the caller — only an internal log line. An app author who has
not read Theorem 15 will not expect this.

This is a **documentation** ticket. No behavior change. The plan ticket
`design-consensus-hygiene-notes` asks to document the behavior and its window
where app authors will actually see it, and to cross-reference the (already
parked) ejection-locality concern rather than duplicate it.

## Expected behavior (of the docs)

- App-facing documentation states plainly, where an application author reads
  about consistency (primary: the consistency / read-path section of
  `docs/transactions.md`; cross-linked from
  `packages/quereus-plugin-optimystic/README.md` for SQL users):
  - A read may be served by a node that is not the responsible one (graceful
    degradation); this succeeds without a caller-visible error.
  - In the default `lazy` read-repair mode, a locally-present block is not
    re-verified against the cluster until it is older than `readRepairWindowMs`
    (**default 10 s**), so a read can return data that is behind the cluster's
    latest commit.
  - **Worst-case staleness bound:** within the read-repair window a caller can
    observe state up to one commit behind current; after the window, the next
    read triggers repair and converges. Give the concrete knobs
    (`readRepairMode`, `readRepairWindowMs`) and note that `paranoid` mode
    verifies on every read (no window) for callers that cannot tolerate any
    staleness.
  - Point to `docs/correctness.md` Theorem 15 for why this is safe *inside a
    transaction* (validation catches stale reads), so authors can decide whether
    a bare soft read is acceptable for their use.
- `docs/right-is-right.md`'s ejection/reputation-locality open question gains a
  one-line cross-reference to the backlog ticket
  `backlog/speculative/7.5-gossip-reputation-blacklisting.md` — that ticket is
  the home; **do not restate or duplicate the analysis here.**

## Edge cases & interactions (things the doc must get right)

- **State the bound, not just the behavior.** "May be stale" is not enough — the
  author needs the *worst case* (bounded by `readRepairWindowMs`, one commit
  behind) to judge acceptability. Name the exact default (10 000 ms) and the
  config knob.
- **Transaction vs. bare read.** Be explicit that the safety argument
  (Theorem 15) applies to reads *used as transaction read-dependencies*; a read
  whose value is consumed and never committed carries no such check. This is the
  crux of the footgun — do not blur it.
- **Do not overstate risk.** This is documented *degradation*, not a bug; the
  tone is "here is a knob and its bound", not "warning, broken". Keep it
  actionable.
- **Ejection item is reference-only.** The cross-ref must not re-open or
  re-argue the reputation-locality question — a single pointer line, so the
  design-review finding is accounted for without a second home.
