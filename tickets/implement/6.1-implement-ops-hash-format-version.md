# Pin and version the operations-hash byte format

## Background — what already shipped

The "operations hash" is the fingerprint a transaction coordinator sends in
`PendRequest.operationsHash`, which every validator independently recomputes and
compares to decide the transaction's operations match. The **canonical byte
form** for that hash already landed in the completed ticket
`txn-operations-hash-non-canonical` — the live implementation is
`packages/db-core/src/transaction/operations-hash.ts`:

- operations sorted by the tuple `(collectionId, blockId, type)` (string compare
  on the first two; `TYPE_RANK` insert<update<delete on the third),
- `canonicalStringify` (recursively sorts object keys, preserves array order,
  mirrors `JSON.stringify` leaf semantics),
- SHA-256 → base64url via `hashString`, emitted as `ops:${hash}`.

So the *serialization is already canonical*. Two gaps remain, both from the
plan ticket `design-consensus-hygiene-notes`:

1. **No version tag.** `hashOperations` already carries a `NOTE:` (lines
   ~132-137) spelling out the exact hole: the canonical encoding differs from
   the old raw-JSON encoding, so the hash value changed, and there is **no
   protocol-version gate**. In a mixed-version cluster an old node and a new
   node disagree and reject each other with the generic `Operations hash
   mismatch` — indistinguishable from a genuine content disagreement or a
   Byzantine lie. This ticket closes that: make the hash self-describing about
   its format version.

2. **Docs are stale / non-normative.** `docs/transactions.md` still shows two
   *illustrative* `hashOperations` bodies that do NOT match the shipped code:
   the throwaway 32-bit `((hash<<5)-hash)` example (~955-968) and a
   `hash(JSON.stringify(...))` sketch (~1219-1229). Neither is the real
   canonical form. The canonical serialization must be **specified normatively**
   in the docs, pointing at `operations-hash.ts` as the source of truth, so all
   implementations hash the same bytes and the format can evolve.

## Design decision (resolved — do not re-litigate)

**The version tag lives inside the ops-hash string itself, not in a new
`TransactionStamp` field or a `PendRequest` type change.** Change the token from
the bare `ops:` prefix to a versioned prefix — `ops.v1:` (choose one concrete
spelling and keep it; `ops.v1:` recommended so the delimiter can't collide with
base64url alphabet). Rationale:

- The hash already travels the wire as an opaque string compared for equality;
  a self-describing string needs **zero** protocol/type changes to
  `PendRequest` or `TransactionStamp`.
- A validator that receives a token whose version it does not recognize can
  raise a **distinct, clear error** ("unsupported operations-hash format version
  vN") instead of the ambiguous `Operations hash mismatch`. Version skew becomes
  diagnosable rather than looking like a content fault.

**Two versioned inputs, sitting alongside each other, together define the bytes
every node must agree on — document both:**

- `engineId` (already stamped in `TransactionStamp`, e.g. `quereus@0.5.3`) —
  versions the *operation content* an engine produces from the same statements.
- the ops-hash **format version** token — versions the *serialization* of that
  content into bytes.

This is what "specified and versioned alongside `engineId`" means in the plan
ticket: they are the two dimensions that must both match for two honest nodes to
agree, and the docs must name both.

**Migration is a non-issue for stored history, but must be handled on the wire.**
The operations hash is recomputed fresh on both sides and is **never persisted**
(confirmed in the `txn-operations-hash-non-canonical` review: "both sender and
validator recompute fresh, nothing persisted"). It is distinct from
`transaction.id` (`createTransactionId`, which hashes `stamp.id + statements +
reads` and IS persisted in logs/block references) — **do not touch
`createTransactionId` or its inputs.** Because the ops-hash is not in history,
bumping the format version cannot retroactively invalidate committed
transactions. The only compatibility surface is a live mixed-version cluster,
handled by the distinct-error behavior above.

## Expected behavior

- The ops-hash string is self-describing: it carries an explicit format-version
  token (`ops.v1:…`). `hashOperations` emits the current version; a single
  exported constant names the current version so there is one source of truth.
- The validator, when it recomputes and the **version tokens differ**, surfaces
  a distinct error naming the version mismatch — NOT the generic content
  `Operations hash mismatch`. When the versions match but the bodies differ, the
  existing content-mismatch error is unchanged.
- `docs/transactions.md` specifies the canonical serialization normatively:
  the sort key `(collectionId, blockId, type)`, `canonicalStringify` rules, the
  SHA-256/base64url step, and the versioned `ops.vN:` token — with a pointer to
  `operations-hash.ts` as the authoritative implementation. The two stale
  illustrative `hashOperations` bodies are replaced by this spec (or by a
  faithful excerpt of the real one), and the doc names both `engineId` and the
  format-version token as the two things that must match.
- The `NOTE:` at `hashOperations` is updated: the version gate now exists, so
  reword from "there is NO protocol-version gate; if rolling upgrades become
  supported, version-gate this" to describe the token and the distinct-error
  behavior (leave a `NOTE:` only for whatever residual concern remains — e.g.
  that nodes still do not *negotiate* or *cross-compute* older versions, they
  only *detect* skew).

## Edge cases & interactions

- **Detect vs. cross-compute.** Minimum bar is *detection*: a node recognizes a
  foreign version and errors clearly. This ticket does NOT require a node to
  compute historical versions (multi-version compat) — that is future work if
  rolling upgrades ship. State this boundary explicitly in the doc and the
  `NOTE:` so a reader does not assume mixed-version clusters interoperate; they
  fail *legibly*, not *silently*.
- **Token parsing robustness.** A validator must not crash or misclassify on a
  malformed/absent token from a peer — an input with no recognizable version
  prefix is treated as an unsupported/foreign version (distinct error), never as
  an accidental content match. Test with: current token, a bumped-version token,
  a bare legacy `ops:` token, and a garbage string.
- **Do not perturb `transaction.id`.** Add a test (or assertion) that the
  version-token change touches only the operations hash and leaves
  `createTransactionId` output byte-identical for the same inputs — history
  identity must not move.
- **Client-signature forward dependency.** A future ticket
  `design-client-transaction-signatures` will have the client signature cover
  the canonical serialized byte form pinned here. Ensure the versioned canonical
  bytes are cleanly derivable (the exact string `hashOperations` hashes, or a
  documented function returning it) so the signature can bind the *same* bytes
  the validators hash. Note this in the doc; no prereq (that ticket does not
  exist yet).
- **Coordinate with the shipped canonical form.** The ordering/canonicalization
  fix from `txn-operations-hash-non-canonical` is the form to specify — do not
  invent a different ordering. The version token wraps *that* form as v1.
