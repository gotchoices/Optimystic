## Multi-cohort sweep + aggregate counts

Per [matchmaking.md §Multi-cohort sweep L148-154](../../docs/matchmaking.md) and [§Wire formats AggregateCountV1 L409-425](../../docs/matchmaking.md). For hot topics whose providers span many tier-`d ≥ 1` cohorts, a seeker wanting a representative cross-ring sample:

1. Registers at its natural tier as usual.
2. Queries the **root** cohort, which returns `AggregateCountV1` — log-bucketed provider counts per tier-1 prefix shard (`bucketCounts[{targetTier:1, prefixSlot:0..F-1, count}]`), threshold-signed.
3. Selects high-population tier-1 cohorts and queries them directly to assemble the set.

Root-side aggregation (`db-p2p/src/matchmaking/aggregate-counts.ts`): compute and **threshold-sign** `bucketCounts`, but return them **only** when tree depth ≥ `aggregate_count_minimum_tier` (default 1) — cold cohorts that fall through to `NoState` do not produce this. Counts are log-bucketed (not exact entries). The sweep costs more RPCs and is reserved for representativeness-over-latency use cases (voting quorums, capability fairness audits); the common path remains the single-cohort sample from the previous ticket.

`db-core/src/matchmaking/multi-cohort-seeker.ts`: the seeker-side orchestration consuming `AggregateCountV1` and fanning queries to selected shards.

## Adversarial traffic-reporting defense

Per [matchmaking.md §Adversarial cohort traffic reporting L303-310](../../docs/matchmaking.md). `QueryReplyV1.topicTraffic` (and the registration reply's traffic) carry the cohort **primary's single-member signature, not a threshold signature**, because the response is advisory; the seeker re-validates each entry's `registrationSig` independently.

Documented worst-case bounds the implementation must preserve and tests must assert:
- **Over-reporting** (fake hot tier ⇒ seeker hangs out) is bounded by the seeker's `patienceMs`; worst case = wasted patience + one extra `register → walk` hop after timeout. No spatial flood — the decision rule only walks *toward the root*, never speculatively outward, so substrate anti-flood is preserved.
- **Under-reporting** (fake cold tier ⇒ seeker escalates) is bounded to **one extra hop per affected tier**, terminating at the root where aggregated truth is hardest to fake.
- **Cross-check via cohort gossip**: other cohort members can detect a primary whose reported rate diverges from the gossip-derived view; detection routes through the **reputation subsystem** (out of scope to implement; provide integration hooks/signal emission points — see [architecture.md §Reputation]).

```ts
// seeker-side bound enforcement (traffic-validation.ts)
interface TrafficBoundResult { trusted: boolean; capPatienceMs?: number; escalateAfterTiers?: number }
function boundReportedTraffic(reply: QueryReplyV1, walkState: SeekerWalkState, cfg): TrafficBoundResult
```

> **GROUNDING resolutions to document in `docs/matchmaking.md`:**
> - No threshold signature is added per `QueryReplyV1` — the single-member signature stands; the bounded worst-case above does not justify the per-reply threshold cost. A future ticket may revisit if observed abuse warrants.
> - Eligibility filtering for voting may be done either client-side via the reputation subsystem **or** via per-entry `registrationSig` verification on the query reply; matchmaking supports both (it carries identity + signatures, reputation provides the metric). The voting plan ticket decides the policy.

## Public module + cohort-topic integration

Per [matchmaking.md §Overview/Configuration](../../docs/matchmaking.md). Main entry points:

```ts
class MatchmakingProvider {
  register(topic: { kind: MatchTopicKind; label: string }, payload: ProviderAppPayloadV1): Promise<void>
  renew(/* capacity update / signal-full */): Promise<void>
  withdraw(): Promise<void>   // RenewV1 TTL=0
}
class MatchmakingSeeker {
  register(topic, payload: SeekerAppPayloadV1): Promise<void>
  query(q: QueryV1): Promise<QueryReplyV1>
  walk(want: { wantCount; filter?; patienceMs }): Promise<ProviderEntryV1[]>   // hang-out engine + optional sweep
}
```

Wire the public API through `CohortTopicService` (register/renew/lookup/withdraw at T2), exposing config defaults and the wire-type exports. `db-p2p/src/matchmaking/index.ts` and `db-core/src/matchmaking/index.ts` become the package public surfaces; add to package indexes. Repo rules: ES modules, no inline `import()`, no `any`, tabs, cross-platform.

## Voting handoff

This ticket does **not** implement voting. `matchmaking-voting-quorum-assembly` (plan) consumes this module: `topicId=H("quorum"‖proposalHash‖"match")`, eligible voters register as providers with eligibility-proof capabilities, coordinator sweeps + selects. See [matchmaking.md §Voting-quorum L158-176](../../docs/matchmaking.md) and §Worked scenario "Voting on a popular proposal" L463-471.
