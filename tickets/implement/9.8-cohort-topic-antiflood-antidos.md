# Anti-flood and anti-DoS defenses for cohort-topic

This ticket hardens the substrate, folding the two defensive sections of the spec:
**anti-flood** (structural properties that prevent self-inflicted storms) and **anti-DoS**
(defenses against malicious registration traffic). Several anti-flood properties are
already emergent from earlier tickets (single-direction walks, sticky promotion); this
ticket completes the set, adds the jitter/rate/budget/replay machinery, and makes the
guarantees testable.

## Anti-flood (§Anti-flood properties L385-395)

The five named floods and their defenses:

1. **Cold-start storm at the root** — participants probe `d_max` *first* (already in the
   walk ticket); the root only sees the tail of the walk for sparse topics.
2. **Re-registration storm after cohort failure** — stagger re-registration with random
   jitter over `T_rejoin_jitter` (default 30s, scaled with observed FRET cohort failure
   rate). The jitter window is set so inbound rate at the recovering/replacement cohort
   stays ≤ `cap_promote / T_rejoin_jitter`.
3. **Speculative outward probe** — eliminated by construction (outward only via `Promoted`;
   already enforced in the walk ticket; assert it here).
4. **Inward retry storm** — `UnwillingCohort` waits `retryAfter` then restarts from
   `d_max` (already in walk/willingness tickets; assert).
5. **Promotion feedback loop** — bounce in-flight registrants with `Promoted(d+1)`; sticky
   `T_promote_sticky` (already in promotion ticket; assert).

```ts
export interface RejoinJitter {
	scheduleRejoin(now: number): number;   // returns a jittered timestamp within T_rejoin_jitter
}
```

## Anti-DoS (§Anti-DoS L397-406)

```ts
export interface RegisterRateLimiter {
	check(peerId: PeerId, topicId: Uint8Array, now: number):
		{ ok: true } | { ok: false; retryAfterMs: number };  // default 4/min/peer/topic, exponential retryAfter
}

export interface TopicBudget {
	admit(topicId: Uint8Array): boolean;   // false when topics_max (2048) exhausted -> UnwillingCohort
	touch(topicId: Uint8Array, participantCount: number): void;  // LRU bookkeeping
	// eviction: LRU by participant count; zero-recent-registration topics dropped first
}

export interface CorrelationReplayGuard {
	accept(correlationId: Uint8Array, peerId: PeerId, timestamp: number, now: number): boolean;
	// rejects stale-timestamp or replayed correlationId
}

export interface BootstrapEvidence {
	verify(reg: RegisterV1, tier: number): boolean;
	// T0/T1: signed reference to a committed parent topic (no PoW).
	// T2/T3: proof-of-work OR reputation-score signature OR signed parent-topic reference.
}
```

- **Per-peer rate limit:** `register_rate_per_peer` default 4/min per peer per topic at a
  single cohort; exceeded → `UnwillingCohort(retryAfter)` with exponential `retryAfter`.
- **Per-cohort topic budget:** `topics_max` default 2048 forwarder topics; LRU by
  participant count, zero-recent topics dropped first; budget exhaustion → `UnwillingCohort`
  for new instantiations (existing topics continue).
- **Signed registrations + replay guard:** `RegisterV1` carries `correlationId` (16 random
  bytes) + signature over `(topicId, tier, correlationId, timestamp)`; drop
  stale-timestamp / replayed-correlationId.
- **Bootstrap evidence:** cold-root `bootstrap: true` requires PoW / reputation /
  signed-parent-reference per tier (T0/T1 generally need no PoW; T2/T3 do).

The layer does not defend against unbounded Sybil at the registration level — that's
FRET's / the reputation subsystem's concern (note this).

### GROUNDING DoS open question (note, not resolve)

The doc raises whether a malicious peer hitting all `F^d` coordinates at tier `d`
(e.g. F=16, d=3 → 4096 shards × 4/min = ~16K registrations/min network-wide) is acceptable
fan-out. Per-coord rate limits bound per-cohort cost; whether the aggregate is acceptable
is a reputation-escalation question. Reference this open question in the doc; the simulator
DoS scenario (`simulator-metrics-and-scenarios` adversarial scenario) informs whether
reputation must escalate faster. Do not resolve here.
