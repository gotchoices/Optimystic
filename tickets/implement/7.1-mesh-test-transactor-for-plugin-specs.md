## Why

`createTestTransactor()` wraps `MemoryRawStorage → BlockStorage → StorageRepo` into an `ITransactor` shim. The production path — `NetworkTransactor → CoordinatorRepo → StorageRepo` — is never exercised by the plugin test suite. Every schema/engine/migration/distributed spec uses this shortcut, so the `Collection/Tree/Log/Chain ↔ CoordinatorRepo` contract has zero joint coverage at the plugin layer.

## What to build

A `'mesh-test'` transactor mode in `collection-factory.ts` that constructs (per test-run, not per call):

```
MemoryRawStorage
  → BlockStorage
  → StorageRepo
  → ClusterMember + CoordinatorRepo  (via mesh-harness createMesh(1, ...))
  → NetworkTransactor                (via buildNetworkTransactor(mesh))
```

The plugin must see an `ITransactor` indistinguishable from production wiring. The mesh mock network already handles 1-node findCoordinator/findCluster, so no real libp2p is needed.

### Interface decisions

- **Name**: `'mesh-test'`. Keep `'test'` (StorageRepo pass-through) for the narrow cases that *explicitly* target StorageRepo-only behavior. Do not silently redirect `'test'` → real stack; suites that only want a StorageRepo pass-through should opt in.
- **Sharing**: export `buildNetworkTransactor(mesh)` from `mesh-harness.ts` so `fresh-node-ddl.spec.ts` and the plugin factory use the exact same construction — no drift.
- **Cross-package dependency**: `quereus-plugin-optimystic` already depends on `@optimystic/db-p2p`. The mesh helpers live under `packages/db-p2p/test/`; move `mesh-harness.ts` + `buildNetworkTransactor` under `packages/db-p2p/src/testing/` (or add a `db-p2p` test-entry export) so consumers can import it without reaching into `test/`. Keep the existing re-export path for `mesh-sanity.spec.ts` intact.

### Migration

Switch each of the four specs below from `default_transactor: 'test'` → `'mesh-test'`:
- `schema-support.spec.ts`
- `engine-execution.spec.ts`
- `schema-migration.spec.ts`
- `distributed-quereus.spec.ts`

Run them. Each failure is a latent bug; file a separate fix ticket per failure rather than tweaking tests to pass.

### Explicit non-goals

- No real libp2p. The mock `MockMeshKeyNetwork` + `responsibilityK: 1` is sufficient.
- Do not delete `'test'` (StorageRepo pass-through). Keep as an opt-in for targeted StorageRepo regression specs.
- Do not touch `'local'` or `'network'` modes.

## Expected tests

- `schema-support.spec.ts`, `engine-execution.spec.ts`, `schema-migration.spec.ts`, `distributed-quereus.spec.ts` pass under `'mesh-test'`.
- New small spec `packages/quereus-plugin-optimystic/test/mesh-test-transactor.spec.ts` that sanity-checks the factory: `CREATE TABLE` + `INSERT` + `SELECT` under `'mesh-test'`, with an explicit 5s per-test timeout so any hang fails loud.
- `fresh-node-ddl.spec.ts` continues to pass after adopting the exported `buildNetworkTransactor`.
