# Cohort-topic host + change-bridge: production node activation

The seam ticket (`reactivity-change-event-tail-and-bridge-seam`) put the committed `tailId` on the
change event and widened the bridge gate to take the whole event. This ticket flips activation **on** in
the production node: it constructs the cohort-topic host on the running libp2p node and installs the
bridge so a commit landing on a reactivity-cohort-member node actually invokes
`CohortTopicService.onLocalCommit(event, cert)` with the authoritative cluster commit cert.

After this lands, `docs/internals.md` §"Wiring status" flips from *dormant* to *live*.

## What is already in place (reuse, do not rebuild)

- `createCohortTopicHost(node, fret, options)` (`cohort-topic/host.ts`) — post-node host: registers the
  five protocols, sets FRET's activity handler, runs the gossip-cadence driver, exposes
  `{ service, registry, protocols, stop() }`. Needs `options.privateKey` for real threshold signing.
- `createCommitCertStore()` + `makeClusterCommitCertExtractor(store)` (`cluster/commit-cert.ts`).
- `ClusterMember`'s `onCommitCertificate(actionId, cert)` sink, fired **synchronously before**
  `storageRepo.commit`, already threaded from `NodeOptions.onCommitCertificate` → `clusterMember(...)`
  (`libp2p-node-base.ts:464`).
- `attachCohortChangeBridge(node, deps) → { notifier, unsubscribe }` (from the seam ticket): sets
  `node.blockChangeNotifier` to the origination-decorating notifier and returns the catch-all teardown.
- `reactivityTopicId(tailId, hash?)` / `createReactivityTopicAnchor(hash?)`
  (`reactivity/topic-anchor.ts`) — `H(tailId ‖ "reactivity")` over db-core's default `createRingHash()`.
- `createTierAddressing(hash).coord0(topicId)` — `coord_0(_, topicId) = H(0x00 ‖ topicId)`.

## Where activation must live: end of `createLibp2pNodeBase`

**Decision (resolves the ticket's ordering open point): attach the bridge at the end of
`createLibp2pNodeBase`, before it returns the node — not in an external caller.**

Rationale: the Quereus `collection-factory.createNetworkTransactor` captures
`(node as any).blockChangeNotifier` **once**, immediately after `createLibp2pNode(...)` returns
(`collection-factory.ts:168`), and reuses that reference as `localChangeNotifier` for every
`NetworkTransactor` it builds. So the bridge must already be installed by the time the function returns.
The end of `createLibp2pNodeBase` is the only place that is *after* the node + FRET are fully assembled
(`node.start()` done, `fretSvc` available) yet *before* any caller can capture the notifier. This makes
the origination path live for **all** collections created on the node, not just those created after a
late external attach. (The host genuinely is "post-node": it consumes the assembled `node` + `fret` — and
the tail of `createLibp2pNodeBase` is post-node. No module-layering issue: `host.ts` is in the same
`db-p2p` package.)

The bare `(node as any).blockChangeNotifier = storageRepo` at `libp2p-node-base.ts:637` stays as the
default; the activation block below **replaces** it with the bridge notifier when the host is enabled.

## Activation gating

Add a `NodeOptions.cohortTopic` option (default **off**). When absent/disabled, behavior is exactly
today's: `blockChangeNotifier = storageRepo`, no host, no cert store, zero cohort cost.

```ts
/** Opt-in cohort-topic substrate (reactivity/matchmaking origination). Absent → dormant, zero cost. */
cohortTopic?: {
	enabled: boolean;
	/** Requested cohort size; MUST match the host's wantK so the gate's membership set agrees. Default 16. */
	wantK?: number;
	/** Optional pass-through host tuning (profile/minSigs/fanout/gossipIntervalMs/antiDos/promotion). */
	host?: Omit<CohortTopicHostOptions, 'privateKey' | 'wantK'>;
};
```

Keep `wantK` resolved **once** (e.g. `const wantK = options.cohortTopic?.wantK ?? 16`) and pass the same
value into both `createCohortTopicHost(...)` and the gate, so the cohort the gate checks is the cohort
the host serves.

## Wiring (end of `createLibp2pNodeBase`)

### Cert store + onCommitCertificate composition (must precede `clusterMember`)

The cert store is read post-assembly (by the extractor) but written by the cluster member during
consensus, so `store.put` must be the `onCommitCertificate` sink passed at `clusterMember(...)`
construction (line ~451-465). Create the store **early** (top of the function, once `cohortTopic.enabled`
is known) and compose with any caller-supplied sink:

```ts
const certStore = cohortEnabled ? createCommitCertStore() : undefined;
const onCommitCertificate: CommitCertificateSink | undefined =
	certStore
		? (actionId, cert) => { options.onCommitCertificate?.(actionId, cert); certStore.put(actionId, cert); }
		: options.onCommitCertificate;
// …pass `onCommitCertificate` (not `options.onCommitCertificate`) into clusterMember(...).
```

(The sink fires synchronously before `storageRepo.commit`, so the cert is retained before the change
event is emitted and the extractor resolves it — see `cluster-repo.ts:796-812`.)

### Post-assembly: host + gate + bridge

```ts
if (cohortEnabled) {
	const fret = (node as any).services?.fret as FretService | undefined;
	if (!fret) throw new Error('cohortTopic enabled but FRET service is unavailable on the node');

	const host = await createCohortTopicHost(node, fret, {
		...(options.cohortTopic!.host ?? {}),
		privateKey: nodePrivateKey,   // real k−x threshold signing
		wantK,
	});

	// selfIsCohortMember: this node is responsible for the collection's reactivity-topic fan-out iff it
	// is in the FRET cohort around coord_0(H(currentTailId ‖ "reactivity")). Default hashes (createRingHash
	// / createReactivityTopicAnchor / createTierAddressing) are byte-identical to the host's internal
	// `new RingHash()` and to the subscriber-side anchor, so the coord lines up across all three.
	const anchor = createReactivityTopicAnchor();
	const addressing = createTierAddressing(createRingHash());
	const selfPeerStr = node.peerId.toString();
	const selfIsCohortMember = (event: CollectionChangeEvent): boolean => {
		if (event.tailId === undefined) return false;            // promotions / tail-less events never originate
		const topicId = anchor.topicId(blockIdToBytes(event.tailId));
		const coord = addressing.coord0(topicId);
		return fret.assembleCohort(coord, wantK).includes(selfPeerStr);
	};

	const { unsubscribe } = attachCohortChangeBridge(
		node as unknown as { blockChangeNotifier?: IBlockChangeNotifier },
		{
			source: storageRepo,
			service: host.service,
			selfIsCohortMember,
			extractCommitCert: makeClusterCommitCertExtractor(certStore!),
		},
	);

	// Teardown: release the catch-all subscription + stop the host (clears the gossip timer, unhandles
	// the five protocols) before the node's transports go down. Compose with the existing stop wrappers.
	const previousStop = node.stop.bind(node);
	node.stop = async () => { try { unsubscribe(); await host.stop(); } finally { await previousStop(); } };
}
```

Resolve the `event.tailId` (a `BlockId`, which is a string) → bytes conversion the **same way** the rest
of db-p2p encodes block ids for hashing (confirm the existing convention — block ids are strings;
`reactivityTopicId` takes raw bytes). Reuse the established string→bytes helper used elsewhere for ring
hashing of ids (e.g. the `TextEncoder`/`peer-codec`/`uint8arrays` convention already in cohort-topic);
do **not** invent a new encoding — it must match whatever the subscriber side feeds `reactivityTopicId`
(`reactivity/topic-anchor.ts`) so origination and subscription resolve the same coord. Verify against
`subscription-manager.ts` (`tailIdAtAttach` is raw bytes) and the real-crypto spec
(`reactivity-real-crypto.spec.ts` uses `b64urlToBytes(TAIL)`); the production tail is a `BlockId` string,
so pin the exact bytes encoding during implementation and assert it in the test.

### `selfIsCohortMember` — the resolved design (closes the ticket's open points)

- **Tail at emit time**: from `event.tailId` (delivered by the seam ticket) — no async storage read, no
  per-collection registry. `undefined` (read-driven promotion) → not a member → no-op.
- **Topic + coord**: `coord_0(H(tailId ‖ "reactivity"))` via the shared anchor + tier addressing.
- **Membership**: `fret.assembleCohort(coord, wantK)` (peer-id strings) includes
  `node.peerId.toString()`. Live FRET read per commit; membership churn between commits is fine
  (reactivity is hint-only).
- **Hash/wantK alignment**: gate uses db-core default `createRingHash()` (== host `new RingHash()` ==
  subscriber default) and the same `wantK` as the host. This is load-bearing — a mismatch means the gate
  checks a different cohort than subscribers register against and origination silently never fires.

## Host-construction failure policy

When `cohortTopic.enabled` is true, a failure to construct the host (or a missing FRET service) is a
**hard error** that fails node startup — the operator explicitly opted in, so silently degrading to the
bare notifier would hide misconfiguration. (Reactivity being hint-only governs *runtime* loss tolerance,
not *activation* of a requested subsystem.) Document this; a future flag could soften it if a deployment
wants best-effort activation.

## Edge cases & interactions

- **Host disabled (default)**: no cert store, `onCommitCertificate` stays caller-supplied,
  `blockChangeNotifier = storageRepo`. Existing node/integration suites must stay green unchanged.
- **FRET unavailable while enabled**: throw at activation (clear message). Test it.
- **Promotion / tail-less event**: gate short-circuits on `event.tailId === undefined`.
- **No retained cert**: `makeClusterCommitCertExtractor` returns `undefined` (logged) → bridge no-ops;
  never fabricates a cert. (Already covered by the bridge, but assert end-to-end here.)
- **onLocalCommit not yet set**: when no reactivity origination consumer has installed `onLocalCommit`
  (the `ReactivityOriginationManager` install + notification emit transport are owned by the reactivity
  origination wiring, NOT this ticket), the bridge no-ops at its `if (!hook) return;` guard. This ticket
  is "live" in the sense that the **bridge invokes** `onLocalCommit` whenever a consumer is attached; the
  integration test attaches a probe `onLocalCommit` on `host.service` to prove the invocation. Do not
  install the origination manager or a stub emit transport here.
- **Teardown**: `node.stop()` must call `unsubscribe()` + `host.stop()` before transports close, and
  compose with the existing arachnode + clusterMember stop wrappers (each wraps `node.stop` and calls its
  captured `previousStop` last). Verify no double-stop / no leaked gossip `setInterval` after stop.
- **Single node / forming mode**: with `bootstrapNodes: []`, FRET assembles a cohort containing at least
  self, so self is a member of every reactivity topic — the integration test exploits this for a
  deterministic, fast, single-node assertion (the multi-node real-libp2p exercise is `14-substrate-e2e-real-libp2p-tier`).
- **Concurrency**: the gate is synchronous and pure (a FRET read + hashing); concurrent commits firing
  it are safe.
- **wantK / hash drift**: a future change to the host's default `wantK` or ring hash must update the gate
  in lockstep — keep them sourced from one place and assert alignment in the test.

## Key tests (TDD targets — new `host-node-activation.spec.ts`, single forming node)

- **Live origination**: create a node with `cohortTopic: { enabled: true }` and a validator/cluster that
  produces commit certs (or drive a commit through the local cluster path so `onCommitCertificate`
  fires); set a probe `host.service.onLocalCommit = (event, cert) => …`; commit a block whose `tailId`
  anchors a topic whose `coord_0` cohort includes self (true on a forming node); assert the probe fires
  once with the right `collectionId`/`rev`/`actionId` and a `cert` whose `thresholdSig`/`signers` match
  the consensus cert byte-for-byte (no re-sign).
- **Cert resolution from the live store**: assert the cert delivered to `onLocalCommit` is the one the
  cluster captured via `onCommitCertificate` (same bytes), proving the store→extractor path, with no
  test-only cert shim.
- **Non-member / tail-less no-op**: a commit whose reactivity coord excludes self (force via a stubbed
  membership or a coord the gate rejects), and a read-driven promotion (undefined tail), each leave
  `onLocalCommit` un-invoked.
- **Disabled node**: `cohortTopic` absent → `node.blockChangeNotifier` is the bare `storageRepo`
  (feature-detect: it has `onAnyCollectionChange`/`onCollectionChange` but no origination decoration);
  no host; per-collection reactive-watch still works.
- **Teardown**: after `node.stop()`, a further commit (if reachable) does not invoke the probe and the
  gossip interval is cleared (no dangling timer).
