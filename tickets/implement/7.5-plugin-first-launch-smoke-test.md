## Why this spec exists

Phase-1–4 coverage map:

| Layer                                        | Real libp2p? | Cold-restart? | DDL via plugin? |
| -------------------------------------------- | ------------ | ------------- | --------------- |
| db-core Tree over solo libp2p (Phase 1)      | yes          | no            | no              |
| db-core Tree over 3-node mesh (Phase 3)      | no (mock)    | no            | no              |
| empty-state contract (Phase 4)               | no (mock)    | no            | no              |
| real-libp2p integration (ticket-4 review)    | yes          | yes (sc. 5)   | no (pend/get)   |
| existing plugin specs (schema-support etc.)  | no           | no            | yes             |

The intersection "real libp2p + cold-restart + DDL through the plugin" has no coverage today. That is precisely the surface a PR can silently break and neither the fast suite nor ticket-4's integration gate will notice. One extra spec in the `OPTIMYSTIC_INTEGRATION=1` lane closes the gap.

This is **not** a sereus-health canary. That framing is abandoned because (a) the on-device bug in ticket 5-chain-add is RN-specific and will not reproduce in Node, and (b) mimicking `CadreService.doStart` creates a drift dependency we cannot maintain. Instead the spec exercises *the plugin's own* documented entry point (`Database.exec` with `USING optimystic(...)`), which the plugin repo owns and controls.

## Shape of the spec

A single spec file, gated at the top via:

```ts
before(function () {
  if (!process.env.OPTIMYSTIC_INTEGRATION) this.skip();
});
```

One `describe` block, two `it`s (a third optional — see Test Plan below). Helpers local to the file; promote if a second plugin-integration spec lands.

### `it 1` — first-launch happy path (hard requirement)

Cold-boot a real libp2p node. Register the plugin against a `transactor` that resolves to the real `NetworkTransactor` wired to that node. Run:

```sql
CREATE TABLE canary (
  id INTEGER PRIMARY KEY,
  value TEXT NOT NULL
) USING optimystic('tree://canary-smoke/canary');

INSERT INTO canary (id, value) VALUES (1, 'first');
SELECT value FROM canary WHERE id = 1;  -- expect 'first'
```

Success criteria: no throw, no mocha timeout, row round-trips. Failure = regression in the plugin's first-DDL path over the real stack.

### `it 2` — cold-restart over shared storage (hard requirement)

Pattern lifted from `real-libp2p.integration.spec.ts` scenario 5:

1. Boot node A over a pre-constructed `MemoryRawStorage` instance with a pre-constructed Ed25519 `privateKey`.
2. Register plugin + run the `CREATE TABLE` + `INSERT` from `it 1`.
3. Stop node A (`await node.stop()`). Do **not** touch the storage or key.
4. Boot node B over the **same** `MemoryRawStorage` instance and the **same** `privateKey`. Register a fresh `Database` + plugin against node B.
5. Run `SELECT value FROM canary WHERE id = 1` — expect `'first'`.

Success criteria: the row persists across the restart, no schema-reload error, no "non-existent chain" or `{state:{}}` throw on the second session. This is the Node-side proxy for "app relaunch after first-launch DDL."

### `it 3` — two sequential DDLs in one session (optional; include if cheap)

A single `Database` session running two `CREATE TABLE` statements back-to-back with an `INSERT` between them, to exercise the "second DDL in same session" path referenced in the blocked-ticket Q3. Skip if it adds >2 s or complicates the helpers — `it 1` + `it 2` are the required minimum.

## Wiring notes for the implementer

The existing plugin specs register with `default_transactor: 'mesh-test'` (see `schema-support.spec.ts:26`). The canary needs a real `NetworkTransactor`. Two possible routes — pick whichever lands cleaner:

- **Route A (preferred if supported):** the plugin accepts a programmatic transactor instance at `register()` time or via a registry. If so, build the `NetworkTransactor` the same way `fresh-node-ddl-libp2p.spec.ts` does (lines 44–60) and inject it. This keeps the spec self-contained.
- **Route B (fallback):** the plugin only accepts string names that resolve via a transactor registry. Register a new named transactor (`'libp2p-canary'` or similar) in the spec's `before()` that returns the real `NetworkTransactor`, and configure `default_transactor: 'libp2p-canary'`. Grep `plugin.ts` / `register` / `default_transactor` in `quereus-plugin-optimystic/src` to find the registry surface before writing the spec.

If neither route exists, **do not add one for the canary** — instead mark this ticket blocked on "plugin needs a programmatic transactor injection API" and file a separate plan ticket for that API. A canary that requires bespoke plugin changes to run is a net negative.

Other wiring:

- Node config: `port: 0`, empty bootstrap, `networkName: 'plugin-first-launch-canary'`, `clusterSize: 1`, `arachnode: { enableRingZulu: true }`. Identical to `fresh-node-ddl-libp2p.spec.ts`.
- Suite timeout: 30 s. Per-`it` timeout: 15 s (arachnode init dominates).
- Cleanup: `afterEach` stops every node via `Promise.allSettled`. If `it 2` stops node A mid-body, splice it out of the tracked list to avoid double-stop (pattern from `real-libp2p.integration.spec.ts`).
- Key + storage construction for `it 2`: `generateKeyPair('Ed25519')` and `new MemoryRawStorage()` before the first boot; pass both to **both** `createLibp2pNode` calls. Construct each `Database` fresh per boot — a stale `Database` object pointing at a stopped node is not the restart scenario.

## Package wiring

Mirror `packages/db-p2p/package.json`'s gate exactly:

```json
"test:integration": "OPTIMYSTIC_INTEGRATION=1 npm test"
```

POSIX-form; the existing `real-libp2p.integration.spec.ts` top-of-file comment covers the Windows `$env:` alternative — copy that comment into the new spec verbatim so cross-platform usage is self-documenting.

Default `npm test --workspace @optimystic/quereus-plugin-optimystic` must remain green with the spec showing as pending (skipped). `OPTIMYSTIC_INTEGRATION=1 npm test --workspace @optimystic/quereus-plugin-optimystic` must show the canary as passing.

## Non-goals

- No React Native / device / emulator coverage. The blocked ticket (`5-chain-add-on-fresh-collection-throws-non-existent-chain`) remains blocked; this canary is **not** expected to reproduce it. If it ever does, that is a bonus, not a success criterion.
- No multi-node mesh. A single-node canary is the smallest spec that exercises the plugin→NetworkTransactor→libp2p path with a restart; multi-node is already covered in Phase 3 at the db-core layer.
- No promotion to default `npm test`. The gate stays.
- No changes to `optimystic-module.ts` or `schema-manager.ts`. Those files are referenced read-only.

## Test plan (what the implementer should verify before moving to review)

- `npm test --workspace @optimystic/quereus-plugin-optimystic` → spec pending, no regressions to existing plugin specs.
- `OPTIMYSTIC_INTEGRATION=1 npm test --workspace @optimystic/quereus-plugin-optimystic` → spec passes end-to-end.
- `npm run build --workspace @optimystic/quereus-plugin-optimystic` → clean.
- Spec completes in under 10 s with the env var set (arachnode init is the bottleneck; if it exceeds 15 s on first run, raise the per-`it` timeout before investigating — it is not a bug to chase in this ticket).
- Intentional break check: temporarily broken `SchemaManager.storeSchema` (e.g. throw at top of method) must make `it 1` fail, not hang. If it hangs, the spec's timeout budget is wrong and needs tightening before the ticket ships.
