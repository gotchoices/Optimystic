# Tier addressing (coord_d), tier ladder config, and network-size-driven d_max

This ticket folds three closely-coupled concerns into one cohesive module: the tier
**addressing** hash, the tier-**ladder** configuration (T0-T3 + Edge/Core profiles), and
the network-size-driven **d_max** computation. They are coupled because `d_max` is the
start tier of every walk and is computed from `F` (an addressing constant) and the FRET
size estimate, and because the tier ladder gates which `tier` values an addressing call is
even allowed to produce for a given node profile.

## Tier addressing

Per `docs/cohort-topic.md` В§Tier addressing (L58-67):

```
coord_0(_, topicId) = H(0x00 вЂ– topicId)
coord_d(P, topicId) = H(d вЂ– prefix(P, dВ·logв‚‚F) вЂ– topicId)   for d в‰Ґ 1
```

- `H` is SHA-256 truncated to FRET's ring width `B` (256 bits / 32 bytes вЂ” reuse FRET's
  `COORD_BYTES`/`RING_BITS` rather than hardcoding).
- `prefix(P, n)` = the `n` most-significant bits of peer ID `P`, left-padded if shorter.
- `F` = fan-out (default 16, `logв‚‚F = 4`); tier `d` has exactly `F^d` coordinates.

Interface:

```ts
export interface TierAddressing {
	readonly F: number;          // fan-out, default 16
	coord0(topicId: Uint8Array): RingCoord;
	coordD(d: number, peerId: Uint8Array, topicId: Uint8Array): RingCoord;  // d>=1
	coord(d: number, peerId: Uint8Array, topicId: Uint8Array): RingCoord;   // dispatches d===0
}
```

Properties to preserve (and test): sibling cohorts at tier `d` (peers sharing the first
`dВ·logв‚‚F` bits) deterministically converge on one coord; different topics and different
tiers of the same topic produce **uncorrelated** coords. Reuse FRET's hash/coord
primitives (`hash.ts`); do not reimplement ring math. `RingCoord` is FRET's coord type.

## Tier ladder configuration

Per `docs/cohort-topic.md` В§Tier ladder (L148-167):

```ts
export const enum Tier { T0 = 0, T1 = 1, T2 = 2, T3 = 3 }  // essential..luxury

export interface NodeProfile {
	kind: "edge" | "core";
	willingTiers: ReadonlySet<Tier>;  // edge: {T0,T1}; core: {T0,T1,T2,T3}
}

export function edgeProfile(overrides?: Partial<...>): NodeProfile;   // T0+T1
export function coreProfile(overrides?: Partial<...>): NodeProfile;   // all tiers
```

- **Edge** (mobile/browser/IoT): T0 + T1 only; refuses forwarder roles for T2/T3.
- **Core** (servers): T0-T3, subject to per-node operator overrides (an operator may
  restrict a core node to fewer tiers).
- The decline policy table (T0 never declined while serving, T3 declined freely) is
  *willingness* behavior consumed by a later ticket; this ticket only encodes the static
  tier classes, the profiles, and the per-node override surface. The Edge per-tier config
  overrides from В§Configuration L630-637 (`ttl=60s`, `ping_interval=20s`, T2/T3 willingness
  permanently off, sticky-cached backups) belong to the willingness/TTL tickets вЂ” surface
  the profile *flags* here, not the runtime behavior.

## Network-size-driven d_max

Per `docs/cohort-topic.md` В§Maximum useful depth (L76-86):

```
d_max = max(0, вЊЉlog_F(n_est)вЊ‹ в€’ 1)
```

If `n_est` confidence < `confidence_min` (default 0.3), clamp to
`d_max = вЊЉd_max_cap / 2вЊ‹` (`d_max_cap` default 60). Computed **lazily**; participants
don't need it precise.

```ts
export interface DMaxComputer {
	dMax(): number;  // reads FRET estimate lazily, applies clamp
}

export function makeDMaxComputer(deps: {
	estimate: () => SizeEstimate;   // wraps FRET estimateSizeAndConfidence(store, m)
	F: number;
	confidenceMin: number;          // default 0.3
	dMaxCap: number;                // default 60
}): DMaxComputer;
```

Integrate FRET's `estimateSizeAndConfidence(store, m)` вЂ” do not reimplement size
estimation. The clamp engages when the returned confidence is below `confidence_min`.

## Simulator dependency

Prereqs `fold-simulator-findings-into-design-docs` because `F` and `d_max_cap` (and the
`confidence_min` threshold) are exactly the parameters the simulator validates:
`simulator-promotion-convergence` confirms steady-state depth в‰€ `ceil(log_F(N/cap_promote))`
and `simulator-fret-cohort-model` confirms the clamp engages below the confidence band.
Build to the folded-back values; if the simulator revised `F` from 16, this module's
default must match the revised doc.
