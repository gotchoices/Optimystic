# Cohort-topic package layering: pure core vs. transport binding

This ticket is the **single source of truth** for where cohort-topic / reactivity /
matchmaking code lives. Every substrate ticket prereqs it (directly or transitively via
`cohort-topic-wire-formats`) and must follow the rule below. It exists because
`docs/architecture.md` defines **db-core as having "no network or disk assumptions"** while
the substrate is fundamentally DHT-backed — so the substrate must be *split*, not dropped
wholesale into either package.

## The rule

- **`packages/db-core/src/cohort-topic` — pure.** Logic that depends only on a hash
  function and byte-array peer IDs: wire-format types + codecs, tier addressing (`coord_d`,
  `prefix`, `d_max`), the registration store + TTL/eviction, the willingness state machine,
  the promotion/demotion state machine + hysteresis, deterministic primary/backup sharding,
  the capacity barometer and topic-traffic math, the walk **decision** logic, the
  gossip-**merge** logic, and (for the apps) the reactivity dedupe/replay/checkpoint/resume
  data structures and the matchmaking hang-out arithmetic. **db-core never imports FRET or
  libp2p.** This layer is reusable by any DHT-backed distributed network db-core targets —
  which is the bar the human set for putting it in core.
- **`packages/db-p2p/src/cohort-topic` — transport binding.** Concrete implementations of
  the ports below over FRET + libp2p: the four `/optimystic/cohort-topic/1.0.0/*` protocol
  handlers, `RouteAndMaybeAct` routing, FRET cohort assembly, threshold-sig collection,
  cohort gossip transport, `MembershipCertV1` fetch, and the size-estimator feed. Same split
  for reactivity (`packages/db-p2p/src/reactivity` = origination hook + delivery;
  `packages/db-core/src/reactivity` = pure replay/dedupe/resume) and matchmaking
  (`packages/db-p2p/src/matchmaking` = query/registration RPC; `packages/db-core/src/matchmaking`
  = hang-out decision + record types).

If a module touches FRET, libp2p, multiaddrs, or sockets, it is db-p2p. Otherwise it is
db-core, behind a port.

## Port interfaces (db-core defines; db-p2p implements)

```ts
// packages/db-core/src/cohort-topic/ports.ts — INTERFACES ONLY. No FRET/libp2p import here.

/** Ring coordinate (SHA-256 truncated to ring width B). db-core owns the type; db-p2p maps to FRET's. */
export type RingCoord = Uint8Array;
export interface PeerRef { readonly id: Uint8Array; }

/** Wraps FRET RouteAndMaybeAct so the walk/registration logic stays transport-agnostic. */
export interface ITopicRouter {
	routeAndAct(key: RingCoord, activity: Uint8Array, opts: { wantK: number; minSigs: number }): Promise<Uint8Array>;
	/** Direct dial to a cached primary; falls back to routeAndAct on failure (caller decides). */
	dialMember(member: PeerRef, activity: Uint8Array): Promise<Uint8Array>;
}

/** Intra-cohort gossip transport (FRET cohort gossip underneath). */
export interface ICohortGossipTransport {
	broadcast(coord: RingCoord, msg: Uint8Array): void;
	onMessage(handler: (from: PeerRef, msg: Uint8Array) => void): () => void;
}

/** Authoritative cohort membership snapshots (FRET MembershipCertV1 / stabilization underneath). */
export interface IMembershipSource {
	current(coord: RingCoord): Promise<Uint8Array | undefined>;   // encoded MembershipCertV1
	fetch(coord: RingCoord): Promise<Uint8Array | undefined>;     // force one refresh (stale-cache retry)
}

/** Network-size estimate feeding d_max. */
export interface ISizeEstimator {
	estimate(): { nEst: number; confidence: number };
}

/** Hash + ring math. db-core supplies its OWN SHA-256 (it already hashes logs/blocks); ring width is config. */
export interface IRingHash {
	H(bytes: Uint8Array): RingCoord;   // SHA-256 truncated to ring width B
	readonly ringBits: number;
}
```

db-core substrate modules take these ports by injection (constructor / factory args). db-p2p
constructs the concrete FRET/libp2p-backed implementations and composes them with the db-core
logic in `CohortTopicService`'s transport host (see `cohort-topic-core-module-fret-integration`).

## Why coord_d uses db-core's own hash, not FRET's

`coord_d(P, topicId) = H(d ‖ prefix(P, d·log₂F) ‖ topicId)` needs only SHA-256 truncated to
the ring width. db-core already owns SHA-256 (logs, block IDs), so the addressing module uses
`IRingHash` backed by db-core hashing — **not** an import of `Fret/.../ring/hash.ts`. db-p2p,
when it maps `RingCoord` onto FRET's coordinate type, is responsible for ensuring the byte
representation matches FRET's ring width so routing keys line up. This keeps db-core
FRET-free while guaranteeing on-the-wire coord compatibility.

## Corrections this ticket imposes on sibling tickets

These were authored with FRET imports inside db-core; they must route through the ports above:

- `cohort-topic-tier-addressing-dmax` — `H` comes from `IRingHash` (db-core SHA-256), not
  `Fret/ring/hash.ts`; `d_max` consumes `ISizeEstimator`, not `Fret/estimate` directly.
- `cohort-topic-gossip-membership-certs` — the gossip **merge** + epoch-drift logic and the
  cert **verification** logic are db-core, consuming `ICohortGossipTransport` /
  `IMembershipSource`; the FRET cohort-gossip adapter and cert publisher are db-p2p.
- `cohort-topic-walk-promotion-coldstart` — `WalkEngine` decision logic is db-core over
  `ITopicRouter`; the `registerMaybeAct`/`sendMaybeAct` binding is db-p2p.

## Constraints

ES modules, no inline `import()`, no `any`, tabs, cross-platform. db-core has **zero** FRET /
libp2p dependency after this lands (enforce with a no-FRET-import check in db-core's build/test
where practical). Treat FRET read-only.
