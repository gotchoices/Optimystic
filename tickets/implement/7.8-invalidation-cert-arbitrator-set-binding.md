# Invalidation certificate: bind votes to the legitimately-selected arbitrator set

This is **part 2 of 2** of the forgeable-invalidation-certificate fix
(`tickets/fix/invalidation-certificate-arbitrator-binding.md`), and the critical binding (#1). Part 1
(`invalidation-cert-target-binding-and-dedup`) closed replay (#2) and vote-duplication (#3) and
established the versioned vote payload; this ticket builds on that wire format to confirm that every
counted vote came from a peer that was a **legitimately-selected arbitrator** for the dispute.

## The hole

`verifyInvalidationCertificate` confirms each vote's signature matches the Ed25519 key embedded in its
claimed `arbitratorPeerId`, but **nothing ties that peer to the legitimate arbitrator set**. Arbitrators
are chosen deterministically by `selectArbitrators` (`arbitrator-selection.ts`) â€” the next K peers
beyond the original cluster, by XOR distance to the disputed block id. An attacker who can mint Ed25519
keypairs can generate N fresh keys, sign a 2/3 super-majority of target-bound `agree-with-challenger`
votes (target binding from part 1 does **not** stop this â€” the attacker simply binds to the real
target), and produce a passing certificate with zero genuine arbitrator participation.

Part 1's dedup ensures each *peer-id* counts once; this ticket ensures each counted peer-id is one of
*the actual arbitrators*, and that the set itself is the one the deterministic selection produces.

## Why this is the hard one â€” the anchoring problem

To confirm "this voter was a legitimately-selected arbitrator," a verifying member must know the
eligible set. There are two ways to get it, each with a gap:

1. **Recompute `selectArbitrators` at apply time.** Cleanest trust model, fully independent of the
   originator â€” but it needs the historical peer/DHT topology (`KnownPeer[]`) the challenger saw. A
   member that joined after the dispute, or whose DHT view has churned, cannot reconstruct it, so a
   pure recompute would falsely reject legitimate certificates.

2. **Carry the eligible set in the proof.** Only helps if the carried set is itself anchored to a trust
   root the member already holds. A bare challenger signature over the set binds the *originator* (a
   third-party relay can't swap in its own sybils without the challenger's key) but does **not** stop a
   *fully-malicious challenger*, who legitimately holds its own key and can sign a sybil set it minted.

The genuinely-closing anchor for peer identity is the cohort-topic / FRET membership-cert trust chain
â€” the **related but not identical** surface tracked by
`tickets/plan/cohort-topic-membership-cert-trust-anchoring.md` (which anchors `MembershipCertV1` to a
chain-to-genesis attestation). That ticket anchors *cohort* membership; arbitrators are DHT-selected
peers *beyond* the cohort, so it does not directly cover this â€” but it provides the same kind of
network-rooted peer-identity attestation this binding ultimately needs. Design this binding to **layer
cleanly on top of** that anchoring when it lands, and to degrade safely until then.

## Design

Build on part 1's versioned payload (bump to `'v3'`). The certificate now carries and binds the
eligible arbitrator set, and the verifier confirms membership + legitimacy in defense-in-depth layers,
strongest-available-wins:

### 1. Carry + challenger-bind the eligible set (always)

- Carry the selected arbitrator peer-ids (the K the challenger computed via `selectArbitrators`) on the
  proof: `DisputeResolutionProof.arbitratorSet: ReadonlyArray<string>` (peer-id strings).
- Fold the set into the binding: each vote's v3 signed payload includes a digest of the sorted
  `arbitratorSet` (e.g. extend `targetHash` to also commit to `sortedArbitratorSet`), and the
  **challenger signs** the `(disputeId, target, arbitratorSet)` tuple. The challenger's key is anchored
  to something the member already trusts: it is a member of `originalRecord.peers`, and the original
  transaction's commit certificate (which this member honored when it committed the now-disputed
  transaction) attests that membership.
- Verifier requires: every counted vote's `arbitratorPeerId` âˆˆ `arbitratorSet`; the challenger
  signature over the set is valid; combined with part 1's dedup, each set member counts at most once.

This closes the **third-party-relay / originator** sybil vector and binds the originator. It does
**not** by itself close a fully-malicious challenger (who signs its own sybil set) â€” that needs layer 2
or 3.

### 2. Recompute-when-feasible (best effort)

When the member can assemble a sufficient peer/DHT view (it has `peerNetwork` / FRET routing on the
cluster side â€” `ClusterMember` holds `peerNetwork`; the dispute side has the injected
`selectArbitrators`), recompute `selectArbitrators(blockId, excludePeers=originalClusterPeers, K)` and
require the carried `arbitratorSet` to match the recomputed eligible set (or be a subset of a tolerance
window around it, to absorb minor churn). This closes the **malicious-challenger** vector whenever the
member can reconstruct the topology. Make the recompute an injected capability so the pure
`verifyInvalidationCertificate` stays usable without a network (its callers pass the capability when
they have one).

### 3. Anchor-when-available (full closure)

When recompute is not feasible, fall to the membership/trust-anchor chain
(`cohort-topic-membership-cert-trust-anchoring`): accept the carried set only if it is attested by a
trusted membership source for the relevant coord/topic. Until that anchoring lands, this layer is a
documented **residual limitation**, not a silent accept.

### Member behavior when none of the above resolves

Define and document it explicitly (the parent fix ticket asks for this): a member that can neither
recompute the historical topology nor anchor the carried set via the trust chain accepts on the basis
of layer 1 alone (challenger-bound set + membership + dedup) and **logs** that it applied an
invalidation it could not fully anchor â€” never a silent accept-by-default, and never a false reject
that would break liveness for late-joiners. This is the interim posture; once the trust-anchor chain
lands it is upgraded to a hard gate.

## Test coverage (required)

- **Sybil-key votes rejected (#1)** â€” N fresh keypairs (not in `arbitratorSet`, or an `arbitratorSet`
  that fails recompute) produce a certificate that returns `false`. This is the headline fix: the
  current green `accepts a challenger-wins resolutionâ€¦`-shaped forgery must now be rejected.
- **Vote from a non-set peer dropped** â€” a signature-valid, target-bound vote whose `arbitratorPeerId`
  is not in `arbitratorSet` is not counted.
- **Tampered set rejected** â€” mutating `arbitratorSet` (adding sybils) breaks the challenger signature
  and/or the recompute match â†’ reject.
- **Recompute path** â€” with an injected topology that reproduces the historical selection, the genuine
  set verifies and a forged set fails.
- **Degradation path** â€” with no recompute capability and no anchor, a layer-1-valid certificate is
  accepted *and logged* as not-fully-anchored (assert the documented interim behavior).
- **Legit path still accepted** â€” genuine, in-set, target-bound 2/3 super-majority verifies.
- Cluster-path coverage in `test/cluster-invalidation.spec.ts`.
