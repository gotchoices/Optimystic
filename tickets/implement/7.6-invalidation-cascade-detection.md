# Read-dependency cascade detection + re-evaluation

Builds on `7.5-invalidation-log-entry-and-reversal` (the durable reversal primitive). That ticket
reverses one transaction in one collection. This ticket finds the **transitive read-dependents** of an
invalidated transaction across collections and re-evaluates each — invalidating only those that no
longer hold, retaining the rest. It also makes the root invalidation **multi-collection atomic**.

## Background (resolved design)

### Dependency relation

A committed transaction `T2` is a **read-dependent** of an invalidated `T_inv` iff `T2`'s read set
contains a `(blockId, revision)` that `T_inv` produced — i.e. `blockId ∈ writes(T_inv)` and
`revision ∈ [rev(T_inv), nextWriterRev(blockId))`. Reads are block-granular `(blockId, revision)` pairs
(`ReadDependency`, `packages/db-core/src/transaction/transaction.ts`; captured during execution by
`TransactorSource.getReadDependencies`, `packages/db-core/src/transactor/transactor-source.ts`) and are
exactly the signal `TransactionValidator` already uses for stale-read detection
(`docs/correctness.md` Theorem 5). The cascade reuses that machinery in reverse: instead of "is this
read stale vs current tip", ask "did this read observe a now-invalidated revision".

### The read-set persistence gap (must close first)

Cascade detection needs the **read set of already-committed** later transactions. Today the collection
log stores only `ActionEntry` (`actionId`, `actions`, `blockIds`, `collectionIds`) — **not** the
`Transaction.reads`. `TransactionSession.commit` computes `reads` and ships them in the PEND
`Transaction`, but the committed log entry drops them. **First task:** persist each committed
transaction's read set so dependents are discoverable — extend `ActionEntry` with
`reads?: ReadDependency[]` (preferred: keeps it inline with the entry the cluster already replicates),
or store the full `Transaction` keyed by `actionId`. Audit `Log.addActions` and the cluster
commit/checkpoint path; thread `reads` through. This is a wire/format change — version it and cover it
with a migration/back-compat note (older entries with no `reads` are treated as "unknown deps" →
conservatively re-evaluated, see horizon below).

### Re-evaluation, not blanket reversal (folds in the Reapplication question)

The cascade is a **re-evaluation pass**, which is also the answer to "can the cascade be partially
reapplied?":

1. Walk each affected collection's log forward from `T_inv`. Collect candidates whose read set
   intersects an invalidated `(blockId, rev)`.
2. For each candidate `T2`, in log order, re-execute/re-validate against **post-invalidation** state:
   - **Still valid** (reads consistent against reverted state, operations reproduce) → **retain**.
     `T2` appeared in the dependency chain but did not actually depend on the invalid value; its
     revision stands untouched. This is "partial reapplication of non-dependent transactions."
   - **No longer valid** → **invalidate `T2`**: append its own InvalidationEntry (`cascadeRoot = T_inv`,
     from the core ticket) via the same consensus path, then **recurse** — `T2`'s reverted writes feed
     the next round of candidates.
3. Continue until no fresh dependents remain (fixpoint) or the horizon is hit.

Truly-invalidated dependents are **not** auto-reapplied with rewritten operations (that would change a
transaction's content id / identity). The client resubmits them through the normal optimistic loop
(handled in `7.5-invalidation-client-notification`). Optional auto-resubmission is parked in
`backlog/speculative/7.5-invalidation-auto-resubmit`.

### Cascade scope / bound

- **Natural bound:** a dispute can only be raised within the dispute TTL (≈ 2× transaction TTL,
  `DisputeChallenge.expiration`), so `T_inv` is recent — this bounds the log suffix that must be walked.
- **Re-evaluation prune:** most candidates retain, so the live cascade is far smaller than the suffix.
- **Hard horizon:** configurable `maxCascadeDepth` and `maxCascadeTransactions`
  (add to `ClusterConsensusConfig` / dispute config). On overflow, **stop cascading** and flag the
  affected collection(s) for **operator-escalated full re-sync** (emit a health signal; do not silently
  truncate — `log()` what was left un-cascaded). Unbounded automatic reversal is never attempted.

### Multi-collection / partial invalidation

`T_inv`'s root InvalidationEntry must be appended **atomically to every collection `T_inv` wrote**
(its `collectionIds`), using the same GATHER/PEND/COMMIT 2PC as the original commit
(`docs/correctness.md` Theorem 3). Non-dependent transactions in those collections keep their
revisions; only read-dependents are re-evaluated. Dependents may live in **other** collections
(cross-collection read deps), so the forward walk and candidate set span every collection reachable via
read-set edges, not just the ones `T_inv` wrote.

## Edge cases & interactions

- **Diamond / shared dependents.** `T3` may depend on both `T_inv` and another invalidated `T2`. Process
  in log (revision) order and dedup by `actionId` so `T3` is evaluated once, after all its invalidated
  ancestors are reverted — never twice, never before an ancestor.
- **Cycle-freedom.** Dependencies follow strictly increasing revision (a read observes an earlier
  write), so the dependency graph is a DAG — assert the walk only ever moves forward; a back-edge
  indicates corruption and must error, not loop.
- **Candidate with missing reads (legacy entry).** A pre-migration entry without a stored read set is
  treated as an unknown dependency: conservatively re-evaluate it (re-execute and compare), and if it
  cannot be re-evaluated, count it against the horizon and escalate rather than guessing.
- **Re-evaluation needs historical base state.** Re-validating `T2` requires the block content as of
  `T2`'s read revisions, recomputed against the reverted history — reuse `materializeBlock` and the
  core ticket's as-if-absent computation. A member missing those revisions fetches via restore/sync.
- **Concurrent new commits during cascade.** New transactions may commit on these collections while the
  cascade runs. Because each invalidation takes a serialized revision slot, a new commit is ordered
  either before (already a candidate) or after (reads post-revert state) each child invalidation — but
  add an explicit test for a commit landing mid-cascade and assert no dependent is skipped.
- **Partial-failure mid-cascade.** If a child invalidation's consensus fails, the cascade must be
  resumable (persisted progress) and must not leave the collection with some-but-not-all dependents
  reverted in an inconsistent way. Define: reverting an ancestor without its dependents is **safe but
  stale** (dependents read invalid data but are detectable as `committed-disputed`/pending re-eval), so
  the cascade is restartable and converges; it is never left silently half-done without a health flag.
- **Horizon overflow mid-chain.** When the horizon trips partway, the already-applied child
  invalidations stand (durable), and the remainder is escalated for full re-sync — record exactly where
  it stopped.
- **Cross-collection atomicity vs cascade.** The **root** `T_inv` reversal is atomic across its
  collections; **child** invalidations are independent per-transaction events (each its own consensus),
  not part of the root's atomic set — document this distinction so a reviewer doesn't expect one giant
  atomic multi-collection operation.
- **Self-dependency / structural blocks.** Read sets include structural blocks (B-tree nodes, log chain
  nodes per Theorem 5), so the candidate set can be conservative (false dependents). That is fine — they
  re-evaluate and retain. Assert a structural-only overlap does not force invalidation.

## Key tests

- Linear chain `T_inv → T2 → T3` where each genuinely reads the prior's write: all three reverted in
  order; `priorHash` chains intact.
- Mixed chain where `T2` reads a block `T_inv` wrote but a field `T_inv` did **not** change: `T2`
  retained (re-evaluation prune), `T3`-via-`T2` unaffected.
- Cross-collection dependent: `T_inv` in collection A, `T2` in collection B reads A's invalidated
  revision → `T2` invalidated.
- Diamond: `T3` depends on two invalidated ancestors → evaluated once, after both.
- Horizon: a cascade exceeding `maxCascadeTransactions` stops, applies what it did, and raises the
  re-sync health flag; assert nothing past the stop point is silently dropped.
- Multi-collection atomic root: `T_inv` spanning A+B is reverted in both or neither under induced
  partial failure.
- Read-set persistence round-trip: committed transaction's `reads` survive log write/read and sync.
