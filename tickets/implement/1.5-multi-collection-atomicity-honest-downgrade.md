## Decision (settled in plan)

Optimystic commits a multi-collection transaction **collection-by-collection with no cross-collection undo**, so a mid-loop failure can leave the transaction half-applied. Two ways forward were on the table:

- **A — real cross-collection two-phase commit:** a durable coordinator decision record so a crashed/failed coordinator drives every collection to one commit-or-abort outcome.
- **B — honest downgrade to atomic-intent / eventual-visibility:** the transaction atomically records intent; collections become visible eventually; partial landings are reported and reconciled.

**This ticket implements B.** Why B, not A — the reasoning the implementer should preserve, not relitigate:

1. **The concrete trigger is a *permanent* stale loss, not a crash.** The partial commit fires when a racing transaction advances one collection's log tail between that collection's PEND and COMMIT (see `commitPhase` / `commitCollection`, `coordinator.ts:889-911` — a returned `{success:false}` is a permanent stale loss that is deliberately *not* retried). A 2PC journal solves *crash* recovery, but it cannot help here: the lost collection's commit slot is gone and the identical request can never win, while the collections that already committed have **no cross-collection undo**. So bolting a journal on would still be non-atomic in exactly the scenario the design names.
2. **Real atomicity requires a much deeper change** than a journal: either PEND must durably *reserve* the log-tail slot and hold it through COMMIT so no racing transaction can steal it (a change to promise/consensus semantics, with liveness cost — a crashed coordinator holds reservations until expiration), or a cross-collection *compensating reversal* must un-commit the winner (extending the single-collection `InvalidationEntry` machinery — today single-collection only, gated on a dispute certificate). Both presume an agreed supercluster (depends on `design-cluster-membership-agreement`).
3. **The architecture already points at B.** `crdt-sync.md` states the intended direction plainly: "Atomicity-of-intent is structural; atomicity-of-visibility is eventual" (§97, §213). Option A becomes the future *strong-mode* per-collection profile (crdt-sync Stage 5), not the near-term default.
4. **B's runtime surface is already built.** `coordinator.commit()` throws `CoordinatorPartialCommitError(committedCollections, failedCollections, reason)`; `coordinator.execute()` surfaces `committedCollections`/`failedCollections` on `ExecutionResult`; `txn-bridge.ts` special-cases it. So B is mostly making the *promise* honest and *proving* the surface with tests — not new runtime machinery.

Option A is not discarded — it is parked as backlog `feat-cross-collection-atomic-commit` (the strong-mode upgrade). This ticket must **not** re-decide the model; it makes the docs truthful and hardens the reporting contract.

## What "atomic-intent / eventual-visibility" means here (state it in the doc)

Define these plainly on first use — the reader is deciding, not guessing:

- **Atomicity of intent:** a multi-collection transaction has one content-addressed identity; every participating collection's PEND carries that same identity. The *intent to commit the whole set* is recorded atomically at the point all collections have pended.
- **Eventual visibility (not atomic visibility):** the COMMIT phase lands each collection independently. Normally all land. When one permanently loses (stale loss) or the coordinator dies mid-loop, some collections are durably committed and others are not — a **partial landing**. The system does not silently claim success: it names the durable set (`committedCollections`) and the not-landed set (`failedCollections`), and the application (or a future reconciler) resolves the split. This is the same shape `crdt-sync.md` describes.
- **What is NOT promised:** that a reader observing collection A's write will also observe collection B's write of the same transaction at that instant. Cross-collection visibility is eventual and, on a partial landing, requires reconciliation.

## Coordinator-crash-mid-loop under this model

Spell this out in the doc so it's not left implied: the db-core `TransactionCoordinator` is an in-memory orchestrator with **no cross-collection journal**. If it dies mid-commit-loop, the collections that committed stay committed; the pended-but-uncommitted collections are released by each cluster's independent expiration cleanup (Theorem 7). The observable outcome is identical to a stale-loss partial landing — intent was recorded, visibility is partial, reconciliation applies. The per-*cluster* persistence that *does* exist (`ITransactionStateStore` / `PersistedCoordinatorState` in db-p2p, resuming only the `broadcasting` phase — `cluster-coordinator.ts:776-810`) is a per-block-cluster durability aid, **not** a cross-collection coordinator decision journal; the rewritten Theorem 3 must not conflate the two (today's §1.2 "restore persisted 2PC state" and Theorem 3's "persisted 2PC state journal records the transaction phase" read as if a cross-collection journal exists — it does not).

## Edge cases & interactions

- **Racing log-tail advance between PEND and commit (the trigger).** The end-to-end test must simulate a two-collection commit where collection B's commit returns a permanent stale `{success:false}` while A succeeds, and assert `commit()` throws `CoordinatorPartialCommitError` with `committedCollections=[A]`, `failedCollections=[B]`. Today `coordinator.spec.ts:252-297` tests `commitPhase` *in isolation*; the gap is the **end-to-end `commit()` and `execute()`** paths (the local-treatment split at `coordinator.ts:205-228` and `:478-493`).
- **Coordinator crash mid-commit-loop.** No new recovery code under this model; the doc must state the outcome (partial landing, reconcile) rather than promise recovery-to-atomic. No test can portably kill the process mid-loop — assert instead that the committed subset got the success-path local treatment (`recordCommitted` + `tracker.reset` + `clearPendingActions`) and the failed subset was restored, so local memory matches durable state regardless of when the loop stopped.
- **Caller-visible partial state — define the contract.** A caller receiving `CoordinatorPartialCommitError` (session path) or a failed `ExecutionResult` with non-empty `committedCollections` (execute path) MUST NOT blindly retry the whole transaction and MUST NOT treat it as a clean abort; it must reconcile the named committed set against the failed set. Document this in the error's doc comment and in transactions.md. `txn-bridge.ts:346` already tears down without `rollbackTransaction()` — keep that behavior; add/confirm a test asserting it does not falsely report a clean rollback.
- **Idempotent replay / no double-apply.** On the partial path the committed collections have their pending actions dropped (`clearPendingActions`, `coordinator.ts:213`) and the failed collections are restored for a clean re-append. Add a test: after a partial-commit throw, re-driving `commit()` for the same coordinator must re-attempt only the failed collections (no duplicate log entry on the already-committed one). Add a `NOTE:` tripwire at the partial-path site stating that correctness here depends on `clearPendingActions` running for every committed collection before any retry.
- **Interaction with cluster membership agreement.** The real-2PC upgrade (backlog `feat-cross-collection-atomic-commit`) presumes an agreed supercluster and thus depends on `design-cluster-membership-agreement`; note that dependency in the backlog ticket, not here. The honest-downgrade path has **no** such dependency — it ships independently.
- **Empty-committed clean failure stays clean.** Confirm the existing behavior is preserved and tested: when PEND fails or the whole commit fails with nothing durable, every tracker is restored and a plain `Error` (not `CoordinatorPartialCommitError`) is thrown, leaving trackers pristine for retry (`coordinator.ts:231-237`).

## Key tests & expected outputs

- **`commit()` end-to-end partial landing** (new, `coordinator.spec.ts`): two collections, B's commit returns permanent stale `{success:false}`. Expect `commit()` to reject with `CoordinatorPartialCommitError`; `err.committedCollections` deep-equals `[A]`; `err.failedCollections` deep-equals `[B]`; collection A's tracker is reset with pending cleared (durable state), collection B's tracker restored to its pre-append snapshot.
- **`execute()` end-to-end partial landing** (new): same setup via the `execute()` path. Expect `result.success === false`, `result.committedCollections === [A]`, `result.failedCollections === [B]`; A `recordCommitted` + `tracker.reset` applied.
- **Clean failure stays clean** (new or extend): PEND fails for one collection. Expect a plain `Error`, no `CoordinatorPartialCommitError`, every tracker pristine.
- **No-double-apply on retry** (new): after a partial-commit throw, a second `commit()` re-appends/pends only the failed collection; the already-committed collection produces no new log entry.
- **Doc consistency** (manual/review, no test): Theorem 3, §1.2, §6.3, composition guarantee #4, and the appendix dependency graph all state atomic-intent/eventual-visibility consistently; no remaining sentence claims unconditional multi-collection atomicity.
