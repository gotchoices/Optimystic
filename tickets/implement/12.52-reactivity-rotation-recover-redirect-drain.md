# Reactivity rotation — recover drain-redirect serve + subscriber honor (db-p2p)

Drives `TailDrainGate` (db-core, 11.5) from the live **recover** serve handler, so an outgoing tail cohort
that has rotated bounces requests to the new tree for `T_drain`, and wires the subscriber to honor that
redirect. Builds on `reactivity-rotation-resume-handoff-and-redirect-codec` (the `kind:"rotated"` recover
reply + `inheritedCheckpoint` resume serving).

## Background — why recover, and why the old cohort must "know it rotated"

A subscriber reaches a serving cohort over exactly two reactivity surfaces: **notify** (down, notifications)
and **recover** (up, backfill/resume). New subscribes ride generic cohort-topic `service.register`, which
reactivity cannot intercept. So the redirect that moves a stuck/waking subscriber off a rotated tail rides
**recover**:

- A waking subscriber that reaches the **new** cohort with a stale `latestKnownTailId` already gets
  `tail_rotated` from `serveResume` (its `currentTailId` differs). That path works today.
- A subscriber that reaches the **old** (rotated, still-draining) cohort gets served `backfill` /
  `checkpoint_window` from the data the old cohort still holds — and is **never told to move**, because the old
  cohort's served `PushState.tailIdAtJoin` is still the old tail, so `serveResume` classifies it as a normal
  lag, not `tail_rotated`. It catches up to `rotationRevision` and then goes silent forever (the old tail no
  longer originates).

The fix: the old cohort must **record that it rotated** (start a `TailDrainGate`), and its recover serve must
return the `kind:"rotated"` redirect while draining. After `T_drain` the old cohort releases its `PushState`
(forwarder demotes naturally) → no served state → no reply → the subscriber re-walks/chain-reads.

## What to build

### A. Rotation state on the forwarder host

`ReactivityForwarderHost` (`forwarder-host.ts`) owns the live per-topic `PushState`; it is the natural holder
of the drain gate.

- Add `markRotated(oldTopicId: Uint8Array, redirect: { newTailId: string; effectiveAtRevision: number }, now:
  number): void`. It constructs a `TailDrainGate({ rotatedAt: now, newTailId, effectiveAtRevision })` (db-core
  derives `newTopicId` internally via `reactivityTopicId`) and stores it keyed by the **old** topicId. Idempotent
  — a second `markRotated` for the same successor is a no-op; a `markRotated` to a *later* successor replaces.
- Add `rotationRedirectFor(oldTopicId: Uint8Array, now: number): RotationRedirectV1 | undefined`. Returns the
  gate's `rotationRedirect` while `gate.isDraining(now)`; once the drain window closes, evict the gate entry
  **and** the served `PushState` for that topic (the old tail is done — this also reclaims the
  `served`/`ingestTails` maps the 12.31 review flagged as un-evicted on rotation) and return `undefined`.
- The trigger for `markRotated` is the origination side observing the collection's `event.tailId` change; that
  wiring is `reactivity-rotation-host-wiring-e2e`. This ticket only provides + unit-tests the seam (call
  `markRotated` directly in tests).

### B. Recover serve returns the redirect while draining

`createRecoverRequestHandler` / `RecoverServeDeps` (`recover-transport.ts`).

- `RecoverServeDeps` gains `rotationFor?: (req: { topicId?: Uint8Array; collectionId: string }, now: number)
  => RotationRedirectV1 | undefined` (the node wiring binds it to `forwarderHost.rotationRedirectFor`,
  resolving the topic from the request — `reactivityTopicId(latestKnownTailId)` for resume; for backfill,
  the collection's current served topic).
- **Resume**: after verify/admit, if `rotationFor({ topicId: reactivityTopicId(latestKnownTailId), collectionId
  }, now)` returns a redirect, reply `{ v:1, kind:"rotated", rotated }`. Otherwise serve as today, **and pass
  `ps.inheritedCheckpoint` into `serveResume`** (the new `ResumeServingDeps.inheritedCheckpoint` from the
  prereq) so a cross-rotation resume to a *new* tail is answered from the inherited checkpoint.
- **Backfill**: a backfill underflowing the current ring on a rotated cohort returns the redirect too (a
  best-effort secondary path; the primary mechanism for active subscribers is notify-driven detection). If
  `rotationFor` returns a redirect, reply `kind:"rotated"`; else serve `serveBackfill` as today.
- No reply (stream abort) on any verify/replay/resolve failure, unchanged.

### C. Outbound transport surfaces the redirect

`Libp2pReactivityRecoverTransport` (`recover-transport.ts`).

- `backfillTransport` / `resumeTransport`: when `exchange(...)` returns `reply.kind === "rotated"`, throw a
  typed `RotationRedirectError` (carry the `RotationRedirectV1`) rather than the current "reply missing its
  body" error. A `kind:"rotated"` reply is terminal for that exchange (do **not** fall through to the next
  cohort target — the cohort answered authoritatively).
- Export `RotationRedirectError`.

### D. Subscriber honors the redirect

`ReactivitySubscriptionManager` (`subscription-manager.ts`).

- In `escalateBackfill` and `resume`, catch `RotationRedirectError`: invalidate the sticky cohort-hint cache
  (`cohortHintCache.invalidate(collectionIdB64)`) and surface the rotation through the **same** seam a delivered
  pre-announce uses — build `planReRegistration({ hint: { newTailId: redirect.newTailId }, lastRevision:
  this.subscriber.lastRevision, now: this.clock(), jitter: this.rejoinJitter })` and fire
  `options.onRotation({ newTailId, preAnnounced: false, plan })`, guarded once-per-successor by the existing
  `rotationHandledFor`. This unifies the redirect path with the notification-driven path: both end in a
  `RotationNotice` the host's re-registration scheduler (`reactivity-rotation-rereg-scheduler`) consumes.
- A redirect must never throw out of `resume()`/the gap seam to the commit/delivery path (the gap seam is
  already detached + catch-guarded; keep that property).

## Edge cases & interactions

- **Idempotent / chained rotation.** `markRotated(old→A)` then `markRotated(old→B)` (a second rotation while
  the first still drains): the gate advances to the later successor (higher `effectiveAtRevision`); the
  subscriber's `rotationHandledFor` self-corrects across OLD→A→B (same property the existing `detectRotation`
  path relies on).
- **Drain window boundary.** A request at exactly `rotatedAt + T_drain` is `drained` (`TailDrainGate.isDraining`
  is strict `<`); the gate + `PushState` are evicted and the next request gets no reply → re-walk. Test both
  sides of the boundary against the injected clock.
- **Redirect is terminal in the cohort-walk.** A `kind:"rotated"` reply must stop the `selectTargets`
  fallthrough — the dialed member spoke for the cohort. Contrast a *dial failure*, which still falls through.
- **Self-dial / co-located serve.** The rotated old cohort and the subscriber may be the same node; the
  redirect must still route (the in-process path) — `markRotated` + `rotationRedirectFor` are node-local, so a
  co-located subscriber's recover (if any) sees it.
- **Resume reaching the new tail vs. old tail.** New tail → existing `serveResume` `tail_rotated` /
  inherited-checkpoint branch. Old draining tail → `kind:"rotated"` redirect. Both converge on the manager's
  re-register. A resume reaching the old tail **after** drain → no served state → no reply → `onChainRead`.
- **Backfill keyed only by collectionId.** `pushStateForCollection` returns the *current* (new) tail's
  PushState once both coexist (highest `lastRevision`, already implemented in 12.31). So a backfill arriving at
  a node that serves both tails resolves the new tail; the redirect for backfill is therefore only emitted when
  the node serves **only** the old (draining) tail. Document this asymmetry; do not over-engineer the
  marginal backfill case.
- **No `onRotation` configured.** The redirect still invalidates the sticky cache (so a later resume re-walks)
  and is a no-op otherwise — same contract as the pre-announce path.
- **Forward-compat.** A peer that predates the `kind:"rotated"` codec fails the decode closed → the transport
  treats it as a dial failure → falls through to the next cohort member / eventually chain-reads. Safe.
- **Memory.** `markRotated`/eviction must not leak gate entries: a gate whose drain elapsed is dropped on the
  next `rotationRedirectFor`, and the lazy `served`-map eviction closes the 12.31-flagged leak.
