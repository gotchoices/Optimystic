# Reactivity recover RPC — wire framing + canonical request signing (db-core)

## Why this ticket exists

This is the **db-core foundation** for the reactivity backfill/resume recovery transport
(`12.4-reactivity-backfill-resume-rpc-transport`, the originating plan ticket). That plan ticket split into
two prereq-chained implement tickets:

1. **this ticket** — the pure db-core wire framing + canonical signing-payload helpers (unit-testable, no
   libp2p);
2. `reactivity-recover-rpc-transport` (db-p2p) — the live libp2p protocol handler + dial transport + node
   wiring that consumes what this ticket produces.

11.5 (`reactivity-backfill-resume-checkpoints`) landed the recovery **logic** (`serveBackfill`,
`serveResume`/`classifyResume`, `createBackfillRequester`, `applyResumeReply`) wired through **injected**
`sign(unsigned) => string` hooks with dummy signers in unit tests. There is **no canonical signing-payload
helper** defining the exact bytes signed (cf. the cohort-topic `registerSigningPayload` /
`renewSigningPayload` / `cohortGossipSigningPayload` in
`packages/db-core/src/cohort-topic/wire/payloads.ts`), and `BackfillV1` carries no freshness field, so the
serving side cannot run replay protection. A real transport needs both, or two honest peers will not agree
on what was signed.

## Design decisions (resolved — do not re-litigate)

These were settled in the plan pass; the rationale is recorded so the implementer and reviewer don't
reopen them.

### 1. Keep the in-message `signature`; add canonical signing-payload helpers

The existing `BackfillV1.signature` / `ResumeV1.signature` fields stay (they match the authoritative
wire-format block in `docs/reactivity.md` §Wire formats and the `BackfillTransport`/`ResumeTransport` seam
shape, which this ticket must **not** redesign). What was missing is the canonical byte image those
signatures cover. Add, mirroring `cohort-topic/wire/payloads.ts`:

```ts
// canonical signed byte image = utf8(JSON.stringify([ "<tag>", ...ordered non-signature fields ])).
export type BackfillSignable = Omit<BackfillV1, "signature">;
export function backfillSigningPayload(body: BackfillSignable): Uint8Array;

export type ResumeSignable = Omit<ResumeV1, "signature">;
export function resumeSigningPayload(body: ResumeSignable): Uint8Array;
```

Determinism comes from encoding an **explicitly-ordered JSON array** (array order is stable; object key
order is not) as UTF-8, exactly like the cohort-topic helpers. Lead each array with a literal type tag
(`"BackfillV1"` / `"ResumeV1"`) so a backfill image can never collide with a resume image. There are no
optional non-signature fields on these types, so no placeholder-normalization is needed (unlike
`registerSigningPayload`'s `bootstrap ?? false`).

### 2. Add `timestamp` to `BackfillV1` (freshness for replay protection)

`ResumeV1` already carries `timestamp`; `BackfillV1` does not. Replay protection at the serving handler
(the db-p2p ticket) needs a freshness value bound into the signature, so add:

```ts
export interface BackfillV1 {
  v: 1;
  collectionId: string;   // base64url
  fromRevision: number;
  toRevision: number;
  timestamp: number;      // NEW — unix ms; bound into backfillSigningPayload so freshness is authenticated
  signature: string;      // base64url, peer-key sig over backfillSigningPayload(unsigned)
}
```

The signing payload **must** include `timestamp` so a captured request cannot be replayed with a forged
fresh timestamp (the forged timestamp would invalidate the signature, which the attacker cannot re-sign).

**Why no explicit `correlationId` nonce (the rejected alternative).** The cohort-topic `RegisterV1`/`RenewV1`
carry a random `correlationId` keyed by the `CorrelationReplayGuard`. We deliberately do **not** add one to
the recover requests, because: (a) the authoritative `docs/reactivity.md` §Wire formats codec blocks for
`ResumeV1`/`BackfillV1` do not list `correlationId` (the §Authentication "include correlationId and
timestamp" line describes the Subscribe/renew RPCs, which ride a real `RegisterV1` envelope; the recover
RPCs do not); (b) the request **signature** is itself a unique, deterministic, authenticated token over the
full request, so the db-p2p ticket keys the replay guard on the signature bytes + `timestamp` window —
secure without an extra field. Adding `correlationId` would be a larger wire-type redesign for no security
gain. Keep the change to the single `timestamp` field.

### 3. Discriminated `RecoverRequestV1` / `RecoverReplyV1` frame (one protocol, explicit kind)

The db-p2p ticket runs one request-reply protocol (`/optimystic/reactivity/recover/1.0.0`) carrying either
a backfill or a resume exchange. Rather than rely on structural disambiguation of two payload shapes off the
wire, define an explicit discriminated wrapper here (db-core owns all reactivity wire codecs):

```ts
export interface RecoverRequestV1 {
  v: 1;
  kind: "backfill" | "resume";
  backfill?: BackfillV1;   // present iff kind === "backfill"
  resume?: ResumeV1;       // present iff kind === "resume"
}
export interface RecoverReplyV1 {
  v: 1;
  kind: "backfill" | "resume";
  backfillReply?: BackfillReplyV1;   // present iff kind === "backfill"
  resumeReply?: ResumeReplyV1;       // present iff kind === "resume"
}
```

Provide `validate*` + `encode*`/`decode*` for both, reusing `encodeCohortMessage`/`decodeCohortMessage` and
the `wire-validate.ts` helpers exactly like `backfill.ts`/`resume.ts` do. On decode, enforce that the
populated branch matches `kind` and the other branch is absent (a `kind: "backfill"` frame carrying a
`resume` body is malformed — `failWire`).

## Key tests (write up front, TDD)

- `backfillSigningPayload` / `resumeSigningPayload`: byte-for-byte deterministic for a fixed body; a change
  to **any** non-signature field changes the bytes; backfill and resume images with otherwise-identical
  fields never collide (the type tag differs). Round-trip: a body signed with a real Ed25519 key (use the
  same noble primitive the cohort-topic tests use, or a fixture key) verifies, and a mutated field fails to
  verify.
- `BackfillV1` validation: a frame missing `timestamp` is rejected; a non-finite `timestamp` is rejected;
  a valid frame round-trips through `encodeBackfillV1`/`decodeBackfillV1` byte-for-byte.
- `RecoverRequestV1`/`RecoverReplyV1`: `kind: "backfill"` with a `backfill` body round-trips; `kind:
  "resume"` with a `resume` body round-trips; a `kind`/branch mismatch (e.g. `kind: "backfill"` carrying a
  `resume` body, or both branches present, or neither) is rejected on decode; oversized frame respects the
  `maxMessageBytes` bound like the sibling codecs.
- `createBackfillRequester` stamps `timestamp` from the injected `clock` into the signed image (assert the
  request passed to the transport carries the clock's value).

## Edge cases & interactions

- **Backward-incompat wire change:** adding required `timestamp` to `BackfillV1` makes any pre-existing
  encoded frame undecodable. This is pre-release substrate (the recover transport is not yet live), so a
  hard break is acceptable — but `grep` for every `BackfillV1` literal / `encodeBackfillV1` call across
  `packages/` and fix them (db-core tests, and any db-p2p test fixtures — coordinate with the consuming
  ticket).
- **`fromRevision`/`toRevision` ordering** is already enforced by `reqIntInRange(obj, "toRevision", what,
  fromRevision)`; do not regress it when inserting `timestamp` into the validator.
- **Signing-payload field order is load-bearing:** signer and verifier must agree byte-for-byte. The
  array order in the helper is the contract — pin it with the determinism test so a later field reorder is
  caught.
- **`subscriberCoord` is part of the resume signing payload:** the db-p2p ticket sources it from the
  subscriber's real ring coordinate (not the collectionId placeholder). Because it is signed, the verifier
  recomputes over whatever value the request carries — no special handling here, but the field must remain
  in the resume signing-payload array so the two sides agree.
- **No clock in db-core proper:** keep `clock` injected (the requester dep), consistent with the rest of
  db-core's no-ambient-clock rule — do not call `Date.now()` in `backfill.ts`.
