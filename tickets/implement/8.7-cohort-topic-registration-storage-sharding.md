# Registration record storage, primary/backup sharding, and TTL renewal/failover

This ticket builds the cohort-side soft-state store and the deterministic load-sharding
that sits under it, plus the TTL lifecycle that keeps it fresh. It folds three coupled
concerns: record storage, primary/backup slot assignment, and TTL renewal/failover. They
are coupled because the slot function determines which member is `primary` for a stored
record, and TTL renewal/failover both read and mutate that assignment.

## Registration record store

Per `docs/cohort-topic.md` §Registration mechanics (L256-272):

```ts
export interface RegistrationRecord {
	topicId: Uint8Array;
	participantId: PeerId;
	tier: number;
	primary: PeerId;          // cohort member assigned to serve
	backups: PeerId[];        // 1..2 warm-failover members
	attachedAt: number;       // unix ms
	lastPing: number;         // unix ms
	ttl: number;              // ms, default 90000 (Edge 60000)
	appState?: Uint8Array;    // opaque; layer never interprets it
}

export interface RegistrationStore {
	put(rec: RegistrationRecord): void;
	getByParticipant(topicId: Uint8Array, participantId: PeerId): RegistrationRecord | undefined;
	listByTopic(topicId: Uint8Array): readonly RegistrationRecord[];
	delete(topicId: Uint8Array, participantId: PeerId): void;
	directParticipants(topicId: Uint8Array): number;   // stock count driving promotion
	evictStale(now: number): readonly RegistrationRecord[];  // now - lastPing > ttl
}
```

`appState` is the application's slot (reactivity stores `lastDeliveredRev`, matchmaking
stores provider metadata). The store never interprets it. Records are indexed for both
`getByParticipant` and `listByTopic`. Actual cross-member replication runs over cohort
gossip (next ticket); this ticket owns the **local** store and its indexes, and the
deterministic functions that the gossip layer and TTL loop call.

## Primary/backup sharding

Per `docs/cohort-topic.md` §Primary and backup sharding (L294-301):

```
order(cohortMembers) = sort(cohortMembers, by PeerId ascending)
slot(participantId, cohortEpoch) = H(participantId ‖ cohortEpoch) mod k
primary  = order[slot]
backups  = order[slot+1 .. slot+2  (mod k)]
```

```ts
export function assignSlots(participantId: PeerId, cohortEpoch: Uint8Array,
	cohortMembers: readonly PeerId[]): { primary: PeerId; backups: PeerId[] };
```

Deterministic given `(participantId, cohortEpoch, members)`; shards roughly evenly across
the `~k` members. Reuse FRET hash primitives for `H`.

### Membership-rotation handoff

Per §Membership rotation and primary handoff (L303-313): when membership changes, the slot
may move a registration's primary. The deterministic handoff:

1. New membership stabilizes; all members compute the new `cohortEpoch`.
2. Members exchange a "primary inventory" listing records they hold as primary.
3. For each record, the new computed primary either already holds it (no-op) or pulls it
   from the previous holder.
4. **Resolved open question (GROUNDING):** the previous primary continues to serve until
   the new primary acknowledges receipt — dual-serve until ack — to prevent a delivery gap.
5. Participants discover the new primary on their next ping (forwarded by old primary) or
   next inbound delivery; `cohortHint` refreshes on either path.

This ticket implements the slot recomputation, the inventory-exchange + pull + dual-serve
state machine, and the ack. The gossip *transport* for inventory comes from the next
ticket; expose the handoff as a function over an injected gossip/transport interface so it
can be unit-tested with a mock.

## TTL renewal and failover

Per §TTL and renewal (L274-282) and §Failure modes (L412-422):

- Participant pings `primary` every `ttl/3` (default 30s; Edge 20s).
- Success → primary updates `lastPing`, gossips the touch.
- Three consecutive failures → participant promotes `backups[0]` via a re-attach RPC
  carrying the existing record (no full re-registration); backup verifies it sees the
  record in its local replica and confirms.
- **Resolved open question (GROUNDING):** backup failover refreshes the participant's
  `cohortEpoch` hint on the *next* ping/renewal (not eagerly at failover time).
- All of primary+backups fail → participant re-runs lookup from `d_max` (walk ticket).
- Cohort-side: evict where `now − lastPing > ttl`; eviction is gossiped so members
  converge on the active set.

```ts
export interface RenewalParticipant {
	pingLoop(): void;             // every ttl/3; 3-fail -> reattach(backups[0])
	reattach(target: PeerId): Promise<RenewReplyV1>;
}
export interface RenewalCohortSide {
	onRenew(msg: RenewV1, now: number): RenewReplyV1;   // touch + maybe primary_moved
	sweepStale(now: number): readonly RegistrationRecord[]; // evictStale + gossip touch
}
```

`RenewReplyV1.primary_moved` is returned when the renew arrives at a member that is no
longer this record's primary after a rotation.

## TTL defaults

Edge `ttl` = 60s, Core `ttl` = 90s (per §Configuration). `ping_interval = ttl/3`.
