# Reactivity change-event tail + bridge membership-gate seam

The cohort-topic origination bridge gates each commit on `selfIsCohortMember(...)` before handing
`(event, commitCert)` to `CohortTopicService.onLocalCommit`. The concrete predicate (supplied by the
node-activation ticket) must decide membership for the collection's **reactivity topic**, which is
`topicId = H(currentTailId ‖ "reactivity")` — a *rotating*, tail-anchored topic (`docs/reactivity.md`
§Anchor). So the predicate needs the collection's **current tail block id** at emit time.

Today `CollectionChangeEvent` carries `{ collectionId, blockIds, actionId, rev }` — **not** the tail.
The tail cannot be recovered from `blockIds` (it is a set, the tail need not be among them, and the
reactivity cohort at `coord_0(H(tail‖"reactivity"))` is a *different* ring coordinate from the storage
cluster that holds any given block — so guessing is wrong), and reading the collection header block to
find the tail is async while the change-listener contract is synchronous. The authoritative,
synchronous source is the `CommitRequest.tailId` that `StorageRepo.commit` already receives — it is the
collection's chain tail (`NetworkTransactor` commits it as the distinguished tail block; the collection
header's `tailId` is the same chain tail). This ticket plumbs that tail onto the change event and feeds
the whole event to the bridge gate.

## Scope: this is the seam only

No libp2p node assembly, no host construction, no FRET membership logic — those land in
`cohort-topic-host-node-activation`. This ticket ships only the data-flow plumbing + the bridge
predicate/teardown shape, all unit-testable without a real node.

## What to change

### 1. db-core — `CollectionChangeEvent.tailId`

Add an **optional** field:

```ts
export type CollectionChangeEvent = {
	readonly collectionId: CollectionId;
	readonly blockIds: readonly BlockId[];
	readonly actionId: ActionId;
	readonly rev: number;
	/**
	 * The collection's chain tail block id at the time of this commit (the `CommitRequest.tailId`).
	 * Anchors the rotating reactivity topic `H(tailId ‖ "reactivity")`. Present on commit-driven
	 * events; `undefined` on read-driven promotions (the `StorageRepo.get` path has no commit tail —
	 * those never originate anyway, and are cert-gated out downstream).
	 */
	readonly tailId?: BlockId;
};
```

Optional → backward-compatible: every existing consumer ignores it; assertions that read named fields
(not whole-object deep-equal) are unaffected.

### 2. db-p2p — `StorageRepo`: thread `request.tailId`

`commit(request)` already groups newly-committed blocks by collection and calls
`emitCollectionChanges(collectionBlocks, request.actionId, request.rev)` after the critical section.
Thread `request.tailId` through so each commit-path event carries it. A commit is for a single
collection's chain in practice (one `tailId` per `CommitRequest`); if `collectionBlocks` ever holds
more than one collection, every event from that commit carries the same `request.tailId` — harmless,
because the gate for a non-matching collection derives the wrong topic and simply fails the membership
check (origination is hint-only and additionally cert-gated). Document this single-collection-per-commit
assumption inline.

The **read-driven promotion** path (`get` → `emitPromotions` → `emitCollectionChanges`) has no commit
request and thus no tail: pass `tailId: undefined` there. Those events must not originate (the cohort
cert store is populated only on the consensus-commit path, so the extractor already returns `undefined`
and the bridge no-ops — but an undefined tail makes the gate fail first, which is the intended order).

Keep `emitCollectionChanges`'s signature change minimal (add a `tailId?: BlockId` parameter; the
promotion grouping passes `undefined`).

### 3. db-p2p — `change-bridge.ts`: widen the gate + thread teardown

- **Widen the predicate.** Change `selfIsCohortMember: (collectionId: CollectionId) => boolean` to
  `selfIsCohortMember: (event: CollectionChangeEvent) => boolean`, and call it as
  `deps.selfIsCohortMember(event)` in `originate`. The predicate is still membership-gate agnostic; it
  simply now receives the full event (it needs `event.tailId`, and `event.collectionId` stays
  available). Update the JSDoc on `CohortTopicChangeNotifierDeps.selfIsCohortMember`.
- **Thread the catch-all unsubscribe.** `makeCohortTopicChangeNotifier` subscribes to
  `source.onAnyCollectionChange(...)` and discards the unsubscribe (node-lifetime by intent). Change
  `attachCohortChangeBridge` to **return** `{ notifier, unsubscribe }` where `unsubscribe` tears down
  that catch-all subscription, so the node-activation ticket can release it on node stop (alongside
  `host.stop()`). Keep `makeCohortTopicChangeNotifier`'s existing `IBlockChangeNotifier` return for the
  unit tests; factor the shared subscribe/originate wiring so the two entry points don't diverge (e.g.
  `attachCohortChangeBridge` captures the unsub directly rather than delegating to
  `makeCohortTopicChangeNotifier`). Idempotent unsubscribe (the underlying `onAnyCollectionChange` unsub
  already is).
- **Tidy the node typing.** `attachCohortChangeBridge(node: { blockChangeNotifier?: IBlockChangeNotifier }, …)`
  is fine as a structural type; ensure the call site in the activation ticket can pass the libp2p node
  without an `as any` on `.blockChangeNotifier` (the activation ticket may cast once to this structural
  type). No change required here beyond confirming the exported type is usable; do not narrow it.

## Edge cases & interactions

- **Read-driven promotion events** carry `tailId: undefined` → gate returns false (no tail → no topic).
  Add a `storage-repo.spec.ts` case asserting a promotion-path event has `tailId === undefined` and a
  commit-path event has the committed `tailId`.
- **Multi-collection commit** (theoretical): all events from one commit share `request.tailId`. Assert
  (or at least document) the single-collection-per-commit invariant; no crash either way.
- **Delete commits** (`newBlock` undefined → collectionId from prior header): tail still comes from
  `request.tailId`, independent of the block-presence fallback. The existing delete test should still
  pass with an added `tailId` assertion.
- **Backward compatibility**: `tailId` optional; existing `onCollectionChange` subscribers (Quereus
  reactive-watch) ignore it. Confirm storage-repo + change-bridge suites stay green after the
  predicate-signature and return-shape updates.
- **change-bridge.spec.ts churn**: the 7 existing bridge cases pass `selfIsCohortMember: (c: CollectionId) => …`;
  update each to `(e) => …` (e.g. `(e) => e.collectionId === ('collection-1' as CollectionId)`), and
  update any usage of `attachCohortChangeBridge`'s return shape (only `makeCohortTopicChangeNotifier` is
  used directly today, so the new `{ notifier, unsubscribe }` is additive — add a focused case asserting
  `unsubscribe()` stops further origination).

## Key tests (TDD targets)

- StorageRepo: a `pend`+`commit` with `tailId: T` fires a catch-all event whose `tailId === T`; a
  read-driven promotion fires an event whose `tailId === undefined`.
- change-bridge: gate receives the full event (assert it can branch on `event.tailId`); a member commit
  with a retained cert still invokes `onLocalCommit(event, cert)`; `attachCohortChangeBridge(...)`
  returns a working `unsubscribe` that stops subsequent origination while leaving per-collection
  `onCollectionChange` delegation intact.
