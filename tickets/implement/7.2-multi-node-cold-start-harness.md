## Why

`fresh-node-ddl.spec.ts` runs on a 1-node mesh. Ticket 5 class bugs (pending-only metadata, chain-add on fresh collection, coordinator-repo blockId extraction) all have multi-node failure modes that were invisible under `clusterSize=1` because the "peerCount<=1 fast path" in `verifyResponsibility` bypasses most of the interesting logic. Mesh-sanity's Suites 1/2 exercise multi-node paths, but only with synthetic `Transforms` — they never drive `Collection`/`Tree` on top.

## What to build

New spec `packages/db-p2p/test/fresh-node-ddl-multi.spec.ts` with two scenarios:

### Scenario A — 3-node cold-start, DDL on A / SELECT on B

Fresh `createMesh(3, { responsibilityK: 3, clusterSize: 3, superMajorityThreshold: 0.67 })`.

Build one `NetworkTransactor` per node (each sees the same mock key network but `getRepo` returns the local `coordinatorRepo` for self and a thin in-mesh proxy for peers). Then:

1. On node A's transactor: `Tree.createOrOpen('t')` + `tree.replace([[1, {...}]])` (first DDL/DML ever on the mesh).
2. On node B's transactor: `Tree.createOrOpen('t')` + `tree.get(1)` — assert the value written via A round-trips.
3. On node C's transactor: identical read — confirms all three replicas converged.
4. Per-test timeout 10s (multi-node consensus is slower than solo).

### Scenario B — 5-node cold-start with one peer down at boot

`createMesh(5, { responsibilityK: 5, clusterSize: 5, superMajorityThreshold: 0.6 })`, then pre-populate `mesh.failures.failingPeers = new Set([nodes[4].peerId.toString()])` *before* the first transaction.

Super-majority = 0.6 of 5 = 3 required; 4 peers available still satisfies super-majority, so:

1. Node A: `Tree.createOrOpen` + `replace`. Must complete despite node E being unreachable.
2. Node B: `tree.get` — must return the value.
3. Per-test timeout 15s.

### Helper

Add `buildNetworkTransactors(mesh)` to the Phase 2 `buildNetworkTransactor` helper: returns `Map<peerIdStr, ITransactor>` so tests can drive operations from any node. Alternatively, a single `buildNetworkTransactor(mesh, localPeerIndex)` called per node — whichever reads cleaner.

## Explicit non-goals

- No real libp2p. Mock mesh only.
- No partition / network-delay chaos. Those belong in byzantine-fault-injection.spec.ts, not here.
- No performance assertions. Correctness only.

## Expected outcomes

- Either the scenarios pass on `main` and we have durable multi-node cold-start coverage, or one of them fails — in which case we file a fix ticket with the deterministic repro (which is what this harness exists to produce).
- Future multi-node cold-start bugs are caught in CI, not on multi-device canaries.
