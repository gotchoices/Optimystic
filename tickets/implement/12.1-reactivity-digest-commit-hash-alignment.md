# Align `NotificationV1.digest` with the threshold-signed commit-vote bytes

## Decision (design resolved → Option 1, refined)

Reactivity reuses the commit certificate's threshold signature **bit-for-bit** and never re-signs. A
subscriber/forwarder verifies it by recomputing the cohort threshold check over `payload =
b64urlToBytes(n.digest)` (`packages/db-core/src/reactivity/verify.ts:64`). For a **real** Ed25519
threshold-verify to succeed, `digest` must carry the exact bytes each member signed.

**What each member actually signs.** The commit cert's `thresholdSig` is the aligned concatenation of
the cohort's per-member `approve` **commit** signatures (`buildCommitCert`,
`packages/db-p2p/src/cluster/commit-cert.ts`). Each such signature is a raw Ed25519 signature over the
cluster commit-vote preimage (`packages/db-p2p/src/cluster/cluster-repo.ts:528-537`):

```
signedPayload = utf8( commitHash + ":approve" )
commitHash    = base64url( sha256( messageHash ‖ canonicalJson(message) ‖ canonicalJson(promises) ) )
```

This preimage is **byte-identical across every approving signer** (they all sign the same
`commitHash:approve` string) and is reproducible on the committing node from the agreed `ClusterRecord`.

**Why Option 1, not Option 2.** The ticket's Option 2 (establish `actionId == commit-hash`) is
**impossible**: `event.actionId` is the transaction id (`hash(stampId ‖ statements ‖ reads)`,
`docs/transactions.md` §Transaction Identity) — a different value from `commitHash`, and the signed
preimage is not even the bare `commitHash` but `utf8(commitHash + ":approve")`. The only correct fix is
**Option 1**: surface the exact signed preimage on `CommitCert` and have `buildNotificationV1` set
`digest` from it. Today origination sets `digest = b64url(utf8(actionId))`, which is why every
crypto-path test here uses a pass-crypto stub (`ICohortThresholdCrypto.verify = () => true`) or a fake
`NotificationVerifier`.

The verify side already does the right thing (`payload = b64urlToBytes(n.digest)`), so **no logic change
in `verify.ts`** — only origination + the cert shape + the cluster capture path change.

## Architecture

### `CommitCert` gains the signed preimage (db-core)

`packages/db-core/src/transactor/change-notifier.ts` — add to the `CommitCert` type:

```ts
export type CommitCert = {
	readonly thresholdSig: Uint8Array;
	readonly signers: readonly string[];
	readonly minSigs: number;
	/**
	 * The exact byte preimage each `signers[i]` signed to produce its 64-byte chunk of
	 * {@link thresholdSig} — the cluster's per-member commit-vote payload `utf8(commitHash + ":approve")`,
	 * identical across all approving signers. Reactivity sets a notification's `digest` to
	 * base64url(signedPayload) so a subscriber's threshold-verify over `digest` reproduces the exact
	 * signed image. Opaque to db-core; minted only by the cluster layer.
	 */
	readonly signedPayload: Uint8Array;
};
```

### Origination uses it (db-core)

`packages/db-core/src/reactivity/notification.ts` — `buildNotificationV1`:

```ts
// before:  digest: bytesToB64url(utf8.encode(event.actionId)),
// after:
digest: bytesToB64url(commitCert.signedPayload),
```

Drop the now-unused `utf8.encode(event.actionId)` for `digest` (keep `utf8` if still used by `sigDigest`).
Rewrite the `buildNotificationV1` doc comment: `digest` now carries the commit-vote signed payload that
deterministically reproduces the threshold-signed image; the seam is **closed**, not "remaining."

### Cluster capture computes and carries it (db-p2p)

`packages/db-p2p/src/cluster/commit-cert.ts` — `buildCommitCert(record, minSigs)` →
`buildCommitCert(record, minSigs, signedPayload: Uint8Array)`; set `signedPayload` on the returned cert.
Keep it pure (caller supplies the bytes; do **not** recompute `commitHash` here — `canonicalJson` is a
private static on `ClusterMember`; duplicating it risks drift).

`packages/db-p2p/src/cluster/cluster-repo.ts` — in `applyConsensusOperation`'s `commit` branch
(around line 794-799), compute the preimage with the existing private helpers **before** the
synchronous capture+commit sequence, then pass it through:

```ts
if ('commit' in operation) {
	const commit = operation.commit;
	// computed before the synchronous capture+commit below — see ordering note
	const commitSignedPayload = this.computeSigningPayload(await this.computeCommitHash(record), 'approve');
	this.captureCommitCert(record, commit.actionId, commitSignedPayload);
	...
}
```

`captureCommitCert(record, actionId, signedPayload)` stays **synchronous** (no `await` inside) and
forwards `signedPayload` into `buildCommitCert(record, minSigs, signedPayload)`. The `await
computeCommitHash` happens *before* `StorageRepo.commit()` is called, preserving the
"cert retained before commit() emits its synchronous change event" invariant the existing comment at
`cluster-repo.ts:796-798` guards.

## Edge cases & interactions

- **Required-field migration.** Adding `signedPayload` makes every `CommitCert` literal a compile error.
  Grep for `CommitCert` literals and `thresholdSig:` and fix all: the `buildCommitCert` call site,
  `packages/db-p2p/test/cohort-topic/change-bridge.spec.ts` (the `buildCommitCert`/`CommitCertStore`/
  extractor literals — give them a `signedPayload`, e.g. `utf8('hash-1:approve')`),
  `packages/db-p2p/test/cluster-consensus-divergence.spec.ts`, and any others surfaced by `tsc`.
- **Synchronous capture ordering (do not regress).** The preimage must be computed *before* the
  synchronous `captureCommitCert` + `StorageRepo.commit()` run, because `commit()` emits the
  `CollectionChangeEvent` synchronously and the reactivity bridge resolves the cert from the store at
  that instant. Do **not** make `captureCommitCert` itself async / defer the `store.put` past `commit()`
  — that reintroduces a race where origination silently skips. (Reuse an already-awaited `commitHash` if
  one exists earlier in the path; otherwise the one extra `sha256` is negligible.)
- **Cross-node / cross-signer determinism.** All approving members sign the same
  `utf8(commitHash + ":approve")`; the capturing node recomputes the same `commitHash` from the agreed
  `record` it already verified signatures against. A diverged record would yield a preimage that fails
  verify — which is the correct, fail-closed outcome.
- **Signer round-trip.** Origination maps each cluster peer-id string via `encodeSigner`
  (db-p2p: `s ⇒ bytesToB64url(peerIdToBytes(s))`); verify maps each wire signer via `signersToBytes`
  (default `b64urlToBytes`) → `verifyPeerSig` → `bytesToPeerIdString` → `peerIdFromString(...).publicKey`.
  The round-trip `peerIdToBytes`↔`bytesToPeerIdString` (UTF-8 of the peer-id string, per `peer-codec.ts`)
  must hold; exercise it with **real** Ed25519 peer ids in the integration test.
- **`minSigs` source.** The verifier uses **its own** configured `minSigs` (`MembershipVerifierDeps`,
  default `DEFAULT_MIN_SIGS = 14`), **not** `cert.minSigs`. The integration test must set the verifier's
  `minSigs ≤` the number of approving signers it constructs. Below-threshold signer count → `untrusted`.
- **Chunk stride.** `thresholdSig` must be exactly `64 × signers.length`; `buildCommitCert` excludes
  `reject` votes, so only fixed-width approve sigs are concatenated. A non-64-byte chunk desyncs the
  stride → `verifyCollectedMultisig` returns false → `untrusted` (already handled; assert it stays so).
- **`mergedDigest` ripple (checkpoint).** `checkpoint.ts:196` folds `b64urlToBytes(e.payload.digest)`
  into `mergedDigest`. The new `digest` is still a deterministic, per-revision, cross-member-identical
  value, so the fold stays convergent under gossip; it remains a **hint** (not cryptographically
  verified — the bracketing endpoints are). Only the doc wording changes; no fold-logic change.
- **Pre-cached cert skips self-consistency.** `MembershipVerifier.verifyMessage` trusts a `cache()`d
  cert without the self-consistent-quorum check (that check runs only on the fetch path,
  `verifier.ts:110-121`). The integration test can therefore `cache()` a hand-built `MembershipCertV1`
  whose `members` ⊇ the signer member-ids without minting a valid cert thresholdSig — production fetch
  still self-validates.
- **Empty approvals.** No `approve` commits ⇒ empty `thresholdSig`/`signers` ⇒ `untrusted`; origination
  already requires a cert (bridge skips when absent). Unchanged.

## Key tests (write first, TDD)

- **Unit — cert carries preimage** (`change-bridge.spec.ts`): `buildCommitCert(record, n, payload)`
  returns `signedPayload === payload`; round-trips through `CommitCertStore` and the extractor unchanged.
- **Unit — origination sets digest from preimage** (db-core notification spec): `buildNotificationV1`
  yields `n.digest === bytesToB64url(commitCert.signedPayload)` and is independent of `event.actionId`.
- **Integration — REAL threshold over `digest`, no pass-crypto stub** (new spec under
  `packages/db-p2p/test/cohort-topic/`, e.g. `reactivity-real-crypto.spec.ts`):
  1. Generate N real Ed25519 keypairs (`generateKeyPair('Ed25519')`); pick a `tailId`.
  2. Produce real per-member `approve` commit signatures over `utf8(commitHash + ":approve")` (drive a
     real `clusterMember` commit and capture via the `onCommitCertificate` sink, **or** construct a
     `ClusterRecord` with real signatures and call `buildCommitCert`) → a `CommitCert` with real
     `thresholdSig`, `signers`, and `signedPayload`.
  3. `buildNotificationV1(event, cert, { ..., encodeSigner: s ⇒ bytesToB64url(peerIdToBytes(s)) })`.
  4. Build a db-core `CohortSigner` whose `ICohortThresholdCrypto.verify` is the **real**
     `verifyCollectedMultisig` (exported from `packages/db-p2p/src/cohort-topic/threshold-crypto.ts`) —
     no `() => true` stub. Wire it into a `MembershipVerifier`; `cache()` a `MembershipCertV1` whose
     `cohortCoord === bytesToB64url(coord0(reactivityTopicId(tailId)))` and whose `members` include each
     signer's member-id bytes; set verifier `minSigs = N`.
  5. `createNotificationVerifier({ verifier, ... }).verify(notification)` ⇒ **`"verified"`**.
  6. Negative cases ⇒ **`"untrusted"`**: (a) flip one byte of `digest`; (b) reconstruct the notification
     the *old* way (`digest = b64url(utf8(actionId))`) to prove the old encoding fails real crypto;
     (c) drop a signer below `minSigs`; (d) truncate one chunk of `thresholdSig`.
