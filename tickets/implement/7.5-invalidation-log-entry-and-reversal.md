# Invalidation log entry + single-collection reversal

This is the **core** of the transaction-invalidation cascade (the broader cascade and
client notification land in `7.5-invalidation-cascade-detection` and
`7.5-invalidation-client-notification`, both `prereq`'d on this). This ticket establishes the
durable reversal primitive for a **single** invalidated transaction in **one** collection. It does
not yet walk read-dependents or span multiple collections — it builds the entry type, the
compensating-state computation, the consensus path that lands it, and the certificate that proves it.

## Background (resolved design)

When the Right-is-Right dispute protocol resolves with `outcome: 'challenger-wins'`
(`packages/db-p2p/src/dispute/types.ts` → `DisputeResolution`), the committed transaction `T_inv`
has been *proven invalid* by a 2/3 super-majority of independently-signed arbitrator votes. Today
that outcome only flips an **in-memory** status (`DisputeService.getDisputeStatus` →
`committed-invalidated`, held in a per-node `Map`); the committed state is never actually reversed,
and nothing is persisted or replicated. This ticket makes the reversal real and durable.

### Why a compensating entry (not void-in-place, not rewrite)

The collection log is an append-only, prior-hash-linked chain (`Log` /
`packages/db-core/src/log/log.ts`, `LogBlock.priorHash`); durability and sync integrity
(`docs/correctness.md` Theorems 6 & 14) depend on entries being immutable once committed. Therefore:

- **Rewriting the log (option c)** breaks every downstream `priorHash` link and the commit
  certificates that anchor them — rejected.
- **Marking the entry void in place (option b)** mutates a signed, content-addressed committed entry —
  breaks integrity and replays — rejected.
- **Appending a compensating "invalidation" entry (option a)** preserves the audit trail and every
  existing invariant. **This is the chosen representation.**

The invalidation is itself a consensus-ordered, replicated, signed event — recoverable on sync exactly
like a commit.

### New log entry variant

`LogEntry<TAction>` (`packages/db-core/src/log/struct.ts`) gains a third optional arm alongside
`action?` and `checkpoint?`:

```typescript
/** A compensating entry that reverses a previously-committed action proven invalid by dispute. */
export type InvalidationEntry = {
	/** actionId of the committed ActionEntry being reversed */
	readonly invalidatedActionId: ActionId;
	/** rev of the invalidated entry — pins which block revisions to roll back */
	readonly invalidatedRev: number;
	/** Authoritative proof: a challenger-wins DisputeResolution carrying the signed arbitrator votes */
	readonly resolution: DisputeResolutionProof;
	/** Per-block compensating result this entry produced (new monotonic revision restoring pre-T_inv content) */
	readonly reverted: ReadonlyArray<{ blockId: BlockId; fromRev: number; restoredContentHash: string }>;
	/** When this invalidation is a cascade step, the root invalidation that triggered it (unused this ticket; see cascade ticket) */
	readonly cascadeRoot?: ActionId;
};

export type LogEntry<TAction> = {
	readonly timestamp: number;
	readonly rev: number;
	readonly action?: ActionEntry<TAction>;
	readonly checkpoint?: CheckpointEntry;
	readonly invalidation?: InvalidationEntry;
};
```

`DisputeResolutionProof` is the subset of `DisputeResolution` needed for independent verification
(outcome, the `ArbitrationVote[]` with their Ed25519 signatures, the disputeId, and the originating
`messageHash`). Keep it in `db-core` (the log layer is db-core and must not import db-p2p);
`db-p2p`'s `DisputeResolution` maps onto it. **Do not** widen db-core's dependency graph — define the
proof type in db-core and have the dispute layer construct it.

### Reversal semantics (append-forward, never delete a revision)

Block revisions are monotonic and prior revisions are retained
(`packages/db-p2p/src/storage/block-storage.ts` → `materializeBlock`, `restoreBlock`, `ensureRevision`).
The invalidation does **not** delete `T_inv`'s revisions. Instead, for each block `T_inv` wrote, it
produces a **new** revision whose content equals the block's state *as if `T_inv` had not committed* —
i.e. `materializeBlock(blockId, invalidatedRev - 1)` with any surviving later actions replayed on top.
In this single-collection/no-cascade core, "surviving later actions" = every committed action after
`T_inv` on that block (none are reverted yet); the cascade ticket replaces this with re-evaluation.
The compensating transform is the forward delta from the current tip to that recomputed content.

### Consensus path

The invalidation is committed through the **same critical-cluster machinery** as any transaction
(`ClusterMember` / `cluster-repo.ts` → `handleConsensus` / `applyConsensusOperation`), so it takes a
revision slot and serializes against concurrent commits with **no new global lock**. Each member
applies the same InvalidationEntry deterministically (it carries the `reverted` targets and the proof),
exactly as `applyConsensusOperation` applies committed operations on every peer.

### Invalidation certificate

A member accepts/applies an InvalidationEntry only if its `resolution` is a valid invalidation
certificate: `outcome === 'challenger-wins'`, the arbitrator `votes` carry verifiable Ed25519
signatures, and the `agree-with-challenger` votes meet the 2/3 decisive-vote threshold
(mirrors `DisputeService.resolveDispute`). This is the reversal analogue of the commit certificate
("Commit Certificate Is Authoritative", `docs/transactions.md`): a compromised peer can withhold an
invalidation but **cannot forge** one.

## Edge cases & interactions

- **Idempotent re-receipt.** The same `challenger-wins` resolution arriving twice (rebroadcast, sync,
  retry) must append the InvalidationEntry **exactly once**. Key dedup on
  `(invalidatedActionId, disputeId)`; a second arrival is a no-op (reuse the
  `executedTransactions` / `stateStore.wasExecuted` dedup pattern in `cluster-repo.ts`).
- **`T_inv` already superseded.** A later committed action may have already overwritten every block
  `T_inv` touched (the read-dependents). Reversal must still recompute the correct
  "as-if-`T_inv`-never-ran" content for the *current* tip, not blindly restore `invalidatedRev - 1`.
  In the no-cascade core, if a later non-dependent action exists, recompute by replaying it on the
  rolled-back base; flag genuinely-dependent successors as out-of-scope-for-this-ticket (the cascade
  ticket handles them) but **log** that they were left untouched.
- **`T_inv` not found locally.** A member that never saw `T_inv` (was behind / just joined) must fetch
  it via the existing restore/sync path before applying the invalidation, or defer until it has it —
  never apply a reversal against missing base state.
- **Invalid / forged certificate.** `outcome !== 'challenger-wins'`, missing votes, bad signatures, or
  sub-threshold agreement → reject the InvalidationEntry, do not append, log. Never trust a single
  peer's say-so.
- **Concurrent commit racing the invalidation.** A normal transaction pending on the same critical
  block while the invalidation commits: the existing per-block commit serialization orders them; the
  later one observes the earlier's revision. Add a test that interleaves a commit and an invalidation
  on one block and asserts both land in a consistent order with monotonic revisions.
- **Crash mid-invalidation.** Persisted 2PC/consensus state must let recovery complete or cleanly
  abandon a half-applied invalidation (same guarantees as commit recovery, `cluster-repo.ts` +
  `BlockStorage.recover`). The InvalidationEntry is all-or-nothing per the consensus apply.
- **Reads during the window.** Between resolution and the InvalidationEntry landing, status is
  `committed-disputed`; reads serve the last committed (still-invalid) revision. This is acceptable —
  eventually-consistent reads, serialized writes. Document it; do not add a read freeze.
- **Determinism across engines.** The compensating content is computed from stored revisions, not by
  re-running the engine, so it does not depend on engine availability — but the `restoredContentHash`
  must match on every member. Assert cross-member agreement in a multi-node test.

## Key tests

- Append an InvalidationEntry for a single-block `T_inv`; assert a new revision restores the
  pre-`T_inv` content and `priorHash` chain stays intact.
- Certificate verification: a `majority-wins` / `inconclusive` / unsigned-vote resolution is rejected;
  a valid `challenger-wins` is accepted.
- Idempotency: applying the same resolution twice yields one entry / one revision.
- Multi-node: all cluster members converge on the same restored content hash and revision.
- Crash-recovery: kill a member mid-apply; on recover the invalidation is either complete or absent,
  never partial.
- Status: after the entry lands, the transaction is durably reported `committed-invalidated` from the
  log (groundwork for the notification ticket).
