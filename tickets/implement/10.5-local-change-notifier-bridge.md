# Bridge local IBlockChangeNotifier into the cohort-topic notification origination point

This ticket connects the **existing local single-node** change-notification primitive to
the **networked** cohort-topic substrate. It is the origination point: a commit that lands
on a node which is a cohort member for that collection's topic emits into the substrate, so
reactivity (and later matchmaking) can fan out. Without this bridge, the substrate has no
source of change events.

## What already exists (GROUNDING — do not rebuild)

- `packages/db-core/src/transactor/change-notifier.ts`: `IBlockChangeNotifier.onCollectionChange`
  + `CollectionChangeEvent { collectionId, blockIds, actionId, rev }`. Listeners run AFTER
  the commit's critical section, synchronously, in commit order; a throwing listener must
  not break the commit (log + continue).
- `packages/db-p2p/src/storage/storage-repo.ts`: emits the event when a commit's critical
  section completes **on that node** (`emitCollectionChanges`).
- `packages/db-core/src/transactor/network-transactor.ts` (~L75-80): forwards to an
  optional `localChangeNotifier`, else a logged no-op.
- `packages/db-p2p/src/cluster/cluster-repo.ts`: assembles the commit certificate carrying
  the cluster-consensus threshold signature (signers + `minSigs`).

The signal today only reaches listeners **in the same process** where the commit applied.
This bridge feeds that signal into `CohortTopicService` at that node.

## The bridge

Provide a `localChangeNotifier` implementation that, on each `CollectionChangeEvent`,
checks whether this node is a member of the cohort responsible for the collection's
reactivity topic; if so, hands the event (with the commit-certificate threshold signature
attached, unchanged) to `CohortTopicService` as the origination point.

```ts
export interface LocalChangeHook {
	onCollectionChange(event: CollectionChangeEvent, commitCert: {
		thresholdSig: Uint8Array;   // extracted from cluster-repo commit certificate, forwarded UNCHANGED
		signers: PeerId[];
		minSigs: number;
	}): void;
}

export function makeCohortTopicChangeNotifier(deps: {
	service: CohortTopicService;
	selfIsCohortMember: (collectionId: CollectionId) => boolean;   // membership check at this node
	extractCommitCert: (event: CollectionChangeEvent) => CommitCert; // from cluster-repo
}): IBlockChangeNotifier;   // or the localChangeNotifier shape network-transactor expects
```

- **Member commit** → invoke `service.onLocalCommit` (the integration hook from the
  core-module ticket) with `collectionId`, `rev`, and the commit certificate's threshold
  signature **forwarded unchanged** (no re-sign — reactivity reuses the commit cert
  directly).
- **Non-member commit** → no-op (this node isn't responsible for fan-out of this topic).
- Throwing must not break the commit (matches `IBlockChangeNotifier` contract: log +
  continue).

The signature extraction reads the existing commit certificate from `cluster-repo.ts`; it
must forward the same bytes/signers/minSigs the cluster consensus produced — the bridge
never re-signs.

## Out of scope

- Reactivity `NotificationV1` production / fan-out / replay (its own ticket
  `reactivity-origination-replay-delivery`).
- Matchmaking.
This ticket only delivers the event + commit-cert into the substrate's origination hook.

## Constraints

ES modules, no inline `import()`, no `any`, tabs. The db-core notifier shape stays
cross-platform; the membership check + cert extraction live in db-p2p (libp2p/cluster
context). Don't break existing transactor/storage tests.
