## Summary

A transaction's `stamp` carries the metadata a validating node trusts: `peerId`,
`timestamp`, `schemaHash`, `engineId`, `expiration`, `nonce`. `stamp.id` is a SHA-256 hash
of exactly those six fields (`stamp:<base64url>`), and the client signature is made over a
payload built from `stamp.id` + statements + reads.

Nothing re-verifies that `stamp.id` actually equals the hash of the fields. So an attacker
who captures a validly-signed transaction can mutate a stamp field **while holding
`stamp.id` and `signature` constant** — the signature payload (built from the unchanged
`stamp.id`) still verifies, so the transaction is accepted with a forged field.

The fix: `TransactionValidator.validate` must recompute the id from the received stamp
fields and reject on mismatch, **before** it trusts any of those fields.

### Reproduced (fix stage)

A throwaway spec confirmed the exploit: build a signed transaction with `ttlMs=1` (so it
would expire immediately), then set `stamp.expiration = Date.now() + 10_000_000` without
touching `stamp.id` or `signature`. Today `validate` returns `{ valid: true }` — the forged
expiration is accepted and the TTL-bounded replay window is defeated. `peerId` is partially
protected (the real verifier derives the pubkey from `stamp.peerId`, so tampering it breaks
a victim's signature), but `expiration` / `engineId` / `schemaHash` / `timestamp` / `nonce`
have no second line of defense.

## Current code

`packages/db-core/src/transaction/transaction.ts` — the id is computed inline, with a fixed
field order that MUST be preserved to keep existing ids/signatures stable:

```ts
const stampData = JSON.stringify({ peerId, timestamp, schemaHash, engineId, expiration, nonce });
const id = `stamp:${await hashString(stampData)}`;
```

`packages/db-core/src/transaction/validator.ts` — `validate()` reads `stamp.expiration`
(step 0), `stamp.peerId` (step 0.5 signature), `stamp.engineId` (step 1), `stamp.schemaHash`
(step 2) straight off the wire, all before any integrity check exists.

## Design

Extract a pure helper so creation and validation derive the id **identically** — the whole
point is that there is exactly one place that defines "the hash of the stamp fields".

```ts
/** The stamp fields that feed the id hash (everything except the id itself). */
export type StampFields = Omit<TransactionStamp, 'id'>;

/**
 * Compute the canonical stamp id from its fields. MUST match the historical field order
 * (peerId, timestamp, schemaHash, engineId, expiration, nonce) — changing it churns every
 * existing stamp id and client signature. Both createTransactionStamp and the validator's
 * integrity check derive the id through this one function.
 */
export async function computeStampId(fields: StampFields): Promise<string> {
	const { peerId, timestamp, schemaHash, engineId, expiration, nonce } = fields;
	const stampData = JSON.stringify({ peerId, timestamp, schemaHash, engineId, expiration, nonce });
	return `stamp:${await hashString(stampData)}`;
}
```

Note: build the JSON from the destructured locals in fixed order (as above) rather than
passing `fields` straight to `JSON.stringify` — that guards against a caller who slipped an
`id` (or extra) property into the object changing the serialized bytes.

`createTransactionStamp` then calls it after generating the nonce:

```ts
const nonce = uint8ArrayToString(randomBytes(16), 'base64url');
const id = await computeStampId({ peerId, timestamp, schemaHash, engineId, expiration, nonce });
return { peerId, timestamp, schemaHash, engineId, expiration, nonce, id };
```

`TransactionValidator.validate` gains **step 0.0**, before the expiration check:

```ts
const { stamp } = transaction;

// 0.0. Integrity: the id must be the true hash of the stamp fields. Runs FIRST because
// every later step (expiration, signature, engine/schema selection) trusts a stamp field.
// Always-on (one SHA-256, cheap) and independent of whether a signature verifier is wired —
// a tampered stamp corrupts expiration and engine selection regardless of signatures.
const expectedId = await computeStampId(stamp);
if (expectedId !== stamp.id) {
	return { valid: false, reason: 'Tampered transaction stamp' };
}

// 0. Check expiration ...
```

Passing `stamp` (a full `TransactionStamp`) where `StampFields` is expected is fine —
`computeStampId` only reads the six field props and ignores `id`.

### Decisions (already made — implement as stated)

- **Always-on**, not gated on the signature verifier. It is one SHA-256 and the fields are
  trusted on every validation path.
- **Ordered before expiration** (step 0.0), since expiration itself consumes an untrusted
  field.
- **Do not change** the hash formula or `clientSignaturePayload` format — this is purely an
  added verification, so existing ids/signatures stay valid.

## Tests (add to the `Client Transaction Signatures` block)

The block already has `makeValidator` / `buildSignedTx` helpers (transaction.spec.ts
~line 3996). Add cases that tamper a stamp field **after** signing while holding `stamp.id`
and `signature` constant, and assert rejection with reason `Tampered transaction stamp`:

- `expiration` extended far into the future (mirror the reproduced exploit; assert it is
  now rejected — this is the headline regression guard).
- `engineId` changed.
- `schemaHash` changed.
- `timestamp` changed.
- Regression: a legitimately-built signed transaction still validates (`valid: true`).
- The integrity check fires even with **no verifier wired** — `makeValidator(undefined)`,
  tamper `stamp.expiration`, expect `Tampered transaction stamp` (proves it is not gated on
  signatures).

Tamper by casting, matching the existing style, e.g.
`(tx.stamp as { expiration: number }).expiration = Date.now() + 10_000_000;`.
