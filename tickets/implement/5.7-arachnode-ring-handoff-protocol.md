Implement the **advertise → confirm-replication → release** handoff from
`docs/arachnode-ring-handoff.md` § Part 2, and the concurrent-move / crash / membership rules in
§ Part 3, so a ring shift (or a lost-block rebalance) never drops a key below its replication floor
`N`. This ticket protects the **replication-floor invariant** stated in the spec.

There are two live code paths that currently release responsibility before confirming replication;
both must be brought under the gated release:

1. **Ring shift (`libp2p-node-base.ts:1017-1027`)** — replace the unilateral
   `setArachnodeInfo(newRing)` flip with the three-phase state machine: set `status='moving'` and
   publish the target ring (Phase A, keep serving the old range); confirm every block in the shed
   range is on ≥ N post-move holders (Phase B); only then set `status='active'` at the new ring and
   release the shed range (Phase C). On any confirmation failure, roll back to `active` at the old
   ring. Move-**in** uses Phase A only (it sheds nothing) — pull the gained half via the existing
   `RestorationCoordinator` path.
2. **Rebalance release (`libp2p-node-base.ts:982-988`)** — today a `lost` block is `untrackBlock`ed
   synchronously while the push runs fire-and-forget. Gate the `untrackBlock` (release) on the push
   result: a block is released only after `BlockTransferCoordinator` confirms it landed on the new
   owners (`pushBlocks` reports it not `missing`) to the floor; a block whose push failed / was
   partition-skipped stays tracked and served.

The confirm primitive already exists: `BlockTransferClient.pushBlocks` returns `missing`. Surface a
per-block confirm result from `BlockTransferCoordinator` (do not release blocks whose confirmation
did not reach the floor). Post-move holders and `N` come from `RebalanceEvent.newOwners` /
`RebalanceMonitor.getCohortSize()`.

## Edge cases & interactions

- **Replication-floor invariant (the core test)** — assert that at no simulated instant during a
  move-out does the count of `active`-serving responsible holders for any shed key fall below `N`:
  before Phase B completes the mover is still counted; after Phase C ≥ N *others* are. Drive a
  fake-clock scenario through all three phases and check the count at each transition.
- **Concurrent adjacent moves (§ Part 3)** — a peer that is `status='moving'` and advertises a
  target partition shedding the *same* sub-range must **not** count toward a confirmer's floor for
  that sub-range. Test two adjacent movers over the same overlap: at most one confirms; the other
  fails to find `N` qualifying holders and rolls back (keeps the range). Neither releases the
  overlap.
- **Failure mid-handoff / crash between advertise and release** — the node never ran Phase C, so it
  never stopped serving and never GC'd the range. Simulate a restart with stale `status='moving'`
  metadata: the node re-derives responsibility, resumes serving if still responsible, and refreshes
  status to `active`. Peers must treat a `moving` node as still-responsible for its **old** range
  (fail-toward-old-holder) — verify a `moving` peer is still counted as a holder for its old range.
- **Partition during handoff** — Phase B must abort (not release) when `partitionDetector` reports a
  partition, consistent with the existing push guard. Test: partition mid-confirm → rollback, range
  retained.
- **Membership-agreement interaction (§ Part 3)** — during Phases A–B the affected blocks' cohort is
  a *superset* (old + new holders), a bounded ~one-member delta that stays inside the admission
  gate's `clusterSizeTolerance` (`cluster-repo.ts admitMembership`). Verify a shift does **not** trip
  the self-shrink floor or tolerance rejection at a peer's `deriveExpectedCluster` view; the
  single-step rule (from the damping ticket) is what bounds the delta.
- **Confirm-timeout / unreachable holders** — Phase B must bound its wait (per-block timeout +
  retry, reusing `BlockTransferCoordinator` semantics) and, if the floor is unmet, abort the whole
  shift rather than release partially. No block in the shed range is released unless *every* block in
  it confirmed.
- **Local bytes vs. tracking** — `untrackBlock` only stops spread; releasing must also gate any GC /
  storage-sweep eligibility on Phase C so the sweep (`st-storage-sweep-archival-and-capacity-estimate`)
  cannot reclaim an unconfirmed range. Coordinate the eligibility signal, don't just untrack.
