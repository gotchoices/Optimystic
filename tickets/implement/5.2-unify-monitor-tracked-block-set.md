# Unify the owned-block tracked set across SpreadOnChurnMonitor and RebalanceMonitor

## Background

After `rebalance-monitor-wiring-and-reaction` lands, **both** monitors run on a live node, and both
keep their **own** `private readonly trackedBlocks = new Set<string>()`
(`spread-on-churn.ts:86`, `rebalance-monitor.ts:42`), each fed by a separate
`storageRepo.onAnyCollectionChange` subscription. The two sets can drift: e.g. spread's
no-local-data self-prune (`spread-on-churn.ts:216-221`) shrinks only spread's set, and there is no
responsibility-loss eviction at all.

This ticket makes both monitors share **one** owned-block set, fed once, and drives eviction from
`RebalanceMonitor`'s `lost` signal — the authoritative "this node is no longer responsible" event —
so spread stops re-pushing blocks the node no longer owns instead of relying solely on its
lazy no-local-data prune.

## Design

### Shared owned-block set

Introduce a single mutable string set that both monitors reference instead of owning. Two viable
shapes — pick the lighter one that keeps the monitors' existing internal iteration working:

- **Option A (preferred): inject a `Set<string>`.** Add an optional `trackedBlocks?: Set<string>` to
  `SpreadOnChurnDeps` and `RebalanceMonitorDeps`. Each monitor uses the injected set when present,
  else constructs its own (preserves all existing unit tests and standalone construction). Replace
  the `private readonly trackedBlocks = new Set<string>()` field initializer with
  `this.trackedBlocks = deps.trackedBlocks ?? new Set<string>()`. `trackBlock` / `untrackBlock` /
  `getTrackedBlockCount` / the `for...of this.trackedBlocks` loops stay unchanged.
- **Option B: a tiny `OwnedBlockRegistry` class** (`add/delete/has/[Symbol.iterator]/size`) injected
  the same way. Only do this if the implement agent finds a reason the bare `Set` is insufficient
  (e.g. wanting change callbacks). The bare `Set` is expected to be enough.

`RebalanceMonitor.responsibilitySnapshot` (`rebalance-monitor.ts:43`) stays **per-monitor** — it is
not owned-block tracking, it is rebalance's own was-responsible memory. Only `trackedBlocks` is
shared. Note `untrackBlock` also deletes from `responsibilitySnapshot` (`rebalance-monitor.ts:104`);
keep that, but be aware that when the shared set is mutated by spread's self-prune the snapshot is
not pruned — acceptable (a stale snapshot entry for an absent block is re-derived; document it).

### Single feed + responsibility-loss eviction in `libp2p-node-base`

- Construct **one** `const ownedBlocks = new Set<string>()` before wiring the monitors, pass it into
  both `initSpreadOnChurnMonitor` and `initRebalanceMonitor` (thread a new `trackedBlocks` param
  through `NetworkManagerService.initSpreadOnChurnMonitor` /
  `initRebalanceMonitor` into the respective `Deps`).
- Collapse the **two** `storageRepo.onAnyCollectionChange` subscriptions (spread's at
  `libp2p-node-base.ts:698`, plus the one `rebalance-monitor-wiring-and-reaction` added) into
  **one** that adds to `ownedBlocks` — which both monitors now read. Keep a single unsubscribe in
  the stop wrapper.
- In the `rebalanceMonitor.onRebalance(...)` handler, drive eviction: for each `event.lost` blockId,
  `ownedBlocks.delete(blockId)` (and optionally `rebalanceMonitor.untrackBlock(blockId)` to also
  clear its snapshot); for each `event.gained` blockId, `ownedBlocks.add(blockId)` so a
  newly-responsible block is tracked even before its next commit/replica touches the feed. Do this
  **alongside** dispatching to `coordinator.handleRebalanceEvent(event)` (both can run; ordering does
  not matter for correctness — the coordinator reads `event`, not the set).

This makes responsibility loss the authoritative untrack and complements (does not replace) spread's
no-local-data self-prune.

### Concurrency / iteration safety

Both monitors iterate the shared set inside `async` loops with `await`s
(`spread-on-churn.ts:194`, `rebalance-monitor.ts:167`). Deleting a `Set` entry mid `for...of` is
safe in JS (the spread monitor already relies on this for its self-prune). With sharing, an eviction
from the rebalance handler can now mutate the set while spread is mid-iteration — still safe (no
throw; entries added/removed during iteration are visited best-effort). This is acceptable
best-effort behavior for a resilience mechanism; document it in a comment rather than adding locking.

### Initial-scan seed — deferred, not in this ticket

The original ticket asked to "consider" seeding the set at startup with blocks already durable from
a previous run. `IRawStorage` (`packages/db-p2p/src/storage/i-raw-storage.ts`) exposes **no**
enumerate-all-block-ids API, so a seed would require adding an enumeration method across every
backend (`MemoryRawStorage`, any file/leveldb backend) — out of scope here. Filed as
`optimystic-owned-block-initial-scan-seed` (backlog). Leave the existing "not re-tracked until next
touched" comment accurate.

## Acceptance

- One `Set<string>` instance backs both monitors on a live node (assertable: track via the feed,
  observe `getTrackedBlockCount()` agree across both monitors; evict via a `lost` event, observe
  both drop it).
- A `RebalanceEvent` with `lost: [b]` removes `b` from the shared set so a subsequent
  `SpreadOnChurnMonitor.checkNow()` no longer attempts to spread `b`.
- Existing standalone monitor unit tests still pass (injected set is optional; default-constructed
  set preserves current behavior).
