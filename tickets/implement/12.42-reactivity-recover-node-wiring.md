## Resolved & landed (this run ‚Äî the sync-signer design point)

The seam question was: the subscription-manager `signBackfill` / `signResume` ports are **synchronous**
(`(unsigned) => string`) ‚Äî the db-core `createBackfillRequester` driver builds the unsigned image
internally, so a pre-signed value is impossible ‚Äî but libp2p `PrivateKey.sign` is **async**.

**Decision: synchronous noble-ed25519 sign over the node's raw Ed25519 seed**, keeping the existing
synchronous `(unsigned) => string` seam unchanged. The async-seam alternative was rejected: it would force
`createBackfillRequester`'s `sign` async, a cross-package db-core change overlapping
`reactivity-recover-wire-signing` (12.4) for no benefit. The sync path needs **no** db-core change, mirrors
the existing synchronous `verifyPeerSig`, and round-trips with it (both `@noble/curves/ed25519`, RFC8032).

Landed:
- `signPeerSig(privateKey, payload): Uint8Array` in `packages/db-p2p/src/cohort-topic/peer-sig.ts` ‚Äî the
  synchronous mirror of `signPeer`. Reads the libp2p Ed25519 `raw` (64 bytes = 32-seed ‚Äñ 32-pub; confirmed
  reachable as `nodePrivateKey.raw`), signs with noble over the 32-byte seed. Throws on a non-Ed25519 key.
- `createRecoverRequestSigners(privateKey): { signBackfill, signResume }` in
  `packages/db-p2p/src/reactivity/recover-transport.ts` ‚Äî builds the manager's signer seam over db-core's
  `backfillSigningPayload` / `resumeSigningPayload`, base64url-encoded. This is the exact glue the production
  subscribe factory feeds the manager once 12.33's wiring exists.
- Tests: `packages/db-p2p/test/reactivity/recover-signers.spec.ts` ‚Äî sign/verify round-trip; **byte-identical
  to the async `signPeer`** (deterministic Ed25519 cross-impl check); non-Ed25519 key rejected; and an
  **end-to-end** check that signatures from `createRecoverRequestSigners` are verified + served by the real
  inbound `createRecoverRequestHandler` (and a wrong-key request is rejected with no reply).
- Doc: `docs/reactivity.md` ¬ßAuthentication ‚Äî the "Recover RPCs" bullet now documents the synchronous
  subscriber-side signing path and that wiring it into the running node rides 12.33's node wiring.

Full `@optimystic/db-p2p` suite green after the change (797 passing, 27 pending ‚Äî the env-gated integration
tests, incl. the recover socket test, skipped because `OPTIMYSTIC_INTEGRATION` is unset).

  `registerRecoverHandler(node, protocols.recover, { pushStateFor: fwd.pushStateFor.bind(fwd),

- **Feed the subscribe factory.** The production factory that builds a `ReactivitySubscriptionManager`

  plain pruned-on-access map).
- **Self-dial / co-located serve.** The transport already excludes `selfPeerId` from dial candidates

  in-process short-circuit from the local `PushState` is a zero-RT optimization, not a correctness need.
  Decide + document either way (mirror the notify transport's self-guard).

## Edge cases & interactions

- **Any cohort member can serve** (ring + checkpoint are gossiped); the sticky hint is the one-RT happy
  path, the cohort-walk fallback reaches any member.
- **Sticky-hint staleness**: invalidate only on confirmed staleness (a `tail_rotated` reply / rotation
  detection ‚Äî already handled by the manager), never on a transient dial failure (the transport's walk
  fallback handles that).
- **Edge node** subscribes + dials recover (never serves ‚Äî no `PushState`); a recover dial to an Edge node
  finds no `PushState` and gets no reply (harmless; Edge is never a recover target in a correct walk).
- **Teardown ordering**: the recover unhandle rides the existing `node.unhandle(reactivityProtocolList)` in
  the stop wrapper, before `host.stop()`.
- **Foreign/oversized/malformed recover frame**: already bounded + no-reply in the handler; confirm the
  node-level `maxBytes` matches the notify transport's.
