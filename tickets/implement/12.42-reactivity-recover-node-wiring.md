# Reactivity recover RPC — live node wiring + real-socket integration test

## Why this ticket exists

The reactivity recover **machinery** is built and unit-tested by `reactivity-recover-rpc-transport`
(`recover-transport.ts`: `Libp2pReactivityRecoverTransport`, `createRecoverRequestHandler` /
`registerRecoverHandler`, `createLibp2pRecoverDialer`; `forwarder-host.pushStateForCollection`; the
subscription-manager escalation + `subscriberCoord` plumbing; the db-core `recover.ts` wire foundation).
What is **not** done is composing it into the live libp2p node — because that composition sits inside the
reactivity block of `libp2p-node-base.ts` that `reactivity-notification-transport` (12.33) owns and which
**errored mid-flight** (the `cohortEnabled` block currently installs only the cohort-topic host + change
bridge; there is no `fwd` forwarder host, subscriber registry, notify transport, origination manager,
`resolveCohort`/`CohortPeerResolver`, or subscribe factory to compose with).

This ticket finishes the recover direction once 12.33's node wiring exists. It is intentionally small: the
transport + handler were structured as injectable units so this is a thin composition step, not new logic.

## What to wire (in `libp2p-node-base.ts`, the `if (cohortEnabled)` reactivity block — after 12.33's `fwd`)

Compose with 12.33's reactivity wiring (forwarder host `fwd`, subscriber registry, notify +
push-state-gossip handlers, the shared `cohortHintCache`, and the cohort gossip transport's
`CohortPeerResolver`/FRET assembly):

- **Construct the dialer + transport.**
  `const recover = new Libp2pReactivityRecoverTransport({ dialer: createLibp2pRecoverDialer(node,
  protocols.recover), selfPeerId: node.peerId.toString(), cohortHintCache, resolveCohort, maxBytes })`,
  where `resolveCohort(topicId)` reuses the cohort gossip transport's FRET `CohortPeerResolver` (resolve the
  recover topic's coord → dialable **peer-id strings**, matching the notify transport's dial-target space).
- **Register the inbound handler.**
  `registerRecoverHandler(node, protocols.recover, { pushStateFor: fwd.pushStateFor.bind(fwd),
  pushStateForCollection: fwd.pushStateForCollection.bind(fwd), replayGuard:
  createCorrelationReplayGuard(), maxBytes })` — one node-level replay guard shared across recover requests.
- **Feed the subscribe factory.** The production factory that builds a `ReactivitySubscriptionManager`
  supplies, in addition to its existing options:
  - `backfillTransport: recover.backfillTransport(topicId, collectionIdB64)` and
    `resumeTransport: recover.resumeTransport(topicId, collectionIdB64)`;
  - `signBackfill: (u) => bytesToB64url(await? signPeer(...))` — **note the seam is synchronous**
    (`signBackfill`/`signResume` return `string`, not `Promise<string>`); `signPeer` (libp2p
    `PrivateKey.sign`) is **async**. Resolve this: either pre-sign is impossible (the unsigned image is
    built inside the driver), so the manager's signer seam needs an **async** variant, OR a synchronous
    Ed25519 signing primitive over the raw private key (mirroring `verifyPeerSig`'s synchronous
    `@noble/curves/ed25519` verify) must be used. **Decide and implement** — this is the one real design
    point in this ticket. The synchronous-noble-sign path keeps the existing `(u) => string` seam; confirm
    the node's raw Ed25519 private key bytes are reachable for it (`nodePrivateKey`).
  - `subscriberCoord`: the subscriber's **real ring coordinate** (the `participantCoord` it registers under
    at the cohort-topic tier), not the collectionId placeholder;
  - the shared `cohortHintCache` (so the transport's sticky-primary lookup and the manager's
    rotation-invalidation see the same cache).
- **Protocol registration + teardown.** `node.handle` the recover protocol with the other reactivity
  protocols; the existing `node.stop` wrapper's `node.unhandle(reactivityProtocolList(protocols))` already
  covers it (recover is now in the list). No new timer (the replay guard is a plain pruned-on-access map).
- **Self-dial / co-located serve.** A subscriber co-located with a serving cohort member must not dial
  itself. The transport already excludes `selfPeerId` from dial candidates; decide whether a co-located
  resume should instead be served in-process directly from the local `PushState` (simpler + zero-RT) and
  document the choice (mirrors the notify transport's self-guard).

## Real-socket integration test (env-gated)

In `substrate-real-libp2p.integration.spec.ts` (gated the same way as 12.33's notification socket-delivery
test): a remote subscriber that slept past the live tail's last delivered revision sends one `ResumeV1`
over a real socket to a real tail-cohort member and is brought current (the backfill variant) — the
recovery analogue of 12.3's notification socket-delivery test. If the gate is unset, note it is gated and
rely on the in-process `recover-transport.spec.ts` coverage.

## Edge cases & interactions

- **Any cohort member can serve** (ring + checkpoint are gossiped); the sticky hint is the one-RT happy
  path, the cohort-walk fallback reaches any member.
- **Sticky-hint staleness**: invalidate only on confirmed staleness (a `tail_rotated` reply / rotation
  detection — already handled by the manager), never on a transient dial failure (the transport's walk
  fallback handles that).
- **Edge node** subscribes + dials recover (never serves — no `PushState`); a recover dial to an Edge node
  finds no `PushState` and gets no reply (harmless; Edge is never a recover target in a correct walk).
- **Teardown ordering**: the recover unhandle rides the existing `node.unhandle(reactivityProtocolList)` in
  the stop wrapper, before `host.stop()`.
- **Foreign/oversized/malformed recover frame**: already bounded + no-reply in the handler; confirm the
  node-level `maxBytes` matches the notify transport's.
