# Reactivity rotation — origination observe-rotation + node composition + e2e (db-p2p)

The capstone: make a running `cohortTopic`-enabled node detect a tail rotation, start the outgoing tail's
drain (`markRotated`), compose the recover-redirect serve and the re-registration scheduler, and prove an
end-to-end rotation over the mock-tier mesh (and note the real-libp2p tier). Builds on
`reactivity-rotation-recover-redirect-drain` (the `markRotated` seam + redirect serve/honor) and
`reactivity-rotation-rereg-scheduler` (the timer component).

## Background — the load-bearing constraint: the successor tail id is not knowable early

The reactivity topic anchors on the collection's tail block id (`reactivityTailBytes(event.tailId)`). Block
ids today are **random** (`TransactorSource.generateId() → randomBytes(32)`; deterministic derivation is the
blocked backlog item `6.5-block-id-derivation`). So at the **filling** commit the host **cannot** know the
successor tail id — which means the pre-announce `rotationHint{ newTailId }` (the optimization that gives
subscribers a jitter head start *before* the switch) cannot be built on a live node.

The authoritative, **observable** rotation signal on the host is therefore `event.tailId` **changing** between
commits. That is a *hard* rotation: notifications then ride the new tail, and active subscribers already
detect it via the manager's `detectRotation` (delivered `tailId` differs → `RotationNotice`, `preAnnounced:
false`). This ticket wires that observation; the pre-announce remains exercised in the mock-tier harness and
the design simulator (both can synthesize the successor id) and is documented as gated on `6.5`.

## What to build

### A. Origination observes the tail-id change → starts the drain

`ReactivityOriginationManager` (`origination-manager.ts`) + its node-base construction.

- Track the last-seen `event.tailId` per collection (keyed by `collectionId`). On a commit whose
  `reactivityTailBytes(event.tailId)` differs from the last-seen for that collection:
  - The **previous** tail's topic has rotated to the new one. Call `forwarderHost.markRotated(oldTopicId,
    { newTailId: bytesToB64url(reactivityTailBytes(event.tailId)), effectiveAtRevision: event.rev }, now)` so
    the old cohort's recover serve begins redirecting (the seam from
    `reactivity-rotation-recover-redirect-drain`).
  - Update the last-seen tail for the collection.
- The origination context's `rotationHint` stays `undefined` on a live node (gated on `6.5`); the manager
  still passes `ctx.rotationHint` through unchanged, so a future deterministic-id world only flips the source.
  Add the `markRotated` callback as a new `ReactivityOriginationManagerOptions` field (the node binds it;
  absent ⇒ origination is unchanged, preserving existing tests).
- **`BlockFillTracker` for warm-up (best-effort).** Construct a per-collection `BlockFillTracker` and feed it
  `onCommit()` per commit; on a `warmup` signal, **log/no-op** the anticipatory pre-dial bias (the next
  `tailId` is not knowable, so this is signal-only per `docs/reactivity.md` §Anticipatory warm-up — confirm
  acceptable and document; do **not** fabricate a successor coord). The `filling` signal is **not** used to
  emit a hint on a live node (no successor id); it may be logged.

### B. Compose the rotation pieces in `libp2p-node-base.ts`

In the `if (cohortEnabled)` block (currently `origination-manager.ts` wiring sets `rotationHint` =
"reactivity-tail-rotation-transport's (12.5)"):

- Bind `origination`'s `markRotated` to `forwarderHost.markRotated`.
- Bind the recover serve handler's new `rotationFor` (from `reactivity-rotation-recover-redirect-drain`) to
  `forwarderHost.rotationRedirectFor` (resolve resume topic from `latestKnownTailId`; backfill from the
  collection's current served topic).
- Construct a `RotationReRegistrationScheduler` with an unref'd `setTimeout`-backed `setTimer`. Expose it on
  the node (`(node as any).reactivityRotation = scheduler`) alongside `reactivitySubscribers` /
  `reactivityRecover`. Its `reRegister(plan)` seam is wired by the **subscribe factory** that constructs
  managers (the deferred Quereus `Database.watch` bridge, `optimystic-network-reactive-watch-integration-test`):
  on fire it builds a fresh `ReactivitySubscriptionManager` under `plan.newTopicId` carrying `plan.lastRevision`,
  registers it, and swaps the `ReactivitySubscriberRegistry` entry (unregister old topic handler → register
  new). Until that factory lands, the scheduler is constructed + exposed + unit/mesh-tested but not driven by a
  node-internal manager — exactly as 12.33 exposed `reactivitySubscribers`/`reactivityRecover` without a live
  manager constructor. Document this honestly.
- **Load-bearing:** wherever a manager's `rejoinJitter` is constructed in the (future) subscribe factory it
  MUST be `createRejoinJitter({ capPromote: DEFAULT_CAP_PROMOTE_FAST })` — leave a prominent comment at the
  exposure site so the watch-bridge ticket honors it.
- Teardown: `scheduler.stop()` in the node `stop` wrapper, before `host.stop()`.

### C. End-to-end mock-tier proof

Extend the mesh harness + `mesh-tail-rotation.spec.ts` to drive the rotation **through the transport seams**,
not just the db-core logic:

- Harness: after `rotateTail`, call `forwarderHost.markRotated` (or model it) so a subscriber that resumes
  against the **old** tail gets the `kind:"rotated"` redirect → fires `onRotation` → the harness's scheduler
  fires `reRegister` → the subscriber re-attaches under the new tail and continues with **no gap**.
- Drive the `RotationReRegistrationScheduler` over the harness virtual clock (`advanceTime`) so the timer fires
  deterministically.
- Assert the cross-rotation **resume served from the inherited checkpoint**: a subscriber whose `fromRevision`
  is below the new tail's young ring but covered by the inherited checkpoint gets `checkpoint_window` (not
  `out_of_window`) and ends up current.

### D. Real-libp2p tier (note, do not block)

Name the real-libp2p assertion in `substrate-real-libp2p.integration.spec.ts` (the
`substrate-e2e-real-libp2p-tier`): after a tail-id change, a recover dialed at the old cohort returns a
`kind:"rotated"` frame over the real socket. If standing up a real two-tail rotation on real libp2p exceeds an
agent-runnable budget, add a `[unimplemented:real-tier]`-tagged pending test + a one-line deferral note
(mirror the existing real-tier pending tests) rather than a flaky long-running test.

## Edge cases & interactions

- **First commit of a collection** (no last-seen tail) → no rotation; just records the baseline tail.
- **tail-less commit** (`event.tailId === undefined`, read-driven promotion) → never originates and never
  rotates (the existing early-return holds); do not record/clear the last-seen tail on it.
- **Tail id encoding** — the observe-rotation comparison MUST use `reactivityTailBytes(event.tailId)` (utf8),
  the same encoding origination/membership-gate use, NOT `blockIdToBytes` (double-hash). A mismatch silently
  never detects rotation (green tests, dead feature) — pin it with a test asserting the `markRotated`
  `oldTopicId` equals the topic the subscriber subscribed under.
- **markRotated for a topic this node doesn't serve** (Edge / non-member) → `rotationRedirectFor` returns
  `undefined` (no gate without a served `PushState`); harmless.
- **Concurrent rotations across collections** → per-collection last-seen map; one collection's rotation never
  disturbs another's.
- **Scheduler timer outliving the node** → unref'd + `stop()` on teardown; a `node-wiring.spec.ts` teardown
  assertion must show no pending timer pins the process (mirror the gossip-driver teardown assertion).
- **Re-register swap race** — the subscribe factory's `reRegister` must register the new-topic handler in
  `ReactivitySubscriberRegistry` *before* unregistering the old, so a notification mid-swap is never dropped;
  spell this ordering out at the exposure site for the watch-bridge ticket (and assert it in the mesh e2e where
  the harness plays the factory role).
- **Continuity invariant** — delivered revisions stay monotone + gap-free across the boundary (revisions
  N..rotationRevision on the old tail, rotationRevision+1.. on the new), even though `tailId` differs across
  the seam (the existing `mesh-tail-rotation` continuity test, now driven through the redirect+scheduler).
