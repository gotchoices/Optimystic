# Crash-failover backup promotion ↔ deterministic slot assignment

## Background (resolved design)

The TTL-failover participant path and the cohort-side `onRenew` were unit-tested independently
and are incoherent when composed: on a *crash* (no membership rotation, so `cohortEpoch` is
unchanged) `assignSlots` still names the dead member as primary, so a re-attach to `backups[0]`
is answered `primary_moved { newPrimary: <dead primary> }` and the participant bounces between the
promoted backup and the corpse until FRET rotation eventually changes the epoch. See the source
plan ticket for the full failure trace.

The three open questions are resolved as follows; **do not re-open them**, implement them:

### Q1 — how a backup recognizes a legitimate promotion → signed re-attach discriminator

Add an optional, **signed** `reattach?: boolean` to `RenewV1` (absent/false = a normal ping;
`true` = "I could not reach my primary, promote you"). A plain ping must **never** trigger
promotion — that would let a stray/misrouted renew silently usurp a live primary. Only an explicit,
participant-attested re-attach may. Because `reattach` is part of the signed body, a member can
trust the attestation and a MITM cannot flip a ping into a promotion (or back).

The cohort accepts a re-attach when **both**: (i) it holds the record in its local replica
(`store.getByParticipant` ≠ undefined — the doc's "verifies it sees the record in its local
replica and confirms"; cross-member replication from `cohort-topic-gossip-membership-certs` makes
this the common case), **and** (ii) `self` is a computed backup for the record under the current
epoch (`self ∈ assignSlots(participantId, epoch, members).backups`) — or already the computed
primary, or already dual-serving via the rotation `isServing` hook.

### Q2 — what stops two primaries → participant-directed primary + epoch-scoped override, no new primitive

The single-primary invariant for *renewal* holds because a participant pings exactly one member
(its `record.primary`). The promotion is recorded as an **epoch-scoped failover override** on the
accepting member (mirroring the handoff's `dualServing`/`isServing` exception) so its *subsequent
plain pings* keep being served instead of bouncing. A revived dead primary receives no pings from
the migrated participant and goes stale on TTL. The override is **cleared on epoch change**, so the
next stabilization's deterministic assignment + rotation handoff reasserts authority and collapses
any ambiguity. The brief *delivery*-level dual-serve window (a revived primary plus the promoted
backup both pushing) is the same bounded window the doc already tolerates for the rotation handoff;
application-level delivery dedup owns it (§Application policies). **No new threshold/tie-break
mechanism is introduced** — consistent with the layer's soft-state philosophy.

### Q3 — failover() honors primary_moved

A legitimate crash-failover re-attach now returns `ok`, so `failover()` promoting on `ok` is
correct. A re-attach that returns `primary_moved` means a *real* rotation moved primary to a
different live member (the dead node has left the cohort, so the rotated `assignSlots` names a live
target); `failover()` must **adopt that payload** (like `applyPrimaryMoved` — primary/backups/epoch)
instead of blindly promoting the backup it contacted. Guard: if `primary_moved.newPrimary` equals
the just-failed primary, **ignore it** and continue to the next backup / relookup (defensive against
the exact bounce the plan describes).

## Why an explicit override and not "serve when stored record.primary == self"

Tempting shortcut, rejected: making `onRenew` serve whenever the stored `record.primary == self`
regresses the rotation handoff. After a rotation+ack the *old* primary's local replica still has
`primary == old-self` (nothing re-stamps it; `onAck` only clears `dualServing`), so it would serve
forever and never redirect the participant to the new primary. The deterministic computed-primary
stays the default serve condition; crash-failover is layered on as a **new, separate, epoch-scoped
exception**, exactly as dual-serve was. The re-stamp of `record.primary = self` is still done (it is
what makes the next rotation's `handoff.start()` recognize the promoted member as the holder, and
what gossip converges other replicas onto), but it does **not** drive `onRenew` serving — the
override does.

## Interfaces / shapes

```ts

export interface RenewV1 {
  v: 1;
  topicId: string;
  participantId: string;
```

```ts
private buildRenew(reattach: boolean): RenewV1 { /* include reattach in the SIGNED UnsignedRenew body */ }

// renewal.ts — cohort side: an internal epoch-scoped override, sibling to the existing isServing dep.
//   key = recordKey(topicId, participantId) → cohortEpoch under which the takeover was accepted.
private readonly failoverServing = new Map<string, Uint8Array>();
```

### onRenew control flow (replaces the current body)

```
rec = store.getByParticipant(topicId, participantId)
if rec === undefined            → return unknown_registration
{ primary, backups } = slots.assignSlots(participantId, cohortEpoch, members)

if msg.reattach === true:                               // crash-failover promotion request
  if self == primary                                   → touchAndServe(); return ok   // already the computed primary
  if self ∈ backups:                                   // legitimate backup takeover
     store.put({ ...rec, primary: self, backups: rec.backups.filter(≠ self), lastPing: now })
     failoverServing.set(key, cohortEpoch)
     gossip.touch(restamped); return ok
  if isServing?(topicId, participantId) === true        → touchAndServe(); return ok   // rotation dual-serve covers it
  → return primary_moved { computed primary/backups/epoch }   // not a valid takeover target → redirect

else:                                                   // plain ping
  serving = self == primary
         || isServing?(topicId, participantId) === true
         || (failoverServing.get(key) matches current cohortEpoch)
  if serving                                            → touchAndServe(); return ok
  // housekeeping: if self == primary, drop any now-redundant failoverServing[key];
  //               if failoverServing[key] epoch ≠ current cohortEpoch, delete it (stale across rotation)
  → return primary_moved { computed primary/backups/epoch }
```

`touchAndServe()` = `store.put({ ...rec, lastPing: now }); gossip.touch(touched)` — the existing
touch path, factored out so the reattach and plain paths share it.

### failover() control flow (participant)

```
for backup in current.backups:
  reply = tryReattach(backup)                 // sends RenewV1 with reattach=true
  if reply == undefined: continue
  switch reply.result:
    'ok':            promote(backup); return
    'primary_moved': if newPrimary != current.primary: applyPrimaryMoved(reply); return
                     else: continue           // guard: never adopt the just-failed primary
    'unknown_registration' | default: continue
relookup(); reset consecutiveFailures
```

`promote()` and `applyPrimaryMoved()` already exist; reuse them. `applyPrimaryMoved` already does
the lazy `cohortEpoch` refresh, so honoring `primary_moved` here keeps the resolved lazy-epoch
semantics intact.

## Edge cases & interactions

- **Subsequent-ping stability (the core fix).** After promotion, `participant.record.primary = B`;
  its next `pingLoop()` sends a *plain* renew to B. B must serve via the `failoverServing` override
  (computed primary is still the dead node) and return `ok`, **not** `primary_moved`. Test that the
  participant's `record.primary` stays on B across several subsequent pings (no bounce).
- **Replication lag.** A re-attach to a backup that has **not yet** received the record via gossip
  → `unknown_registration`; `failover()` moves to the next backup, else `relookup()`. Don't promote
  on an unconfirmed record. (Rare once the prereq lands; still must be handled.)
- **Cascading failover.** Primary `s` dies → promote `backups[0]=s+1`; if `s+1` also dies →
  re-attach `backups[1]=s+2`. Under the unchanged epoch, computed backups are still `[s+1, s+2]`, so
  `s+2 ∈ computed backups` → accept. After both backups exhausted → `relookup` (bounded, matches the
  doc's "all of primary and backups fail").
- **Override cleared on rotation.** A `failoverServing` entry tagged with epoch-1 must be ignored
  once `cohort().cohortEpoch` advances to epoch-2; the rotation handoff then governs (either `self`
  is the new computed primary and serves anyway, or the record is pulled away). Test: set override
  under epoch-1, advance epoch, confirm a plain ping to the ex-backup now returns `primary_moved`
  (or that it is the new computed primary and still serves) — never serves on a stale override.
- **Re-attach to a non-backup / wrong member.** `reattach=true` but `self ∉ backups` and `self !=
  primary` and not dual-serving → `primary_moved` (redirect, do not promote). Guards against a
  participant with a stale `backups` view forcing an arbitrary member to take over.
- **Plain ping never promotes.** A plain renew (`reattach` absent/false) landing on a backup that
  holds the record must **not** promote — it returns `primary_moved` (the existing redirect). Assert
  no override entry is created on a plain ping.
- **`primary_moved` points at the dead primary.** `failover()` must drop such a reply and continue
  rather than adopting it (the defensive guard); otherwise the bounce returns.
- **Signature/codec.** `reattach` is inside the signed `UnsignedRenew` body (`buildRenew` includes
  it before signing) and round-trips through `encode/decodeRenewV1`. `validateRenewV1` accepts it via
  `optBool` (absent → field stays off the decoded object, per `assignDefined`); a non-boolean
  `reattach` is rejected with `CohortWireError`. Backward-compatible: an old encoder omits it →
  decodes as a normal ping.
- **Interaction with rotation dual-serve.** The `isServing?` hook is independent of
  `failoverServing`; both are OR-ed into the serve decision. A record can be dual-served (rotation)
  and never enters `failoverServing`, and vice-versa. Ensure the two exceptions don't clobber each
  other (distinct state; the reattach path checks `isServing` only after the backup-accept branch).
- **Gossip convergence.** The re-stamped record (`primary = B`) is gossiped via the existing
  `gossip.touch(rec)`; other replicas converge so a later `handoff.start()` on B recognizes B as
  holder and a further failover finds a sensible record. (The override *set* itself is not gossiped —
  only B is pinged.)

## Key tests (extend `test/cohort-topic/registration.spec.ts` and `wire.spec.ts`)

Participant side:
- Re-attach sends `RenewV1` with `reattach === true`; a normal `pingLoop` sends `reattach` falsy.
  Assert the flag is present in the signed body the transport receives.
- Crash failover: primary fails 3× → re-attach to `backups[0]` returns `ok` → `record.primary == backups[0]`,
  and **subsequent plain pings to it keep returning `ok` and leave `record.primary` on the backup**
  (the anti-bounce regression test — this is the bug).
- `failover()` honors `primary_moved`: re-attach reply `primary_moved { newPrimary: live-member, cohortEpoch }`
  → participant adopts `live-member` and refreshes the epoch hint (not blind-promote of the contacted backup).
- Guard: re-attach reply `primary_moved { newPrimary: <the just-failed primary> }` → participant does
  **not** adopt it; falls through to the next backup / relookup.
- `unknown_registration` on re-attach → next backup tried, else relookup.

Cohort side:
- `onRenew` with `reattach=true` where `self ∈ computed backups` and the record is held → `ok`,
  `lastPing` touched, gossip fired, `record.primary` re-stamped to `self`, and a *subsequent plain*
  `onRenew` from the same participant returns `ok` (override serves it under the unchanged epoch).
- `onRenew` plain ping on a backup that holds the record (no prior re-attach) → `primary_moved`
  (no promotion, no override created).
- `onRenew` reattach where `self ∉ backups` and `self != primary` → `primary_moved` (redirect).
- Override is epoch-scoped: after accepting under epoch-1, advance `cohort()` to epoch-2 → a plain
  `onRenew` no longer serves on the stale override (returns `primary_moved`, or serves only if `self`
  is the new computed primary).
- `reattach=true` where `self == computed primary` (rotation already moved primary to this member) →
  `ok` without creating a redundant override.

Wire:
- `wire.spec.ts`: `RenewV1` with `reattach: true` round-trips through `encode/decodeRenewV1`; absent
  `reattach` decodes to a `RenewV1` without the field; a non-boolean `reattach` throws `CohortWireError`.
