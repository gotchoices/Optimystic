## Cohort-side capability filter

Per [matchmaking.md §Capability filter L136-145](../../docs/matchmaking.md): evaluation is **local to the cohort** and **advisory only** — it biases which providers the cohort returns but is not admission; the seeker re-validates the returned set.

```ts
function matchesFilter(p: ProviderEntryV1, f: CapabilityFilter | undefined): boolean
// must:     every tag in f.must present in p.capabilities
// mustNot:  no tag in f.mustNot present in p.capabilities
// minBudget: p.capacityBudget >= f.minBudget (when set)
```

The cohort returns its locally-known direct provider registrations matching `filter`, up to `limit` (≤ `query_limit_max` 256). Set `truncated=true` when more matched than `limit`. A pathological filter that matches almost nothing is acceptable (the seeker re-validates and escalates) — see edge case 4 below.

### Seeker-side `registrationSig` re-validation — MUST resolve the signing-scope tension first

The advisory trust model (matchmaking.md §Wire formats L582) hinges on the seeker re-validating each
returned entry's `registrationSig`. **This is currently unbuildable as specified** and was flagged by
the foundation ticket's review — do not skip it:

`providerSigningPayload`/`seekerSigningPayload` (in `db-core/src/matchmaking/wire.ts`) bind the
provider/seeker registration signature over a tuple that **includes `correlationId`** (provider scope:
`(topicId, capabilities, capacityBudget, correlationId)`; seeker scope mirrors it). But the forwarded
`ProviderEntryV1` / `SeekerEntryV1` carry **no `correlationId`** field, so a seeker **cannot reconstruct
the exact signed image** to verify the forwarded `registrationSig`. (Note also that the matchmaking
`correlationId` is unrelated to the cohort-topic `RegisterV1.correlationId`, which the substrate
generates per-probe internally.) Pick one and apply it across `wire.ts` codecs, the cohort query
handler that builds the entries, the seeker verifier, **and** `docs/matchmaking.md` §Wire formats:

- **(a)** add `correlationId: string` (base64url) to `ProviderEntryV1` and `SeekerEntryV1`, forwarded
  verbatim from the registration — or
- **(b)** drop `correlationId` from `providerSigningPayload`/`seekerSigningPayload` so the signed image
  is fully reconstructable from the forwarded entry fields alone.

Recommendation: (b) unless `correlationId` is doing real replay-binding work that the entry needs — it
keeps the signed image self-contained in the forwarded entry. Whichever is chosen, the seeker verifier
(`db-core/src/matchmaking` pure `verifyProviderEntry`/`verifySeekerEntry`) and a round-trip test
(sign on the registration path → forward → verify on the seeker path) land in this ticket.

## Seeker hang-out-vs-continue engine

Per [matchmaking.md §Hang-out vs. continue L198-297](../../docs/matchmaking.md). After receiving `Accepted` with `topicTraffic` at tier `d`:

1. **Immediate-match check.** Issue `QueryV1`. If matching providers ≥ `wantCount`, done.
2. **Hang-out feasibility.**
   ```
   expectedNewMatches ≈ arrivalsPerMin × filterAcceptRatio × (patienceMs / 60000)
   contentionFactor   ≈ min(1 + (queriesPerMin × meanWantCount) / max(arrivalsPerMin, 1), contention_factor_cap)
   hang out  iff  currentMatches + expectedNewMatches ≥ wantCount × contentionFactor
   ```
   On hang-out: keep the seeker registration alive via TTL renewals; re-query every `requery_interval_ms` (default 1 s) until `wantCount` met or `patienceMs` elapses.
   `filterAcceptRatio` starts at `filter_accept_ratio_initial` (1.0) and is refined from observed query yields across the walk; `meanWantCount` is a small constant (or learned).
3. **Otherwise walk toward root.** Withdraw the seeker registration (`RenewV1` TTL=0; polite but optional — correctness does not depend on it, see foundation ticket), re-register at `d − 1`, repeat. The patience budget keeps draining across hops, so each escalation makes hang-out slightly less attractive (correct — root is terminal). At `d = 0` there is nowhere to walk: hang out for the remaining patience, then return whatever matched (possibly < `wantCount`).

Decision state types:

```ts
interface SeekerDecisionInputs {
  currentMatches: number          // from the immediate QueryV1
  directParticipants: number
  arrivalsPerMin: number
  queriesPerMin: number
  childCohortCount: number
  wantCount: number
  patienceMsRemaining: number
  filterAcceptRatio: number       // running, decays from 1.0
  meanWantCount: number
}
type SeekerDecision = { action: "done" } | { action: "hangOut"; requeryIntervalMs: number } | { action: "escalate" }
function decide(in: SeekerDecisionInputs, cfg: HangOutConfig): SeekerDecision
```

`db-core/src/matchmaking/seeker-walk.ts` holds the pure decision (unit-testable from the doc's worked examples). `db-p2p/src/matchmaking/seeker-walk-client.ts` wires `QueryV1`/`RenewV1` over cohort-topic and drives the requery timer.

## Edge cases (must be explicit — [matchmaking.md L268-293](../../docs/matchmaking.md))

1. **`topicTraffic` absent on reply** → treat cohort as zero-rate, walk one tier toward root without hanging out (no estimation against absent inputs).
2. **`arrivalsPerMin = 0` right after a cohort epoch change** → do **not** withdraw on a single zero reading; issue one `QueryV1` first — if it returns ≥ `wantCount`, the cohort was quiet not empty; escalate only when the query also yields below threshold.
3. **`UnwillingCohort` before `topicTraffic`** → hang-out not entered (never received `Accepted`); standard substrate back-off applies.
4. **Filter matches almost nothing** → `filterAcceptRatio` decays toward zero, `expectedNewMatches` collapses, threshold fails at every tier, seeker walks to root (correct — root's aggregated pool gives the filter its best chance).
5. **Many seekers competing** → high `queriesPerMin` raises `contentionFactor`, pushing more seekers toward the root; capped at `contention_factor_cap = 4.0` so a runaway query rate cannot pin every seeker to the root. Self-balancing.
