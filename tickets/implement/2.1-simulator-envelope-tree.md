# Cohort-topic *tree* operating-envelope boundaries

Three boundaries for the cohort-topic tree, each an axis + a predicate over an existing claim, driven
by the `findBoundary` harness from `simulator-envelope-core`. These are the heavier full-tree axes
(deliberately split out of core), gated by N as `sweep.ts` gates its full-tree walk-hop measurement.
The R* / `root-not-overloaded` axis already lives in the core ticket — this ticket adds the remaining
three tree rows of the parent ticket's candidate table.

This is new **measurement** over the existing tree model — reuse `TopicTree`, `ParticipantWalk`,
`runConvergence`, `expectedDepth`, the willingness vocabulary. Do not re-model tree behaviour.

## Boundary 1 — depth-law vs peer-ID prefix skew

`uniformLadder` shards perfectly evenly (`index mod F^d`), so observed depth tracks
`⌈log_F(N/cap_promote)⌉` exactly. Real sha256 addressing is only approximately uniform; a hot shard
(non-uniform prefix distribution) concentrates participants into fewer buckets, deepening the tree
past the law for the same N.

- **axis** `prefixSkew ∈ [0, 1]` — 0 = the existing uniform ladder; 1 = maximally concentrated (all
  participants funnel into one tier-`d` bucket). Add a `skewedLadder(index, dMax, F, skew)` generator
  beside `uniformLadder`: at skew `s`, a participant lands in its uniform bucket with probability
  `1−s` and the hot bucket with probability `s` (seeded, deterministic per index). Skew is
  monotone-in-harm (more concentration ⇒ deeper tree).
- **claim** `depth-law` — `holds(s)` grows a full tree at fixed N via `register(skewedLadder(...))`
  and returns `tree.maxOccupiedTier(topic) === expectedDepth(N,F,cap)`.
- **designAssumption** `prefixSkew = 0` (the design assumes ~uniform sha256 sharding). Output: the
  skew `s*` at which observed depth first exceeds the law, `margin = s* − 0`.
- **Gate by N** (`walkSampleMaxN`-style): grows a full tree, so default to a modest N (e.g. 10k) and
  make the large-N point opt-in; record the skip when gated.

## Boundary 2 — promotion/demotion stability vs churn (T_demote hysteresis)

At steady state the tree should be flat: `oscillations === 0` in the `PromotionTracer` readout. Under
arrival↔departure churn, if the churn rate outruns the `cap_promote`/`cap_demote` + `T_demote`
hysteresis, a cohort promotes and demotes repeatedly (flapping) instead of settling.

- **axis** `churnRate` — departures+re-arrivals per gossip round as a fraction of the cohort
  population (monotone-in-harm). Drive it with the existing `attach`/`detach` + per-round
  `evaluatePromotion`/`evaluateDemotion` (both already on `TopicTree`; `detach` is the existing
  departure path) around a cohort held near the `cap_promote` boundary so the hysteresis is the thing
  under test.
- **claim** `promotion-demotion-stable` — `holds(r)` runs the churn loop and returns
  `oscillations === 0` (reuse `PromotionTracer.result(...).oscillations`, which already counts
  observed-depth decreases).
- **designAssumption** the nominal churn the gossip cadence tolerates — define as `0` (a settled tree)
  and report `r* = churnRate` at first flap as the margin; document that `T_demote` is what buys the
  margin (the parent ticket's "each default justified by the envelope margin it buys").

## Boundary 3 — walk `no-give-ups` / hop bound vs unwilling fraction

A landing walk should attach without giving up and within the hop bound. As more members reply
`UnwillingMember`/`UnwillingCohort`, walks retry/escalate; past some unwilling fraction they give up
or the hop count blows past the bound.

- **axis** `unwillingFraction ∈ [0, 1]` — the fraction of members replying unwilling (monotone-in-harm).
  Inject via the willingness vocabulary the walk already understands.
- **claim** `no-give-ups` (and the `≤ d_max + 2` hop bound) — `holds(f)` runs a population of
  `ParticipantWalk`s over a grown tree with that unwilling fraction and returns
  `gaveUp === 0 && maxHops ≤ d_max + 2`.
- **designAssumption** `unwillingFraction = 0`. Output: `f*` at which walks first give up or breach the
  hop bound, `margin = f* − 0`.
- **Gate by N** like Boundary 1 (full-tree growth + walks).

## Edge cases & interactions

- **Determinism under gating.** Both full-tree axes must be N-gated and deterministic from `(seed,
  config, axisValue)`; the gated/large-N point must be opt-in and the skip recorded (mirror
  `sweep.ts` `walkMeasured`/`walkHopsSkipped`).
- **Skew degeneracy.** `skewedLadder` at `skew = 1` must still produce valid nesting coords (the hot
  bucket at tier `d` must refine its tier-`d−1` ancestor) — a fully-concentrated ladder must not
  corrupt the tree or throw; if perfect single-bucket concentration cannot nest, cap the skew model at
  the deepest still-valid concentration and document it.
- **Monotonicity assumption.** Each axis is asserted monotone-in-harm; if `findBoundary` reports
  `monotoneViolated` (e.g. skew that *shallows* the tree by emptying deep buckets, or churn that
  happens to settle at a higher rate), surface it rather than trusting the edge — the boundary spec
  must be tuned so harm is actually monotone, or the violation documented.
- **Flap detection window.** Boundary 2 must run enough rounds past peak churn for a flap to register
  in `oscillations`; too short a window yields a false `holds`. Pin the window and assert a known
  high-churn rate *does* flap (a positive control) so the boundary isn't vacuously "stable."
- **Hysteresis sticky floor.** `evaluateDemotion` enforces `T_promote_sticky` and `childCohortCount
  == 0`; the churn axis must exercise a cohort where demotion is actually reachable, else
  `oscillations` is structurally 0 and the boundary is vacuous (compare the `heal-convergence`
  structural-guarantee caveat in `simulator-strengthen-scenario-adversariality`).
- **Walk give-up vs hop-bound which-fails-first.** Boundary 3's predicate is a conjunction; record
  *which* sub-condition flipped (give-up vs hop-bound breach) so the readout is diagnosable.
- **Cross-ticket overlap.** This shares the tree subsystem with `simulator-envelope-core` (R* axis) and
  cross-refs `simulator-strengthen-scenario-adversariality` (which hardens these same point-claims).
  Design as if that ticket lands; do not block on it.
