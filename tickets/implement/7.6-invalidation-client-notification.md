# Surface invalidation to clients (push + pull)

Builds on `7.5-invalidation-log-entry-and-reversal`. Once an invalidation is a durable, committed log
entry, clients that previously received `committed` for that transaction must be able to learn it was
reversed. Two paths, mirroring the existing commit model: a **push** hint for connected subscribers and
an **authoritative pull** on the next status query.

## Background (resolved design)

### Push ‚Äî reuse the reactivity channel

An InvalidationEntry is a committed collection change, so it already flows through the existing
notification path: the commit fires `IBlockChangeNotifier.onCollectionChange`
(`packages/db-core/src/transactor/change-notifier.ts`) ‚Üí `onLocalCommit(event, commitCert)` ‚Üí
`ReactivityOriginationManager.originate` (`packages/db-p2p/src/reactivity/origination-manager.ts`) ‚Üí
`buildNotificationV1` ‚Üí emit. The invalidation's consensus produces its own **commit cert**
(`packages/db-p2p/src/cluster/commit-cert.ts`), reused bit-for-bit as the notification signature ‚Äî no
re-signing (the "notifications are hints; the cert is authoritative" model, `docs/transactions.md`
Design Decision 12). A subscriber wakes, sees the affected collection advanced to a new revision, and
reads the authoritative reverted state.

The notification must let a subscriber distinguish an **invalidation** from an ordinary commit so it can
react (drop derived results) rather than just refresh. Add a minimal typed marker to the change event /
notification (e.g. an `invalidation` flag plus the `invalidatedActionId`); keep it a hint ‚Äî the
subscriber still verifies against committed state.

### Pull ‚Äî authoritative durable status

The transaction status query must return `committed-invalidated` from **durable** state, not the
in-memory `DisputeService` map (`getDisputeStatus`, lost on restart and per-node). After the core
ticket, invalidation lives in the log; surface it through the status path
(`NetworkTransactor.getStatus`, `packages/db-core/src/transactor/network-transactor.ts`, whose current
`committed`/`pending`/`aborted` logic explicitly does **not** consult historical commits). Extend it to
report `committed-invalidated` when the queried action has an InvalidationEntry against it. The
in-memory dispute map may stay as a fast cache, but the log is the source of truth.

### Client reaction

A client holding a result derived from now-invalidated state must treat it like a stale read: on
learning of the invalidation (push or pull), drop the local result and resubmit through the existing
optimistic-concurrency loop (`Collection.updateAndSync`, `packages/db-core/src/collections`). This needs
no new protocol ‚Äî invalidation collapses into the same "your base moved, re-sync and retry" path the
client already runs on stale-read rejection.

## Edge cases & interactions

- **Disconnected client.** Misses the push entirely ‚Äî must learn on next pull. Assert a client that was
  offline during the invalidation gets `committed-invalidated` on its next `getStatus`.
- **Push without trust.** A subscriber that distrusts/misses the notification loses nothing: it can
  always read the authoritative committed (reverted) state directly. Notifications accelerate, never
  gate ‚Äî preserve this; never make correctness depend on push delivery.
- **Forged/dropped notification.** A compromised forwarder can drop or delay but cannot forge an
  invalidation notification (it would need the invalidation's `k‚àíx` threshold signature it does not
  hold). Verify the reused cert on the subscriber exactly as for commit notifications.
- **Cascade notifications.** Each child invalidation from `7.6-invalidation-cascade-detection` is its
  own committed entry and fires its own notification ‚Äî a client with several dependent transactions may
  receive several. Dedup/coalesce on the client by `invalidatedActionId`; do not assume one
  notification per dispute.
- **Notification ordering vs revision.** A subscriber may receive the invalidation notification before
  it has synced the intervening revisions. Reuse the existing reactivity backfill/resume-checkpoint
  handling ‚Äî the invalidation is just another revision in the tail.
- **Status of an in-flight (pending) transaction whose base got invalidated.** It will fail validation
  (stale read) on its own; ensure the status query distinguishes "pending ‚Üí will-be-rejected" from
  "committed-invalidated" and does not mislabel.
- **Idempotent client resubmit.** Resubmission must not double-apply if the original somehow still
  partially landed ‚Äî rely on transaction content-addressing / `StampId()` dedup so a resubmit is a
  distinct, safe transaction.

## Key tests

- Connected subscriber receives an invalidation notification carrying the reused invalidation commit
  cert; signature verifies against the same cohort membership cert as a commit notification.
- Offline-then-online client gets `committed-invalidated` from `getStatus` (durable, survives a node
  restart of the queried peer).
- Notification marks invalidation distinctly from a plain commit; a generic subscriber still refreshes
  correctly, an invalidation-aware client drops + resubmits.
- `updateAndSync` resubmits a result built on invalidated state and succeeds against the reverted base.
- Forged invalidation notification (bad threshold sig) is rejected by the subscriber.
