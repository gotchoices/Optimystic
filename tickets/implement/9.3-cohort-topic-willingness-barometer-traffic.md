# Willingness/admission control, capacity barometer, and topic-traffic signal

This ticket implements the admission/load brain of the substrate, folding three coupled
signals: per-member **willingness** (the admit decision), the **capacity barometer** (the
load buckets that flip willingness and trigger fast promotion), and the **topic-traffic**
counters (the advisory flow signal returned to applications). They are coupled because the
barometer's bucket crossings flip willingness bits, and the traffic signal is computed
from the same per-topic counters the barometer summarizes — all three ride the cohort
gossip established in the prereq.

## Willingness / admission

Per `docs/cohort-topic.md` §Willingness (L170-194). When `RouteAndMaybeAct` lands a
registration on a member, that member runs the willingness check based on:

- its advertised tier set (does it serve this op's tier — from the `NodeProfile` in the
  addressing/ladder ticket);
- its current per-tier load (from the barometer below);
- per-topic budget (how many topics it's already primary for at this tier);
- operator config overrides.

Outcomes:

```ts
export type WillingnessOutcome =
	| { kind: "accepted" }
	| { kind: "unwilling_member"; candidateMembers: PeerId[] }   // others will serve
	| { kind: "unwilling_cohort"; retryAfterMs: number };        // no member will serve

export interface WillingnessCheck {
	evaluate(reg: RegisterV1, self: NodeProfile, now: number): WillingnessOutcome;
}
```

- **Willing** → become primary (assign slots via sharding ticket) → `Accepted`.
- **Unwilling personally, gossip says a sibling will** → `UnwillingMember(candidateMembers)`.
- **Unwilling, gossip says no member will** → `UnwillingCohort(retryAfter)`.

The cohort gossips a coarse willingness vector (1 bit/tier/member, refreshed each round —
from the gossip ticket), so any member answers member-vs-cohort without polling siblings.
Per §Why the caller doesn't walk on UnwillingCohort (L187-194), `UnwillingCohort` is a
**temporal** back-off — callers do not move spatially. The `retryAfter` staggers
re-registration so a cohort under load does not cascade (no flood) — exponential back-off
on repeated rejection (per anti-DoS ticket's rate limiter, but the base willingness
back-off lives here).

Quorum-level willingness: a cohort takes on tier-T duties for a topic only if a quorum of
members is willing to serve T (per §Tier ladder L159); otherwise registrations get
`UnwillingCohort`. Derive the quorum view from the barometer/willingness gossip.

## Capacity barometer

Per §Capacity barometer (L198-216):

```ts
export interface LoadBarometer {
	tier: Tier;
	bucket: number;   // 0..7, log-bucketed utilization
}
```

- 3-bit per-tier load buckets (4 tiers) + a willingness bit fit in 16 bits; gossiped in
  `CohortGossipV1.loadBuckets`/`willingnessBits` (already on the wire).
- A member observing its own per-tier utilization cross a threshold flips its willingness
  bit for that tier; siblings see it within one gossip round.
- **Early-promote trigger:** when the cohort is hot at the tier it serves
  (`bucket ≥ bucket_overload`, default 6), promotion fires earlier than `cap_promote`
  (the promotion ticket consumes this; here, expose the `bucket ≥ bucket_overload` signal).
- Not aggregated across the tree — children promote independently.

## Topic-traffic signal

Per §Topic traffic signal (L220-246):

```ts
// TopicTrafficV1 already defined in wire ticket:
// { windowSeconds, arrivalsPerMin, queriesPerMin, directParticipants, childCohortCount }

export interface TrafficCounters {
	recordArrival(topicId: Uint8Array, now: number): void;   // fresh register OR renewal (combined)
	recordQuery(topicId: Uint8Array, now: number): void;     // application-level query
	snapshot(topicId: Uint8Array): TopicTrafficV1;           // from gossip-derived view, lags <= 1 round
}
```

- `arrivalsPerMin` deliberately **combines** fresh registrations + renewals (one scalar);
  `queriesPerMin` is application-level query rate over `windowSeconds` (default 60).
- Sent as **exact integers** (not bucketed like the barometer).
- Computed locally, gossiped within the cohort; the reply uses the responder's most-recent
  gossip-derived view (lags ≤ one round; no recompute at reply time).
- **Resolved open question (GROUNDING):** traffic is returned **only** on `accepted` and
  `promoted` replies; absent on `no_state`, `unwilling_member`, `unwilling_cohort` (matches
  the `RegisterReplyV1.topicTraffic` comment in the wire ticket and the locked design in
  `tickets/complete/cohort-topic-traffic-signal`). A participant getting `Promoted(d+1)`
  still gets the outgoing cohort's traffic so it can estimate whether the redirect target
  is hot.
- Counters reset to zero on `cohortEpoch` change; first round after rotation may
  under-report (consumers tolerate a single zero).
- The layer does not interpret the signal (advisory only).

### Resolved open question (GROUNDING): T3 willingness gradations

Decision: keep willingness at **1 bit per tier** (no finer T3 gradations such as
subscriber-count buckets) for now; the barometer bucket already provides coarse load and
finer gradations buy little vs. gossip cost. Document this decision; reactivity's Edge/Core
policy ticket may revisit if simulator/e2e shows need.

## Simulator dependency

The barometer/willingness thresholds (`bucket_overload`, the per-tier utilization flip
thresholds) and the `retryAfter` back-off curve are simulator-validated
(`simulator-churn-and-willingness` confirms back-off drops accepted/sec to a
capacity-matched level with no cascade). Build thresholds to the folded-back values; this
ticket's prereqs already include the gossip ticket which transitively follows the
fold-back via the addressing prereq, but treat the numeric thresholds as "use the values
in docs/cohort-topic.md §Configuration as folded back by the simulator".
