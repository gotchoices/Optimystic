# Reactivity rotation — resume↔handoff coordination + `RotationRedirectV1` wire codec (db-core)

This is the **pure db-core** half of wiring the tail-rotation lifecycle onto a live node. It has no
host/transport dependencies and lands first; the three db-p2p tickets (`reactivity-rotation-recover-redirect-drain`,
`reactivity-rotation-rereg-scheduler`, `reactivity-rotation-host-wiring-e2e`) build on the seams it exposes.

## Background

The rotation lifecycle logic exists in `packages/db-core/src/reactivity/rotation.ts` (11.5) and is exercised
logic-only. Two db-core seams it leaves open block the host wiring:

1. **Handoff ↔ resume coordination.** `applyRotationHandoff` records the outgoing tail's final checkpoint on
   the new tail's `PushState.inheritedCheckpoint` (`push-state.ts`). But `classifyResume` / `serveResume`
   (`resume.ts`) consult only the live rolling `checkpoint`, so a `ResumeV1` whose `fromRevision` falls below
   the new tail's young ring **and** below its (still-empty) rolling checkpoint, but **is** covered by the
   inherited checkpoint, is misclassified `out_of_window` — the subscriber needlessly chain-reads even though
   the new tail "holds the old checkpoint" exactly to answer it (`docs/reactivity.md` §Tail rotation step 5).

2. **`RotationRedirectV1` has no codec.** `rotation.ts` defines `RotationRedirectV1` (`{ v, result:"rotated",
   newTailId, newTopicId, effectiveAtRevision }`) and `TailDrainGate` produces it, but nothing serializes it.
   The drain-window redirect (the db-p2p follow-on) needs it framed over the **recover** request-reply
   protocol (`/optimystic/reactivity/1.0.0/recover`), the only reactivity surface a subscriber reaches a
   serving cohort on. (A fresh subscribe rides generic cohort-topic `service.register`, whose walk only
   understands tier-`Promoted`, never a topic redirect — so the redirect cannot ride registration; it rides
   recover.)

## What to build

### A. Resume serves from the inherited checkpoint

Extend the resume classifier/server so a cross-rotation resume is answered from the inherited checkpoint.

- `ResumeServingDeps` (`resume.ts`) gains an optional `inheritedCheckpoint?: CheckpointSummary` (the new
  tail's `PushState.inheritedCheckpoint`).
- `classifyResume` decision order becomes: `tail_rotated` → `backfill` → `checkpoint_window` (rolling) →
  **`checkpoint_window` (inherited)** → `out_of_window`. Add an `inherited?: CheckpointSummary` parameter;
  when the rolling checkpoint does **not** cover `fromRevision` but the inherited one does
  (`checkpointCovers(inherited, fromRevision)` from `rotation.ts`), classify `checkpoint_window`.
- `serveResume`, on the inherited-checkpoint branch, returns a `checkpoint_window` reply carrying the
  **inherited** `CheckpointSummary` plus the live ring's `recentEntries` (same shape as the rolling-checkpoint
  branch). The subscriber-side `applyResumeReply` is **unchanged** — it already verifies bracketing endpoints,
  applies the digest, `rebaseline`s, and replays recent entries; an inherited checkpoint is just a
  `CheckpointSummary` like any other.
- Precedence: the rolling checkpoint wins when both cover `fromRevision` (it is the fresher, narrower window).
  The inherited checkpoint is only consulted when the rolling one misses.

### B. `RotationRedirectV1` codec on the recover envelope

Give `RotationRedirectV1` a codec by adding a third `RecoverReplyV1` kind, `"rotated"`, in `recover.ts`.

- `RecoverKind` stays `"backfill" | "resume"` on **requests** (a subscriber never *asks* for a redirect).
- `RecoverReplyV1` gains `kind: "backfill" | "resume" | "rotated"` and an optional `rotated?:
  RotationRedirectV1` body, present iff `kind === "rotated"` (mirror the existing `rejectStrayBranch`
  discipline: a `"rotated"` reply must carry `rotated` and neither `backfillReply` nor `resumeReply`).
- Add `validateRotationRedirectV1` (move/define alongside the `RotationRedirectV1` interface in `rotation.ts`,
  or in `recover.ts` next to the envelope — keep it with the other reactivity validators using
  `wire-validate.ts` helpers): require `v === 1`, `result === "rotated"`, base64url `newTailId` / `newTopicId`,
  and a non-negative integer `effectiveAtRevision`.
- `validateRecoverReplyV1` handles the `"rotated"` branch; `encodeRecoverReplyV1` / `decodeRecoverReplyV1`
  round-trip it (no signature on a reply — replies are self-verifying via the data they carry / are
  unauthenticated hints exactly like the existing reply variants).
- Export `validateRotationRedirectV1` from `index.ts` (the `RotationRedirectV1` type is already exported).

## Edge cases & interactions

- **Inherited vs. rolling checkpoint both cover `fromRevision`** → rolling wins (assert in a test). A resume at
  exactly the rolling checkpoint's low edge classifies rolling, not inherited.
- **Inherited checkpoint present but `fromRevision` above the ring low** → `backfill` (ring wins; inherited is
  the deepest, last-resort window).
- **Non-abutting inherited checkpoint.** `applyResumeReply` already rejects a `checkpoint_window` reply whose
  `summary.fromRevision > lastRevision + 1` (forged/buggy → chain-read). The honest serve must therefore only
  pick the inherited branch when it abuts the subscriber's contiguity head; since the server cannot see
  `lastRevision`, it picks on `checkpointCovers(inherited, fromRevision)` (inclusive span) — verify the
  subscriber-side guard still holds for an inherited reply (a test feeding a non-abutting inherited summary
  must yield `checkpoint_untrusted`, unchanged behavior).
- **Empty inherited checkpoint / `undefined`** → behaves exactly as today (`out_of_window`).
- **`RotationRedirectV1` branch/kind mismatch** — a `"rotated"` frame carrying a stray `resumeReply`, or a
  `"resume"`/`"backfill"` frame carrying a stray `rotated`, must `failWire` (mirror existing tests).
- **Malformed `newTopicId` / `newTailId`** (not base64url) → `failWire` on decode.
- **`effectiveAtRevision` negative / non-integer** → `failWire` (use `reqIntInRange(..., 0)`).
- **Forward-compat:** an old decoder seeing `kind:"rotated"` already fails closed (the current `reqKind`
  rejects anything but backfill/resume) — note this in the JSDoc so the db-p2p side knows a peer that predates
  this change cannot honor a redirect (it falls back to chain-read, which is safe).
