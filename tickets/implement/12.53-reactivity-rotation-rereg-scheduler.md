# Reactivity rotation — re-registration timer host seam (db-p2p)

The subscription manager surfaces a `RotationNotice{ newTailId, preAnnounced, plan: ReRegistrationPlan }`
(once per successor tail) but nothing schedules the timer or performs the move. This ticket builds the
standalone, unit-testable host component that does. It depends only on already-landed seams
(`RotationNotice` / `ReRegistrationPlan` / `createRejoinJitter` / `DEFAULT_CAP_PROMOTE_FAST`); the node
composition that feeds it lives in `reactivity-rotation-host-wiring-e2e`.

## What to build

A `RotationReRegistrationScheduler` (`rotation-rereg-scheduler.ts`):

- Consumes a `RotationNotice`: schedules a one-shot timer for `max(0, plan.fireAt - now())`, and on fire
  invokes an injected `reRegister(plan: ReRegistrationPlan): Promise<void>` seam.
- Injected `setTimer(fn, delayMs): () => void` (returns a cancel handle) and `now(): number` clock, so the
  scheduler is deterministic under test (no wall clock). The production node binds `setTimer` to
  `setTimeout`/`clearTimeout` with an **unref'd** timer (must not pin an idle process — mirror the
  push-state-gossip driver's unref'd timer).
- De-dupes by successor `newTopicId` (base64url): a second notice for a successor already scheduled/fired is
  ignored (the manager already fires once per successor, but the scheduler must be independently idempotent
  since a redirect and a pre-announce can both surface the same successor).
- `cancel(newTopicId?)` / `stop()`: cancel one or all pending timers (teardown).
- `reRegister` failures are isolated + logged (a failed move must never throw out of a timer callback); an
  optional bounded retry is acceptable but not required this pass — document whichever you choose.

### Load-bearing jitter contract

The `plan.fireAt` in the `RotationNotice` is drawn by the **manager's** `rejoinJitter`. The wave path must
bound the new tail's inbound to `cap_promote_fast` (32), so wherever the manager's `rejoinJitter` is
constructed (the subscribe factory, `reactivity-rotation-host-wiring-e2e`) it **MUST** be
`createRejoinJitter({ capPromote: DEFAULT_CAP_PROMOTE_FAST })` — the default `createRejoinJitter()` cap is the
cohort-failure `cap_promote = 64` and silently overruns the rotation burst ceiling (11.5
`planReRegistrationWave` JSDoc). The scheduler itself only *consumes* `plan.fireAt`, but its JSDoc must state
this contract loudly so the composing ticket honors it (and so a reviewer checks the construction site).

## Edge cases & interactions

- **`fireAt` already in the past** (`fireAt <= now`) → fire immediately (delay clamped to 0), not negative.
- **Duplicate successor notice** (redirect after pre-announce, or vice-versa) → second is a no-op while the
  first is pending/fired.
- **Chained rotation OLD→A→B** before A's timer fires → A and B are distinct successors → two independent
  timers; both may fire (the manager/scheduler do not cancel A on B — re-registering at A then immediately
  rotating A→B is self-correcting via the manager's `rotationHandledFor`). Document this; do not try to cancel
  superseded timers (a superseded re-register is harmless and rare).
- **`stop()` during a pending timer** → the timer is cancelled and never fires (no `reRegister` after stop).
- **`reRegister` throws / rejects** → logged + swallowed; other pending timers unaffected.
- **Timer cancel-handle correctness** → cancelling after fire is a safe no-op; cancelling twice is a no-op.
