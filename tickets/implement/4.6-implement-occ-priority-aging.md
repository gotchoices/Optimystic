## Background

When two transactions are pending on the same block *at the same time*, the cluster runs
`resolveRace(existing, incoming)` (`cluster-repo.ts` ~line 1470). Today the order is:

1. more promise signatures wins;
2. tie → lexicographically higher `messageHash` wins.

Because each retry produces a fresh transaction id (new hash), a starved transaction gets a
brand-new ~coin-flip in every race — its probability of losing *k* in a row is 2⁻ᵏ but it never
accrues any advantage from having already lost. Large transactions touch more blocks, so they
lose these races more often (and lose more stale-read validations — addressed separately by
`implement-occ-structural-read-exclusion`). Theorem 9 openly admits "no deterministic starvation
prevention exists."

This ticket adds an **aged priority** that a transaction carries and that `resolveRace` honors,
so a repeatedly-losing transaction eventually beats fresh ones in the concurrent-race case.

### Scope boundary (read this before implementing)

Priority-in-`resolveRace` guarantees progress only for the **concurrent-contention** case
(two transactions pending on the same block simultaneously). It does NOT by itself defeat
**sequential** starvation — a stream of quick transactions that each *commit and finish* inside
the window while the large transaction is still re-reading, so they never co-pend and never meet
`resolveRace` at all; the large transaction just keeps getting stale-rejected at PEND. Closing
that residual needs a cluster-side admission reservation (deferring fresh conflicting pends for an
aged high-priority transaction). That is deliberately deferred to backlog
`feat-occ-priority-reservation` — it is a distinct, stateful, Byzantine-sensitive cluster
subsystem and does not belong in this pass. This ticket must not overreach into it; it must
document the residual honestly (see Theorem 9 update below).

## Design

### Priority field on the transaction (db-core)

Add a bounded, monotonic `priority: number` to `Transaction` (default 0):

- Starts at 0 for a first attempt.
- Rises by 1 per failed attempt (attempt-count-derived — deterministic, needs no wall clock,
  and clock-skew-free, consistent with §7.4). The retry loop in `Collection.sync` (and the
  multi-collection coordinator retry from the prereq) sets it from `consecutiveFailures`.
- **Capped** at a small constant `MaxPriority` (e.g. 8). Capping is essential: it bounds the
  fairness effect and bounds the throughput cost (below). Two capped-out transactions fall back
  to the existing promise-count / hash tiebreak, so behavior degrades gracefully to today's.
- Priority is **advisory / fairness-only** and MUST NOT influence validity, the operations
  hash, stale-read checks, or which transaction is *correct* — only which of two
  simultaneously-pending conflicting transactions the race keeps. This keeps it out of the
  safety argument entirely (Theorem 1 is unaffected).

The transaction id must stay a pure function of `(stamp.id, statements, reads)` — do NOT fold
priority into `createTransactionId` or the client-signature payload (`clientSignaturePayload`),
or every retry's identity/signature churns and anti-replay/OCC binding breaks. Priority is a
mutable retry-scoped field, not part of content identity.

### Carry priority into the signed record (db-p2p)

`cluster-coordinator.ts` builds the `ClusterRecord.message` that members sign (its digest feeds
`messageHash`/`promiseHash`/`commitHash`). Include priority in that message so it is
integrity-protected in transit — a relaying peer cannot strip or inflate it without invalidating
signatures. (The *originating* coordinator can still self-assert its own priority; that is a
fairness concern, not a safety one — see Byzantine note below.)

### Honor priority in resolveRace (db-p2p)

New order in `resolveRace`:

1. **higher priority wins** (new, first);
2. more promise signatures wins;
3. tie → higher `messageHash` wins.

Priority first (ahead of promise count) is what lets an aged transaction win even when a fresh
rival has briefly gathered an equal/greater promise count. Keep the comparison total and
deterministic so every honest member computes the identical winner (Theorem 1 Case 2 premise).

### Progress bound (state precisely in Theorem 9)

Under the honest-majority assumption the correctness doc already relies on:

- In the concurrent-race case, once a starved transaction's priority exceeds any fresh
  transaction's initial priority (0) — i.e. after at most `MaxPriority` losing attempts — honest
  cluster members deterministically pick it over every fresh rival in `resolveRace`, so it wins
  every subsequent *concurrent* race. Combined with backoff+jitter (prereq) thinning
  self-inflicted contention, the concurrent-starvation probability collapses from "2⁻ᵏ but
  memoryless" to "bounded number of losses, then deterministic win."
- Residual (documented, NOT fixed here): sequential starvation by sub-window quick transactions
  is not defeated by `resolveRace` priority; see `feat-occ-priority-reservation`.

Update Theorem 9's "Bound" paragraph to reflect this (it currently says "No deterministic
starvation prevention exists" — soften to "deterministic among concurrently-pending conflicts;
sequential-stream fairness is bounded by the reservation follow-up").

## Edge cases & interactions

- **Priority-aging vs livelock (the guarantee).** Test: transaction A repeatedly loses to a
  sequence of fresh transactions in *concurrent* races; assert that within `MaxPriority + 1`
  rounds A's priority makes `resolveRace` pick A, and A commits. This is the core progress bound.
- **Cap prevents runaway advantage.** Two aged, capped-out (priority == MaxPriority) conflicting
  transactions must resolve by the *existing* promise/hash tiebreak — deterministic and
  symmetric, no deadlock. Unit-test `resolveRace` directly at equal max priority.
- **Priority never affects safety.** Test that a higher-priority transaction that is *stale*
  (bad read set) is still rejected at PEND validation — priority does not buy a stale transaction
  a commit. Priority must be absent from `operationsHash`, read-dep checks, and validity.
- **Identity/signature stability.** Two attempts of the same logical transaction differ in
  priority but produce transaction ids and client signatures per the *unchanged* payload
  (priority excluded). Round-trip test: same statements+reads ⇒ same tx id regardless of priority.
- **Integrity in transit.** A record whose priority was altered after signing fails
  message-hash/signature verification at the member (priority is inside the signed message). Test
  a tamper that bumps priority ⇒ `Message hash mismatch`.
- **Byzantine coordinator self-asserting max priority (fairness DoS, not safety).** Document and
  test that a coordinator declaring `priority == MaxPriority` on every transaction cannot cause
  two conflicting commits (safety intact) — it only wins races it might have ~50% won anyway,
  degrading to at-worst-status-quo fairness. This is the same graceful-degradation class as
  spam under honest-majority; hardening (binding priority to provable age) is explicitly out of
  scope and noted as a tripwire.
- **Fairness vs throughput bound.** Aging a large transaction up makes more short transactions
  lose *that block's* races while it is pending. Bound the cost: the effect is limited to
  conflicts on the specific block(s) the aged transaction pends, for the duration of its pend,
  and only until it commits (≤ `MaxPriority` extra losses imposed on rivals per aged transaction).
  Note this reasoning as a `NOTE:` at the `resolveRace` site.
- **Interaction with hot-log-tail (design-hot-log-tail-sharding-guidance).** `resolveRace` and
  the record message are the same path a future HLC/crdt-sync redesign touches. Keep priority a
  self-contained additive field on the message + one comparison key, so it composes with (does
  not block) that redesign. Add a `NOTE:` cross-reference at the resolveRace site.
- **Legacy / unversioned records.** A record with no priority field is treated as priority 0
  (backward compatible; an older coordinator's transactions simply never age). Verify a
  mixed-version race resolves deterministically.
