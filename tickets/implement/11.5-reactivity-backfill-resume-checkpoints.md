# Reactivity recovery: backfill RPC, parent checkpoints, and resume protocol

This ticket implements recovery for subscribers that fall behind, from a short gap (served from
the replay buffer) through a long sleep (served from a parent checkpoint) to out-of-window
(chain read) and across a tail rotation. It extends the `PushState`/`replayBuffer` substrate
built by [reactivity-origination-replay-delivery](#) and consumes the `requestBackfill` seam
that ticket defined.

## Architecture

### Backfill RPC

Served from the replay buffer (`docs/reactivity.md` §Backfill RPC L252–259):

```ts
interface BackfillV1 {
	v: 1;
	collectionId: string;   // base64url
	fromRevision: number;
	toRevision: number;
	signature: string;      // subscriber peer-key sig
}

interface BackfillReplyV1 {
	v: 1;
	entries: NotificationV1[];        // signed, end-to-end verifiable
	available: { from: number; to: number };
}
```

A subscriber MAY request a sub-range smaller than `[fromRevision, toRevision]`; the cohort
returns the **intersection** with its replay buffer and reports `available` so the subscriber
knows whether to fall back further (to a checkpoint or a chain read). Entries retain the
original threshold signature, so backfill responses are verifiable end-to-end. Any cohort
member can serve (the buffer is gossiped, per the origination ticket).

### Parent checkpoint summaries

A `W`-revision replay buffer (256 at 1 cps ≈ 4 min) is too short for a phone asleep overnight.
To extend recoverable range without ballooning replay memory, every parent forwarder cohort
(and the tail cohort) maintains a rolling `CheckpointSummary` (`docs/reactivity.md` L233–250):

```ts
interface CheckpointSummary {
	collectionId: Uint8Array;
	fromRevision: number;
	toRevision: number;            // toRevision - fromRevision ≈ W_checkpoint (default 4096)
	mergedDigest: Uint8Array;      // application-defined fold of per-entry digests
	mergedDelta?: Uint8Array;      // optional, bounded; coalesced delta if collection supports
	bracketingSigs: [Uint8Array, Uint8Array]; // threshold sigs of the entries at from/to
}
```

The checkpoint is a **hint summary, not a chain replacement**. The two bracketing signatures
let a subscriber verify the endpoints are real committed revisions; `mergedDigest` tells the
application what changed across the range (enough for KV-shaped collections to invalidate caches
without a chain read; collections needing exact intermediate state fall back to the chain).
`W_checkpoint` defaults to 16× `W` (4096 ≈ 1 h at 1 cps), configurable per collection. Tier
`d ≥ 1` cohorts are the primary checkpoint holders; the tail cohort holds the current rolling
checkpoint, advancing `fromRevision` as revisions retire from the replay buffer.

This fills the `parentCheckpoint` field reserved in `PushState` by the origination ticket.

### Resume protocol

A subscriber resuming from sleep sends (`docs/reactivity.md` §Resume L218–231, §Wire formats
L373–404):

```ts
interface ResumeV1 {
	v: 1;
	collectionId: string;
	fromRevision: number;
	latestKnownTailId: string;
	subscriberCoord: string;
	timestamp: number;        // unix ms
	signature: string;        // subscriber peer-key sig
}

interface ResumeReplyV1 {
	v: 1;
	result: "backfill" | "checkpoint_window" | "out_of_window" | "tail_rotated";
	// backfill:
	entries?: NotificationV1[];
	currentRevision?: number;
	// checkpoint_window:
	checkpoint?: {
		fromRevision: number; toRevision: number;
		mergedDigest: string; mergedDelta?: string;
		bracketingSigs: string[]; // length 2
	};
	recentEntries?: NotificationV1[];
	// out_of_window:
	currentTailId?: string;
	// tail_rotated:
	newTailId?: string;
	newRevisionAtRotation?: number;
}
```

Classification at the serving cohort:

- `fromRevision` within the replay window → **Backfill** (`entries`, `currentRevision`). One RT.
- older than the buffer but within parent-checkpoint range → **CheckpointWindow** (`checkpoint`
  + `recentEntries`). Subscriber applies the merged digest, then replays the recent entries
  deduped against `lastRevision`. One RT.
- older than even the checkpoint → **OutOfWindow** (`currentTailId`, `currentRevision`).
  Subscriber falls back to a chain read, then issues a fresh subscribe.
- `latestKnownTailId` stale → **TailRotated** (`newTailId`, `newRevisionAtRotation`). Subscriber
  re-registers under the new tail and replays from the new tree. (The rotation *lifecycle* —
  drain, jittered re-registration, buffer-to-checkpoint handoff — is owned by
  [reactivity-rotation-backpressure-policy](#); this ticket only needs to *detect* a stale tail
  and emit the `TailRotated` reply so resume composes with rotation. Coordinate the
  `latestKnownTailId`-staleness check with that ticket.)

`cohortHint` is sticky-cached for Edge so resume after a brief flap can hit the cached primary
directly (one RT) instead of re-walking from `d_max`.

### Design questions to resolve (from GROUNDING / `docs/reactivity.md` open questions)

- **`mergedDigest` semantics.** Docs say "application-defined fold." Confirm against the
  transaction-layer digest field (`NotificationV1.digest` originates from the tx commit digest):
  decide whether `mergedDigest` is a collection-defined fold of per-entry digests or a
  system-level deterministic field. Document the chosen path in `docs/reactivity.md` and, if it
  is system-level, reference `docs/transactions.md` for the digest definition.
- **Delta coalescing vs `delta_max`.** If `W_checkpoint = 4096` and `delta_max = 4 KB`, a single
  coalesced `mergedDelta` may blow past `delta_max`. Decide: omit `mergedDelta` when the coalesced
  size would exceed `delta_max`, or split. Document the rule.
- **Resume on rotation.** Current design does a full walk from `d_max` on `TailRotated`. Decide
  whether to optimize with a pre-dial toward the pre-announced successor coord, or keep the
  simple full walk. Defer the pre-dial to the rotation ticket if it requires rotation-side state.

### Simulator-dependent parameters

`W_checkpoint` (default 4096) and the delta-coalescing policy are **provisional** pending
[simulator-reactivity-replay](#) (which measures resume RPC counts/latency by wake-lag and the
`W` vs `W_checkpoint` ratio) and the [fold-simulator-findings-into-design-docs](#) bridge.
Source `W_checkpoint` from the shared reactivity defaults table so the fold-back can revise it.
