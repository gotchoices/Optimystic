The pure `db-core` half (`implement-client-tx-signature-core`) added the signature field, the canonical signing bytes, and the sign/verify *ports*, but left them un-backed (no port injected ŌćÆ transactions are unsigned and unverified). This ticket supplies the real libp2p Ed25519 signer and verifier, and adds the config flag that decides when verification is enforced.

## Design decisions (settled ŌĆö do not re-open)

**Identity = the libp2p peer id; no separate key distribution.** For a libp2p Ed25519 identity the peer-id string **embeds the raw public key** (`peerIdFromString(str).publicKey.raw`). `TransactionStamp.peerId` is that string, so a verifying node derives the client's public key straight from it ŌĆö no lookup, no cert. Reuse the existing primitives:
- **Sign**: the node's libp2p `PrivateKey`. Use `signPeer` (async) from `db-p2p/src/cohort-topic/peer-sig.ts`, base64url-encode (same `bytesToB64url` helper the reactivity signer uses).
- **Verify**: `verifyPeerSig(peerId, payload, sigBytes)` from the same module (synchronous, total ŌĆö returns false on any malformed/non-Ed25519 input, never throws), after base64url-decoding the signature. `verifyPeerSig` already derives the pubkey from the peer-id and checks the Ed25519 signature, so it *is* the identity binding for this path; `peerIdBindsPublicKey` is available if an explicit binding assertion is wanted, but `verifyPeerSig` subsumes it.

Non-Ed25519 identities are out of scope: `verifyPeerSig` returns false for them, so with enforcement on they are rejected. This matches the cohort-topic substrate's standing Ed25519 assumption ŌĆö document it, don't try to support other key types.

**Signer injection ŌĆö at the session seam.** `txn-bridge.ts:~260` calls `TransactionSession.create(this.coordinator, this.engine, peerId, schemaHash)`. Add the `signer` argument. The signer closure captures the node's libp2p `PrivateKey`; source it from `collectionFactory` (it already owns the node handle ŌĆö see `collection-factory.ts` `node.peerId` usages ~184/382). Add a `getSigner(options)` (or `getPrivateKey(options)`) accessor on the collection factory alongside the existing `getPeerId(options)`, returning a `TransactionSigner` bound to that node's key, or `undefined` when no node/key is available (legacy/local mode) ŌĆö in which case the session is created without a signer (unsigned, as today).

**Verifier injection + enforcement flag ŌĆö at the validator seam.** `quereus-validator.ts:85` builds `new TransactionValidator(engines, createValidationCoordinator, options.blockStateProvider)`. Add the `verifyClientSignature` port (4th arg) as a closure over `verifyPeerSig` + base64url-decode. Gate it on a new option:
- `requireClientSignature: boolean` (default **false** for now ŌĆö phased rollout; flip the default in a later hardening pass once clients sign in the field).
- When `true` ŌåÆ inject the verifier port (so `validate()` rejects missing/invalid signatures at pend).
- When `false` ŌåÆ do **not** inject it (verification skipped; unsigned transactions accepted). This keeps existing single-node / dev / not-yet-migrated deployments working.

Thread `requireClientSignature` through whatever options object feeds `quereus-validator` (mirror how `blockStateProvider` is already threaded).

**Backward compatibility / migration.** Because enforcement is off by default, the rollout order is: (1) this lands, clients that have a node key start *signing* immediately (signer is injected whenever a key exists ŌĆö signing is always safe, it just adds a field); (2) once a deployment observes clients signing, it flips `requireClientSignature: true` to start *rejecting* unsigned/invalid. Already-committed unsigned history is never re-verified ŌĆö verification happens only at pend (the acceptance point), so old committed transactions are untouched.

## Edge cases & interactions

- **Signing round-trips through verification** ŌĆö a transaction signed by `signPeer` over `clientSignaturePayload(...)` must verify via `verifyPeerSig` over the same payload. Cover with an integration test that drives a real (or test-harness) libp2p keypair through sign ŌåÆ validate.
- **peerId ŌēĀ signing key** ŌĆö if a client sets `stamp.peerId` to identity A but signs with key B, verification must fail (the pubkey derived from A won't match B's signature). This is the core "can't impersonate another client" property. Test it: sign with one keypair, stamp another peer's id, expect reject.
- **Non-Ed25519 / malformed peerId** ŌĆö with enforcement on, a transaction whose `peerId` is non-Ed25519 or malformed rejects (verifier returns false). Confirm no throw.
- **Base64url round-trip** ŌĆö signature encode (sign path) and decode (verify path) must use the same alphabet/padding convention as the existing reactivity signer (`bytesToB64url` / its inverse); a mismatch silently fails every verification. Add a unit assertion.
- **Enforcement off, signed transaction present** ŌĆö a signed transaction submitted to a node with `requireClientSignature: false` must still commit (the extra field is ignored, not rejected). Ensures mixed-fleet rollout works.
- **Enforcement on, no signer at client** ŌĆö a client with no node key (legacy/local mode) produces unsigned transactions; against an enforcing node these reject at pend with `Missing client signature`. Document this as the expected migration failure mode (operators must not flip enforcement on before clients sign).
- **Signer failure** ŌĆö if `signPeer` throws (e.g. wrong key type), the session commit should surface a clean error, not a partially-built unsigned transaction that later mysteriously rejects. Wrap and return a failure from `commit()`'s signer call.
- **Recovery re-verification (Theorem 13 anchor)** ŌĆö this ticket does not build recovery, but confirm the signature survives the pend/commit persistence path (cluster record / log) so a future recovery reader can re-run `verifyPeerSig`. A round-trip assertion through the transactor's serialization is enough here.
