# Reactivity recover RPC transport — live libp2p protocol + dial + node wiring (db-p2p)

## Why this ticket exists

This is the **db-p2p transport layer** for the reactivity recovery (pull) direction — the companion to the
notification (push) transport (`reactivity-notification-transport`, 12.3). It consumes the wire framing +
signing-payload helpers landed by its prereq `reactivity-recover-wire-signing` and makes the
backfill/resume RPC **live** on a running `cohortTopic`-enabled node.

Today (after 11.5 + the wire-signing prereq) db-core has the pure servers (`serveBackfill`,
`serveResume`/`classifyResume`) and the subscriber-side drivers (`createBackfillRequester`,
`applyResumeReply`, `ReactivitySubscriptionManager.resume()` + the backfill seam), wired only through
**injected** `BackfillTransport`/`ResumeTransport` function seams. No libp2p protocol decodes a recover
frame off the wire and calls the server, and no dial path sends one from a subscriber to its serving
cohort. This ticket fills those seams with a real protocol. **It does not redesign the seams** — the
db-core function shapes (`(req) => Promise<reply>`) are unchanged.

## Architecture

```
 subscriber node                                        serving cohort member (any member — ring+checkpoint gossiped)
 ───────────────                                        ──────────────────────────────────────────────────────────
 gap detected / wake                                    node.handle("/optimystic/reactivity/recover/1.0.0", …)
   │                                                      │  read RecoverRequestV1 frame (bounded)
   ├─ build BackfillV1 / ResumeV1 (db-core)               ├─ verifyPeerSig(remotePeer, <sig payload>, sig)   ← reject on fail
   │  sign over backfill/resumeSigningPayload (peer key)  ├─ CorrelationReplayGuard.accept(sigBytes, remotePeer, ts, now) ← reject replay/stale
   ├─ wrap RecoverRequestV1{kind, body}                   ├─ resolve live PushState (by topicId / collectionId)
   ├─ dial serving cohort:                                ├─ serveBackfill(buffer, req, collId)  OR
   │    sticky ReactivityCohortHint.primary (1-RT)        │     serveResume(req, {buffer, checkpoint, currentTailId, currentRevision, …})
   │    └─ stale/miss → cohort-walk (CohortPeerResolver)  └─ reply RecoverReplyV1{kind, …Reply} on the SAME stream
   └─ requestResponse() → RecoverReplyV1
        backfill   → createBackfillRequester replays entries (verify→contiguity→deliver→dedupe)
        resume     → applyResumeReply (backfill/checkpoint replay; out_of_window/tail_rotated escalate)
```

The recover protocol is **request-reply** (one frame each way over a single stream), so reuse
`requestResponse` (outbound) and `handleRequestResponse` (inbound) from
`packages/db-p2p/src/cohort-topic/stream-util.ts` — do **not** use the one-way `sendOneWay`/notify helpers
(those would desync the dialer). This mirrors the cohort-topic `sign` request-reply protocol.

### Outbound: `Libp2pReactivityRecoverTransport` (new, `recover-transport.ts`)

A single class exposing the two db-core seams against one libp2p node:

```ts
class Libp2pReactivityRecoverTransport {
  constructor(node: Libp2p, opts: {
    recoverProtocol?: string;          // default PROTOCOL_REACTIVITY_RECOVER
    selfPeerId: string;                // node.peerId.toString()
    cohortHintCache: StickyCohortHintCache;     // shared with the subscription manager
    resolveCohort: (topicId: Uint8Array) => string[]; // FRET walk → peer-id strings (cohort-walk fallback)
    maxBytes?: number;
  });
  backfillTransport(topicId: Uint8Array, collectionId: string): BackfillTransport;
  resumeTransport(topicId: Uint8Array, collectionId: string): ResumeTransport;
}
```

Each returned transport: pick a target — **sticky hint first** (`cohortHintCache.get(collectionId)?.primary`,
decoded to a peer-id string), else **cohort-walk** (`resolveCohort(topicId)` → first reachable member);
frame `RecoverRequestV1`; `requestResponse` to the target over the recover protocol; decode
`RecoverReplyV1`; return the inner `backfillReply`/`resumeReply` (throw `failWire` on a kind mismatch). On a
dial failure against the sticky primary, fall back to the cohort-walk target once before throwing (the
caller's retry/escalation policy, below, handles the rest).

**Dial-target encoding bridge (load-bearing — the same trap 12.3 calls out, one layer over).** The sticky
`ReactivityCohortHint.primary` and `cohortHint[]` are **base64url of cohort member-id bytes**
(`bytesToB64url(peerIdToBytes(peerIdString)) = base64url(utf8(peerIdString))`), per `resume.ts`'s
`ReactivityCohortHint` JSDoc ("All ids base64url"). `requestResponse` calls `peerIdFromString(target)`, so
the transport MUST convert: `bytesToPeerIdString(b64urlToBytes(hint.primary))`. The `resolveCohort` walk,
by contrast, already returns peer-id strings (it's the cohort-topic `CohortPeerResolver` space). Feeding a
base64url-of-bytes string straight to `peerIdFromString` throws → swallowed → recovery silently never
reaches the cohort. Pin both directions with an assertion (below).

### Inbound: recover protocol handler (in `libp2p-node-base.ts` node wiring)

`handleRequestResponse(node, PROTOCOL_REACTIVITY_RECOVER, async (frame, fromPeer) => { … })`:

1. `decodeRecoverRequestV1(frame, maxBytes)` (swallow a decode failure → no reply → `handleRequestResponse`
   aborts the stream; never throw out of the handler).
2. **Verify the request signature** against `fromPeer` (the dialing subscriber's peer id IS the signer —
   no signer-id field on the wire): `verifyPeerSig(fromPeer.toString(), backfillSigningPayload(unsigned) |
   resumeSigningPayload(unsigned), b64urlToBytes(req.signature))`. Reject (no reply) on failure.
3. **Replay guard.** A node-level `CorrelationReplayGuard` (`createCorrelationReplayGuard()` from db-core
   `antidos/replay-guard.js`): `guard.accept(b64urlToBytes(req.signature), peerIdToBytes(fromPeer.toString()),
   req.timestamp, Date.now())` — the **signature bytes are the anti-replay key** (unique, authenticated;
   see the wire-signing ticket's decision 2). Reject (no reply) on a stale/future/replayed request.
4. **Resolve the live PushState** for the request and dispatch:
   - **resume:** derive `topicId = reactivityTopicId(b64urlToBytes(req.resume.latestKnownTailId))`; look up
     the served `PushState` for that exact topic via the forwarder host (`fwd.pushStateFor(topicId)`). Call
     `serveResume(req.resume, { buffer: ps.replayBuffer, checkpoint: ps.checkpoint, currentTailId:
     ps.tailIdAtJoin, currentRevision: ps.lastRevision, rotationRevision: ps.lastRevision,
     expectedCollectionId: ps.collectionId })`. If the node holds **no** PushState for that tail's topic
     (it migrated / never served it), the cohort can still answer `tail_rotated`/`out_of_window` only if it
     can resolve the collection's current tail — resolve the current served PushState for
     `req.resume.collectionId` (see collectionId index below) and serve from it (its `currentTailId` differs
     from the request's stale `latestKnownTailId` ⇒ `classifyResume` returns `tail_rotated`). If neither is
     found, send no reply (the subscriber's transport falls back to a cohort-walk / chain read).
   - **backfill:** `BackfillV1` carries no tailId — resolve the **current** served PushState for
     `req.backfill.collectionId` (collectionId index below) and call `serveBackfill(ps.replayBuffer,
     req.backfill, ps.collectionId)`.
5. Wrap the result as `RecoverReplyV1{ kind, backfillReply | resumeReply }`, `encodeRecoverReplyV1`, return
   it (the handler sends it on the stream).

**collectionId → live PushState lookup.** The forwarder host (`ReactivityForwarderHost`) keys its served
state by **topicId**; a backfill request has only a collectionId. Add a small lookup to the forwarder host:
`pushStateForCollection(collectionId: string): PushState | undefined` returning the served `PushState`
whose `.collectionId` matches (and, when a collection has more than one served topic during a rotation
drain, the one with the **highest `lastRevision`** = the current tail). Bind the recover handler's resolver
to `fwd.pushStateForCollection`. Document that across a rotation drain the old + new tail may both be served
briefly; the highest-revision pick is the current tail (12.5 owns the drain-window redirect refinement).

### Subscription-manager wiring (`subscription-manager.ts`)

The manager already accepts `backfillTransport`/`signBackfill`/`resumeTransport`/`signResume`/
`subscriberCoord`/`cohortHintCache` options and builds the db-core drivers. This ticket supplies the **real**
values from the production subscribe factory (in `libp2p-node-base.ts`), not redesigns the manager. Required
changes:

- **Backfill retry/escalation policy (decide here — resolved):** `onBackfillGap` currently fires the
  backfill driver fire-and-forget and lets a transport failure surface as an **unhandled rejection**.
  Replace with: `void this.backfillRequester(from, to).catch(() => this.escalateBackfill(from, to))`, where
  `escalateBackfill` does a **bounded retry** (default 1 retry; the sticky-hint dial + one cohort-walk
  fallback already live in the transport, so this is a single application-level re-attempt) and, if still
  failing, escalates to `this.resume()` when a resume transport is configured, else invokes
  `onBackfillUnderflow`/`onChainRead`. **Must never block or fault the commit/delivery path** — backfill is
  hint-only; all of this stays off the inbound-notification critical path (it already runs from the
  subscriber's gap seam, not the deliver path). Add a `backfillMaxRetries?: number` option (default 1).
- **`subscriberCoord` sourced from the real ring coordinate.** The manager defaults `subscriberCoord ??
  collectionIdB64` (a placeholder). The production factory MUST pass the subscriber's **real ring
  coordinate** (the same `participantCoord` it registers under at the cohort-topic tier — available from the
  node's peer-id → ring coord, or the registration handle). Because the recover protocol replies on the
  **same stream**, `subscriberCoord` is not used for reply routing by this transport; still, source it
  correctly so the signed field is meaningful and a future out-of-band reply path is unblocked. Drop the
  `?? collectionIdB64` placeholder in favor of the supplied value; keep a documented fallback only if the
  factory cannot yet compute the coord, and log when the fallback is used.
- **`currentTailId`/`rotationRevision` (subscriber side)** are already correct: `ResumeV1.latestKnownTailId
  = tailIdAtAttach`. Serving-side `currentTailId`/`rotationRevision` come from the served `PushState`
  (above). Coordinate with `reactivity-rotation-backpressure-policy`/12.5: the live current-tail value the
  `tail_rotated` classification compares against is `PushState.tailIdAtJoin` for a non-rotated cohort; 12.5
  refines the post-rotation drain behavior. No cross-ticket blocker — design as if 12.5 lands.

### Node wiring (`libp2p-node-base.ts`, the `if (cohortEnabled)` block)

Compose with 12.3's reactivity wiring (forwarder host `fwd`, subscriber registry, notify + push-state-gossip
handlers):

- Construct `const recover = new Libp2pReactivityRecoverTransport(node, { selfPeerId: node.peerId.toString(),
  cohortHintCache, resolveCohort, ... })`, reusing the cohort gossip transport's `CohortPeerResolver` /
  FRET assembly for `resolveCohort` (resolve the recover topic's coord → peer-id strings).
- Register the inbound recover handler (`handleRequestResponse` over `PROTOCOL_REACTIVITY_RECOVER`) bound to
  the serve logic above (`fwd.pushStateFor` / `fwd.pushStateForCollection`, `verifyPeerSig`, the node-level
  replay guard).
- The production **subscribe factory** that constructs a `ReactivitySubscriptionManager` supplies
  `backfillTransport`/`resumeTransport` from `recover.*Transport(topicId, collectionId)`, the real
  `signBackfill`/`signResume` (peer-key sign over `backfillSigningPayload`/`resumeSigningPayload` via
  `signPeer(privateKey, …)` → `bytesToB64url`), `subscriberCoord` (real ring coord), and the shared
  `cohortHintCache`.
- **Protocol registration + teardown.** Add `recover` to `ReactivityProtocols` /
  `DEFAULT_REACTIVITY_PROTOCOLS` / `makeReactivityProtocols` / `reactivityProtocolList`
  (`packages/db-p2p/src/reactivity/protocols.ts` — the `recover` slot is already reserved in a comment;
  promote it to a real field `PROTOCOL_REACTIVITY_RECOVER = ${REACTIVITY_BASE}/recover`). `node.handle` it
  with the other reactivity protocols; the existing `node.stop` wrapper's `node.unhandle(reactivityProtocolList(...))`
  then covers it. No new timer (request-reply is stateless besides the replay guard, which is a plain map
  pruned on access).

- **db-p2p unit (`managers.spec.ts` / a new `recover-transport.spec.ts`):**
  - A `Libp2pReactivityRecoverTransport.backfillTransport` over a fake/in-memory libp2p pair: a backfill

  - **Signature verification:** a request signed with the wrong key is rejected (no reply); the dialer's
    `requestResponse` surfaces the dropped stream as a transport failure → the manager's escalation fires.
  - **Replay guard:** the identical request frame sent twice within the freshness window — the second is

  - **Dial-target encoding (mandatory, mirrors 12.3's no-dial regression test):** a sticky

    `peerIdFromString` **throws** (pinning the bug both ways) and that the transport uses the decoded
    peer-id string.
  - **Backfill escalation:** a backfill whose transport always rejects retries `backfillMaxRetries` times
    then escalates to `resume()` (when a resume transport is configured) — assert the resume transport was
    called; assert no exception escapes onto the caller (no unhandled rejection).
  - **`subscriberCoord`:** a constructed `ResumeV1` carries the supplied ring coordinate, not the
    collectionId placeholder.
  - **collectionId → PushState resolution:** `fwd.pushStateForCollection` returns the highest-`lastRevision`

- **Real-libp2p integration (`substrate-real-libp2p.integration.spec.ts`, env-gated):** a remote subscriber

## Edge cases & interactions

- **Any cohort member can serve.** The ring + rolling checkpoint are gossiped (12.3's push-state-gossip), so
  a recover dial to any member that holds the topic's `PushState` is answerable — the dial does not have to
  hit the primary. The sticky hint targets the primary for the one-RT happy path; the cohort-walk fallback
  reaches any member.
- **Sticky hint stale after a flap vs. after a rotation.** A brief flap → the cached primary is still valid
  → one RT. A rotation → the manager already `invalidate`s the sticky cache (on a `tail_rotated` reply and
  on rotation detection); the next recover re-walks. Do not invalidate the sticky cache on a mere transient
  dial failure (the transport's cohort-walk fallback handles that) — only on confirmed staleness, matching
  the `StickyCohortHintCache` contract.
- **Backfill underflow vs. resume escalation:** `createBackfillRequester` already returns early with
  `onUnderflow` when `available.fromRevision > from` (the ring no longer covers the gap) — that path must
  escalate to `resume()`/chain read, distinct from a **transport failure** escalation. Wire both into the
  manager's escalation seam without double-firing.
- **Self-dial:** a subscriber co-located with a serving cohort member must not dial itself — the transport's
  `selfPeerId` guard short-circuits to an in-process serve, or (simpler) the subscribe factory routes a
  co-located resume through the local `PushState` directly. Pick one and document; the `selfPeerId` guard
  mirrors the notify transport.
- **Malformed / oversized recover frame:** bounded by `maxBytes`; a decode failure in the handler sends no
  reply (stream aborts) and never throws — exactly like the notify handler.
- **Foreign-collection request:** `serveBackfill`/`serveResume` already `failWire` when `collectionId`
  mismatches `expectedCollectionId`; the handler must catch that (no reply) rather than letting it abort
  uncaught. A request whose `collectionId` matches no served PushState → no reply (fallback to walk/chain).
- **Concurrent recovers for the same collection** are independent reads of the gossiped `PushState`
  snapshot (serve is pure over the snapshot) — no per-topic serialization needed on the serve side (unlike
  ingest, which mutates the ring). The replay guard is the only shared mutable state; its map is
  concurrency-safe for single-threaded event-loop access.
- **Edge node as subscriber, Core node as server:** Edge subscribes + dials recover (never serves — it has
  no `PushState`); Core serves. The handler on an Edge node finds no `PushState` and sends no reply
  (harmless — Edge is never a recover target in a correct walk).
- **Mock-tier harness stays in-process.** Like the notify transport, the recover RPC over real sockets
  cannot run on the in-process FRET mesh; `reactivity-mesh-harness` / `mesh-mobile-resume.spec.ts` already
  *model* this transport seam in-process — leave them unchanged. The real-socket coverage is the env-gated
  integration test above.
- **Teardown ordering:** the recover protocol unhandle rides the existing
  `node.unhandle(reactivityProtocolList(...))` in the stop wrapper (before `host.stop()`); no separate
  timer to clear.

## Honest gaps to flag for review

- **Rotation drain-window serving** (the post-rotation period where both old and new tail cohorts serve
  recover requests, and the redirect/`RotationRedirectV1` honoring) is `reactivity-tail-rotation-transport`
  (12.5)'s. This ticket serves `tail_rotated` from the current `PushState.tailIdAtJoin` and the
  highest-revision collection lookup; the precise drain/redirect choreography is 12.5's refinement.
- **`subscriberCoord` reply routing** is unused by this request-reply transport (reply rides the same
  stream). The field is sourced correctly and signed; an out-of-band reply path (if ever added) is future
  work.
- **Replay-guard scope:** a single node-level guard keyed on signature bytes is sufficient (signatures are
  globally unique); if recover volume ever warrants per-topic guards or a tuned window, that is a follow-on.
