# Invalidation certificate: bind votes to the target transaction + dedup arbitrators

This is **part 1 of 2** of the forgeable-invalidation-certificate fix
(`tickets/fix/invalidation-certificate-arbitrator-binding.md`). It closes two of the three missing
bindings — **#2 target binding** and **#3 per-arbitrator dedup** — both of which are fully verifiable
by any member from the proof alone, with no peer-topology or trust-anchor infrastructure. The third
and hardest binding (**#1 arbitrator-set legitimacy**) is split into the follow-up ticket
`invalidation-cert-arbitrator-set-binding`, which depends on this one and additively extends the same
wire format.

## Background (confirmed reproduction)

`verifyInvalidationCertificate` (`packages/db-p2p/src/dispute/invalidation.ts`) is the independent
re-check every member runs over a `DisputeResolutionProof` before applying an invalidation — the
threat model is an originator that lies. Today each vote signs only
`utf8(`${disputeId}:${vote}:${computedHash}`)` (`DisputeService.makeVote`), the verifier confirms that
signature against the Ed25519 key embedded in the claimed `arbitratorPeerId`, and counts
`agree-with-challenger` votes against a 2/3 decisive-vote threshold. Two consequences this ticket fixes:

- **Replay (#2).** Nothing in the signed payload commits to *which transaction* the votes are about.
  Neither `applyInvalidation` nor `ClusterMember.applyConsensusInvalidation` checks that the proof
  corresponds to the `invalidatedActionId` / `blockIds` it is reverting. A *genuine* `challenger-wins`
  proof for dispute D (about transaction X) can be carried in an `InvalidateRequest` whose
  `invalidatedActionId`/`blockIds` point at an unrelated, innocent transaction Y, and it verifies.
- **Duplication (#3).** The count loop tallies every signature-valid vote with no dedup on
  `arbitratorPeerId`. One genuine `agree-with-challenger` vote, replicated N×, manufactures a
  super-majority.

The existing test `accepts a challenger-wins resolution with a 2/3 super-majority of signed votes`
(`test/invalidation.spec.ts`) is the live reproduction of the broader hole: three unrelated keypairs
over an arbitrary `disputeId`/`computedHash`/`messageHash` return `true`. After this ticket that test
must be updated to construct a *target-bound* proof and verify against the *matching* target; a new
test asserts the same proof verified against a *different* target returns `false`.

`verifyInvalidationCertificate` has exactly two production callers — `applyInvalidation`
(invalidation.ts) and `applyConsensusInvalidation` (cluster-repo.ts) — and **both already hold the
reversal target** (`invalidatedActionId`, `blockIds`, and `proof.messageHash`). So threading the
target into the verifier is a local change with no new plumbing.

## Design

### Wire/format change — versioned, target-bound vote payload (v2)

The signed arbitration-vote payload becomes target-bound and explicitly versioned. Old (v1) votes —
which lack the binding — are **rejected, not accepted-by-default**.

```
targetHash = hashString( `${messageHash}|${invalidatedActionId}|${sortedBlockIds.join(',')}` )
v2 vote signed payload = utf8( `v2:${disputeId}:${vote}:${computedHash}:${targetHash}` )
```

- `messageHash` — already carried on `DisputeResolutionProof`; the original transaction's cluster-record hash.
- `invalidatedActionId` — the committed action being reversed.
- `sortedBlockIds` — the invalidated action's `blockIds`, lexically sorted so the binding is
  order-independent across members.
- `hashString` is the existing db-core helper already used in invalidation.ts (`computeRevertedBlock`),
  so the binding hash is computed the same way on every member.

The arbitrator has all three inputs at vote time: the `DisputeChallenge` carries `originalRecord`
(its `messageHash`, and the committed action's `actionId`/`blockIds` derivable exactly as
`DisputeService.extractInvalidationTarget` already does it). `makeVote` must take the target (or a
precomputed `targetHash`) and sign the v2 payload; `handleChallenge` already builds the target context
and passes it down.

A `version` discriminator must be explicit on the proof/vote shape (a `'v2'` literal field, or a
`v2:` payload prefix that the verifier requires) so a v1 vote cannot silently slip through — and so the
follow-up arbitrator-set ticket can bump to a `'v3'` shape additively. Reject any vote whose version is
absent or unrecognized **before** counting it.

### Verifier signature change

```ts
type CertificateTarget = {
  readonly invalidatedActionId: ActionId;
  readonly blockIds: ReadonlyArray<BlockId>;
};

verifyInvalidationCertificate(proof: DisputeResolutionProof, target: CertificateTarget): Promise<boolean>
```

The verifier recomputes `targetHash` from `proof.messageHash` + `target.invalidatedActionId` +
`sorted(target.blockIds)`, and verifies each vote's signature over the v2 payload built from that
`targetHash`. Because the arbitrators signed the *real* transaction's `targetHash`, feeding any other
target (a replay against Y) makes every signature fail → 0 decisive votes → reject. Both call sites
pass their own request target:

- `applyInvalidation` → `{ invalidatedActionId: params.invalidatedActionId, blockIds: params.blockIds }`
- `applyConsensusInvalidation` → `{ invalidatedActionId: request.invalidatedActionId, blockIds: request.blockIds }`

This *is* the "apply path verifies the binding against the request's target before writing" requirement
— the check is the certificate verification itself, so no compensating revision or log entry is written
for a mismatched target.

### Per-arbitrator dedup (#3)

In the count loop, track `const counted = new Set<string>()` keyed on `arbitratorPeerId`. A signature-
valid vote whose `arbitratorPeerId` is already in `counted` is **skipped** (counted at most once). If
the same arbitrator appears twice with *different* decisive votes (equivocation), treat that arbitrator
as non-decisive — drop both occurrences (do not let an arbitrator be on both sides). Document the chosen
rule in the verifier doc-comment. Dedup happens after signature verification and composes with the
set-membership check the follow-up ticket adds.

### Struct / type changes

- `packages/db-core/src/log/struct.ts` — `ArbitrationVoteProof` gains the explicit `version` marker
  (and any field needed to carry the bound payload, e.g. nothing beyond what's already there if the
  verifier recomputes `targetHash` from `proof.messageHash` + the apply target). Update the
  `DisputeResolutionProof` doc-comment: the `messageHash` + apply-target now bind the votes; keep the
  "cannot forge" language **caveated** (see docs task) — full unforgeability is only restored by the
  follow-up ticket.
- `packages/db-p2p/src/dispute/types.ts` — `ArbitrationVote` gains the matching `version` marker so
  origination and the proof projection stay in lock-step.
- `packages/db-p2p/src/dispute/invalidation.ts` — `toVoteProof` carries the version through;
  `verifyVoteSignature` builds the v2 payload (requires `targetHash` + version); `applyInvalidation`
  passes its target.
- `packages/db-p2p/src/cluster/cluster-repo.ts` — `applyConsensusInvalidation` passes
  `request`-derived target into `verifyInvalidationCertificate`.

### Back-compat

Nothing originates or applies an invalidation end-to-end yet (parent gap #1 wiring is unlanded), so
there are **no persisted v1 proofs to migrate** — back-compat is purely "reject any vote/proof that is
not v2." Make the version gate explicit and tested. The format may be bumped again (additively) by the
follow-up arbitrator-set ticket; that is acceptable pre-launch.

target-bound payload.

- **Replay rejected (#2)** — a genuine `challenger-wins` proof for target X verified against a different

- **Duplicate-arbitrator counted once (#3)** — one genuine `agree-with-challenger` vote replicated 3×
  does **not** reach the super-majority (counts as one); the legitimate 3-distinct-arbitrator case still
  passes.
- **Unversioned/v1 vote rejected** — a vote lacking the v2 version marker is dropped before counting.
