# Wire FRET adapter into RingSelector.determineRing()

description: RingSelector accepts an ArachnodeFretAdapter but never reads it; ring-depth math uses hard-coded constants for "estimated total network demand". Wire the adapter through so ring selection reflects observed network state.

## Context

`RingSelector` (`packages/db-p2p/src/storage/ring-selector.ts`) was created in commit `697c51f` with an `ArachnodeFretAdapter` constructor parameter that was never read. Commit `f5bb326` renamed the field to `_fretAdapter` to silence the unused-variable lint instead of wiring it up. Inside `determineRing()` (lines 47-49):

```ts
// Estimate total network size from FRET
const avgBlockSize = 100 * 1024;     // 100KB typical block
const estimatedTotalBlocks = 1000;   // Conservative estimate
const estimatedTotalData = estimatedTotalBlocks * avgBlockSize;
```

The comment claims "Estimate total network size from FRET" but the code is a stub. As a consequence, every node — regardless of capacity or network conditions — calculates ring depth against the same fictitious 100 MB target. This breaks the promise of `docs/arachnode.md` §"Capacity Management and Ring Adjustment", which calls for sampling overlap and demand across rings to drive ring selection.

The adapter already exposes the methods we need: `getRingStats()`, `getKnownRings()`, `findPeersAtRing()`, `getMyArachnodeInfo()`. The single non-test caller `libp2p-node-base.ts:387` already passes a real adapter — only the consumer is broken.

## Design

`determineRing()` should derive estimated total network demand from observed peer state instead of hard-coded constants.

**Algorithm**: aggregate observed available capacity across all known rings as a proxy for total network capacity:

```
networkCapacityEstimate = Σ over rings: peerCount × avgCapacity
```

`getRingStats()` already returns `{ ringDepth, peerCount, avgCapacity }` per known ring, where `avgCapacity` is the mean *available* bytes for peers at that ring. Summing `peerCount × avgCapacity` gives observed total free byte capacity across the network — a usable proxy for "estimated total data the network is sized to handle". When my own `available` is divided by this sum, `coverage` becomes "my fraction of the network's spare capacity," which translates naturally to ring depth via `-log2(coverage)`. More peers → smaller share → higher ring depth (more specialization). Fewer peers → larger share → lower ring depth.

**Bootstrap path**: when `getRingStats()` returns empty (joining the network, no peers seen yet), fall back to the existing constant-based heuristic so first-boot behavior is deterministic.

**Floor**: keep `Math.max(0.001, coverage)` to bound the log argument, and the final `Math.min(ringDepth, 16)` cap.

**`shouldTransition()`**: it already calls `determineRing()` to compute `currentRingDepth`, so the new logic flows in automatically. No additional FRET sampling needed in this ticket — local capacity pressure (the existing `usedPercent` test) plus the now-network-aware `determineRing()` is sufficient. (Overlap-factor sampling for transition decisions is deeper architecture work tracked separately by `docs/arachnode.md`.)

**Add a doc comment** above the new aggregation pointing at `docs/arachnode.md` §"Capacity Management and Ring Adjustment" so the formula has a paper trail.

## Test expectations (`test/ring-selector.spec.ts`)

The mock `MockFretAdapter` currently returns empty arrays — that case must continue to work (bootstrap path). Add coverage for the populated case:

- `getRingStats` returning a single ring with N peers and known `avgCapacity` → ring depth derived from `myAvailable / (N × avgCapacity)`.
- Multiple rings → sum across rings.
- Verify that when network capacity grows (more peers), ring depth increases for a fixed local capacity.
- Verify that when network capacity shrinks, ring depth decreases.
- Existing `MockFretAdapter` returning empty arrays should still produce the bootstrap-fallback value.
