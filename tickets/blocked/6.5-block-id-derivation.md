## Problem (unchanged from plan)

Block IDs are currently 256-bit cryptographically random values (`TransactorSource.generateId()` → `randomBytes(32)`). Since block IDs determine DHT coordinator assignment (via `blockIdToBytes` → SHA256 → FRET routing), a malicious node can grind IDs to land blocks on coordinators it controls.

## Proposed Approach (unchanged from plan)

```
blockId = hash(allocationSeed || chipCode)
seed_{n+1} = hash(seed_n || serialized(block_n))
seed_0 = hash(collectionId)
```

With `chipCode` from ChipCode (NIST-validated, expiring nonce) stored in the `BlockHeader` for re-derivation by validators.

## Why Blocked

Moving from `plan` to `implement` is premature: the plan enumerates six open questions, each of which requires an architectural decision before code is written. Research below clarifies the questions but does not resolve them — the decisions need user input (or at least explicit acceptance of tradeoffs).

## Open Questions & Tradeoffs

### 1. When is a block "fully constructed" for seed-advance purposes?

**Current reality**: `createBlockHeader()` (`transactor-source.ts:15`) is called synchronously during block *construction*. The resulting block can still be mutated via `tracker.update()` afterwards — both inside the action handler and via `Atomic.commit()` flushing updates at the end of an action. So "fully constructed" is ambiguous.

Options:
- **(A) Advance at `Atomic.commit()` time**, iterating inserts in insertion order. Requires hooking into commit. Loses the ability of subsequent blocks within the same action to use the advanced seed.
- **(B) Advance at `tracker.insert()` time** using the block as-inserted. But later `tracker.update()` calls then wouldn't be included in the hash. Downstream blocks allocated *after* the update still wouldn't know about the update.
- **(C) Defer ID generation entirely** — allocate a placeholder, finalize IDs in a post-pass at `Atomic.commit()` ordered by insertion. Requires rewriting every `createBlockHeader` call-site that currently passes the returned ID into child data structures (B-tree node pointers, chain next/prev, log tail IDs, etc.). Massive churn.

This is the most structurally important question. **Recommendation**: Option (A) with a stable serialization of the insert set at commit-time, accepting that seed evolution happens in "batches" per atomic commit.

### 2. What exactly gets hashed when advancing the seed?

- The block object contains `header.id` — which was just derived from the previous seed. Including it is fine mathematically but redundant.
- Field ordering must be deterministic across replayers. JSON.stringify is not deterministic (no guaranteed key order for non-string keys; varying float reps).
- The block may contain arbitrary application data with its own serialization quirks.

Options:
- Canonical JSON (RFC 8785 / JCS) — adds a dependency and serialization cost.
- CBOR with sorted keys — already available via multiformats? Needs audit.
- Domain-specific: hash a tuple of (header.id, header.type, update-operation-list) — cheaper, still sufficient for unpredictability.

### 3. How is the seed persisted / restored?

- **Option A**: Recompute from log replay on `Collection.createOrOpen()`. Correct-by-construction but expensive for long-lived collections.
- **Option B**: Store current seed in the `CollectionHeader` block. Cheap but requires the header to be written on every action (currently header is only written on creation), or a separate "state" block.
- **Option C**: Store in log entries alongside `ActionId`/`rev`. Natural fit since the seed advances in lockstep with actions.

### 4. Determinism across Quereus + Optimystic phases

Block allocation happens in two phases:
1. **Mutation phase**: action handlers call `tracker.insert()` via atomic (inside `Collection.internalTransact`).
2. **Sync phase**: `Collection.syncInternal()` calls `Log.addActions()` which allocates log blocks.

Questions:
- Is `Log.addActions()` allocation order deterministic? (chain append can trigger header-block rewrites; needs audit)
- If a validator replays only the mutation phase, the seed diverges from the original node's (which also allocated log blocks). Validators must replay both phases in the same order.
- Record iteration order on `Transforms.inserts` relies on JS string-key insertion order; stable but brittle. Canonicalizing (e.g. sorted by insertion sequence tracked separately) is safer.

### 5. ChipCode integration

- ChipCode is not currently a dependency; adding it means cross-platform audit (browser, node, RN per AGENTS.md).
- Policy parameters needed: `ageMs` (expiration), acceptance window on validators, clock-skew tolerance.
- `BlockHeader` currently `{id, type, collectionId}` — adding `chipCode: Uint8Array` inflates every header and every wire message. Tradeoff: only carry chipCode on *insert* messages, not on replay-from-storage; validators verify once at pend-time.
- Bootstrap block (first block in a collection) has known seed `hash(collectionId)`. ChipCode's entropy validation is the only barrier to targeting. Is that sufficient, or do we need commit-reveal (a separate ticket) for just the bootstrap?

### 6. Validation protocol integration

The plan says "validators replay the deterministic transaction." Current validators (`ClusterMember`, `TransactionValidator`) don't replay — they verify signatures, revision monotonicity, and read dependencies. Adding derivation checks means:

- Validators need the seed-at-time-of-this-action to re-derive. Must be readable from prior committed state (see Q3).
- Validators need to re-run the exact serialization used in Q2.
- Failure mode: a block with a mismatched ID. What's the consensus response? Reject the whole transaction? Penalty to the coordinator? (`PenaltyReason` already exists for equivocation — does this warrant a new reason?)
- Does this interact with the cascading-consensus/right-is-right protocol for disputes?

### 7. Migration / backward compatibility

Existing collections have block IDs with no derivation. The allocation-seed scheme only works for *new* allocations. Options:
- Gate by collection version — old collections stay on random IDs.
- Flag-day upgrade requiring all nodes to support the new derivation.

Per AGENTS.md "Don't worry about backwards compatibility yet", this is probably just a matter of deciding the gate mechanism, but it needs a decision.

## What's needed to unblock

A short design session (or written decisions from the owner) on:
1. **Commit-time batch vs. per-insert advance** (Q1 — structural)
2. **Canonical serialization choice** (Q2)
3. **Seed persistence location** (Q3)
4. **ChipCode policy parameters + header-bloat tradeoff** (Q5)
5. **Validator protocol extension shape** (Q6)

Questions 4 and 7 are lower-risk and can be resolved during implementation once 1/2/3/5/6 are settled.

Convert this into a short design doc under `docs/` once the owner has made calls on Q1–Q3.
