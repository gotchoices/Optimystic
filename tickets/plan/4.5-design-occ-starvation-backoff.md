Ship a default retry policy that backs off and ages up priority so big transactions eventually win.

Theorem 9 openly admits there is no deterministic starvation prevention, and the design makes the problem systematically worse for large transactions. Read-dependency capture is block-granular and includes B-tree interior nodes and log-chain nodes, so a transaction that touches more blocks is structurally more likely to observe a stale read and lose a race. Short transactions keep winning; large ones can lose indefinitely — a fairness inversion that is honestly flagged but has no fence primitive built to counter it.

## Expected behavior

Provide a coordinator-side default that gives large transactions a path to eventually commit, rather than deferring fairness entirely to application authors:

- **Exponential backoff with priority aging** as the shipped default retry policy: a transaction that repeatedly loses races backs off, but its effective priority rises with age/attempts so it eventually wins against a stream of fresh short transactions.
- **Consider excluding purely structural reads from the conflict set where safe** — interior B-tree/log-chain nodes read only for navigation, not for their value, need not always participate in conflict detection. Define precisely when dropping a structural read from the conflict set is safe (i.e. does not admit a correctness violation) versus when it must be retained.

## Edge cases & interactions

- **Priority-aging vs. livelock** — aging must actually guarantee eventual progress, not merely reduce probability; define the bound.
- **Structural-read exclusion safety** — dropping interior-node reads from the conflict set must not let a transaction commit against a tree shape that changed underneath it; enumerate the cases where the structural read is load-bearing (e.g. a split/merge that moves the key) and must stay in the set.
- **Fairness vs. throughput** — aging a large transaction up will cause more short transactions to lose; bound the throughput cost.
- **Interaction with hot log-tail** (design-hot-log-tail-sharding-guidance) — both concern the same conflict/`resolveRace` path; keep the backoff policy compatible with any future HLC/crdt-sync redesign.
- **Configurability** — application authors may still tune it, but the default must be safe and fair out of the box.
