## Why this is still a plan, not implement

Ticket-7 Phase 1 already gives us a real-libp2p solo-node cold-start spec. It passes on main. The bugs caught by the sereus-health mobile canary (ticket 5-chain-add) do NOT reproduce in that Node spec, strongly suggesting the residual bug surface is React-Native runtime behavior (async scheduling, persistent storage, app-kill mid-DDL) that a Node CI canary cannot exercise.

So the open question: does a JS-level canary beyond what Phase 1 already provides pay for itself in CI minutes?

## Options

**A. Do nothing.** Phase 1 + Phase 3 (multi-node) + Phase 4 (empty-state) already cover the Node-reachable bug surface. Accept that RN-specific bugs need a device canary, and invest in making the device canary fast/automated upstream instead. Lowest CI cost.

**B. Thin canary — `cadre-smoke.spec.ts` in `quereus-plugin-optimystic`.** A single test that, on real libp2p + real SchemaManager, does the exact sequence `doStart` performs: open schema tree, read schema, store new schema, insert row, restart node (reopen memory storage), read back. Five minutes to write; catches the slice of "whole-stack first-launch" bugs that are Node-reproducible. CI cost: one extra ~10s spec.

**C. Device harness in CI.** Run an Android emulator in CI with the sereus-health bundle. Realistically catches RN bugs. High CI cost; high maintenance burden; cross-repo concern.

## Recommendation (for the next pass to confirm)

Option **B** unless we already have Option A's coverage via Phase 3. The marginal cost of one extra spec that exercises the whole plugin-to-libp2p path with first-launch semantics (including a memory-storage "restart") is low, and it gives us a single go/no-go signal for "did I break the first-launch path" per PR — useful independent of whether it catches the specific RN bug.

If we go with Option B, this ticket becomes an implement ticket for `packages/quereus-plugin-optimystic/test/cadre-smoke.spec.ts`.

Option C is out of scope for this repo and should be filed against the sereus-health repo if pursued.

## Questions to resolve before transitioning to implement

1. Is Phase 3 (multi-node cold-start) already going to cover the "first DDL + DML + restart" path end-to-end via the plugin? If yes, Option B is redundant.
2. Does `MemoryRawStorage` survive a "teardown and reconstruct the stack over the same storage" well enough to simulate a cold-restart? (Phase 1's spec doesn't do this yet.)
3. Who owns keeping the canary's schema-manager sequence in sync with whatever sereus-health actually does? If it drifts, it stops being a canary.

## Unblock path

After Phase 2/3/4 implement tickets land, re-read the resulting coverage and answer Q1–Q3. If Option B is still net-positive, convert this to an implement ticket.
