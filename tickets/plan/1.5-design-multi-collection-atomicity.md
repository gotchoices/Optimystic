description: A transaction that spans several data collections is supposed to be all-or-nothing, but the code commits them one at a time with no way to undo an earlier commit if a later one fails and no durable record to recover from — so a transaction can end up half-applied. Either make it truly all-or-nothing, or change the promise to match what the code does and tell callers when a transaction landed only partially.

----

`correctness.md` Theorem 3 claims multi-collection atomicity, but the implementation does not realize it. `TransactionCoordinator.commitPhase` (`coordinator.ts:732-804`) loops over collections calling `transactor.commit` one at a time. On a mid-loop failure, `cancelPhase` explicitly excludes already-committed collections (`:604-610, 812-821`) — those commits stand. So partial commit is a reachable outcome, not a theoretical edge: commit can genuinely fail after a successful PEND when a racing transaction advances the log tail between prepare and commit.

The recovery state Theorem 3 leans on lives per-*cluster* in `cluster-coordinator.ts` and only resumes the `broadcasting` phase (`:775-805`). There is no cross-collection two-phase-commit journal at all — nothing durably records the coordinator's overall commit/abort decision across the set of collections.

This finding overlaps the Transaction-section review item (referred to there as the transaction-layer 2PC/atomicity fix). This plan ticket is the design home for the atomicity decision; a transaction-layer implementation ticket should depend on this one rather than re-deciding the model.

## Expected behavior

Resolve, in this plan, which of these the system commits to:

- **Real cross-collection two-phase commit.** Give the coordinator a durable decision record (a 2PC journal over the supercluster) so that after a crash or a mid-loop failure the coordinator can drive every collection to the same commit-or-abort outcome, achieving genuine atomicity.
- **Honest downgrade to atomic-intent / eventual-visibility.** Adopt the `crdt-sync.md` framing: the transaction atomically records intent, and collections become visible eventually, with reconciliation. Rewrite Theorem 3 to state this weaker, accurate guarantee.

Either way, and regardless of which is chosen, the failure result must surface `committedCollections` (which collections did land) so callers know reconciliation is required rather than being told the whole transaction failed when part of it committed.

## Edge cases & interactions

- **Racing log-tail advance between PEND and commit** — the concrete trigger for partial commit; the design must handle a commit that fails only for a subset of collections.
- **Coordinator crash mid-commit-loop** — recovery must reach a single global decision; today only the per-cluster broadcasting phase resumes, which does not cover the multi-collection decision.
- **Caller-visible partial state** — define the shape of the failure result and what a caller is expected to do with `committedCollections`.
- **Interaction with cluster membership agreement** (design-cluster-membership-agreement) — a durable 2PC decision record presumes an agreed supercluster; note the dependency if the chosen design needs it.
- **Idempotent replay** — a resumed commit must not double-apply a collection that already committed.
