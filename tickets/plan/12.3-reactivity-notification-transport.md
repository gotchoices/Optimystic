# Reactivity notification transport (production fan-out over libp2p)

## Problem

The reactivity hot path is logic-complete but has **no live transport**. Three seams remain in-process
only:

1. **Origination → subscribers/children fan-out.** `ReactivityOriginationManager` builds a
   `NotificationV1` and hands it to an injected `emit(notification)` callback. Nothing dials a
   subscriber's primary or a child cohort to actually deliver the frame.

2. **Forwarder forward decision → transport.** `createReactivityForwarder.receive` returns
   `"forward" | "duplicate" | "untrusted"` after verify → dedupe → append, but the `"forward"` action
   (fan the unmodified `NotificationV1` to this forwarder's direct subscribers and child cohorts) has no
   wire binding. Inbound notifications are likewise not received from any registered protocol — the
   subscription manager's `onNotification` is only called by tests.

3. **PushState gossip not on the bus.** `PushState.serializeGossip` / `mergeGossip` (replay ring +
   dedupe window + scalars) are complete and tested, but nothing drives them over the host's
   `CohortGossipBus` cadence (`CohortTopicService.cohortGossip()`). So "any cohort member can serve a
   replay" holds in-process but not over the wire.

## What to build

A reactivity application protocol handler mirroring the existing cohort-topic protocol handlers in
`packages/db-p2p/src/cohort-topic/`:

- **Register a libp2p protocol** (e.g. `/optimystic/reactivity/notify/1.0.0`) that frames
  `NotificationV1` with `encodeNotificationV1` / `decodeNotificationV1` (already implemented).
- **Inbound:** on receipt, route to the forwarder receive path (forwarder primary) and/or the
  subscriber delivery path (direct subscriber), per this node's role for the topic.
- **Outbound fan-out:** on origination `emit` and on a forwarder `"forward"` decision, dial each direct
  subscriber's primary and each child cohort (`PushState.childCohorts`) and send the unmodified frame.
  Respect failure isolation (a dead subscriber must not break the fan-out or the commit).
- **PushState gossip:** fold `serializeGossip` / `mergeGossip` into the cohort gossip bus obtained from
  `CohortTopicService.cohortGossip()`, on the same cadence cohort-topic uses for membership-cert gossip.

Decisions to resolve in the plan:

- Push vs. the subscriber pulling; retry/timeout and backpressure interaction with
  `reactivity-rotation-backpressure-policy` (the per-subscriber bounded queue).
- How child-cohort primaries are resolved/dialed (reuse cohort-topic routing / `CohortRef.primary`).
- Composition with `cohort-topic-host-node-wiring` (which makes origination *live* by wiring the
  change-notifier bridge into the production node) — this transport is what that bridge's emitted
  notifications actually travel over.

## Done when (plan)

- Protocol surface, framing, routing, fan-out, retry/failure-isolation, and gossip-cadence integration
  specified; implement ticket(s) emitted.
- Test strategy named: in-process the mock-tier e2e (`reactivity-e2e-mock-tier`) drives this over the
  mock mesh; over real libp2p, `substrate-e2e-real-libp2p-tier`.
