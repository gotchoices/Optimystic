# Reactivity tail-rotation transport + host wiring

## Problem

`reactivity-rotation-backpressure-policy` (11.5) landed the **pure rotation lifecycle logic** in
`packages/db-core/src/reactivity/rotation.ts` and exercised it as logic-only:

- `TailDrainGate` classifies an inbound request as `serve` / `redirect(RotationRedirectV1)` / `drained`.
- `planReRegistration` / `planReRegistrationWave` produce jittered `fireAt` re-registration plans.
- `BlockFillTracker` emits `warmup` / `filling` signals.
- `buildRotationHandoffCheckpoint` / `applyRotationHandoff` migrate the buffer→checkpoint handoff.
- The db-p2p `ReactivitySubscriptionManager` surfaces a `RotationNotice` (with a plan) once per successor
  tail and invalidates the sticky cohort-hint cache.

But **no running node drives any of it.** The sibling notification-transport ticket
(`reactivity-notification-transport`, 12.3) builds the fan-out/forward/gossip protocol and owns the
per-subscriber dequeue→dial delivery loop + its backpressure interaction. The notification-transport ticket
does **not** cover the rotation-specific seams; this ticket owns them.

## What to build (resolve in plan)

- **`RotationRedirectV1` on the wire.** It is a reactivity-specific shape (redirect to a *new topic*, not a
  cohort-topic tier-based `Promoted`) and has **no codec** today. Decide its framing (reuse the reactivity
  protocol surface from 12.3) and how a forwarder/old-tail primary returns it when `TailDrainGate.classify`
  yields `redirect`. Wire the receiving subscriber to honor it (re-register at `newTopicId`).

- **Re-registration timers.** The subscription manager surfaces a `RotationNotice{ plan: { fireAt,
  newTopicId, lastRevision } }` but does **not** schedule the timer or perform the re-register. Specify the
  host seam: schedule at `plan.fireAt`, then register under `plan.newTopicId` carrying `plan.lastRevision`.
  **Load-bearing:** the wave path must bound the new tail's inbound to `cap_promote_fast` (32), so the host
  MUST build the `RejoinJitter` with `capPromote = DEFAULT_CAP_PROMOTE_FAST` — the default
  `createRejoinJitter()` cap is the cohort-failure `cap_promote = 64` and silently overruns the rotation
  burst ceiling. (11.5's `planReRegistrationWave` JSDoc now spells this contract out.)

- **`BlockFillTracker` → origination wiring.** A per-collection `BlockFillTracker` on the tail-cohort
  primary must feed `OriginationCollectionContext.rotationHint` on the `filling` commit (today
  `resolveContext` returns a hand-supplied hint in tests). The `warmup` signal should bias FRET pre-dialing
  toward likely-successor coords (best-effort, signal-only today — confirm acceptable or wire the pre-dial).

- **Drain gate driven from the forwarder host.** Once a live reactivity forwarder host exists (the
  receive/fan-out loop — see 12.3), drive `TailDrainGate` from its inbound classification so the old tail
  bounces new subscriptions while serving renewals/replays for `T_drain`.

- **Handoff ↔ resume coordination.** `applyRotationHandoff` records `PushState.inheritedCheckpoint`, but the
  resume classifier (`classifyResume`/`serveResume`, owned by `reactivity-backfill-resume-checkpoints`) does
  not yet consult it, so a resume whose span crosses a rotation is not served from the inherited checkpoint.
  Likely a small follow-on in that module; decide whether it lands here or there.

## Memory / lifecycle note (from 11.5 review)

`SubscriberBackpressure` has a `remove(subscriberId)` but nothing calls it, and `enqueueForSubscribers` has
no consumer. The dequeue→dial delivery loop (12.3) must also evict a departed/TTL-expired subscriber's
queue, or the per-subscriber map grows unbounded. The documented memory bound
(`cohort_subscribers × queue_max × notification_size`) only holds once that eviction is wired.

  specified; implement ticket(s) emitted.
- Test strategy named (mock-tier e2e `reactivity-e2e-mock-tier` drives an end-to-end rotation; real libp2p
  via `substrate-e2e-real-libp2p-tier`).
