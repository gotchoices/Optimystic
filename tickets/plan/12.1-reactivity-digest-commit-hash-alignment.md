# Reactivity ↔ cluster: align `NotificationV1.digest` with the threshold-signed commit bytes

## Problem

Reactivity reuses the commit certificate's threshold signature **bit-for-bit** — it never re-signs.
A subscriber/forwarder therefore verifies `sig` by recomputing the cohort threshold check over some
payload and comparing to `cert.thresholdSig`. The verifier
(`packages/db-core/src/reactivity/verify.ts`) feeds `payload = b64urlToBytes(n.digest)` into
`MembershipVerifier.verifyMessage(signers, coord, tier, payload, sig)`.

But origination (`buildNotificationV1`, `packages/db-core/src/reactivity/notification.ts`) sets:

```ts
digest: bytesToB64url(utf8.encode(event.actionId))
```

i.e. the UTF-8 bytes of the `actionId` **string**, base64url-encoded. The commit certificate's
threshold signature, however, is computed by the transaction/cluster layer over the **commit hash**
(see `docs/transactions.md`). Unless `utf8(actionId)` is byte-identical to those signed bytes, a real
Ed25519 threshold-verify over `digest` will FAIL. This is why every cryptographic-path test in this
area currently uses a fake `NotificationVerifier` or a pass-crypto stub (`ICohortThresholdCrypto.verify
= () => true`); no test exercises a genuine signature over `digest`.

This is the single most important integration seam in the reactivity hot path: until it is closed,
reactivity cannot be verified end-to-end against real crypto, and the real-libp2p e2e tier
(`substrate-e2e-real-libp2p-tier`) cannot assert authenticity.

## What to decide / specify

The deliverable is a design that makes `digest` carry (or deterministically reproduce) the exact
threshold-signed byte image. Resolve one of:

1. **Expose the signed payload on `CommitCert`.** Add the signed commit-hash bytes (the exact preimage
   the cluster threshold-signed) to the `CommitCert` shape in
   `packages/db-core/src/transactor/change-notifier.ts`, and have `buildNotificationV1` set
   `digest` from those bytes rather than from `actionId`. Cleanest end-to-end; requires the cluster
   commit-cert store/extractor to surface the preimage.

2. **Establish and encode an `actionId == commit-hash` invariant.** If the transaction layer guarantees
   `actionId` *is* the commit hash, specify the exact byte encoding (raw hash bytes, NOT
   `utf8(hexString)`), document the invariant in both `docs/reactivity.md` §Notification origination and
   `docs/transactions.md`, and fix the encoding in `buildNotificationV1` accordingly.

Whichever path: the verifier and origination must agree on the byte image, and the design must specify
how `signers` (peer-id → member-id bytes; the origination `encodeSigner` /
verifier `signersToBytes` loop) interacts with the threshold check over that payload.

## Done when (plan)

- One of the two approaches chosen and justified (coordinate with the transaction-layer owner).
- An implement ticket specifying the `CommitCert`/`buildNotificationV1`/`verify` changes and a test
  that exercises a **real** threshold signature over `digest` (no pass-crypto stub) end to end.
- `docs/reactivity.md` and `docs/transactions.md` updated to state the invariant.

## Notes

- The mock-transport e2e tier (`reactivity-e2e-mock-tier`) does NOT depend on this (it verifies against
  fabricated membership certs). The **real-libp2p** tier (`substrate-e2e-real-libp2p-tier`) does — add a
  `prereq` on this work there when it is scheduled.
