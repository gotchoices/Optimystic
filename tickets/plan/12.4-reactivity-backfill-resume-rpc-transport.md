# Reactivity backfill/resume RPC transport (the pull/recovery direction over libp2p)

## Problem

`reactivity-backfill-resume-checkpoints` (11.5) landed the **logic** of the recovery path but no live
transport for it. The sibling transport ticket `reactivity-notification-transport` (12.3) covers the
**push** direction (origination → subscriber/child fan-out, forwarder forward decision, PushState gossip)
— it does **not** touch backfill/resume. The **pull/recovery** direction is therefore homeless:

- db-core has the pure servers (`serveBackfill`, `serveResume`/`classifyResume`) and the subscriber-side
  drivers (`createBackfillRequester`, `applyResumeReply`, `ReactivitySubscriptionManager.resume()` /
  backfill seam), but they are wired only through **injected** `BackfillTransport` / `ResumeTransport`
  function seams. No libp2p protocol decodes a `BackfillV1` / `ResumeV1` frame off the wire and calls the
  server, and no dial path sends one from a subscriber to its serving cohort.
- **Request signing is injected, not canonical.** `BackfillV1` / `ResumeV1` take a
  `sign(unsigned) => string` hook; there is no canonical signing-payload helper (cf. cohort-topic's
  `registerSigningPayload`) defining the exact bytes signed, and the requests are not wrapped in a
  cohort-topic replay-protection envelope (the doc says replay protection is the cohort-topic layer's
  job — that wrapping is currently unwired). A real transport needs both, or two honest peers will not
  agree on what was signed.

## What to build

A reactivity recovery protocol handler, mirroring the cohort-topic protocol handlers in
`packages/db-p2p/src/cohort-topic/` and composing with the notify protocol from 12.3:

- **Register a request-reply libp2p protocol** (e.g. `/optimystic/reactivity/recover/1.0.0`) framing
  `BackfillV1`/`BackfillReplyV1` and `ResumeV1`/`ResumeReplyV1` with the existing codecs.
- **Inbound (serving cohort):** decode the request, verify the request signature + replay-protection
  envelope, dispatch to `serveBackfill` / `serveResume` over the live `PushState` (ring + rolling
  checkpoint + current tail), and reply. Any cohort member can serve (the ring/checkpoint are gossiped).
- **Outbound (subscriber):** supply the `BackfillTransport` / `ResumeTransport` that dials the serving
  cohort — using the sticky `ReactivityCohortHint` cache for a one-RT re-attach after a flap, falling
  back to a cohort walk when the hint is stale/invalidated. Feed `currentTailId` / `rotationRevision`
  from the rotation lifecycle (`reactivity-rotation-backpressure-policy`).
- **Canonical request signing.** Define and register the `BackfillV1` / `ResumeV1` signing payload
  (canonical byte image over the unsigned fields) and wrap requests in the cohort-topic replay-protection
  envelope, replacing the injected dummy signers used in 11.5's unit tests.

## Notes / context for the implementer

- 11.5 deliberately left the seam shape injectable so the db-core logic stays unit-testable; this ticket
  fills the seams with a real protocol, it does not redesign them.
- The `subscriberCoord` field of `ResumeV1` is a reply-routing hint; the 11.5 manager currently defaults
  it to the collection id as a placeholder. This ticket should source it from the subscriber's real ring
  coordinate.
- The backfill seam in the manager is fire-and-forget (a transport failure surfaces as an unhandled
  rejection). Decide the retry/error policy for the real transport (hint-only path — a bounded retry then
  escalate to resume/chain is reasonable; do not block the commit path).
- Coordinate `currentTailId` sourcing with `reactivity-rotation-backpressure-policy` (the rotation
  lifecycle owns the real current-tail value the `tail_rotated` classification compares against).
