Specify a safe handoff protocol and add damping before any of this is built.

`arachnode.md` describes nodes shifting rings based on capacity and demand, but specifies no protocol for atomic responsibility transfer. When a node moves outward it sheds half its keyspace; if the inward/outward references that should now cover that slice have not loaded it elsewhere, those keys lose a replica. Separately, `ringDepth = ceil(-log2(capacity/demand))` is an undamped reactive control loop — a classic thrashing setup near ring boundaries, where a node oscillates in and out as the ratio hovers around a threshold.

This is mostly not-yet-implemented, so the fix is a design specification to land *before* the migration machinery is built, not a code change to existing behavior.

## Expected behavior

- **Specify a two-phase responsibility handoff:** advertise-new-ring → confirm-replication → release-old-range. A node must not release responsibility for a key range until it has confirmed the range is replicated to whoever now covers it, so no key ever drops below its replication floor during a move.
- **Add hysteresis / damping to the ring-depth signal:** the move decision must not flip on small fluctuations around a boundary — introduce a dead-band, smoothing, or minimum-dwell so a node does not thrash rings.

Deliver this as a written protocol/spec in `arachnode.md` (and any companion design doc) that a later implementation ticket can build from.

## Edge cases & interactions

- **Concurrent moves** — two adjacent nodes moving at once must not both release overlapping ranges believing the other still covers them; define ordering/locking or a safe overlap window.
- **Failure mid-handoff** — a node that crashes between advertise and release must leave the range covered (fail toward keeping the old holder responsible).
- **Boundary thrash** — specify the exact damping parameters or the method for choosing them, and how demand is measured/smoothed.
- **Interaction with cluster membership** (design-cluster-membership-agreement) — a ring shift changes which nodes are responsible for a block; the handoff must produce a membership change the rest of the system agrees on, not a unilateral local move.
- **Replication-floor invariant** — state the invariant explicitly (no key below N replicas at any instant during a move) so the reviewer can test it.
