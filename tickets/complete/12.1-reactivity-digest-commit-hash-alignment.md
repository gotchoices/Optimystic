# Align `NotificationV1.digest` with the threshold-signed commit-vote bytes — COMPLETE

Closes the cluster↔reactivity authenticity seam from `11-reactivity-origination-replay-delivery`.
Before, origination set `digest = b64url(utf8(event.actionId))` — a value the threshold signature was
**never** computed over — so every crypto-path test had to stub `ICohortThresholdCrypto.verify = () => true`.
Now `digest` carries the exact bytes each cohort member signed (`utf8(commitHash + ":approve")`), so the
unchanged `verify.ts` (which recomputes the threshold check over `b64urlToBytes(n.digest)`) succeeds against
**real** Ed25519.

## What landed (implementer summary, verified accurate)

- **`CommitCert.signedPayload: Uint8Array`** (db-core, *required*) — opaque to db-core, minted only by the
  cluster layer.
- **`buildCommitCert(record, minSigs, signedPayload)`** (db-p2p) — takes the preimage as a param; does NOT
  recompute `commitHash` (the canonical-JSON helper is private to `ClusterMember`; duplicating it risks drift).
- **`cluster-repo.ts` commit branch** — computes `computeSigningPayload(await computeCommitHash(record),
  'approve')` then threads it through `captureCommitCert` → `buildCommitCert`. The single `await` lands before
  the synchronous capture, preserving the "cert retained before `commit()` emits its change event" invariant.
- **`buildNotificationV1`** — `digest: bytesToB64url(commitCert.signedPayload)`.
- **`verify.ts` unchanged** — already `payload = b64urlToBytes(n.digest)`; the point of the ticket.

## Review findings

### What was checked
- **Implement diff read first, fresh, before the handoff** (`git show c062235`): all 8 source/doc files + 6
  test files.
- **Crypto chain end-to-end.** Traced `signVote` → `computeSigningPayload(hash,'approve')` (no rejectReason ⇒
  preimage `commitHash + ":approve"`) ⇒ byte-identical to the captured `computeSigningPayload(commitHash,
  'approve')`. `computeCommitHash` (cluster-repo.ts:522) hashes only `messageHash + message + promises` (NOT
  `commits`), so the hash recomputed *after* self adds its own commit equals the hash every signer signed —
  the load-bearing property. `verifyCollectedMultisig` (threshold-crypto.ts:44) splits `thresholdSig` into
  `signers.length × 64` chunks and Ed25519-checks each over `payload` — exactly what `verify.ts` feeds it.
- **Ordering invariant (concern #2).** Confirmed `validateRecord → validateSignatures` (cluster-repo.ts:240,
  487) runs before `handleConsensus → applyConsensusOperation` on both phase paths (`OurCommitNeeded` and
  `Consensus`). The captured preimage therefore derives from an already-validated record. **Residual, by
  design:** `validateSignatures` runs on the *incoming* record, while capture runs on the possibly-*merged*
  `currentRecord`; correctness rests on the protocol's pre-existing "promise set frozen before commit phase"
  invariant (so `computeCommitHash` is stable across the commit phase). This is a pre-existing protocol
  property, not introduced here, and the failure mode is **safe/fail-closed**: if it were ever violated the
  digest would not match the signed image and real crypto would return `"untrusted"`, dropping reactivity back
  to an authoritative read (the hint-only contract). The cluster-consensus spec asserts the 2-peer happy
  path; a >2-peer / merge-heavy path is not directly asserted — acceptable given fail-closed semantics.
- **Migration completeness (concern #3).** Swept every `CommitCert` literal (`notification.spec.ts:20`,
  `change-bridge.spec.ts:61/240/257`, `managers.spec.ts:220`) and every `buildCommitCert` caller (src +
  3 specs) — all carry `signedPayload`. Other `thresholdSig:` literals belong to unrelated types
  (`MembershipCertV1`, `PromotionNoticeV1`/`DemotionNoticeV1`, `assemble()` returns). Since `signedPayload`
  is required and `db-p2p`'s `tsc` build typechecks `test/`, the compiler vouches for full coverage.
- **Integration test honesty (concern #4).** `reactivity-real-crypto.spec.ts` wires the **real**
  `verifyCollectedMultisig` (imported from `threshold-crypto.ts`, no `() => true`); confirmed no stub leaked.
  The cached `MembershipCertV1` with a dummy `thresholdSig` is deliberate and documented in-test — it
  exercises the *notification's* real Ed25519 check, not the cert's self-quorum check (production fetch still
  self-validates).
- **`mergedDigest` ripple (concern #5).** checkpoint.ts: doc wording only; the fold still folds
  `b64urlToBytes(e.payload.digest)`. New digest is per-revision deterministic and cross-member-identical, so
  the fold stays convergent and remains a hint. Doc accurately describes the new digest.
- **Docs.** Read all touched docs against the new reality. `docs/transactions.md` and `docs/reactivity.md`
  §Notification-origination + the wire diagram are accurate.
- **Build + tests (run in full):** `db-core` `yarn build` green; `db-p2p` `yarn build` green (typechecks all
  specs); `db-core` **721 passing**; `db-p2p` **630 passing, 9 pending**, exit 0. (The `parent unreachable`
  console line in `host-antidos-coldstart.spec.ts` is an expected logged error from a failure-handling test —
  it passes.) No pre-existing failures surfaced; `tickets/.pre-existing-error.md` not written. **Lint:** the
  repo configures no eslint (no config, no `lint` script); the strict `tsc` build is the static-analysis gate

### What was found and done (minor — fixed inline)
- **Zero-cost-no-sink regression (was concern #1).** The commit-vote preimage (`sha256` + an extra `await`
  microtask) was computed **unconditionally** on every cluster commit, even on nodes with no reactivity sink
  wired — where `captureCommitCert` then no-ops. This silently contradicted the existing test
  `does not assemble a cert when no sink is wired (zero-cost default)`. **Fixed:** gated the computation +
  capture on `if (this.onCommitCertificate)` in `applyConsensusOperation` (cluster-repo.ts:807), restoring the
  genuine zero-cost default. The `await`-before-synchronous-capture ordering is preserved; the inner guard in
  `captureCommitCert` is kept as defense-in-depth. Re-built and re-ran the full db-p2p suite after the change —
  still 630 passing (incl. the zero-cost test and the capture test).
- **Stale trust-model doc line (was concern #6).** `docs/reactivity.md:149` claimed a subscriber
  "automatically trusts notifications without additional verifiers" — false once this ticket adds a real
  cryptographic verifier (`createNotificationVerifier` / `verify.ts`), and inconsistent with the same file's
  §Authentication (L320). **Fixed:** reworded to state the subscriber threshold-verifies the sig against the
  tail cohort's membership over `b64urlToBytes(digest)` against real Ed25519, while introducing no new signing
  authority beyond the commit certificate.

### Major findings → new tickets
None. No correctness, security, type-safety, resource-cleanup, or error-handling defect rose to a follow-up
