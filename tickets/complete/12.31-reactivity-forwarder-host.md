# Reactivity forwarder host — review complete

## What it is

`ReactivityForwarderHost` (`packages/db-p2p/src/reactivity/forwarder-host.ts`) turns db-core's in-process
forward *decision* (`ReactivityForwarder.receive`) into live fan-out over `ReactivityNotifyTransport`:
`ingest` (receive→forward→fan-out, per-topic serialized, never rejects), `onInbound` (subscriber +
forwarder roles), `fanOut` (per-subscriber bounded queues, lazy GC of departed subscribers, child-cohort
dial), plus the `reactivityNotificationTopicId` and `reactivityDirectSubscribers` adapters. It is an
unwired building block; the live libp2p assembly is `reactivity-notification-transport` (12.33, implement).

## Verdict

Implementation is sound: per-topic serialization is correct (both `.then` arms route through the
throw-isolating `ingestSerialized`, so no rejection leaks and no ring/dedupe interleave), Edge gating and
`null`-caching are right, per-target/local delivery isolation holds, departed-subscriber GC bounds the
queue map, and fan-out forwards the byte-identical (same-reference) frame. Build clean (`tsc`); full
`@optimystic/db-p2p` suite **758 passing, 27 pending, 0 failing** after the review edits (was 757 — one
test added). The `cohort-topic cold-start: parent unreachable` line is a log inside a passing test, not a
failure.

## Review findings

### Checked
- **Per-topic serialization / rejection isolation** — traced `ingest`'s promise-chain tail: both
  fulfilled/rejected arms call `ingestSerialized` (full-body try/catch), so the chain never accumulates a
  rejection and concurrent same-topic ingests stay contiguous (covered by the existing concurrency test).
- **Receive-decision routing** — `"forward"` fans out; `"duplicate"`/`"untrusted"` do not, untrusted never
  reaches the ring (verified against db-core `forwarder.ts`).
- **Edge / subscriber-only gating** — `instantiateForwarderPushState` → `undefined` on Edge, cached as
  `null`; never forwards, still delivers via `deliverLocal`.
- **Failure isolation** — per-target `send` throw/reject swallowed; `deliverLocal` throw swallowed; verifier
  throw resolves `ingest` without fan-out.
- **Memory bound** — `reconcileQueues` reclaims departed subscribers' queues (one round late, as
  documented); map bounded by the live subscriber set.
- **Subscriber dedupe contract** — confirmed db-core `createReactivitySubscriber` dedupes by
  `(collectionId, revision)`, which is what makes the `onInbound` double-delivery idempotent (see minor #1).
- **Docs** — read `docs/reactivity.md` §Propagation; it described the receive/forward/fan-out path with no
  `> **Implemented**` callout for the new host, inconsistent with every sibling section (origination,
  backpressure, policy, resume all carry one). Fixed (minor #2).

### Major (filed / handed off — not fixed in this pass)
- **Subscriber-id / dial-target encoding mismatch (silent no-dial).** Member-id bytes are
  `peerIdToBytes(peerId) = utf8(peerIdString)` (`cohort-topic/peer-codec.ts`). `reactivityDirectSubscribers`
  originally returned `bytesToB64url(participantId)` = base64url(utf8(peerIdString)), but
  `Libp2pReactivityNotifyTransport.send` calls `peerIdFromString(target)` — feeding base64url throws, the
  transport swallows it, and **origination silently never reaches subscribers** (green tests, dead feature).
  The implementer flagged this (their gap #1) but the owning wiring ticket (12.33) only pinned the unrelated
  *tail-bytes→coord* encoding; its step-2 pseudocode (`bytesToB64url(participantId)`) was itself the broken
  path. Disposition:
  - **Fixed at the source (inline, minor-cost):** `reactivityDirectSubscribers` now returns
    `bytesToPeerIdString(participantId)` — the peer-id-string space the only existing transport requires
    (zero transport wrapper). JSDoc updated to state the whole host runs in peer-id-string space.
  - **Handed off (12.33):** added a "Load-bearing constraint: subscriber-id / dial-target encoding" section
    requiring all four seams (`selfPeerId`, `directSubscribers`, `resolveChildPrimary`, `send` target) to be
    peer-id strings, with a mandatory `peerIdFromString` round-trip assertion; fixed the step-2 pseudocode
    to `bytesToPeerIdString(participantId)`.

### Minor (fixed in this pass)
1. **`onInbound` double-delivery contract undocumented.** A co-located subscriber+forwarder invokes
   `deliverLocal` **twice** for one notification (subscriber role + forwarder self-delivery via `fanOut`),
   relying on the manager's `(collectionId, revision)` dedupe. Analysis shows the race is not realistically
   reachable (the forwarder role's own `await receive` lets the first delivery's verify settle before the
   second starts), and the dedupe collapses it. The `deliverLocal` JSDoc said nothing about this. Fixed:
   documented the idempotency requirement; added a test pinning both the raw double-call and that a deduping
   sink yields exactly one delivery.
2. **Docs §Propagation missing the Implemented callout.** Added one for `12.31-reactivity-forwarder-host`,
   matching the file's convention and honestly marking it a not-yet-live building block.

### Minor (documented, not changed — accepted with rationale)
- **Backpressure is a pass-through under the current model.** `fanOut` drains each per-subscriber queue
  inline and fires `send` fire-and-forget, so drop-oldest is not organically reachable in steady state. This
  matches the ticket's pseudo-code; whether genuine in-flight backpressure (a per-subscriber async delivery
  worker) is wanted is a wiring/redesign decision. Handed to 12.33's honest-gaps.
- **Inbound forwarder-role gating is upstream.** `onInbound` runs the forwarder role unconditionally; a Core
  node receiving a notify for a topic it doesn't serve fans out to an empty set (harmless, cached, wasteful).
  Handed to 12.33's honest-gaps (route by served topic, or accept the no-op).
- **`reactivityDirectSubscribers` does not cross-check `collectionId`.** Safe today (one `topicId` ⇒ one
  collection, since tail block ids are unique per collection). Add a `collectionId` filter only if that
  invariant loosens.
- **Per-topic `served` / `ingestTails` maps never evict a left cohort.** Bounded by the node's served-topic
  count (itself bounded by `topics_max`), but a topic whose cohort the node leaves after a tail rotation is
  not reclaimed. Negligible today; worth a glance when rotation-driven topic migration lands.
- **Child fan-out unexercised end-to-end** until `cohort-topic-parent-child-link` (backlog) populates
  `childCohorts`; dial path is correct and covered by the synthetic test. Already tracked.

### Empty categories
- **New fix/plan/backlog tickets filed: none.** The one major finding was best closed at its source (the
  adapter, inline) plus a constraint handed to the already-existing owning ticket 12.33 (in implement/, not
  yet built, prereq on this ticket) — filing a separate ticket would have duplicated 12.33's scope. The
  remaining items are minor and either fixed inline or documented with rationale.
