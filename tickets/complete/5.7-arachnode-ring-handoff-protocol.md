# What shipped

The **advertise → confirm-replication → release** ring handoff (`docs/arachnode-ring-handoff.md`
§ Part 2/3). A ring shift (or a lost-block rebalance) can no longer drop a key below its replication
floor `N`: a node keeps serving a range until it has confirmed ≥ N *other* holders have a current
replica, then releases. See the implement handoff (git `c563f9a`) for the full mechanism
description — `RingShiftCoordinator` three-phase state machine, `BlockTransferCoordinator.confirmReplicated`
primitive, `arachnode-partition.ts` shared responsibility predicates, gated rebalance release,
`RebalanceEvent.floor`, `reconcileOnStart`, and `recordShiftSettled`.

**Correctness (major, fixed inline — was a latent floor-violation bug)**
- `RingShiftCoordinator.rollback(undefined)` did a bare `setStatus('active')`. By that point Phase A
  has already replaced self with the `moving` **target** (narrower ring/partition + `moveFrom`), so
  flipping only the status left the node advertising **active at the aborted narrower ring** — i.e.
  it stopped claiming coverage of the shed range while never confirming any replica landed elsewhere.
  That is exactly the replication-floor violation the handoff exists to prevent. Reachable, not
  merely dormant: `RingSelector.shouldTransition` explicitly tolerates a node with no advertised
  `ArachnodeInfo` (`info?.ringDepth ?? …`, "production always advertises at bootstrap" is a *should*,
  not a guarantee), so a no-prior-info node can trigger a move-out and hit this rollback. **Fixed** to
  restore a clean active ring-0 (whole-keyspace) advertisement — no stray target partition, no
  `moveFrom`. Added regression test `rolls back to a CLEAN active ring-0 advertisement when the node
  had no prior info` asserting active/ring-0/no-partition/no-moveFrom and that the node is still a
  serving holder of the shed key after rollback.

**Config semantics (minor, noted — tripwire NOTE added, no behavior change)**
- `handleRebalanceEvent` now routes the lost path through `confirmReplicated`, which — unlike the
  legacy `pushBlocks` — does **not** honor `enablePush`. So `enablePush:false` no longer suppresses
  network pushes on the rebalance/handoff release path. This is deliberate: confirmation
  fundamentally must push to verify replication, and honoring the flag would leave every block
  unconfirmed → never released → the node never sheds. (The pre-existing code released synchronously
  regardless of the flag, so neither old nor new honored it for *release*.) Left as-is; recorded a
  `NOTE:` at the `confirmReplicated` site so a future reader meets it — if a config ever needs "never
  move data at all", gate the release *wiring*, not this primitive.

**Verified, no action**
- `confirmReplicated` in-flight filter (`!inFlight.has('confirm:'+id)`) silently drops a
  concurrently-confirming block from *both* result arrays. Both consumers stay safe by construction:
  ring-shift's `confirmShedRange` requires `confirmed.length === shed.length` → a dropped block forces
  rollback (range retained); the rebalance handler only untracks `released`, so a dropped block stays
  tracked/served. No floor violation — conservative by design.
- Confirm counting: counts DISTINCT owners (a `Set`) reporting the block *not* `missing`, stops at
  `floor`, retries the whole round up to `maxRetries`. Correct. rev-1 idempotent push
  (`saveReplicatedBlock` never downgrades) trusted per implement note — matches the existing
  `pushBlocks` path.
- Making `RebalanceEvent.floor` required did not break other event constructions: `rebalance-reaction`
  / `rebalance-monitor-node-wiring` specs build events from the live monitor (which now sets `floor`);
  `spread-on-churn` only *reads* `triggeredAt`. `tsc` confirms.
- `computeShedBlocks` / `RingShiftCoordinator` async iteration over the shared live `ownedBlocks` set:
  concurrent mutation can miss/include a block, but every outcome is conservative (missed shed block →
  not released, stays served; since-removed block → no local data → rollback). Same pattern the
  implementer already flagged for the rebalance eviction path. No action.
