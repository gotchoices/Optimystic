## Summary of completed work

**Problem.** On a solo node (empty bootstrap, no listen addrs) the first DDL —
pend of the `optimystic/schema` block — surfaced as:
```
Some peers did not complete: self[block:optimystic/schema](in-flight); root: No coordinator available for key (all candidates excluded)
```
The batch-retry path ran `findCoordinator` with self excluded, which threw
"all candidates excluded" on a solo node, overwriting the actual first-attempt
cause. Callers saw a misleading message and could not diagnose the real
failure (or, in the mobile bootstrap case, that no real failure existed and
the retry itself was nonsensical).

**Fix — three coordinated changes.**

1. **Preserve the original first-attempt error**
   (`batch-coordinator.processBatches`). Retry setup
   (`createBatchesForPayload` + recursive `processSet`) is wrapped in an inner
   `try/catch`; the outer `.catch(async e => …)` always rethrows the original
   `e`, so `batch.request.error` carries the true root cause.

2. **Distinguishable error codes from `findCoordinator`**
   (`Libp2pKeyPeerNetwork`). New `FindCoordinatorError` with
   `.code ∈ { SELF_COORDINATION_BLOCKED, SELF_COORDINATION_EXHAUSTED,
   NO_COORDINATOR_AVAILABLE }`. On HWM ≤ 1 with self excluded, throws
   `SELF_COORDINATION_EXHAUSTED` — a clear signal that the prior attempt's
   error is authoritative.

3. **Aggregate error diagnostics** (`NetworkTransactor.get / pend /
   commitBlocks`). New `firstBatchError()` helper returns the first per-batch
   error across the batch tree (root first, retries next). Aggregate messages
   prefer that over any outer retry error, and batch detail lines now include
   `cause=<message>` for failed batches. Callers always see the real reason.

4. **Restoration-callback self-skip**
   (`RestorationCoordinator`). Accepts optional `selfPeerId`, filters self out
   of ring-peer candidates so it doesn't dial itself (which hangs on nodes
   with no listen addrs). Short-circuits with `undefined` when no non-self
   peer and no inner rings remain — the correct "block not present" outcome
   for a brand-new solo node. Wired at construction in
   `libp2p-node-base.ts` via `node.peerId.toString()`.

## Usage

For a solo bootstrap / mobile node:
```ts
createLibp2pNode({
  bootstrapNodes: [],
  listenAddrs: [],
  transports: [...],
  clusterSize: 1,
})
```
Then normal `coordinatorRepo.pend` / `commit` / `get` calls work without
hanging. The `peerCount<=1` short-circuit in `CoordinatorRepo` routes through
`StorageRepo` directly; `RestorationCoordinator` short-circuits to
`undefined` for any missing block. As remote peers are discovered, FRET
populates, HWM grows, and the existing cluster-consensus code paths take
over.

If a real underlying pend/commit error occurs (e.g. transform error, disk
full, storage bug), the aggregate now surfaces it clearly:
```
Some peers did not complete: self[block:...](in-flight) cause=<actual message>; root: <actual message>
```
…rather than the old misleading `"all candidates excluded"`.
