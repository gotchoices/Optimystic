# What Was Built

Changed `ClusterMember.getTransactionPhase()` from unanimity-based promise resolution to threshold-based (super-majority) resolution, aligning member-side logic with the coordinator's existing super-majority tolerance.

## Key Changes

- **ClusterRecord type** (structs.ts): Added `disputed?: boolean` and `disputeEvidence` fields
- **ClusterMember** (cluster-repo.ts): `getTransactionPhase()` uses configurable `superMajorityThreshold` (default 1.0 = unanimity for backward compat). Commits when `approvedPromises >= superMajority`; rejects when `rejections > maxAllowedRejections`
- **ClusterCoordinator** (cluster-coordinator.ts): Sets `disputed: true` with evidence when minority rejects but super-majority approves
- **Wiring** (libp2p-node-base.ts): Single `consensusConfig` object shared by member and coordinator (DRY)
- **Mesh harness**: Passes `consensusConfig` to cluster members for integration tests

## Testing

4 new tests in `threshold-based promise resolution` suite:
- Minority rejection (1/5) allows commit (threshold 0.75, superMajority=4)
- Boundary rejection (2/5) blocks commit (exceeds maxAllowedRejections=1)
- Default no-config preserves unanimity (backward compatible)
- Disputed record evidence shape verification

Full suite: db-core 261 passing, db-p2p 268 passing.

## Usage

- Pass `consensusConfig` with `superMajorityThreshold < 1.0` to enable threshold-based resolution
- Production default is 0.67 (shared between member and coordinator)
- Check `record.disputed` after consensus to detect minority-objection transactions
- Dispute evidence is recorded but not yet acted on — the dispute protocol (separate ticket) consumes it
