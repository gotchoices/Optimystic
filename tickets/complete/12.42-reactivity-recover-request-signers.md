Adversarial review of the **signing seam** sub-deliverable split out of `reactivity-recover-node-wiring`
(12.42). The node-composition + real-socket integration work remains in `implement/`
(`reactivity-recover-node-wiring`), genuinely gated on `reactivity-notification-transport` (12.33). This

## What was reviewed (the diff under review)

Implement commit `feb39cd` ("ticket(implement): reactivity-recover-node-wiring"):

- **`signPeerSig(privateKey, payload): Uint8Array`** — `packages/db-p2p/src/cohort-topic/peer-sig.ts`. The
  synchronous mirror of async `signPeer`: reads the libp2p Ed25519 `raw` key (64 bytes = 32-byte seed ‖

- **Doc**: `docs/reactivity.md` §Authentication — synchronous subscriber-side signing bullet.

## Outcome

**Confirmed correct, well-tested, well-documented. No major findings; no minor fixes required.** Build
clean, all targeted tests green. The design decision (synchronous noble sign over the node's raw Ed25519
seed, keeping the `(unsigned) => string` seam unchanged) holds, and is in fact *more* robust than the
handoff framed it (see "load-bearing claim", below).

## Review findings

### Correctness / design (checked — sound)

- **Seed extraction (`raw.subarray(0, 32)`).** Verified empirically, not just by inspection. Traced libp2p's
  `@libp2p/crypto` Ed25519 backend: in **Node** (the test + server runtime), `privateKey.sign` →
  `crypto.sign(null, msg, keyObject)` — **Node native OpenSSL Ed25519**, an implementation entirely
  independent of noble. The byte-identity test (`signPeerSig` vs `signPeer` over the same 64-byte
  `generateKeyPair('Ed25519')` key) therefore genuinely pins **noble == OpenSSL** for the same seed +
  payload. Were the seed slice wrong, noble would sign over garbage and the bytes would diverge from
  OpenSSL's — the test would fail. It passes ⇒ the `raw` layout is confirmed seed‖pub (seed-first) for the
  libp2p version in use, and the slice is correct. Not a tautology; not a value re-compared to itself.
- **Load-bearing claim is actually over-satisfied.** The handoff rests correctness on "noble's sync sign is
  byte-identical to libp2p's async sign (RFC8032 determinism)." True — but the **production recover path is
  noble on both ends**: `signPeerSig` (sign) → `verifyPeerSig` (verify), both `@noble/curves/ed25519`. So
  the recover round-trip does **not** even depend on noble↔OpenSSL determinism; the cross-impl byte-identity
  is bonus assurance that a noble-signed request is also accepted by any OpenSSL/WebCrypto verifier (and
  vice versa). The integration tests (`signBackfill`/`signResume` → real `createRecoverRequestHandler` →
  `verifyPeerSig`) exercise the true production path directly. Robustness margin, not fragility.
- **base64url consistency.** Signer `bytesToB64url`(signature); handler `b64urlToBytes` before
  `verifyPeerSig` — both the same db-core codec. Signing payload is the *same* db-core function
  (`backfillSigningPayload` / `resumeSigningPayload`) on both sides — no field-ordering or canonicalization
  drift possible. Confirmed by reading the db-core payload builders (explicitly-ordered, type-tagged JSON
  array; backfill/resume images can't collide — leading type tag differs).
- **Throw vs. total asymmetry (intended).** `signPeerSig` throws on non-Ed25519 / bad raw length;
  `verifyPeerSig` is total (`false`, never throws). Confirmed the throw path is only reachable from the
  node's **own** key (the gated subscribe factory feeds `nodePrivateKey`), never from wire input — so the
  asymmetry is right: fail-fast on a node misconfiguration, fail-soft on hostile wire input.
- **No aliasing hazard.** `subarray` returns a view; noble's `sign` reads it without mutation.

### Observations (not findings — pre-existing, no action this ticket)

- **Phantom `@noble/curves` dependency.** `peer-sig.ts` imports `@noble/curves/ed25519.js`, which is **not**
  declared in `packages/db-p2p/package.json` — it resolves transitively via `@libp2p/crypto` (a direct dep).
  This predates the ticket: the pre-existing `verifyPeerSig` already imports it, and the whole cohort-topic
  substrate relies on the same transitive resolution. `signPeerSig` adds a second import site but **no new
  dependency surface**. Left as-is — declaring it directly would ripple `yarn.lock` and is package-wide
  hygiene unrelated to this seam, not a regression introduced here.

### Honest gaps (confirmed genuinely deferred, not overlooked)

- **No live-node coverage.** Tests exercise the signer against the **in-process** handler only. The
  end-to-end "signed request over a real socket" path is the env-gated integration test owned by
  `reactivity-recover-node-wiring`, which cannot land until `reactivity-notification-transport` (12.33)
  provides the node wiring. Correctly out of scope here.
- **Ed25519-only.** A node configured with a non-Ed25519 key would throw at first recover-request sign. This
  matches the cohort-topic substrate's standing assumption (`verifyPeerSig` is also Ed25519-only). Unguarded
  at the node level, but consistent with the substrate; flagged in the gated ticket already.
- **Consuming call-site not yet present.** `createRecoverRequestSigners(nodePrivateKey)` inside the
  subscribe factory is described in `reactivity-recover-node-wiring` but not yet in the tree — expected, it
  rides 12.33.
