# Complete: local change-notifier → cohort-topic origination bridge

Connects the existing local single-node change-notification primitive (`StorageRepo` →
`IBlockChangeNotifier`) to the networked cohort-topic substrate. A commit landing on a node that is a
cohort member for the collection's reactivity topic is emitted into `CohortTopicService.onLocalCommit`,
carrying the cluster commit certificate **forwarded unchanged** so reactivity (next ticket) fans out
without re-signing. Implementation summary is in the prior review ticket / the implement commit
(`62365270`); this file records the review disposition.

### Correctness — verified, no defects requiring inline fix
- **Cert captured before commit.** `captureCommitCert` fires in `applyConsensusOperation` *before*
  `storageRepo.commit`, which emits the `CollectionChangeEvent` synchronously at the end of the call —
  so the cert is retained when the bridge's extractor resolves it. The added 2-peer consensus test
  proves `record.commits` is fully populated at capture (128-byte blob = two concatenated 64-byte
  Ed25519 commit sigs, both signers, `minSigs=2`).
- **Emit restructure is behavior-preserving** for per-collection subscribers: an event is now also
  built when only a catch-all listener exists, but per-collection dispatch is byte-identical (shared
  `fireChangeListeners` helper, snapshot-iterated, throws isolated + logged). Unchanged storage-repo
  spec still green.
- **Bridge is a pure pass-through + safe decorator.** `onCollectionChange` delegates straight to the
  source (reactive-watch path intact); origination runs on the catch-all feed; member-gate → hook →
  cert-extract ordering means no extract cost when no hook is attached; a throwing hook is isolated so
  a commit can never break. All exercised by the new spec (member/non-member, byte-for-byte forward,
  throwing hook, no-cert skip, no-hook skip, decorator subscribe/unsubscribe, end-to-end stored cert).
- **`CommitCertStore`** TTL+cap eviction and `buildCommitCert` (approve-only, ascending-signer sort,
  byte-exact concat, reject excluded) are unit-tested and correct. `IBlockChangeNotifier` surface is
  only `onCollectionChange`, so the returned notifier is complete. Removed `LocalCommitChange`
  placeholder confirmed to have zero remaining references.

### Minor observations — acceptable as-is, no change made
- `selfIsCohortMember` is evaluated on every commit even before any `onLocalCommit` hook is attached.
  It's a cheap injected predicate and the hook is effectively always attached once reactivity is live;
  not worth reordering.
- `CommitCertStore.put` runs an O(n) `dropExpired` sweep per insert, but n is bounded (`MAX_ENTRIES`
  4096, 60 s TTL keeps it far smaller in practice) — negligible.
- Certs captured for a commit that subsequently *diverges* (behind/ahead, no local emit) are orphaned
  in the store and TTL-expire harmlessly; the store is keyed by `actionId`, so a mismatched actionId
  simply yields no cert → origination skipped.
- `minSigs` is the **cluster** super-majority (`ceil(peers × superMajorityThreshold)`), which is a
  different threshold concept from the cohort `k − x`; the field doc conflates them. The cert bytes are
  forwarded unchanged, so this is purely a verification-payload concern for the **consumer** — see
  major finding below; not a defect in this origination ticket.

### Major findings → new ticket filed
The bridge mechanism + both seams are complete and tested, but the implementer **deliberately and
correctly** left production activation dormant (the cohort host is constructed post-node, so wiring
can't live in `createLibp2pNodeBase`). That leaves three deferred items (the implementer's honest gaps
#1, #2, #5) with **no owning ticket**: constructing the `CohortTopicService` + `CommitCertStore` in the
production node and calling `attachCohortChangeBridge`, the concrete `selfIsCohortMember` predicate,
and the bridge/catch-all teardown decision. Filed **`tickets/plan/13.5-cohort-topic-host-node-wiring.md`**
(prereq: `reactivity-origination-replay-delivery`) to own that activation before the real-libp2p e2e
tier (`14-substrate-e2e-real-libp2p-tier`) exercises reactivity end-to-end.

The remaining honest gaps (#3 cert-verify byte/payload reconciliation, #4 capture-on-every-peer +
store sizing) are owned by the existing consumer ticket **`11-reactivity-origination-replay-delivery`**
(already in `implement/`, lists `change-notifier.ts` + `cluster-repo.ts`), which sets `onLocalCommit`
and builds `NotificationV1` from `(event, commitCert)` — the correct place to reconcile the verify
payload. No additional ticket needed for those.
