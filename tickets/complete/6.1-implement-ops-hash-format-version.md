# Complete: pin and version the operations-hash byte format

## What shipped

The **operations hash** — the fingerprint one node sends and every other node
recomputes to confirm they agree on exactly what a transaction changes — now carries
a format-version token: `hashOperations` emits `ops.v1:<hash>` instead of the bare
`ops:<hash>`. The hashed body is unchanged (sort by `(collectionId, blockId, type)`,
`canonicalStringify`, SHA-256/base64url); only the prefix moved.

- `OPS_HASH_VERSION = 'v1'`, `OPS_HASH_PREFIX = 'ops.v1:'` — single source of truth.
- `opsHashVersion(token)` — total parser returning the `vN` segment or `null` for
  legacy/garbage/non-string tokens.
- `canonicalOperationsPayload(operations)` — the exact preimage string (no token),
  exposed for the future client-signature ticket.
- `validator.ts` step 9 splits a hash disagreement: a foreign/legacy/unparseable
  version → distinct `Unsupported operations-hash format version: local=v1, sender=<vN|unrecognized>`;
  same version, different bytes → the unchanged `Operations hash mismatch`.
- `docs/transactions.md` gained a normative "Operations Hash — Canonical
  Serialization" section; the two stale illustrative `hashOperations` sketches were
  replaced with pointers to the real module.

## Review findings

Adversarial pass over the implement diff (commit `169dd94`) from every angle — SPP,
DRY, modularity, error handling, type safety, resource cleanup, docs accuracy.

**Checked — clean:**

- **Centralization / DRY.** Grepped every producer and consumer of the ops-hash on
  the wire (`hashOperations`, `operationsHash`, `opsHashVersion`, literal `ops:`).
  Only the coordinator and validator produce the token via `hashOperations`; only the
  validator parses the version. No stray `ops:` literal parses the token anywhere in
  `src/`. The prefix change is fully centralized — no drift surface.
- **Parser totality.** `opsHashVersion` handles `ops.v1:`→`v1`, `ops.v2:`→`v2`, bare
  `ops:`→`null`, and `''`/`'ops.'`/`'ops.:x'`/`'ops.v1'`(no colon)/non-strings→`null`.
  The `.` and `:` delimiters are both outside the base64url alphabet the hash uses, so
  the version segment always slices cleanly. A `null`/foreign version routes to the
  version-skew branch, never to `valid`. Verified by unit tests and now by validator
  integration tests (below).
- **`transaction.id` did not move.** The version token lives only in the ops-hash
  string; `createTransactionId` (persisted identity) is byte-identical for fixed
  inputs — asserted in `operations-hash.spec.ts`, and confirmed no coordinator/log path
  substitutes the ops-hash where the tx id belongs.
- **Payload derivability.** `canonicalOperationsPayload(ops)` + `OPS_HASH_PREFIX` +
  `hashString` reconstructs `hashOperations(ops)` exactly and carries no token —
  asserted.
- **Docs accuracy.** Read every touched file against the docs; the normative section
  and the validator pseudocode split match the shipped code.

**Found and fixed inline (minor — test-coverage gap):**

- The new step-9 branch in `validator.ts` (version-skew vs. content-mismatch) had
  **zero validator-level coverage** — the only tests were the `opsHashVersion` unit and
  happy-path `valid: true` validations. The wire-facing error strings a peer actually
  receives were unpinned. Added three tests to `transaction.spec.ts` driving
  `validator.validate()` end-to-end: (1) same-version token with wrong bytes →
  `reason === 'Operations hash mismatch'`; (2) foreign `ops.v2:` token → version-skew
  error naming `local=v1`, `sender=v2`; (3) bare legacy `ops:` token → version-skew
  error with `sender=unrecognized`. All pass.

**Tripwires (parked, not ticketed — index only; analysis lives at the site):**

- *Detect-not-cross-compute* boundary and the *no version negotiation* residual: a
  node detects a foreign version but cannot reproduce another version's bytes; mixed
  clusters fail legibly, not interoperably. Parked as the `NOTE`-style paragraph in the
  `hashOperations` doc comment (`operations-hash.ts`) and the docs' normative section.
  Re-picked here if rolling upgrades ship.
- *No version whitelist*: `opsHashVersion` returns any well-formed `vN` without checking
  a registry of known versions; the validator only tests equality with the local
  version. Fine while only `v1` exists. Parked under the "No version whitelist" note.

**Major findings:** none — no new tickets filed.
