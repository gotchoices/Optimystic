# Complete: cohort-topic package layering (pure core vs. transport binding)

Established and reviewed the single-source-of-truth layering rule for the cohort-topic substrate:
pure logic in `db-core` behind ports, FRET/libp2p implementations in `db-p2p`, enforced by a
db-core test guard. Every downstream substrate ticket prereqs this.

## What landed (as implemented + review fixes)

- **db-core ports** (`cohort-topic/ports.ts`): `ITopicRouter`, `ICohortGossipTransport`,
  `IMembershipSource`, `ISizeEstimator`, `IRingHash`, plus db-core-owned `RingCoord` (`Uint8Array`)
  and `PeerRef`. Interfaces only, no transport import.
- **db-core ring hash** (`cohort-topic/ring-hash.ts`): `RingHash` / `createRingHash` over
  `@noble/hashes/sha2`. At `RING_BITS=256` returns the full digest (byte-identical to a raw
  SHA-256); sub-byte widths truncate to `ceil(ringBits/8)` bytes and mask the trailing partial
  byte. Range-validated `[1,256]`.
- **db-p2p stubs**: `Fret{TopicRouter,CohortGossipTransport,MembershipSource,SizeEstimator}`, each
  implementing its db-core port and throwing `notWiredToFret(...)` — honest placeholders pointing
  at the integration ticket.
- **Guard** (`db-core/test/no-fret-import.spec.ts`): walks `db-core/src/**` and fails on any
  import/export/dynamic-import resolving to `p2p-fret`, `libp2p`, or `@libp2p/*`.
- **Docs**: architecture.md §Package Layering, internals.md §Cohort-Topic Port Boundary,
  cohort-topic.md §FRET integration pointer.

## Review findings

**Scope checked:** the full implement diff (`cec2464`) read first with fresh eyes; ring-hash
masking math at boundary widths; the guard regex against every import form; stub honesty; port
signature sufficiency vs. the consumer tickets; db-core/db-p2p builds; full db-core test suite;
accuracy of all three touched docs; layering (`db-core/src/**` confirmed FRET/libp2p-free at HEAD).

### Major (new tickets)
None. The split is sound, the stubs are honest, the ring-hash logic is correct, and the docs match
the code.

### Minor (fixed inline this pass)
- **Guard missed bare side-effect imports.** The regex required `from`, so `import 'libp2p';` —
  the *exact* case the ticket claims the guard catches — slipped through (verified: it returned no
  match). Added a third alternative for `import '<spec>';` and refactored extraction into an
  `importSpecifiers()` helper. `import.meta.url` (used by the spec itself) correctly does not match.
- **Guard was not self-validating.** The original spec only asserted "no violations exist today" —
  it never proved the detector actually fires, so it could silently rot into a no-op. Added two
  cases: one asserting every forbidden import form (named, side-effect, type-only, `export … from`,
  dynamic, multi-line) is flagged; one asserting allowed specifiers and `import.meta.url` are not.
  Documented in-test that the regex is intentionally not comment-aware (over-matches a commented
  import → safe direction, false-positive never false-negative).

### Routed to an existing downstream ticket
- **FRET byte-compatibility is asserted by design, not by a test** (implementer's known gap).
  db-core can't import FRET, so the `RingHash.H(x) === FRET hashKey(x)` byte-for-byte assertion has
  no home here. Added it to Phase 3 of `10-cohort-topic-core-module-fret-integration` (the package
  allowed to import both). Not a new ticket — folded into the work that will create the seam.

### Verified, no change needed
- **Ring-hash math:** `tailMask` correct at 256 (→0xff, full-digest fast path), 64 (byte-aligned),
  12 (0xf0 partial-byte), 1 (0x80); full-digest path returns a fresh per-call array (noble v2),
  no aliasing. Range validation rejects 0/257/1.5.
- **Stub honesty:** every adapter method throws `notWiredToFret` with a clear pointer; no silent
  no-ops; unused args `_`-prefixed per AGENTS.md.
- **Port sufficiency:** signatures (encoded `Uint8Array` in/out, `{nEst,confidence}`, unsubscribe
  handle on `onMessage`) match what the addressing / gossip-cert / walk-promotion consumers need
  per the design (merge/verify stay in db-core over the ports).
- **Type safety / layering:** no `any`; `import type` for db-core types in db-p2p; non-null
  assertions justified; `db-core/src/**` is FRET/libp2p-free at HEAD (the `@libp2p/*` entries in
  db-core are devDependencies only, outside the guard's `src` scope as specified).
- **Docs:** architecture.md / internals.md / cohort-topic.md accurately describe the split, the
  five ports, the guard path, and the db-core-owned SHA-256.
