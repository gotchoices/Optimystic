# Complete: reactivity change-event tail + bridge membership-gate seam

The plumbing seam that lets the cohort-topic reactivity bridge derive the rotating reactivity topic
`H(tailId ‖ "reactivity")` from a commit. `CollectionChangeEvent` now carries the committed collection's
chain `tailId`; `NetworkTransactor` threads that tail through every per-block commit so it reaches each
committing node's `StorageRepo.commit`, which stamps it onto the emitted event. The cohort-topic bridge
gates on the **whole event** (so the node-activation predicate can read `event.tailId`) and
`attachCohortChangeBridge` returns `{ notifier, unsubscribe }` so the node releases the catch-all
origination feed on stop. Implemented alongside 13.6 (`cohort-topic-host-node-activation`).

## What the seam delivers (verified end-to-end)

The tail flows: `Coordinator` builds `CommitRequest.tailId = logTailBlockId` →
`NetworkTransactor.commit` threads `request.tailId` into `commitBlock`/`commitBlocks` → each per-block
`getRepo(peer).commit({ actionId, blockIds, rev, tailId })` → wire/consensus carries it
(`{ commit: request }`, `RepoService` → `CoordinatorRepo`/`ClusterRepo`) → committing node's
`StorageRepo.commit` reads `request.tailId` and emits `CollectionChangeEvent` carrying it → bridge gate
reads `event.tailId`. Read-driven promotions (the `get` path) have no commit request → `tailId` is
`undefined`, and those never originate (cert-gated downstream). `tailId?` is optional at every layer, so
bare per-block paths (churn replication, block-transfer) keep emitting `tailId: undefined` and never
originate.

## Review findings

Adversarial pass over commit `8f2eac9` (the implement commit; the 13.5 half was not separately
committed). Reviewed the diff with fresh eyes before the handoff, traced the full tailId data-flow, and
checked backward-compat, serialization, multi-block/multi-collection commits, the bridge teardown
contract, docs, build, and tests.

**Checked & clean:**
- **Type soundness / backward compat.** `RepoCommitRequest.tailId?` and `CollectionChangeEvent.tailId?`
  are optional everywhere; no caller now *requires* a tail. `IRepo.commit(RepoCommitRequest)` vs
  `StorageRepo.commit(CommitRequest)` is the pre-existing (bivariant) structural relationship — the only
  field newly read at the emit site (`request.tailId`) is present on every commit-path request and safely
  `undefined` on read-driven promotions. Verified `db-core` + `db-p2p` + `quereus-optimystic` all `tsc`
  build clean against the changed types.
- **Tail-flow correctness.** Confirmed the consensus apply path (`cluster-repo.ts` →
  `storageRepo.commit(commit)`) and the direct path (`coordinator-repo.ts`) both forward the
  tailId-bearing request; the FRET coordinator-key recording (`repo/client.ts`
  `extractKeyFromOperations`, `cluster/client.ts`) now receives a real tail rather than `encode(undefined)`
  → empty bytes — a latent improvement, as the handoff noted. Not changed further (out of seam scope).
- **Multi-block / multi-collection.** One `NetworkTransactor.commit` is one collection's chain in
  practice; every per-block commit in it carries the same top-level `request.tailId`. A delete commit
  (materialized block undefined) still gets the tail from `request.tailId`, not the block. `StorageRepo`
  groups by `collectionId` and stamps the same tail on each — documented and correct for the
  single-collection commit invariant.
- **Bridge teardown.** `buildCohortTopicChangeBridge` de-dups `makeCohortTopicChangeNotifier` and
  `attachCohortChangeBridge`; the returned `unsubscribe` releases only the catch-all origination feed and
  leaves per-collection `onCollectionChange` delegation (which lives on the `StorageRepo`) intact.
  Idempotency is exercised by the new test.
- **Docs.** `docs/internals.md` updated to `selfIsCohortMember(event)` and the tail-anchored topic
  derivation; reflects the new reality.
- **Lint.** Repo `lint` script is a no-op echo (`'Lint not configured for all packages'`) — nothing to
  run.
- **Tests.** Suites green: `db-core` 721 passing; `db-p2p` 639 passing / 9 pending; `quereus-optimystic`
  builds clean (212 passing per implement handoff, type-additive change). New tests cover the gate
  receiving the whole event (`event.tailId`), `tailId` present on a commit event and `undefined` on a
  read-driven promotion, and the `attachCohortChangeBridge` unsubscribe (origination stops, per-collection
  delegation survives, idempotent). End-to-end tail through the real transactor is asserted in 13.6's
  `host-node-activation.spec.ts`.

**Major (filed, not fixed here):**
- `tickets/fix/commit-redirect-key-anchors-on-wrong-block.md` — **`RepoService` keys the commit
  redirect on `operation.commit.tailId`** (line ~211), while the commit handler
  (`CoordinatorRepo.commit`) anchors consensus and `verifyResponsibility` on `blockIds[0]`, and all other
  ops (`get`/`pend`/`cancel`) redirect on their own first block. Before this seam the per-block commit
  `tailId` was `undefined` on the wire (→ constant `H("")` key — inert garbage); the seam makes it a real
  block id, so for non-tail per-block batches the redirect now keys on the collection tail's cluster
  rather than the cluster of the blocks actually being committed. Latent only — every current test runs
  in a small mesh where `checkRedirect` returns `null` regardless of key, so this is **not** a regression
  of currently-exercised behavior — but it is a real redirect-routing bug for distributed (multi-cluster)
  collections, newly reachable. Candidate fix (`blockKey = operation.commit.blockIds[0]`) is a one-liner
  but touches consensus-adjacent routing with no multi-cluster test coverage, so it is filed for proper
  validation rather than changed inline in a reactivity-seam review.

**No other findings.** No security, resource-cleanup, or error-handling issues found in the seam: the
origination path isolates throws (`originate`'s try/catch), the change-listener contract stays
synchronous (no async header read introduced), and teardown is explicit and idempotent. The
NetworkTransactor tailId threading has no dedicated unit test (covered transitively by the 13.6 e2e) —
judged an acceptable gap, not worth a ticket.
