# Complete: read-dependency cascade detection + re-evaluation

Builds on `7.5-invalidation-log-entry-and-reversal`. Adds the **cascade**: from one reversed `T_inv`, find
its transitive read-dependents across collections and re-evaluate each, invalidating only those that no
longer hold. Implementation summary is in the original handoff (see git `ticket(implement):
invalidation-cascade-detection`, commit `42b1f17`); this file records the review outcome.

## What was implemented (verified)

- **Read-set persistence (db-core).** `ActionEntry.reads?: ReadDependency[]`; `Log.addActions` threads an
  optional `reads` param inserted before the trailing `timestamp`; `coordinator.ts` passes
  `transaction.reads` on commit. Confirmed only two `addActions` callers exist and neither passed
  `timestamp` positionally, so the new param is back-compatible. `reads === undefined` (legacy/unknown)
  is kept distinct from `reads === []` (read nothing).
- **Cascade engine (`db-p2p/src/dispute/cascade.ts`).** `cascadeInvalidate` — frontier seeded from the
  root's reverted pairs, per-round forward log walk collecting read-dependents, deterministic
  `(rev, collectionId, actionId)` processing, pluggable `Reevaluate` (default `contentEqualityReevaluator`),
  child `InvalidationEntry` via the 7.5 `applyInvalidation` primitive carrying `cascadeRoot`, fixpoint +
  two hard horizons with escalation, same-collection back-edge guard.
- **Config / plumbing.** `ClusterConsensusConfig.maxCascadeDepth` (32) / `maxCascadeTransactions` (1000);
  `ApplyInvalidationParams.cascadeRoot`; shared `hashBlockContent`; cascade API exported from `dispute/index.ts`.
- **Docs.** `right-is-right.md` (Read-Dependency Cascade subsection + file map) and `correctness.md`
  (Theorem 8b). Read against the code — accurate for what is built.

### Checked — correctness of the engine

- **Single-collection semantics are sound.** Linear chain, diamond (evaluated once after both ancestors),
  re-evaluation prune (retain on unchanged content), injected field-granular retain, structural-only false
  dependent, legacy-escalation, horizon overflow, idempotent restart, back-edge guard — all verified against
  the code, not just the test names. The `retained` array (reset per round) correctly ends up holding the
  final still-retained set because the frontier only grows, so a dependent never stops being a candidate and
  is re-evaluated every round until the terminating round captures it.
- **Termination / convergence** match Theorem 8b: monotonic frontier, permanent `invalidatedIds`, finite
  candidate population, two hard horizons; deterministic content-addressed children give per-member
  convergence without a per-child consensus round. The idempotent-restart path correctly reuses
  `applyInvalidation`'s `already-applied` return (which carries `existing.reverted`) to keep growing the
  frontier on re-run.
- **Frontier pair semantics** (key `blockId rev`, restored-hash comparison, owning-collection storage
  resolver) are internally consistent between seed, child-feedback, and the default re-evaluator. Relies on
  block ids being globally unique (they are — generated/CID ids), so cross-collection pairKey collisions are
  a non-issue.

### Found — MAJOR (filed as new tickets, not fixed in this pass)

1. **Multi-collection dependent reverted in only one collection.** The children dedup keys on `actionId`
   alone, but a transaction spanning N collections shares one `actionId` across N log entries (each with its
   own `blockIds`). The engine processes the first collection-entry, marks the `actionId` invalidated, and
   skips the rest — so the transaction's writes in all other collections are never reverted and no child
   entry is appended there. **Confirmed with a two-collection reproduction during review** (collection A kept
   only the root entry; A's `t2` write was never reverted). The single-collection tests can't catch it and
   the docs' "Multi-collection" section only covers cross-collection read *edges*, not a dependent that
   itself spans collections. Latent today (engine not yet wired to a composition root) but violates the
   all-or-nothing reversal guarantee once wired. → **`tickets/fix/cascade-multi-collection-dependent-reversal.md`**.

2. **Delete-restore / created-block reversal still deferred — and throws at `rev > 1`.** 7.5's review pointed
   block-creation reversal at "the cascade ticket"; the cascade ticket deferred it again. `computeRevertedBlock`
   only recognizes creation at `invalidatedRev <= 1` (records the `DEFERRED_DELETE_RESTORE` sentinel rather
   than physically removing), and a block *created* at `rev > 1` makes it read a nonexistent prior revision
   and **throw** (the implementer hit this and worked around it by pre-creating the block at genesis). Both
   cases are now reachable from the cascade (a dependent that creates a block). Needs an `IBlockStorage`
   delete-transform API → design work. → **`tickets/backlog/invalidation-delete-restore-and-created-block-reversal.md`**.

### Found — MINOR (acknowledged, no change — safe and documented)

- **Default re-evaluator is block-granular conservative.** It can over-invalidate a dependent that read an
  unchanged *field* of a changed block; never wrongly retains. Safe (over-invalidation just resubmits).
  Field-granular pruning requires an engine-backed `Reevaluate` (injection point provided). Flagged in the
  handoff (gap #1); a decision to ship an engine-backed re-evaluator belongs to the live-wiring work, not here.
- **`TransactionValidator` not literally reused** (gap #2) — its revision-equality-vs-tip check is
  incompatible with the reverted-state model (compensating revisions are *higher* revs). Content comparison /
  injected re-evaluator is the correct substitute. Intentional divergence; accepted.
- **Escalation `remainder` over-reports.** On a horizon, `remainder` is the whole current candidate set,
  including genuinely-retained candidates that don't need re-sync. Conservative (surfaced, never dropped) and
  documented. No change.
- **Full per-round log rescan.** Each round walks every in-scope collection log end-to-end, O(rounds × log).
  Horizon-bounded and the dispute TTL bounds the relevant suffix; not a correctness issue. Acceptable for the
  library stage; revisit if/when wired at scale.
- **`maxCascadeDepth` counts rounds, not graph depth** (gap #9). A linear chain in rev order collapses to ~one
  progressing round; cross-collection ancestor-ordering may cost extra rounds. Documented; acceptable as a
  horizon.
