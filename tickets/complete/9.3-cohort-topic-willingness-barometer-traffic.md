# Complete: willingness / barometer / topic-traffic

Implements the admission/load brain of the cohort-topic substrate: the **capacity barometer**
(per-tier log-bucketed load), per-member **willingness** (the admit decision), and the
**topic-traffic** counters (advisory flow signal on replies). All three are pure db-core logic over
injected ports (gossip view, registration store) — no FRET/libp2p imports.

## What landed

- **Barometer** (`load/barometer.ts`): `utilizationBucket(u) = clamp(7 + ⌊log₂ u⌋, 0, 7)`;
  `LoadBarometerState` with `observe`/`bucket`/`reading`/`loadBuckets`/`isOverloaded`/`loadWilling`;
  `DEFAULT_OVERLOAD_BUCKET = 6` — the single threshold for both willingness-flip and early-promote.
- **Willingness** (`willingness.ts`): `createWillingnessCheck(deps).evaluate(reg, self, now)` →
  `accepted` | `unwilling_member{candidateMembers}` | `unwilling_cohort{retryAfterMs}`, mirroring the
  simulator's `classifyAdmission` adapted to gossiped (not live) sibling willingness. Bit packing
  (`tierBit`/`packWillingnessBits`/`willingnessBitsHex`/`selfWillingnessBits`) and capped-doubling
  back-off (`backoffRetryMs`, base 1s / ×2 / cap 60s) folded back verbatim from the simulator.
- **Traffic** (`traffic.ts`): windowed exact-integer counters (`recordArrival` combines
  fresh+renewal, `recordQuery`), `publish` freezes per-round counts, `snapshot` is gossip-derived
  (own + sibling cohort-wide sums, `directParticipants` from the replicated store), `attachTopicTraffic`
  wires the signal onto `accepted`/`promoted` replies only, `reset` zeros on epoch change.
- **Docs**: §Willingness / §Capacity barometer / §Topic traffic signal carry "Resolved" /
  "Implementation" notes pinning 1-bit-per-tier willingness, `bucket_overload = 6`, and
  traffic-on-accepted+promoted-only.

### Checked — correctness / logic
- **Barometer math** — boundary table verified (`u=0.5→6`, `u=1.0→7`, `1/64→1`, `1/128→0`),
  NaN→0, +Infinity→7, clamping both ends, `loadBuckets()` returns a defensive copy. Correct.
- **Willingness `evaluate`** — quorum gate / accepted / unwilling_member / unwilling_cohort ordering
  matches the simulator's `classifyAdmission`, correctly adapted: siblings counted via **gossiped**
  willingness bits (self excluded), self via **live** willingness (profile ∧ load ∧ budget). The
  gossiped coarse bit (profile ∧ load, no budget) vs the live finer gate is intentional and documented.
  Out-of-range op tier → `unwilling_cohort`. Correct.
- **Back-off** — `backoffRetryMs` identical to the simulator's `backoffDelay` (formula, params,
  validation); huge `attempt` saturates at cap with no overflow. Correct.
- **Traffic snapshot** — own (`lastPublished`) + sibling gossiped summaries summed for the
  partitioned arrival/query rates; `directParticipants` read from the replicated store (not summed);
  `childCohortCount` = max across siblings unless the promotion-layer override is supplied (and a `0`
  override is honored, since `?? `is nullish). Self-echo de-dup by `selfMember`. Window pruning at
  publish-time only. Correct.
- **Reply wiring** — `attachTopicTraffic` sets `topicTraffic` only on `accepted`/`promoted`, no-op
  elsewhere; matches the `RegisterReplyV1.topicTraffic` wire comment. Correct.

### Found & fixed inline (minor)
- **Doc/impl mismatch (docs accuracy).** §Topic traffic signal L330 read "the reply uses the
  responding member's **own** most-recent gossip-derived view — *the same per-topic entry the member
  last gossiped*", implying own-only — but `snapshot()` aggregates own **+ sibling** cohort-wide
  sums. Since arrivals shard ~`1/k` across members (FRET routes each registration to one member),
  cohort-wide aggregation is the semantically required reading (matchmaking consumes
  `arrivalsPerMin` as a cohort rate). Rewrote L330 to state the aggregation, and extended the
  §Topic-traffic "Resolved" note to pin own+sibling-sum / store-derived-`directParticipants` /
  max-`childCohortCount`. The implementer flagged this as an open interpretation; it is now resolved
  in the doc to match the code.
- **Test floor raised (+9 specs).** Added coverage for previously-untested error/edge paths:
  `utilizationBucket` NaN/Infinity; barometer `overloadBucket` `RangeError` (0/8/3.5);
  `reading`/`loadBuckets` (incl. defensive-copy); `willingnessBitsHex` nibble emission + high-bit
  masking; `evaluate` with an out-of-range tier → `unwilling_cohort`; `backoffRetryMs` negative /
  non-integer `RangeError` and huge-attempt saturation; `createTrafficCounters` non-positive
  `windowSeconds` `RangeError`; custom-window echo in snapshot; never-observed-topic zeros.

### Found & deferred to a new ticket (major — design decision)
- **Admission-quorum semantics are unpinned.** The willingness check defaults `quorum` to strict
  majority `⌊k/2⌋+1` (= 9 for k=16), but `docs/cohort-topic.md` only ever says "a quorum" and pins no
  number; the simulator left it a free parameter. Whether this should be strict majority or aligned
  with the cohort-signature threshold `minSigs = k − x = 14` (a *different* concern — signing vs
  admission) is a policy decision needing sign-off. Filed
  `tickets/backlog/cohort-topic-admission-quorum-semantics.md`. Not blocking: the value is
  configurable via `WillingnessConfig.quorum`, and the downstream integration ticket can set it
  explicitly.

### Checked & accepted as-is (documented, owned downstream)
- **`maxPrimaryTopicsPerTier = 2048` default** reuses the *cohort-wide* `topics_max` for a
  *per-member-per-tier* budget — scopes differ, but the number is configurable and the doc pins no
  per-member figure. Left as a documented default; the integration/config ticket can tighten it.
- **`countWithin` assumes ascending `now`** (monotonic clock). Out-of-order `recordArrival` would
  under-prune. True for production callers; documented assumption. Verify when the register/renew
  handlers are wired (ticket 10).
- **Two overload thresholds** — the gossiped bit uses the barometer's own `overloadBucket` via
  `loadWilling`; the live gate uses the willingness check's `overloadBucket`. `createWillingnessCheck`
  takes an externally-constructed barometer, so a caller must keep the two in sync (the
  `*WithIdleBarometer` helper does). Foot-gun, but intentional separation; note for integration.
- **No integration wiring** — `publish` cadence, `recordArrival` from register/renew handlers,
  `reset` on the gossip-bus drift handler, and folding `selfWillingnessBits`/`loadBuckets` into the
  outbound `CohortGossipV1` are owned by `cohort-topic-core-module-fret-integration` (which lists this
  ticket as a prereq) and the anti-DoS ticket (`attempts` counter). Confirmed correctly out of scope.
- **`childCohortCount`** authoritative value owned by the promotion ticket via the injected override.

### Not applicable / empty
- **Resource cleanup** — pure in-memory `Map`s; `reset()`/eviction-via-`countWithin`-prune keep them
  bounded. No handles/timers/listeners to release. Nothing to check.
- **Concurrency** — synchronous single-threaded logic over injected state; no async, no shared
  mutable cross-call state beyond the per-instance maps. Nothing to check.
