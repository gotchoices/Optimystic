## What shipped

Real libp2p Ed25519 backing for the client-transaction-signature ports that the db-core half
(`implement-client-tx-signature-core`) defined but left un-backed, plus the rollout switch:

- **Signing (client side).** `collection-factory.ts` `getSigner(options)` returns a `TransactionSigner`
  bound to the node's libp2p identity key (`async payload => bytesToB64url(await signPeer(nodeKey,
  payload))`). `txn-bridge.ts` passes it into `TransactionSession.create(...)` as the 6th arg. No node/key
  → `undefined` → unsigned session (unchanged legacy behavior). Signing only *adds* a `signature` field.
- **Key exposure.** `libp2p-node-base.ts` attaches the private key as `(node as any).peerPrivateKey`, on
  the same `(node as any).*` surface already used for `coordinatedRepo`/`keyNetwork`/etc. Ed25519 by
  construction.
- **Verify + enforcement (node side).** `quereus-validator.ts` `requireClientSignature?: boolean`
  (default `false`). When `true`, constructs the `ClientSignatureVerifier` port (`verifyPeerSig` over the
  peer-id-derived key, total: catches decode errors → `false`) and passes it to `TransactionValidator`,
  so `validate()` rejects missing (`Missing client signature`) or invalid (`Invalid client signature`)
  signatures at pend. When `false`, the port is omitted: signed and unsigned both pass (migration
  posture).
- **Signer-failure safety.** `session.commit()` wraps the signer call in try/catch → clean `{ success:
  false, error: 'Failed to sign transaction: …' }` before `coordinator.commit` runs, so a failed sign
  cannot leak a partial unsigned tx into consensus.

**Identity model (settled):** an Ed25519 libp2p peer-id string embeds the raw public key, so the verifier
derives the client key straight from `stamp.peerId` — no key distribution. Non-Ed25519 identities are out
of scope (`verifyPeerSig` → `false`, rejected under enforcement). Stamp-field tampering is caught upstream
by the prereq `bind-stamp-fields-to-tx-signature`'s step-0.0 integrity check (`stamp.id` recomputed from
fields before the signature step), so mutating `peerId`/`expiration`/`engineId`/`schemaHash` fails as
`Tampered transaction stamp` regardless of the signature.

## Review findings

Reviewed the implement diff (`42f6d47`) with fresh eyes against SPP, DRY, modularity, error handling,
type safety, and — most importantly — attack surface and test completeness.

**Correctness / crypto wiring — CONFIRMED SOUND.** Signer payload (`clientSignaturePayload(stamp.id,
statements, reads)`) matches the verifier payload exactly; `transaction.reads` is the same array signed.
The signature transitively binds all stamp fields via `stamp.id` + the always-on step-0.0 integrity check,
so peer-id substitution / stamp tampering is caught. `verifyPeerSig` is total (never throws) and the
plugin closure additionally wraps `b64urlToBytes` in try/catch — verified against malformed peer-id,
non-base64url signature, and `'AAAA'` (valid b64 / wrong length) inputs. Signature-step ordering
(after integrity + expiration, before engine/schema) is correct and defended by a comment. No issues.

**Test coverage — one gap fixed inline (minor).** Impersonation (wrong key, right peer-id) was covered,
but *content* integrity — a relayer mutating `statements` after a valid sign while keeping the stale
signature — was not directly asserted. Added `rejects a tampered statement…` to
`client-tx-signature.spec.ts` (now 10 passing). It proves the payload binds `statements`, a distinct
property from key-binding.

**Major gap — filed as a ticket.** No live cross-node enforcement test, and **no production code sets
`requireClientSignature: true`** — enforcement is proven only at the validator seam (single-process),
never through a live cluster PEND path, and the mesh harness does not thread the flag. Filed
`backlog/debt-mesh-client-signature-enforcement.md` (prereq: this ticket) for a mesh test that flips
enforcement on across nodes and asserts unsigned/mismatched clients are rejected at pend, plus surfacing
the flag as a real config so a deployment can opt in.

**Tripwires (conditional — parked, not ticketed):**
- *Recovery re-verification through on-disk serialization is unverified.* The `signature` field's doc
  claims it "rides along through serialization/persistence," but only JSON round-trip + an in-memory
  session commit are tested — NOT the db-p2p cluster-repo on-disk transaction-log path. No recovery
  reader exists yet. Tagged `NOTE:` at the field definition
  (`db-core/src/transaction/transaction.ts` `signature?`) so a future recovery author meets it. Only
  becomes work when a recovery path lands.
- *Injected-node silent-unsign.* A node registered via `registerLibp2pNode` but not built by
  `createLibp2pNode` carries no `peerPrivateKey`, so signing silently disables. Already `NOTE:`-tagged at
  the `getSigner` code site by the implementer. Harmless now (enforcement off); work only if a deployment
  enforces against such a node.
- *Non-Ed25519 client identities rejected under enforcement.* `verifyPeerSig` → `false` for them. Matches
  the cohort-topic substrate's standing Ed25519 assumption; work only if the project ever needs
  non-Ed25519 client identities. Documented, no code site.

**Not re-covered (owned upstream):** stamp-field tampering (expiration/engineId/schemaHash mutated while
holding `stamp.id` + signature) is handled by the prereq `bind-stamp-fields-to-tx-signature` step-0.0
integrity check with existing db-core tests.

**Docs:** the `signature` field doc in `transaction.ts`, and the JSDoc on `getSigner`,
`requireClientSignature`, `createClientSignatureVerifier`, and the `session.commit` signing block, were
read and reflect the shipped reality (one `NOTE:` added, above). No stale docs found.
