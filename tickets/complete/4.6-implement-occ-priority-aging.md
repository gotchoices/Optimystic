## What was built (implement stage)

A transaction now carries an **advisory, aged `priority`** (default 0) that rises by one per failed
retry and is capped at `MaxPriority` (8). A cluster member consults it as the **first tiebreak** in
`resolveRace`, so a repeatedly-losing transaction eventually out-ranks fresh (priority-0) rivals.
Priority is carried inside the signed cluster `message` (multi-collection: `pend.transaction.priority`;
single-collection: top-level `pend.priority`), read back via `clampPriority(...)`, and was kept out of
the transaction id, client signature, operations hash, and stale-read/validity checks.

See the git history (`ticket(implement): implement-occ-priority-aging`, d837df3) for the full data
flow and the implementer's own honest-gaps list.

## Review findings

**Scope of review.** Read the full implement diff fresh, then the handoff. Scrutinised the
consensus-arbitration change (`resolveRace` reorder) against the promise/commit protocol in
`cluster-repo.ts`, the two retry loops that set priority, `clampPriority` bounds, the carrier
plumbing, the doc rewrite, and both new test suites. Ran the **full** db-p2p suite (1214 passing)
and the **full** db-core suite (1226 passing) — both green (the implementer had only run targeted
specs; the broader run was gap #4 in their handoff, now closed).

### MAJOR — filed as a ticket

- **`resolveRace` priority-first is a likely safety regression, not a fairness-only change.**
  → `fix/occ-priority-first-breaks-promise-monotonicity`.
  Placing priority *ahead of* the promise count removes the monotonicity that the old "more
  promises wins" order provided. In this protocol a member commits purely on promise supermajority
  (`handleCommitNeeded` has no conflict re-check), `validatePendOperations` does not reject a second
  *pending* conflicting transaction, so `resolveRace` is the **sole** arbiter among concurrent
  conflicting pends — a safety mechanism, not a liveness tweak. Old order guaranteed: once X reaches
  a promise supermajority, no conflicting rival can also reach one (quorum-intersection member
  rejects the lower-promise rival). Priority-first lets a high-priority Y displace an
  already-quorum-reached X on its promising members; X still finishes via its already-collected
  promises (no commit-path conflict check), so **both conflicting transactions can commit** →
  split-brain. Reachable in the feature's intended use case (a starved tx overtaking a rival that is
  currently winning), not a corner case. The 4.6 tests pass because they exercise `resolveRace` and
  the retry loops in isolation and never the promise→commit composition — which is exactly where the
  regression lives. Confidence: reasoning-confirmed against the code paths; not yet reproduced on a
  live multi-node cluster (the ticket's first task). This also means `docs/correctness.md` Theorem 9's
  "safety intact" claim is currently unproven — the ticket re-derives it after the ordering is fixed.
  A `WARNING:` breadcrumb was added at the `resolveRace` site pointing at the ticket (comment-only,
  no behavioural change), because the surrounding comments assert "safety intact" unqualified.

### Verified correct (no action)

- **`clampPriority` bounds** — `[0, MaxPriority]`, floors, NaN/≤0→0, over-cap/+Inf→cap. Read and
  write both clamp, so a Byzantine self-asserted value cannot exceed the cap. Tested.
- **Priority excluded from id / client signature / operations hash / stale-read** — confirmed in
  `createTransactionId`, `clientSignaturePayload`, and `validate` (stale read still rejects a
  `MaxPriority` transaction). Tested. This part of the "fairness-only" claim holds.
- **Integrity in transit** — priority rides inside the hashed `message`; a post-signing bump trips
  `Message hash mismatch`. Tested. `cluster-coordinator.ts` needed no hashing change (gap #3
  confirmed — the whole `message` is already hashed).
- **Dual carrier** (gap #2) — `pend.transaction?.priority ?? pend.priority` is acceptable; the
  single-collection sync path has no `Transaction`, so a top-level carrier is unavoidable. Both live
  inside the signed message, both clamp on read. No issue beyond the ordering defect above.
- **Backward compatibility** — a record with neither carrier reads as priority 0; mixed-version
  races stay deterministic. Tested.
- **Retry loops** — coordinator sets `priority` only when `staleLosses > 0`; `Collection.sync`
  passes `clampPriority(consecutiveFailures)` (0 on first attempt), so the common first-attempt pend
  is byte-identical to before. Verified by reading; `Collection.sync` climb is thin-tested (gap #5)
  but is subsumed by the fix ticket's adversarial harness.
