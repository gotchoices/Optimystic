# Completed — single-source + fail-fast-couple the super-majority threshold

## Summary of shipped work

`superMajorityThreshold` (fraction of a cluster's peers that must promise before a
transaction proceeds) had three drifting *absent-config defaults*: member `?? 1.0`,
coordinator `?? 0.75`, node composition root `?? 0.67`. Now:

- **One exported constant** `DEFAULT_SUPER_MAJORITY_THRESHOLD = 0.75` in
  `packages/db-core/src/cluster/structs.ts`, re-exported via the `@optimystic/db-core`
  barrel. Every fallback default references it (member, coordinator policy, node base,
  and — added during review — the test mesh-harness).
- **Effective-threshold getters** on `ClusterMember` and `CoordinatorRepo` reporting the
  value each component actually uses for vote counting.
- **`assertSuperMajorityCoupling(member, coordinator)`** — a small unit-testable helper
  that throws at node construction if the two resolved thresholds differ, with a message
  naming both values and their provenance. Called in `libp2p-node-base.ts` right after
  both components are built.

Build + targeted tests green (96 passing across the six consensus suites; re-verified 21
passing on coupling + mesh-sanity after the review fix).

## Review findings

**Scope checked:** the full implement diff (all 7 source/test files) read before the
handoff; every vote-counting site that consumes `superMajorityThreshold`
(`cluster-repo.ts:756`, `:1339`, `cluster-coordinator.ts:280`); constructor signatures vs.
the test call sites; the two factory functions feeding the startup assertion; every
`superMajorityThreshold` default/literal across `src/` and `test/`; and `docs/cluster.md`.

**Behavior-change soundness (verified, no defect):** The member's old `?? 1.0` fallback was
effectively dead in the real node — `libp2p-node-base` always populated
`consensusConfig.superMajorityThreshold` (previously 0.67), so the member never actually ran
at 1.0 in production; the `1.0` only surfaced when a `ClusterMember` was built with *no*
config (tests/harness). So the real-node default moves coordinator+member together 0.67→0.75,
which stay coupled. Confirmed `ceil(peerCount * threshold)` for the common small cluster
sizes: the member/coordinator required-promise counts match at 1–3 peers; they only diverge
at ≥4 peers, where production always supplies an explicit threshold anyway.

**Minor — FIXED inline:** `mesh-harness.ts` hardcoded `?? 0.75` on *both* the member
(`:144`) and coordinator (`:265`) sides — a literal duplicate of the constant that directly
undercuts the ticket's single-source-of-truth goal (change the constant and tests would keep
defaulting to 0.75, silently diverging from production). Rewired both to
`?? DEFAULT_SUPER_MAJORITY_THRESHOLD`. Both sides still share one expression, so no coupling
regression; rebuilt and re-ran coupling + mesh-sanity (21 passing).

**Wiring correctness (verified):** `clusterMember()` and `coordinatorRepo()` are concrete
factories returning `ClusterMember` / `CoordinatorRepo` — not overridable via `NodeOptions` —
so the `as ClusterMember` / `as CoordinatorRepo` casts feeding the assertion cannot be a
foreign impl lacking the getter. The assertion sits at the composition root before
transaction recovery / request acceptance, as intended.

**Docs (verified accurate, no change needed):** `docs/cluster.md` documents 0.75 as the
default in all relevant spots (`:157`, `:185`, `:644`, `:718`, `:734`) and never claimed the
member defaulted to unanimity, so nothing went stale. It does not name the new constant or the
coupling assert — an enhancement, not a correctness gap; left as-is per the implementer's call.

**Coverage gaps (accepted as a reasonable floor, no ticket):** No end-to-end test boots
`createLibp2pNode` through the real assertion — it is covered via the extracted helper plus the
real class getters, and the wiring is a 3-line call over verified concrete types. Forcing a
genuine mismatch requires reaching into internals since both sides share one `consensusConfig`
by construction. Low residual risk; not worth a heavyweight node-boot test.

**Tripwires (recorded, not ticketed):**
- The coupling assertion is un-failable on today's wiring (member + coordinator share one
  `consensusConfig`); it only earns its keep if someone later threads the two from *separate*
  config sources. Rationale already lives in the helper's doc comment at the site — no new
  code note needed.
- `assertSuperMajorityCoupling` uses strict float `!==`. Safe today because both operands are
  one stored config number, never a computed sum. If a future path ever *computes* a threshold
  (e.g. `3/4` on one side, a literal `0.75` on the other), exact equality could false-trip — but
  that path does not exist. Parked here; not worth an epsilon-compare while both values share a
  single source.

**Major findings:** none — no new fix/plan/backlog tickets filed.
