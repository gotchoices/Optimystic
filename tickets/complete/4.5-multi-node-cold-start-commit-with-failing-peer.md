## Summary

### Root cause

`NetworkTransactor` uses two different coordinator-selection algorithms across the two phases of a write:

- **pend** uses `consolidateCoordinators` (findCluster + greedy set-cover across block clusters) — tends to consolidate to a single coordinator per write.
- **commit** uses `batchesForPayload` (XOR-distance `findCoordinator` per block) — picks the closest peer to each block, independently.

With `superMajorityThreshold=0.6` on a 5-node cluster (3 approvals required) and one peer (E) unreachable at boot, pend reaches super-majority (A/B/C/D save the pending action, E is skipped). But commit can land on a *different* coordinator than pend — in particular, `findCoordinator` can pick E for some blocks by distance. In that case:

1. E's `ClusterCoordinator.executeClusterTransaction` for the commit runs. Other peers sign approve (idempotently commit their local pending). E itself signs approve on the commit payload.
2. In the broadcast phase, each peer sees majority commits → enters `TransactionPhase.Consensus` → `handleConsensus` → `storageRepo.commit`. On E, this throws "Pending action … not found" because E missed the pend. `handleConsensus`'s catch deletes E from `executedTransactions` and rethrows; the rejection is swallowed by the coordinator's `Promise.allSettled`.
3. `executeClusterTransaction` returns normally. Back in `CoordinatorRepo.commit`, `localExecuted` is false, so the fallback `this.storageRepo.commit(request)` runs on E — and throws the identical error, which propagates to `NetworkTransactor` as a batch-level failure.
4. `processBatches` retries with `excludedPeers=[E]`, but the mock `findCoordinator` used to ignore that option, so retries looped back to E.

In practice the test was already passing because (a) `StorageRepo.commit` is idempotent per block (crash-c ticket) and (b) `NetworkTransactor.commit` tolerates non-tail commit errors when the tail has committed. But every run spammed many `console.warn` lines and burned retry cycles.

### Fix

**`CoordinatorRepo.commit`** now distinguishes "cluster didn't execute" from "cluster executed elsewhere but local peer is behind". After `executeClusterTransaction`:

- If `localExecuted` → return success (unchanged).
- Otherwise attempt `storageRepo.commit(request)` locally. If it throws **and** the returned `ClusterRecord` has a simple majority of approving commit signatures, the cluster is authoritative — return success and log `coordinator-repo:commit-local-failed-cluster-succeeded`. This peer will reconcile via the existing `clusterLatestCallback` sync path.
- If there was no cluster consensus, throw the original error (unchanged).

This targets the exact stranded case: a peer that is the coordinator for commit but wasn't part of the pend quorum. It does not widen any success path when consensus was not reached.

**`MockMeshKeyNetwork.findCoordinator`** now honors `excludedPeers`, matching the real `Libp2pKeyPeerNetwork.findCoordinator` at `packages/db-p2p/src/libp2p-key-network.ts:316-429`. Previously the mock always returned `sorted[0]`, so `processBatches` retries with excluded peers looped back to the same coordinator. With this change, retries converge to a reachable peer in one hop.

### Why the fix is safe

- Majority commit signatures mean a super-majority of peers signed the commit payload with their valid keys. Cluster consensus is reached in the protocol sense — the operation is committed cluster-wide.
- A signed commit doesn't strictly prove each signer *finished* applying the transform to local storage, but signers that failed locally will re-sync via the normal reconciliation paths; they are not quorum members that rolled back.
- The fallback is only skipped when a majority signed. When consensus was not reached, the original behavior (throw) is preserved — so `mesh-sanity` Suite 2 "promise phase failure with default threshold causes transaction to fail" (3-of-3 at 0.75, 1 peer down → 2 approvals → reject pend) still fails pend as expected (verified).

### Review notes

- `clusterReachedCommitConsensus` uses `approvedCommits > peerCount / 2`, identical arithmetic to member-side `hasMajority` in `packages/db-p2p/src/cluster/cluster-repo.ts:550-552`. Keep the two in sync if either changes.
- `CoordinatorRepo.pend` was considered for analogous treatment; it is not needed today because `StorageRepo.pend` is idempotent for new pending actions and rarely throws in the cold-start case.

### Residual read-side flakiness (pre-existing, not caused by this fix)

Scenario B shows ~10% flakiness on isolated repeated runs (`--grep "Scenario B"`, 10 iterations) both before and after this fix — the failing iterations return `undefined` from `treeB.get(42)` rather than throwing from the commit path. Stashing the `coordinator-repo.ts` change and rerunning reproduces the same failure rate, so this is pre-existing. The `fresh-node-ddl-multi.spec.ts` full-suite run in CI order has been stable across multiple back-to-back executions. The residual flakiness appears to be on the **read** path (coordinator selection / sync-on-read timing when peer E is first-picked for a block B never had), not the write path this ticket addresses — a natural follow-up ticket can start with `NetworkTransactor.get`'s retry predicate (`hasBlockInResponse`) interaction with fresh-coordinator empty-state responses, and `CoordinatorRepo.fetchBlockFromCluster` timing when the first-picked coordinator is the unreachable peer.

## Use cases / where this matters

- Cold-start writes where one cluster member is unreachable at boot — the writing client does not know which peer will be picked as the commit-phase coordinator (distance-based) vs pend-phase coordinator (cluster-intersection set cover). If they diverge to an unreachable peer for commit, the cluster still commits; the lagging peer catches up via sync.
- More generally: any scenario where `CoordinatorRepo.commit` runs on a peer that missed the pend phase (dropped during pend, freshly joined, transient partition during pend) — commit now succeeds based on cluster majority rather than erroring out on the local coordinator's missing pending-action state.
- Mock harness behavior now matches `Libp2pKeyPeerNetwork` for the `excludedPeers` option, so retry logic under `processBatches` actually converges in tests rather than looping back to the original failing peer.
