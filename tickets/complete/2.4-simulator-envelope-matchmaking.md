# Complete: matchmaking operating-envelope boundaries

Two `BoundaryAxisSpec`s layered on the generic `findBoundary` harness, folded into a `Metrics` sink and
exported from `src/index.ts`. New **measurement** over the existing matchmaking model — `SeekerWalk`,
the decision math, and `seekerPoolContentionWouldFlip` are reused read-only. Reviewed adversarially;
accepted with two minor fixes applied in-pass (a README docs gap and an added regression test). Build
(`tsc`) clean, full suite **257 passing** (was 256; +1 added in review).

## What was built (as implemented)

`runMatchmakingBoundaries({ seed })` → `MatchmakingBoundaryReport { boundaries, metrics, …diagnostics }`.

- **Boundary 1 — `bounded-harm` × `lyingFraction`** (seeded seeker walks). Axis `f ∈ [0,1]` = fraction
  of the seeker's non-root path tiers whose primary mis-reports `topicTraffic` via the per-query-flip
  adversary. `holds(f) = liedMatched ∧ extraHops ≤ lyingTierCount ∧ extraDrain ≤ patienceMs`, measured
  relative to an honest baseline. Located `f* ≈ 1.5/startTier ≈ 0.1875`, **positive** margin.
- **Boundary 2 — `hang-out-fairness` × `seekerProviderRatio`** (pure function). Axis `ρ` = seeker-pool
  demand per provider-arrival. `holds(ρ) = !seekerPoolContentionWouldFlip(…)`. Located `ρ* = 2.5`,
  `designAssumption = cap−1 = 3`, margin **−0.5** (negative *by construction* — the misfire precedes the
  design's assumed worst-case contention, the principled signal that the exact-`Σ wantCount` refinement
  is warranted).

## Review findings

**Method.** Read the implement diff (`git show 15ff55f`) with fresh eyes before the handoff summary,
then traced every moving part: `boundary.ts` (scan/bracket/bisect, the three open-bracket cases,
`marginRatio = NaN` handling), `seeker-walk.ts` (per-tier query/escalate/hang-out, the
`patience_per_tier_fraction` budget split, `SeekerTrace`), `matchmaking.ts`
(`contentionFactor`/`expectedNewMatches`/`decideHangOut`), and `refinement-signal.ts`
(`seekerPoolContentionWouldFlip`). Hand-verified both boundaries' algebra; ran an out-of-band probe to
confirm the B1 mechanism; ran build + full suite.

- **Correctness / algebra — checked, sound.** B2 hand-derived end to end: `newMatches = 60·1·(24000/60000)
  = 24`, `lhs = 4+24 = 28`, `approxContention = min(1+60/60,4) = 2` (always hangs out),
  `exactContention = min(1+ρ,4)`; the exact decision escalates once `8·(1+ρ) > 28` ⇒ `ρ* = 2.5`,
  `exactContentionAtEdge = 3.5 < cap`, `margin = 2.5−3 = −0.5`. Monotone across the whole `[0,10]` scan
  (exact term rises then saturates at the cap; the decision never flips back). B1 mechanism traced: top
  lied tier under-reports (parity 0 ⇒ escalate), the *second* lied tier over-reports (parity 1 ⇒ hang
  out), and with `patience_per_tier_fraction = 1.0` that one hang-out drains the entire budget at a
  provider-less tier ⇒ match failure at `round(f·startTier) = 2`. The per-query `queryCount` parity is
  sound because the reporter is invoked exactly once per tier (hang-out re-queries in `onRequery` do
  **not** call the reporter) and tiers are visited strictly top-down.

- **Baseline fairness — checked, sound.** The honest (`0x50`) and lied (`0x55`) walks use different
  ladder markers but both `tree.ensure` their cohort at `startTier`, so both land at `startTier` and the
  `extraHops`/`extraDrain` deltas are a fair honest-relative measure, not polluted by marker-dependent
  registration. Escalations are counted in the matchmaking layer only (registration hops excluded).

- **Headline finding is genuine, not an artifact — verified (ticket gap #1).** Out-of-band probe
  swept `patience_per_tier_fraction`: at the shipped `1.0` the edge is finite at `f ≈ 0.19`
  (`match-failure`); at `0.5` and `0.25` the boundary moves *out* past `f = 1` (`boundaryFound = false`,
  `liedMatched = true`, `within-bounds`) — the seeker escalates past the over-reporting tier and matches
  at the root, harm degrading to graded escalation. This confirms the `1.0`-fraction fragility is the
  real mechanism (and quantifies the need for `matchmaking-per-tier-patience-splitting`), not an
  over-tuned tier model. **Locked in as a new regression test** ("the edge is the
  patience_per_tier_fraction = 1.0 fragility, not a tier-model artifact").

- **Determinism — checked.** `NaN` `marginRatio` deep-equals across runs (chai SameValueZero); both
  boundaries fold into the metrics sink keyed by `(claim, axis)`; the existing determinism spec already
  pins byte-identical reports.

- **Docs — one gap found and fixed (minor).** `README.md`'s validity-envelope finder bullet enumerated
  every other `boundary-*.ts` module but omitted the new `boundary-matchmaking.ts`. Added it to the
  module list and described both axes (parallel to the reactivity/churn descriptions, noting B2's
  deliberately negative margin). No other doc references the envelope rows; `docs/matchmaking.md` is the
  decision-rule spec and needs no change (this ticket adds no decision-rule behaviour).

- **Maintainability note (no change).** `measureSeekerContention` re-derives `approxContention` /
  `exactContention` / `lhs` independently from `seekerPoolContentionWouldFlip` to expose the terms as
  diagnostics; the two are consistent today. If the decision formula in `refinement-signal.ts` ever
  changes, both sites must move together. Left as-is — the duplication is intentional (diagnostic
  surface) and the `wouldFlip` verdict itself still comes from the single source of truth.

- **Performance note (no change).** B1's `measureBoundedHarm` recomputes the honest baseline on every
  `holds()` call even though it is fraction-independent; `findBoundary` invokes it ~20–30× per boundary.
  Negligible (full B1 boundary runs in single-digit ms) — not worth caching in measurement code.

- **Cross-refs — accurate.** `matchmaking-per-tier-patience-splitting` and
  `matchmaking-contention-from-seeker-pool` exist in the board and are reused read-only via
  `refinement-signal.ts`; no code here depends on them landing. The two known gaps the implementer
  flagged (#2 the unexercised `+1-hop` under-report dimension, #3 the somewhat-arbitrary B2 constants)
  are honest decomposition trade-offs, not defects: B2's constants are robust because
  `exactContentionAtEdge = 3.5` sits comfortably inside `(1, cap=4)`, and the under-report dimension is
  already covered by `AdversarialReportingScenario.measureUnderReport`.
