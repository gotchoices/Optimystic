# Complete: cohort-topic host + change-bridge production-node activation

Flips reactivity origination from **dormant** to **live** on opt-in nodes. When
`NodeOptions.cohortTopic.enabled` is set, `createLibp2pNodeBase` creates a `CommitCertStore` early
(composed into the `onCommitCertificate` sink threaded to `clusterMember`), and post-assembly builds the
cohort-topic host, a real FRET-backed `selfIsCohortMember` gate over `coord_0(H(tailId ‖ "reactivity"))`,
and installs the origination bridge as `node.blockChangeNotifier`. A node without `cohortTopic.enabled` is
byte-identical to today (bare `blockChangeNotifier = storageRepo`, no host, no cert store).

The implementation is sound and well-documented; the implementer's handoff was unusually honest about the
solo-node cert-gating limitation. Review confirmed that framing is correct.

### Findings & disposition

- **MINOR (fixed inline) — caller cert-sink could defeat origination.** The composed
  `onCommitCertificate` ran the caller-supplied sink BEFORE `certStore.put`. Because
  `ClusterMember.captureCommitCert` isolates the whole composed call in a try/catch, a caller sink that
  threw would skip `certStore.put`, and that commit would silently never originate. **Fixed:** reordered so
  the internal `certStore.put` (which never throws) runs first, then the caller sink — origination's cert
  capture is now independent of the caller sink. (`libp2p-node-base.ts`.)

- **MINOR (fixed inline) — started node leaked on activation hard-fail.** When enabled, a missing FRET
  service or a `createCohortTopicHost` throw hard-fails by rejecting out of `createLibp2pNodeBase` — but
  the node had already `start()`ed (transports open, FRET running), so the rejection leaked a started node.
  **Fixed:** both hard-fail paths now `await node.stop()` (which runs the already-installed arachnode +
  clusterMember teardown wrappers and closes connections) before rethrowing. (`libp2p-node-base.ts`.)

- **MAJOR-as-cross-ticket-constraint (recorded) — subscriber-side encoding must match.** The pinned
  `BlockId → tail bytes` encoding (`reactivityTailBytes = utf8(tailId)`) is load-bearing: if the subscriber
  transport derives its `tailIdAtAttach` coord from differently-encoded bytes (e.g. db-core's async
  `blockIdToBytes`, which `sha256`s first), origination and subscription resolve different coords and
  origination silently never reaches subscribers. The subscriber side lives in `12.3-reactivity-
  notification-transport` (still in `plan/`), which did **not** state this constraint. **Added** a
  "Load-bearing constraint" section to that plan ticket pinning `reactivityTailBytes` and requiring a
  coord-equality test. No separate ticket filed — the constraint belongs on the owning ticket.

- **NOTED (no new ticket) — `resolveFretEngine` reaches the engine via the wrapper's private `ensure()`.**
  A real but minor design smell (the `Libp2pFretService` wrapper does not publicly expose
  `setActivityHandler`). It is guarded (falls back to the value as-is for a raw injected engine), works,
  and is exercised by the integration test. A cleaner public accessor is genuinely out-of-scope here
  (it is an upstream p2p-fret change). Left as-is with the existing in-code rationale; not worth a backlog
  ticket on its own given the substrate is still pre-1.0 and the seam is documented.

- **NOTED (no new ticket) — solo-node cannot prove the cert-bearing headline.** Confirmed the handoff's
  honesty: `CoordinatorRepo.commit` short-circuits consensus at `peerCount <= 1`, so a solo node never
  produces an authoritative cert and the bridge correctly cert-gates origination to a no-op there (never
  fabricates an unsigned cert). The cert → `onLocalCommit` byte-identical pass-through IS proven with real
  components in `change-bridge.spec.ts`; the full real-consensus-cert origination on a ≥2-node activated
  node is already owned by `14-substrate-e2e-real-libp2p-tier` (env-gated, discovery-based). **Deferral
  accepted** — solo genuinely cannot produce a cert, and ticket 14 owns the multi-node proof. No new ticket.
