# Complete: reactivity notify protocol surface + transport primitives

## What shipped

Pure transport plumbing for the 12.3 reactivity chain — frame → dial → deliver of `NotificationV1`
bytes over libp2p. No fan-out orchestration, no role decision, no gossip (those are siblings
`reactivity-forwarder-host` and `reactivity-pushstate-gossip`). Mirrors `packages/db-p2p/src/cohort-topic/`.

- `protocols.ts` — reactivity protocol family: `REACTIVITY_BASE`, `PROTOCOL_REACTIVITY_NOTIFY`,
  `PROTOCOL_REACTIVITY_PUSH_STATE_GOSSIP` (declared here, consumed by the pushstate-gossip sibling),
  `${BASE}/recover` reserved (named in comment, not defined), plus `ReactivityProtocols`,
  `DEFAULT_REACTIVITY_PROTOCOLS`, `makeReactivityProtocols(networkName)`, `reactivityProtocolList(p)`.
- `notify-transport.ts` — `ReactivityNotifyTransport` / `Libp2pReactivityNotifyTransport`
  (unicast `send`, `onNotification` subscribe, `deliver` decode-and-fan), and `registerNotifyHandler`
  (one-way inbound handler: read one bounded frame → `deliver` → close; abort on read error).
- `index.ts` — re-exports both new modules.

## Review findings

**Approach:** Read the implement diff (`3bb5e56`) with fresh eyes before the handoff summary, then
cross-checked against the cohort-topic siblings it mirrors (`stream-util.ts`, `cohort-gossip-transport.ts`,
`peer-codec.ts`, `protocols.ts`, `host.ts`) and the db-core `NotificationV1` codec (`reactivity/wire.ts`).
Scrutinized SPP/DRY/modularity, error handling, resource cleanup, type safety, and the wire bounds.
Ran db-core build (clean), db-p2p `tsc --noEmit` (clean), and the reactivity unit specs.

**Checked — error handling / failure isolation:** `send` never rejects (dial rejection swallowed +
logged; encode/peer-id errors caught). `deliver` never throws (decode failure logged + dropped).
`registerNotifyHandler` aborts the stream on a read error and never throws on the stream. All correct
and consistent with the cohort-topic one-way handlers. **No issue.**

**Checked — resource cleanup:** stream lifecycle (read → close, or abort-on-error) is delegated to the
shared `sendOneWay` / `readAllBounded` helpers; the handler closes on success and aborts (guarded
double-abort) on failure. **No issue.**

**Checked — type safety / SPP / DRY:** no `any`, no casts in source; framing/dial reused from
`cohort-topic/stream-util.js`; peer-id↔bytes reused from `cohort-topic/peer-codec.js`. The notify
`deliver` decodes (returning `NotificationV1` to subscribers) where the cohort gossip `deliver` passes
raw bytes — a deliberate, correct difference (notify subscribers want the decoded notification).
**No issue.**

**Minor — fixed in this pass:** the `registerNotifyHandler` read-error → `stream.abort` branch had zero
test coverage. Added a unit test (`registerNotifyHandler aborts the stream and delivers nothing when the
bounded read fails`) that drives a frame past a tiny read ceiling so `readAllBounded` rejects, asserting
`abort` fires and no subscriber is delivered. Spec now 10 passing (was 9).

**Reviewed — accepted as designed (no change):**
- *`send` swallows local programming errors (malformed notification / bad peer-id string), not just the
  dead-target dial rejection.* This preserves the load-bearing "one bad target never breaks a
  commit/fan-out" property and is explicitly documented. For a fire-and-forget, hint-only transport this
  is the right tradeoff; surfacing it would weaken the invariant the tests rely on. Left as-is.
- *`deliver` uses the transport's stream-level `maxBytes` (default 512 KiB `DEFAULT_STREAM_MAX_BYTES`) as
  the `decodeNotificationV1` message-codec ceiling, rather than db-core's `DEFAULT_MAX_MESSAGE_BYTES`
  (1 MiB).* Verified safe and internally consistent: `registerNotifyHandler` already bounds the read at
  the same 512 KiB, so the decode bound is ≤ the read bound and never rejects a frame the read accepted on
  size alone. 512 KiB matches the cohort-topic family's stream bound. Left as-is.

**Maintainability note (no action — forwarder-host's wiring responsibility):** the read bound
(`registerNotifyHandler`'s `maxBytes` param) and the decode bound (the transport's `maxBytes`) default
independently. They agree at the defaults (both 512 KiB); a deployment that customizes one but not the
other could create an inconsistent pair (safe — the smaller bound wins — but surprising). The
`reactivity-forwarder-host` ticket that wires both should pass the same `maxBytes` to the transport and to
`registerNotifyHandler`.

**Pre-existing, repo-wide (not this ticket):** doc comments reference `docs/reactivity.md` (and
`docs/cohort-topic.md`), neither of which exists in `docs/` (only `cluster.md`, `repo.md`, `storage.md`).
The db-core reactivity sources already reference `docs/reactivity.md` the same way, so this is a
pre-existing convention/gap across the codebase, not introduced here. No action taken.

**Coverage deliberately deferred (not regressions):** no real-libp2p / on-wire round-trip (all tests use
a fake node) — deferred until `reactivity-forwarder-host` wires `registerNotifyHandler` into an actual
node; self-delivery of a co-located subscriber is the forwarder-host's call (the transport only skips the
self-dial). The existing `mesh-*` reactivity specs predate notify and don't exercise it.
