# Complete: walk / promotion / cold-start (db-core cohort-topic dynamics)

Three FRET-free db-core modules over injected ports (`ITopicRouter`, `TierAddressing`,
`DMaxComputer`, `CohortSigner`, `RegistrationStore`, `ParentRegistrar`, `RegisterMessageFactory`):

- **`walk.ts`** — `WalkEngine.register` walks toward the root from `d_max`, one probe per RPC,
  dispatching `accepted` / `no_state` (inward, root re-issues once with `bootstrap`) / `promoted`
  (the one outward move) / `unwilling_member` (same-coord sibling dial) / `unwilling_cohort`
  (temporal back-off). A `maxSteps` valve bounds malformed-tree oscillation.
- **`promotion.ts`** — `PromotionLifecycle` keyed by `topicId`: eager per-arrival promotion (cap /
  hot-fast-path / slope) and gossip-tick demotion (cap_demote held `T_demote`, no live children,
  `T_promote_sticky` floor, root-never-demotes). Both transitions threshold-signed; methods are
  `async` and return the signed notice.
- **`coldstart.ts`** — `shouldInstantiate = (bootstrap ∨ followOn) ∧ quorumWilling`; the
  `Forwarder` / `ColdStartManager` link-up state machine (`awaiting_parent` → `serving` on parent
  ack; root serves immediately); `promotedRedirectReply` is the just-promoted-burst bounce.

## Review findings

Adversarial pass over the implement diff (commit `9e6e214`), read before the handoff summary.
Scrutinized for SPP/DRY/modularity, scalability, maintainability, performance, resource cleanup,
error handling, and type safety, and cross-checked against the ports/wire/sig modules the change
depends on.

### Checked and found correct (no change)
- **Walk single-direction discipline.** Inward-only on `NoState`; the *only* outward move is a
  `Promoted` redirect (`d = targetTier`, coord recomputed). `dialTarget`/`memberAttempts`/`bootstrap`
  are correctly reset on every non-member transition. The root cold-start re-issue (`d = 0`,
  `bootstrap = true`) and its give-up path are correct.
- **Sibling-retry cycling.** `candidates[memberAttempts % len]` with the `maxMemberRetries` cap
  terminates as a temporal cohort decline — traced against the existing exhaustion test.
- **Promotion triggers.** Cap (`≥ cap_promote`), hot fast-path (`bucket ≥ bucket_overload ∧
  ≥ cap_promote_fast`), and slope extrapolation are correct; slope guards `samples.length < 2`,
  `span ≤ 0`, and `slope ≤ 0`. The growth-window trim keeps ≥1 sample and discards a stale slope
  after a long gap (verified by trace).
- **Demotion hysteresis.** Root-never-demote, sticky window, children-block, `cap_demote` (live
  re-read), and `T_demote` anchor are each enforced; the existing combined test confirms the low-load
  clock stays anchored while children block demotion across the `T_demote` hold.
- **Cold-start gate + forwarder lifecycle.** Truth table, root-serves-immediately, accepts-but-holds
  pre-ack, idempotent-per-topic, and `parentCoord`-required-for-tier>0 are correct.
- **Burst bounce.** `promotedRedirectReply` is always `Promoted`, never `UnwillingCohort`, and
  attaches traffic when supplied; `targetTier ≥ 1` validated.
- **Type safety / FRET-freeness.** No FRET/libp2p imports; ports honored; byte/b64url conventions
  consistent with `wire/types.ts` and `sig/threshold.ts`. `tsc --strict --noUncheckedIndexedAccess`
  clean over src+test.
- **`maxSteps` default** (`2·(d_max+2) + maxMemberRetries + 8`): at `d_max ≈ 6` → 40, generously
  above a legitimate deep walk (~7 inward probes + one redirect + sibling retries). Heuristic but
  adequate; only bites in malformed trees.

### Minor — fixed in this pass
- **Test coverage gaps filled (tests were a floor).** Added three specs for previously-uncovered
  paths:
  - `walk.spec.ts` — `maxSteps` terminates a tree that **alternates `NoState`/`Promoted` at the same
    tier** (the ticket's named adversarial case; had **zero** coverage): asserts `retry_later` with
    exactly `maxSteps` probes and no router-dry throw.
  - `walk.spec.ts` — a `Promoted` redirect with **no explicit `targetTier`** defaults to `d+1` and
    recomputes the coord there.
  - `coldstart.spec.ts` — a **failed parent registration** leaves the forwarder `awaiting_parent`
    (accepts participants, holds parent-ops, no spurious flip to `serving`) and surfaces the failure
    via `console.warn` rather than swallowing it.

### Noted — left as-is, flagged for the binding layer (no code change)
- **Async promote/demote race.** `onParticipantCountChange` checks `!state.promoted` synchronously
  but only sets `state.promoted` *after* awaiting the async threshold signer; two concurrently
  *un-awaited* calls could both emit a notice. Not a defect in db-core (notices are idempotent-ish
  and the lifecycle is single-cohort state), but the db-p2p driver MUST serialize per-topic
  lifecycle calls (await each before the next). Belongs to the activity-handler/gossip-loop contract
  in `cohort-topic-core-module-fret-integration`, not here.
- **`isPromoted` vs cold-started forwarders.** `PromotionLifecycle` (promoted = "redirects to d+1")
  and `ColdStartManager` (tracks forwarders) hold independent state by design; a cold-started
  tier-`d` forwarder is correctly `isPromoted = false` (it serves, not redirects) and may still
  demote once underutilized. Consistent, but the two-state-machine relationship is subtle — the
  binding layer should not assume they share state.

### Major — filed as a new ticket
- **`followOn` derivation is unresolved** (the implement handoff's "single biggest open item").
  `followOn` is not on the wire; until the db-p2p host can derive it, a participant joining a
  freshly-promoted topic whose child tier is uninstantiated oscillates `Promoted ↔ NoState` and only
  `retry_later`s (now regression-tested as *bounded*, not as *resolved*). Filed
  **`tickets/backlog/cohort-topic-followon-derivation.md`** — a design decision (wire field vs.
  host-side routing inference vs. parent-vouched token) that gates a correct cold-start path in
  `cohort-topic-core-module-fret-integration` and may touch the wire format / anti-flood work.

### Deferred gaps confirmed already tracked (not re-filed)
- No db-p2p binding / lifecycle driver / e2e yet (handoff gaps #1, #3): owned by
  `cohort-topic-core-module-fret-integration` (implement) and `cohort-topic-e2e-mock-tier`, both of
  which list this ticket as a prereq.
- Admission-quorum *number* (distinct from `followOn`): owned by

- No cold-start parent-registration retry loop (handoff gap #5): reasonable for now; behavior is now
