# Reactivity recovery: backfill RPC, parent checkpoints, resume protocol — COMPLETE

Recovery beyond the live notification stream: a short gap served from the replay ring (**Backfill**), a
long sleep served from a parent **Checkpoint**, out-of-window (**chain read**), and across a tail
rotation (**TailRotated**). Built on the `PushState` / `replayBuffer` substrate from
`reactivity-origination-replay-delivery`.

The implement-stage work (commit `7cfd489`) was sound: codecs, `serveBackfill`/`serveResume`,
`createBackfillRequester`, `RollingCheckpoint` fed by ring eviction, `applyResumeReply`, `rebaseline`,
the sticky cohort-hint cache, and the db-p2p manager wiring all landed with good unit coverage. Review
found **two reachable correctness/integrity bugs** (both adversarial-path; both masked by the unit tests'
seam shape) and one stale doc comment — all fixed inline with regression tests. Substantial follow-on
(the on-wire RPC transport) was filed as a new ticket.

Final state: **db-core 765 passing**, **db-p2p 643 passing / 9 pending / 0 failing**. `yarn build` clean
for both. (No `lint` script is configured in this repo — `tsc` via `yarn build` is the type gate.)

### Found and fixed inline (minor — localized fixes + regression tests)

1. **Backfill underflow re-entrancy → unbounded amplification** (`backfill.ts`,
   `createBackfillRequester`). On underflow (held window can't reach the gap's low edge) the requester
   replayed the non-contiguous partial entries *before* firing `onUnderflow`. Because the db-p2p manager
   wires the subscriber's `requestBackfill` seam **back to this same driver**, replaying entries that sit
   above an unfillable gap re-fires `gap → requestBackfill` for every entry, each spawning another backfill
   RPC — an unbounded async amplification on a reachable condition (subscriber lag past the ring). The
   db-core unit test masked it (its subscriber had no `requestBackfill` wired). **Fix:** detect underflow
   first, escalate via `onUnderflow`, and skip the replay — aligning with `docs/reactivity.md` §Backfill
   RPC ("it escalates to a checkpoint resume or a chain read"). **Regression test** wires the seam back to
   the requester and asserts one transport call, zero deliveries, no recursion.

2. **Non-abutting verified checkpoint silently skips revisions** (`resume.ts`, `applyResumeReply`).
   `verifyCheckpointEndpoints` proves the bracketing endpoints are real committed revisions but **not**
   that the checkpoint's low edge connects to the subscriber's contiguity head. A malicious/buggy cohort
   could return a verified checkpoint `[100,116]` to a subscriber at `last=11`; `rebaseline(116)` would
   jump the head forward, **silently skipping revisions 12..99**. (This is exactly the implementer's own
   review-focus question #2.) The honest path is unaffected — `classifyResume` only picks
   `checkpoint_window` when the checkpoint covers `lastRevision + 1` — but `applyResumeReply` is the trust
   boundary and applies *any* reply. **Fix:** guard `summary.fromRevision <= subscriber.lastRevision + 1`
   before applying; otherwise chain-read (return `checkpoint_untrusted`, no state advance). **Regression
   test** added.

3. **Stale module doc** (`push-state.ts`): the header comment still described `parentCheckpoint` as
   "reserved … populated by the sibling tickets [reactivity-backfill-resume-checkpoints]" — i.e. as future
   work that *this* ticket performs. Updated to state the rolling `checkpoint` is fed by ring eviction here
   and that only `perSubscriberQueue` remains reserved (for the rotation ticket).

### Found and deferred (major — new ticket filed)

- **No on-wire backfill/resume RPC transport.** db-core's recovery logic and the db-p2p manager are wired
  only through injected `BackfillTransport`/`ResumeTransport` seams; there is no libp2p protocol that
  decodes a `BackfillV1`/`ResumeV1` off the wire and dispatches to the server, no dial path, no canonical
  request-signing payload, and no replay-protection envelope wrapping. This is the **pull/recovery**
  direction; the existing `plan/12.3-reactivity-notification-transport` ticket covers only the **push**
  fan-out + gossip and does not mention backfill/resume. Filed
  **`plan/12.4-reactivity-backfill-resume-rpc-transport`** (prereq: `reactivity-notification-transport`),
  which also absorbs the canonical-signing / replay-envelope gap and the `subscriberCoord` placeholder.

### Accepted caveats (by design; owned by named follow-on tickets, not regressions)

- **Checkpoint is per-member, not gossiped** in `PushStateGossipV1`. Acceptable: each member rebuilds it
  deterministically from the gossiped ring's evictions via the deterministic fold, so members converge. A
  member that never observed the lower revisions simply won't hold them — the resume then classifies
  out_of_window and the subscriber chain-reads. Documented.
- **`W_checkpoint` adaptive hook** (`resolveWCheckpoint`) exists but isn't bound to a live cps measurement
  (default 4096 from the shared config table). Simulator fold-back is pending and out of scope here.
- **`rotationRevision` / `currentTailId` are caller-supplied** to `serveResume`; the rotation lifecycle
  (`reactivity-rotation-backpressure-policy`) owns the real values. Coordination noted in 12.4.
