# Complete: surface invalidation to clients (push + pull)

Library-stage vertical slice for surfacing durable invalidations to clients, building on
`7.5-invalidation-log-entry-and-reversal` and `7.6-invalidation-cascade-detection`. Push marker + reused
cert, durable `committed-invalidated` pull status, and the client drop+resubmit reaction are all built and
unit-tested. The node-side **emit** remains the deliberate composition-root injection point shared with the
parent tickets (see Review findings → carried gaps).

## What shipped (unchanged from implement, verified in review)

- **Push**: `CollectionChangeEvent` / `NotificationV1` carry `invalidation?: boolean` + `invalidatedActionId?`
  (`reqBoolean` added to the wire validator; codec round-trips + rejects non-boolean/non-string markers).
  `buildNotificationV1` threads the marker and reuses the **invalidation's** commit cert bit-for-bit as `sig`.
  `ClusterMember.applyConsensusInvalidation` captures that cert via `buildCommitCert`, keyed by the
  deterministic `invalidationActionId(invalidatedActionId, disputeId)`, gated on the `onCommitCertificate` sink.
- **Pull**: `BlockActionStatus['statuses']` gains `'committed-invalidated'`; `NetworkTransactor.getStatus`
  refines an otherwise-`aborted` slot via `Log.findInvalidation` (durable, restart-surviving; only `aborted`
  slots are checked; `pending` left alone).
- **Client reaction**: `Log.getInvalidationsFrom(startRev)` surfaces reversals since last sync;
  `Collection.updateInternal` drops reverted blocks from the read cache and replays pending work against the
  reverted base; `coalesceInvalidatedActionIds` collapses cascade-child notifications by `invalidatedActionId`.

### Checked — what was scrutinized

  `case '<status>'` / `=== '<status>'` site. The only consumer is `refineInvalidatedStatuses` itself; no

  (tsc build confirms). Verdict: clean.
- **`getInvalidationsFrom` early-break correctness**: it stops at the first entry with `rev <= startRev`,
  unlike `getFrom` (which must walk past the checkpoint to assemble the pending set). Verified revs are
  monotonic across action/checkpoint/invalidation entries, so the early break is sound. Verified the
  rev-bookkeeping does not double-process: `getFrom` seeds `context.rev` from the **tail** entry's rev
  regardless of entry kind, so when an invalidation is the tail, `actionContext.rev` advances past it and the
  next `update` won't re-surface it. Verdict: correct.
- **Client-reaction wiring (Tracker/CacheSource/replayActions)**: confirmed `Tracker` keeps no independent
  read cache (reads pass through `CacheSource` live), so `sourceCache.clear(revertedBlockIds)` alone makes a
  subsequent read re-fetch the reverted base; the pending>0 path additionally `replayActions()` (reset +
  replay). Verdict: correct.
- **Cert capture (cluster-repo)**: capture happens after certificate verification, before the sink, keyed by
  `invalidationActionId` (not the reversed action id); dedup guards a re-receipt; a rolled-back sink failure
  leaves only an idempotent-by-key cert. The reused-cert push path verifies with **real Ed25519** against the
  tail-cohort membership cert and rejects forged/sub-threshold sigs (existing tests). Verdict: correct.
- **Docs**: read both changed docs in full against the code. `docs/right-is-right.md` §Client Notification and

### Found + fixed inline (minor)
- **Vacuous client-reaction assertion** (`test/invalidation-client.spec.ts`). The original
  "drops the reverted block from cache" test ended with `expect(observed?.value).to.not.equal('v2')` after a
  `bump` to `'v3'`. The `bump` handler replaces `value` wholesale (`applyOperation` assigns directly when the
  inserted value is not an array), so `observed.value` is **always** `'v3'` whether or not the cache-drop
  fired — the assertion proved nothing about the feature. Fixed by splitting into two real tests:
  (1) a no-pending `update()` that asserts `equal('v0')` — directly proving the unconditional cache-drop reads
  the reverted base (this would read the stale `'v2'` without the drop); (2) a pending-replay test asserting
  the resubmit lands as a new durable log action entry (`committed.length === 3`) and the collection reads its
  own `'v3'` result, exercising the `pending>0 → anyConflicts → replayActions` branch that test (1) does not.
  Net: 905 → 906 passing.

### Carried gaps (major — NOT closed here; tracked, by design)
- **Node-side emit is not wired** (parent gap #1). Nothing emits a `CollectionChangeEvent` with
  `invalidation: true` from a live node yet; the `InvalidationApplySink` (composition root) must, when applying
  an invalidation, emit with `invalidatedActionId` and `actionId = invalidationActionId(...)` so the bridge's
  cert extractor resolves the captured cert. This is gated behind the certificate-arbitrator-binding fix and
  is already tracked by `tickets/fix/invalidation-certificate-arbitrator-binding.md` (the unforgeability fix
  that must land before any live invalidation wiring). No **new** ticket filed — this ticket explicitly scoped
  the emit out, the cert-capture key and the documented emit contract line up (verified), and the binding fix
  already owns the wiring gate. Until then the push path is proven at the assembler+verify level, and
  `coalesceInvalidatedActionIds` is a correct-but-as-yet-unconsumed helper (push-consumer side not wired).
- **`getStatus` opens a `Log` per aborted ref** (parent gap #2): a status query for an aborted action now
  triggers extra chain reads (cached per `(collection, action)` within a call; faults swallowed → stays
  `aborted`). Acceptable; left as documented. Not promoted to a ticket — it's a known cost trade-off on a path
  that already does network I/O, isolated and best-effort.
- **Client reaction is over-inclusive** (parent gap #3): `updateInternal` replays pending work whenever ANY
  invalidation landed since the client's rev and pending exists, without intersecting the pending read-set
  with the reverted blocks. Safe (over-invalidation only resubmits, never wrongly retains) and matches the
  cascade re-evaluator's stance; left coarse by design.
- **`getInvalidationsFrom` adds a tail walk to every `update`** (parent gap #4): even with zero invalidations
  it walks the tail back to the client's rev. Could fold into `getFrom`'s single pass; kept separate for
  clarity. Left as-is.

All carried gaps were judged minor-cost-trade-off or already-tracked; none warranted a new fix/plan/backlog
ticket beyond the existing `invalidation-certificate-arbitrator-binding` gate.
