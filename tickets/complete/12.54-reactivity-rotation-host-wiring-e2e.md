# Complete — reactivity rotation: origination observe-rotation + node composition + e2e

The capstone of the tail-rotation series landed and passed review. A `cohortTopic`-enabled node observes a
tail rotation (the only live signal available — `event.tailId` changing), arms the outgoing tail's drain via
`forwarderHost.markRotated`, binds the recover serve's `rotationFor` to `rotationRedirectFor`, and constructs
+ exposes an unref'd-timer `RotationReRegistrationScheduler` (`node.reactivityRotation`). The full redirect →
re-register → continue path is proven end-to-end over the mock-tier mesh; the scheduler's `reRegister` move
remains a logged no-op until the deferred subscribe factory lands (by design, mirrors 12.33/12.42).

## What shipped (unchanged from the implement handoff)

- **A. Origination observes the tail-id change** (`origination-manager.ts`): per-collection last-seen tail
  anchor; on a change fires `markRotated(oldTopicId, { newTailId, effectiveAtRevision: event.rev }, now)` with
  `oldTopicId = reactivityTopicId(b64urlToBytes(lastSeenTailB64))` — byte-identical to the topic a subscriber
  subscribed under. First commit records baseline; tail-less commit never records/clears. `markRotated` +
  `blockFill` are optional (absent ⇒ origination byte-for-byte unchanged). Observe path isolated in its own
  try/catch so it can never block the notification emit. Block-fill warm-up/filling are logged-only on a live
  node (successor tail id not knowable; gated on `6.5-block-id-derivation`).
- **B. Node composition** (`libp2p-node-base.ts`): `markRotated` bound to the forwarder host; recover serve's
  `rotationFor` resolves the stale topic for a resume (`req.topicId`) or the collection's current served topic
  for a backfill (new `resolveCurrentServedTopic` helper) then consults `rotationRedirectFor`; the scheduler
  constructed with the default unref'd `setTimeout`, exposed as `node.reactivityRotation`, and torn down at the
  top of the reactivity stop wrapper.
- **C. Mock-tier e2e** (`reactivity-mesh-harness.ts` + `mesh-tail-rotation.spec.ts`): `rotateTail(coll, {
  autoReattach: false })` arms a real `TailDrainGate`; a stale resume raises `RotationRedirectError` while
  draining; a per-subscriber scheduler drives re-attach over the virtual clock. Two new tests — redirect-driven
  re-registration with no gap (continuous 1..12), and cross-rotation resume from the inherited checkpoint
  (`checkpoint_window`, not `out_of_window`). `rotateTail` is now async (re-seeds the new cohort's willingness);
  all callers updated to `await`.
- **D. Real-libp2p tier**: tagged `it.skip` with a deferral note (a real two-tail rotation exceeds an
  agent-runnable budget at this small-N tier).
- **Docs**: `docs/reactivity.md` §Tail rotation + `docs/internals.md` reactivity-wiring bullet updated to
  "Tail rotation is now live."

## Review findings

**Scope reviewed:** the full implement diff (`fc8dbbd`) with fresh eyes before the handoff summary — the
origination observe-rotation path, the node composition, the recover-serve `rotationFor` binding, the mesh
harness model, all new/changed tests, and both doc files. Cross-checked the unchanged supporting code the diff
relies on (`forwarder-host.ts` `markRotated`/`rotationRedirectFor`/`pushStateForCollection`,
`recover-transport.ts` resume/backfill `rotationFor` call sites, `rotation-rereg-scheduler.ts`,
`PushState.topicId` encoding).

**Correctness / encoding (the load-bearing pin):** Verified `oldTopicId = reactivityTopicId(b64urlToBytes(
bytesToB64url(ctx.tailId)))` collapses to `reactivityTopicId(reactivityTailBytes(tail))` in production
(`resolveContext` sets `tailId: reactivityTailBytes(event.tailId)`), byte-identical to both the topic a
subscriber subscribes under and the resume's `staleTopic` in the recover serve. The `managers.spec.ts` test
asserts exact byte equality and would catch a regression to the double-hashing `blockIdToBytes`. ✅ No issue.

**Resume vs backfill redirect binding:** Confirmed the node's `rotationFor` uses `req.topicId` (the resume's
`staleTopic`) when present and falls back to `resolveCurrentServedTopic` (highest-`lastRevision` served topic)
for a backfill — matching `recover-transport.ts`'s `serveResumeReply`/`serveBackfillReply` call sites. The
drain-window behavior (old tail redirects while it is the sole served state; the new tail resolves once served,
no gate → no redirect) matches the recover-serve doc. ✅ No issue.

**Tests:** `yarn build` (tsc) exit 0; `test/reactivity/**/*.spec.ts` → **145 passing**;
`test/cohort-topic/host-node-activation.spec.ts` → **7 passing**. Covered: happy path (no-gap continuity),
edge (cross-rotation inherited checkpoint, baseline/same-tail/tail-less observe cases), error path (markRotated
absent ⇒ inert), teardown (scheduler refuses to schedule after `stop()`), regression (all prior rotation tests
still green with `rotateTail` now async — every caller awaits). ✅

**Resource cleanup / teardown:** scheduler `stop()` runs before `host.stop()`; the unref'd timer never pins the
process; node-wiring test asserts `pendingCount` returns to 0 and a stopped scheduler arms nothing. Harness
`stop()` tears down every subscriber's scheduler. ✅ The `lastSeenTail`/`fillTrackers` maps grow per-collection
(bounded by collection count, no per-commit growth) — acceptable, not a leak.

**Minor — fixed in this pass:** The register-NEW-before-unregister-OLD swap ordering for the deferred subscribe
factory (handoff gap #6) was spelled out only in `libp2p-node-base.ts` source comments and `docs/reactivity.md`
— the named landing-site backlog ticket (`optimystic-network-reactive-watch-integration-test`) said nothing
about the rotation `reRegister` wiring or the swap ordering. Added an "Also lands here" section to that backlog
ticket capturing both the `onRotation → schedule` binding and the swap-ordering requirement, so the contract is
not lost when the factory is implemented.

**Minor — noted, not fixed (negligible):** On a non-redirected backfill, `pushStateForCollection` is scanned
twice — once inside the node's `resolveCurrentServedTopic` (via `rotationFor`), once again in
`serveBackfillReply` after the redirect returns `undefined`. The served map is small; the cost is negligible
and not worth threading the resolved state through the `rotationFor` seam.

**Considered, not a defect (by design, documented):** In production the node that *fires* `markRotated` (the
new tail's originator) may not be the node that *serves* recover requests for the old tail (the old cohort), so
the drain gate is not guaranteed to be armed on every old-tail serving member. This is acceptable: the recover
redirect is an explicitly **best-effort secondary path** (the recover-serve doc states the primary mechanism
for an active subscriber is notify-driven detection); a subscriber that misses the redirect falls back to
notify-driven rotation detection or a chain read. No correctness hole — left as-is.

**Honest coverage gaps carried forward (deferred, not defects):**
1. The production socket path for rotation is not exercised end-to-end — the node exposes the scheduler but
   `reRegister` is a logged no-op until the subscribe factory lands. Proven only at the mock tier + unit level.
   Tracked by `optimystic-network-reactive-watch-integration-test` (now annotated with the wiring contract).
2. The real-libp2p two-tail rotation is `it.skip` (§D) — beyond an agent-runnable budget; a human/CI runs it
   out-of-band.
3. The mesh harness models `markRotated` with a `TailDrainGate` (not a real `ReactivityForwarderHost`) and its
   `reRegister` re-attaches the same manager rather than constructing a fresh one under `plan.newTopicId` — the
   equivalent no-gap move, but it does not exercise the registry swap. By design ("or model it" per the
   originating ticket); the registry swap is the deferred factory's responsibility (now captured in the backlog
   ticket).

**Pre-existing flaky test:** `mesh-slow-subscriber.spec.ts` (Edge subscriber, `CohortBackoffError`) was
