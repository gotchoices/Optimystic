# Complete — reactivity rotation: re-registration timer host seam (db-p2p)

`RotationReRegistrationScheduler` is the standalone, unit-testable host that schedules the jittered move of a
subscriber to a rotated tail. The `ReactivitySubscriptionManager` surfaces a `RotationNotice{ newTailId,
preAnnounced, plan: ReRegistrationPlan }` once per successor; the scheduler de-dupes by successor `newTopicId`,
arms a one-shot timer for `max(0, plan.fireAt − now())`, and on fire invokes an injected `reRegister(plan)`. The
node composition that binds it to the manager's `onRotation` observer remains deferred to
`12.54-reactivity-rotation-host-wiring-e2e` (by design).

### Found + fixed inline (minor)
1. **Inaccurate "load-bearing jitter contract" (the implement diff's centerpiece claim) — corrected.** The diff
   asserted, loudly, in `docs/reactivity.md`, the module JSDoc, and the class JSDoc, that the composing site MUST
   build the manager's `rejoinJitter` as `createRejoinJitter({ capPromote: DEFAULT_CAP_PROMOTE_FAST })` or the
   rotation burst "silently overruns `cap_promote_fast`." This is **wrong for the path the manager actually uses**:
   the manager calls `planReRegistration` → `RejoinJitter.scheduleRejoin`, which draws a **uniform** offset over
   `T_rejoin_jitter` and **never consults `capPromote`** (`jitter.ts:93-96`). `capPromote` only governs the *wave*
   planner `scheduleWave` / `planReRegistrationWave` (used by the mesh harness, not the production manager). Per
   the docs' own §Rotation cost / Worked scenario, the per-subscriber path is window-bounded (≈ `subscribers /
   T_rejoin_jitter` arrivals/s) and the new-tail burst is absorbed on the *receiving* side by the new tail
   cohort's `cap_promote_fast` fast-promotion — a separate cohort-topic mechanism. Setting `capPromote` on the
   manager's jitter would be a silent no-op. Rewrote the contract in all three places (and the `fire()` recovery
   comment) to describe reality: the load-bearing knob is the **window**; `capPromote` matters only if the
   composing site deliberately adopts the wave planner.
2. **Propagated the correction to the downstream ticket.** `12.54-reactivity-rotation-host-wiring-e2e` (still in
   `implement/`) carried the same false "Load-bearing: MUST build `createRejoinJitter({ capPromote:
   DEFAULT_CAP_PROMOTE_FAST })`" instruction in two places + a docs note. Corrected them so the wiring implementer
   does not add a no-op comment / wire on a false premise.
3. **`fire()` recovery comment overstated recovery.** It claimed a failed move is "recovered when the next
   notification re-detects the (still-current) rotation" — but `seen` (and the manager's `rotationHandledFor`)
   de-dupe a re-notice for the same successor, so that is **not** a recovery path. Corrected to name the real
   backstop: the subscriber's recover/re-walk path. (Confirms the accepted "no retry this pass" decision is sound,
   but the stated mechanism was inaccurate.)
4. **Test gap — added one case.** The implement spec proved isolation of a failing seam but not that `seen`
   integrity holds on the **failure** path. Added *"a re-notice after a FAILED move stays deduped (no implicit
   retry to the same successor)"*, which also pins the corrected recovery semantics. Suite 13 → 14; full reactivity
   139 passing.

### Found + filed as new ticket (major)
- **Unbounded `seen` ledger growth.** On the normal schedule→fire path a successor key stays in `seen` forever
  (pruned only by `cancel`/`stop`). One scheduler per long-lived subscription means a very-high-frequency-rotation
  subscription slowly accumulates unreclaimed memory. Negligible at typical cadence (~22 keys/day for a 64-min
  rotation), real for seconds-scale rotation over a long-lived subscription. A naive "delete on fire" is **wrong**
  (the persistence guards superseded-successor re-surface), so the fix needs a bounded/TTL ledger — a design
  choice → filed `backlog/reactivity-rotation-rereg-seen-ledger-unbounded.md`.

### Not found (explicitly)
- **No correctness bug on the common path.** The scheduler behaves exactly as specified for fire-at-`fireAt`,
  past-`fireAt` clamp-to-0, dedup, cancel/stop teardown, and seam-failure isolation.
- **No DRY / dead-code issues** beyond the intended `defaultSetTimer` mirroring of the gossip driver.
- **No security / injection surface** — the module schedules timers and calls one injected seam; no I/O, no
  parsing, no untrusted input beyond opaque `Uint8Array` topic ids used only as map keys.
