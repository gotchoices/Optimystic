## Summary

A transaction's `stamp` carries the metadata a validating node trusts (`peerId`,
`timestamp`, `schemaHash`, `engineId`, `expiration`, `nonce`). `stamp.id` is a SHA-256
of exactly those six fields; the client signature is made over `stamp.id` + statements +
reads, NOT over the raw fields. Nothing re-verified that `stamp.id` actually equals the
hash of the fields, so an attacker who captured a validly-signed transaction could mutate
a stamp field (headline: extend `expiration` to defeat the TTL replay window) while holding
`stamp.id` and `signature` constant, and it still validated.

The implement stage added a shared pure helper `computeStampId(fields)` used by both
`createTransactionStamp` and a new validator **step 0.0** that re-derives the id and
rejects a mismatch with reason `Tampered transaction stamp`. Always-on (one SHA-256, not
gated on the signature verifier), ordered first because the very next step (expiration)
already consumes an untrusted field. Hash formula and `clientSignaturePayload` format are
byte-unchanged, so existing stamp ids and signatures stay valid.

## Review findings

**Verdict: implementation is correct, well-tested, and well-documented. No major findings;
no new tickets filed. Minor doc drift fixed inline.**

Checked and clear:
- **Single validation path.** `TransactionValidator.validate` is the only `validate`
  implementation; `createQuereusValidator` (quereus-plugin) merely constructs a
  `TransactionValidator`, so it inherits step 0.0 for free. No sibling validator bypasses
  the check.
- **No un-covered stamp-id construction.** Every stamp id in the codebase is produced by
  `createTransactionStamp` → `computeStampId`; no inline `stamp:`+hash remains. The
  coordinator/session `isTransactionExpired` calls (coordinator.ts, session.ts) operate on
  a node's own self-built transaction, not an untrusted received one — not an attack
  surface, correctly untouched.
- **Defensive helper design.** `computeStampId` destructures the six fields and rebuilds
  the object in fixed order before `JSON.stringify`, so passing the full `TransactionStamp`
  (with its `id`) — as the validator does — or any object with stray extra properties cannot
  perturb the serialized bytes. Type-safe: `TransactionStamp` is assignable to
  `StampFields = Omit<TransactionStamp,'id'>`.
- **Test coverage (happy/edge/error/regression).** 102 passing. Tamper-after-signing cases
  for `expiration` (headline replay guard), `engineId`, `schemaHash`, `timestamp`; a
  legit-signed regression (`valid:true`); a no-verifier tamper proving the check is not
  gated on signatures; the reworked peerId test that recomputes `stamp.id` to reach the
  signature layer (peerId's second line of defense). Both layers meaningfully exercised.
- **Ordering.** Step 0.0 precedes expiration; the `timestamp`-tamper and future-`expiration`
  tests confirm integrity fires ahead of the expiration verdict.

Found and fixed inline (minor):
- **Stale interface doc.** `ITransactionValidator.validate`'s step list (transaction.ts)
  enumerated only steps 1–7, omitting the integrity check plus the expiration/signature
  steps added by earlier tickets. Updated to list steps 0.0 / 0 / 0.5, pointing at
  `TransactionValidator.validate` as authoritative.
- **Stale `docs/transactions.md`.** Three places described `stamp.id` as a hash of only
  four fields (peerId, timestamp, schemaHash, engineId) — pre-existing drift from before
  `expiration`/`nonce` were added, and directly relevant to this security ticket's field
  set. Corrected all three to the six fields, added `nonce` to the `TransactionStamp` code
  block, and documented the new re-derive-and-reject integrity check.

Empty categories (explicit):
- **New tickets: none.** No major/latent defect surfaced — the change is a purely additive
  verification with no behavioral regression to existing valid transactions.
- **Tripwires: none.** No conditional "fine-now-until-X" concern was found; the check is
  O(1) per validation and does not scale with transaction size.

Noted, no action (implementer's flags confirmed reasonable):
- `nonce` has no dedicated tamper test. It feeds the hash and is covered by the mechanism;
  the four field tests + no-verifier test already prove step 0.0 catches any single-field
  mutation. A `nonce` case would be redundant — agreed, not added.
- Statement/read tampering remains covered only by the signature path (they are not stamp
  fields, so integrity does not touch them). Unsigned + no-verifier posture still accepts
  statement/read tampering — unchanged by this ticket, out of scope. Real Ed25519
  verifier wiring is `implement-client-tx-signature-p2p`.
- Non-canonical read ordering in `createTransactionId` remains deferred to
  `design-consensus-hygiene-notes` — unrelated to this ticket.
