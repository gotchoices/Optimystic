## What was built

Documentation-only, no behavior change:

- Added a one-line comment above `partitionDetector?: PartitionDetector;` in the `ClusterMemberComponents` interface (`packages/db-p2p/src/cluster/cluster-repo.ts:44`) marking the field as reserved for the partition-healing consumer (backlog ticket `6.5-partition-healing`).
- Added a matching one-line comment above the `_partitionDetector?: PartitionDetector` constructor parameter (`packages/db-p2p/src/cluster/cluster-repo.ts:100`) noting it is held but not yet consumed. The leading underscore stays so the unused-parameter lint rule is satisfied.
- Expanded `tickets/backlog/6.5-partition-healing.md` with a **ClusterMember integration points** section listing the three plausible consumption points (`processUpdate` / `handlePromiseNeeded`, `propagateIfNeeded`, status reporting via `detectPartition()`) and the rename-back step from `_partitionDetector` to `partitionDetector` (with `private readonly`) when the future implementer wires it up.

## Decision context

Option A from the source plan ticket: keep the parameter, document intent. Option B (drop now, re-add later) was rejected because `tickets/backlog/6.5-partition-healing.md` already names `cluster-repo.ts` as a target file and the dependency is wired up at every call site (`libp2p-node-base.ts:293,311`, `network-manager-service.ts:73,98`, plus four test harnesses) — removing it would have churned five callers for a parameter we expect to re-add.

- No new tests; the change is comment-only.

## Usage notes

When the partition-healing backlog ticket (`6.5-partition-healing`) is picked up, the implementer should:

1. Rename `_partitionDetector` → `partitionDetector` and add `private readonly` to the constructor parameter so it's stored on `this`.
2. Wire consumption at one (or more) of the three documented integration points.
3. Remove the two reserved-for comments — they become stale once the field is read.
