# Registration storage, primary/backup sharding, TTL renewal/failover — completed

New module `packages/db-core/src/cohort-topic/registration/` implementing the local soft-state
registration store, deterministic `assignSlots`, participant/cohort TTL renewal, and the
membership-rotation primary-handoff state machine. All transports are injected; db-core stays
FRET/libp2p-free (the `no-fret-import` guard passes). `yarn build` (tsc) and `yarn test` are green
for db-core: **403 passing** (one regression test added in review).

See commit `5500f81` (implement) for the original handoff summary. The cross-member replication and
FRET binding remain out of scope (downstream tickets `cohort-topic-gossip-membership-certs` and
`cohort-topic-core-module-fret-integration`).

## Review findings

### Method

Read the full implement diff (`git show 5500f81`) with fresh eyes before the handoff summary, then
read every touched source and test file plus the wire types (`wire/types.ts`), wire codec
(`wire/codec.ts`), and `ports.ts` (`IRingHash`) that the module depends on. Re-derived the sharding
math, the TTL boundary semantics, the participant failover state machine, the cohort-side
`onRenew`, and the rotation handoff against `docs/cohort-topic.md` §TTL and renewal / §Primary and
backup sharding / §Membership rotation and `docs/internals.md` §Registration-record lifecycle. Ran
`yarn build` + `yarn test` (402 → 403 passing). No lint configured for the repo (root `lint` is a
no-op echo); `tsc` is the type gate and passes.

### Verified correct (checked, nothing found)

- **Sharding math** — `modK` is correct MSB-first full-digest modulo with no bigint and no Number
  overflow for any realistic `k`. Backups are distinct from the primary and from each other
  (offsets `1..min(2, k−1)` mod `k` never collide with slot 0), wrap correctly, and cap at `k−1`.
  Determinism is order-independent (input members are sorted). Distribution test (16k over 16)
  lands within ±30%. Empty cohort throws.
- **Store** — double indexing is consistent; `put` replaces; `delete`/`evictStale` prune empty
  topic maps; map mutation during iteration in `evictStale` is the legal delete-current-entry form.
  `evictStale`/`sweepStale` use the strict `now − lastPing > ttl` boundary, tested at the exact
  boundary (`== ttl` not stale, `ttl+1` stale).
- **TTL renewal participant** — success resets the failure count; promotion fires on *exactly* the
  3rd consecutive failure and promotes `backups[0]` only; the `cohortEpoch` hint is held at
  failover and refreshes lazily on the next `primary_moved`. Both resolved open questions
  (dual-serve-until-ack, lazy epoch refresh) are implemented and annotated in the docs.
- **Cohort `onRenew`** — touch+gossip on the computed primary; `unknown_registration` when absent;
  `primary_moved` (carrying `newPrimary`/`newBackups`/`cohortEpoch`, base64url round-trips through
  the codec) when self is no longer the computed primary; the optional `isServing` dual-serve
  override is wired and tested.
- **Rotation handoff** — 40 records survive a full epoch rotation with each landing on its new
  primary; pulled records are re-stamped (`primary`/`backups`) so a subsequent rotation recognizes
  the new holder; old primary dual-serves until ack then stops; no-op when the slot doesn't move;
  `pulling` set prevents a duplicate inventory from double-pulling.
- **Layering / type decisions** — peer ids as raw `Uint8Array` (matching `PeerRef.id`) rather than
  re-exporting the structural `network/types.ts` `PeerId` is correct and avoids an ambiguous barrel
  export. Docs (`cohort-topic.md`, `internals.md`) were updated to reflect the new module and both
  resolved questions, and match the code.

### Minor — fixed in this pass

- **Relookup hammering (`renewal.ts`).** When the primary *and* all backups are dead, `failover()`
  called `transport.relookup()` but did not reset `consecutiveFailures`. Because `relookup()`
  returns `void` and does not update the participant's record (it still points at the dead nodes),
  every subsequent `pingLoop()` immediately re-failed → re-ran `failover()` → re-ran the expensive
  `d_max` relookup walk on **every** ping cycle. Fixed by resetting `consecutiveFailures = 0` after
  `relookup()` (mirroring the reset `promote()` already does), so a still-dead primary backs off to
  one relookup per `MAX_PING_FAILURES` cycles. Added regression test
  *"backs off relookup to one per MAX_PING_FAILURES cycles when everything stays dead"* (6 failing
  pings → exactly 2 relookups). 402 → 403 passing.

### Major — filed as a new ticket

- **Crash-failover ↔ `onRenew` incoherence** → `tickets/plan/9.7-cohort-topic-crash-failover-reattach-acceptance.md`
  (prereq `cohort-topic-gossip-membership-certs`).

  The participant side and the cohort side were unit-tested in isolation with mocks and are
  inconsistent when composed. A backup-promotion *re-attach* is sent as a plain `RenewV1`
  (no re-attach discriminator), and the only inbound handler is `onRenew`. On a crash the
  `cohortEpoch` is unchanged, so `assignSlots` still names the dead member as primary; `onRenew`
  therefore replies `primary_moved { newPrimary: <dead primary> }` to the promoted backup. The
  participant promotes the backup locally but discards that payload, and its next ping bounces
  `record.primary` back to the dead node — an indefinite loop until membership rotation changes the
  epoch. Cross-member replication gives the backup the record but does **not** fix the redirect
  (the backup still computes itself as not-the-primary). Resolving it needs a design decision (a
  re-attach signal on the wire vs. an `onRenew` "I am a live backup, primary unreachable → accept"
  path vs. relying on fast rotation), so it is a plan ticket rather than an inline fix.

### Known gaps left in place (acknowledged by the implementer, no action — correct boundaries)

- **No cross-member replication / no FRET binding** — owned by `cohort-topic-gossip-membership-certs`
  and `cohort-topic-core-module-fret-integration`. The `RenewalGossip`/`HandoffTransport`/
  `RenewalParticipantTransport` interfaces are the injection seams.
- **No signature verification** — `onRenew` trusts the decoded `RenewV1`; signing is a separate
  layer. Acceptable boundary for the local store.
- **Handoff partial-failure / no ack timeout** — a dropped ack leaves the old primary dual-serving
  indefinitely; documented as a concern for the replication ticket. Not in scope here.
- **Stale `appState`/`lastPing` on handoff pull** — the record is moved as-is and only
  `primary`/`backups` are re-stamped. Semantically the application state should carry over; the
  `lastPing` staleness is bounded (rotations move live records and a fresh ping follows). Left as a
  documented gap.
