# Cohort-topic anti-flood + anti-DoS defenses (db-core substrate)

The anti-flood and anti-DoS sections of the cohort-topic spec were folded into transport-agnostic,
injection-driven `db-core` logic under `cohort-topic/antiflood/` and `cohort-topic/antidos/`, matching
the module's conventions (ES modules, tabs, no `any`, factory + injected-deps, byte-array peer ids).

## Anti-flood (`cohort-topic/antiflood/`)
- **`RejoinJitter`** (`jitter.ts`) — staggers a post-cohort-failure re-registration wave.
  `scheduleRejoin` draws a uniform offset (bound in expectation); `scheduleWave` spaces arrivals at
  `T_rejoin_jitter / cap_promote` so any window holds ≤ `cap_promote` arrivals by construction. Window
  widens with the observed FRET failure rate. Defaults `T_rejoin_jitter = 30 s`, `cap_promote = 64`.
- **Invariant predicates** (`invariants.ts`) — pure, dependency-free checks over a recorded walk
  trace: `outwardMovesArePromoted` (claim 3), `inwardStepsFollowNoState`, `retriesRestartAtDMax`
  (claim 4), `stickyHolds` (claim 5). Mirror the simulator's `walk-metrics.ts` instrumentation.

## Anti-DoS (`cohort-topic/antidos/`)
- **`RegisterRateLimiter`** — sliding-window counter per `(peer, topic)`, default 4/60 s; over-rate ⇒
  `{ ok:false, retryAfterMs }` with the willingness back-off curve; strikes reset after a quiet window.
- **`TopicBudget`** — LRU cap `topics_max = 2048`; evicts the coldest zero-participant resident, refuses
  a new topic when all residents are populated. Deterministic monotonic-seq LRU (no wall clock).
- **`CorrelationReplayGuard`** — drops stale / implausibly-future timestamps and replayed
  `correlationId`s; remembers ids for one `maxAge` window with amortized pruning.
- **`BootstrapEvidence`** — tier policy over injected verifiers: T0/T1 require a signed parent ref;
  T2/T3 accept PoW OR reputation signature OR parent ref; non-bootstrap always admitted.

Docs: `docs/cohort-topic.md` §Anti-flood / §Anti-DoS implementation callouts + the `F^d` open
reputation-escalation question; `docs/debugging.md` cohort-topic DEBUG sub-namespaces.

Build + tests green for `db-core`: `yarn build` clean, `yarn test` **533 passing** (was 531; +2 added

## Review findings

Adversarial pass over commit `05faf5f`. The implement-stage diff was read first with fresh eyes,
then the handoff. Checked against SPP/DRY, modularity, scalability, maintainability, performance,
resource cleanup, error handling, type safety; tests audited for edge/error/regression coverage; every
touched (and should-have-touched) doc re-read.

### What was checked and is sound (no action)
- **Type safety / build.** Strict `tsc` (`noUncheckedIndexedAccess`, no `any`, no inline `import()`);
  builds clean. All cross-module deps (`backoffRetryMs`, `recordKey`/`bytesKey`,
  `DEFAULT_T_PROMOTE_STICKY_MS`, `RegisterV1`/`RegisterResult`) resolve and are used per their contracts.
- **DRY / reuse.** Back-off curve reused from `willingness.ts` rather than re-derived; map keys reuse
  `registration/bytes.js`; sticky window re-exported from `promotion.ts`. No duplication.
- **Error handling.** Every factory validates its config (`RangeError` on non-positive rate/window/cap,
  `failureRateScale < 1`, non-integer counts). Good defensive surface for injection-driven primitives.
- **Resource cleanup / memory.** Replay-guard pruning is amortized and bounds memory to ~2 windows;
  topic-budget is hard-bounded by `topics_max`; rate-limiter per-key history is `O(ratePerWindow)`.
- **Anti-flood bound.** `scheduleWave` even-spacing holds ≤ `cap_promote` per window even with the
  `Math.floor` rounding — re-derived and confirmed by the sliding-window storm test (2,000 → ≤64).
- **Bootstrap tier matrix.** T0/T1 parent-ref-only (PoW+reputation alone insufficient), T2/T3 any-of-
  three, none ⇒ reject, non-bootstrap ⇒ accept — all covered and correct.

### Minor findings — fixed inline this pass
1. **`rate-limiter.ts` docstring was false.** It claimed "idle keys are pruned on access to bound
   memory," but `check()` never deletes a key from the `states` map (only trims each key's accept
   array). Corrected the doc to describe the real behavior (per-key history is `O(ratePerWindow)`;
   global eviction of long-idle keys is the host service's lifecycle concern). No behavior change —
   the primitive is unconsumed, and the map is bounded by the cohort's working set.
2. **`replay-guard.ts` dead field.** `SeenEntry.peer` was recorded but never read. Made it useful:
   the replay-rejection log now attributes the replay to the *original* accepter's peer (the
   correlationId, not the peer, is the anti-replay key, so a spoofed source is still caught). Improves
   anti-DoS observability instead of carrying dead weight.
3. **`antidos.spec.ts` misleading comment.** Called `t[1]` "least-recently used" when it was the
   *most*-recently touched; the test passed only because participant-count is the primary eviction key.
   Comment corrected to state why the eviction holds.
4. **Test gap — topic-budget LRU tiebreak (added).** Every existing test had at most one zero-
   participant resident, so the `seq` LRU tiebreaker in `coldestEvictable` was never exercised — a
   regression in it would have passed. Added a test with three zero-participant residents asserting the
   least-recently-used (lowest seq) is the victim.
5. **Test gap — replay-guard prune path (added).** The `maybePrune` deletion path was never asserted.
   Added a test: a correlationId is re-admitted with a fresh timestamp a full window later (its aged
   record was pruned), while an immediate exact replay is still rejected.

Result after fixes: `yarn build` clean, `yarn test` **533 passing**.

### Major finding — filed as a separate ticket
- **Scope leak: an unrelated quereus-plugin feature was bundled into the implement commit.** Commit
  `05faf5f` also contains a deferred-DML staging refactor in `packages/quereus-plugin-optimystic`
  (`TransactionBridge.markDirty`/`dirtyTrees`, `optimystic-module.ts`, `schema/index-manager.ts`,
  `collection-factory.ts`) plus its db-core enablers `Collection.discardPending` and
  `Tree.stage`/`sync`/`discardChanges`. These are **unrelated to anti-flood/anti-DoS** (almost
  certainly a stray `git add -A` over in-flight work). They compile (`quereus-plugin-optimystic` builds
  green) and are internally coherent, but: (a) had no owning ticket, and (b) a comment inside
  `txn-bridge.ts` self-acknowledges that the **session/consensus commit path is not covered by any
  real-DML test** and points to a `fix/optimystic-session-mode-commit-composition` ticket that **did
  not exist**. Filed `tickets/fix/optimystic-session-mode-commit-composition.md` to own the
  verification + missing coverage. Not reverted: the code compiles, is already in `main`, and appears
  to be wanted work — un-bundling it from history is out of scope and riskier than tracking it forward.

### Noted, acceptable as-is (no action needed)
- **Invariants not auto-wired into `WalkEngine`.** Predicates run on caller-supplied traces, validated
  against hand-authored traces only. This is the documented gap; the natural home — recording a real
  trace from the engine — is the existing `13-cohort-topic-e2e-mock-tier` ticket (confirmed present in
  `implement/`). Deferral is sound.
- **Claim-4 invariant differs subtly from the simulator.** db-core's `retriesRestartAtDMax(traces[])`
  asserts each *separate* re-registration walk starts at `d_max`; the simulator's
  `unwillingRetriesRestartAtDMax(trace)` asserts within-trace post-`unwilling_cohort` probes restart at
  the top. Different-but-defensible decomposition — the real engine re-registers as a fresh walk after
  a time back-off, so a per-walk first-probe check is the right shape for db-core. The "mirrors the
  simulator" docstring is slightly loose (the simulator also lacks `inwardStepsFollowNoState` /
  `stickyHolds`), but the predicate is correct. Not worth churning.
- **`F^d` aggregate DoS** remains an open reputation-escalation question, deliberately deferred to the
  simulator adversarial scenario. Doc framing is accurate and acceptable.
- **`RejoinJitter` / `RegisterRateLimiter` have no production caller yet.** Expected — they are db-core
  primitives awaiting the db-p2p `CohortTopicService` host; only specs exercise them today.
