# Complete: tier addressing (coord_d), tier ladder, and d_max

Three pure-logic db-core modules implementing the cohort-topic addressing substrate:

- **`addressing.ts`** — `TierAddressing` / `HashTierAddressing` (factory `createTierAddressing`,
  default `F = 16`). `coord_0 = H(0x00 ‖ topicId)`, `coord_d = H(d ‖ prefix(P, d·log₂F) ‖ topicId)`
  for `d ≥ 1`, hashing through the injected `IRingHash`. `coord(d,…)` dispatches `d===0` to
  `coord0`. `prefixBits(P, n)` extracts the `n` MSBs MSB-first into `⌈n/8⌉` bytes. Guards: `F`
  power-of-two `≥ 2`, `1 ≤ d ≤ 255`.
- **`tiers.ts`** — `Tier` (T0..T3 as a const-object + value-union), `ALL_TIERS`, `NodeProfile`,
  `edgeProfile`/`coreProfile`. Edge base `{T0,T1}`, Core base `{T0,T1,T2,T3}`; operator overrides
  can only **narrow** (intersect with base). Static flags only — no willingness/TTL behaviour.
- **`dmax.ts`** — `makeDMaxComputer({ estimator, F, confidenceMin?, dMaxCap? })`. Reads the injected
  `ISizeEstimator` lazily, returns `max(0, ⌊log_F(n_est)⌋ − 1)` capped at `dMaxCap`, clamped to
  `⌊dMaxCap/2⌋` when `confidence < confidenceMin`. Defaults `0.3` / `60`.

## Review findings

**Scope reviewed:** the implement-stage diff (`e788db1`) read fresh before the handoff; all three
source modules + their specs; the consumed ports (`ports.ts`, `ring-hash.ts`); the simulator
ground-truth (`packages/substrate-simulator/src/size-model.ts`); and every touched doc section.
Build + full db-core suite re-run.

**Correctness — no bugs found.** All three modules faithfully implement the `docs/cohort-topic.md`
formulas **and** match the simulator's validated reference:
- `dmax.ts`'s confidence clamp returns `Math.floor(dMaxCap/2)` exactly as the simulator's
  `computeDMax` (`size-model.ts:33-34`). The db-core `floorLogF` is in fact **more precise** than
  the simulator (it adds a power-of-`F` boundary correction the simulator's raw `Math.floor(logF)`
  lacks), so exact powers like `16³` never lose a tier to float error. Verified against the
  powers-of-`F`, tiny-network, clamp-boundary, custom-config, cap, and lazy-reread specs.
- `addressing.ts` `coord_d` byte layout (`d ‖ prefix ‖ topicId`) is unambiguous: the leading
  tier byte differs across tiers, the prefix length is deterministic per `d`, and `topicId` is a
  fixed-width hash — so no length-framing delimiter is needed. The collision spec enumerates 1,536
  coords and observes 0 aliases, mirroring the simulator finding.
- `tiers.ts` override narrowing uses set-intersection over the base order; Edge can never be widened
  to T2/T3. Confirmed by the override specs.

**`enum` → const-object substitution — VALIDATED as necessary (not just acceptable).** The runtime
is Node **v24.2.0**, where native type stripping is default-on for `.ts` and the test run emits
`ExperimentalWarning: Type Stripping`. Strip-only mode rejects non-erasable `enum` syntax, so a
literal `export const enum Tier` (as the spec wrote) would abort the suite. The const-object +
value-union form (`Tier.T0` member access, `0|1|2|3` type) is the correct erasable equivalent and
the in-file rationale is accurate. (Note: `register.mjs` registers `ts-node/esm`, but native
stripping handles `.ts` first under this Node version — the substitution is correct regardless.)

**Doc sync — one inconsistency found and FIXED inline (minor).** The §Configuration table described
`confidence_min` as "halve `d_max`", which contradicts both the §Maximum useful depth formula
(`clamp to ⌊d_max_cap/2⌋`) and the implementation/simulator (a clamp to a fixed `⌊d_max_cap/2⌋`,
**not** a halving of the computed `d_max` — they differ materially for small networks). Corrected the
table cell to `clamp d_max to ⌊d_max_cap/2⌋`. The §Tier addressing and §Maximum useful depth
implementation callouts were checked line-by-line and are accurate.

**Reviewed, no action (with reasons):**
- *`prefixBits` left-pad direction* — the high `(n − bits(P))` positions are zeroed and `P` sits at
  the low end. Never exercised on the real path (peer IDs are 256-bit ≫ any `n = 4·d` at
  `d ≤ d_max_cap`); covered only by a dedicated unit test. The db-p2p FRET-binding ticket should
  confirm the direction it feeds, but keys there are also full-width, so it is moot in practice.
- *Low-confidence clamp can exceed the computed `d_max`* (e.g. small network + low confidence →
  `30`, not `0`). This is the doc-specified, simulator-validated behaviour (it bounds the start tier
  at half the cap when `n_est` is untrusted, rather than minimizing it); out of scope to relitigate
  a folded-back design decision. Left as-is.
- *`Math.min(clampValue, dMaxCap)` in `dmax.ts`* — `clampValue = ⌊dMaxCap/2⌋ ≤ dMaxCap` always, so
  the `Math.min` is redundant defensive code. Harmless; left to avoid churn.
- *Duplicated `F` validation in `addressing.ts` vs `dmax.ts`* — not a DRY violation: the constraints
  differ intentionally (addressing needs power-of-two for integer `log₂F`; dmax only needs integer
  `≥ 2` for the `log_F` base).
- *`coordD` `d ≤ 255` guard untested* — unreachable given `d_max_cap = 60`; not worth a test.

**Tests.** Existing specs cover happy path, edge cases (powers of `F`, tiny networks, clamp
boundary `confidence === confidence_min`, lazy re-read, sibling convergence/divergence, cross-
topic/tier decorrelation), error paths (construction guards, `d < 1`, invalid `F`), and the
collision regression. Coverage judged sufficient; no gaps warranting additions.
