# Complete — Reactivity recover RPC wire framing + canonical request signing (db-core)

## What this was

db-core's pure wire substrate for reactivity recovery requests:

- `BackfillV1.timestamp` (unix-ms) bound into the signing payload for freshness/replay protection.
- `backfillSigningPayload` / `resumeSigningPayload` — canonical, explicitly-ordered, type-tagged JSON-array
  signing images so signer (subscriber) and verifier (serving cohort) agree byte-for-byte.
- `createBackfillRequester` stamps the timestamp from an injected `clock` (no ambient `Date.now()`).
- `RecoverRequestV1` / `RecoverReplyV1` — a discriminated envelope (`kind: "backfill" | "resume"`) framing the
  single libp2p recover protocol, with `encode/decode` + structural validators.

The live libp2p transport + node wiring that consumes this lives in db-p2p
(`reactivity-recover-rpc-transport`, landed earlier in commit `d83e781`). The db-core substrate was landed
inline with that consumer in the same commit; the `9a358f4` implement pass for this ticket therefore added
no code and only verified + built the already-present work.

## Review findings

Adversarial audit of the committed db-core substrate (`recover.ts`, the `backfill.ts`/`resume.ts` signing
helpers, the envelope validators, and the docs). Disposition: one minor test-coverage gap fixed inline; no
major findings, no new tickets filed.

### Checked — correctness / canonical-bytes contract
- **Field-order determinism is load-bearing and correct.** `backfillSigningPayload` /
  `resumeSigningPayload` read fields by explicit name into a fixed-order array (not object iteration), so the
  signed image is independent of JSON key order — exactly the property the design requires. Confirmed the
  orders match the JSDoc contract and the determinism/collision tests pin them.
- **No divergent re-derivation of the signing bytes.** Verified the only other place that computes these
  bytes — db-p2p's serve handler (`recover-transport.ts` `createRecoverRequestHandler`) and signer factory
  (`createRecoverRequestSigners`) — **imports** `backfillSigningPayload`/`resumeSigningPayload` from db-core
  rather than re-implementing them. Signer and verifier are byte-identical by construction.
- **Pattern consistency.** The type-tagged JSON-array signing image faithfully mirrors the established
  `registerSigningPayload` / `promotionNoticeSigningPayload` / `membershipCertSigningPayload` convention in
  `cohort-topic/wire/payloads.ts` and `cohort-topic/sig/payloads.ts`. Leading `"BackfillV1"`/`"ResumeV1"`
  tags (plus differing array lengths, 6 vs 7) make cross-type image collision impossible.
- **Requester ↔ signer ↔ verifier round trip.** `createBackfillRequester` builds the unsigned image
  (`{v, collectionId, fromRevision, toRevision, timestamp}`) and signs over it; the decoded `BackfillV1`
  on the serving side recomputes over the same signable fields (ignoring `signature`). Integer + base64url
  string fields round-trip through JSON exactly, so the bytes match. Cross-package byte-identity (noble
  sync `signPeerSig` vs libp2p async signer) is proven in db-p2p `recover-transport.spec.ts` —
  out of this package but the load-bearing consumer.
- **Envelope validators.** `validateRecoverRequestV1`/`validateRecoverReplyV1` enforce `v===1`, narrow
  `kind`, reject the stray opposite branch (`rejectStrayBranch`), and delegate the populated branch to the
  existing inner validators (which throw on a missing/garbage body via `asObject`). `encode*` re-validates
  before framing, so a malformed in-memory message cannot be serialized.
- **`timestamp` validation.** Validated by `reqFiniteNumber` (rejects missing / `NaN` / `Infinity`),
  consistent with the sibling `ResumeV1.timestamp` and `NotificationV1.timestamp`. A non-integer or negative
  timestamp is structurally allowed but harmless: it is signed, so a verifier recomputes over whatever the
  request carries, and the node-level freshness guard (db-p2p) rejects stale values. Not a defect.

### Found + fixed inline (minor)
- **Reply-side envelope validator had weaker test coverage than the request side.** The request-side tests
  exercise wrong-body-for-kind, missing-body, and unknown-kind; the reply side only tested "both branches
  present". Since `validateRecoverReplyV1` has its own `reqKind` / `rejectStrayBranch` / inner-validator
  paths, I added three mirror cases to `recover.spec.ts`: a reply whose body is the wrong branch for its
  `kind`, a reply whose declared `kind` has no matching body, and a reply with an unknown `kind`. Suite went
  from 845 → 848 passing.

### Noted — no action (out of this package / deliberate design)
- **No `correlationId` nonce on recover requests — deliberate.** Replay protection keys on the
  signature bytes + `timestamp` at the db-p2p node-level guard; the signature is itself a unique authenticated
  token. Reviewed and agree the argument holds; the absence is not a gap.
- **Protocol-id prose mismatch (cosmetic, db-p2p-owned).** The originating ticket prose wrote
  `/optimystic/reactivity/recover/1.0.0`; the implemented code + docs use
  `/optimystic/reactivity/1.0.0/recover`. Both ends (handler + dialer) agree on the implemented string and it
  is internally consistent. The protocol id lives in db-p2p, not db-core. No action here.
- **`subscriberCoord` real-ring-coord sourcing** is a db-p2p responsibility (the resume request carries it
  and it is signed); db-core has no handling to get wrong. Out of scope.
