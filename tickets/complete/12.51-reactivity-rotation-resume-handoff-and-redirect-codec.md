# Complete — Reactivity rotation: resume↔handoff coordination + `RotationRedirectV1` recover codec (db-core)

The pure db-core half of wiring the tail-rotation lifecycle onto a live node. Two seams for the follow-on
db-p2p tickets:

- **A.** `classifyResume`/`serveResume` consult the new tail's inherited (cross-rotation) handoff checkpoint
  when the rolling checkpoint misses, so a resume spanning a rotation is served as `CheckpointWindow` rather
  than `OutOfWindow`.
- **B.** `validateRotationRedirectV1` + the `RecoverReplyV1` `kind: "rotated"` branch — the drain-window
  redirect rides the recover reply envelope; forward-compat is fail-closed (an old decoder rejects `"rotated"`
  and chain-reads).

Implementation landed in commit `5ffc6fe`. This review pass found one **major** correctness defect in seam A
(fixed inline as a safe stop-gap + filed a design follow-on), and confirmed seam B is sound.

## Review findings

### Checked

- **Read the implement diff (`5ffc6fe`) first, fresh**, then every source file it touched (`resume.ts`,
  `recover.ts`, `rotation.ts`) and the ones it depends on but did not touch (`checkpoint.ts`,
  `push-state.ts`, `subscriber.ts`) to understand the real window/contiguity semantics.
- **Seam A — inherited resume serving:** classifier decision order, the `serveResume` summary-selection,
  rolling-vs-inherited precedence, and (critically) the **subscriber-side apply** path through
  `applyResumeReply` → `ReactivitySubscriber` contiguity.
- **Seam B — redirect codec:** `validateRotationRedirectV1` field validation, the `RecoverReplyV1`
  `reqReplyKind` narrower, stray-branch rejection both directions, encode/decode round-trip, forward-compat
  fail-closed claim, transitive export from `index.ts`.
- **Cross-cutting:** import-cycle risk (`resume.ts` → `rotation.ts` for `checkpointCovers`), the db-p2p
  consumer of `serveResume` (`recover-transport.ts`), type-level exhaustiveness of the widened reply kind,
  build (db-core + db-p2p), and the full test suites.

### Found

- **MAJOR — high-edge gap in the inherited resume-serving path (`resume.ts`).** `serveResume` answers the
  inherited window with `inherited summary + the live ring's recentEntries`. This is only gap-free when the
  inherited window abuts the ring's low edge. In steady state after a rotation the new tail's **own** rolling
  checkpoint (`PushState.checkpoint`, distinct from the inherited one — `applyRotationHandoff` does not feed
  it) sits **between** the inherited window and the ring — e.g. inherited `[1,8]`, rolling `[9,16]`, ring
  `[17,20]`. Serving inherited `[1,8]` + ring `[17,20]` **silently skips `[9,16]`**. Applied subscriber-side,
  `applyResumeReply` only guards the checkpoint's **low** edge (vs. the subscriber's own contiguity head), not
  this high-edge gap: it rebaselines to `inherited.toRevision`, replays the ring, and returns
  `checkpoint_applied` (**false success**) while `[9,16]` is never delivered. The implementer's own headline
  test (`resume.spec.ts`) asserted exactly this gapped serve shape (`inherited [1,8]` + ring `[17,18,19,20]`)
  and never ran it through a subscriber, so the defect was untested.
- **MINOR — docs over-promise.** `docs/reactivity.md` §Resume + §Tail rotation step 5 implied the inherited
  window restores the full `W + W_checkpoint` cross-rotation range; with a single-checkpoint reply that is not
  achievable once the new tail evicts (see below).
- **Seam B (codec): clean.** `validateRotationRedirectV1` and the `kind:"rotated"` reply branch are correct
  and symmetrically validated; round-trip + negative tests are thorough (stray branches both ways, missing
  body, malformed ids, negative/non-integer revision, version, non-object). Forward-compat fail-closed is
  correct (`reqReplyKind` rejects `"rotated"` on an old decoder). No issues.
- **No import cycle.** `rotation.ts` imports neither `resume.ts` nor `recover.ts`; `checkpointCovers` is a
  hoisted function decl. Confirmed.
- **db-p2p consumer unaffected.** `recover-transport.ts`'s `serveResumeReply` does not pass
  `inheritedCheckpoint`, so the inherited branch is inert there and the fix below is a strict no-op for it.

### Done

- **Fixed the major defect inline (safe stop-gap).** Added a **gap-free guard** to `classifyResume`: the
  inherited window is used only when it still abuts the ring's low edge (`inherited.toRevision >= ringLow − 1`);
  otherwise the request falls to `OutOfWindow` (an honest chain read) instead of emitting a reply with a
  high-edge gap. Every served `checkpoint_window` reply is now contiguous. Updated the `classifyResume` /
  `ResumeServingDeps.inheritedCheckpoint` / `serveResume` doc-comments accordingly.
- **Tests.** Rewrote the inherited headline test to an abutting (gap-free) scenario; **added a subscriber-side
  end-to-end test** proving the reply applies contiguously with nothing skipped; **added a test** that a
  non-abutting inherited window classifies/serves `OutOfWindow`. db-core reactivity: **203 passing** (was
  201); full db-core suite **870 passing**; db-p2p reactivity **108 passing**; `yarn build:db-core` +
  `yarn build:db-p2p` clean. (`yarn lint` is a no-op — lint not configured for these packages; the builds are
  the type gate.)
- **Docs.** Corrected `docs/reactivity.md` §Resume and §Tail rotation step 5 to state the single-checkpoint
  **abut limitation** (cross-rotation resume covers only ≈`W` revisions past a rotation, not the full
  `W_checkpoint`) and to point at the follow-on.
- **Filed the follow-on (major, design decision):** `tickets/plan/reactivity-rotation-inherited-window-bridge.md`
  — restore the documented `W + W_checkpoint` cross-rotation range by bridging all three stacked windows
  (inherited + new-rolling + ring) in one reply. Lays out three candidate approaches (compose summaries at
  serve time / multi-checkpoint reply / accept-and-document) and requires end-to-end proof through
  `applyResumeReply`, plus a decision on `mergedDigest` fold composability.

### Not changed (with reason)

- **Seam B codec / `recover.ts` / `rotation.ts` redirect surface** — reviewed, correct, well-tested; nothing
  to change.
- **`applyResumeReply` subscriber guards** — left as-is. The high-edge gap is now prevented at the serving
  classifier; adding a redundant subscriber-side high-edge check would be belt-and-suspenders against an
  honest cohort and is folded into the follow-on if the multi-checkpoint reply shape is chosen.
- **`checkpointCovers` (rotation.ts) vs `RollingCheckpoint.covers` (checkpoint.ts) duplication** — minor and
  intentional (different types; the ticket specified the `rotation.ts` source). No import cycle, so left in
  place.
