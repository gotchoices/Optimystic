## What this delivered

A B-tree point lookup (`BTree.get` / `Tree.get`) reads interior branch nodes on the way down to
the leaf it actually wants. Those interior "navigation" reads used to enter the optimistic-
concurrency conflict (read) set, so an unrelated concurrent change to a structural block a
transaction merely *walked through* could stale-reject it — a false positive that worsens the more
blocks a transaction touches (the fairness inversion behind the starvation work).

This change lets those navigation reads be dropped from the conflict set, but **only** for a
value-of-key point lookup, and **only** the interior nodes strictly between the root and the target
leaf. The target leaf is always retained. Everything else — range scans, `find`, `first`/`last`,
counts, writes, the root, the collection header, and all log-chain reads — stays retained. Default
is always "retain": a read must be *positively* classified as `navigation` to be droppable, so a
forgotten tag can only over-reject, never miss a conflict.

Safety crux: a point lookup's result depends only on the terminal leaf's content. Any concurrent
change that would alter the result (set/delete the key, or a split/merge/rebalance that relocates
it) necessarily modifies that leaf, bumping its revision — caught by the retained leaf read.

Mechanism: `ReadPurpose = 'value' | 'navigation'` threads through the read chain (`tryGet` →
Tracker → CacheSource → TransactorSource) into a shared `ReadDependencyCollector` that merges
revision max-wins and purpose value-wins; `getReadDependencies()` returns only `value` reads. The
`get` descent tags interiors `navigation` and calls `markReadValue(leafId)` (forwarded down the
store chain) to re-retain the leaf.

**Approach:** read the implement diff (196023b) with fresh eyes before the handoff, re-derived the

that redirects a key across a leaf boundary must modify that leaf), traced the full store chain and
every `find`/`get`/`getPath`/scan/write call site, checked the validator's use of the read set, and
read every touched file plus the docs that *should* have been touched. Ran build, full suite, lint.

**Verification run (all green):**

- `eslint` on all touched source + test files — clean.
- New spec `occ-structural-read-exclusion.spec.ts` — 16 passing.

**Correctness — checked, nothing found.**
- `navigate=true` is reached *only* from `get()`. Every other descent (`find`, `findFirst`/
  `findLast` for `range`, `upsert`/`insert`/`replaceKey`/`deleteAt` write paths) uses the
  `navigate=false` default → interiors retained. Confirmed by reading all `find(`/`getPath(` sites.
- The leaf re-retention (`markReadValue`) forward chain is complete across every store layer that
  can sit under a BTree in the collection wiring: `AtomicProxy` (fresh-tree path) → `Tracker`
  (reopen path) → `CacheSource` → `collector.markValue`. `TransactorSource` also records the
  navigation read into the *same shared* collector, and value-wins collapses the two — so the
  single upgrade re-retains it regardless of which layer served the read.
- Determinism (Theorem 4) holds robustly: `createTransactionId` sorts reads
  (`transaction.ts:248`), so `Map` insertion order is immaterial; and both TransactorSource and
  CacheSource record into the shared collector, so read-set content is identical cold vs. warm
  cache (tested).
- The validator (`validator.ts`) checks the **shipped** `transaction.reads` as-is (step 3) and does
  not re-derive reads from its re-execution (re-execution only recomputes the *write* operations
  hash) — so tree-height-at-validation-time cannot desync the conflict set. The navigation
  exclusion is purely a recording-side reduction.
- `nodeCapacity` is threaded into **all three** live BTree constructions in `Tree.createOrOpen`
  (create-header, reopen, write-path `actionTree`) — verified complete. `readView` correctly omits
  it: it is read-only, and fan-out only governs split/merge, never traversal.
- The Tracker memo short-circuit (handoff probe 4) is pre-existing and orthogonal: it only skips
  recording for blocks the transaction has already staged writes to (already in the write set);
  `markValue` no-ops safely on such an unrecorded leaf.

**Minor findings — FIXED inline this pass.**
- *Stale docs.* `docs/internals.md` "Read Dependency Validation" still claimed "ALL block reads
  (including internal structural blocks) are tracked — maximally correct." Corrected to describe
  `ReadPurpose`, the value-wins/max-wins merge, and the point-lookup navigation exclusion, with a
  pointer to Theorem 5. (`docs/correctness.md` Theorems 5/8b/9 were already updated by the
  implementer.)
- *Coverage gap.* The reopen-tree read path (`BTree → Tracker → CacheSource → collector`, no
  AtomicProxy) was only covered transitively; only the fresh-tree (AtomicProxy) path had an
  integration test. Added a focused unit test asserting `Tracker` forwards both `purpose` and
  `markReadValue` down to the CacheSource collector (drop-then-re-retain). +1 test (1242 total).

**Tripwire — parked (not a ticket).**
- `nodeCapacity` is a per-call construction param and is **not persisted** in the collection
  header. Creating a tree with a non-default fan-out and later reopening it (a separate
  `createOrOpen`) without the same capacity silently falls back to 64, so subsequent writes split
  at a fan-out the existing nodes were not built for. Fine today — only tests pass it, and they pass
  it consistently within one `createOrOpen`. Recorded as a `NOTE:` at the `nodeCapacity` param in
  `tree.ts`; the fix, if a persisted tree ever needs a custom fan-out, is to store it in the header
  and read it back on reopen.

**Doc nuance noted, intentionally not edited.**
- `docs/correctness.md:408` (Theorem 7, Byzantine freshness) says a transaction "captures read
  dependencies for every block accessed." Slightly imprecise post-change, but the security claim
  holds unchanged: navigation reads by definition do not affect the result, so their freshness is
  immaterial — you remain protected from staleness in every read you *depend on*. Theorem 5 (the
  canonical statement) already carries the precise wording. Left as-is to avoid doc churn.

**Major findings:** none. No new tickets filed.
