## What was built

Swapped the faulty extraction in `CoordinatorRepo.pend`:

```ts
// before
const allBlockIds = Object.keys(request.transforms);
await this.verifyResponsibility(allBlockIds);
// ...
return { success: true, pending: [], blockIds: Object.keys(request.transforms) };

// after
const allBlockIds = blockIdsForTransforms(request.transforms);
await this.verifyResponsibility(allBlockIds);
// ...
return { success: true, pending: [], blockIds: allBlockIds };
```

`blockIdsForTransforms` (exported from `@optimystic/db-core`, defined in `packages/db-core/src/transform/helpers.ts:41`) deduplicates the block ids across `inserts`, `updates`, and `deletes`. The local `allBlockIds` is reused for the success payload so the verification and reporting sites cannot drift again.

## Why the bug was latent

On solo / `clusterSize <= 1` deployments (sereus-health), `pend` hits the `peerCount <= 1` fast path and delegates to `storageRepo.pend`, so the faulty ids never reached `verifyResponsibility` in a rejecting way. The bug would have surfaced as soon as multi-node clusters enforced proximity — either rejecting legitimate writes with `Not responsible for block(s): inserts, updates, deletes`, or incorrectly passing hashes of those literals through to cluster routing.

## Testing / validation

`packages/db-p2p/test/coordinator-repo-proximity.spec.ts`:

- **New regression test** `pend block id extraction (regression for Object.keys(transforms) bug)` — issues a `pend` with inserts + updates + deletes across three distinct block ids, captures every key `findCluster` is queried for, and asserts:
  - `"inserts"`, `"updates"`, `"deletes"` are NOT in the verified keys
  - the three real block ids ARE in the verified keys
- **Fixed existing tests** — several proximity tests had been passing malformed `transforms: { [blockId]: [] }` payloads, which only appeared to work because the old bug treated the top-level keys as block ids. They now use the correct `Transforms` shape `{ inserts, updates, deletes }`; test intent (throw/not-throw, errors mention specific block ids, etc.) is unchanged.

Validation:
- `packages/db-core` → `npm test`: 287 passing.
- `packages/db-p2p` → `npm test`: 396 passing (including the new regression).
- `packages/db-p2p` → `npm run build`: clean.

## Usage notes

No public API change. Callers of `CoordinatorRepo.pend` who had been observing the bogus `blockIds: ['inserts','updates','deletes']` in the returned `PendSuccess` will now see the actual affected block ids — the behavior the type always claimed.
