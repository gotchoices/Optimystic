## What shipped

Backoff + jitter is now the built-in default retry policy for optimistic-concurrency (OCC) losses in
db-core:

- **Single-collection `Collection.sync`** — the exponential backoff between stale-failure retries now
  multiplies by a random factor in `(0.5, 1]`, so a herd of clients that lost the same race no longer
  re-attempts in lockstep. The abort-sleep and abort-error helpers were extracted to a shared
  `utility/backoff.ts` (the private copies in `collection.ts` were removed).
- **Multi-collection `TransactionCoordinator.commit`** — previously threw on any loss and left retry
  to the caller. It now wraps the single-attempt work (`commitOnce`) in a bounded, jittered retry
  that re-drives **only a clean stale loss** (`CoordinatorStaleLossError`): between attempts it backs
  off and re-reads every registered collection to fresh revisions. A partial landing
  (`CoordinatorPartialCommitError`) and every hard failure (expired txn, unavailable transactor,
  storage/policy pend rejection) propagate immediately without retry. Defaults are safe out of the
  box (10 attempts, base 100 ms, cap 5000 ms); `SyncOptions` (`maxAttempts` / `deadlineMs` / `signal`
  / `rand`) tunes it, threaded through `session.commit(options?)`.

See implement commit `d1d44e7` for the full implementation write-up.

## Review findings

Reviewed the implement diff with fresh eyes against SPP/DRY/modularity/scalability/maintainability/
performance/resource-cleanup/error-handling/type-safety, ran the full db-core suite and lint, and
re-read every file the change touched plus the ones it should have (`docs/transactions.md`, the
Quereus `txn-bridge.ts` commit path).

**Checked and clean:**
- Retry control flow in `commit()` — attempt cap, deadline (progress-agnostic, enforced only after
  the first loss — intentional and documented), abort mid-backoff, and non-retry of partial landings
  are all correct and covered by tests.
- `staleLoss` threading through `commitPhase`/`coordinateTransaction` — `commitPhase` correctly holds
  `staleLoss` only when every failure is stale and none is hard (`anyStale && !anyHard`).
- The inter-attempt `collection.update()` re-read correctly preserves staged actions (they live in
  `this.pending` and replay against the refreshed tracker), matching how `Collection.sync` re-reads.
- Only `session.commit()` calls `coordinator.commit()`; the Quereus adapter routes through it. No
  other caller is affected by the new default retry.
- `CoordinatorStaleLossError.failedCollections` is over-inclusive on a pend-phase loss (names all
  participants), but no caller keys logic off it — it is diagnostic-only (txn-bridge only tests
  `instanceof CoordinatorPartialCommitError`; tests only assert `instanceof`). No action.

**Fixed in this pass (minor):**
- *Order-dependent pend classification (correctness).* `pendPhase` classified the whole pend as a
  retryable stale loss based on **only the first settled failure**, whereas `commitPhase` requires
  *every* failure to be stale. A mixed pend (one retryable conflict + one hard rejection) where the
  conflict settled first was therefore misclassified as retryable, and `commit()` would spin the full
  retry budget on a hard failure the design intends to fail fast — nondeterministically, depending on
  collection iteration order. Rewrote `pendPhase` to mirror `commitPhase`
  (`staleLoss = anyConflict && !anyHard`, order-independent). Added a regression test
  (`MixedPendTransactor`, "a MIXED pend failure … fails fast, not as a retryable stale loss").
- *Stale docs.* `docs/transactions.md` still described the coordinator throwing a plain error on any
  clean session-mode failure for the caller to retry. Updated to describe the new default: clean
  stale loss → auto-retry then `CoordinatorStaleLossError` on exhaustion; hard failure → immediate
  plain error; `execute()` still not retry-wrapped.

**Filed as a new ticket (major — test coverage):**
- `backlog/debt-coordinator-retry-real-competing-committer-test` — every coordinator retry test loses
  via `FlakyCommitTransactor`, which refuses **without** advancing a real revision, so the
  "re-read → rebase → win" property (a genuine rival committing a competing rev mid-retry) is
  unproven end-to-end for the multi-collection path. Coverage-only.

**Confirmed intentional (no action):**
- *PEND asymmetry between the two loops.* `Collection.sync` retries any stale failure including a bare
  `reason`; the coordinator retries a pend loss only on a real conflict (`missing`/`pending`), failing
  fast on a bare `reason`. This is the deliberate, more-conservative coordinator contract (avoids
  spinning the default 10× on a hard failure), and is safe.
- `makeAbortError` message changed to the generic `'The operation was aborted'` when shared. Only
  `err.name === 'AbortError'` is asserted anywhere; cosmetic.

## Tripwires recorded

- `coordinator.ts` re-read loop (pre-existing `NOTE:`) — refreshes **all** registered collections, not
  just this transaction's participants. Harmless today; narrow to participants only if a coordinator
  ever holds many collections and retry latency shows up.
- `txn-bridge.ts` commit site (new `NOTE:`) — `session.commit()` with no options now inherits the
  coordinator's default bounded retry (~seconds under sustained contention) where it previously failed
  fast. Intended safe default; thread a `SyncOptions` through `commit()` if the SQL layer ever needs
  faster failure.
