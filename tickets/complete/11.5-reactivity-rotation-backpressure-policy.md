# Complete: reactivity tail rotation, slow-subscriber backpressure, Edge/Core policy

Adversarial review of implement commit `d60a393`. Implements three reactivity concerns as pure db-core
modules plus thin db-p2p wiring: the rotating tail-anchor + drain protocol, per-subscriber bounded-queue
backpressure, and the Edge/Core T3 policy + consolidated config. No simulator code (already landed by
`simulator-reactivity-replay`).

## Review findings

- `yarn build:db-core`, `yarn build:db-p2p` — **clean** (full type-check; the `checkpoint.ts` refactor and
  the `BoundedQueueRef`→`SubscriberBackpressure` field-type change compile across both packages).
- db-core full suite — **806 passing** (reactivity subset 160).
- db-p2p full suite — **647 passing / 9 pending**. The `parent unreachable` line is the expected
  failure-path log from `host-antidos-coldstart.spec.ts`, not a regression (test passes).
- Verified the origination pass-through end-to-end: `buildNotificationV1` copies `ctx.rotationHint` onto the
  emitted `NotificationV1` (`notification.ts:74`), and `managers.spec.ts` asserts emission on the filling

### Correctness / logic (checked)
- **Rotation detection precedence** — a delivered `tailId` mismatch (hard rotation) is preferred over a
  `rotationHint` pre-announce; chained rotations (OLD→NEW1→NEW2) self-correct via `rotationHandledFor`.
  Cache invalidation fires once per successor even when `onRotation` is absent. Correct.
- **`BlockFillTracker`** — `filling` (`>= blockFillSize`, then reset) is checked before `warmup`
  (`=== warmAt`), and `warmAt = blockFillSize − warmThreshold` is always strictly < `blockFillSize`
  (validated `warmThreshold < blockFillSize`), so the two signals can never collide. Correct.
- **Handoff window** — `buildRotationHandoffCheckpoint` folds `[checkpoint.toRevision + 1, rotationRevision]`
  (abuts the rolling checkpoint's high edge), reusing the outgoing tail's own `foldOptions` so the merged
  digest is byte-identical to a steady-state checkpoint. Empty-ring and "nothing above the checkpoint" both
  return `undefined`. `adoptRotationCheckpoint` only advances (idempotent, no rewind). Matches spec
  §Tail rotation step 5 (the old tail's *own* rolling checkpoint is intentionally NOT migrated — a deeper
  resume falls back to chain re-read).
- **`checkpoint.ts` refactor** — confirmed `buildCheckpointSummary` / `coalesceMergedDelta` are behavior-
  identical to the prior inline `summary()` / `buildMergedDelta()` (same digest fold, same omit-when-oversize
  / omit-when-no-delta / `deltaMaxBytes <= 0` branches, same `bracketingEntries`). Existing
  checkpoint/resume specs pass unchanged.
- **Gossip exclusion** — `serializeGossip()` omits `perSubscriberQueue` and `inheritedCheckpoint`
  (asserted by `push-state.spec.ts`). Correct: both are primary-local / one-time-migration state.

### Minor — fixed in this pass
- **`planReRegistrationWave` cap contract was under-specified.** The rotation invariant
  (`docs/reactivity.md` §Tail rotation) is that the re-registration wave holds the new tail within
  `cap_promote_fast = 32`, but `createRejoinJitter()`'s default `capPromote` is the cohort-failure
  `cap_promote = 64`. A host using the default jitter would silently bound to 64/window and overrun the
  documented rotation ceiling; the wave test passes only because it injects `capPromote: 32` explicitly.
  Tightened the `planReRegistrationWave` JSDoc to make the caller contract explicit and reference
  `DEFAULT_CAP_PROMOTE_FAST`. (No production caller exists yet, so this was latent, not breaking.)
  Re-ran the rotation spec — 20 passing.

### Major — filed as follow-on
- **Rotation-lifecycle host wiring is owned by no ticket.** The implementer correctly flags (gaps 1, 4, 5)
  that the drain-redirect transport, `RotationRedirectV1` codec, re-registration timer scheduling, and the
  `BlockFillTracker`→origination hookup are host-integration follow-ons. The notification-transport plan
  (`reactivity-notification-transport`, 12.3) owns the dequeue→dial delivery loop and the backpressure
  interaction but **not** these rotation-specific seams. Filed
  `tickets/plan/12.5-reactivity-tail-rotation-transport.md` (prereq `reactivity-notification-transport`) to
  own them, plus the `SubscriberBackpressure` queue-eviction lifecycle and the handoff↔resume coordination
  (`classifyResume` reading `inheritedCheckpoint`).

### Observations — noted, not changed (acceptable for this milestone)
- **`SubscriberBackpressure` has no automatic pruning** (`remove` exists, no TTL caller) and
  `enqueueForSubscribers` has no consumer/delivery loop. The documented memory bound holds only once the
  host wires TTL eviction + the dequeue loop. Correct given no forwarder host exists yet; captured in 12.5.
- **`resolveQueueMax` with `cohortSubscribers` but no `scaleBaseline`** defaults `baseline = 1`, producing
  very large queues (e.g. 256 subscribers → 8192 depth). A footgun, but it is an explicitly
  simulator-validated-pending adaptive hook with no production caller and every test passes an explicit
  `scaleBaseline`. Left as-is; the simulator fold-back owns the eventual real scaling input.
- **Provisional params** (`T_drain` / `queue_max` / `block_fill_size`) are sourced from the single config
  table and marked simulator-validated-pending, so the fold-back can revise without protocol changes.
