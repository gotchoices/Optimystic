# Invalidation log entry + single-collection reversal (reviewed)

The core durable reversal primitive for a single invalidated transaction in one collection:
the `InvalidationEntry` log type, compensating-state computation, the invalidation certificate,
and the consensus-apply path. Implemented as a vertical slice — db-core primitive + db-p2p
compute/verify/apply fully built and unit-tested; composition-root wiring (live multi-node)
deliberately left as injection points (`onInvalidate`, `onInvalidation`).

## What landed

See the implement commit (`ticket(implement): invalidation-log-entry-and-reversal`) for the full
diff. In brief: `LogEntry` gains an `invalidation?` arm; `Log.addInvalidation` / `findInvalidation`;
`InvalidateRequest` + a 4th `RepoMessage` operation arm; `db-p2p/src/dispute/invalidation.ts`
(proof builder, certificate verifier, `computeRevertedBlock`, `applyInvalidation`); cluster-repo
`onInvalidate` sink + `applyConsensusInvalidation`; dispute-service `onInvalidation` originator.

## Review findings

### Scope of the review

Read the implement diff with fresh eyes before the handoff summary. Traced the new types through
the log layer (`getFrom`/`getActionContext`/`findInvalidation` skip semantics), the certificate
verifier against `DisputeService.resolveDispute`, `computeRevertedBlock` against the real
`BlockStorage`/`StorageRepo` revision model (sparse per-block revs, `getBlock(rev)`/`ensureRevision`
range semantics, `saveReplica` monotonic guard), the cluster consensus-apply arm, and the
dispute-service origination path. Verified the doc changes against the implemented reality. Ran the
floor tests and the full db-p2p suite.

### Checked — found correct (no change needed)

- **Log skip semantics.** `getFrom` and `getActionContext` correctly skip invalidation entries in the
  pending/committed set (they take a rev slot only); `findInvalidation` scans newest→oldest and
  returns the most recent match. `getFrom`'s post-checkpoint loop already guards `entry.action`.
- **Certificate threshold parity.** `verifyInvalidationCertificate`'s `totalDecisive === 0 → reject`
  and `challengerVotes >= ceil(totalDecisive * 2/3)` exactly mirror `resolveDispute`. The signed-vote
  payload `${disputeId}:${vote}:${computedHash}` matches `makeVote` in dispute-service.
- **Compensating-state computation vs. the real storage model.** Block revisions are *sparse* in the
  global rev space (`internalCommit` writes a revision only at the commit's global rev), but
  `getBlock(invalidatedRev − 1)` materializes the highest rev ≤ target via `materializeBlock`'s
  descending walk, and `pend` seeds `ranges = [[0]]` (open), so `ensureRevision` does not throw on a
  rev where this block was not modified. The pre-`T_inv` base is recovered correctly. The
  `listRevisions(invalidatedRev+1, fromRev)` guard against the descending-range footgun is sound.
- **Monotonic rev / no under-tip collision.** `rev = max(fromRev)+1` is strictly above every reverted
  block's tip, so `saveReplica`'s monotonic guard never no-ops the compensating write.
- **Determinism + idempotency.** `stableStringify` sorts keys recursively (including objects nested in
  arrays); the convergence test confirms two independent members agree on hash + rev. Dedup is keyed
  on `(invalidatedActionId, disputeId)` at both the log layer (durable) and the cluster in-memory map
  (fast path, pruned on the executed-tx TTL, rolled back on sink fault).
- **Consensus-apply fault tolerance.** The `invalidate` arm verifies the certificate before the sink,
  tolerates a missing/throwing sink without resetting the stream, and adds `invalidate.blockIds` to
  `getAffectedBlockIds` for conflict serialization.

### Major — filed as a new ticket

- **`tickets/fix/invalidation-certificate-arbitrator-binding.md` — the invalidation certificate is
  forgeable.** `verifyInvalidationCertificate` checks only that each vote's signature matches its
  claimed peer-id's embedded key. It does **not** bind votes to the legitimately-selected arbitrator
  set (sybil-generated keypairs produce a passing certificate for any target), does **not** dedup by
  arbitrator (one genuine vote duplicated fakes a super-majority), and the apply path never checks
  `proof.messageHash` against the `invalidatedActionId`/`blockIds` it reverts (a genuine proof can be
  replayed against an innocent transaction). The votes sign only `disputeId:vote:computedHash`, with
  no commitment to the target. This contradicts the documented "cannot forge one" guarantee. It is a
  *latent* hole, not a live exploit, because no invalidation is originated or applied end-to-end yet
  (the wiring is the parent ticket's gap #1) — but it **must** be closed before that wiring lands.
  Filed `prereq: invalidation-log-entry-and-reversal`, difficulty hard, with concrete forgery-vector
  test requirements. The cascade and client-notification follow-ups consume the same verifier and
  inherit the fix.

### Minor — fixed inline this pass

- **Honest docs.** `docs/right-is-right.md` § Durable Invalidation previously asserted an unqualified
  "cannot forge one." Added an interim caveat block documenting exactly what the current verifier does
  and does not bind, pointing at the fix ticket, and noting it is not yet a live exploit. The fix
  ticket will restore the unqualified guarantee and remove the caveat once the binding lands.

### Noted — left to the wiring/cascade follow-ups (not defects to fix here)

- **`rev` is not actually threaded through consensus.** `InvalidateRequest` carries no target `rev`,
  and `applyConsensusInvalidation` passes none, so every member computes `rev = max(localTip)+1`
  locally — yet the `ApplyInvalidationParams.rev` / `InvalidateRequest` doc comments describe a
  "consensus-assigned slot the consensus path passes." Determinism therefore relies on identical local
  block tips across members rather than an agreed slot. This is the same surface as the parent ticket's
  gaps #2/#7; the composition-root wiring ticket should reconcile the comments with reality (either
  thread an agreed rev or document that convergence depends on tip equality).
- **Blind later-action replay (parent gap #5).** `computeRevertedBlock` replays surviving later
  actions' delta transforms verbatim on the rolled-back base; a delta built against post-`T_inv`
  content can corrupt when applied to pre-`T_inv` content. By design for the no-cascade core; the
  cascade ticket replaces it with read-dependent re-evaluation. Also `getTransaction` returning
  `undefined` is silently skipped (`continue`), which could diverge members if a later transform is
  missing — acceptable under full replication, worth hardening when the cascade reworks this path.
- **`getActionContext` still lists the invalidated action in `committed` (parent gap #6).** Reversal is
  surfaced via `findInvalidation`, which the client-notification ticket consults. Acceptable split.
- **Delete-restore deferred (parent gap #4).** A `T_inv` that created a block records the
  `DEFERRED_DELETE_RESTORE` sentinel and logs rather than physically deleting; `IBlockStorage` exposes
  no delete-transform write. Cascade ticket.
