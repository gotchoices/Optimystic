# Complete — reactivity rotation: recover drain-redirect serve + subscriber honor (db-p2p)

The drain-redirect path that moves a stuck/waking subscriber off a rotated, still-draining tail. The old
serving cohort records its rotation (`ReactivityForwarderHost.markRotated` → a `TailDrainGate` keyed by the
old `topicId`) and, while draining, answers recover requests (resume; best-effort backfill) with a
`kind: "rotated"` `RotationRedirectV1`. The outbound transport raises a terminal `RotationRedirectError`; the
subscriber honors it through the shared `surfaceRotation` / `onRotation` re-registration seam a notify
pre-announce uses. After `T_drain` the gate + old tail's served `PushState` are evicted lazily.

## Review findings

**Scope reviewed.** The full implement diff (`0cac17a`): `forwarder-host.ts` (`markRotated` /
`rotationRedirectFor` / `rotationGates`), `recover-transport.ts` (`RotationRedirectError`, redirect-first
serve in `serveResumeReply` / `serveBackfillReply`, `inheritedCheckpoint` threading, the `rotationFor` dep,
terminal redirect in `exchange`), `subscription-manager.ts` (the extracted `surfaceRotation` seam,
`honorRotationRedirect`, `resume()` / `escalateBackfill` redirect handling), `docs/reactivity.md`, and the
three spec files. Cross-checked the db-core seams the change rides: `TailDrainGate.isDraining` (strict `<`
boundary), `rotationRedirect`, `serveResume`'s `inheritedCheckpoint` branch, and `decodeRecoverReplyV1`.

**Correctness / behavior — checked, sound.**
- **Redirect-vs-serve ordering** (the core race): in both `serveResumeReply` and `serveBackfillReply` the
  `rotationFor` redirect is emitted *ahead of* `pushStateFor*` resolution — a draining old cohort redirects
  rather than serving stale data up to `rotationRevision` then going silent. Confirmed.
- **`reply.rotated!` non-null assertion is safe.** `validateRecoverReplyV1` (db-core `recover.ts:132`)
  validates `rotated` as a well-formed `RotationRedirectV1` whenever `kind === "rotated"` (throws otherwise),
  so the assertion in `exchange` can never deref undefined. The terminal `throw` (no walk fallthrough) is
  correct — the dialed member answered authoritatively.
- **`RotationRedirectError` cannot escape to the commit/delivery path.** `resume()` maps the redirect to the
  `"tail_rotated"` outcome (no throw); `escalateBackfill` catches it in its retry loop; the initial backfill
  rides the detached, `.catch`-guarded gap seam. The `backfillMaxRetries: 0` corner falls through to
  `resume()`, which surfaces the redirect. Confirmed end-to-end against the tests.
- **Backfill asymmetry is sound.** The backfill redirect fires only while the node serves *solely* the old
  draining tail — once both tails coexist `pushStateForCollection` resolves the new tail (highest
  `lastRevision`), so the wiring adapter keys `rotationRedirectFor` off the new topic (no gate) and serves
  normally. Documented as a deliberate best-effort secondary path.
- **Latent fix bundled in:** `serveResumeReply` now threads `ps.inheritedCheckpoint` into `serveResume`, so a
  cross-rotation resume to the new tail is served from the inherited handoff (`checkpoint_window`) instead of
  falling to `out_of_window` — previously the transport dropped it. Covered by a new recover-transport test.
- **Build + tests:** `yarn build:db-core` + `yarn build:db-p2p` clean (tsc, 0 errors). Reactivity suite
  **125 passing** (was 124 + the one test added below). No lint is configured project-wide (`lint` is a stub).

**Minor — fixed inline.**
- *Test gap on the central claim.* The `surfaceRotation` extraction's whole point is that notify-driven
  detection and recover-driven redirects share one `rotationHandledFor` once-per-successor guard, but the
  suite only tested each path in isolation. Added `managers.spec.ts` → *"shares the once-per-successor seam
  across notify-driven and recover-driven surfacing"*: a notify pre-announce surfaces successor A, a recover
  redirect to the **same** A is cross-deduped (one `onRotation`), and a notify announce of a **chained**
  successor B re-fires through the same guard. Passes.

**Minor — flagged, no code change (owned downstream / acceptably low-risk).**
- *`markRotated` hardcodes `T_DRAIN_MS`.* It constructs the `TailDrainGate` without a `tDrainMs`, so it always
  uses the 60 s default even though `ReactivityConfig.tDrainMs` exists. Harmless today (the only caller is the
  unit tests; the default *is* the documented `T_drain`), but the host-wiring ticket **`12.54`** binds
  `markRotated` live and should plumb the configured drain duration through if a non-default `T_drain` is ever
  wanted — the current `markRotated` signature cannot carry it. Recorded here for `12.54` (no separate ticket:
  the wiring is its scope, and it already owns the trigger + `rotationFor` binding).
- *Non-monotonic re-surfacing.* `surfaceRotation` remembers only the **last** successor, so an out-of-order
  surfacing (handle B, then a stale notification/redirect for the superseded A) would re-fire `onRotation` for
  A. This is pre-existing in `checkRotation` and merely broadened by the recover path; low-risk in practice
  (notifications are revision-ordered and chained rotations strictly increase `effectiveAtRevision`). Left as
  documented behavior, not a regression.

**Honest gaps confirmed acceptable (from the implement handoff).**
- *No live e2e of the full loop.* The `markRotated` trigger and the `rotationFor` → `rotationRedirectFor`
  binding adapter are deferred to **`12.54-reactivity-rotation-host-wiring-e2e`** — verified that ticket
  explicitly owns both (its §A/§B and TODO). The seam contracts here are unit-pinned; the integration is not,
  by design. No action.
- *Lazy, on-demand eviction.* The drained gate + served `PushState` are reclaimed only on the next
  `rotationRedirectFor` after `T_drain`. The `12.31` leak is now *reclaimable* (reduced), not proactively
  swept. Acceptable for this ticket; a proactive sweep, if wanted, belongs with the host demotion path. No
  action.
- *Forward-compat decode-closed only.* A peer predating `kind:"rotated"` fails the reply decode → the
  transport treats it as malformed and chain-reads (safe). Covered at the db-core codec layer
  (`recover.spec.ts`); not re-tested at db-p2p. Acceptable.

**Categories with nothing to report.** Type safety, resource cleanup, and error handling were each examined
and are sound: the `rotationFor` dep is optional (existing recover wiring still compiles), the gate/served
maps are reclaimed on drain-elapse, and every recover-driven throw is funnelled away from delivery. **No
major findings → no new fix/plan/backlog tickets filed.**
