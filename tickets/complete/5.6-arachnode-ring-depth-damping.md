# What shipped

Damping for `RingSelector`'s ring-shift decision (spec: `docs/arachnode-ring-handoff.md` § Part 1),
self-contained to `RingSelector` + its config + its single construction/flip site. The two-phase
responsibility handoff (Part 2) is deliberately untouched — sibling ticket
`arachnode-ring-handoff-protocol`.

Three mechanisms, all in `shouldTransition()`:

1. **Smoothing (EWMA).** Each tick folds available capacity and estimated network data
   (`Σ peerCount×avgCapacity`) into two per-instance EWMAs (α default 0.2), seeded from the first
   real sample. The decision works off the smoothed continuous depth
   `d = clamp(-log2(smoothedAvailable/smoothedTotalData), 0, 16)`.
2. **Dead-band / hysteresis.** Anchored on the node's currently-**advertised** ring `R`. Move out
   only when `d ≥ R+1−h` and `usedPercent > moveOut`; move in only when `d ≤ R−1+h`,
   `usedPercent < moveIn`, and `R>0`. Default `h=0.5` ⇒ a full ring separates the two triggers.
3. **Minimum dwell + single-step.** No new move within `minDwellMs` (default 10 min); no move while
   `status==='moving'`; the target is always `R±1`.

Supporting: `RingSelectorConfig` gained `smoothingAlpha`, `deadband`, `minDwellMs`, and an injectable
`now` clock; `determineRing()` refactored to a pure instantaneous read (shares `estimateNetworkData`,
behavior unchanged) and stays the deterministic first-boot ring; `createArachnodeInfo` takes an
optional `ringDepthOverride` so the advertised ring matches the ring the transition just chose;
`libp2p-node-base.ts` wires the config and passes the explicit target at the flip site. Return shape
`{shouldMove, direction, newRingDepth}` unchanged.

# Review findings

**Method.** Read the implement diff (`f9d9f30`) with fresh eyes before the handoff summary, checked it
against the Part 1 spec, traced every caller of the three changed methods
(`createArachnodeInfo`/`shouldTransition`/`determineRing` — only `libp2p-node-base.ts` and the spec
consume them), and confirmed the production wiring. Validated: `yarn build` clean; ring-selector spec
**30 passing**; full db-p2p suite **1222 passing, 36 pending, 0 failing**; `eslint` on the three
changed files exit 0.

**Correctness — no defects found.** The three damping mechanisms match the spec. I worked the math on
each: the dead-band + single-step make the move-out and move-in triggers mutually exclusive (a node
that just moved on high usage cannot also satisfy `usedPercent < moveIn`), dwell bounds frequency at
one move / `minDwellMs`, and the `advertisedRing` anchor keeps the target from disagreeing with the
node's own advertised ring on the next tick. Confirmed the hysteresis anchor actually resolves in
production: `getMyArachnodeInfo()` works because `libp2p-node-base.ts:896` injects `selfPeerId`; the
`round(d)` fallback is unreachable in production (bootstrap advertises before the first tick).

**Minor — fixed inline in this pass:**

- **Smoothing mechanism was never isolated by a test.** The "boundary hover" test's jitter values all
  sit *inside* the dead-band `(2.5, 3.5)`, so it proves the dead-band, not the EWMA — every raw sample
  is a no-move with or without smoothing. Added `smoothing absorbs a single boundary-crossing spike…`:
  the node settles at `d≈2.0`, then a single raw spike to `d=2.9` (past the 2.5 boundary — an
  unsmoothed signal *would* move out) is absorbed by the accumulated EWMA (`α=0.2` drags smoothed `d`
  only to ≈2.23), so no move fires. Directly exercises mechanism #1 and contrasts the first-sample
  seeding. (spec floor was 10 damped tests; now 11.)
- **Doc/impl mismatch on the smoothing form** (a gap the implementer flagged). `docs/arachnode-ring-handoff.md`
  §1.1 pseudocode smoothed the coverage *ratio* directly, but the code smooths the two inputs
  separately then divides — a distinct EWMA. Updated the pseudocode and prose to describe the
  input-wise form the code actually uses, note the distinction, and state the first-sample seeding.

**Observations / tripwires — no code change (recorded here, the index; homes noted):**

- **A single *large* spike from a settled ring integer can still trigger one move.** `α=0.2 + h=0.5`
  attenuates but does not veto: a 2-ring spike (raw `d` 2.0→4.0) drags smoothed `d` to ≈2.68, crossing
  2.5. This is by design — dwell (10 min) + single-step prevent *thrashing*, which is what Part 1
  targets; smoothing is a low-pass filter, not a hard gate. The parameter tradeoff already lives in
  `docs/arachnode-ring-handoff.md` §1.4; no new home needed. Not a defect.
- **`lastMoveAt` stamps on move *trigger*, not *completion*.** Already tagged `NOTE:` at the stamp site
  by the implementer; correct while Part 2 handoff is unlanded (status never becomes `moving` in
  production yet). Re-stamp-on-completion is the sibling `arachnode-ring-handoff-protocol` ticket's
  job. Tripwire stands; no action here.
- **`usedPercent` `total===0` guard is untested.** Defensive path (yields `usedPercent=0`, so a
  zero-total node can never move out). Low value to test; left uncovered deliberately.
- **`determineRing` effectively caps at ring 10, not 16**, because of the *pre-existing*
  `Math.max(0.001, coverage)` floor (`-log2(0.001)≈9.97`), which this diff did not touch — the
  `Math.min(…,16)` and the "caps at 16" test (asserts only `at.most(16)`) never bite. Out of scope for
  this ticket; noted for awareness, not filed.

**Major — none.** No new fix/plan/backlog tickets. The implementation faithfully covers Part 1 and
every edge case the spec calls out (dead-band both directions, ring-0 floor, in-flight `moving`,
seeding, dwell, single-step, boundary hover).
