# Complete: unify the owned-block tracked set across SpreadOnChurnMonitor and RebalanceMonitor

## What shipped (Option A — inject a bare `Set<string>`)

Both monitors previously kept their own `private readonly trackedBlocks = new Set<string>()`, each fed
by a *separate* `storageRepo.onAnyCollectionChange` subscription in `libp2p-node-base`, with no
responsibility-loss eviction. After this ticket:

- **`SpreadOnChurnDeps` / `RebalanceMonitorDeps`** gained an optional `trackedBlocks?: Set<string>`.
  Each monitor's field is `private readonly trackedBlocks: Set<string>`, assigned in the constructor as
  `deps.trackedBlocks ?? new Set<string>()` — injected ⇒ shared; omitted ⇒ a fresh private set
  (standalone construction / existing unit tests unchanged). `RebalanceMonitor.responsibilitySnapshot`
  stays **per-monitor** (rebalance's own was-responsible memory, not owned-block tracking).
- **`NetworkManagerService.initSpreadOnChurnMonitor` / `initRebalanceMonitor`** each take a new required
  `trackedBlocks: Set<string>` param and pass it straight into the respective `Deps`. Only
  `libp2p-node-base` calls these (confirmed: no other src callers, no test callers).
- **`libp2p-node-base`** constructs **one** `const ownedBlocks = new Set<string>()`, passes it to both
  monitor inits, and collapses the **two** `onAnyCollectionChange` subscriptions into **one**
  (`ensureOwnedBlockFeed()`, registered lazily by whichever monitor wires first, torn down exactly once
  in a dedicated up-front stop wrapper). In `onRebalance`, alongside `coordinator.handleRebalanceEvent`,
  it drives eviction: `event.gained → ownedBlocks.add`, `event.lost → rebalanceMonitor.untrackBlock`
  (clears both the shared-set entry and the snapshot). Responsibility-loss is the authoritative untrack,
  complementing spread's lazy no-local-data self-prune.

### Checked — and what was found
- **Correctness of the shared set + eviction (SPP / core invariant):** Verified `gained` is only ever
  derived by iterating `trackedBlocks` (`performRebalanceCheck` loops `this.trackedBlocks`), so the
  `gained → ownedBlocks.add` is genuinely a no-op today — defensive/forward-looking, documented inline.
  Kept as-is. The `lost → untrackBlock` eviction removes from the shared set **and** clears the stale
  snapshot entry; the coordinator reads the captured `event` (not the set), so eviction ordering vs the
  async coordinator dispatch is irrelevant. **No issue.**
- **Resource cleanup / teardown ordering:** Traced the three composed stop wrappers. Call order is
  rebalance.stop() → spread.stop() → feed teardown → original `node.stop`. Both monitors are fully
  stopped (listeners removed, debounce timers cleared) *before* the single feed is unsubscribed, and the
  up-front teardown wrapper fires regardless of which monitor registered the feed (or if neither did —
  `offOwnedBlockFeed` is undefined-guarded, so it is an inert extra wrapper). **No leak; no double-free.**
- **Type safety / call-site coverage:** The new required `trackedBlocks` param on both init methods is
  passed by the only callers (`libp2p-node-base`); `find_references` + a test-dir grep confirm no other
  src or test callers. The optional `deps.trackedBlocks` keeps every standalone construction
  backward-compatible. `yarn build` (tsc) clean, exit 0. **No issue.**
- **Error handling:** Spread and rebalance init are each wrapped in non-fatal try/catch (resilience, not
  correctness); the eviction handler is synchronous and the async coordinator dispatch keeps its
  `.catch(log)`. **No issue.**
- **`responsibilitySnapshot` staleness:** Confirmed the documented reasoning. When spread self-prunes a
  block (direct `trackedBlocks.delete`, *not* via `untrackBlock`), rebalance's snapshot retains a
  lingering entry. It is inert (`performRebalanceCheck` only iterates `trackedBlocks`; a removed block is
  never re-read) and re-derived if the block is re-fed. This is not a regression — pre-ticket, rebalance
  never evicted at all, so the snapshot grew unboundedly there too; the `lost` path now *clears* it.
  **Acceptable; no change.**
- **Best-effort iteration safety under shared mutation:** The rebalance handler can `delete` from
  `ownedBlocks` while `SpreadOnChurnMonitor.performSpread` is mid `for…of` over the same Set inside an
  `await` loop. JS Set mutation during iteration does not throw (entries visited best-effort);
  acceptable for a resilience mechanism. **Confirmed; documented inline; no change.**
