# Complete: matchmaking multi-cohort sweep, adversarial traffic bounds, public module

Adversarial review of implement commit `4b527d0`. Three deliverables landed, all behind injected ports
(no live libp2p; mock-tier e2e is the documented `13-matchmaking-e2e-mock-tier` follow-on):

1. **Multi-cohort sweep + aggregate counts** — db-core pure orchestration (`multi-cohort-seeker.ts`:
   `selectShards` / `runMultiCohortSweep`), db-p2p root producer (`aggregate-counts.ts`:
   `buildAggregateCount`, depth-gated + log-bucketed + threshold-signed), and the
   `AggregateCountV1.signers` wire addition.
2. **Adversarial traffic bounds** — `traffic-validation.ts` (`boundReportedTraffic`, `maxWalkHops`,
   `reportTrafficCrossCheck`) makes the over-/under-report worst-case bounds explicit and bridges
   cross-check signals into `IPeerReputation`.
3. **Public module** — `module.ts` (`MatchmakingProviderSession` / `MatchmakingSeekerSession` /
   `createMatchmakingQuorumDiscovery`) + `SeekerWalkResult.maxChildCohortCount`.

## Review findings

Scrutinized for SPP/DRY/modularity, scalability, resource cleanup, error handling, type safety, the
advisory trust model end-to-end, doc accuracy, and test coverage (happy path, edge, error, regression,
interactions). Two correctness/honesty defects found and **fixed inline**; the implementer's six
honest-flag deferrals were verified to be genuinely deferred and already tracked.

### Fixed inline (minor)

- **Hotness signal dropped on the `done` short-circuit (`seeker-walk-client.ts`).**
  `maxChildCohortCount` was folded only *after* the immediate-match `done` return and the
  `traffic === undefined` return inside `handleAccepted`. So when a single hot cohort's immediate query
  already met `wantCount`, the walk returned with `maxChildCohortCount = 0` — contradicting the field's
  own doc ("max ... across **every** `Accepted` reply this walk saw") and, materially, defeating the
  sweep escalation for a *small quorum on a genuinely hot topic*: `createMatchmakingQuorumDiscovery.walk`
  reports `childCohortCount`, and `VotingQuorumAssembler` escalates only when `> 0`, so the assembler
  would have taken a prefix-biased single-cohort sample on exactly the hot proposals the sweep exists to
  cover. **Fix:** record the hotness signal as soon as the reply's `topicTraffic` is available, before the
  `done`/undefined short-circuits. Control flow is otherwise untouched. Regression tests added
  (`seeker-walk-client.spec.ts`): the done-path-still-records case and the fold-the-max-across-tiers case.

- **`aggregateTrusted` claimed trust that was never established (`multi-cohort-seeker.ts`).**
  `runMultiCohortSweep` returned `aggregateTrusted: true` on success even when **no `verifyAggregate`
  port was injected** — nothing was verified — contradicting the field doc ("`false` when absent or
  invalid"). The sweep correctly *proceeds* without a verifier (every entry is `registrationSig`-re-
  validated, so a forged aggregate can at worst mis-steer shard selection, never inject providers), but
  the flag must stay honest. **Fix:** `aggregateTrusted` is now asserted only when a verifier actually ran
  and passed. No caller gates on the flag (`module.ts` / the quorum binding read only `.providers`), so
  behaviour is unchanged; only the reported metadata is now correct. Regression test added
  (`multi-cohort-seeker.spec.ts`): no-verifier ⇒ `aggregateTrusted = false`, providers still unioned.

### Checked — no change needed

- **Advisory trust model end-to-end.** Confirmed every shard/walk entry is re-validated via
  `verifyProviderEntry(topicId, …)` before counting (sweep, walk, and `VotingQuorumAssembler`), so a
  lying shard primary or a forged aggregate buys nothing. The `AggregateCountV1` threshold envelope
  (`signature` + `signers`) mirrors `PromotionNoticeV1`/`DemotionNoticeV1`; the divergence from the
  previously-frozen single-`signature` shape is justified (a threshold blob is unverifiable without the
  aligned signer set) and documented in `matchmaking.md` §Wire formats — **accepted**.
- **`logBucketCount` / `selectShards` over-provision invariant.** Rounds down (power-of-two floor) so the
  summed bucketed counts under-estimate the real population; selection over-provisions, never under-
  selects. Verified against the producer's emission and the `DEFAULT_SWEEP_OVERPROVISION = 1` default.
- **Wire validation.** `validateAggregateCountV1` requires `signers` as a base64url string array;
  `aggregateCountSigningPayload` sorts buckets by `(targetTier, prefixSlot)` so signer/verifier recompute
  identical bytes (order-independence test present). `validateAggregateCountV1` does not enforce
  `signers.length ≥ minSigs` — correctly left to the (deferred) verification layer.
- **Reputation bridge.** `reportTrafficCrossCheck` maps to the real `IPeerReputation.reportPeer(…,
  PenaltyReason.ProtocolViolation, …)` API (verified signature/enum), skips unattributed signals, and
  scores nothing itself — the aggregation/decay/penalty policy stays the reputation subsystem's job.
- **Error handling / resource cleanup.** Session `requireManager()` throws a clear pre-`register()` error
  (tested); the walk withdraws politely before each escalation; no leaked timers/handles (clock + sleep
  injected). Type safety: both packages `tsc`-clean; the optional-property spreads are exactOptionalProperty-safe.
- **Docs.** `matchmaking.md` (sweep / adversarial / public-API "implemented, mock-tier-e2e pending" +
  the two GROUNDING resolutions + the `signers` envelope), `architecture.md` (matchmaking now implemented
  across its full surface; Doc Sync mock-tier honestly still pending), and `internals.md` (§Matchmaking
  subsystem additions) read true against the code after the two fixes (which align the code *to* the
  already-correct field docs — no doc edits required).

### Deferred (not this ticket — already tracked, verified honest)

The implementer's six known gaps are genuine follow-on work, not omissions: (1) no FRET-host wiring of
`shardCounts`/walk transports in `cohort-topic/host.ts`; (2) the `AggregateCountV1` threshold-*verification*
binding (split blob + check against membership cert) is unbuilt — the sweep trusts unless a verifier is
injected (now flagged honestly, see fix above); (3) `boundReportedTraffic` is not yet *called inside*
`SeekerWalkClient` (the walk enforces the bounds structurally; this is the explicit signal-emission
wiring); (4) untuned `overReportSuspectRatio`/under-report heuristics with no simulator backing; (5)
`logBucketCount` round-down coupling to `selectShards`; (6) sweep queries all selected shards (no early
stop). All are bounded-harm and land in **`13-matchmaking-e2e-mock-tier`** (implement,
`prereq: matchmaking-sweep-adversarial-module`), which builds the mock-mesh harness that necessarily binds
these ports. No new ticket filed — filing one would duplicate ticket 13's scope.
