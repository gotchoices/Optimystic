# Review (complete): invalidation certificate — target binding (#2) + per-arbitrator dedup (#3)

Part 1 of 2 of the forgeable-invalidation-certificate fix. Closes **#2 target binding** (a genuine
`challenger-wins` proof for transaction X can no longer be replayed to revert an unrelated, innocent
transaction Y) and **#3 per-arbitrator dedup** (one signed vote replicated N× can no longer manufacture
a super-majority). **#1 arbitrator-set legitimacy** remains the follow-up
`7.8-invalidation-cert-arbitrator-set-binding` (filed, present in `tickets/review/`) and is intentionally open.

## Implementation summary (as reviewed)

- **Wire format v2 (target-bound).** Arbitration votes sign `v2:${disputeId}:${vote}:${computedHash}:${targetHash}`
  with `targetHash = hashString(`${messageHash}|${invalidatedActionId}|${sortedBlockIds}`)`. `blockIds`
  are lexically sorted so the binding is order-independent. A required `version: 'v2'` discriminator was
  added to `ArbitrationVoteProof` (db-core) and `ArbitrationVote` (db-p2p); the verifier rejects
  absent/unrecognized versions before counting (v1 never accepted-by-default).
- **Verifier** `verifyInvalidationCertificate(proof, target: CertificateTarget)` recomputes `targetHash`
  from `proof.messageHash` + the apply-path target and verifies each signature over the v2 payload. A
  mismatched target fails every signature → 0 decisive → reject. Decisive verdicts are deduped per
  `arbitratorPeerId`; an arbitrator with two *different* decisive votes is marked `equivocated` and
  dropped from both tallies.
- **Callers.** `applyConsensusInvalidation` (network-facing) verifies against the request's own
  `{invalidatedActionId, blockIds}` — the replay boundary. `applyInvalidation` defaults to its own
  target; `cascadeInvalidate` threads the *root's* target (`rootCertificateTarget`) into each child
  because children reuse the root proof. `DisputeService` derives the vote target from the challenge's
  `originalRecord` via `extractInvalidationTarget`, identical to the originator/verifier derivation.

## Review findings

### Verified (and where)
- **Read the full implement diff first** (commit `d508402`): `struct.ts`, `types.ts`, `invalidation.ts`,
  `cluster-repo.ts`, `dispute-service.ts`, `cascade.ts`, `index.ts`, `docs/right-is-right.md`, and every
  touched test, before reading the handoff summary.
- **Target-binding correctness** — `computeTargetHash` sorts `blockIds` (order-independent) and uses the
  shared `hashString`; verifier and signer build the identical v2 payload string. Exercised by
  `invalidation.spec.ts` (matching-target accept, different-action/different-blocks reject,
  block-order-independence) and `cluster-invalidation.spec.ts` (apply-path replay reject without
  invoking the sink).
- **Dedup / equivocation** — traced multiple vote orderings by hand (dup→counts once; conflicting→
  `equivocated` regardless of arrival order; inconclusive never tallied). Matches `invalidation.spec.ts`
  "duplicated arbitrator counted once" and "drops an equivocating arbitrator from both sides".
- **Security boundary of the cascade decoupling** — confirmed via grep that `cascadeInvalidate` has **no
  production caller** (only `index.ts` re-export + tests) and `applyInvalidation` is called only from
  `cascade.ts` in `src/`. The network path (`applyConsensusInvalidation`) never sets `certificateTarget`,
  so the only code that decouples verified-target from reverted-target is the local, deterministically-
  derived cascade. No path lets a peer-supplied request reach `applyInvalidation` with an attacker-chosen
  `certificateTarget`. The rationale in the handoff holds.
- **Wire-safety of the new `version` field** — dispute messages serialize via `JSON.stringify`/`JSON.parse`
  (`dispute/service.ts`); there is no protobuf schema for these proofs (`network/struct.ts` only
  references the type). The new field round-trips like the existing `signature`/`computedHash` fields.
- **Cross-side target consistency** — originator (`maybeInvalidate`), arbitrator
  (`computeChallengeTargetHash`), and verifier (`applyConsensusInvalidation`) all derive the target from
  the same `extractInvalidationTarget` over the same record, so they agree by construction.
- **Build + tests** — `db-core` build clean, `yarn test` **906 passing**; `db-p2p` build clean (tsc
  includes `test/`), `yarn test` **906 passing / 30 pending / 0 failing**. Targeted dispute suite green.

### Fixed inline (minor)
- **Closed the flagged end-to-end gap.** The handoff noted there was "no end-to-end dispute→originate→apply
  test asserting a real `DisputeService`-produced proof verifies against the real `InvalidateRequest`
  target." Added assertions to the existing `dispute.spec.ts` "originates an invalidation …" test: the
  real arbitrator-produced proof now must `verifyInvalidationCertificate(... request's own target ...) ===
  true` and `=== false` against an unrelated action id. This exercises the cross-side consistency that was
  previously only "by construction" (and would catch a future divergence in `extractInvalidationTarget`).
  `dispute.spec.ts` rebuilds clean and the suite stays green (assertions added inside an existing `it`,
  so the case count is unchanged).

### Observations (examined, no action — correct as-is)
- **A single deduped decisive challenger vote passes** (`total=1`, `threshold=ceil(1·2/3)=1`, `1≥1`). This
  is not a regression: it mirrors `DisputeService.resolveDispute`, which is the protocol's own rule.
  Establishing a *minimum legitimate cohort size* is precisely #1's job (arbitrator-set binding), correctly
  deferred to 7.8. Dedup still does its job: it stops a sub-threshold genuine set (e.g. 1 challenger + 1
  majority) from being inflated to a super-majority by replication — verified by the dedup test.
- **`resolveDispute` does not dedup**, but the verifier re-derives the challenger super-majority
  independently from the deduped votes and only trusts `outcome === 'challenger-wins'` as a claim, so the
  un-deduped origination tally cannot be trusted into a pass.
- **Empty-target votes** (arbitrator can't extract a target from `originalRecord`) are signed but can never
  match a real reversal, because the originator returns early when no target is extractable. Fails closed.

### Major findings → new tickets
- **None.** The remaining unforgeability gap (#1 arbitrator-set legitimacy) is already filed as

  `ApplyInvalidationParams.certificateTarget`, and the cascade code comments; it is inert today (cascade is

## Out of scope (deferred to 7.8)
Arbitrator-set legitimacy (#1): votes are still not bound to the legitimately-selected cohort, so a peer
that can mint Ed25519 keypairs can present a synthetic set. `docs/right-is-right.md` §Durable Invalidation
correctly states #2/#3 hold and #1 is open; the unqualified "cannot forge" guarantee was deliberately not
restored.
