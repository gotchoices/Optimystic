# Pre-existing (flaky) test failure

## Command

```
cd packages/db-p2p && yarn test
```

(package: `@optimystic/db-p2p`)

## Failing test

```
Fresh-node DDL (multi-node, real production stack)
  Scenario B — 5-node cold-start with one peer down at boot
    DDL on A completes with peer E unreachable; SELECT on B sees the write

AssertionError: expected undefined to deeply equal { key: 42, value: 'despite-E-down' }
  at Context.<anonymous> (test/fresh-node-ddl-multi.spec.ts:116:47)
```

## Why this is pre-existing / not caused by this ticket

- **Flaky, not deterministic.** On the first full-suite run it failed (464 passing, 1 failing).
  On an immediate re-run of the identical command it passed (465 passing, 0 failing).
  Run in isolation (`mocha test/fresh-node-ddl-multi.spec.ts`) both scenarios pass in ~60ms.
  The flake only appears under full-suite timing/load (multi-node libp2p cold-start with a
  peer intentionally down — sensitive to scheduling/propagation timing).
- **Outside this ticket's diff.** This ticket (`1-optimystic-block-change-notification`) only
  adds per-collection change notification at the storage commit funnel
  (`db-core/src/transactor/change-notifier.ts`, `StorageRepo.commit`/`internalCommit`,
  `NetworkTransactor.onCollectionChange`, node `blockChangeNotifier` exposure). It touches none
  of the multi-node DDL / consensus / propagation path the failing test exercises. The added
  `emitCollectionChanges` is a no-op when no listeners are registered (this test registers none),
  and `internalCommit`'s new return value is ignored on the read path — commit behavior/results
  are otherwise unchanged.

## Work-around applied

None needed — the flake does not block this ticket's own tests. The 7 new change-notification
tests in `test/storage-repo.spec.ts` pass deterministically in isolation and within the full suite.
